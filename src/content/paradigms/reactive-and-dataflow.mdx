# Reactive and dataflow programming

**Reactive programming** treats values as **streams that change
over time**, with computations that *automatically re-run* when
their inputs change. The slogan: "spreadsheets, but for
programs."

**Dataflow programming** is the broader paradigm: programs are
graphs of operations, values flow along edges, downstream
operations run when upstream values arrive. Reactive is a
dataflow variant where the primary thing flowing is "values
changing over time."

The two paradigms have been quietly taking over UI and data
pipelines. They're worth understanding as a paradigm — not just
as a "thing RxJS does."

## The spreadsheet analogy

In Excel, `C1` says `=A1 + B1`. When you change `A1`, `C1`
updates automatically. You didn't write code to re-add — the
*formula* declared the relationship, and the engine kept
things in sync.

That's the reactive paradigm in a nutshell:

- **Cells** are values that can change.
- **Formulas** are computations that depend on other cells.
- The engine **propagates** changes through the dependency
  graph, re-running formulas whose inputs changed.

Now imagine an entire program structured that way: state as
"cells," derived values as "formulas." That's what modern
reactive UI frameworks are doing.

## Signals: the modern primitive

In 2024–2026, the dominant reactive primitive is the
**signal** (Solid, Preact Signals, Vue 3 refs, Svelte 5 runes,
Angular signals, and increasingly React via projects like
Jotai/Recoil). Conceptual API:

```ts
const count = signal(0);
const doubled = computed(() => count() * 2);

effect(() => {
  console.log('count is', count(), 'doubled is', doubled());
});

count.set(5);
// effect logs automatically: 'count is 5 doubled is 10'
```

The magic: `computed` and `effect` *track which signals they
read at runtime.* When any of those signals updates, the
engine knows exactly which computeds/effects to re-run, and
in what order — derived from the dependency graph it built on
the fly.

You didn't subscribe. You didn't write a listener. You wrote
`doubled = count() * 2` and the system kept it correct.

## Streams: the older reactive primitive

Before signals went mainstream, **streams** (RxJS, ReactiveX,
Bacon.js, Akka Streams) were the dominant reactive shape. The
abstraction: a stream is a *sequence of values over time*, and
you apply functional operators (map, filter, debounce, merge,
combineLatest) to transform them.

```ts
fromEvent(input, 'input')
  .pipe(
    map((e) => (e.target as HTMLInputElement).value),
    debounceTime(300),
    switchMap((q) => fetch(`/api/search?q=${q}`).then((r) => r.json())),
  )
  .subscribe((results) => render(results));
```

Reading top to bottom: events → values → debounced → mapped to
fetches → rendered. The whole search-with-debounce-and-cancel
flow is one pipeline. That's the dataflow paradigm pushed
hard.

The trade with streams vs signals:

- **Streams** are great for *events over time* — keypresses,
  network responses, user gestures. They handle async,
  cancellation, and composition naturally. They can feel
  heavy for static "this depends on that."
- **Signals** are great for *state that changes* — the
  current count, the user's name. Simpler mental model, less
  ceremony. Async needs adapters (`from(promise)` style).

Mainstream reactive frameworks have been quietly moving toward
*both*: signals for state, streams for events. Solid, Svelte 5,
and Vue 3 are roughly there.

## Dataflow: the broader paradigm

Step back from UI: the **dataflow paradigm** is "a program is
a graph; values flow along the edges; nodes compute when
inputs arrive."

You've used dataflow systems even if you've never called them
that:

- **Build systems** (Make, Bazel) — files flow through
  compilation rules.
- **Stream processors** (Kafka Streams, Flink, Spark
  Structured Streaming) — events flow through stateful
  operators.
- **ETL pipelines** (Airflow, Dagster, Prefect) — datasets
  flow through stages with explicit dependencies.
- **Audio/video pipelines** (GStreamer, ffmpeg filtergraphs) —
  samples/frames flow through filters.
- **ML frameworks' computation graphs** (TensorFlow's
  original eager-vs-graph mode, JAX, PyTorch's TorchScript) —
  tensors flow through ops.
- **GPU shader pipelines** — vertex → fragment → output,
  parallel across thousands of cores.

Dataflow scales spectacularly when:

1. The graph is large and the system can parallelize across
   it.
2. Inputs are reusable — if a stage's inputs are unchanged,
   the cached output stands (this is what makes Bazel fast).
3. Nodes are pure (no side effects) — they can be re-run
   anywhere, in any order, on any machine.

Functional purity and dataflow are kindred spirits.

## Push vs pull

There are two ways to implement "propagate changes":

- **Push (eager).** When a source changes, immediately re-run
  every downstream. RxJS subscriptions, classic
  ReactiveExtensions, signal libraries with eager
  propagation.
- **Pull (lazy).** Downstream values are recomputed *when
  asked*. The system tracks which sources have changed and
  invalidates cached values; the next read recomputes. React
  with `useMemo`, MobX, lazy signals.

Pull is more efficient when most of your derived state isn't
being looked at every tick. Push is simpler to reason about
when everything *is* being looked at. Real systems usually
blend (push invalidations + lazy recomputation, also called
"reactive cache" — what Solid and Preact Signals roughly do).

## The "glitch" problem

A subtlety reactive systems have to solve: if `c = a + b` and
both `a` and `b` change, does `c` get computed once (with both
new values) or twice (once after `a` changes with old `b`,
then again after `b` updates)?

- **Glitches** are intermediate inconsistent states observed
  by the consumer.
- Good systems use **batching** + **topological propagation**:
  collect all changes in a tick, then propagate in dependency
  order, computing each derived value at most once per tick.

This is one of the reasons Excel doesn't update `C1` for each
intermediate change when you paste a range — it batches.

## Reactive in business code: events and ETL

Beyond UI, reactive thinking dominates **event-driven backends**
and **streaming data**:

```
Kafka topic ──> Kafka Streams ──> aggregations ──> output topic
                     │
                     ├── windowing
                     ├── joins with other streams
                     └── state stores
```

Kafka Streams (and Flink, Spark Structured Streaming) treats a
topic as an unbounded stream. You write transformations as
operators. The framework handles partitioning, state, and
fault recovery.

The win: a query like "5-minute moving average of clicks per
user, joined with the user's plan tier" is a few lines of
declarative pipeline. The framework figures out the dataflow.

## When reactive/dataflow is the right paradigm

- **UI.** Modern UI is reactive top to bottom because UI
  *is* "state changing over time, rendered to a screen."
- **Streaming data.** Anything event-shaped with windows,
  aggregations, joins.
- **Pipelines with caching.** Build systems, ETL,
  ML training. Dataflow with cached outputs is how these
  scale.
- **Realtime collaborative anything.** CRDTs + reactive
  glue. Multiple sources of truth merging continuously.
- **Spreadsheets, naturally.** The original killer app.

## Where it's a bad fit

- **Imperative, ordered workflows.** "Then do this. Then do
  that. If error, roll back step 2." A migration script.
  Don't force this into reactive pipelines.
- **Heavy random-access state.** Reactive shines when data
  flows in one direction; cyclic graphs and arbitrary mutation
  break the model.
- **Small programs.** A 200-line script with one `for` loop
  doesn't need a reactive framework. The model is overhead
  until the dataflow graph gets meaningful.

## The takeaway

Reactive programming treats values as streams (or signals)
that change over time, with derived values automatically kept
in sync. Dataflow generalizes: programs are graphs of pure
ops, values flow, the runtime schedules computation. UIs,
build systems, streaming data pipelines, and ML graphs all
share this paradigm. The wins are automatic propagation,
parallelism, and incremental computation; the cost is a model
that feels overkill until the graph grows enough to pay back
its setup.
