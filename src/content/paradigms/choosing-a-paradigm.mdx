# Choosing a paradigm

A senior engineer picks the right paradigm for the piece of
code in front of them. A staff engineer picks the right
paradigm for the **codebase**, the **team**, and the **decade**
— knowing that paradigm choice is partly technical, partly
social, and partly economic.

The technical choice is the easy part. This page is about the
rest.

## The forces in tension

A paradigm choice is rarely "what's best in the abstract."
It's the intersection of:

- **Problem shape.** Some problems are intrinsically
  entity-shaped, others transformation-shaped, others
  event-shaped. Fighting the shape costs you forever.
- **Team familiarity.** The best paradigm for the problem is
  worthless if the team can't read it next month. A codebase
  that requires onboarding a custom mental model for each new
  hire is taxing growth.
- **Language and ecosystem grain.** Every language has a
  paradigm it's *easy* to write and others that fight the
  language. Java with monad transformers is doable and ugly.
  Haskell with mutable global state is doable and missing the
  point.
- **Toolchain maturity.** Debuggers, profilers, IDE support,
  library quality. Mainstream paradigms in mainstream
  languages get the best tools.
- **Hiring market.** "We use F# / Haskell / Rust / Clojure" is
  both a magnet (for some) and a wall (for others).
- **Cost of being wrong later.** Some paradigm choices are
  reversible cheaply; some are baked into 50,000 files.

A staff engineer is reasoning about all of these at once. The
answer isn't always "use the most powerful paradigm" — it's
often "use the boring one that the team already knows, unless
the problem genuinely demands more."

## A decision framework

The questions that have actually moved real decisions:

**1. What is the shape of the dominant work?**

- Mostly stateful business entities → OOP-leaning.
- Mostly data transformation pipelines → FP-leaning.
- Mostly events / reactive state → event-driven + reactive
  with FP or OOP as glue.
- Mostly graph queries / search / deduction → reach for a
  solver or declarative DSL; don't write the search yourself.
- Mostly straight-line algorithms → procedural is fine.

**2. Where does the complexity live?**

- In the *data*? FP / declarative. Algebraic data types,
  pattern matching, immutable values.
- In the *coordination*? OOP / actor / event-driven. Entities
  with identity, supervised lifecycle, messages.
- In the *math*? Lean as functional as the language allows.
- In the *integration with external systems*? OOP-shaped
  service objects with clean interfaces.

**3. What's the team's center of gravity?**

- If 80% of the team is fluent in one paradigm, the *good*
  argument for switching has to be strong, because you're
  going to absorb a long onboarding cost.
- If the team is mixed, lean toward the *boring middle* —
  modern OOP with FP-flavored data layers is the boring middle
  in 2026.

**4. What's the language's grain?**

- Haskell, OCaml, Elm: lean FP, fight to do OOP.
- Java, C#, Kotlin, C++: lean OOP, but happy to do FP at the
  module/method level.
- TypeScript, Python, Rust, Swift, Scala: genuinely
  multi-paradigm. The choice is more cultural than technical.
- Go: deliberately *not* multi-paradigm. Procedural-with-
  interfaces. Trying to do "real" FP or OOP is fighting Go.
- SQL, Datalog, Prolog: declarative/logic — use them *for the
  queries*, not as your whole codebase.

**5. How often does this code change, and by whom?**

- Code that rarely changes can afford to be clever.
- Code that changes every week needs to be boring.
- Code that many teams touch needs the *most legible* paradigm
  — the one that imposes the lowest mental tax on outsiders.

## The staff-engineer failure modes

Things I've watched go wrong (and occasionally caused):

**1. Paradigm bingo.** "We're a microservices, event-driven,
DDD, functional-core shop." Five paradigms stacked because each
sounded compelling on a conference talk, nobody bought in
deeply, all five half-implemented. The result is the *cost* of
five paradigms with the *benefit* of one and a half. Pick one
or two; commit deeply; let the others be tactical.

**2. Premature abstraction by paradigm.** Introducing a full
DI container, an effect system, or an actor framework on a
small codebase because "we'll need it when we scale." You
won't, and the abstraction is taxing your hiring and onboarding
*now*. Reach for these when their cost is small relative to
the system you're already paying for.

**3. Religious purity.** Refusing to mutate anything because
"we're FP," even when a hot inner loop screams for it.
Refusing to let a service depend on anything but interfaces,
even when there's only one implementation. Letting the
paradigm dictate when the code disagrees.

**4. Underselling the team.** Assuming the team can't learn
something new because *you've* never used it. Often the right
move *is* to introduce a paradigm carefully — but earn it by
showing the wins on a small, contained piece first.

**5. Overestimating swapping cost.** Some paradigm choices
*are* easy to reverse. A handful of services rewritten over a
quarter, sure. A 200k-line codebase whose every domain object
is a class hierarchy — that's not getting refactored to be
FP-first in a quarter no matter how much you want to.

## The economics

A paradigm change has costs that don't show up in code review:

- **Hiring.** A Haskell role takes longer to fill than a Java
  role; a Clojure role takes longer than a Python role. Maybe
  the candidates are better; maybe not.
- **Onboarding.** New engineers ramp slower on unfamiliar
  paradigms. Months, not weeks.
- **Tooling.** Mainstream paradigms get profilers, debuggers,
  AST tools, code generators, AI completions. Off the path,
  you're hand-rolling.
- **Library ecosystem.** A boring paradigm in a boring
  language gives you a *huge* library ecosystem. A novel
  paradigm gives you fewer batteries and more home-grown
  wheels.
- **AI assistance quality.** As of 2026, AI coding tools are
  noticeably stronger on mainstream paradigms in mainstream
  languages because that's where the training data is.

None of this means "always pick the boring thing." It means
the bar for picking the un-boring thing should be *real*: a
specific class of problem the paradigm uniquely solves,
expected to dominate the codebase, with a team that can carry
it.

## A pragmatic default for 2026

If you were starting a new business-software codebase today
without strong reasons otherwise:

- **TypeScript** (or Kotlin / C# / Rust depending on the
  domain).
- **OOP at the service / module boundary**, with dependency
  injection via constructor parameters. Not because OOP is
  best, but because the rest of the world will be able to read
  it without translation, and most service-shaped code wants
  service-shaped abstractions.
- **FP-style domain code inside**: immutable values, pure
  functions, ADTs (discriminated unions in TS), pattern
  matching where supported.
- **Declarative where engines exist**: SQL for data, regex
  for patterns, IaC for infra, declarative UI frameworks.
- **Event-driven** for integration boundaries and UI.
- **Reactive primitives** for UI state (signals, hooks).
- **Concurrency** via async/await + channels or actors, not
  raw threads + locks. (See the concurrency topic.)

This isn't the *coolest* stack. It's the stack with the lowest
hidden taxes, where the paradigm choices have the most
support, and where the team can grow without each new hire
needing a paradigm tutorial.

## When to break the default

There are real cases where the boring default is wrong:

- **Domain is dominated by algebraic reasoning.** Compilers,
  type checkers, theorem provers, query optimizers. Reach for
  Haskell/OCaml/Rust + functional style.
- **Domain is dominated by hard concurrency.** Telecom,
  multiplayer game servers, distributed coordination. Reach
  for actors (Erlang/Elixir) or strict CSP (Go).
- **Domain is dominated by hard performance.** Game engines,
  HFT, databases, codec internals. Reach for C/C++/Rust + data-
  oriented design.
- **Domain is dominated by declarative reasoning.** ETL,
  analytics, infra. Reach for SQL, Datalog, dataflow systems.
- **Strong team buy-in.** A team of senior FPers will be
  vastly more productive in F# than the same team forced into
  Java. The team is the largest single variable.

The reverse: a team of junior OOPers forced into Haskell will
likely write some of the most opaque code you've ever seen.
Not because Haskell is bad, but because the *cultural fit*
matters.

## The takeaway

Paradigm choice is partly about the problem and largely about
the team, the language, the ecosystem, and the cost of being
wrong. The boring default — mostly-OOP at the boundaries, FP
in the domain core, declarative where engines exist, reactive/
event-driven at the edges — is boring because it's reasonable.
Deviating from it should be a deliberate move tied to a real
class of problem and a team that can sustain the choice.

Mostly, the cost of getting the paradigm slightly wrong is
small if the *code* is well-factored. Encapsulated state, pure
functions where you can swing them, narrow interfaces, small
modules — these matter more than the label. Get the
fundamentals right and the paradigm becomes a label rather
than a verdict.
