# Classes vs prototypes vs traits

OOP isn't one mechanism — it's several. The way "an object has
behavior" actually works under the hood differs between
languages, and the differences shape how the language *feels*
to use. Three big families:

- **Class-based** (Java, C#, C++, Python, modern JS) — behavior
  is defined on classes; objects are instances.
- **Prototype-based** (JavaScript under the hood, Lua,
  Self) — objects inherit directly from other objects; there
  are no classes in the strict sense.
- **Trait / protocol-based** (Rust, Swift, Scala, Haskell type
  classes) — behavior is defined separately from data, and
  attached to types that opt in.

Understanding which one a language *actually* uses explains a
lot about why its OOP feels the way it does.

## Class-based OOP

The dominant industrial flavor. A class is a static blueprint:
fields, methods, a constructor. Objects are instances of a
class. Each instance gets its own fields; methods are shared
via the class.

```java
class Counter {
    private int n = 0;
    public void increment() { n++; }
    public int value() { return n; }
}
```

Method lookup is conceptually simple: on `c.increment()`, the
runtime looks up `increment` on `Counter`. If not found, walk
up the superclass chain.

Strengths:

- **Predictable.** What the class declares is what an instance
  has. Tooling (autocomplete, refactoring) is easy.
- **Compile-time checks.** The compiler knows the layout. You
  can have static dispatch where possible, and even when
  dispatch is virtual, the vtable is fixed at class definition.
- **Inheritance + polymorphism in one mechanism.** Familiar to
  most working programmers.

Weaknesses:

- Adding behavior to a type you don't own is awkward (no
  "extension methods" in classic Java; you wrap or extend).
- The class hierarchy is a *commitment* at definition time.

## Prototype-based OOP

JavaScript is the famous example, but Self (Xerox PARC, 1986;
later Sun) was the original. There are no classes — only objects. An object has
**properties**, and a pointer to another object called its
**prototype**. On a property lookup, the runtime checks the
object itself; if absent, follows the prototype chain.

```js
const animal = {
  speak() { return `${this.name} makes a sound`; },
};

const dog = Object.create(animal);
dog.name = 'Rex';
dog.bark = function () { return `${this.name} woofs`; };

dog.speak();   // 'Rex makes a sound' — found on prototype
dog.bark();    // 'Rex woofs' — found on dog itself
```

`class` syntax in JavaScript (ES2015+) is **sugar** over this:
it creates a constructor function and sets up the prototype
chain. The underlying model is still prototypal — and you can
still see this when you `Object.getPrototypeOf(instance)` or
when monkey-patching `Array.prototype`.

Strengths:

- **Extremely flexible.** Add a method to one object without
  affecting others; mutate a prototype to affect everything
  inheriting from it (rarely a good idea, but possible).
- **No class/instance distinction.** Objects can extend other
  objects directly.

Weaknesses:

- **`this` is dynamic** and confusing — depends on the call
  site, not where the function is written.
- **Performance was historically bad** because lookups had to
  walk a chain. Modern engines (V8) use **hidden classes** to
  recover class-like speed: they infer a hidden "shape" from
  property additions and optimize accordingly. This is also
  why creating an object's properties in a consistent order
  matters for V8 performance.
- **Tooling is harder** when shape isn't declared up front.
  TypeScript exists partly to add the declarations back.

## Trait / protocol-based OOP

Traits (Rust), protocols (Swift), type classes (Haskell), and
type-class-like interfaces with default methods (Scala, Java 8+)
share a key idea: **separate data from behavior, then attach
behavior to types after the fact.**

```rust
trait Speak {
    fn speak(&self) -> String;
}

struct Dog { name: String }
struct Cat { name: String }

impl Speak for Dog {
    fn speak(&self) -> String { format!("{} woofs", self.name) }
}
impl Speak for Cat {
    fn speak(&self) -> String { format!("{} meows", self.name) }
}

fn announce<T: Speak>(thing: &T) {
    println!("{}", thing.speak());
}
```

`Dog` and `Cat` are plain structs. `Speak` is a trait declared
*separately*. Implementations are attached with `impl Speak
for Dog`. Code that wants polymorphism takes anything that
implements `Speak`.

Why this is interesting:

1. **No inheritance, no diamond problem.** A type can
   implement many traits; traits don't inherit state, only
   require methods. The combinatorial mess of multiple
   inheritance is sidestepped.
2. **You can implement traits for types you don't own.**
   Subject to coherence rules (Rust's "orphan rule"), you can
   teach `String` a new trait from your crate. Java forced
   adapter/wrapper classes for this; Rust just lets you `impl`.
3. **Static dispatch is the default.** `announce<T: Speak>`
   monomorphizes — the compiler emits a specialized version
   per type, like C++ templates. You can opt into dynamic
   dispatch with `dyn Speak`.

Haskell **type classes** are the original idea (Wadler,
Blott — 1989). Equality, ordering, serialization, all defined
as type classes rather than baked into a class hierarchy.

## Why this matters in practice

- Writing **JavaScript** without understanding prototypes is
  fine until you hit a weird `this` bug or try to extend a
  built-in. Then it isn't.
- Writing **Rust** as if it were Java is a recipe for
  frustration. Rust *intentionally* doesn't have inheritance —
  the answer is composition + traits, not "fight the compiler
  until it lets me extend."
- Writing **TypeScript**, you have full classes, plus
  structural interfaces, plus the JS prototype chain under
  the hood. The structural typing especially makes TypeScript
  feel a bit trait-like — a function can take `{ speak():
  string }` and any matching shape works, without explicit
  `implements`.
- Languages adding **default interface methods** (Java 8+, C#
  8+) are quietly turning interfaces into traits. The line
  between "interface" and "trait" is blurring.

## The takeaway

OOP isn't one model. Classes are the dominant industrial form;
prototypes are JS's underlying machinery (and the reason `this`
is weird); traits/protocols are the modern alternative that
sidesteps inheritance's pitfalls while keeping polymorphism.
Knowing which model a language uses tells you a lot about *why*
its OOP code is written the way it is — and what idioms (like
"extend this base class" vs "implement this trait") will fit
the grain of the tool.
