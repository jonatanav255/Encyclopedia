# Logic programming

**Logic programming** is the paradigm where you state **facts**
and **rules**, and the runtime *searches* for values that make
your queries true. You don't write algorithms — you write a
specification, and a solver does the work.

The canonical language is **Prolog** (1972, Colmerauer &
Roussel). It's niche in industry now, but its descendants are
everywhere you don't expect them: type inference, query
languages like Datalog, package solvers (apt, npm/pnpm peer
deps, Cargo's resolver), constraint engines, formal
verification, theorem provers.

## The flavor: Prolog in five lines

```prolog
parent(tom, bob).
parent(bob, alice).
parent(bob, charlie).

ancestor(X, Y) :- parent(X, Y).
ancestor(X, Y) :- parent(X, Z), ancestor(Z, Y).

?- ancestor(tom, alice).   % true
?- ancestor(tom, X).        % X = bob ; X = alice ; X = charlie
```

What just happened:

- Three **facts** (`parent(...)`).
- Two **rules** (`ancestor(...)` defined in terms of itself
  and `parent`).
- A **query** asks the system to find values that satisfy a
  goal. The runtime performs **unification** (matching
  variables against terms) and **backtracking search** (trying
  alternatives when a branch fails).

You didn't write a loop, a graph traversal, or a base case.
You stated *what `ancestor` means*. The engine found all the
answers.

## How the engine works (in broad strokes)

A Prolog-style runtime tries to prove a goal by working
backwards:

1. **Match the goal against a clause.** `ancestor(tom, X)`
   matches the first rule by unifying `X_rule = tom` and
   `Y_rule = X`. Now the engine needs to prove
   `parent(tom, X)`.
2. **Try facts.** `parent(tom, bob)` matches; the engine binds
   `X = bob`. The first answer is `X = bob`.
3. **Backtrack.** Asked for more answers? Undo the binding,
   try the *second* rule: `parent(tom, Z), ancestor(Z, X)`.
   Find a `Z` that's a child of `tom`, then recurse.
4. **Repeat until no more proofs work.**

The interesting properties:

- **You can run the program backwards.** `?- ancestor(tom, X)`
  finds descendants of Tom. `?- ancestor(X, alice)` finds
  ancestors of Alice — *same rules, different binding pattern.*
  Most programming languages don't let you do that.
- **Search is built in.** No need to write DFS yourself; the
  evaluation strategy *is* depth-first search with
  backtracking.

## Datalog: the practical descendant

Pure Prolog is hard to scale (search trees can blow up, the
language has imperative escape hatches that complicate things,
queries can be undecidable). **Datalog** strips Prolog down to
a decidable subset and turns out to be *very* useful for
querying graph-shaped data.

Datalog underpins:

- **Datomic** (database whose query language is Datalog).
- **Soufflé** (high-performance Datalog used in static
  analysis tools).
- **Logica** (Google's analytical Datalog over BigQuery).
- **Recursive CTEs in SQL** (which encode a form of Datalog).
- **Static analyzers** for security, taint tracking, alias
  analysis — many use Datalog under the hood because graph
  queries with transitive closure are exactly what it's good
  at.

A typical SQL query expresses "which user IDs join through
roles join through permissions to X?" — and gets ugly fast.
The same query in Datalog is a few rules.

## Where logic programming shows up without you noticing

You've used logic programming today, even if you've never
written Prolog:

- **Type inference.** Hindley–Milner-style unification is the
  same machinery Prolog uses. The compiler is *solving for*
  the types that make your program well-typed.
- **Regex engine compilation.** A regex pattern is compiled
  to a state machine; the engine searches for a match.
  Different paradigm in form, similar in spirit.
- **Package resolvers.** npm/pnpm/yarn (peer-dep resolution),
  Cargo, apt, dnf — all solve constraint problems. Modern
  pip switched to a real SAT-style solver because the older
  greedy resolver kept producing broken installs.
- **Build systems.** Bazel computes which targets are stale
  and what order they can run in.
- **Constraint propagation in spreadsheets, layout engines,
  and Sudoku solvers.**
- **Theorem provers and verifiers** — Coq, Isabelle, Lean,
  TLA+, Z3. The latter (Z3, an SMT solver) is used inside
  modern static analyzers and symbolic execution tools.

When you see "the system figures out a valid assignment of
values," you're seeing logic programming, even if the surface
syntax looks nothing like Prolog.

## Strengths

- **Specifications, not algorithms.** Stating the rules is
  often shorter and clearer than writing the search.
- **Same program, many queries.** One model of the data, many
  questions you can ask of it.
- **Excellent for relations, graphs, deductions.** Anything
  that's "given these connections, find this." Genealogy is
  the toy example; bug-pattern matching in code is the real
  one.
- **Strong at "explain this."** Many logic engines can give
  you the *proof tree* of why an answer holds — useful for
  static analyzers explaining a found bug.

## Weaknesses

- **Search explodes.** Unconstrained Prolog can loop forever
  or take exponential time. Real programs need cut, tabling,
  or moving to Datalog to stay tractable.
- **Performance is hard to predict.** You wrote a
  specification; the engine chose the search. If the search
  picks a bad order, you may have to rewrite rules to nudge it.
- **I/O and state are awkward.** Pure logic programs reason
  about static relations. Doing the work of a typical
  application (read input, mutate state, write output) needs
  escape hatches that fight the paradigm.
- **Tooling is thin.** Debuggers, profilers, IDE support — all
  decades behind mainstream languages.

This is why Prolog itself isn't widely used as a general-
purpose application language. But the *ideas* (declarative
rules, unification, search) live on in places that play to
those strengths.

## A worked angle: type inference as logic programming

When TypeScript infers a type, it's solving:

> "Find types `T1, T2, ...` such that every expression's type
> is consistent with how it's used."

The compiler accumulates constraints (`x : T1`, `T1 = number
| string`, `f(x) : Promise<T2>`, etc.) and tries to unify
them. If a single consistent assignment exists, you get a
type-checked program. If not, you get a type error pointing at
the conflict.

The same engine, dressed in different syntax, is what Prolog
does for facts and queries. Once you see this, "why does TS
infer this weird type?" becomes "what constraints did the
compiler collect, and which assignment satisfied them?"

## When logic programming is the right paradigm

- **Querying graph-shaped data** with transitive relations.
- **Constraint problems** (scheduling, resource assignment,
  packing).
- **Static analysis** — bug patterns are naturally rules.
- **Configuration & deduction** — "given these settings, what
  must follow?"
- **Theorem proving / verification** — you want machine-
  checked proofs, not loops.
- **Domain DSLs** — many internal DSLs at companies end up
  Datalog-shaped without admitting it.

For most application code, you'll use the *ideas* (declarative
rules, solvers as libraries) rather than writing the whole
program in Prolog.

## The takeaway

Logic programming describes facts and rules, then lets a
solver search for answers. Prolog is the original; Datalog is
the practical descendant; the ideas show up in type
inference, package resolvers, static analyzers, and SAT/SMT
solvers under the hood of modern tooling. It's not a paradigm
you'll write every day, but recognizing when a problem is
*really* a constraint-solving problem — and reaching for the
right solver instead of writing it yourself — is one of the
more leveraged moves you can make as an engineer.
