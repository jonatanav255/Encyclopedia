# Imperative vs declarative

The most fundamental split in paradigms isn't OOP vs FP — it's
**imperative vs declarative**. Almost every other paradigm is a
flavor of one or the other.

- **Imperative**: tell the computer *how* to do something, step
  by step, by changing state.
- **Declarative**: tell the computer *what* you want; let the
  runtime figure out how.

OOP and procedural are imperative. FP, SQL, HTML, CSS, regex,
Prolog, and build systems like Make are declarative.

## The defining contrast

A trivial example — "give me the names of users older than 18":

```ts
// Imperative: walk the list, build a new one
const adults = [];
for (let i = 0; i < users.length; i++) {
  if (users[i].age > 18) {
    adults.push(users[i].name);
  }
}
```

```ts
// Declarative-ish (FP): describe the pipeline
const adults = users.filter((u) => u.age > 18).map((u) => u.name);
```

```sql
-- Declarative (SQL): describe the result
SELECT name FROM users WHERE age > 18;
```

The imperative version explains the *mechanism*: index, compare,
push. The declarative versions explain the *result*: "the names
of users whose age is over 18." The runtime (V8, Postgres) picks
how to actually do it — and for SQL, the database is allowed to
choose any execution plan it likes, as long as the answer is
correct.

## What "declarative" actually buys you

When you describe *what*, not *how*, the runtime can:

1. **Optimize freely.** SQL planners pick indexes, join orders,
   parallelism. You wrote `SELECT`, you didn't write a nested
   loop. The planner can swap in a hash join when it's cheaper.
2. **Run on different hardware.** The same SQL query runs on a
   laptop, a sharded cluster, and a columnar warehouse, each
   picking a different execution plan.
3. **Stay readable.** Intent is on the page. There's no
   bookkeeping (`i`, `j`, accumulators) for a reader to mentally
   simulate.
4. **Be analyzed.** A type system, a linter, or an optimizer can
   reason about declarative code more easily because it doesn't
   need to track how state evolves over time.

The trade is that you give up fine-grained control. If the
planner picks a bad plan, you can't easily "just write the loop
yourself."

## What "imperative" actually buys you

Imperative isn't a worse paradigm — it's the one closest to the
hardware. CPUs *are* imperative machines: load register,
compare, jump, store. So:

1. **Predictable performance.** When you write the loop, you
   know what the loop does. No planner is going to surprise you.
2. **Fine control of state.** Some algorithms (in-place sort,
   union-find, hash table internals) are easier to express as
   step-by-step mutation than as transformations of immutable
   data.
3. **Low-level work.** Drivers, allocators, hot inner loops — if
   you're counting cache lines, you want imperative.

## The mix in real code

Most code you'll write is a mix. A typical web handler:

```ts
async function getAdultNames(req, res) {
  // declarative: describe the result
  const adults = await db.user.findMany({
    where: { age: { gt: 18 } },
    select: { name: true },
  });

  // imperative: send the response
  res.status(200).json(adults);
}
```

The data query is declarative (you described a result; the ORM
+ database choose the execution). The response handling is
imperative (set status, then send body — order matters, state
is being mutated on the `res` object).

Neither part is fighting the other. Each is using the style
that fits its job.

## A useful framing

Ask yourself, when you're stuck deciding how to write something:

- *Do I care about the steps, or only the answer?*
- *Will a smart engine probably do a better job than me?* (SQL,
  CSS layout, build dependency graphs — usually yes.)
- *Am I doing this once or millions of times?* (Hot loops lean
  imperative; data shaping leans declarative.)

## The takeaway

Imperative is *how*. Declarative is *what*. Real programs blend
them: declarative where intent matters and a smart runtime can
do the work; imperative where you need control or proximity to
the hardware. Spotting which mode a piece of code is in is the
first step to deciding whether it's written well.
