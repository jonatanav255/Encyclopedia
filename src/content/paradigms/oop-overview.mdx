# OOP: the core idea

**Object-oriented programming** organizes a program around
**objects**: bundles of data (state) and behavior (methods) that
talk to each other through messages. The unit of structure is
the object; the unit of interaction is the method call.

OOP became the dominant industrial paradigm in the 1990s
(Smalltalk → C++ → Java → C#) because it gave teams a shared
vocabulary for modeling business domains: a `User`, an `Order`,
a `BankAccount` — each with its own state, its own rules, and a
clear interface for the rest of the system to talk to.

## The four pillars (the usual list)

You'll see this list everywhere:

1. **Encapsulation** — hide internal state; expose methods.
   Callers can't reach in and corrupt your invariants.
2. **Abstraction** — model "what this thing does" separately
   from "how it does it." Interfaces, not implementations.
3. **Inheritance** — let one class reuse another class's
   behavior and add to it.
4. **Polymorphism** — one interface, many implementations. The
   caller doesn't need to know which concrete type it has.

This list is a teaching device, not a contract. Plenty of OOP
code is excellent without ever using inheritance. The two
pillars that actually carry the weight are **encapsulation**
and **polymorphism**.

## A minimal example

A bank account, in idiomatic OOP:

```ts
class Account {
  private balance = 0;

  deposit(amount: number) {
    if (amount <= 0) throw new Error('amount must be positive');
    this.balance += amount;
  }

  withdraw(amount: number) {
    if (amount > this.balance) throw new Error('insufficient funds');
    this.balance -= amount;
  }

  getBalance() {
    return this.balance;
  }
}
```

What's actually going on:

- `balance` is **encapsulated**: outsiders can't set it
  directly. The class owns its invariant ("balance never goes
  negative") because it controls every write.
- `deposit` / `withdraw` are the **interface**. They describe
  *what you can do* with an `Account`, not how it's stored.
- A future `SavingsAccount extends Account` could add interest
  (**inheritance**), and a function `printBalance(a: Account)`
  works on any subclass (**polymorphism**).

## What "object-oriented" actually means

Two different camps disagree on this, and both have a point.

- **Class-based** (Java, C#, C++, Python): an object is an
  instance of a class. Behavior is defined on the class, shared
  by all instances. This is what most working programmers think
  of as OOP.
- **Message-oriented** (Smalltalk, Erlang, Ruby in spirit): an
  object is anything that responds to messages. The point isn't
  classes — it's that you *don't* reach into another object's
  guts; you ask it to do something.

Alan Kay (who coined "object-oriented") leaned hard on the
second meaning. He's said multiple times he regrets that the
term got attached to C++/Java-style class hierarchies, because
the point was supposed to be **encapsulation + message
passing**, not inheritance trees.

## Identity, state, behavior

The signature trait of OOP is that an object has **identity**.
Two `Account` instances with the same balance are still
different accounts. If you transfer $100 from one to the other,
something has actually changed in the world — and the object is
the natural place to encode that change.

This is why OOP shines for **entities that exist over time**: a
user session, an order in flight, a connection, a game
character. Each has a lifecycle and accumulates state. It's
also why OOP can feel forced for things that are really *just
data* (a 2D point, a parsed date) or *just a calculation* (a
sort, a hash) — those don't have identity or lifecycle, and
wrapping them in classes adds ceremony without insight.

## The common failure modes

OOP gets a bad reputation when teams:

- Build **deep inheritance trees** instead of using
  composition. By the time `OrderSpecialPromotionalSummerEdition
  extends Order` exists, change is painful.
- Treat **getters and setters as encapsulation**. Exposing
  `getBalance` *and* `setBalance` is just public state with
  extra steps; the class no longer owns its invariant.
- Model **everything as a class**. A `UserService`,
  `UserManager`, `UserHelper`, `UserUtil`, `UserFactory`... all
  of which just hold functions. Those are functions; let them
  be functions.
- Confuse OOP with **mutable shared state**. They aren't the
  same thing. Encapsulation lets you *control* mutation; it
  doesn't require mutation everywhere.

Most of the rest of this topic is about doing OOP without
falling into these traps.

## How OOP fits with FP

In modern code, OOP and FP aren't enemies. A common pattern:

- **OOP for boundaries**: services, request handlers,
  long-lived entities, anything with identity. Classes give you
  a clear seam to mock, swap, and inject.
- **FP inside**: the actual work — transforming data, computing
  results — is mostly pure functions and immutable data inside
  those classes. Easier to test, easier to reason about.

That blend is what most real codebases look like today. The
paradigms are tools; you reach for whichever one fits the part
of the problem in front of you.

## The takeaway

OOP models a program as objects with state, behavior, and
identity, talking through methods. Its real power is
**encapsulation** (objects own their invariants) and
**polymorphism** (callers depend on interfaces, not concrete
types). Inheritance is a tool, not the point. Use OOP for
things that have identity over time, and reach for other
paradigms (FP, declarative) for the pieces that don't.
