# Encapsulation

Encapsulation is **the** core idea of OOP. Strip away
inheritance, polymorphism, even classes — you still have OOP if
you have encapsulation. Lose encapsulation and you have a
struct with functions glued to it.

The definition: an object owns its state, and the outside world
can only interact with that state through methods the object
chose to expose. The object is responsible for keeping its own
invariants. Nobody else can break them, because nobody else can
reach in.

## Why this is load-bearing

Take a `BankAccount` with the invariant *"balance never goes
negative."* If `balance` is a public field, that invariant is
maintained by *every place in the codebase that writes to it.*
A new teammate adds `account.balance -= fee` somewhere on a
Tuesday afternoon and now your invariant is everyone's problem.

If `balance` is private and the only mutator is `withdraw`,
which checks the invariant, then the invariant is maintained by
**one place**. The class is the boundary of trust. Outside the
class, you don't have to think about whether `balance` is
consistent — it is, by construction.

This is why encapsulation matters in any language that supports
it, even ones (like Python or JavaScript pre-`#private`) where
"private" is a convention rather than a hard rule.

## What encapsulation isn't

A frequent confusion: **getters and setters are not
encapsulation.** A class like this:

```ts
class Account {
  private balance = 0;
  getBalance() { return this.balance; }
  setBalance(b: number) { this.balance = b; }
}
```

is a struct with extra steps. Anyone can call `setBalance(-1000)`.
The invariant isn't owned by the class; it's been delegated to
"whoever is polite enough to call `setBalance` carefully." The
field is public; the syntax just changed.

Real encapsulation looks like:

```ts
class Account {
  #balance = 0;

  deposit(amount: number) {
    if (amount <= 0) throw new Error('positive only');
    this.#balance += amount;
  }

  withdraw(amount: number) {
    if (amount > this.#balance) throw new Error('insufficient');
    this.#balance -= amount;
  }

  get balance() { return this.#balance; }   // read-only is fine
}
```

Now there is **no path** through this class to a negative
balance, period. A reader of the rest of the codebase doesn't
have to wonder.

In TypeScript specifically, `private` is checked by the compiler
but disappears at runtime — the field is still on the JS object.
`#field` is enforced by the JS runtime itself; nothing outside
the class can read it, not even via reflection. For real
encapsulation in TS, prefer `#`.

## What you actually expose

The job of designing a class is mostly the job of designing its
**public surface**: the smallest set of methods that lets
callers do what they need, with no method that breaks the
invariants.

Two useful heuristics:

1. **Verbs, not getters.** `account.deposit(100)` and
   `account.withdraw(50)` describe *what the caller is trying
   to do.* `account.balance += 100` describes *how it works.*
   The first lets the class evolve (add fees, audit logs,
   currency conversion) without callers changing. The second
   doesn't.
2. **Tell, don't ask.** Instead of pulling state out, asking
   questions, and then putting state back, ask the object to
   do the whole operation. Code that reads `if
   (account.getBalance() > x) account.setBalance(...)` is
   leaking the invariant out into the caller — make it a single
   method on `Account`.

## Encapsulation and immutability

Encapsulation and immutability solve overlapping problems but
they aren't the same.

- **Encapsulation** says: *I control who can mutate this state.*
  Mutation is fine, as long as it goes through me.
- **Immutability** says: *nobody mutates this state, ever.* You
  return new values instead.

A class can be encapsulated and mutable (`Account` above), or
encapsulated and immutable (a `Money` value object that returns
a new `Money` from every operation). For long-lived entities
with identity, encapsulated-mutable is natural. For values
(dates, points, money amounts, configurations),
encapsulated-immutable is usually cleaner — see the
**immutability** page.

## The leaks that matter

The point of encapsulation is *invariants don't leak out*. The
common ways they leak anyway:

- **Returning mutable internals.** `getItems()` returns the
  actual internal array; callers `.push()` into it; the
  invariant is gone. Return a copy, a readonly view, or an
  iterator.
- **Exposing collaborators.** `account.repository.delete(...)`
  bypasses the account entirely. The repository should be a
  private collaborator, not part of the surface.
- **God-method shortcuts.** A "for testing" backdoor
  (`__setBalance`) that production code finds and starts using.
  If tests need to set up state, do it through the constructor
  or a builder, not a hidden mutator.

## The takeaway

Encapsulation is a fence around state, with a small gate
labelled "methods." Inside the fence, the object owns its
invariants. Outside the fence, callers think in verbs, not
fields. Get this one thing right and most of OOP's other ideas
(polymorphism, composition, testability) become easier to
reach. Get it wrong and you have a public struct with a
misleading `class` keyword in front of it.
