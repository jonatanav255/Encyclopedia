# Polymorphism

**Polymorphism** literally means "many forms": one interface,
many implementations. It's the feature that lets a caller write
code against a shape without knowing — or caring — which
concrete type it has.

It's also a word that's been overloaded with three different
meanings, which trips up a lot of people. Worth pulling them
apart.

## Three flavors

Modern languages have all three. Java has subtype (extends),
parametric (`<T>`), and overloading. Rust has subtype-ish (trait
objects), parametric (generics), and ad-hoc (trait
implementations). The names are formal but the ideas are
everyday.

## Subtype polymorphism: the OOP version

The classic one:

```ts
interface Notifier {
  send(message: string): void;
}

class EmailNotifier implements Notifier {
  send(m: string) { /* SMTP */ }
}

class SlackNotifier implements Notifier {
  send(m: string) { /* HTTP to Slack */ }
}

function notifyAll(notifiers: Notifier[], message: string) {
  for (const n of notifiers) n.send(message);
}
```

`notifyAll` has no idea what concrete types it's dealing with.
It depends on the **interface**, not the implementations. That
decoupling is the point — you can add a `SmsNotifier` later
without touching `notifyAll` at all.

This is the **Open/Closed Principle** in action (open for
extension, closed for modification). Adding a new notifier is
new code; it isn't surgery on existing code.

## Parametric polymorphism: generics

Parametric polymorphism is one piece of code that works for
*any* type, with the type passed in as a parameter:

```ts
function identity<T>(x: T): T { return x; }

function first<T>(xs: T[]): T | undefined { return xs[0]; }

class Box<T> {
  constructor(private value: T) {}
  get(): T { return this.value; }
}
```

The function `identity` doesn't know or care what `T` is —
that's the point. It can't call any methods on it (it doesn't
know if `T` has them) and it can't compare it (might not be
comparable). What it *can* do is treat `T` as an opaque value
and shuffle it around — which is exactly what `identity`,
`first`, and `Box` need.

This is called **parametricity**: a function that works for any
`T` is forced to be honest. `identity<T>(x: T): T` can only
return its input. There's no other value of type `T` available
to it.

Generics are how containers, mapping/filtering, and large parts
of FP-leaning libraries are typed without losing safety.

## Ad-hoc polymorphism: overloading and operators

Same operator or method name, different implementation per
type:

```ts
1 + 2          // number addition
'a' + 'b'      // string concatenation
[1] + [2]      // string concatenation, because JS
```

`+` is overloaded. In strongly typed languages like Java or C#,
method overloading is common: `print(int)`, `print(String)`,
`print(double[])`. The compiler picks the right one based on
argument types.

Haskell's **typeclasses**, Rust's **traits**, and Swift's
**protocols** generalize this: declare an interface (`Eq`,
`Ord`, `Show`), and implement it separately for each type.
Call sites stay clean (`x.show()`, or even just `show(x)`),
with the runtime/compiler dispatching to the right
implementation.

## Static vs dynamic dispatch

When the program calls `notifier.send(msg)`, *something* has to
pick which `send` runs. Two answers:

- **Static dispatch** (compile time): the compiler picks the
  implementation. Fast (often inlinable), but only works when
  the concrete type is known at compile time. Generics with
  monomorphization (Rust, C++ templates) do this.
- **Dynamic dispatch** (runtime): the object carries a pointer
  to its method table (a **vtable**). The call looks it up at
  runtime. Slower (one indirect call, can't inline easily), but
  works for any subtype. Java/C# virtual methods, Rust `dyn
  Trait`, C++ virtual methods do this.

Most OOP-style polymorphism is dynamic dispatch. Most
FP/generic-style polymorphism is static dispatch. Both have a
place; the trade is flexibility vs. speed.

## How polymorphism replaces conditionals

A common code smell: long `switch` or `if-else` chains on a
type tag.

```ts
// Smelly
function area(shape: Shape) {
  switch (shape.kind) {
    case 'circle':    return Math.PI * shape.radius ** 2;
    case 'square':    return shape.side ** 2;
    case 'rectangle': return shape.w * shape.h;
  }
}
```

Every time a new shape is added, every such `switch` needs
updating. Polymorphism turns it inside out:

```ts
interface Shape { area(): number; }

class Circle implements Shape {
  constructor(public r: number) {}
  area() { return Math.PI * this.r ** 2; }
}
class Square implements Shape {
  constructor(public s: number) {}
  area() { return this.s ** 2; }
}
```

Now `shape.area()` works for any shape, and adding a `Triangle`
is one new class — no central switch to update.

There's a counter-argument from the FP side: **tagged unions
plus exhaustive pattern matching** give you the same safety,
and put the logic for each operation together (one place to
read all the area formulas) instead of scattered across
classes. TypeScript's discriminated unions + `never` checks
make this a real choice. Both are valid; it's a question of
*"do you add new types more often, or new operations more
often?"*

- New types often → polymorphism wins (open to new shapes).
- New operations often → tagged unions win (open to new
  operations; adding `perimeter()` is one switch, not N classes
  to edit).

## The takeaway

Polymorphism — one interface, many implementations — is what
makes OOP code (and well-designed FP code) extensible without
surgery. **Subtype** polymorphism is the OOP default; **generics**
are the parametric kind; **operator/method overloading** is the
ad-hoc kind. The practical payoff: callers depend on shapes
(interfaces), not on concrete types, which is what lets a
codebase grow without every new feature touching every
existing file.
