# Coupling and cohesion

If SOLID is the diagnostic vocabulary for *what's wrong with a
class*, coupling and cohesion are the diagnostic vocabulary for
*how a system's parts hang together.* They predate OOP (Larry
Constantine, 1968) but apply just as well to modules, packages,
microservices, and front-end components.

The two-line summary:

- **Low coupling**: modules don't depend on each other's
  internals; changes don't ripple across the codebase.
- **High cohesion**: each module's contents belong together;
  things that change together live together.

The goal is **loosely coupled, highly cohesive** modules. Most
"my codebase is a mess to change" stories are a coupling
problem, a cohesion problem, or both.

## Coupling: kinds, worst to best

Constantine's ranking still holds up. From most painful to
least:

1. **Content coupling** — one module reaches into another's
   internals (private fields, by-name introspection). Touching
   one breaks the other in subtle ways. Reflection abuse and
   monkey-patching live here.
2. **Common coupling** — modules share global mutable state.
   Anyone can change it; no one is responsible. The "shared
   globals" antipattern.
3. **External coupling** — modules share an externally imposed
   format (a file layout, a wire protocol) and both need to be
   updated in lockstep when it changes.
4. **Control coupling** — module A passes a flag that controls
   *which branch* module B executes. B isn't a black box
   anymore; A knows about its internals.
5. **Stamp coupling** — A passes a big object to B; B only
   uses two fields. B is now coupled to the whole shape of the
   object even though it doesn't need most of it.
6. **Data coupling** — A passes B exactly the values it needs;
   nothing more. Clean.
7. **Message coupling** — A sends B a message via a well-known
   interface; A doesn't know how B handles it. The loosest
   useful form.

You almost never get to zero coupling — modules have to
interact somehow. The goal is to push the dependencies *down
the list* whenever a refactor reasonably allows.

## A concrete example

```ts
// Stamp coupling — sendEmail takes the whole User, only uses email + name
function sendEmail(user: User, body: string) {
  smtp.send(user.email, `Hi ${user.name}, ${body}`);
}

// Data coupling — only what's needed
function sendEmail(to: string, name: string, body: string) {
  smtp.send(to, `Hi ${name}, ${body}`);
}
```

The first version is coupled to the *whole `User` shape*. A
schema change to `User` can break `sendEmail` even though the
email logic didn't actually care about those fields. The
second is coupled only to the data it uses.

This isn't always worth refactoring — passing three arguments
everywhere has costs too. The point is to notice when stamp
coupling is biting you (test setup gets painful, types churn
across unrelated files) and then narrow the interface.

## Cohesion: kinds, worst to best

Symmetrically:

1. **Coincidental** — things in the file have no relation;
   they ended up there for historical reasons. (The
   `utils.ts` graveyard.)
2. **Logical** — things share a category but not a purpose.
   ("All the parsing functions" — but XML parsing and JSON
   parsing share nothing in common except the word "parse.")
3. **Temporal** — things that happen at the same time (boot,
   shutdown). Better than coincidental but still arbitrary.
4. **Procedural** — steps in a process. Cohesive *if the
   process is the thing*; coupled if you'll want the steps
   separately.
5. **Communicational** — operations on the same data. Often
   good.
6. **Sequential** — output of one step feeds the next. Often
   good.
7. **Functional** — every part contributes to a single
   well-defined task. The ideal.

The shorthand: a cohesive module passes the *"summarize it in
one sentence without using 'and'"* test. "This module manages
the user session" — one job. "This module manages the user
session **and** logs to a third-party service **and** does CSV
exports" — three jobs.

## How they interact

Coupling and cohesion are linked: if you split a cohesive
module the wrong way, you create coupling between the pieces.
If you merge two unrelated modules to "reduce file count,"
you destroy cohesion. They're two sides of the same question:
*where do the lines between modules go?*

The good news: a cohesive module is almost always loosely
coupled to the rest of the system, because it has a narrow,
purposeful interface. If the module does one job, it only
needs to expose the operations of that job — not a grab-bag
of methods.

## Some practical heuristics

- **The grep test.** If you change file A, how many other
  files need updating? More than a couple → coupling is
  probably too high.
- **The new-developer test.** Can a new teammate guess where
  to make a change from the directory structure? If "where
  does this code live?" requires institutional knowledge, the
  cohesion is wrong.
- **The "and" test.** Try to describe each module's job in one
  sentence. Every "and" in the description is a hint at a
  split.
- **The constructor count.** A class whose constructor takes
  eight unrelated collaborators is doing too many jobs. Split
  by *which collaborators it uses together.*
- **Things that change together belong together.** If editing
  feature X always means editing files in three different
  modules, those modules aren't really separate — the seam is
  in the wrong place.

## Coupling and cohesion at every scale

The same vocabulary applies up and down:

- **Within a class:** are methods sharing state, or just
  parked together? (Communicational/functional cohesion good;
  coincidental cohesion bad.)
- **Within a module:** are the exported functions on the same
  problem, or a grab-bag?
- **Across services:** does Service A know about Service B's
  internal database schema? (Content coupling — disaster.) Or
  only its public API? (Message coupling — fine.)
- **Across teams:** if teams need to coordinate releases for
  every change, the systems are coupled too tightly.

Microservices are largely a *coupling* argument: services
talking through well-defined HTTP/gRPC contracts are message-
coupled, not content-coupled, and can deploy independently.
They're a bad answer when the boundaries are drawn poorly —
two services that always change together at the same time are
secretly one service split across a network.

## The takeaway

Low coupling, high cohesion. The names are old but the failure
modes — global mutable state, fat parameter objects, grab-bag
modules, lockstep changes — are present in code today. When a
codebase feels hard to change, the question to ask first isn't
"do we have enough tests?" or "should we use dependency
injection?" It's *"are the lines between modules in the right
places?"* Most refactoring is moving those lines.
