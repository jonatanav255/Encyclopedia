# Higher-order functions and composition

A **higher-order function** is a function that takes a function
as an argument, returns a function as a result, or both.
That's it — the definition is two lines, but the implications
are most of what makes FP code feel different.

If first-class functions are the "atoms," higher-order
functions are the "molecules": they're how you build big
behaviors out of small reusable pieces.

## The three you already know

You've used these in every JavaScript codebase ever:

```ts
[1, 2, 3, 4].filter((n) => n % 2 === 0);   // [2, 4]
[1, 2, 3, 4].map((n) => n * 10);           // [10, 20, 30, 40]
[1, 2, 3, 4].reduce((sum, n) => sum + n, 0);   // 10
```

`filter`, `map`, and `reduce` are higher-order because they
take a function as input and apply it to each element. They
replace whole categories of `for` loop with a more declarative
expression.

The pattern: identify *the thing that varies* (the predicate,
the transformation, the combiner), make it a parameter, leave
the iteration generic. That's also why `Array.prototype.sort`
takes a compare function and why React's `useEffect` takes a
cleanup-returning function.

## Functions as return values

The other half of "higher-order" — returning a function:

```ts
const greaterThan = (n: number) => (x: number) => x > n;

const above10 = greaterThan(10);
above10(5);    // false
above10(15);   // true

[3, 12, 7, 25].filter(greaterThan(10));   // [12, 25]
```

`greaterThan(10)` *returns a new function* that closes over the
`10`. The returned function has access to `n` even after
`greaterThan` itself has returned. That's a **closure**.

Closures are the mechanism that makes higher-order functions
work in practice. Without them, returning functions would be
nearly useless (the returned function couldn't remember
anything).

## Currying and partial application

Two related techniques you'll see in FP-leaning code:

- **Currying.** Transform a function of multiple arguments
  into a chain of functions of one argument each.

  ```ts
  // uncurried
  const add = (a: number, b: number) => a + b;
  add(2, 3);   // 5

  // curried
  const addC = (a: number) => (b: number) => a + b;
  addC(2)(3);   // 5
  ```

- **Partial application.** Pre-supply some arguments, get back
  a function expecting the rest.

  ```ts
  const greet = (greeting: string, name: string) => `${greeting}, ${name}`;
  const hello = greet.bind(null, 'Hello');
  hello('Ada');   // 'Hello, Ada'
  ```

In Haskell, every function is curried by default. In JS/TS,
you do it explicitly when it's useful (it's useful for piping
data through transformations: `users.map(prop('name'))` reads
better than `users.map(u => u.name)` once you have a `prop`
helper). Don't curry everything for the sake of it — it's a
tool, not a style guide.

## Composition

**Composition** is gluing functions together: take the output
of one and feed it to the next.

```ts
const compose = <A, B, C>(f: (b: B) => C, g: (a: A) => B) =>
  (a: A): C => f(g(a));

const trim = (s: string) => s.trim();
const lower = (s: string) => s.toLowerCase();
const normalize = compose(lower, trim);   // first trim, then lower

normalize('   HELLO  ');   // 'hello'
```

Or chained explicitly (left-to-right `pipe`, more readable to
most people):

```ts
const pipe = <T>(...fns: ((x: T) => T)[]) =>
  (x: T) => fns.reduce((v, f) => f(v), x);

const normalize = pipe(trim, lower);
normalize('   HELLO  ');   // 'hello'
```

JavaScript's `|>` pipe operator was proposed for exactly this;
Elm and F# bake it into the language (`|>`). Until then,
method chaining on arrays (`.filter().map()`) is the same idea.

The win: each piece (`trim`, `lower`) is independently testable
and reusable, and the composed pipeline reads as a description
of the transformation. The cost: another small layer of
abstraction; for one-off code it's overkill.

## Higher-order functions across the language

Once you start looking, they're everywhere — well beyond
`filter`/`map`/`reduce`:

- `Array.prototype.sort(compareFn)` — sort with a strategy.
- `setTimeout(fn, ms)`, `setInterval` — schedule a function.
- `addEventListener('click', handler)` — register a callback.
- `Promise.prototype.then(onFulfilled, onRejected)` — apply
  a continuation when the promise settles.
- `useState(initialValue)` returns `[state, setState]` — a
  pair where `setState` is the function the consumer uses.
- `useEffect(fn, deps)` — runs `fn` after render.
- `JSON.parse(text, reviver)` — transform values during
  parsing.
- React component children: a render-prop component takes a
  function and calls it with internal state.
- Middleware in Express/Koa/Redux: each middleware is a
  function returning a function returning a function (it's a
  curried higher-order pattern).

OOP design patterns are often higher-order functions in
disguise:

- The **Strategy pattern** = passing a function as an
  argument.
- The **Observer pattern** = registering a callback.
- The **Visitor pattern** = `reduce` for trees.
- The **Decorator pattern** = a function that wraps another
  function and returns a new one.
- The **Command pattern** = a value representing a function
  call.

This isn't a put-down of design patterns. They were written
in the era of Java, before lambdas were ergonomic. In a
language with first-class functions, the pattern is usually
just *a function*.

## Decorators: HOFs wrapping behavior

A common idiom is the **higher-order function that wraps
behavior** — adding logging, timing, retries, caching to an
existing function without modifying it.

```ts
function withRetry<A extends unknown[], R>(
  fn: (...args: A) => Promise<R>,
  tries = 3,
) {
  return async (...args: A): Promise<R> => {
    let lastErr: unknown;
    for (let i = 0; i < tries; i++) {
      try { return await fn(...args); }
      catch (e) { lastErr = e; }
    }
    throw lastErr;
  };
}

const fetchUserWithRetry = withRetry(fetchUser);
```

`withRetry` takes a function and returns a *new function* with
retry logic baked in. The original `fetchUser` is unchanged.
Compose it with `withLogging`, `withTiming`, `withCache` — and
you've assembled cross-cutting behavior without inheritance,
without decorators-the-language-feature, without
metaprogramming. Plain functions.

This is also how middleware works conceptually: each
middleware is `next => req => /* do something, maybe call
next(req) */`. Compose them, and you have a pipeline.

## Where higher-order functions are awkward

- **`this` binding (JS).** Methods passed as values lose
  `this`. The common fix is arrow functions or `.bind(obj)`.
  `setTimeout(obj.method, 100)` is almost never what you want.
- **Type inference.** Generic higher-order functions can
  confuse type inference, especially in heavily-generic FP
  libraries. TypeScript has gotten much better at this, but
  long curried generic pipelines still occasionally need help.
- **Debugging stacks.** Stack traces through deep composition
  pipelines can be noisy. Modern devtools handle async stacks
  better than they used to, but it's a real cost.
- **Over-abstraction.** A four-step `pipe(...)` is great; a
  twenty-step one where half the functions are one-off
  `compose`s is unreadable. Like any tool, it has a sweet spot.

## The takeaway

A higher-order function takes or returns a function. It's the
mechanism behind `map`/`filter`/`reduce`, callbacks, promises,
middleware, hooks, and most GoF design patterns rewritten in a
language with first-class functions. Combined with closures
and composition, it's the FP toolkit for building large
behaviors out of small reusable parts — the same way OOP
builds them by composing collaborators. Two roads, often
to the same destination.
