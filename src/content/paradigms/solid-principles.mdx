# SOLID

**SOLID** is five OOP design principles, popularized by Robert
Martin around 2000. They're not a rulebook — they're a checklist
of *common ways OOP code goes wrong*, each principle naming one
failure mode and the habit that avoids it.

A few of them are obvious once named. A few are subtle. All
five are still cited because the failure modes they describe
still happen every day in real code.

## S — Single Responsibility Principle

> A class should have one, and only one, reason to change.

The often-quoted version is "do one thing." The more useful
phrasing is Uncle Bob's later refinement: a class should have
**one stakeholder** whose decisions cause it to change.

The failure mode: a `UserService` that does authentication
*and* email notifications *and* profile updates *and* CSV
export. Three different teams will eventually want to change
it, in conflicting ways. The class becomes the bottleneck.

Fix: split by *reason to change*. Auth logic → `AuthService`.
Notifications → `NotificationService`. CSV export → an
export module that consumes user data. Each can evolve without
disturbing the others.

The danger of taking this too literally: splitting until every
class has one public method. The principle is about
**reasons to change**, not method count. Two methods that
change together belong together.

## O — Open / Closed Principle

> Open for extension, closed for modification.

Adding new behavior should usually mean *new code*, not
*editing existing code*. The classic example:

```ts
// Closed against new shapes — every new shape edits this function
function area(shape) {
  if (shape.kind === 'circle')    return Math.PI * shape.r ** 2;
  if (shape.kind === 'rectangle') return shape.w * shape.h;
  // ... every new shape, edit here
}
```

```ts
// Open to new shapes — adding Triangle is one new class
interface Shape { area(): number; }
class Circle implements Shape { /* ... */ }
class Rectangle implements Shape { /* ... */ }
class Triangle implements Shape { /* ... */ }   // new code, no edits
```

The mechanism is **polymorphism**. Define the contract once;
let new types satisfy it.

This is *not* a rule to predict every possible extension up
front. Designing for extension points you'll never use is
overengineering. The principle bites when you add the *second*
or *third* variant — at that point, refactor to an interface
so the *fourth* doesn't require open-heart surgery again.

## L — Liskov Substitution Principle

> Subtypes must be usable as their supertypes without
> surprising the caller.

If `Dog extends Animal`, any code that takes an `Animal` must
work when given a `Dog`. The failure mode is the rectangle/
square example on the **inheritance** page — `Square extends
Rectangle` looks fine in geometry class, but mutable squares
break callers that mutate width and height independently.

The practical rules a subtype must respect:

- **Preconditions** can be the same or *weaker* (accept at
  least as much input).
- **Postconditions** can be the same or *stronger* (return at
  least as much guarantee).
- **Invariants** of the supertype must be preserved.
- **Exceptions thrown** should be a subset of those the
  supertype declared.

Violating LSP isn't a syntax error — it's a *semantic* error.
Your compiler is fine with `Square extends Rectangle`; your
callers are not.

LSP is also the reason inheritance can't be used purely for
code reuse. The moment a subtype changes the *meaning* of a
method (not just its implementation), you've broken the
contract callers were relying on.

## I — Interface Segregation Principle

> Clients should not be forced to depend on methods they don't
> use.

The failure mode: a fat interface that everyone implements but
each implementer only cares about a slice.

```ts
// Fat
interface UserRepository {
  findById(id: string): User | null;
  save(u: User): void;
  delete(id: string): void;
  exportToCSV(): string;
  computeMonthlyMetrics(): Metric[];
  syncWithLDAP(): void;
}
```

A class that just needs `findById` now has to mock six methods
in tests, can't ship a minimal implementation, and is coupled
to changes in any of them.

Fix: split by client. The auth flow wants `UserFinder` (one
method). Admin tools want `UserExporter`. Sync wants
`UserSync`. A single class can still implement all of them —
the *implementation* can be unified; the *contracts* should be
narrow.

This is also why TypeScript's structural typing is so handy: a
caller can declare `(repo: { findById(id: string): User | null
})` and depend only on what it actually uses, without inventing
a named interface every time.

## D — Dependency Inversion Principle

> High-level modules shouldn't depend on low-level modules.
> Both should depend on abstractions.
> Abstractions shouldn't depend on details. Details should
> depend on abstractions.

The shorter version: **depend on interfaces, not concretes.**

The failure mode is the `OrderService` that constructs its own
`PostgresUserRepository` — high-level business logic
hard-coded to a low-level storage detail. Tests are painful,
swapping the store is a migration project.

The fix is **dependency injection**: pass the collaborator in
as an interface.

```ts
class OrderService {
  constructor(private users: UserRepository) {}   // interface
}
```

See the **abstraction-and-interfaces** page for the full
treatment. The "inversion" in the name refers to flipping the
arrow: instead of high-level code knowing about low-level
code, both know about a contract in the middle.

## What SOLID is not

It's not a religion, it's not a metric, and it's not a
checklist to bludgeon code review with. A small program that
violates several SOLID principles can be excellent. A large
codebase that follows all of them slavishly can be a maze of
indirection.

The honest framing: SOLID names common ways OOP code becomes
hard to change. If your code is easy to change, you're already
following the spirit, whether or not you can recite the
letters. If it's hard to change, SOLID is a decent place to
look for *why*.

Two extra cautions worth knowing:

- **DI doesn't require a DI framework.** Passing a constructor
  argument *is* dependency injection. You don't need Spring or
  InversifyJS unless you're wiring large object graphs.
- **Open/Closed isn't "predict the future."** Apply it after
  the second variant appears, not in anticipation of one.

## The takeaway

SOLID is five named failure modes of OOP code: god classes
(S), edit-the-world to add a variant (O), subtypes that lie
about being subtypes (L), fat interfaces (I), and business
logic chained to infrastructure (D). The habits — small
responsibilities, polymorphism for variation, honest subtyping,
narrow contracts, dependency injection — are durable for a
reason. Treat them as a diagnostic vocabulary, not a syntactic
mandate.
