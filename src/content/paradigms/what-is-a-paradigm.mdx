# What is a programming paradigm?

A **paradigm** is a style of thinking about programs — a set of
ideas about what the basic building blocks are, how they fit
together, and what kinds of problems become easy or hard. It's
not a language, not a framework, and not a rulebook. It's a
mental model.

The same problem written in two paradigms can look completely
different even in the *same* language, because the paradigm
decides what shape the solution takes.

## The big ones you'll meet

- **Imperative** — "do this, then this, then this." Step-by-step
  instructions that change state. C, assembly, the body of every
  `for` loop you've ever written.
- **Procedural** — imperative + grouped into named procedures
  (functions). C is the canonical example.
- **Object-oriented (OOP)** — model the world as objects that
  bundle data with behavior, talking to each other through
  methods. Java, C#, Smalltalk, Ruby.
- **Functional (FP)** — model computation as the evaluation of
  pure functions over immutable data. Haskell, Elixir, Clojure,
  OCaml.
- **Declarative** — describe *what* you want, not *how* to get
  it. SQL, HTML, CSS, Prolog, regex.
- **Logic** — declare facts and rules; the runtime searches for
  values that satisfy them. Prolog, Datalog.
- **Reactive / dataflow** — wire up streams or signals; values
  flow through, and downstream things recompute when upstream
  changes. RxJS, spreadsheets, React (in part), Svelte signals.
- **Event-driven** — the program reacts to events from the
  outside world. GUIs, Node.js servers, game loops.

## Paradigms are not religions

Languages are usually **multi-paradigm**. Python has classes
(OOP), first-class functions (FP), and `for` loops (imperative)
in the same file. TypeScript is the same. Even "pure" OOP
languages like Java have grown lambdas and streams; "pure" FP
languages like Haskell have IO actions that look procedural.

The useful question is rarely *"is this OOP or FP?"* It's
*"what's the right shape for this piece of code, given the
problem and the team?"* A request handler might be FP-shaped
(transform input → output). A user session might be OOP-shaped
(an entity with state and lifecycle). A schema definition is
almost certainly declarative.

## Why the choice matters

Each paradigm makes some things cheap and other things
expensive. Picking the wrong one for the problem doesn't fail —
it just makes everything harder than it needed to be.

- **OOP** is great when you have **entities with identity and
  lifecycle**: a user, an order, a connection. It's awkward when
  the "thing" is really a transformation, not an entity.
- **FP** is great for **data pipelines and transformations**:
  parse, validate, map, reduce. It's awkward when you genuinely
  need mutable shared state.
- **Declarative** is great when you can write down the
  invariants and let a solver/engine figure out the rest (SQL,
  build systems). It's awkward when you need fine control over
  *how* the work happens.
- **Imperative** is great when the algorithm itself is the
  point (sorting, parsing, low-level bit twiddling). It's
  awkward when you're really just describing data.

## A tiny illustration

The same job, three paradigms:

```ts
// Imperative: sum even numbers, step by step
let total = 0;
for (let i = 0; i < nums.length; i++) {
  if (nums[i] % 2 === 0) total += nums[i];
}
```

```ts
// Functional: describe the pipeline
const total = nums.filter((n) => n % 2 === 0)
                  .reduce((a, b) => a + b, 0);
```

```sql
-- Declarative: ask the database
SELECT SUM(n) FROM nums WHERE n % 2 = 0;
```

All three compute the same number. They differ in what the
reader is asked to think about: instructions, transformations,
or a question about a set.

## The takeaway

Paradigms are lenses, not laws. Knowing a few of them well —
especially **OOP** and **FP**, plus enough **declarative** to
not be afraid of SQL and HTML — lets you pick the shape that
fits the problem, instead of bending every problem to the
shape you happen to know.
