# Composition over inheritance

If there's one piece of OOP advice that's survived three
decades unscathed, it's this: **prefer composition over
inheritance.** It comes from the Gang of Four (1994) and every
serious OOP style guide since.

The advice isn't *"never inherit."* It's *"reach for composition
first; inherit only when there's a real reason."* The reason
the default flipped is that inheritance turned out to couple
classes far more tightly than it appeared, while composition
gives you the same code reuse with much weaker coupling.

## What "composition" means here

**Composition** is "X **has-a** Y": X holds a reference to a Y
and delegates to it.

**Inheritance** is "X **is-a** Y": X extends Y and inherits
its surface and its implementation.

```ts
// Inheritance — Logger IS-A ConsoleWriter
class ConsoleWriter {
  write(line: string) { console.log(line); }
}
class Logger extends ConsoleWriter {
  info(msg: string) { this.write(`[info] ${msg}`); }
}
```

```ts
// Composition — Logger HAS-A Writer
interface Writer { write(line: string): void; }

class ConsoleWriter implements Writer {
  write(line: string) { console.log(line); }
}

class Logger {
  constructor(private writer: Writer) {}
  info(msg: string) { this.writer.write(`[info] ${msg}`); }
}
```

Both versions work today. The composition version is far easier
to live with tomorrow.

## Why composition is the safer default

**1. You can swap collaborators at runtime.** Want a logger
that writes to a file? `new Logger(new FileWriter('app.log'))`.
With inheritance, `Logger extends ConsoleWriter` is baked in —
there's no `LoggerOnFile` without writing a whole new class.

**2. Tests get trivial.** `new Logger(new InMemoryWriter())`
captures output in memory, no console, no filesystem. With
inheritance you'd subclass-and-override, which is exactly the
fragile coupling you're trying to avoid.

**3. Coupling is smaller and explicit.** `Logger` depends on
`Writer` — three methods, say. With `extends ConsoleWriter`, it
inherits *everything* on `ConsoleWriter`, including methods it
didn't want, doesn't use, and is now responsible for not
breaking.

**4. You dodge the fragile base class problem.** A composed
collaborator's *interface* is what you're coupled to. Its
*implementation* can change freely. With inheritance, you're
coupled to internals — the `Counter` / `LoggingCounter` example
on the **inheritance** page.

**5. You can combine multiple capabilities.** Composition is
naturally multi-source. A `Logger` can hold a `Writer` *and* a
`Formatter` *and* a `Clock`. Inheritance gives you one parent
per class.

## The "is-a" test, in practice

If you find yourself reaching for `extends`, ask: *would a
domain expert say "an X is a Y"?* Not "an X has the same
methods as a Y" — that's a tooling coincidence.

- `Order extends LinkedList` ❌. An order has *items*; it isn't
  a list.
- `AdminUser extends User` ⚠️. Maybe. Or maybe an admin is a
  user with a `role` field. Depends on whether the difference
  is *what they are* or *what they're allowed to do.*
- `NotFoundError extends HttpError extends Error` ✅. A
  not-found error genuinely is an HTTP error.

If the "is-a" feels like a stretch, it's a stretch. Compose.

## Delegation: the composition trick people miss

The thing inheritance silently gives you is **forwarding**:
when you don't override a method, it just works because the
parent provides it. With composition, you do that forwarding
yourself.

```ts
class Cache<K, V> {
  private map = new Map<K, V>();
  get(k: K) { return this.map.get(k); }
  set(k: K, v: V) { this.map.set(k, v); }
  has(k: K) { return this.map.has(k); }
  delete(k: K) { return this.map.delete(k); }
}
```

That's the cost. Five forwarding methods. In return, your
`Cache` doesn't expose `forEach`, `entries()`, `clear()`, the
spread iterator protocol, and whatever else `Map` happens to
ship — which means it can never accidentally promise behaviors
it doesn't want to support. The forwarding is the *whole
point*: you choose your surface.

Some languages have shorthand for this: Kotlin's `class Logger
by writer`, Groovy's `@Delegate`. Most don't, and you write the
methods. It's still cheap compared to the cost of getting
inheritance wrong later.

## "Composition over inheritance" is not "no inheritance"

There are still cases where `extends` is the right tool:

- **Framework extension points.** When a framework's contract
  *is* "subclass this and override these methods" (React class
  components pre-hooks, Android `Activity`, JUnit base classes),
  do that — the base is designed for it.
- **Exception hierarchies.** Real "is-a" + you want `catch`
  blocks to dispatch on type.
- **Sealed hierarchies / ADTs.** Languages with sealed
  classes (Kotlin, Scala, Java records + sealed) let you model
  closed sets of subtypes — a great fit for things like
  `Result<T> = Ok<T> | Err`. Pattern matching on these is
  often clearer than visitor patterns.

The common thread: inheritance is the right tool when the
*subtyping* is what you're after, not the *code reuse*.

## A worked example: the strategy pattern

The classic "use composition" pattern, the **strategy** pattern,
makes the win obvious:

```ts
// Bad: every variant is a subclass
class Sorter { sort(xs: number[]) { /* default */ } }
class ReverseSorter extends Sorter { sort(xs: number[]) { /* ... */ } }
class StableSorter extends Sorter { sort(xs: number[]) { /* ... */ } }

// Good: the variant is a value you compose with
type CompareFn = (a: number, b: number) => number;

class Sorter {
  constructor(private compare: CompareFn) {}
  sort(xs: number[]) { return [...xs].sort(this.compare); }
}

const reverseSorter = new Sorter((a, b) => b - a);
```

Strategies as objects (or functions) compose; strategies as
subclasses combinatorially explode. The composition version is
the one that scales when you have *multiple* axes of variation
(ordering × stability × locale × ...).

## The takeaway

Default to composition. Hold collaborators as fields, depend on
their interfaces, delegate as needed. Reach for inheritance
when there's a real `is-a` relationship, the base is designed
for extension, or polymorphism (not reuse) is the goal. The
codebases that aged best didn't ban inheritance — they used it
sparingly, in shallow hierarchies, where it actually
represented the domain.
