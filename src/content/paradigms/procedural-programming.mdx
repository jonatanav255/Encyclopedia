# Procedural programming

**Procedural programming** organizes code around **procedures**
(named, parameterized blocks of imperative code) that operate
on data held outside them. It's what most people mean when they
say "imperative" in 2026, and it's the paradigm of C, early
Pascal, and most of the shell.

Procedural is often dismissed as "the paradigm before we knew
better." That dismissal misses what's good about it — and why
plenty of modern, performant code is procedural by deliberate
choice.

## The basic model

A procedure takes some arguments, possibly mutates some state,
maybe returns a value. Data is passed in as arguments or
pointers, not "owned" by an object. The program is a graph of
procedure calls; the structure is the call graph.

```c
typedef struct { double x, y; } Point;

double distance(Point a, Point b) {
    double dx = a.x - b.x;
    double dy = a.y - b.y;
    return sqrt(dx * dx + dy * dy);
}
```

`distance` doesn't belong to `Point`. It's a procedure that
operates on `Point` values. The data is plain (a struct of
two doubles); the behavior is a function.

You could call this "OOP without classes," and that's basically
fair. Procedural code with structs + functions is the same
shape as OOP with public fields + methods. The differences are
mostly *encapsulation* (procedural typically doesn't enforce it
beyond `static`/header conventions) and *dispatch* (procedural
doesn't have virtual methods built in — you fake them with
function pointers).

## What procedural gets right

- **It maps directly to the machine.** CPUs execute
  instructions sequentially, branch, call, return. Procedural
  code is the smallest abstraction over that.
- **Performance is predictable.** No vtable lookups, no GC
  pressure from method binding, no allocation per `Object`.
  When you write a `for` loop over an array of structs, you
  know exactly what the CPU is doing.
- **It composes by call.** Want behavior B after A? Call A,
  then call B. No framework, no DI container, no plugin
  system.
- **It scales down well.** A 30-line procedural program is
  immediately readable. A 30-line OOP program with an
  interface, a factory, and a registry is mostly ceremony.
- **It's the lingua franca.** Every language can read
  procedural code. Python, C, Lua, shell, even SQL stored
  procs all have a recognizable procedural core.

This is why **systems code** is mostly procedural. The Linux
kernel, the Redis source, SQLite, FFmpeg, Postgres, Nginx —
all written in procedural C. Not because their authors didn't
know about OOP. Because the constraints (predictable layout,
no allocation surprises, no hidden control flow) align with
what procedural gives you.

## What procedural lacks

The classic complaints are real:

- **No encapsulation by default.** A struct's fields are
  public. Convention (`s->internal_thing`) and module-level
  `static` are how C does information hiding; it's polite,
  not enforced.
- **No polymorphism without effort.** You want to handle
  multiple "shapes" through one interface? Tag the struct
  (`enum kind`), `switch` on it everywhere. Or function
  pointers (`struct ops { read, write, close }`) — which is
  basically a vtable, just hand-rolled.
- **State management gets messy in the large.** Without
  encapsulation, invariants are maintained by *every place
  that touches the data.* A large procedural codebase is a
  maintenance challenge unless the team is disciplined about
  modules.

Most of OOP's vocabulary (encapsulation, abstraction,
polymorphism) is a *response* to those weaknesses showing up
in large procedural codebases. The fixes were valuable; the
weaknesses are real.

## Procedural vs functional

People sometimes conflate the two. They're not the same.

A clean FP program and a clean procedural program don't look
alike: the FP one is a tree of expressions; the procedural one
is a sequence of statements.

## Procedural vs OOP

The honest take: a small OOP program with no inheritance and
data-class-style records *is* procedural, with `obj.method()`
syntax instead of `method(obj)`. Most "OOP" code at the leaf
level is procedural under the hood.

The OOP additions that genuinely matter — encapsulation,
polymorphism, interfaces — kick in *when programs get big*.
For small, single-purpose, performance-sensitive code,
procedural is fine and often the right call.

## When procedural is the right paradigm

- **Hot inner loops.** Sorting an array, encoding a frame,
  parsing bytes. You want predictable layout and no dispatch
  overhead.
- **Embedded / kernel / firmware.** No GC, no virtual
  dispatch, tight memory budgets.
- **Scripts and one-off tools.** A 50-line bash/Python
  script with a few helper functions doesn't need objects.
- **Algorithms.** Graph search, sorting, parsing, numerical
  routines — these are *transformations of data*, not
  *entities with behavior*. Procedures fit naturally.
- **Game engines (the perf-sensitive core).** Often
  data-oriented design: structs of arrays, tight loops, no
  virtual calls. The wider game code might be OOP; the inner
  ECS often isn't.

The common thread: when the dominant concern is *performance*
or *the algorithm is the point*, procedural usually wins.
When the dominant concern is *modeling a domain with many
collaborating entities*, OOP usually wins.

## A modern procedural style: data-oriented design

There's been a quiet revival of procedural thinking under the
name **data-oriented design** (DoD). The pitch: instead of
modeling things as objects (each holding their own data), lay
out *the data* for what the CPU actually does, and write plain
procedures that march over it.

The motivating example: in a game with 10,000 enemies, an
**array of `Enemy` objects** has cache locality issues — each
object holds heterogeneous fields, the inner loop touches a
few of them, the rest waste cache lines. **Struct-of-arrays**
(one array of positions, one of velocities, etc.) lets the
inner loop hit only the cache lines it needs.

```c
// Object-of-arrays: cache-unfriendly for tight inner loops
struct Enemy { float x, y, vx, vy, hp; const char* name; };
Enemy enemies[10000];

// Struct-of-arrays: cache-friendly
float xs[10000], ys[10000], vxs[10000], vys[10000];
// inner loop only touches xs, ys, vxs, vys
```

Entity Component Systems (ECS — Unity DOTS, Bevy, Flecs) are
this idea taken to its conclusion. The code is procedural; the
data layout is the design.

## The takeaway

Procedural programming is imperative code organized into named
procedures over plain data. It's not "OOP without the good
parts" — it's a paradigm that trades encapsulation and
polymorphism for predictability, simplicity, and proximity to
the hardware. For systems code, hot inner loops, scripts, and
algorithms, it's often the right call. For large business
domains with stateful entities, OOP's additions earn their
keep. Knowing both means picking the right one for the part of
the problem in front of you.
