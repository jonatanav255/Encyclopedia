# OOP vs FP in practice

The "OOP vs FP" debate is almost always less interesting than
people make it. Both are tools. Real codebases use both,
because some pieces of a system are naturally object-shaped
(things with identity and lifecycle) and others are naturally
function-shaped (transformations of data).

The interesting question isn't "which paradigm wins?" It's
*which pieces of my program want which shape, and how do they
interact?*

## Where the typical web app already mixes them

Take a typical Node/TypeScript web service. It has:

- **OOP-shaped pieces.**
  - A `UserService` injected with a `UserRepository` and an
    `EmailService`. Classes with constructors, methods,
    interfaces. The whole "depend on interfaces, inject
    implementations" pattern is OOP.
  - A `Connection` to a database or queue. Lifecycle:
    connect, use, close. Identity matters (it's *this*
    connection, not another).
  - HTTP handlers organized into controllers.
- **FP-shaped pieces.**
  - Validation: a pure function over the request body.
  - Business rules: pure functions over plain data
    structures (`computeInvoice(items, taxRate, discounts)`).
  - Data mapping: `users.map(toUserDTO).filter(isActive)`.
  - Reducers: state changes modeled as pure transitions.
  - JSON serialization, parsing, transformations — all FP.
- **Reactive/event-driven glue.**
  - The HTTP server itself, the message bus consumer, the
    websocket layer.

This isn't a weird hybrid. It's normal. Modern languages (TS,
Kotlin, Swift, Rust, Python, even modern Java) make the
hybrid ergonomic on purpose.

## React: the canonical hybrid

React is interesting because it has *all three* paradigms
running visibly:

- **FP shape** at the component level: a component is a pure
  function of props and state. Same inputs → same UI.
- **OOP shape** at the framework level: until 2018, components
  were classes with lifecycle methods. The class component
  model is still supported, and many large codebases still
  have them.
- **Reactive shape** in the rendering model: state changes
  trigger re-renders through React's diffing. Hooks
  (`useState`, `useEffect`, `useMemo`) are reactive
  primitives.

The hooks redesign was a deliberate *shift from OOP toward FP*
within the framework, because the FP shape generalized better
(custom hooks compose; lifecycle methods don't).

## Common shapes — and the paradigm that fits

A few recurring shapes, with the paradigm that usually wins:

- **HTTP handler.** Mostly FP shape (request in → response
  out). Side effects at the boundary.
- **Domain entity.** OOP. A `BankAccount` with invariants,
  state, methods.
- **Domain calculation.** FP. `applyDiscount(items, code)`.
- **Service object.** OOP-ish. A thin shell that wires
  dependencies and orchestrates calls; the actual *work*
  inside is mostly FP.
- **Repository / data access.** OOP, because it has a lifecycle
  and is a natural interface boundary for tests.
- **Reducer / state machine.** FP. Pure transitions over
  immutable state.
- **Event handler.** OOP/event-driven externally; FP
  internally.

The trick is recognizing the shape early. A function that
*tries* to be a class (lots of internal state held in
closures, passed around implicitly) is fighting itself. A
class that has no fields and one method is a function in
costume.

## What FP got right that OOP absorbed

The last decade has seen OOP-leaning languages quietly absorb
FP ideas:

- **Lambdas / closures everywhere.** Java 8, C# Func/Action,
  Python lambdas (always there), Swift closures, Kotlin
  lambdas. Class-based languages now treat functions as
  first-class values.
- **Streams / LINQ / Sequences.** Functional pipelines as a
  standard library feature (`stream().filter().map().collect()`
  in Java, LINQ in C#, sequences in Kotlin).
- **Records / data classes / value types.** Tight, immutable
  data containers without ceremony. Java records, Kotlin
  `data class`, C# records, Scala case classes, Rust structs.
- **Pattern matching.** Java 21, C# 9+, Python 3.10+. Tagged
  union dispatch in OOP languages.
- **`final` / `readonly` / `val` by default in modern style.**
  Immutability is now idiomatic.

The lesson: the things OOP languages adopted from FP were the
ideas that *make state safer* — immutability, pure
transformations, first-class functions. None of those
weakened OOP; they made it better.

## What OOP got right that FP absorbed

The other direction is real too:

- **Modules.** Even Haskell programs grow modules with
  encapsulated state and a public interface. Different syntax,
  same instinct.
- **Records with named fields.** OCaml, Haskell, Elm all have
  records; not classes, but structured data with named
  members. (FP without records is masochistic.)
- **Type classes / traits as a polymorphism mechanism.**
  Haskell/Rust/Swift learned to dispatch behavior the same way
  OOP does, just without inheritance.
- **Effect systems.** Modern FP languages (Scala ZIO,
  Effect-TS) bring back service-style organization, dependency
  injection, and resource management — patterns OOP had
  forever, now reimagined functionally.

The convergence point is that OOP and FP, in their *good*
modern forms, agree on a lot: small modules, narrow
interfaces, immutable data where possible, side effects at
the boundary, polymorphism over conditionals, composition
over inheritance/macros.

## The actual disagreements

What's left, after the convergence?

- **Default for state.** OOP defaults to mutable-but-encapsulated;
  FP defaults to immutable. This is the loudest difference.
- **Default for dispatch.** OOP defaults to dynamic dispatch via
  vtables; FP defaults to static dispatch via parametricity and
  pattern matching.
- **Default for organization.** OOP organizes around objects
  (data + behavior bundled); FP organizes around functions
  (behavior separate from data, data is plain).
- **Default for effects.** OOP treats effects as normal
  (methods can do anything); FP treats them as something to
  isolate or track.

These are *defaults*. None of them is a hard line. A Java
program with immutable records, switch expressions, and static
helpers can be "FP-shaped." A Haskell program with `IORef`s
and an `IO`-heavy interface can be "OOP-shaped."

## Picking the shape for the part

A pragmatic rule, restated:

- If the thing has **identity** and **state evolving over
  time** (a session, a connection, a long-lived entity), use
  OOP.
- If the thing is a **transformation** (input → output, no
  identity), use FP.
- If the thing is **events arriving asynchronously**, use
  reactive/event-driven on top of either.

A single feature usually has all three. An order checkout:
the **Order** is OOP (identity, state). The **price
calculation** is FP (items + discounts → total). The
**payment confirmation** is event-driven (await webhook).

## What's actually a bad idea

The bad ideas aren't "use OOP" or "use FP." They're:

- **Mutable shared state without encapsulation.** Globals,
  singletons-as-globals, public fields. Both paradigms agree
  this is bad.
- **Deep inheritance hierarchies.** Both paradigms agree.
- **Pretending to be the other paradigm.** Java code written
  as if it were Haskell (function pointers as fields,
  ten-level lambda nests) is usually worse than the
  paradigm-native version.
- **Forcing the wrong shape.** A pure function shouldn't be
  wrapped in a class; an entity shouldn't be threaded through
  a chain of mappers.

## The takeaway

OOP and FP aren't competitors — they're complementary lenses,
and most modern code uses both. Use OOP for things with
identity and lifecycle; use FP for transformations and
pipelines; use reactive/event-driven for the things that
arrive over time. The languages have converged on the *good
parts* of both, so picking the shape that fits the problem is
both easier and lower-stakes than the religious debate
suggests.
