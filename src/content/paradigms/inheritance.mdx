# Inheritance

Inheritance lets one class **extend** another, reusing its
fields and methods and adding or overriding behavior. It's the
mechanism people most associate with OOP, and the one that
causes the most regret in production.

The mechanism is simple. The hard part is *when not to use it.*

## What inheritance gives you

```ts
class Animal {
  constructor(public name: string) {}
  describe() { return `${this.name} is an animal`; }
}

class Dog extends Animal {
  bark() { return `${this.name} says woof`; }
  describe() { return `${this.name} is a dog`; }   // override
}

const d = new Dog('Rex');
d.bark();       // 'Rex says woof'
d.describe();   // 'Rex is a dog'   ← Dog's version wins
```

What you got:

- **Code reuse.** `Dog` didn't redeclare `name`.
- **Subtype relationship.** A `Dog` is-a `Animal`. Anything
  expecting an `Animal` accepts a `Dog`.
- **Open recursion.** When `Animal` calls `this.describe()`
  internally, it dispatches to `Dog`'s version, even though the
  call site is in `Animal`. This is the magic and the danger.

## "Is-a" vs "has-a"

The classic test: does this relationship make sense as
**"is-a"**?

- `Dog extends Animal` — *a dog is an animal.* Plausible.
- `Square extends Rectangle` — *a square is a rectangle.*
  Famously broken — see the Liskov section below.
- `Stack extends ArrayList` — *a stack is an arraylist.* No.
  The arraylist exposes `get(i)`, `add(i, x)`, `remove(i)` —
  things a stack shouldn't allow. `Stack` should **have** an
  `ArrayList` (composition), not extend it. Java's standard
  library did this and it's a textbook mistake.

If you can't say "X is-a Y" with a straight face, you want
**composition** (X has-a Y), not inheritance. The composition
page covers this in detail.

## The Liskov Substitution Principle (the L in SOLID)

> If `Dog` extends `Animal`, then anywhere code uses an
> `Animal`, a `Dog` must work without surprises.

The canonical violation:

```ts
class Rectangle {
  constructor(public width: number, public height: number) {}
  setWidth(w: number) { this.width = w; }
  setHeight(h: number) { this.height = h; }
  area() { return this.width * this.height; }
}

class Square extends Rectangle {
  // A square must have equal sides, so:
  setWidth(w: number) { this.width = w; this.height = w; }
  setHeight(h: number) { this.width = h; this.height = h; }
}

function stretch(r: Rectangle) {
  r.setWidth(5);
  r.setHeight(4);
  console.log(r.area());   // expects 20
}

stretch(new Rectangle(1, 1));   // 20
stretch(new Square(1, 1));      // 16, because setHeight clobbered width
```

The bug isn't in `Square`. The bug is in *modeling.* Geometric
squares are rectangles; mutable `Square` objects that share
`Rectangle`'s interface are not — they have stronger invariants
the interface can't express. Inheritance lets you declare a
subtype relationship that isn't actually a subtype. The compiler
won't catch it; runtime will.

## The fragile base class problem

Inheritance secretly couples subclasses to the **implementation
details** of the base class. Consider:

```ts
class Counter {
  protected n = 0;
  increment() { this.n++; }
  incrementBy(k: number) {
    for (let i = 0; i < k; i++) this.increment();   // ← calls this.increment
  }
}

class LoggingCounter extends Counter {
  increment() { console.log('inc'); super.increment(); }
}
```

`new LoggingCounter().incrementBy(3)` logs three times. Fine.

Now someone refactors `Counter`:

```ts
incrementBy(k: number) { this.n += k; }   // optimization
```

`LoggingCounter` is silently broken. It now logs zero times for
`incrementBy(3)`. The base class didn't change its public
contract — it still increments by `k` — but it changed how it
*internally* does the work, and the subclass's override no
longer intercepts.

This is called the **fragile base class problem**: changes to
the base, even ones that look like internal cleanups, can break
subclasses arbitrarily. Inheritance is a far stickier coupling
than people realize. It's why most modern style guides (and
libraries) lean composition-first.

## Multiple inheritance, mixins, traits

Most languages with inheritance let a class extend exactly one
parent (Java, C#, JavaScript). Some allow multiple (C++, Python,
Eiffel). Multiple inheritance opens the **diamond problem**:

```
      A
     / \
    B   C
     \ /
      D
```

If `D` extends both `B` and `C`, and both override a method
from `A`, whose version wins?

Different languages answer differently:

- **C++**: ambiguous unless you use virtual inheritance and
  resolve explicitly. Famous source of pain.
- **Python**: deterministic via C3 linearization (MRO — method
  resolution order). Works, but the order isn't always
  obvious.
- **Java / C#**: outlawed for classes; allowed for interfaces
  (with default methods, with rules to resolve clashes).
- **Rust / Scala / Swift**: **traits / protocols** — multiple
  inheritance of *behavior*, but not state. This is the modern
  answer; it sidesteps the diamond.

See the **class-vs-prototype-vs-trait** page for the mechanics.

## When inheritance is actually right

Inheritance pays off when:

1. The "is-a" relationship is real and stable, not a metaphor.
2. The base class is **designed for extension** — its protected
   surface is small, documented, and stable. Frameworks
   (React component classes pre-hooks, Android `Activity`,
   `Exception` hierarchies) often fit.
3. **Polymorphism is the goal**, not code reuse. You want
   callers to treat subtypes uniformly. If you just want to
   share helpers, share helpers.
4. The hierarchy is **shallow** — one or two levels deep.
   Anything deeper tends to ossify.

A canonical good case: an exception hierarchy.
`HttpError extends Error`, `NotFoundError extends HttpError`. A
caller can `catch (e instanceof HttpError)` and treat all HTTP
errors uniformly. The relationship is real, the hierarchy is
shallow, and the base class is stable.

## The takeaway

Inheritance gives you code reuse and subtyping in one
mechanism. The reuse part is usually better served by
composition; the subtyping part is usually better served by
**interfaces** (or traits). Reach for `extends` when there's a
genuine, stable "is-a" relationship and polymorphism is the
goal. Otherwise, prefer composition — your future self will
have an easier time. The **composition-over-inheritance** page
goes deeper.
