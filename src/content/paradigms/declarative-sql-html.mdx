# Declarative programming: SQL, HTML, and friends

**Declarative** programming means you describe *what you want*
and the runtime figures out *how to get it*. The two languages
every working developer already uses declaratively are **SQL**
(what data) and **HTML/CSS** (what UI). Several other useful
declarative systems share the same shape: regular expressions,
build systems (Make, Bazel, Nix), Terraform/Kubernetes
manifests, GraphQL queries, and configuration-as-code.

The point of declarative programming isn't "less code." It's
"hand the *how* to a smart engine that's better at it than you
are."

## SQL: the canonical declarative success

You write:

```sql
SELECT u.name, COUNT(o.id) AS orders
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
WHERE u.country = 'US'
GROUP BY u.name
HAVING COUNT(o.id) > 5
ORDER BY orders DESC
LIMIT 10;
```

You don't write the join algorithm, the index lookup, the
parallelization, the I/O ordering. The **query planner** turns
that statement into an **execution plan** — possibly a nested
loop join, possibly a hash join, possibly an index-only scan
— and runs it. The same query on the same data may produce
different plans tomorrow if statistics change.

What you got:

- **Hardware portability.** The same query runs on a laptop
  and on a sharded warehouse, each with a different plan.
- **Automatic optimization.** Adding an index changes the
  plan; you didn't have to rewrite anything.
- **Composability.** Subqueries and CTEs let you build big
  queries out of small ones, and the planner re-optimizes
  the whole.
- **Statistical reasoning.** The planner uses estimated row
  counts to pick algorithms. A human writing the loop by hand
  doesn't get that automatically.

The cost: when the planner picks a bad plan, you can't easily
say "no, do the other thing." You can hint, you can rewrite
the query to nudge the planner, but you don't have full
control. This is the canonical declarative trade.

## HTML and CSS: the other declarative language you use daily

```html
<article>
  <h1>Title</h1>
  <p>Body text.</p>
</article>
```

```css
article { max-width: 60ch; margin-inline: auto; }
```

You didn't tell the browser to allocate pixels, run a font
shaper, do line breaking, render at retina, position elements,
or handle resize. You described **what the document is** and
**what it looks like**, and the browser's layout engine
(Blink, WebKit, Gecko) does the rest — different on every
device, every viewport size, every accessibility setting.

HTML/CSS is also where the **separation of concerns** version
of declarative shines: the HTML says *what*; the CSS says
*how it looks*; the JS says *how it behaves*. Each lens stays
independently understandable.

## React: declarative UI on top of an imperative DOM

React's pitch is fundamentally declarative: "describe the UI
as a function of state; React figures out the DOM mutations."

```tsx
function Counter({ count }: { count: number }) {
  return <button>{count}</button>;
}
```

You don't call `document.createElement`, `textContent`,
`removeChild`. React diffs the virtual DOM against the
previous render and emits the minimum imperative changes. The
mental model is declarative; the machinery underneath is
imperative.

This is also why "don't mutate state" matters in React: the
declarative reasoning *requires* that current state describes
the current UI. Mutate state in place and the next render
might or might not see the change — declarative reasoning
breaks down.

Other declarative-UI systems: SwiftUI, Jetpack Compose,
Flutter, Vue's templates, Svelte's reactivity. They share the
"UI = f(state)" stance.

## Other places declarative wins

- **Build systems.** Make/Bazel/Nix declare *files depend on
  files via rules*; the engine figures out the build graph
  and what's stale. You don't say "compile this then link
  that" — you say "this file is derived from these inputs."
- **Infrastructure as code.** Terraform/Kubernetes manifests
  say *what the desired state is*; the controller reconciles
  reality to match. You don't run a sequence of API calls; you
  apply a desired state.
- **Regex.** A single line `^[a-z]+@[a-z]+\.[a-z]{2,}$`
  describes a pattern; the regex engine compiles it to a state
  machine and runs it. You didn't write the parser.
- **GraphQL queries.** Client declares *which fields it wants
  from which types*; the server figures out which resolvers to
  call and in what order.
- **Constraint solvers.** SAT solvers, MIP solvers, datalog —
  declare constraints/facts, let the engine search.

The common pattern: there's a hard problem (query
optimization, layout, build ordering, search) that a smart
runtime can solve better than humans writing the steps by
hand. Declarative lets the runtime own it.

## When declarative falls down

- **Bad plans.** The SQL planner sometimes picks a terrible
  join order. You're stuck rewriting the query or adding
  hints. The declarative abstraction *leaked*.
- **Hidden complexity / performance cliffs.** A CSS selector
  that triggers a full-page reflow, a Terraform diff that
  rewrites a database from scratch, a regex that's polynomial
  on certain inputs. The "what" hid expensive "how."
- **Debugging is harder.** You can't step through a SQL query
  in a debugger. You read the `EXPLAIN` plan. That's a
  different skill, and many engineers never learn it.
- **Stateful, sequential workflows are awkward.** A migration
  script with ordered steps and rollback isn't declarative-
  shaped. Workflow engines (Temporal, Step Functions) try, but
  they always end up letting you describe the steps in some
  procedural-ish way.
- **Edge cases break the abstraction.** "Just one more
  feature" eventually pushes declarative systems toward Turing
  completeness, after which they smell suspiciously like
  programming languages with worse ergonomics (looking at
  you, advanced Helm templating).

## When to reach for declarative

A useful prompt: *is this problem one where a smart engine
exists, or could exist, that would beat me at picking the
how?*

- **Data queries:** SQL.
- **UI layout:** CSS, declarative UI frameworks.
- **Build graphs:** Bazel, Nix, Make.
- **Infrastructure:** Terraform, Kubernetes.
- **Search / matching:** regex, datalog, query DSLs.
- **Schema validation:** JSON Schema, Zod, Yup — describe the
  shape; the engine validates.

When the answer is "yes, an engine can do this for me," lean
declarative. When it's "no, the algorithm is the point and I
want full control," lean imperative.

## The takeaway

Declarative programming says *what*, not *how*, and trusts a
runtime to figure out the rest. SQL and HTML are the
declarative languages every developer already uses. The wins —
optimization, portability, separation of concerns — are real,
but you pay in lost control when the engine picks badly. Use
declarative wherever a smart engine outperforms you on the
*how*. Keep imperative where you genuinely need to specify the
steps.
