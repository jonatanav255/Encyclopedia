# Functional programming: the core idea

**Functional programming** models computation as the
**evaluation of pure functions over immutable data**. The unit
of structure is the function; the unit of interaction is the
return value. Where OOP says "objects send each other
messages," FP says "data flows through functions."

The lineage runs from lambda calculus (Church, 1930s) through
Lisp (1958), ML, Haskell, OCaml, Clojure, F#, Scala, Elm,
Elixir, and into the FP-influenced corners of every modern
mainstream language (JS, Python, Rust, Kotlin, Swift).

## The three central ideas

1. **Pure functions.** Same input → same output. No side
   effects: no mutation of outside state, no I/O, no hidden
   randomness. Calling the function ten times with the same
   arguments produces the same answer ten times, and nothing
   else in the world changes.
2. **Immutable data.** You don't mutate; you return a new
   value. `arr.push(x)` becomes `[...arr, x]`. The old `arr`
   is unchanged. Everyone who held a reference to it still
   sees the same data.
3. **First-class functions.** Functions are values: pass them
   as arguments, return them from other functions, store them
   in variables. This is what lets you build big behaviors out
   of small ones.

Add **composition** (combining small functions into bigger
ones) and **higher-order functions** (`map`, `filter`,
`reduce`, and friends), and you have the FP toolkit.

## A tiny example

A typical FP pipeline:

```ts
type User = { name: string; age: number; active: boolean };

const activeAdultNames = (users: User[]): string[] =>
  users
    .filter((u) => u.active)
    .filter((u) => u.age >= 18)
    .map((u) => u.name)
    .sort();
```

What's notable:

- Nothing is mutated. `users` is the same array afterward.
- Each step is a tiny pure function (`u => u.active`).
- The code reads as a *description of the result*, not a
  recipe of state changes.
- You can test any step in isolation.
- You can swap, reorder, parallelize.

The imperative equivalent does the same job with `for` loops,
indices, and a working array — and a future reader has to
mentally execute it to understand what comes out.

## Why purity matters

A pure function is a black box with no surprises:

- **Referential transparency.** You can replace any expression
  with its result without changing program behavior. `double(3)`
  can become `6` anywhere it appears. Reasoning about code
  reduces to algebra.
- **Testable.** No setup, no mocks, no teardown. Pass input,
  assert output.
- **Cacheable.** If the result depends only on the input, you
  can memoize freely. React's `useMemo` is exactly this bet.
- **Parallelizable.** Two pure calls can run on two cores;
  there's no shared state to race over.
- **Reorderable.** Independent pure expressions can be
  evaluated in any order. Compilers love this.

Most concurrency bugs (races, deadlocks, time-of-check/
time-of-use) are *about shared mutable state*. Take away the
shared mutable state and the bugs go too.

## But real programs do I/O

If a program is *only* pure functions, it can't read a file,
send a request, or print anything. So FP languages handle the
boundary in one of a few ways:

- **Push impurity to the edges.** Most of the code is pure;
  a thin shell at the top reads input, calls the pure core,
  writes output. This is the "functional core, imperative
  shell" pattern (Gary Bernhardt) — practical in *any*
  language.
- **Wrap effects in types.** Haskell's `IO` type is the
  canonical example: an `IO String` isn't a string; it's a
  *description* of an action that, when run, yields a string.
  The pure functional core builds these descriptions; the
  runtime executes them. Effects are first-class values you
  can compose, log, retry, and reason about.
- **Effect tracking.** Modern systems (Scala ZIO, Effect-TS,
  OCaml 5 effect handlers) make effects visible in types
  without requiring a monad transformer stack.

You don't need to go all the way to Haskell to get the benefit.
A typical TypeScript codebase keeps business logic as pure
functions over plain data, with I/O in the request handler
and the database client. That's already most of the win.

The last two — **ADTs** + **pattern matching** — are the

Once you've used them, modeling "this is one of these N
cases" without them feels clumsy.

## FP vs OOP, briefly

This deserves a page of its own (and gets one: see
**oop-vs-fp-in-practice**). The short version:

- OOP organizes around **identities** (things that exist over
  time, with state and behavior bundled).
- FP organizes around **transformations** (data flowing
  through functions, no hidden state).

They're complementary. A web app handler is FP-shaped
(transform request to response). A user session is OOP-shaped
(identity, state, lifecycle). Most real code is a mix.

## Common misconceptions

- **"FP means no state."** No — FP means no *hidden,
  mutable, shared* state. Explicit state (passed in,
  returned, threaded through) is fine. The state monad,
  `useReducer`, Redux: all FP.
- **"FP is slow because of immutability."** It can be, naively.
  But **persistent data structures** (Clojure, Scala, Immer)
  share structure between versions — updating a 10,000-element
  vector is O(log n), not O(n). And the simpler reasoning
  often enables optimizations (parallelism, memoization) that
  outpace the overhead.
- **"FP requires monads."** No — monads are one way to model
  effects in a pure language (Haskell needs them; TS doesn't).
  You can write very functional JS without ever using the M-word.
- **"FP and OOP are enemies."** They're tools. Most modern
  code mixes them. Even Haskell programs end up looking
  module-shaped, and even Java has streams.

## The takeaway

FP models computation as pure functions over immutable data.
The wins are referential transparency (easier reasoning),
testability (no mocks), and concurrency safety (no shared
mutation). The bet is that pushing impurity to the edges and
keeping the core pure pays off more than it costs. You don't
need Haskell to get the benefits — a TypeScript codebase that
keeps business logic pure and isolates I/O at the boundary is
already doing most of it.
