# Abstraction and interfaces

**Abstraction** is the act of saying *what something does*
without committing to *how it does it.* In OOP, the main tools
for abstraction are **interfaces** (and, in some languages,
**abstract classes**) — named contracts that types can
implement.

The reason abstraction matters is testability and change.
Code that depends on a concrete `PostgresUserRepository` can
only be tested against a real Postgres. Code that depends on a
`UserRepository` interface can be tested against an in-memory
fake, and the real implementation can be swapped without
touching the caller.

## Interface vs implementation

The line between them is the entire point. A few examples:

- `Array.prototype.push` is part of an interface; the V8 hidden
  class machinery that makes it fast is the implementation.
  Your code calls `push`; you don't care how V8 stores the
  array.
- `fs.promises.readFile` is an interface — give it a path, get
  bytes. The implementation involves libuv, a thread pool,
  syscalls, kernel I/O scheduling. None of which is your
  problem at the call site.
- `Map.get(k)` is an interface; the underlying hash table
  (collision strategy, load factor, resize policy) is the
  implementation.

Good abstractions are **stable** at the interface level even
when the implementation churns. That stability is what lets a
team change the engine under the hood without breaking
everyone's code.

## What an interface actually is

In statically typed OOP languages, an interface is a named set
of method signatures with no implementations:

```ts
interface UserRepository {
  findById(id: string): Promise<User | null>;
  save(user: User): Promise<void>;
  delete(id: string): Promise<void>;
}
```

Multiple classes can `implement UserRepository`. Code that takes
a `UserRepository` parameter doesn't know or care which one it
got.

In structural type systems (TypeScript, Go, OCaml), a class
*implicitly* implements any interface whose shape it matches —
no `implements` keyword required. In nominal systems (Java, C#,
Rust traits), you have to declare the relationship explicitly.

Both work. Structural is more flexible (you can retrofit an
interface onto a third-party type); nominal is more explicit
(the compiler tells you exactly which types claim to satisfy
which contract).

## Why depend on interfaces, not classes

The textbook reason is the **Dependency Inversion Principle**
(the D in SOLID). The practical reason is: depend on the part
that doesn't change.

```ts
// Tightly coupled
class OrderService {
  private db = new PostgresUserRepository();   // ← concrete
  async ship(orderId: string, userId: string) {
    const user = await this.db.findById(userId);
    // ...
  }
}
```

This `OrderService`:

- Can only be tested with a real Postgres.
- Knows your DB driver in a class that should be about orders.
- Will need editing the day you migrate to a different store.

Versus:

```ts
class OrderService {
  constructor(private users: UserRepository) {}   // ← interface
  async ship(orderId: string, userId: string) {
    const user = await this.users.findById(userId);
    // ...
  }
}
```

Now `OrderService` knows nothing about Postgres. Tests pass an
in-memory `UserRepository`. Production passes the real one.
Migrating to a different store is one new class, not a hunt
through the codebase.

This pattern — pass collaborators in through the constructor as
interfaces — is called **dependency injection**. The
"injection" part is just "the caller chooses the
implementation."

## Interfaces vs abstract classes

Most OOP languages also offer **abstract classes**: classes
that can have some implemented methods and some unimplemented
ones, with subclasses required to fill in the gaps.

A common pattern: an **abstract base class** implementing the
shared scaffolding, with the language's interface system used
for the public contract. Frameworks frequently look like this
(Spring controllers, Android `Activity`, etc.).

## The wrong kind of abstraction

Abstraction has a failure mode: abstractions that don't
actually hide anything, or hide the wrong thing.

- **Pass-through interfaces.** `UserService` whose every
  method just forwards to a method on `UserRepository` with the
  same name. No invariants, no logic — just a layer for the
  sake of a layer. Delete it.
- **Premature interfaces.** An `IFoo` for a class that has
  exactly one implementation, will only ever have one, and
  exists because "we might need it someday." This is the
  YAGNI antipattern — you pay for the indirection now and get
  the benefit never.
- **Leaky abstractions.** The interface promises one thing but
  callers have to know *how it's actually implemented* to use
  it correctly. (Joel Spolsky's classic example: TCP is meant
  to abstract "reliable byte stream," but throughput, latency,
  and packet loss leak through anyway.) Some leaks are
  unavoidable; pretending they aren't is worse than naming
  them.

A good rule: introduce an interface **when you have a real
second implementation**, or a test fake that's elaborate
enough to justify the type. One real class + one in-memory fake
for tests is enough; one real class with no second
implementation in sight is overengineering.

## Abstraction is a tax

Every layer of abstraction is one more thing a reader has to
hold in their head, one more file to jump through, one more
name to remember. That's the **tax**. It's worth paying when
the abstraction:

- hides genuine complexity (a database, a protocol, a
  third-party service);
- has a stable interface (won't churn every sprint);
- has multiple implementations or genuinely needs to swap
  (tests, environments, vendors).

It's not worth paying when the abstraction:

- forwards every call unchanged;
- has one implementation forever;
- exists because a style guide said "always program to an
  interface."

## The takeaway

Abstraction is about *what*, not *how*. Interfaces are the OOP
tool for naming a contract you depend on without committing to
an implementation. Depending on interfaces makes code testable
and swappable; introducing them speculatively makes code harder
to read for no return. The rule of thumb: abstract real
boundaries (external systems, modules, replaceable parts) — and
let everything else be a class.
