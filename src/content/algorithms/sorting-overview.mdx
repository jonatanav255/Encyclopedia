# Sorting overview

Sorting is the most-studied algorithmic problem in computer science.
You don't write sort routines anymore — your language's standard
library is faster than anything you'll write — but understanding
which sort runs under the hood matters when behavior surprises you.

The headline: **`Array.prototype.sort` in V8 uses Timsort and has
been guaranteed stable since ES2019.**

## The contenders

| Algorithm | Time (avg) | Time (worst) | Space | Stable | In-place |
| --- | --- | --- | --- | --- | --- |
| Quicksort | O(n log n) | O(n^2) | O(log n) | No | Yes |
| Mergesort | O(n log n) | O(n log n) | O(n) | Yes | No |
| Heapsort | O(n log n) | O(n log n) | O(1) | No | Yes |
| Timsort | O(n log n) | O(n log n) | O(n) | Yes | No |
| Insertion sort | O(n^2) | O(n^2) | O(1) | Yes | Yes |

The big four real-world sorts are mergesort, quicksort, heapsort,
and Timsort. Insertion sort matters because the fast sorts use it
as a base case.

## Stability

A sort is **stable** if equal elements keep their relative order:

```js
// Sort by age, but keep name order for equal ages
[
  { name: 'Alice', age: 30 },
  { name: 'Bob', age: 25 },
  { name: 'Carol', age: 30 },
].sort((a, b) => a.age - b.age);

// Stable: [Bob(25), Alice(30), Carol(30)]
// Unstable: [Bob(25), Carol(30), Alice(30)]  — also a valid sort
```

Stability matters when you sort by multiple keys. Sort by name
first, then by age — stable sort preserves the name order within
each age. Unstable sort scrambles it.

ES2019 guarantees `Array.prototype.sort` is stable. Before, V8 used
quicksort for arrays >10 (unstable) and insertion sort for shorter
(stable). The TC39 spec made it stable across all engines.

## In-place

An **in-place** sort uses O(1) extra space (or O(log n) for the
recursion stack). Quicksort and heapsort are in-place. Mergesort
needs O(n) auxiliary space to merge.

For small arrays this doesn't matter. For sorting billions of
records, the memory difference is real.

## Quicksort

Pick a **pivot**. Partition the array into "less than pivot" and
"greater than pivot." Recurse on each side.

```js
function quicksort(arr, lo = 0, hi = arr.length - 1) {
  if (lo >= hi) return;
  const p = partition(arr, lo, hi);
  quicksort(arr, lo, p - 1);
  quicksort(arr, p + 1, hi);
}
```

- **Average**: O(n log n). The fastest practical sort, with the
  smallest constant factor of the O(n log n) sorts.
- **Worst**: O(n^2) — if the pivot is always the smallest or
  largest element. Sorted input with naive pivot triggers this.
- **Cache-friendly**: contiguous partitions, no auxiliary buffers.
- **Unstable** by default.

Modern quicksort uses median-of-three pivot selection or randomized
pivots to avoid worst case. The C++ STL uses **introsort**:
quicksort that falls back to heapsort if recursion depth exceeds
`2 log n` — bounded worst case, fast average case.

## Mergesort

Split the array in half. Sort each half. Merge them.

```js
function mergeSort(arr) {
  if (arr.length <= 1) return arr;
  const mid = arr.length >> 1;
  const left = mergeSort(arr.slice(0, mid));
  const right = mergeSort(arr.slice(mid));
  return merge(left, right);
}

function merge(a, b) {
  const out = [];
  let i = 0, j = 0;
  while (i < a.length && j < b.length) {
    if (a[i] <= b[j]) out.push(a[i++]);
    else out.push(b[j++]);
  }
  return out.concat(a.slice(i), b.slice(j));
}
```

- **Always** O(n log n). Worst case = average case.
- **Stable.**
- **O(n) extra space** for the merge buffer.
- **Parallelizes well** — the two halves are independent.
- **Best for linked lists** — no random access needed, no merge
  buffer (you splice nodes).

Used for external sorts (sorting more data than fits in RAM). You
merge sorted runs streamed from disk.

## Heapsort

Build a max-heap from the array. Repeatedly extract the max and
put it at the end.

- **Always** O(n log n).
- **Unstable.**
- **O(1) extra space** — in-place.
- **Slower in practice** than quicksort or mergesort due to cache
  unfriendliness (heap operations jump around the array).

Rarely used as a primary sort. Used as the fallback in
introsort (the "intro" stands for "introspective") because it
guarantees worst-case O(n log n) without quicksort's pathological
input.

## Timsort

Designed by Tim Peters for Python in 2002. Now also in Java
(Arrays.sort for objects), Android, V8, Rust, and Swift.

Idea: real-world data has **runs** — already-sorted sequences.
Timsort finds them, then merges them efficiently.

- **Best case O(n)** — fully sorted or reverse-sorted input.
- **Average / worst** O(n log n).
- **Stable.**
- **O(n) extra space.**
- Uses **insertion sort** for short runs (typically fewer than 64 elements)
  because insertion sort has the lowest constant factor.

V8 uses Timsort for `Array.prototype.sort` since V8 7.0 (October
2018). Firefox uses a Timsort variant. The reason: real-world
JS sorts often hit nearly-sorted data (appending to a sorted log,
re-sorting after a small change), and Timsort excels there.

## Insertion sort

```js
function insertionSort(arr) {
  for (let i = 1; i < arr.length; i++) {
    const val = arr[i];
    let j = i - 1;
    while (j >= 0 && arr[j] > val) {
      arr[j + 1] = arr[j];
      j--;
    }
    arr[j + 1] = val;
  }
}
```

- O(n^2) in general. O(n) on nearly-sorted input.
- **Stable, in-place, simple.**
- Beats every O(n log n) sort for small `n` (typically `n < 64`)
  because the constant factor is tiny.

That's why every fast sort uses it as a base case once the
recursive partitions are small enough.

## What V8 actually does

```js
[3, 1, 4, 1, 5, 9, 2, 6].sort();   // Timsort, stable
```

V8 dispatches to TorqueSort (a Timsort implementation in V8's Torque
language) for `Array.prototype.sort`. Stable since ES2019.

Two things to know:

1. **Default comparator is lexicographic**, not numeric:
   `[10, 2, 1].sort()` returns `[1, 10, 2]`. Always pass
   `(a, b) => a - b` for numbers.
2. **`sort` is in-place** and returns the same array. If you want a
   sorted copy, use `arr.toSorted()` (ES2023) or `[...arr].sort()`.

## When each sort wins

| Situation | Use |
| --- | --- |
| Default in your language | Whatever it ships (Timsort in JS) |
| Need guaranteed O(n log n), no risk | Mergesort or heapsort |
| Need stability | Mergesort, Timsort, insertion sort |
| Need in-place, fast average | Quicksort (with good pivot) |
| Sorting linked list | Mergesort |
| Sorting data already nearly sorted | Timsort or insertion sort |
| Sorting tiny array (`n < 50`) | Insertion sort |
| Sorting larger-than-memory | External mergesort |
| Sorting integers in a bounded range | Counting sort or radix sort (O(n)) |

## Counting and radix sort

If your data is small integers (or fixed-width keys), you can sort
in **linear time** without comparisons:

```js
// Counting sort: integers in [0, k]
function countingSort(arr, k) {
  const count = new Array(k + 1).fill(0);
  for (const x of arr) count[x]++;
  const out = [];
  for (let i = 0; i <= k; i++) {
    for (let j = 0; j < count[i]; j++) out.push(i);
  }
  return out;
}
```

- O(n + k) time, O(k) space. Beats O(n log n) when `k` is small.
- Used inside radix sort.

Radix sort applies counting sort digit by digit. Used in some
database query engines for sorting fixed-width integer keys.

## Common pitfalls

### Default lex comparator

```js
[10, 2, 1, 20].sort();   // [1, 10, 2, 20]
```

Sorts as strings. For numbers, always pass a comparator.

### Assuming sort is non-mutating

```js
const sorted = arr.sort();   // same array, mutated
sorted === arr;              // true
```

Use `arr.toSorted()` (ES2023) or `[...arr].sort()` for a copy.

### Wrong comparator return value

```js
// BAD — returns boolean
arr.sort((a, b) => a > b);
```

The comparator must return a **number**: negative if `a < b`, zero
if equal, positive if `a > b`. Returning booleans causes
undefined behavior in some sorts.

```js
// GOOD
arr.sort((a, b) => a - b);                    // numeric ascending
arr.sort((a, b) => a.name.localeCompare(b.name)); // locale-aware string
```

### Sorting strings naively across locales

```js
['ä', 'z', 'a'].sort();   // ['a', 'z', 'ä']  — code-point order
```

For user-facing text, use `Intl.Collator` or `localeCompare`.
Code-point order is rarely what users expect.

### Re-sorting after every push

If you're maintaining a sorted collection and inserting often, sort
once and use binary-search insert (`splice`) — or use a structure
designed for sorted insert (heap, sorted set). Re-sorting is
O(n log n) per insert.

## The mental model

Modern sorts are **hybrids**. Timsort uses insertion sort for short
runs and mergesort for merging. Introsort uses quicksort with a
heapsort safety net. The era of "pick one sort and commit" ended
decades ago.

For JS specifically:

- **Use `arr.sort((a, b) => a - b)`** for numbers, with the
  comparator.
- **Use `arr.toSorted(...)` (ES2023)** for non-mutating sorts.
- **Trust that sort is stable** since ES2019 — multi-key sorts
  work.
- **Reach for counting/radix only** when you've measured and
  comparison sort is the bottleneck on integer data.

For everything else, your standard library has thought about this
more than you have.
