# Greedy algorithms

A greedy algorithm makes the **locally optimal** choice at each
step and never reconsiders. When it works, you get a simple,
fast solution — typically O(n log n) for the sort plus O(n) for
the sweep. When it doesn't, it silently returns a wrong answer.

The hard part is **proving** that greedy is correct for your
problem, not implementing it.

## The two correctness properties

A problem admits a greedy solution when it has both:

1. **Greedy-choice property** — a globally optimal solution can be
   built by making locally optimal choices. The local choice
   never paints you into a corner.
2. **Optimal substructure** — after the greedy choice, the
   remaining subproblem is itself an instance of the same problem,
   and its optimal solution combined with the greedy choice is
   globally optimal.

If you can't argue both, you probably need DP, not greedy.

## Activity selection — the canonical example

Given intervals `[start, end)`, select the **maximum number** of
non-overlapping intervals.

Greedy: sort by **end time**, then pick the next interval whose
start is `>= ` the previously picked interval's end.

```js
function maxActivities(intervals) {
  intervals.sort((a, b) => a[1] - b[1]); // earliest end first
  const chosen = [];
  let lastEnd = -Infinity;
  for (const [s, e] of intervals) {
    if (s >= lastEnd) {
      chosen.push([s, e]);
      lastEnd = e;
    }
  }
  return chosen;
}
```

Why earliest-end works: picking the interval that finishes first
leaves the most room for everything else. Any optimal solution
can be swapped to start with the earliest-ending interval without
reducing the count — that's the exchange argument.

Sort by start time, by duration, or by latest end — all wrong.
Construct counterexamples until you internalize that the right
key matters.

## Interval scheduling vs partitioning

Subtle but common confusion:

- **Maximum non-overlapping intervals** → sort by end, greedy
  pick. O(n log n).
- **Minimum number of resources** to schedule all intervals
  (classroom assignment) → sort by start, sweep with a min-heap
  of end times; if the smallest end is before the current start,
  reuse the resource, else add a new one. Same idea, different
  greedy. O(n log n).

Both look like "interval problems" until you read the question
twice.

## Huffman coding — optimal prefix codes

Given character frequencies, build a binary tree that minimizes
the total weighted path length. Greedy rule: **repeatedly merge
the two least-frequent nodes**.

```js
function huffman(freqs) {
  // min-heap of {weight, node}
  const heap = new MinHeap();
  for (const [ch, f] of Object.entries(freqs)) heap.push({ w: f, n: ch });
  while (heap.size() > 1) {
    const a = heap.pop(), b = heap.pop();
    heap.push({ w: a.w + b.w, n: [a.n, b.n] });
  }
  return heap.pop().n;
}
```

O(n log n) with a heap. Used in `gzip` (combined with LZ77),
`zstd`'s entropy stage, JPEG, MP3.

The greedy choice works because of a clever exchange argument:
in any optimal tree, the two least-frequent characters must be
deepest sibling leaves — otherwise you can swap to reduce total
cost.

## Minimum spanning tree

Two greedy algorithms, both optimal:

- **Kruskal** — sort edges by weight, add each edge if it doesn't
  create a cycle (Union-Find for cycle check). O(E log E).
- **Prim** — start from a node, repeatedly add the cheapest edge
  connecting the tree to a new vertex (min-heap). O(E log V).

Pick Kruskal for sparse graphs and when edges are easy to enumerate
(network design, clustering). Pick Prim for dense graphs or when
the graph is implicit.

## Dijkstra is greedy

Dijkstra's shortest-path algorithm is a greedy: at each step,
finalize the unvisited node with the smallest tentative distance.
Correct only because edge weights are non-negative — once you
finalize, no later relaxation could improve the value. With
negative edges, the greedy invariant breaks (see **Dijkstra and
shortest paths**).

## When greedy fails — coin change

Make change for amount `N` using the **fewest** coins from a
set of denominations.

Greedy "always take the largest coin that fits" works for the US
coin system `[1, 5, 10, 25]` but **fails** for `[1, 3, 4]` and
target 6: greedy picks 4, 1, 1 (three coins) but optimal is 3, 3
(two coins).

This is the classic "looks greedy, isn't" problem — fall back to
DP (`O(N · k)` where k = denominations).

## Exchange-argument template

When proving a greedy is correct, the usual structure:

1. Suppose an optimal solution `OPT` doesn't make the greedy
   choice `g` at the first step.
2. Construct a new solution by swapping `OPT`'s first choice for
   `g`.
3. Show the new solution is no worse and is still feasible.
4. By induction, the greedy choice is safe at every step.

If you can't make step 3 work, your greedy is wrong. Counterexamples
are cheap; build a few small instances by hand before trusting it.

## Telltale signs you want greedy

- "Maximum / minimum **number** of ..." (selection problems).
- "Schedule / assign with **earliest** / **latest** / **smallest**."
- The problem has a natural ordering (time, weight, frequency).
- Brute-force is exponential but the structure is "pick one, recurse
  on the rest" with the choice depending only on a sorted key.

If the decision at each step depends on **the rest of the future**
(not just a key on the remaining items), you probably need DP.
