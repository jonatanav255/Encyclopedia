# Graph algorithms (advanced)

BFS, DFS, Dijkstra, and topological sort are the everyday tools.
Beyond them are a smaller set of algorithms that come up in
real backend systems: **max flow**, **min cut**, **bipartite
matching**, **strongly connected components**, **bridges and
articulation points**, **2-SAT**, and **A***. This page is a
practitioner's cheat sheet, not a textbook.

## Strongly connected components (SCC)

In a directed graph, an SCC is a maximal set of vertices where
every pair has a path in both directions.

Two linear-time algorithms:

- **Tarjan's** — single DFS, maintains a stack and `low-link`
  values. O(V + E).
- **Kosaraju's** — two DFS passes (one on `G`, one on `G^T`).
  Same complexity, conceptually simpler.

Use cases: detecting deadlocks in lock-acquisition graphs,
analyzing module-import cycles, finding cycles in CI pipelines
that should be DAGs, compiler dataflow analysis. The
**condensation** of a directed graph (each SCC collapsed to one
node) is a DAG — you can topo-sort it.

## Bridges and articulation points

In an undirected graph:

- A **bridge** is an edge whose removal increases the number of
  connected components.
- An **articulation point** (cut vertex) is a vertex whose
  removal disconnects the graph.

Both found in O(V + E) by a DFS that tracks discovery times and
`low` values (the lowest discovery time reachable via tree edges
and at most one back edge).

Use cases:

- **Network reliability**: which links / routers, if they fail,
  partition the network?
- **Social-graph analysis**: bridge edges connect otherwise
  separate communities.
- **Game-map design**: choke points.

## Max flow / min cut

Given a directed graph with edge capacities, a source `s`, and a
sink `t`: send as much flow from `s` to `t` as possible without
exceeding any edge capacity. The **max-flow min-cut theorem**
says the maximum flow equals the minimum total capacity of any
edge set whose removal disconnects `s` from `t`.

Standard algorithms:

| Algorithm | Complexity | Notes |
| --- | --- | --- |
| Ford-Fulkerson (BFS) = Edmonds-Karp | O(V · E²) | Augmenting paths via BFS |
| Dinic's | O(V² · E), O(E · sqrt(V)) on unit-capacity | Fast in practice |
| Push-relabel | O(V³) or better | Best for dense graphs |

Applications:

- **Bipartite matching** (see below) — model as unit-capacity flow.
- **Image segmentation** — graph cuts (Boykov-Kolmogorov is a
  push-relabel variant).
- **Project selection** / **closure** — pick a subset of projects
  with dependencies maximizing profit; reduces to min cut.
- **Network routing** — multi-commodity flow, traffic engineering.

Almost nobody writes max-flow from scratch in production — use a
library (OR-tools, Boost Graph). But knowing the **reduction** is
the skill: half of "weird combinatorial optimization" problems
are flow problems in disguise.

## Bipartite matching

Given a bipartite graph (vertices split into two sides, edges
only between sides), find the maximum set of edges with no shared
endpoints.

- **Hopcroft-Karp** — O(E · sqrt(V)). The standard.
- Or **reduce to max flow** — add a super-source connected to
  one side, a super-sink to the other, all unit capacities.

Use cases:

- **Job assignment** — workers ↔ tasks, edge if worker can do
  task.
- **Stable matching** — though stability adds the Gale-Shapley
  algorithm on top.
- **Ad serving** — impressions ↔ ad slots.
- **GPU scheduling** — jobs ↔ devices with affinity.

## A* search

Best-first graph search that uses a **heuristic** `h(n)` (estimated
cost from `n` to the goal). Maintains a priority queue keyed by
`f(n) = g(n) + h(n)`, where `g(n)` is the cost from start to `n`.

Correctness requires `h` to be **admissible** — never overestimates
the true remaining cost. For shortest-path on a grid, straight-line
distance is admissible. For road networks, ALT (A-star with
landmarks and triangle inequality) is the high-performance choice.

Use cases:

- **Pathfinding in games** — A* on the navigation mesh.
- **Robotics motion planning** — A* / RRT* / lazy-A*.
- **Route planning** — OSRM and GraphHopper layer A*-style
  algorithms with contraction hierarchies.

When `h(n) = 0` everywhere, A* degenerates to Dijkstra. The
better your heuristic, the fewer nodes A* expands.

## 2-SAT

Boolean satisfiability where every clause has exactly two
literals. Solvable in linear time via SCCs on the **implication
graph**: each clause `(a OR b)` becomes two implications
(`¬a → b`, `¬b → a`); satisfiable iff no variable and its
negation are in the same SCC.

Showing up in production code looks like:

- "Pick one of two configurations for each component such that
  certain pairs are compatible."
- Scheduling with binary on/off choices and pairwise constraints.

Recognizing 2-SAT in disguise turns an exponential brute force
into O(V + E).

## Eulerian and Hamiltonian paths

- **Eulerian** — visits every edge exactly once. Existence
  conditions are simple (degrees), construction is O(E)
  (Hierholzer's algorithm). Used in genome assembly (de Bruijn
  graphs), Chinese postman problem.
- **Hamiltonian** — visits every vertex exactly once. NP-hard.
  Brute-force backtracking or DP-on-subsets (`O(n² · 2^n)`).
  Travelling-salesman is the weighted optimization version.

The shape of the asked-for path matters enormously; conflating
Eulerian with Hamiltonian is a classic blunder.

## Centrality and PageRank

For graph analytics:

- **Betweenness centrality** — how often a vertex sits on
  shortest paths. Brandes' algorithm: O(V · E).
- **PageRank** — eigenvector of the column-stochastic adjacency
  matrix. Power iteration, O(E) per iteration, ~50 iterations
  usually enough.
- **Closeness centrality** — 1 / sum of shortest-path distances.

Used in search ranking, fraud detection ("nodes that bridge
many clusters"), influence analysis.

## When to call a library vs implement

Reach for **NetworkX** (Python), **igraph**, **JGraphT**, or
**graphology** (JS) for production work. Implement these only when:

- You need a tighter integration with your data structures.
- The graph doesn't fit in memory and you need streaming.
- You're targeting an exotic platform without good libraries.

The reductions — "this problem is bipartite matching," "this is
2-SAT," "this is min-cut" — are the real interview / design
value. The implementations are commodity.
