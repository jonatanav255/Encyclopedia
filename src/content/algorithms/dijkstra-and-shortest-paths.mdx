# Dijkstra and shortest paths

For unweighted graphs, BFS gives shortest paths in O(V + E). For
weighted graphs you need more — and the right algorithm depends on
whether your weights are non-negative, can be negative, and
whether you have heuristic information.

The three algorithms worth knowing: **Dijkstra** (non-negative
weights), **Bellman-Ford** (allows negatives), **A\*** (Dijkstra
with a heuristic).

## Dijkstra's algorithm

Find shortest paths from a single source to all other nodes when
all edge weights are non-negative.

The idea: maintain a tentative distance to every node. Repeatedly
pick the unvisited node with the smallest tentative distance,
finalize it, and relax its outgoing edges.

```js
// graph[node] = array of [neighbor, weight]
function dijkstra(graph, source) {
  const dist = new Map();
  for (const node in graph) dist.set(node, Infinity);
  dist.set(source, 0);

  // Min-heap by distance. (Use a real priority queue; demo uses sort.)
  const pq = [[0, source]];

  while (pq.length) {
    pq.sort((a, b) => a[0] - b[0]);   // O(n log n) — use a heap in real code
    const [d, node] = pq.shift();

    if (d > dist.get(node)) continue;   // stale entry

    for (const [next, weight] of graph[node]) {
      const newDist = d + weight;
      if (newDist < dist.get(next)) {
        dist.set(next, newDist);
        pq.push([newDist, next]);
      }
    }
  }

  return dist;
}
```

Complexity:

- With a **binary heap** priority queue: O((V + E) log V).
- With a **Fibonacci heap** (rarely worth implementing):
  O(E + V log V).
- With **naive array scan**: O(V^2). Wins for dense graphs.

The "use a heap" advice is only true for sparse graphs. For dense
graphs (E close to V^2), the simpler array version is faster
because heap operations have non-trivial constants.

JavaScript has no built-in heap. Either ship a small heap class or
use a library (`@datastructures-js/priority-queue`,
`heap-js`). The sort-on-every-pop approach above is fine for
interview problems and ~thousand-node graphs.

## Why Dijkstra breaks on negative edges

Dijkstra's correctness relies on the invariant **"once a node is
finalized, no shorter path can be found later."** With non-negative
weights, this holds — you can't reduce a distance by adding more
edges.

With a negative edge, a longer path might suddenly become shorter:

```
A --1--> B
A --4--> C --(-3)--> B
```

Dijkstra finalizes B at distance 1 from A. It never re-examines B,
so it misses the path A -> C -> B at distance 4 + (-3) = 1, which
happens to tie but in general could be smaller.

If you have negative edges, **use Bellman-Ford**.

## Bellman-Ford

Same problem (single-source shortest path), works with negative
edges, and detects **negative cycles**.

```js
// edges = array of [from, to, weight]
function bellmanFord(nodes, edges, source) {
  const dist = new Map();
  for (const node of nodes) dist.set(node, Infinity);
  dist.set(source, 0);

  // Relax every edge V - 1 times
  for (let i = 0; i < nodes.length - 1; i++) {
    for (const [u, v, w] of edges) {
      if (dist.get(u) + w < dist.get(v)) {
        dist.set(v, dist.get(u) + w);
      }
    }
  }

  // One more pass: if anything still improves, there's a negative cycle
  for (const [u, v, w] of edges) {
    if (dist.get(u) + w < dist.get(v)) {
      throw new Error('Negative cycle detected');
    }
  }

  return dist;
}
```

Complexity: O(V * E). Much slower than Dijkstra. Use only when you
have negative edges or need negative-cycle detection.

**Negative-cycle detection** is what makes Bellman-Ford uniquely
useful. Applications include arbitrage detection in currency
markets (a cycle with negative log-weight = profit) and constraint
solving where infeasibility manifests as a negative cycle.

## A\* — Dijkstra with a heuristic

Dijkstra explores outward in all directions equally. If you know
roughly which direction the goal is in (e.g., Euclidean distance
on a grid), you can bias the search toward the goal.

A* uses `f(n) = g(n) + h(n)`:

- `g(n)` — known shortest distance from source to `n`.
- `h(n)` — **heuristic** estimate of distance from `n` to goal.

The algorithm picks the node with smallest `f(n)`, not smallest
`g(n)`.

```js
function aStar(graph, source, goal, heuristic) {
  const gScore = new Map();
  for (const node in graph) gScore.set(node, Infinity);
  gScore.set(source, 0);

  const pq = [[heuristic(source, goal), source]];

  while (pq.length) {
    pq.sort((a, b) => a[0] - b[0]);
    const [_, node] = pq.shift();

    if (node === goal) return gScore.get(goal);

    for (const [next, weight] of graph[node]) {
      const tentative = gScore.get(node) + weight;
      if (tentative < gScore.get(next)) {
        gScore.set(next, tentative);
        pq.push([tentative + heuristic(next, goal), next]);
      }
    }
  }

  return Infinity;
}
```

For A* to give an **optimal** answer, the heuristic must be
**admissible** — never overestimate the true remaining distance.

Common heuristics:

- **Manhattan distance** for grid pathfinding without diagonals.
- **Euclidean distance** for any-angle pathfinding.
- **Octile distance** for grid pathfinding with diagonals.
- **Zero** — degenerates A* into Dijkstra.

If `h(n) = 0` for all `n`, A* becomes Dijkstra. If `h(n)` is exactly
the true remaining distance, A* expands only nodes on the optimal
path. Most real heuristics fall between these extremes.

A* is the workhorse of **game pathfinding** (NPCs in roguelikes,
RTS unit movement) and **GPS routing** (with great-circle distance
as the heuristic).

## Beyond single-source

| Problem | Algorithm | Time |
| --- | --- | --- |
| Single-source, non-negative | Dijkstra | O((V+E) log V) |
| Single-source, negative ok | Bellman-Ford | O(V * E) |
| Single-source, A to B with heuristic | A* | depends on heuristic |
| All-pairs shortest path | Floyd-Warshall | O(V^3) |
| All-pairs, sparse | Johnson's | O(V^2 log V + V * E) |

**Floyd-Warshall** is the "give me distances between every pair"
algorithm. Brutally simple — three nested loops:

```js
function floydWarshall(dist) {
  const n = dist.length;
  for (let k = 0; k < n; k++)
    for (let i = 0; i < n; i++)
      for (let j = 0; j < n; j++)
        if (dist[i][k] + dist[k][j] < dist[i][j])
          dist[i][j] = dist[i][k] + dist[k][j];
}
```

O(V^3) — feasible for V up to a few hundred. Used for small,
dense graphs and for transitive closure problems.

## Real-world applications

### GPS routing

Modern map services don't run plain Dijkstra on the whole road
network — that would be too slow. They use:

- **Contraction hierarchies**: preprocess the graph to add
  "shortcut" edges that skip uninteresting nodes. Query time
  becomes near-instant.
- **A*** with the Euclidean heuristic.
- **Hierarchical routing**: motorways for long distances, local
  roads only near origin/destination.

But the core "find shortest weighted path" problem is Dijkstra-
shaped.

### Game pathfinding

A* is the de facto standard. Grid-based games (roguelikes, tile
RPGs) use Manhattan or octile distance. Continuous-space games
use navmesh graphs.

When the map is huge, hierarchical A* (HPA*) partitions the map
into clusters, plans coarsely across clusters, and refines within
each cluster.

### Network routing

Internet routers use **distance-vector** (RIP — Bellman-Ford-like)
or **link-state** (OSPF — Dijkstra) protocols to compute paths
through the network. Negative weights aren't a real concern in
networking; Dijkstra dominates.

### Dependency cost

If you have a dependency graph with "cost to build" labels, the
shortest-path algorithm tells you minimum cost to reach a target.
Build systems sometimes use this to decide which path to rebuild
when multiple paths to the same target exist.

### Word ladders

"Convert CAT to DOG, one letter at a time, each intermediate must
be a word." A graph where edges connect words at Hamming distance
1; shortest path is BFS (unweighted). Add costs (e.g., "vowel
changes are cheap, consonant changes are expensive") and it
becomes Dijkstra.

## Negative weights in disguise

Real systems sometimes need negative edges:

- **Currency arbitrage**: edge weight = `-log(exchange_rate)`. A
  negative cycle means risk-free profit.
- **Game theory**: payoffs can be losses.
- **Constraint propagation**: difference constraints (`x - y <= w`)
  map directly to graph edges with weight `w`.

If you might have negatives, Dijkstra is wrong even if it usually
"looks right" on your test data. Use Bellman-Ford.

## Common pitfalls

### Using Dijkstra with negative weights

Silently wrong answers. Dijkstra doesn't detect that it can't
handle negatives — it just returns nonsense. Always verify
non-negativity before reaching for Dijkstra.

### Sorting the queue every pop

```js
pq.sort((a, b) => a[0] - b[0]);   // O(n log n) per pop -> O(n^2 log n) total
```

Fine for small graphs. For real performance, use a binary heap
(insert and extract-min both O(log n)).

### Forgetting "stale entry" check

Naive priority-queue Dijkstra inserts duplicates when a node's
distance improves. The check:

```js
if (d > dist.get(node)) continue;
```

skips stale entries. Without it you may process the same node
multiple times. Still correct, but slower.

### A* with an inadmissible heuristic

If the heuristic *overestimates* remaining distance, A* finds a
path quickly but it may not be optimal. "Greedy best-first search"
uses an aggressive heuristic and accepts suboptimality for speed —
sometimes that's fine. Know which you've built.

### Using shortest path when you want all paths

Dijkstra finds *one* shortest path. For "all shortest paths" (or
the count), you need a modified algorithm that tracks predecessors
as a set, not a single value.

### Forgetting that A B C might tie

Tied path lengths cause non-deterministic results — which path
you get depends on heap implementation, neighbor iteration order,
etc. If determinism matters (testing, replays), break ties with a
secondary criterion.

## The mental model

Shortest-path algorithms are **expansion of a wavefront** — the
wavefront just expands differently:

- **BFS**: in concentric rings of edge count.
- **Dijkstra**: in concentric rings of total weight (only safe with
  non-negative weights).
- **A***: in a teardrop biased toward the goal by the heuristic.
- **Bellman-Ford**: relax all edges V-1 times. Brute force, but it
  handles negatives.

Two practical rules:

1. **Default to Dijkstra** with a binary heap for weighted graphs.
   Reach for A* when you have a useful heuristic (geometric
   distance, embedding similarity).
2. **Bellman-Ford only when you need it.** Negative weights or
   negative-cycle detection are the only valid reasons — it's
   slow.

In production, you usually don't write shortest-path code — you
use a library (graphology, JGraphT, NetworkX). The value of
knowing them is recognizing when your problem **is** a shortest-
path problem in disguise: dependency cost, optimal-action
sequences, scheduling under constraints, conversion arbitrage, even
word similarity. The pattern is everywhere once you see it.
