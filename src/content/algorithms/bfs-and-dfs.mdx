# BFS and DFS

Breadth-first and depth-first search are the two basic ways to walk
a graph. They share the same template — pick a frontier, pop a
node, visit its neighbors, repeat — and differ only in what
container holds the frontier.

That one difference (queue vs stack) changes which problems each
solves naturally.

## Representing a graph

The standard JS representation is an **adjacency list**: each node
maps to an array of neighbors.

```js
// Directed graph
const graph = {
  A: ['B', 'C'],
  B: ['D'],
  C: ['D', 'E'],
  D: ['F'],
  E: ['F'],
  F: [],
};
```

For an undirected graph, store both directions.

For weighted graphs, store `[neighbor, weight]` pairs — covered in
**Dijkstra and shortest paths**.

Space: O(V + E) where V is vertices, E is edges. Adjacency matrices
(V x V boolean grid) are O(V^2) — usually wasteful for sparse
graphs.

## BFS — breadth-first search

Visit all neighbors at distance 1, then distance 2, then distance
3. Use a **queue** (FIFO).

```js
function bfs(graph, start) {
  const visited = new Set([start]);
  const queue = [start];
  const order = [];

  while (queue.length) {
    const node = queue.shift();   // O(n) in JS arrays — see below
    order.push(node);
    for (const next of graph[node]) {
      if (!visited.has(next)) {
        visited.add(next);
        queue.push(next);
      }
    }
  }

  return order;
}
```

**A note on `queue.shift()`**: `Array.prototype.shift` is O(n)
because it reindexes all elements. For small graphs it doesn't
matter. For large ones, use an index pointer:

```js
const queue = [start];
let head = 0;
while (head < queue.length) {
  const node = queue[head++];
  // ...
}
```

Or a real deque implementation. Most BFS code in interviews uses
`shift` and that's fine.

## BFS gives shortest path in unweighted graphs

Because BFS explores in layers — all distance-1 nodes before any
distance-2 — the first time it reaches a node is via a
shortest-edge-count path.

```js
function shortestPath(graph, start, end) {
  if (start === end) return [start];
  const parent = new Map([[start, null]]);
  const queue = [start];
  let head = 0;

  while (head < queue.length) {
    const node = queue[head++];
    for (const next of graph[node]) {
      if (parent.has(next)) continue;
      parent.set(next, node);
      if (next === end) {
        // Reconstruct path by walking parents back
        const path = [];
        let cur = end;
        while (cur !== null) {
          path.push(cur);
          cur = parent.get(cur);
        }
        return path.reverse();
      }
      queue.push(next);
    }
  }

  return null;   // unreachable
}
```

Store `parent` as you go; reconstruct the path by walking back from
`end`.

This only works for **unweighted** graphs (each edge is "1 step").
For weighted graphs, you need Dijkstra.

## DFS — depth-first search

Go deep before going wide. Use a **stack** (LIFO) — or just
recursion, which uses the call stack.

```js
// Recursive
function dfs(graph, node, visited = new Set()) {
  if (visited.has(node)) return;
  visited.add(node);
  for (const next of graph[node]) {
    dfs(graph, next, visited);
  }
}

// Iterative with explicit stack
function dfsIterative(graph, start) {
  const visited = new Set();
  const stack = [start];
  while (stack.length) {
    const node = stack.pop();
    if (visited.has(node)) continue;
    visited.add(node);
    for (const next of graph[node]) stack.push(next);
  }
}
```

For deeply nested graphs (>10K nodes deep), prefer iterative —
recursion blows the stack since V8 doesn't do tail-call
optimization. See **Recursion and tail calls**.

## Pre-order, post-order, and cycle detection

DFS naturally has **two timestamps** per node: when you entered
("discovery time") and when you finished all descendants ("finish
time"). The difference lets DFS answer questions BFS can't.

```js
const WHITE = 0, GRAY = 1, BLACK = 2;

function detectCycle(graph) {
  const color = new Map();
  for (const node in graph) color.set(node, WHITE);

  function visit(node) {
    color.set(node, GRAY);
    for (const next of graph[node]) {
      if (color.get(next) === GRAY) return true;   // back edge = cycle
      if (color.get(next) === WHITE && visit(next)) return true;
    }
    color.set(node, BLACK);
    return false;
  }

  for (const node in graph) {
    if (color.get(node) === WHITE && visit(node)) return true;
  }
  return false;
}
```

The three-color scheme:

- **WHITE** — not yet seen.
- **GRAY** — currently in the recursion stack.
- **BLACK** — finished (all descendants explored).

A **back edge** (edge to a GRAY node) means you've found a cycle.
You can't detect cycles with BFS in directed graphs the same way —
BFS doesn't track "currently in the path."

For undirected graphs: any non-tree edge that isn't to your direct
parent is a cycle. Different but related logic.

## DFS for topological order

Topological sort orders nodes so every edge points "forward" (from
earlier to later). Possible only in a DAG (directed acyclic graph).

DFS gives a topo sort for free: push nodes onto a result list in
**finish order** (when DFS leaves the node), then reverse.

```js
function topologicalSort(graph) {
  const visited = new Set();
  const result = [];

  function visit(node) {
    if (visited.has(node)) return;
    visited.add(node);
    for (const next of graph[node]) visit(next);
    result.push(node);   // post-order
  }

  for (const node in graph) visit(node);
  return result.reverse();
}
```

The intuition: a node is "finished" only after all its dependencies
are finished. Reversed, that gives dependency-first order.

This works only for DAGs — if there's a cycle, no topo order
exists. Combine with cycle detection (`detectCycle` above) before
trusting the result. See **Topological sort** for the BFS-based
alternative (Kahn's algorithm) and the use cases.

## When to use which

**BFS**:

- **Shortest path** in unweighted graphs.
- **Level-order** traversal of trees ("print level by level").
- **Spreading processes** — finding everything within K steps of a
  source.
- **Web crawlers** prioritizing breadth.

**DFS**:

- **Cycle detection** in directed graphs.
- **Topological sort.**
- **Connected components** (count clusters).
- **Path enumeration** (find all paths between two nodes).
- **Backtracking** problems (sudoku, N-queens).
- **Tree traversal** (in-order, pre-order, post-order).

Same code structure, different container, different problems.

## Bidirectional BFS

For "shortest path between A and B" in a large graph, you can search
from both endpoints simultaneously and stop when they meet.

```
BFS from A:  expands sphere around A
BFS from B:  expands sphere around B
When the spheres collide: shortest path found
```

If the branching factor is `b` and the distance is `d`, naive BFS
is O(b^d). Bidirectional is O(b^(d/2) + b^(d/2)) = O(b^(d/2)).
For `b = 10, d = 10`, naive is 10 billion, bidirectional is 200K.

Used in social-graph "degrees of separation" queries.

## Connected components

```js
function countComponents(graph) {
  const visited = new Set();
  let count = 0;

  for (const node in graph) {
    if (visited.has(node)) continue;
    count++;
    // DFS or BFS from this node, marking all reachable
    const stack = [node];
    while (stack.length) {
      const cur = stack.pop();
      if (visited.has(cur)) continue;
      visited.add(cur);
      for (const next of graph[cur]) stack.push(next);
    }
  }

  return count;
}
```

Each unvisited node starts a new component. Useful for clustering,
"find all isolated subgraphs."

For dynamic connectivity (edges added over time), use **union-find**
instead — see the data structures section.

## Common pitfalls

### Forgetting to mark visited

```js
function dfs(graph, node) {
  for (const next of graph[node]) dfs(graph, next);   // infinite loop
}
```

Any cycle will spin forever. Always track visited.

### Using a set for the frontier, not visited

```js
const queue = new Set([start]);   // BAD — Set doesn't have FIFO order
```

Sets are unordered (technically insertion-ordered in JS, but you
can't pop the head efficiently). Use an array as a queue.

### `shift()` performance

```js
while (queue.length) {
  const node = queue.shift();   // O(n) — relayouts the array
}
```

For BFS on large graphs (>100K nodes), use a head pointer or a real
deque. Most of the time it doesn't matter; when it does, it
matters a lot.

### Recursive DFS on deep graphs

```js
function dfs(graph, node, visited) {
  visited.add(node);
  for (const next of graph[node]) {
    if (!visited.has(next)) dfs(graph, next, visited);   // stack overflow
  }
}
```

V8 caps recursion at ~10K frames. For graphs deeper than that
(linked lists modeled as graphs, deeply nested JSON), convert to
iterative DFS with an explicit stack.

### BFS for weighted shortest path

```js
bfs(roadNetwork, home);   // each edge has a distance in km
```

BFS counts edges, not weights. The path with the fewest edges
isn't the shortest. Use Dijkstra.

### Confusing tree and graph traversal

A tree has no cycles, so you don't need a `visited` set for tree
DFS. But the moment you have a graph (or any structure where nodes
can be reached by multiple paths), you need to track visited.

## The mental model

BFS and DFS share **one algorithm with one parameter**: the
container for the frontier. Queue = breadth. Stack = depth.

Two practical rules:

1. **Default to BFS for shortest-path / nearest queries.** Default
   to DFS for topology and "explore the structure" questions.
2. **Always track visited** in graphs, never in trees.

The hard part isn't the traversal — it's modeling your problem as a
graph. Tasks with dependencies, file systems, social networks,
state machines, web pages, package versions — all graphs. Once you
see the graph, BFS/DFS is the easy part.
