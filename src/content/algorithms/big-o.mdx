# Big-O notation

Big-O describes how an algorithm's cost grows as input size grows.
It throws away constants, lower-order terms, and machine details so
you can compare algorithms without benchmarking every machine.

It's an upper bound on growth rate, not a wall-clock prediction.

## O, Theta, Omega

- **O(f)** — upper bound. "Grows no faster than f." Worst case is
  often phrased this way.
- **Omega(f)** — lower bound. "Grows at least as fast as f." Best
  case.
- **Theta(f)** — tight bound. Both O(f) and Omega(f). The "true"
  growth rate.

In practice, engineers say "O(n log n)" and mean Theta. The strict
mathematician's distinction rarely matters at standups.

## The common classes

From fastest to slowest:

| Class | Name | Example |
| --- | --- | --- |
| O(1) | constant | hash lookup, array index |
| O(log n) | logarithmic | binary search |
| O(n) | linear | one pass through an array |
| O(n log n) | linearithmic | mergesort, heapsort, Timsort |
| O(n^2) | quadratic | nested loop over the same array |
| O(n^3) | cubic | three nested loops (naive matrix multiply) |
| O(2^n) | exponential | recursive Fibonacci without memo, subsets |
| O(n!) | factorial | brute-force permutations (TSP) |

For `n = 1,000,000`:

- O(1): 1 step.
- O(log n): ~20 steps.
- O(n): 1 million steps.
- O(n log n): ~20 million steps.
- O(n^2): 1 trillion steps. Hours.
- O(2^n): heat death of the universe.

The gap between O(n log n) and O(n^2) is the line between "scales"
and "doesn't scale."

## What gets dropped

```js
// All of these are O(n)
for (let i = 0; i < n; i++) work();              // n iterations
for (let i = 0; i < n; i++) { a(); b(); c(); }   // 3n iterations
for (let i = 0; i < 5 * n + 100; i++) work();    // 5n + 100
```

Big-O drops:

- **Constants**: `5n` is O(n). `100n` is O(n).
- **Lower-order terms**: `n^2 + n + 7` is O(n^2).
- **Coefficients on the dominant term**: `0.5 n log n` is O(n log n).

This is the *point* of the notation — it lets you compare algorithms
without machine-specific noise. It's also the **trap**: an O(n)
algorithm with a 1000x constant factor loses to an O(n log n)
algorithm for small n.

## What constant factors hide

```js
// Both O(n)
arr.forEach(x => x);                    // tight loop
arr.forEach(x => JSON.parse(slow(x)));  // tight loop with expensive body
```

Same Big-O. Very different runtimes. The notation tells you nothing
about the work *inside* the loop.

Real-world examples where constants dominate:

- **Cache locality**: a linear scan over a contiguous array beats an
  O(log n) tree traversal that misses cache on every step. For
  `n < ~1000`, linear search often wins over binary search.
- **Hash maps vs arrays**: hash lookup is O(1) but with a constant
  factor of 50-100ns. Array index is ~1ns. For small collections,
  the array wins.
- **`Array.prototype.indexOf` (O(n))** beats `new Set(...)` lookup
  for tiny arrays — the Set construction cost dwarfs the lookup
  savings.

Big-O is a guide, not the answer. Benchmark when it matters.

## Time vs space

Big-O applies to both:

- **Time complexity**: how many operations.
- **Space complexity**: how much memory.

```js
// Time O(n), space O(1)
function sum(arr) {
  let s = 0;
  for (const x of arr) s += x;
  return s;
}

// Time O(n), space O(n)
function doubled(arr) {
  return arr.map(x => x * 2);   // allocates n-element array
}
```

The map allocates `n` new elements. The sum allocates one variable.
Same time class, different space classes.

You often trade one for the other:

- **Memoization** — spend O(n) space to drop O(2^n) time to O(n).
- **In-place sorts** (quicksort) — O(1) extra space but harder to
  parallelize.
- **External sorts** — stream from disk in O(n log n) time, O(1)
  RAM.

## How to read a piece of code

Walk the structure:

```js
function example(arr) {
  for (let i = 0; i < arr.length; i++) {      // O(n)
    for (let j = i + 1; j < arr.length; j++) { // O(n)
      console.log(arr[i], arr[j]);              // O(1)
    }
  }
}
```

`n * n * 1 = O(n^2)`. Two nested loops over the same input.

```js
function example(arr) {
  for (const x of arr) {              // O(n)
    if (arr.includes(x * 2)) {        // O(n) — includes is linear
      console.log(x);
    }
  }
}
```

That's O(n) outer * O(n) inner = O(n^2). Easy to miss because
`includes` looks like one operation. Fix with a `Set`:

```js
function example(arr) {
  const set = new Set(arr);            // O(n)
  for (const x of arr) {               // O(n)
    if (set.has(x * 2)) console.log(x); // O(1)
  }
}
```

Now O(n). A common refactor — replace linear scans with hash
lookups.

## Amortized analysis (teaser)

Some operations are cheap *on average* even when one operation can
be expensive:

```js
const arr = [];
for (let i = 0; i < n; i++) arr.push(i);
```

`push` is "O(1) amortized." Most pushes are O(1). But when the
underlying array runs out of capacity, the engine reallocates and
copies — O(n) for that one push. Spread across n pushes, the total
work is O(n), so each push averages O(1).

The full story is in **Amortized complexity** — for now, know that
"amortized O(1)" means "on average per operation across many
operations, not for any single one."

`Array.prototype.push`, hash map insert, dynamic array growth — all
amortized O(1).

## Worst, average, best

A single algorithm has three numbers worth knowing:

- **Worst case**: pathological input. Usually what you report.
- **Average case**: random input. What you usually see.
- **Best case**: ideal input. Rarely useful alone.

Quicksort:

- Worst: O(n^2) on already-sorted input with naive pivot.
- Average: O(n log n).
- Best: O(n log n).

Hash map lookup:

- Worst: O(n) when every key collides.
- Average: O(1).
- Best: O(1).

In production, you mostly care about worst-case bounds for
adversarial inputs (security, untrusted users) and average-case for
typical workloads.

## Common pitfalls

### Counting only the outer loop

```js
for (const x of arr) {           // looks like O(n)
  doSomeWork(x);                  // but what is doSomeWork?
}
```

If `doSomeWork` is O(n) (e.g., does a `.filter` over `arr`), the
total is O(n^2). Look inside every call.

### Ignoring built-in costs

`arr.indexOf`, `arr.includes`, `arr.find` are O(n). String
concatenation in a loop can be O(n^2) in some engines. `JSON.parse`
is O(n) in input size. None of these are free.

### Confusing input size

```js
function isPrime(n) {
  for (let i = 2; i * i <= n; i++) {
    if (n % i === 0) return false;
  }
  return true;
}
```

Looks O(sqrt(n)) in the value of `n`. But the *input size* of `n`
in bits is `log n`. In bit complexity, this is exponential. Mostly
ignored unless you're doing cryptography.

### Premature optimization

"This is O(n^2), let me rewrite to O(n)." For `n = 10`, the O(n^2)
might be 100x faster due to constant factors. Profile first.

## The mental model

Big-O is the **shape** of how an algorithm scales. It tells you the
slope, not the y-intercept.

Two practical rules:

1. **Aim for O(n log n) or better** on anything that processes
   user-controlled `n`. O(n^2) breaks at scale.
2. **Constant factors matter for small n**. The "better" algorithm
   isn't always faster on small inputs — benchmark when it
   matters.

For interview-style problems: identify nested loops, identify hash
lookups vs linear scans, and watch for hidden work inside built-ins.
That covers 80% of complexity analysis.
