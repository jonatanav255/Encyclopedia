# DP patterns

The **Dynamic programming intro** page covers the mental model
and basic 1D state. This page is the next layer: the patterns
you see in interviews, competitive programming, and (yes) real
production code. Each pattern has a recognizable shape, a
canonical example, and pitfalls.

If you can identify which pattern a new problem matches, you've
already done 80% of the solving.

## 1) Linear DP — "decision per element"

State: `dp[i]` = best answer considering the first `i` items.
Transition: a small constant number of choices at each step.

Examples:

- **House robber** — `dp[i] = max(dp[i-1], dp[i-2] + nums[i])`.
- **Climbing stairs** — `dp[i] = dp[i-1] + dp[i-2]`.
- **Max subarray (Kadane's)** — `dp[i] = max(nums[i], dp[i-1] + nums[i])`.
- **Longest increasing subsequence (LIS)** — see **DP intro**.

Recognize when: scanning left-to-right, each position's answer
depends on a small bounded window of earlier positions.

## 2) Interval DP — "best way to split a range"

State: `dp[i][j]` = best answer for the range `[i, j]`.
Transition: try every split point `k` between `i` and `j`.

```js
for (let len = 2; len <= n; len++) {
  for (let i = 0; i + len <= n; i++) {
    const j = i + len - 1;
    dp[i][j] = Infinity;
    for (let k = i; k < j; k++) {
      dp[i][j] = Math.min(dp[i][j], dp[i][k] + dp[k+1][j] + cost(i, k, j));
    }
  }
}
```

O(n³) time, O(n²) memory. Classic examples:

- **Matrix-chain multiplication** — minimize scalar multiplications.
- **Optimal BST** — minimize expected search cost.
- **Burst balloons** — count combinations after popping in best order.
- **Stone merging / palindrome partitioning**.

Pitfall: the natural recursion is "pick one element to be last,"
not "split at k." That formulation often makes the transition
cleaner (e.g., burst balloons: pick which balloon to burst last).

## 3) Knapsack — 0/1 and unbounded

**0/1 knapsack**: each item once. State `dp[i][w]` = best value
using first `i` items with capacity `w`. O(n · W).

**Unbounded knapsack**: each item unlimited. Loop weights in the
**outer** loop (counts combinations) or items in the outer loop
(counts permutations) — order matters and is the #1 bug.

```js
// 0/1 — items outer, capacity inner, capacity loop decreasing
for (const [w, v] of items)
  for (let c = W; c >= w; c--)
    dp[c] = Math.max(dp[c], dp[c - w] + v);

// Unbounded — capacity increasing
for (const [w, v] of items)
  for (let c = w; c <= W; c++)
    dp[c] = Math.max(dp[c], dp[c - w] + v);
```

The capacity-loop direction encodes whether each item is reusable.
Memorize this once.

Pseudo-polynomial: O(n · W) is polynomial in `W` (the value)
but **exponential in the number of bits to represent `W`**. For
large weights, this isn't actually polynomial — that's why
knapsack is NP-hard despite this DP.

## 4) DP on subsets — "bitmask DP"

State: `dp[mask][...]` where `mask` is a bitmask over a small set
(typically `n ≤ 20`).

```js
// Held-Karp: shortest tour visiting all cities, ending at city `i`
// dp[mask][i] = best cost
for (let mask = 1; mask < (1 << n); mask++) {
  for (let i = 0; i < n; i++) {
    if (!(mask & (1 << i))) continue;
    const prev = mask ^ (1 << i);
    if (prev === 0) { dp[mask][i] = (i === 0 ? 0 : Infinity); continue; }
    for (let j = 0; j < n; j++) {
      if (!(prev & (1 << j))) continue;
      dp[mask][i] = Math.min(dp[mask][i], dp[prev][j] + cost[j][i]);
    }
  }
}
```

O(n² · 2^n). The standard exact solver for **TSP**, optimal
assignment, set cover with small ground set, and various
scheduling problems.

Pitfall: it's tempting to "just memoize the recursion." For
n=20 that's already 20M states; iterative bottom-up is usually
2-5× faster than memo recursion due to cache locality and lower
overhead.

## 5) DP on trees

State: `dp[node][...]` computed by post-order traversal. Each
parent combines children's DP values.

Classic problems:

- **Maximum independent set on a tree** — `dp[node][0]` (excluded),
  `dp[node][1]` (included). Children contribute `max(child[0],
  child[1])` when parent is excluded; only `child[0]` when included.
- **Tree diameter** — longest path through each node = sum of
  top-two children's depths.
- **Re-rooting (DP twice)** — answer "for each node, what is the
  result if it were the root?" Compute downward DP, then reroot
  in O(1) per node.

Memory: O(V). Time: O(V) for most tree-DP problems.

## 6) Digit DP

For "count numbers in `[L, R]` with some property of their digits."
State: position, tight-flag (still bounded by upper limit), plus
problem-specific state (digit sum mod k, last digit, etc.).

```js
function countLE(N, isValid) {
  const digits = String(N).split('').map(Number);
  const memo = new Map();
  function dfs(pos, tight, ...state) {
    if (pos === digits.length) return isValid(...state) ? 1 : 0;
    const key = `${pos}|${tight}|${state.join(',')}`;
    if (memo.has(key)) return memo.get(key);
    const max = tight ? digits[pos] : 9;
    let total = 0;
    for (let d = 0; d <= max; d++) {
      total += dfs(pos + 1, tight && d === max, /* updated state */);
    }
    memo.set(key, total);
    return total;
  }
  return dfs(0, true /* initial state */);
}
```

Used for problems like "count numbers ≤ N whose digits sum to k"
or "count numbers where no two adjacent digits differ by more
than 1." Hard to spot, mechanical once you do.

## 7) DP optimization patterns

Once you have a working DP, sometimes you can drop a factor:

- **Rolling array** — if `dp[i]` only depends on `dp[i-1]`, keep
  two rows instead of `n` rows. O(n) → O(1) space (per row).
- **Convex hull trick** — when the transition looks like
  `dp[i] = min_j (dp[j] + a[j] · b[i] + c[i])` with the right
  monotonicity, O(n²) becomes O(n log n) or O(n).
- **Knuth's optimization** / **divide and conquer DP** — for
  certain interval DPs where the optimal split index is monotone,
  reduces O(n³) to O(n²).
- **SOS DP (sum over subsets)** — `dp[mask] = sum over submasks
  of mask`. Naive O(3^n); with the technique, O(n · 2^n).

These are competitive-programming material. In production code,
get the readable O(n²) right first; optimize only if profiling
demands it.

## Pattern-recognition cheat sheet

| Shape of problem | Pattern |
| --- | --- |
| Decision per element, small lookback | Linear DP |
| Best split of a range | Interval DP |
| Pick a subset under capacity | Knapsack |
| Small `n` (≤ ~20), visit-each problem | Bitmask DP |
| Recursion mirrors tree structure | Tree DP |
| Count numbers with digit property | Digit DP |
| Two pointers / monotone DP | Optimization patterns |

When in doubt, write the most-natural recurrence, memoize it,
then check whether the state space is small enough to enumerate
iteratively. Most "I can't solve this DP" problems are really
"I picked the wrong state."
