# Backtracking and branch-and-bound

Backtracking is **DFS through the space of partial solutions**
with the ability to abandon a branch as soon as you know it can't
lead to a valid (or optimal) answer. Branch-and-bound adds a
**bound** (a fast over- or under-estimate of the best achievable
score from a partial state) to prune even harder.

Both are exponential in the worst case. Both can be wildly fast
in practice if the pruning is tight. The skill is in the pruning,
not the recursion.

## The skeleton

Every backtracking solver looks the same:

```js
function solve(state) {
  if (isComplete(state)) {
    record(state);
    return;
  }
  for (const choice of candidates(state)) {
    if (!isFeasible(state, choice)) continue; // PRUNE
    apply(state, choice);
    solve(state);
    undo(state, choice);                       // BACKTRACK
  }
}
```

Three responsibilities to get right:

1. **`isFeasible`** — the constraint check. Tighter checks → more
   pruning. Cheap checks → faster per node. Balance matters.
2. **`apply` / `undo`** — keep the state mutation symmetric.
   Forgetting to undo is the #1 bug. Use immutable structures if
   you can afford the allocation.
3. **`candidates` ordering** — try most-constrained choices first
   (e.g., the Sudoku cell with the fewest legal values). This
   often shrinks the tree dramatically.

## N-queens

Place N queens on an N×N board so no two attack each other.
Backtracking with column / diagonal sets:

```js
function nQueens(n) {
  const cols = new Set(), d1 = new Set(), d2 = new Set();
  const placements = [];
  let count = 0;
  function place(row) {
    if (row === n) { count++; return; }
    for (let c = 0; c < n; c++) {
      if (cols.has(c) || d1.has(row - c) || d2.has(row + c)) continue;
      cols.add(c); d1.add(row - c); d2.add(row + c);
      placements.push(c);
      place(row + 1);
      placements.pop();
      cols.delete(c); d1.delete(row - c); d2.delete(row + c);
    }
  }
  place(0);
  return count;
}
```

The sets are the constraint check — O(1) feasibility test. Naive
double-loop over already-placed queens would be O(n) per check
and dominate the running time on big boards.

## Sudoku

Classic constraint-satisfaction. Pick the empty cell with the
**fewest candidates** (most-constrained-variable heuristic), try
each candidate, recurse, undo. The MCV heuristic alone takes a
"5 second" naive solver down to milliseconds.

Combine with **constraint propagation** (every time you fill a
cell, eliminate that digit from its row / column / box's
candidate sets) for another ~10× speedup. That's how Norvig's
famous Sudoku solver works.

## Permutations and combinations

For combinatorial enumeration, the cleanest pattern is "decision
at each position":

```js
function permutations(arr) {
  const result = [], current = [];
  const used = new Array(arr.length).fill(false);
  function dfs() {
    if (current.length === arr.length) {
      result.push([...current]);
      return;
    }
    for (let i = 0; i < arr.length; i++) {
      if (used[i]) continue;
      used[i] = true; current.push(arr[i]);
      dfs();
      current.pop(); used[i] = false;
    }
  }
  dfs();
  return result;
}
```

For combinations, pass a `start` index so each level only chooses
from items after the last picked, eliminating duplicates without
needing a `Set`.

## Branch-and-bound

Backtracking for **optimization**: you're minimizing (or maximizing)
some cost. At each node, compute a **bound** — a cheap value that's
guaranteed to be no better than the best achievable from this state.

- Minimizing? Bound is a **lower bound** on cost from here. If
  `lower_bound(state) >= best_so_far`, prune.
- Maximizing? Bound is an **upper bound** on profit. If
  `upper_bound(state) <= best_so_far`, prune.

The bound must be **admissible**: it never overestimates the
remaining benefit (or underestimates the remaining cost). A bad
bound prunes nothing or — worse — prunes the optimum.

## TSP via branch-and-bound

For the traveling salesman problem on a small graph (~30 cities),
exact DP (Held-Karp) is O(n² · 2^n). Branch-and-bound can beat it
on average by aggressively pruning.

Lower bound idea: for any unfinished tour, every remaining city
must be entered and exited at least via its two cheapest incident
edges. Sum those over all unvisited cities and divide by 2 — a
valid lower bound. Compute incrementally as the tour grows.

This is roughly how solvers like Concorde solve thousand-city TSP
instances optimally — branch-and-cut with LP-relaxation bounds.

## Branch-and-bound vs DP

- **DP** is for problems with overlapping subproblems and a small
  state space (polynomial). Knapsack with small weights, edit
  distance, matrix-chain.
- **Branch-and-bound** is for problems with huge state spaces
  but tight bounds (NP-hard combinatorial problems). TSP, large
  knapsack, ILP, planning.

If you can characterize the state in O(poly(n)) and use a table,
prefer DP — it's predictable. Reach for branch-and-bound when DP
would need exponential state.

## Iterative deepening

For game trees and AI planning, you often don't know how deep the
solution is. **Iterative deepening DFS (IDDFS)** runs DFS to depth
1, then 2, then 3, etc. Memory stays O(d) (DFS-like) while
guaranteeing the first solution found is shallowest (BFS-like).

The "wasted work" of repeating shallower searches is bounded by a
constant factor in branching-factor `b` (geometric series), so the
asymptotic cost is the same as plain DFS to the answer depth.

Used in chess engines (Stockfish's iterative-deepening alpha-beta)
to keep memory predictable while giving "best move so far" any
time the search is interrupted.

## Don't write backtracking for problems with cleaner algorithms

If the problem has structure — sortable, monotone, matroid-shaped,
graph-shortest-path — there's almost always a polynomial algorithm.
Backtracking is the fallback when nothing better fits. Always
ask "is this actually NP-hard, or am I just being lazy with
analysis?" before reaching for it.
