# Dynamic programming intro

Dynamic programming (DP) is **recursion with memoization** applied
to problems with two properties:

1. **Overlapping subproblems** — the naive recursion solves the
   same subproblem many times.
2. **Optimal substructure** — the optimal answer combines optimal
   answers to subproblems.

If both hold, caching subproblem results turns exponential
recursion into polynomial time.

The name "dynamic programming" is historical (Bellman, 1950s) and
not very descriptive. Think "memoized recursion" — that's 90% of
it.

## The pattern

```js
// Naive recursion — exponential
function fib(n) {
  if (n < 2) return n;
  return fib(n - 1) + fib(n - 2);
}

fib(40);   // ~1 second — recomputes the same values repeatedly
```

`fib(5)` calls `fib(4)` and `fib(3)`. `fib(4)` calls `fib(3)` and
`fib(2)`. `fib(3)` is computed twice. The number of duplicate calls
grows exponentially.

Memoization fixes it:

```js
function fib(n, memo = new Map()) {
  if (n < 2) return n;
  if (memo.has(n)) return memo.get(n);
  const result = fib(n - 1, memo) + fib(n - 2, memo);
  memo.set(n, result);
  return result;
}

fib(40);    // microseconds
fib(1000);  // works, but now hitting BigInt territory
```

Same recursion. Each unique `n` is computed once. O(2^n) becomes
O(n).

## Top-down vs bottom-up

Two implementation styles:

**Top-down (memoized recursion)**: write the recursion naturally;
cache results.

```js
function fib(n, memo = new Map()) {
  if (n < 2) return n;
  if (memo.has(n)) return memo.get(n);
  const r = fib(n - 1, memo) + fib(n - 2, memo);
  memo.set(n, r);
  return r;
}
```

**Bottom-up (tabulation)**: iterate from base cases up to the
answer.

```js
function fib(n) {
  if (n < 2) return n;
  const dp = [0, 1];
  for (let i = 2; i <= n; i++) dp[i] = dp[i - 1] + dp[i - 2];
  return dp[n];
}

// Space-optimized: only need the last two values
function fib(n) {
  let [a, b] = [0, 1];
  for (let i = 0; i < n; i++) [a, b] = [b, a + b];
  return a;
}
```

Both are O(n). Top-down is easier to write (mirrors the recursion).
Bottom-up avoids recursion overhead and often allows space
optimization (you can drop old rows of the table).

In practice: write top-down to understand the problem, convert to
bottom-up if you need the performance or want to drop memory.

## Identifying DP problems

The giveaway is **"optimize over choices."** Each step you have a
choice; you want the best total outcome.

Classics:

- **Fibonacci, climbing stairs** — count paths.
- **Knapsack** — maximize value within a weight budget.
- **Edit distance** — minimum edits to transform A into B.
- **Longest common subsequence (LCS)** — used by diff tools.
- **Longest increasing subsequence (LIS)** — sub-sequence of an
  array.
- **Coin change** — fewest coins to make N.
- **Matrix chain multiplication** — optimal parenthesization.
- **Word break** — can a string be split into dictionary words.

If you see "find the best/longest/shortest/cheapest" plus "choices
at each step," try DP.

## The state and transition

Every DP problem boils down to two questions:

1. **What is the state?** What parameters uniquely identify a
   subproblem?
2. **What is the transition?** How do you compute a state from
   smaller states?

For Fibonacci: state is `n`; transition is `fib(n) = fib(n-1) +
fib(n-2)`.

For LCS of strings `a` and `b`: state is `(i, j)` (positions in `a`
and `b`); transition is:

```js
if (a[i] === b[j]) lcs(i, j) = 1 + lcs(i+1, j+1)
else                lcs(i, j) = max(lcs(i+1, j), lcs(i, j+1))
```

Number of unique states tells you the time complexity: number of
states * work per state.

For LCS: O(|a| * |b|) states, O(1) work per state, so O(|a| * |b|)
time.

## Worked example: edit distance

Given two strings, find the minimum number of single-character
edits (insert, delete, substitute) to turn one into the other.
This powers `diff`, spell-checkers, and DNA-sequence alignment.

```js
function editDistance(a, b) {
  const m = a.length, n = b.length;

  // dp[i][j] = edit distance between a[0..i) and b[0..j)
  const dp = Array.from({ length: m + 1 }, () => new Array(n + 1));

  // Base cases: turning empty string into b[0..j) takes j inserts,
  // and a[0..i) into empty takes i deletes.
  for (let i = 0; i <= m; i++) dp[i][0] = i;
  for (let j = 0; j <= n; j++) dp[0][j] = j;

  for (let i = 1; i <= m; i++) {
    for (let j = 1; j <= n; j++) {
      if (a[i - 1] === b[j - 1]) {
        dp[i][j] = dp[i - 1][j - 1];   // no edit needed
      } else {
        dp[i][j] = 1 + Math.min(
          dp[i - 1][j],     // delete a[i-1]
          dp[i][j - 1],     // insert b[j-1]
          dp[i - 1][j - 1], // substitute a[i-1] -> b[j-1]
        );
      }
    }
  }

  return dp[m][n];
}

editDistance('kitten', 'sitting');   // 3
// k -> s, e -> i, append g
```

How to read it:

- **State**: `(i, j)` = how many chars consumed from each string.
- **Transition**: either the chars match (no edit), or we pay 1 +
  the best of three smaller subproblems (delete / insert /
  substitute).
- **Base case**: turning empty into a string of length k costs k
  inserts.
- **Answer**: `dp[m][n]` — full string to full string.

Complexity: O(m * n) time, O(m * n) space. Can be reduced to O(min(m, n))
space by keeping only the previous row.

To recover the actual edit sequence (not just the count), walk
backward from `dp[m][n]` to `dp[0][0]` choosing the transition that
led to each cell. This is the **traceback**.

Real-world uses:

- `git diff` uses an LCS variant (Myers' algorithm) which is
  related to edit distance.
- Spell-checkers rank suggestions by edit distance.
- Bioinformatics aligns DNA/protein sequences with weighted edit
  distances (Needleman-Wunsch, Smith-Waterman).

## The knapsack archetype

Given items with weights and values, pick a subset with maximum
total value subject to a weight limit.

```js
// 0/1 knapsack — each item used at most once
function knapsack(weights, values, capacity) {
  const n = weights.length;
  const dp = Array.from({ length: n + 1 }, () => new Array(capacity + 1).fill(0));

  for (let i = 1; i <= n; i++) {
    for (let w = 0; w <= capacity; w++) {
      dp[i][w] = dp[i - 1][w];   // skip item i-1
      if (weights[i - 1] <= w) {
        dp[i][w] = Math.max(
          dp[i][w],
          dp[i - 1][w - weights[i - 1]] + values[i - 1],   // take item i-1
        );
      }
    }
  }

  return dp[n][capacity];
}
```

State: `(i, w)` = considering items `0..i)` with `w` capacity
remaining.

Complexity: O(n * capacity). Note this is **pseudo-polynomial** — it
depends on the numeric value of capacity, not just its bit length.
Knapsack is NP-hard in the general case.

The knapsack pattern shows up in resource allocation, budget
optimization, and (loosely) compiler register allocation.

## LIS — longest increasing subsequence

```js
function lis(arr) {
  const dp = new Array(arr.length).fill(1);
  for (let i = 1; i < arr.length; i++) {
    for (let j = 0; j < i; j++) {
      if (arr[j] < arr[i]) dp[i] = Math.max(dp[i], dp[j] + 1);
    }
  }
  return Math.max(...dp);
}
```

`dp[i]` = length of the LIS ending at index `i`. O(n^2). There's
an O(n log n) version using patience sorting; if your interviewer
asks for "better," reach for that.

Patience sorting gives you the **length** of the LIS cheaply: scan
left to right, binary-search each value into a "tails" array. The
final length of `tails` is the LIS length. Reconstructing the
actual subsequence is subtler — `tails` is not the answer itself,
since later writes can clobber earlier indices. You need a parent
pointer per input position recording "what was the predecessor in
the pile this element was placed into," then walk backwards from
the last element placed in the tallest pile.

## Common pitfalls

### Solving non-DP problems with DP

DP requires **overlapping subproblems**. If the recursion has no
overlap (e.g., binary tree problems where each subtree is unique),
memoization is pure overhead. Plain recursion is fine.

### State too big

If the state has 4 parameters each ranging over n values, you have
O(n^4) cells. Often there's a smaller equivalent state if you think
harder. "DP that won't fit in memory" usually means "find a smaller
state."

### Forgetting the base case

```js
function fib(n, memo) {
  if (memo.has(n)) return memo.get(n);
  const r = fib(n - 1, memo) + fib(n - 2, memo);   // n=0 -> n=-1 -> ...
  memo.set(n, r);
  return r;
}
```

Without `if (n < 2) return n`, the recursion never bottoms out.

### Memoizing a side-effecting function

```js
function expensive(x, memo) {
  if (memo.has(x)) return memo.get(x);
  log(x);                          // side effect
  const r = doWork(x);
  memo.set(x, r);
  return r;
}
```

The second call returns the cached value without logging. Memoize
only pure functions.

### Mutating the input

```js
function dp(arr) {
  arr.sort();   // surprise — mutates caller's array
  // ...
}
```

DP often sorts or otherwise reorders. Make a copy or document the
mutation.

### Confusing "fits in DP" with "should be DP"

Some problems have a DP solution but a greedy or graph algorithm is
better. Always ask "is there a simpler algorithm?" before
committing to a DP table.

## The mental model

DP is **a search through choices where the choice tree has
overlapping branches**. Cache the overlaps; turn exponential into
polynomial.

Two practical rules:

1. **Start with naive recursion.** Identify the state. Identify
   the transition. Add memoization. That's DP.
2. **Convert to bottom-up if you need speed or want to optimize
   memory.** Most top-down DPs become loops over a 1D or 2D
   table.

The hard part isn't the algorithm — it's spotting the recursive
structure. Common shapes: "best path through a grid," "subset with
some property," "transform one sequence into another," "fewest
steps to reach a target." When you see these, ask "what's my
state?"
