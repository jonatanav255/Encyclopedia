# Binary search

Binary search finds a value in a **sorted** array in O(log n) by
halving the search range each step. The idea is simple. The
implementation is famously easy to get wrong.

Donald Knuth: "Although the basic idea of binary search is
comparatively straightforward, the details can be surprisingly
tricky." A 1986 study found bugs in published binary search code
from textbooks.

## The basic shape

```js
function binarySearch(arr, target) {
  let lo = 0;
  let hi = arr.length - 1;

  while (lo <= hi) {
    const mid = lo + Math.floor((hi - lo) / 2);
    if (arr[mid] === target) return mid;
    if (arr[mid] < target) lo = mid + 1;
    else hi = mid - 1;
  }

  return -1;   // not found
}
```

Three things to notice:

1. **`lo + (hi - lo) / 2`**, not `(lo + hi) / 2`. See "overflow"
   below.
2. **`lo <= hi`**, not `lo < hi`. Critical for single-element
   ranges.
3. **`mid + 1`** and **`mid - 1`** when narrowing. Without the +/-1,
   you loop forever.

## Off-by-one pitfalls

### The overflow that bit Java

```js
const mid = (lo + hi) / 2;        // BAD in languages with fixed ints
```

In Java/C, `lo + hi` overflows for large arrays. Joshua Bloch
publicly fixed this in JDK's `Arrays.binarySearch` in 2006 — the
bug had sat in the JDK for nearly a decade, and similar bugs have
been found in published textbook code going back to Bentley's
*Programming Pearls* analysis.

In JavaScript, numbers go up to 2^53 - 1 (`MAX_SAFE_INTEGER`), so
arrays small enough to fit in memory will never overflow this way.
Still write `lo + (hi - lo) / 2` out of habit — it generalizes to
other languages.

### The infinite loop

```js
while (lo < hi) {
  const mid = lo + Math.floor((hi - lo) / 2);
  if (arr[mid] < target) lo = mid;     // forgot +1
  else hi = mid;
}
```

When `lo + 1 === hi`, `mid === lo`, and `lo = mid` doesn't advance.
Forever loop. Always advance past `mid` on the side you're keeping.

### Closed vs half-open intervals

Two valid styles:

```js
// Closed: [lo, hi]   - both inclusive
let lo = 0, hi = arr.length - 1;
while (lo <= hi) { ... hi = mid - 1; }

// Half-open: [lo, hi)  - hi exclusive
let lo = 0, hi = arr.length;
while (lo < hi) { ... hi = mid; }
```

Both work. Pick one and stick with it across your codebase.
Mixing them is the most common source of off-by-one bugs. The
half-open form generalizes better to `lower_bound`/`upper_bound`.

## Lower bound and upper bound

In real code, "find the value" is rarer than "find where it would
go." That's `lower_bound` (first index `>= target`) and
`upper_bound` (first index `> target`).

```js
// First index i where arr[i] >= target.
// If target > all elements, returns arr.length.
function lowerBound(arr, target) {
  let lo = 0;
  let hi = arr.length;
  while (lo < hi) {
    const mid = lo + Math.floor((hi - lo) / 2);
    if (arr[mid] < target) lo = mid + 1;
    else hi = mid;
  }
  return lo;
}

// First index i where arr[i] > target.
function upperBound(arr, target) {
  let lo = 0;
  let hi = arr.length;
  while (lo < hi) {
    const mid = lo + Math.floor((hi - lo) / 2);
    if (arr[mid] <= target) lo = mid + 1;
    else hi = mid;
  }
  return lo;
}
```

With these you can:

- **Find** the target: `lowerBound(arr, t)` then check
  `arr[i] === t`.
- **Count** equal values: `upperBound - lowerBound`.
- **Find** the insertion point to keep the array sorted:
  `lowerBound`.
- **Range queries**: "how many values in `[a, b]`?" is
  `upperBound(b) - lowerBound(a)`.

These two functions are more useful than the plain "is it here"
search. They're the standard in C++'s `<algorithm>` for a reason.

## When the array isn't there

Binary search only requires a **monotone predicate** — not an actual
sorted array. If you can answer "is `X` big enough?" with yes/no
and "bigger X always answers yes once smaller X does," you can
binary search on `X`.

```js
// Find smallest X such that predicate(X) is true.
// predicate(X) must be monotone: false ... false true ... true.
function binarySearchOnAnswer(lo, hi, predicate) {
  while (lo < hi) {
    const mid = lo + Math.floor((hi - lo) / 2);
    if (predicate(mid)) hi = mid;
    else lo = mid + 1;
  }
  return lo;
}
```

Classic use cases:

- **"What's the minimum capacity ship that can deliver these
  packages in D days?"** Predicate: `canDeliverWith(capacity)`.
- **"What's the largest minimum distance between K elements?"**
  Predicate: `canPlaceK(distance)`.
- **"What's the largest sum that's still less than X?"** Predicate:
  `prefixSum(i) <= X`.

This pattern shows up constantly in interview problems and
real-world scheduling/sizing problems. Recognize it when you see
"find the minimum/maximum value such that..."

## On floats

```js
function sqrtBinary(x, eps = 1e-9) {
  let lo = 0;
  let hi = Math.max(1, x);
  while (hi - lo > eps) {
    const mid = (lo + hi) / 2;
    if (mid * mid < x) lo = mid;
    else hi = mid;
  }
  return lo;
}
```

Float binary search uses a tolerance (`eps`) instead of integer
arithmetic. Don't `<= hi`, don't `mid + 1`. The loop converges by
shrinking the range below `eps`.

Fixed iteration count (e.g., `for 100`) is often safer than a
tolerance condition — guarantees termination.

## Performance notes

- **O(log n) comparisons**. For `n = 1e9`, that's ~30 comparisons.
- **Cache misses dominate** for large arrays. The first few mid
  positions touch widely-spaced memory. CPU prefetchers struggle.
- **Linear search wins for small n** (typically `n < 50`).
  V8's `Array.prototype.indexOf` is heavily optimized; binary search
  in user code carries function-call overhead.

For sorted-array lookups in JS, consider whether a `Map` or `Set`
would serve better — O(1) lookup beats O(log n) when you don't need
ordering.

## When NOT to use binary search

- **Unsorted data.** Sort first if you'll search many times.
  Sorting is O(n log n); searching once is O(n) — only worth
  sorting if you'll search more than `log n` times.
- **Small arrays.** Linear search wins for ~50 elements or fewer.
- **Frequent inserts/deletes.** Maintaining sorted order is O(n)
  per write. Use a balanced BST, skip list, or just a hash map.
- **Approximate matches** where "close" varies non-monotonically.
  Binary search needs a clean comparator.

## Common pitfalls

### Searching an unsorted array

```js
binarySearch([3, 1, 4, 1, 5, 9, 2], 4);   // undefined behavior
```

Silently returns nonsense. No error. Sort first, or use linear
search.

### Comparing wrong types

```js
binarySearch(['10', '2', '30'], '10');    // strings sort lex, not numeric
```

`'10' < '2'` is true (lexicographic). The "sorted" array isn't
sorted numerically. Always pass a comparator (or convert types)
when working with strings/objects.

### Using `===` with floats

```js
if (arr[mid] === target) ...   // floats rarely match exactly
```

Use `Math.abs(arr[mid] - target) < eps` or use `lowerBound`-style
search.

### Iterating until `lo === hi` then forgetting to check the value

```js
const i = lowerBound(arr, target);
return arr[i];   // wrong if target not present — returns next greater
```

`lowerBound` returns an insertion point. Check `arr[i] === target`
before treating it as "found."

### Mutating during the loop

Don't. Binary search assumes a static sorted view. If the array
changes mid-loop, all bets are off.

## The mental model

Binary search is **a divide-and-conquer search over any monotone
yes/no condition**.

Two practical rules:

1. **Use `lowerBound` as your default**, not the "find or -1"
   version. It composes into range queries and insertions.
2. **Write the half-open form `[lo, hi)`** and stop second-guessing
   the boundary conditions. Test with empty array, single element,
   target absent, target at both ends.

For numerical problems (capacity, scheduling, sizing), recognize
"find the smallest X such that..." as binary-search-on-the-answer.
That pattern unlocks an entire class of problems.
