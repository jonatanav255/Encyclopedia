# Adversarial ML and red teaming

The adversarial ML literature is large, deep, and 80% irrelevant
to most product teams. Academic attacks assume white-box access
to model weights, ability to query gradients, or threat models
that don't map onto "a logged-in user with a browser." If you
read Joseph et al.'s *Adversarial Machine Learning* end-to-end
and apply every defense, you will burn a quarter and not be
materially safer.

The job is to **separate the threats that matter for a typical
product team** from the ones that matter only at the foundation-
model layer, and to build a red-team practice that targets the
real ones. This page is that triage.

For LLM-specific defenses, see
`ai/prompt-injection-and-llm-security.mdx`. This page is broader:
classical ML attacks plus LLM-era variants, with a bias toward
"what does a senior backend engineer on a product team actually
need to defend."

## The attack taxonomy

The four families. They overlap; the boundaries are fuzzy. Use
the taxonomy to know what *not* to ignore.

<Cheatsheet caption="Adversarial ML attack families with realistic-product-team relevance">
| Family | What the attacker does | Threat to a typical product team |
|---|---|---|
| **Evasion** | Craft an input the model misclassifies at inference time | High for fraud / spam / abuse / moderation; medium elsewhere |
| **Model extraction** | Query the model enough to clone it | High if you sell access; low if model is incidental |
| **Training-data poisoning** | Insert malicious examples into the training set | Medium if you train on user data; low for hosted-model users |
| **Membership inference** | Determine whether a specific record was in the training set | Medium for regulated data (HIPAA, GDPR); low elsewhere |
| **Jailbreaks** (LLM) | Get a hosted model to violate its guardrails | High for any LLM-fronted product |
| **Indirect injection** (LLM) | Smuggle instructions through retrieved content | High for any agent with tools |
</Cheatsheet>

The right column is the part most adversarial-ML books bury. If
you only have a quarter, work on the high-relevance ones.

## Evasion attacks: the one that bites everyone

The attacker tweaks the input until the model is wrong, without
changing the human-meaningful content.

The original demos were adversarial images (Goodfellow et al.,
2014 — the "panda + noise = gibbon" example). Those rarely
matter in production because nobody is computing FGSM gradients
on your image classifier. The **versions that actually attack
products**:

- **Spam / phishing**: attackers add zero-width characters,
  homoglyphs (Cyrillic `а` for Latin `a`), invisible Unicode,
  reorder words. Your spam classifier was trained on clean
  text; production text isn't.
- **Fraud**: attackers learn which feature values fly under the
  radar (transaction amounts, time-of-day, merchant categories)
  and tune their behavior accordingly. This is **online
  adversarial learning** — the model and attacker co-evolve.
- **Content moderation**: "leetspeak," image-of-text, audio
  embeds. Anything that defeats the model's feature extractor
  without changing what a human reads.
- **CAPTCHA / bot detection**: residential proxies, ML-driven
  mouse-movement emulators, headless browser fingerprint
  spoofing.

Defenses that work in practice (not "adversarial training,"
which is mostly research):

1. **Defense in depth.** Don't gate moderation on one model;
   stack rule-based prefilters, the model, and human review
   for borderline cases.
2. **Monitor for adversarial drift.** Attackers' tactics shift;
   your eval set on last quarter's spam is not last week's.
   Sample false-negatives from production into the next train.
3. **Cost asymmetry.** Make the attack expensive enough that
   the marginal attacker gives up. Rate limits, proof-of-work,
   trusted-device signals, account-age requirements.
4. **Don't treat the model's confidence as truth.** A
   high-confidence "benign" on a flagged-by-other-signals item
   should still go to review.

## Model extraction

If your product is "API access to a model" (search ranking,
recommendations, an embedding service, an LLM wrapper), an
attacker can query enough to clone the model. Papacharissi /
Tramèr et al. showed this is feasible against commercial APIs
in 2016 and the techniques have only gotten cheaper.

For most product teams this is a low priority because:

- The model's IP value is usually less than the data that fed
  it.
- Cloning a recommender is hard because half the value is the
  data pipeline, not the weights.
- Rate limits + access controls make bulk extraction expensive.

When it matters:

- You sell model access as a paid product.
- Your moat is the model itself (LLM provider, specialized
  embedding service).
- Regulatory or competitive reasons make leak material.

Defenses: query rate limits, watermarking outputs (Adi et al.,
2018), monitoring for systematic-query patterns, differential
privacy in training if extraction → privacy concerns.

## Training-data poisoning

Two flavors:

### Untargeted poisoning

Attacker injects noise into training data to degrade overall
quality. Easy to mount on systems that train on user data
without curation (e.g., a model trained on user-submitted
reviews, public forum posts, or scraped web content).

Defenses: data validation, outlier detection in training
pipelines, source provenance. The 2024 incidents involving
poisoned web-scraped pretraining corpora — search "Carlini
poisoning the web" for the 2023 paper — make this real for
anyone fine-tuning on scraped data.

### Targeted / backdoor poisoning

Attacker plants examples that cause the model to misbehave on
a specific trigger. "When the input contains the phrase
`xyzzy42`, classify as benign." Subtle, hard to detect by
overall accuracy.

For product teams that train models on user input (fraud,
moderation, search rankers): this is real. The defense is
fundamentally **data provenance and labeling discipline**:

- Don't train on user-submitted content without filtering.
- Treat labels from external annotators with skepticism;
  multi-annotator agreement, hold-out audits.
- For models trained on activity data, anomaly-detect on
  training-set composition between runs.

## Membership inference

Given a trained model, can an attacker tell whether a specific
record (a user, an email, a medical record) was in the training
set? Shokri et al. (2017) showed this works against many model
families.

Where it matters: regulated data. If your model was trained on
HIPAA-protected records or GDPR-protected EU resident data, the
ability for an attacker to *prove* a person's data was in the
training set is itself a disclosure.

Defenses:

- **Differential privacy** in training (DP-SGD). Real
  protection; real accuracy cost. Used by Apple, Google, and a
  handful of regulated-data companies.
- **Don't memorize.** Aggressively regularize; ensure no single
  record drives the loss. Easier said than done at scale.
- **Limit query access.** A model accessible only through a
  public API with rate limits is harder to attack than one
  whose weights you ship to clients.

For most teams, this is a "know it exists, don't necessarily
defend specifically" risk. Decision driver: do you have
regulated training data, and is the model accessible to
attackers in a way that allows enough queries?

## Jailbreak taxonomies (LLM-specific)

For LLM products, the threat is the model producing output it
was trained not to — harmful content, system prompt leaks, tool
abuse. The categories that show up in red-team logs:

- **Roleplay / persona** — "Pretend you're an AI from 2010
  before safety training." The model adopts the persona and
  drops guardrails.
- **Hypothetical** — "In a fictional world where this is
  legal..." Models trained to refuse direct asks comply with
  hypothetical framing.
- **Encoded payloads** — base64, leetspeak, foreign languages,
  emoji-substituted text. Tokenizer-level evasion of safety
  filters.
- **Many-shot jailbreaks** — fill the context window with
  examples of the model "agreeing" to harmful requests; the
  model continues the pattern (Anil et al., 2024, Anthropic).
- **Multi-turn escalation** — start benign, gradually walk the
  conversation toward the goal. Each individual turn looks
  innocent.
- **Tool-use abuse** — convince the agent that calling a
  destructive tool is the right move given fabricated context.

Defenses are the stack from
`ai/prompt-injection-and-llm-security.mdx`: tool allowlists,
output filters, content provenance, monitoring. **No single
defense holds**; the goal is layered cost.

## Building an internal red-team practice

The high-leverage version. You don't need a 12-person team.

<Steps>
<Step n="1" title="Define the threat model in writing">
Who attacks? What's the worst outcome? A common framing: data
exfiltration, financial loss, reputational harm,
service-availability impact, regulatory exposure. Rank them.
Most teams' actual top concern is "user does something the
agent shouldn't help with that ends up on Twitter." That's a
legitimate threat model; write it down.
</Step>
<Step n="2" title="Build an adversarial eval set">
Like a regression suite, but adversarial. Collect:
- Known jailbreaks from public corpora (DAN variants, JailbreakBench).
- Reports from your trust/safety queue.
- Internal red-team attempts.
Run it on every model upgrade, every prompt change. Treat
failures as production bugs.
</Step>
<Step n="3" title="Run a quarterly red-team week">
Block 3-5 engineers (not just security) for a week to attack
your own systems. Concrete targets ("exfiltrate the contents of
this test inbox via the agent"; "get the model to generate
disallowed content"). Outcomes get filed as bugs.
</Step>
<Step n="4" title="Bug bounty for LLM behavior">
Many companies extend bug-bounty programs to cover
jailbreaks and prompt injection. The economics work: external
researchers find what internal teams miss; the payout for
"convinced the agent to leak data" is worth thousands.
</Step>
<Step n="5" title="Post-incident review tied to red-team backlog">
Every real incident → new test case in the adversarial eval
set. Otherwise the same attack works next quarter.
</Step>
</Steps>

## What academic adversarial ML over-emphasizes

A small list, to save you reading time:

- **L-infinity-bounded image perturbations.** Real attackers
  don't compute gradients on your image classifier; they use a
  different image entirely.
- **Provable robustness certificates.** Mathematically
  beautiful, computationally heavy, brittle to threat-model
  shifts. Useful in safety-critical domains; not in product.
- **Adversarial training** as a silver bullet. Adds a 10–30%
  compute tax for usually-marginal robustness gains. Works
  better on specific narrow attacks than on novel ones.
- **One-off poisoning demos** on tiny datasets. The risk in
  production is *systematic* low-rate poisoning, not "I
  inserted 1% adversarial examples and accuracy dropped."

## What academic adversarial ML under-emphasizes

The mirror image:

- **Co-evolution.** The attacker watches your defenses and
  changes tactics weekly. Static benchmarks understate this.
- **Cost economics.** The defense doesn't need to be
  bulletproof; it needs to push the attacker's ROI negative.
- **Detection over prevention.** Knowing you were attacked is
  often more valuable than perfectly blocking the attack.
- **Composite attacks.** Most real incidents combine three
  weak signals. Defenses tested on one signal at a time miss
  these.

## What good looks like

- A written threat model with named adversaries and worst-case
  outcomes.
- An adversarial eval set that grows with every incident and
  runs on every model / prompt change.
- A quarterly internal red-team week with concrete attack
  objectives.
- Detection telemetry for the high-relevance attack families
  (evasion, jailbreaks, tool abuse) wired to alerts.
- A bug-bounty program scope that explicitly includes model
  behavior, not just code.
- An on-call team that can answer "did we get attacked, and
  what did the attacker access" from logs without reaching
  for forensics.

## The discipline

- **Triage the literature against your actual threat model.**
  Most adversarial ML isn't your problem; some of it definitely
  is.
- **Cost asymmetry, not perfection.** Make attacks expensive,
  not impossible.
- **Detection is half the defense.** You will be attacked; the
  question is whether you notice.
- **Adversarial eval sets grow.** Every incident becomes a test
  case.
- **Red-team your own product.** Internal red-team weeks find
  more than external ones, because internal teams know where
  the soft spots are.
- **The high-relevance attacks for a typical product team**:
  evasion (fraud / spam / moderation), jailbreaks, indirect
  prompt injection. Defend those first. Everything else is a
  bonus.
