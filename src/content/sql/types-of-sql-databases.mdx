# Types of SQL Databases

"SQL database" is a broad category. They all speak roughly the
same query language and honor ACID, but they differ in feature
set, license, deployment model, and what they're best at. Here are
the ones you'll actually encounter, with honest opinions on when
each fits.

## Postgres

The default for new server-side projects in 2026. Open source
(PostgreSQL License), mature, rich type system, generous
extensibility model.

**Why pick it:**
- `JSONB` columns give you schemaless storage when you need it
  without leaving the relational world. Defeats most "we need
  MongoDB" arguments.
- Arrays, ranges, `tsvector` (full-text search), `PostGIS`
  (geographic), `pgvector` (vector embeddings) — built-in or
  extension.
- Excellent SQL standards compliance.
- `RETURNING` clause on INSERT/UPDATE/DELETE — gets the modified
  rows back in one round trip.
- Generally the most "no surprises" SQL behavior.

**Trade-offs:**
- Connection model is process-per-connection, which makes
  connection pooling mandatory at scale. (PgBouncer or
  application-level pooling.)
- Vacuum + autovacuum needs occasional attention on write-heavy
  tables.

If you don't have a specific reason to pick something else, pick
Postgres. See **Postgres** in the Data topic.

## MySQL

The other dominant open-source option (GPL). Historically the
default for the LAMP stack; still huge in legacy systems, WordPress
hosting, and managed cloud services.

**Why pick it:**
- Battle-tested at extreme scale (Facebook, YouTube, Uber).
- Faster simple primary-key reads than Postgres on equivalent
  hardware (InnoDB is tuned for it).
- Replication setup is straightforward.
- Operationally familiar — every cloud has it as a managed
  service.

**Trade-offs:**
- Looser SQL semantics historically — silent truncation, implicit
  type coercions, the infamous `utf8` (3-byte) vs `utf8mb4`
  (real UTF-8) trap. Set `sql_mode='STRICT_ALL_TABLES'` and use
  `utf8mb4` always.
- Less rich type system than Postgres (no arrays, weaker JSON
  ops, no extensions ecosystem).
- The ownership saga: MySQL is Oracle, MariaDB is the community
  fork. Compatible but diverging slowly.

Pick MySQL when the team or platform already standardizes on it,
or for very high-throughput simple-read workloads. Otherwise
Postgres. See **MySQL & ORMs** in the Data topic.

## MariaDB

Community fork of MySQL, created in 2009 when Oracle bought Sun.
Drop-in compatible for most things, diverging on advanced features
(MariaDB has its own storage engines, sequences, JSON functions).

**Why pick it:**
- Open-source governance you might prefer over Oracle's stewardship.
- Default in many Linux distros (Red Hat, Debian).
- Adds features the MySQL community version doesn't have
  (sequences, system-versioned tables).

**Trade-offs:**
- The MySQL/MariaDB compatibility window is shrinking each release.
  Code that works on both today may not in five years.

Most teams just use MySQL or Postgres; MariaDB is the choice when
you've inherited it or have a strong open-source preference.

## SQLite

Not a server — a **library**. Your app embeds SQLite and reads/
writes a single file on disk. No daemon, no network, no auth, no
users.

This sounds toy-like. It is not.

**Why pick it:**
- Embedded use: every iOS and Android app, Firefox, many desktop
  apps. Probably the most-deployed database in the world.
- Tests — spin up a fresh in-memory DB per test, no setup.
- Small services with one writer. The whole DB is one file you
  can copy.
- Local-first apps: store user data on their device, sync later.
- Server-side via **Litestream / LiteFS / Turso** — replicate the
  SQLite file to S3 or distribute it globally. Surprisingly
  capable for read-heavy small/medium services.

**Trade-offs:**
- One writer at a time (writes are serialized). Fine for thousands
  of TPS, not millions.
- No row-level locking — coarse-grained concurrency.
- No network protocol — every client process opens the file
  directly. Doesn't work for multi-machine setups without
  Litestream-style replication.

SQLite is having a renaissance. Don't dismiss it.

## SQL Server (Microsoft)

The dominant choice in Microsoft-shop enterprises. Strong tooling
(SSMS, Azure Data Studio), excellent integration with .NET, paid
licensing with a generous free Express edition.

**Why pick it:**
- You're in a .NET / Azure ecosystem.
- T-SQL has features Postgres / MySQL had later or still lack:
  table-valued parameters (T-SQL only), `MERGE` (added to PG 15),
  and historically had window functions and CTEs first. Postgres
  has had both since 8.4 (2009), so they are now standard SQL,
  not a T-SQL advantage.
- Excellent at OLAP and BI workloads — strong query optimizer,
  columnstore indexes, integrated reporting.

**Trade-offs:**
- License cost at scale.
- Vendor lock-in to Microsoft ecosystem.
- Less common in startups / Node ecosystem.

Most new Node services won't pick SQL Server. If you're integrating
with one, the protocol is documented and `mssql` driver works.

## Oracle Database

The grand old enterprise database. Dominant in finance, telecom,
large enterprises. Famous for performance, infamous for licensing
cost and complexity.

**Why pick it:**
- You're at a company that already runs Oracle.
- Some workloads — extreme OLTP, RAC clustering for HA — are
  genuinely best-in-class.
- PL/SQL stored-procedure ecosystem is mature.

**Trade-offs:**
- Cost. Licensing is famously expensive and audit-heavy.
- Vendor lock-in. PL/SQL doesn't port.
- Operational complexity. You need an Oracle DBA, plural.

You don't pick Oracle for a new project. You inherit it.

## DB2 (IBM)

Same category as Oracle / SQL Server — enterprise, paid, strong
in finance and mainframe environments. Outside those niches, rare
in new projects.

## CockroachDB

**Distributed SQL.** Postgres wire-compatible, but the storage
layer is a distributed Raft-based replicated key-value store.
Horizontally scalable, multi-region capable, survives node failure
without manual failover.

**Why pick it:**
- You need multi-region active-active writes with serializable
  isolation.
- You want Postgres ergonomics without managing sharding manually.
- You need true horizontal scale beyond what a vertical Postgres
  box gives you.

**Trade-offs:**
- Per-query latency is higher than single-node Postgres (Raft
  consensus on every write).
- More operational surface — running a distributed system.
- License change in 2024 (Source Available, not OSI open-source)
  upset some users.

For most apps: overkill. For genuinely global apps where vanilla
Postgres can't keep up: real.

## YugabyteDB

Like Cockroach: distributed SQL, Postgres-compatible. Different
internals (DocDB based on RocksDB), Apache 2 licensed. The
practical choice between Yugabyte and Cockroach often comes down
to license preference and the specific feature set you need.

## Google Spanner

The original distributed-SQL system, internal to Google for years,
now available via Google Cloud. Backed by **TrueTime** — physical
clocks plus uncertainty intervals — to achieve external consistency
across continents.

**Why pick it:**
- You're on GCP and need globally-distributed strong consistency.
- You have the scale to justify the operational and cost model.

**Trade-offs:**
- Proprietary, GCP-only.
- Expensive.
- SQL dialect is its own (Postgres-compatible interface exists
  but isn't 1:1).

Not a startup choice. A "we have global users and SLAs" choice.

## Amazon Aurora / Aurora DSQL

AWS's managed Postgres/MySQL with a different storage architecture
(distributed, log-structured, 6-way replicated). Compatible with
Postgres and MySQL at the wire-protocol level — you can usually
just point your app at it.

**Aurora DSQL** (released 2024–2025) is AWS's distributed-SQL
answer to Cockroach/Spanner: serverless, multi-region active-
active, Postgres-compatible.

**Why pick it:**
- You're on AWS and want managed Postgres/MySQL with better
  durability and read scaling.
- You want distributed SQL without picking a new database product.

**Trade-offs:**
- AWS lock-in.
- Subtle behavior differences from upstream Postgres (especially
  around extensions).

## TimescaleDB

A Postgres extension that specializes in time-series workloads —
high-volume append-only writes indexed by time, with operators
for downsampling, continuous aggregates, and data retention.

It's still Postgres. You keep SQL, you keep ACID, you keep your
existing tooling — TimescaleDB just makes time-series performant.

**Why pick it:**
- IoT data, metrics, financial ticks, event logs — anything
  written by timestamp and queried by time window.
- You'd otherwise reach for InfluxDB or ClickHouse but want to
  keep one database stack.

Sometimes listed under "NoSQL" because it's specialized. It's
SQL.

## DuckDB

Embedded analytics database — "SQLite for analytics." Columnar
storage, vectorized execution, excellent for OLAP queries on
parquet/CSV files.

**Why pick it:**
- Local analytics on data files (parquet, CSV, JSON) without
  loading into a warehouse.
- Notebook / scripting workflows where you want SQL over
  in-memory data.
- Embedded analytics inside an app.

It's not a transactional database. It's the analytics counterpart
to SQLite.

## The summary table

| Database | Type | When to pick |
| --- | --- | --- |
| **Postgres** | OLTP | Default for new projects |
| **MySQL** | OLTP | Existing infra, simple high-throughput reads |
| **SQLite** | Embedded | Small/single-writer, tests, local-first apps |
| **SQL Server** | OLTP/OLAP | .NET / Azure shops |
| **Oracle** | OLTP/OLAP | Inherited enterprise environments |
| **CockroachDB** | Distributed | Multi-region active-active SQL |
| **YugabyteDB** | Distributed | Multi-region, open-source preference |
| **Spanner** | Distributed | Globally-consistent on GCP |
| **Aurora / DSQL** | Managed | AWS-native Postgres/MySQL or distributed |
| **TimescaleDB** | Specialized | Time-series on Postgres |
| **DuckDB** | OLAP/embedded | Local analytics |

## How to actually choose

The decision tree is shorter than the table suggests:

1. **Is the workload time-series?** → TimescaleDB (or InfluxDB /
   ClickHouse in the NoSQL world).
2. **Do you need multi-region active-active writes with ACID?**
   → CockroachDB / Spanner / Aurora DSQL.
3. **Are you embedded or local-first?** → SQLite.
4. **Are you in a Microsoft shop?** → SQL Server.
5. **Otherwise** → Postgres.

90% of new services land on step 5. That's correct.
