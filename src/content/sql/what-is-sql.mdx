# What is SQL

SQL — **S**tructured **Q**uery **L**anguage — is the language you use
to talk to relational databases. It's been the standard way to
store, query, and update structured data since the 1970s, and
nothing has displaced it in fifty years. If you're going to write
backend code, you will write SQL.

The phrase "SQL database" really means two things bundled together:

1. **The relational data model** — data lives in tables with fixed
   schemas; rows in one table reference rows in another via keys.
2. **The SQL query language** — a declarative way to ask for data
   without specifying how to find it.

Postgres, MySQL, SQL Server, Oracle, and SQLite are all SQL
databases. They differ in features and operations, but the model
and language are 90% the same.

## What "declarative" means

In most programming languages you write **imperative** code: "first
do this, then do that." SQL is **declarative**: you describe what
you want, the database figures out how to get it.

```sql
SELECT name, email
FROM users
WHERE active = true
ORDER BY name
LIMIT 50;
```

You didn't tell the database to walk a B-tree, use an index, sort
in memory, or stop after 50 rows. You said "give me the names and
emails of up to 50 active users, sorted by name." The **query
planner** figures out the rest.

This is the source of SQL's longevity. The language hasn't really
changed since the 1980s. The execution underneath has gotten
dramatically smarter, and your old queries get faster for free.

## The four basic operations

CRUD — Create, Read, Update, Delete — covers 95% of what you'll
write:

```sql
INSERT INTO users (name, email) VALUES ('Jon', 'jon@example.com');
SELECT * FROM users WHERE active = true;
UPDATE users SET name = 'Jonathan' WHERE id = 42;
DELETE FROM users WHERE id = 42;
```

Read those out loud and they mean exactly what they say. SQL was
designed to be readable; lean into that when you write queries.

## Why SQL is still the default

If you're building a new backend service in 2026, "use SQL" is
almost always the right starting point:

- **Maturity.** Postgres and MySQL have been hardened for decades.
  Every weird edge case you might hit, someone has hit and
  documented.
- **Tooling.** ORMs, query builders, migration frameworks,
  observability — the SQL ecosystem dwarfs every other database
  ecosystem combined.
- **Hiring.** Anyone you'd hire knows SQL. They might know MongoDB
  too, but they all know SQL.
- **Flexibility.** A flexible query language means your access
  patterns can change without remodeling the data. Critical for
  reporting, analytics, admin tools, and "the product team needs a
  new dashboard by Friday."
- **ACID guarantees.** All-or-nothing transactions are not
  optional for anything involving money, inventory, or any "two
  things must both happen" rule. See **ACID**.

## The myth: "SQL doesn't scale"

You'll hear this often. It's wrong in a specific, important way.

A single Postgres on a modern machine (64 cores, 512GB RAM, NVMe)
serves tens of thousands of queries per second for most workloads.
The vast majority of products never outgrow that. Read replicas
extend it further. Most "we need to scale beyond SQL" stories are
really "we never tuned our SQL."

The real scale ceiling for vanilla SQL is **write throughput
across geographic regions** — billion-row global tables with
active-active multi-region writes. Distributed SQL systems
(CockroachDB, Spanner) handle that case while keeping SQL.

So: SQL scales further than you think. Worry about scaling when
you have a measurement, not a vibe.

## When you'll encounter SQL

In a typical Node/Express backend, you write SQL via:

- **A query builder** like Kysely or Knex — you write SQL-shaped
  JavaScript that compiles to parameterized SQL.
- **An ORM** like Prisma or Drizzle — you write higher-level
  model code; SQL is generated for you.
- **The driver directly** (`pg`, `mysql2`) — full control, more
  boilerplate.

You'll still hit cases where you write raw SQL, especially for
analytics or migrations. The language is worth learning even if
you mostly use an ORM. See **SQL Basics** in the Data topic.
