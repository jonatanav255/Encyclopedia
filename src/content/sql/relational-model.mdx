# The Relational Model

Edgar Codd published "A Relational Model of Data for Large Shared
Data Banks" in 1970. The core idea: store data in **tables** where
each row has the same columns, and link rows across tables by
**keys**. That's the whole model — but the consequences run deep.

## Tables, rows, columns

A table is a set of rows. Each row has the same fixed shape — the
same columns, in the same order, with the same types.

```sql
CREATE TABLE users (
  id         UUID PRIMARY KEY,
  name       TEXT NOT NULL,
  email      TEXT NOT NULL UNIQUE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

Three things to notice:

1. Every column has a **type** (UUID, TEXT, TIMESTAMPTZ). The
   database enforces it — you can't put `"hello"` in an integer
   column.
2. **Constraints** declare rules: `NOT NULL`, `UNIQUE`, `PRIMARY
   KEY`. Violations are rejected at write time.
3. The schema is fixed. Adding a column is a deliberate change
   (a **migration**), not a side effect of inserting different
   data.

## Primary keys

Every table should have a **primary key** — a column (or combination)
that uniquely identifies a row. Two common choices:

```sql
-- Auto-incrementing integer
id SERIAL PRIMARY KEY,        -- Postgres
id INT AUTO_INCREMENT PRIMARY KEY,   -- MySQL

-- UUID (universally unique identifier)
id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
```

UUIDs are usually the right default for new projects — they don't
leak record counts, they're safe to generate client-side, and they
don't collide if you ever merge databases. Specifically **UUID v7**
(time-ordered) gives you UUID safety + good insert performance.

## Foreign keys — the "relational" part

A foreign key is a column in one table that references the primary
key of another table.

```sql
CREATE TABLE posts (
  id        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  author_id UUID NOT NULL REFERENCES users(id),
  body      TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

`posts.author_id` must point at a real row in `users.id`. The
database **enforces** it:

- You can't insert a post with `author_id = '999...'` if no such
  user exists.
- You can't delete a user who still has posts (without `ON DELETE
  CASCADE` or similar).

This is huge. Whole categories of bugs ("we have orphan rows
referencing deleted users") are impossible.

## Joins — combining tables

You don't store the user's name on every post. You store
`author_id`, and **join** the tables when you query:

```sql
SELECT u.name, p.body
FROM posts p
JOIN users u ON u.id = p.author_id
WHERE p.created_at > NOW() - INTERVAL '7 days';
```

The join condition `u.id = p.author_id` tells the database how
the tables connect. Indexes on the join columns make this fast.

The four join types you'll meet:

| Type | What you get |
| --- | --- |
| `INNER JOIN` | Only rows that match in both tables |
| `LEFT JOIN` | All rows from the left; nulls where no match |
| `RIGHT JOIN` | All rows from the right; rarely used |
| `FULL JOIN` | All rows from both, nulls where no match |

See **Joins & Queries** for more.

## Normalization

The principle: each fact lives in exactly one place. If you store
the user's name on every post, and they change their name, you
have to update every post — and miss some.

Normal form rules get formal (1NF, 2NF, 3NF, BCNF) but the
practical version is:

- **Each table represents one kind of thing.** Users in `users`,
  posts in `posts`, comments in `comments`.
- **Don't duplicate data.** Reference it via foreign key.
- **Don't store derived values.** If `full_name` is `first_name ||
  ' ' || last_name`, don't store it — compute it.

Normalization can be taken too far. For read-heavy systems you
sometimes **denormalize** — duplicate a piece of data to skip a
join. That's a deliberate trade-off, not an accident.

## Schema = enforced contract

The relational model's superpower is that the schema is **part of
the database**, not part of your application code. Five years from
now, when you've forgotten what fields a `User` has, the database
still knows. When someone writes a script in a different language
to ingest data, the database still rejects bad rows.

This is what people mean by "schema-on-write." The check happens
when you write. Compare with "schema-on-read" systems (some
document stores, log files) where every reader has to handle the
fact that records may not have the fields they expect.

## Sets, not loops

Codd's other big idea: SQL operates on **sets of rows**, not one
row at a time. You don't write a loop — you write a predicate, and
the database applies it to every row.

```sql
UPDATE posts SET archived = true
WHERE created_at < NOW() - INTERVAL '1 year';
```

That updates a million rows or zero rows with the same syntax. No
explicit iteration. The database handles parallelism, ordering,
locking. You stated the intent; it figured out the execution.

This is why "I'll write a loop in app code" usually loses to a
single SQL statement — you're fighting the database's strongest
muscle.
