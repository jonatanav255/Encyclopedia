# Event Loop (Node)

JavaScript runs on a single thread. Node's event loop is the scheduler
that decides what runs next. The browser model (one macrotask, drain
microtasks, repaint) is a simplification — Node's loop has **multiple
phases** that fire in a specific order, and the difference shows up the
moment you write `setImmediate` next to `setTimeout` and they don't
behave the way you'd expect.

## The phases

Each turn of the loop walks through phases in order:

1. **Timers** — expired `setTimeout` and `setInterval` callbacks.
2. **Pending callbacks** — deferred I/O callbacks (rare in app code).
3. **Idle, prepare** — internal.
4. **Poll** — fetch new I/O events; execute their callbacks. The loop
   can **block** here waiting for events if nothing else is queued.
5. **Check** — `setImmediate` callbacks.
6. **Close callbacks** — `'close'` events on streams/sockets.

Between each phase, two queues drain in order:

- **`process.nextTick` queue**
- **Microtask queue** (promise callbacks, `queueMicrotask`)

`process.nextTick` runs first, then microtasks. Both drain completely
before the loop moves to the next phase.

## `setTimeout(fn, 0)` vs `setImmediate(fn)`

```js
setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));
// order: non-deterministic when at top level
```

When called from the main module, the order depends on the loop's
current state — not the order you wrote them. But inside an I/O
callback:

```js
fs.readFile('x', () => {
  setTimeout(() => console.log('timeout'), 0);
  setImmediate(() => console.log('immediate'));
});
// always: immediate, then timeout
```

Inside the **poll** phase, `setImmediate` runs in the next **check**
phase (immediately after). `setTimeout(0)` waits for the next loop
iteration to reach the **timers** phase.

Mental model: from inside I/O, `setImmediate` means "next phase",
`setTimeout(0)` means "next iteration."

## `process.nextTick`

Fires before microtasks. Higher priority than promise callbacks:

```js
Promise.resolve().then(() => console.log('promise'));
process.nextTick(() => console.log('nextTick'));
// nextTick, then promise
```

**Don't abuse it.** Recursively scheduling `nextTick` starves the
loop — the phase never advances, I/O callbacks never fire, the
process freezes.

```js
function bad() { process.nextTick(bad); }   // never advances
function ok()  { setImmediate(ok);     }   // yields between iterations
```

If you need to defer-and-loop, use `setImmediate`.

## Microtasks

```js
queueMicrotask(() => console.log('a'));
Promise.resolve().then(() => console.log('b'));
// a, b
```

Both queue onto the microtask queue. Equivalent in scheduling priority.

After every macrotask (timer, I/O, `setImmediate`), the entire
microtask queue drains before the next macrotask. So promise chains
feel "synchronous" within one tick.

## Event-loop lag

The gap between when a callback should run and when it actually
runs. Healthy Node services keep p99 lag under ~10ms. Higher means
something synchronous is monopolizing the thread.

Measure it:

```js
import { monitorEventLoopDelay } from 'node:perf_hooks';

const h = monitorEventLoopDelay({ resolution: 20 });
h.enable();
setInterval(() => {
  console.log({ p50: h.percentile(50) / 1e6, p99: h.percentile(99) / 1e6 });
  h.reset();
}, 5000);
```

p99 lag is the metric to alert on. When it spikes, profile to find
the sync hot path (sync crypto, `JSON.parse` of huge payloads, regex
backtracking, sync FS APIs).

## Why "Node is single-threaded" is technically wrong

The **JavaScript** runs on one thread. Underneath, Node uses
**libuv**'s thread pool (default 4 threads) for file I/O, DNS
resolution, and some crypto. Worker threads (`worker_threads`) and
child processes are separate Node instances entirely.

You can't accidentally write concurrent JS — but you can absolutely
write code that contends with the libuv pool (lots of `fs` calls) or
spawn extra threads explicitly.

## The takeaway

For Express specifically:

- A long sync block in any handler delays **every** queued request.
- `await` is a microtask — it doesn't yield to other macrotasks.
  Use `setImmediate` or `setTimeout(0)` if you need to actually let
  I/O fire between steps.
- Don't block. Don't `nextTick`-recurse. Move CPU work to worker
  threads. Stream large payloads. Cap sync APIs.

Knowing the phases is the difference between "I have no idea why this
hangs sometimes" and "the issue is that this regex backtracks for
2 seconds, blocking the poll phase."
