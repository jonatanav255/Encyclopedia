# Dashboards that work

The default trajectory of a dashboard: starts useful, accumulates
panels, becomes a graveyard. Within a year, no one knows what
half the charts mean or whether they're up-to-date. People scan
the top three and ignore the rest.

This page is about how to build dashboards that stay useful past
month one.

## What dashboards are for

Three distinct uses, often confused:

- **At-a-glance health** — "is the service okay?" Read in 5
  seconds. One screen. Fixed layout. Read during normal ops
  and during incidents.
- **Investigative** — "what's wrong with the service right now?"
  Read by someone debugging. Detailed panels, drill-down by
  endpoint / user / region.
- **Exploratory / analytical** — "I want to understand a
  pattern." Ad-hoc queries, not a fixed dashboard. Use Grafana
  Explore or Honeycomb's BubbleUp, not a pinned dashboard.

The mistake is conflating them. A dashboard that tries to be all
three becomes none of them.

## The "at-a-glance" dashboard

The most important dashboard per service. Optimize ruthlessly for
*scannability*.

A working layout:

```
┌──────────────────────────────────────────────────────────────┐
│                   service-name / prod                        │
│  ✓ healthy                last incident: 12 days ago          │
├──────────────────┬──────────────────┬───────────────────────┤
│  Latency p99     │  Error rate %    │  Throughput / s       │
│  [time series]   │  [time series]   │  [time series]        │
│  current: 240ms  │  current: 0.05%  │  current: 1.2k/s      │
├──────────────────┴──────────────────┴───────────────────────┤
│              Saturation (CPU / Memory / Pool)                │
│              [3 small time series side by side]              │
└──────────────────────────────────────────────────────────────┘
```

Rules:

- **Above the fold**: the four golden signals, no scroll needed.
- **Big text for current values**: not just charts. The current
  number is what an operator scanning for problems looks at.
- **Sane time range**: 1-hour default; the small chart shows the
  trend, the number shows now.
- **One service per dashboard**: don't mix. Make separate
  dashboards for different services.
- **No more than 8-12 panels total**. More than that and no one
  reads them.

## The investigative dashboard

A second dashboard for the same service, deeper. Used during
incidents.

- **Per-endpoint breakdowns**: latency p95 per endpoint, error
  rate per endpoint.
- **Top errors by type**: which exception classes are firing.
- **Downstream dependency health**: latency / errors to each
  downstream call.
- **Recent deploys overlaid**: a vertical line for each deploy
  on every time series. The single biggest investigation aid.
- **Saturation per resource**: per-pod CPU, per-instance memory,
  pool utilization.
- **Links to logs**: click a spike, jump to logs at that time.
- **Links to traces**: click a slow request bucket, jump to
  representative traces.

This dashboard can have 30+ panels because it's read by someone
actively investigating, not someone scanning.

## Avoid these common mistakes

- **Mixing averaged and per-instance metrics on the same chart**:
  one panel with both confuses. Pick one perspective per panel.
- **Y-axes not starting at zero** (for traffic / throughput):
  visually exaggerates changes. Sometimes intended (latency
  variations), but be explicit.
- **Default Grafana panel names**: "Panel Title". Rename
  meaningfully. "Request rate by endpoint" beats "Panel 3".
- **No units on axes**: "200" — milliseconds? seconds? requests?
  unitless score? Always label units.
- **Stale panels that no longer work**: a panel that's been
  showing "no data" for 6 months. Delete it.
- **No description / link to runbook**: an operator who's never
  seen this dashboard can't act. Use Grafana's "panel description"
  field; link to a runbook.

## Per-environment vs combined

Two patterns:

- **One dashboard per environment** (prod / staging / dev).
  Switch via dropdown. Clean per-env view.
- **One dashboard with env as a panel variable**. Filter via
  dropdown.

The combined pattern works if you only ever look at one env at a
time. The separate pattern is safer — no risk of viewing prod
when you meant staging.

Most teams: separate dashboards. Prod is the one that matters;
staging dashboards exist mainly for canary/deploy verification.

## Variables and templating

Modern Grafana / Datadog dashboards support variables: dropdowns
that filter the dashboard. Common ones:

- `env` (prod/staging)
- `region`
- `service` (across multiple)
- `instance` (zoom into one pod)
- `time_range` (1h, 6h, 24h presets)

Useful: one dashboard parameterized to work across all instances
of a service.
Annoying: too many variables, and operators have to configure the
dashboard before reading it. Default to sensible values.

## Linking dashboards together

A typical flow during an incident:

1. Alert fires → service health dashboard.
2. Notice latency spike on one endpoint → click endpoint → drill
   into endpoint dashboard.
3. See error correlation → click to logs filtered by error
   pattern.
4. Find a sample trace ID → click into the trace.

Each click is a navigation. Set them up in advance. Grafana lets
you add **data links** to panels — clicking a chart navigates to
another dashboard with the relevant variables pre-filled. Massive
quality-of-life win during real incidents.

## SLO panels

If you have SLOs, a dashboard panel showing them is high-value:

```
┌──────────────────────────────────────────────────────────────┐
│  SLO: 99.0% of requests successful (30-day window)            │
│                                                               │
│  Current: 99.84%       Error budget: 64% remaining            │
│  [burn rate sparkline: 7-day]                                 │
└──────────────────────────────────────────────────────────────┘
```

Shows how the service is tracking against its commitment. Burn
rate (rate of error-budget consumption) signals trouble before
SLO is breached. See `slo-error-budgets`.

## When to delete a dashboard

A dashboard is owed regular review. Signals it should go:

- No one has opened it in 3 months. Check Grafana usage stats.
- Panels show "no data" or unintuitive trends because metrics
  renamed.
- The owner left the company; no one inherited it.
- The service it documents was deprecated.

Dashboards aren't free — they take up screen real estate, get
quoted in runbooks, and clutter search. Delete aggressively.

## Build for the on-call's first day

The honest test of a dashboard: a new on-call engineer who has
never seen the service before is paged at 3am. They open this
dashboard. Can they figure out what's wrong in 60 seconds?

If no — fix the dashboard. The current readability matters more
than how clever individual panels are. Big numbers, clear titles,
runbook links, deploy markers, a single screen of relevant
charts.

That's what dashboards-that-work look like.
