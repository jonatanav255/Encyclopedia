# The four golden signals

From Google's SRE book, this is the minimum-viable monitoring set
for a service. Track these four, alert on them, and you have a
basic but solid foundation:

- **Latency** — time taken to serve a request.
- **Traffic** — demand on the system.
- **Errors** — rate of failed requests.
- **Saturation** — how full / loaded the system is.

If you have nothing else, build these first. They cover ~80% of
"is the service healthy" questions.

## What each one is and how to measure it

<Steps>
<Step n="1" title="Latency">
Time from request arrival to response. **Critical: split
successful vs failed requests.** A failed request returning a 500
in 5ms looks healthy in raw latency stats but masks reality.

Track:
- p50, p95, p99 (and p99.9 if traffic is high enough).
- Successful and error responses separately.
- Per endpoint (or per route class).

Don't report mean alone — see the performance topic's
`tail-latency-and-percentiles`.
</Step>

<Step n="2" title="Traffic">
A measure of demand: requests per second, transactions per
second, sessions concurrent. Different services have different
natural units:

- HTTP API: requests per second.
- Streaming: messages per second, bytes per second.
- Database: queries per second.
- Job queue: jobs per second.

A traffic drop is often a problem signal (something upstream
stopped sending requests). A traffic spike often precedes
saturation.
</Step>

<Step n="3" title="Errors">
The rate of failed requests. Critical to define what "failed"
means for your service:

- HTTP 5xx is unambiguous: server error.
- HTTP 4xx is *usually* a client problem but worth watching
  (sudden spike in 401s might mean an auth outage).
- Semantic errors: a 200 response that contained an error code
  in the body. Often missed.

Track:
- Error rate (errors / total).
- Errors per endpoint.
- Breakdown by error type / status code.

Alert on rate, not count. "5% error rate" is meaningful;
"50 errors" depends on traffic.
</Step>

<Step n="4" title="Saturation">
How "full" the service is. The hardest signal to define because
it varies by what limits you.

- CPU-bound service: CPU utilization.
- Memory-bound: memory utilization.
- I/O-bound: queue depth on the bottleneck I/O.
- Connection-bound: pool utilization.
- Thread-pool-bound: worker utilization.

Saturation is a **leading indicator** of errors and latency
problems. A pool at 95% utilization will be at 100% any minute,
followed by queueing, followed by timeouts, followed by errors.
</Step>
</Steps>

## A working dashboard

The minimum dashboard for a service shows the four signals at the
top:

```
┌──────────────────────────────────────────────────────────────┐
│                  api-service / prod                          │
├──────────────────────────┬───────────────────────────────────┤
│  Latency (p50/p95/p99)   │  Traffic (req/s)                  │
│  [time series chart]     │  [time series chart]              │
├──────────────────────────┼───────────────────────────────────┤
│  Error rate (%)          │  Saturation (CPU / pool)          │
│  [time series chart]     │  [time series charts]             │
└──────────────────────────┴───────────────────────────────────┘
```

Below this go service-specific breakdowns (per-endpoint latency,
top errors by endpoint, downstream dependency health). The four
golden signals are above the fold; everything else is detail.

## SLO-relevant metrics

The four signals also form the basis of SLOs:

- "99% of requests successful" (errors).
- "p99 latency under 250ms" (latency).
- "system handles peak load X" (traffic + saturation).

If you have well-defined SLOs, the four signals are your SLIs.

See `devops/slo-sli` for SLI/SLO mechanics in detail.

## Alerting from the four signals

The good alerts:

- **Error rate spike**: more than X% errors over Y minutes.
- **Latency spike**: p95 > threshold for Y minutes.
- **Traffic drop**: traffic < Y% of expected baseline (catches
  upstream outages).
- **Saturation high**: utilization > 85% for Y minutes (leading
  indicator).

The bad alerts:

- **Raw count thresholds**: "10 errors" is meaningless without
  traffic context.
- **Single-data-point spikes**: alert flaps on every blip. Use
  rolling windows (5+ minutes).
- **Anything below 1%**: noisier than your service. Set
  realistic thresholds.
- **Alerting on every signal**: pick the one or two that map to
  user pain. Saturation alerting is for engineers; user-pain
  alerting is for the team.

## Symptoms vs causes

A subtle distinction the SRE book emphasizes:

- **Symptom-based alerts**: "users are getting errors." Triggers
  on user impact.
- **Cause-based alerts**: "the database is down." Triggers on
  root cause.

Alert on symptoms. The user pain is what matters; the cause may
not be one of your monitored systems. A symptom-based alert
fires even if you've never thought to monitor the actual cause.

Cause-based alerts can be useful as supplementary: "DB went down,
not yet user-visible because of cache, but it's a clock ticking."
But the primary alerts should be symptom-driven.

## Beyond the four

The four golden signals are the floor. Mature observability adds:

- **Throughput** (separate from request count — e.g., bytes/sec
  for streaming services).
- **Queue depth** (for async services).
- **Dependency health** (latency / errors to downstream
  services).
- **Business metrics** (signups / hr, orders / hr — catches
  outages the technical metrics miss).
- **Distributed traces** (for distributed-system debugging).

But if any of the four golden signals are missing or undefined,
fix that before adding new metrics. They are the highest-leverage
investment in observability.
