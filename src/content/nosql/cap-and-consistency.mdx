# CAP & Consistency (NoSQL focus)

The reason NoSQL exists is that distributed systems force
trade-offs that single-machine SQL ducks. The theory — CAP,
PACELC, the consistency-model hierarchy from linearizable down to
eventual — is covered in **system-design/cap-theorem** and
**system-design/consistency-models-overview**. This page is the
NoSQL-side view: per-system tunability knobs, BASE vs ACID
framing, and the consistency failure modes that bite NoSQL apps
in particular.

## What you actually pick

Real systems offer **tunable consistency** so you can pick per
operation:

### DynamoDB
- **Eventually consistent reads** (default): low latency, may be
  stale.
- **Strongly consistent reads**: higher latency, guaranteed
  freshness.

### Cassandra
Per-query consistency level:
- `ONE`, `LOCAL_QUORUM`, `QUORUM`, `ALL`.
- `LOCAL_QUORUM` writes + `LOCAL_QUORUM` reads = strong in a single
  DC.

### MongoDB
**Read concern** (`local`, `available`, `majority`, `linearizable`)
and **write concern** (`{w: 1}`, `{w: "majority"}`,
`{w: "majority", j: true}`). Combinations give different
guarantees.

### CockroachDB / Spanner
Serializable isolation by default. Follower reads opt-in for stale
reads at lower latency.

The right level depends on the data:

- **User profile updates** → strong consistency (avoid showing the
  old profile to the same user).
- **View counts / like counts** → eventual is fine.
- **Account balances** → strong, transactional.
- **Recommendation cache** → eventual / stale OK.

Per-query choice lets you mix.

## ACID vs BASE

Two acronym camps:

### ACID (SQL world)
- **A**tomicity
- **C**onsistency
- **I**solation
- **D**urability

See **ACID** in the SQL topic. The contract for transactional
systems.

### BASE (NoSQL world)
- **B**asically **A**vailable — usually serves a response.
- **S**oft state — the state may change over time without input
  (replicas converging).
- **E**ventually consistent — reads may be stale; converge over
  time.

BASE trades guarantees for availability and scale. The pitch:
"strict ACID is overkill for most operations; let me trade some
correctness for performance."

The honest framing: **ACID and BASE are endpoints on a spectrum,
not opposites.** Most real systems are some mix.

## Common consistency failure modes

The pain points to know:

### Read-your-writes failure
User updates their email. Their settings page (served from a read
replica with replication lag) shows the old email. They re-submit
the update. Now you have two writes to investigate.

**Mitigation:** route reads from a user back to the primary (or
the same replica they wrote to) for a short window after they
write. Many systems support "session consistency" for this.

### Lost updates
Two clients read the same record, modify it, write it back. The
second write overwrites the first.

**Mitigation:** atomic operations (`UPDATE balance = balance - $1`),
optimistic locking with a version column, or pessimistic locking
(`SELECT ... FOR UPDATE`).

### Split-brain (concurrent writes after partition)
A network partition leaves two sides of the cluster believing
they own the data. Each side accepts writes; after the partition
heals, both versions exist and must be reconciled.

**Mitigation:** quorum writes (avoids accepting writes on a
minority partition), vector clocks (Cassandra, Riak), CRDTs, or
last-write-wins with conflict logging. Strict-consensus systems
(Raft, Paxos) prevent split-brain by refusing writes without a
quorum at the cost of availability.

### Stale reads after failover
Primary fails, replica promotes. Recent writes that hadn't
replicated yet are lost. Reads from the new primary don't see
them.

**Mitigation:** synchronous replication (writes ack'd only after
N replicas confirm), or accept the data loss and configure
monitoring.

### Time-of-check / time-of-use
Read a value, decide based on it, write a new value. The value
changed between the check and the write.

**Mitigation:** transactions with `SERIALIZABLE` isolation, or
atomic predicate updates (`UPDATE ... WHERE expected_value = ?`).

## How to pick a consistency level

A pragmatic decision framework:

1. **Identify the data.** Per-table, per-operation. Don't pick one
   level for everything.
2. **Identify the worst-case if you're wrong.**
   - "User sees stale view count" → eventual is fine.
   - "Two users get assigned the same username" → strong needed.
   - "Money goes missing" → transactional, never anything weaker.
3. **Identify the cost.**
   - Stronger consistency = higher latency + lower availability
     under partition.
4. **Pick the weakest level that handles your worst case.**
   Default to stronger; relax with intent.

The mistake is **defaulting to eventual consistency for everything
because it's faster**. Most apps need strong consistency for
99% of operations and eventual is wrong.
