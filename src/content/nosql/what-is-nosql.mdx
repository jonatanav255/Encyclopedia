# What is NoSQL

"NoSQL" is one of the worst names in computing. It originated in
2009 as a Twitter hashtag for a meetup about non-relational
databases, and it stuck. The name suggests these databases are
defined by what they're *not* — they don't use SQL. Which tells
you essentially nothing about what they actually are.

The useful framing: **NoSQL is an umbrella for several unrelated
families of database**, each solving a specific problem the
relational model isn't great at. They have different data models,
different consistency guarantees, and different access patterns.
Calling them all "NoSQL" is like calling all non-cars
"non-cars" — true, but it doesn't help you pick one.

## The five families

| Family | Examples | What it stores |
| --- | --- | --- |
| **Key-value** | Redis, Memcached, DynamoDB | `Map<key, bytes>` |
| **Document** | MongoDB, Firestore, Couchbase | JSON-shaped documents |
| **Wide-column** | Cassandra, ScyllaDB, HBase | Sparse partitioned tables |
| **Graph** | Neo4j, Dgraph, Neptune | Nodes and edges |
| **Search** | Elasticsearch, OpenSearch, Meilisearch | Inverted indexes over text |

Plus honorable mentions: **time-series** (InfluxDB, TimescaleDB),
**vector** (Pinecone, pgvector, Qdrant), and **object stores**
(S3, GCS).

Each family has its own page in this topic.

## Why NoSQL exists

The relational model is general-purpose. That's its strength —
and the source of its trade-offs:

- **Joins cost.** A query joining 5 tables across millions of rows
  is fundamentally more work than a single key lookup.
- **ACID across many nodes is hard.** Distributed transactions
  require consensus, which costs latency.
- **Schema migrations on huge tables hurt.** `ALTER TABLE` on a
  billion rows is an operational event.

NoSQL systems each give up some part of the relational contract in
exchange for something else:

- A key-value store gives up joins, complex queries, and schema —
  but is wildly faster than SQL for "look up by key."
- A document store gives up transactions across documents — but
  lets each record have its own shape.
- A wide-column store gives up ad-hoc queries — but handles
  millions of writes per second.
- A graph database gives up tabular thinking — but makes
  traversal queries efficient.
- A search engine gives up being a system of record — but
  delivers real text search.

The right question isn't "should we use NoSQL?" It's "does our
workload match what one of these families is good at?"

## NoSQL ≠ schemaless

A common confusion: "NoSQL means no schema, which means we move
fast."

Wrong on two counts:

1. **Many NoSQL stores have schemas.** Cassandra requires you to
   declare a `CREATE TABLE` with types. DynamoDB requires you to
   declare partition keys. Even MongoDB recommends a schema layer
   in your app (Mongoose, Zod) once you stop being a prototype.

2. **The schema doesn't go away — it just moves.** With a
   document store, every reader of the data has to handle the
   fact that records may have any combination of fields. The
   schema lives in N places (every consumer) instead of one
   (the database). This is usually a worse outcome than declaring
   the schema up front.

A real Postgres `JSONB` column gives you "flexible schema where you
need it" without losing the relational guarantees everywhere else.
This is the move that defeats 90% of "we need MongoDB" arguments.

## NoSQL ≠ "scales better"

Another common belief: "SQL doesn't scale, NoSQL scales."

The reality:

- **A single Postgres on a big box** handles tens of thousands of
  QPS for many workloads.
- **Read replicas + caching** push that further.
- **Distributed SQL** (Cockroach, Spanner) gives you horizontal
  SQL.

NoSQL systems often **do** scale further for specific access
patterns — Cassandra handles write rates that single-primary
Postgres can't, DynamoDB delivers single-ms reads at any scale —
but they do it by giving up features (joins, transactions,
ad-hoc queries).

"NoSQL scales" is true if your workload is shaped like what a
specific NoSQL system optimizes for. For a typical CRUD app with
moderate traffic, NoSQL doesn't scale better — it just gives up
features you needed.

## Where the "Not Only SQL" reading comes in

A revisionist take on the name: NoSQL stands for "**N**ot **O**nly
**SQL**." The healthier framing of the same idea.

A typical production stack in 2026 uses several databases:

- **Postgres** (SQL) as the source of truth.
- **Redis** (key-value, NoSQL) for caching and sessions.
- **Elasticsearch** (search, NoSQL) for text search.
- **S3** (object store) for files.

That's polyglot persistence. Each store does what it's best at;
the system is more capable than any one of them alone.

The mistake is treating NoSQL as a replacement for SQL rather than
a complement to it. Most apps need SQL **and** some NoSQL — not
NoSQL instead of SQL.

## What you actually trade away

When you pick a NoSQL system for your main store, you typically
give up at least one of:

- **ACID transactions.** Most NoSQL is single-document atomic at
  best. Cross-document atomicity is limited or unavailable.
- **Joins.** Either impossible, or done in app code, or limited
  to specific patterns (DynamoDB's adjacency lists).
- **Ad-hoc queries.** Many NoSQL stores require you to design
  your access patterns into the data model. New queries → new
  data layout.
- **Strong consistency.** Many NoSQL systems are eventually
  consistent by default. Reads may lag writes by seconds.
- **The ecosystem.** ORMs, migration tools, observability,
  tutorials — SQL has decades of accumulated tooling. NoSQL
  ecosystems are smaller and more fragmented.

These aren't always trades worth making. Sometimes they are. The
job is knowing which.

## When NoSQL is right

Specific use cases, not vibes:

- **You need a cache** → key-value (Redis).
- **You need session storage** → key-value (Redis).
- **You need full-text search with relevance** → search
  (Elasticsearch / Meilisearch / Algolia).
- **You're ingesting millions of time-series points per second**
  → time-series (TimescaleDB stays SQL; InfluxDB / ClickHouse
  don't).
- **You have genuine graph-traversal queries** (5+ hops) →
  graph (Neo4j).
- **You've measured write throughput needs that single-primary
  SQL can't keep up with** → wide-column (Cassandra / Scylla).

For most apps: SQL as the main store, plus Redis and maybe a
search store. Pure NoSQL stacks are rarer than the marketing
suggests.
