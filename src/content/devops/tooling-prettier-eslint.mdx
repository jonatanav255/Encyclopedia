# Prettier and ESLint

The two tools that decide what "well-formatted" and "linted" mean
in your codebase. They're different tools with different jobs;
running them together is the default.

## What each does

- **Prettier**: formats code. No opinions about logic — just
  whitespace, line breaks, quote style, trailing commas.
- **ESLint**: lints code. Finds bugs, enforces patterns, can
  auto-fix some issues. Less opinionated about formatting.

In 2026 the split is: **Prettier handles formatting; ESLint handles
logic.** They used to overlap; modern configs separate them.

## Prettier minimal setup

```bash
pnpm add -D prettier
```

`.prettierrc.json`:

```json
{
  "semi": true,
  "singleQuote": true,
  "trailingComma": "all",
  "printWidth": 100,
  "tabWidth": 2
}
```

`.prettierignore`:

```
node_modules
dist
build
.next
*.min.js
```

In `package.json`:

```json
{
  "scripts": {
    "format": "prettier --write .",
    "format:check": "prettier --check ."
  }
}
```

That's it. `pnpm format` formats everything in place.

## Prettier philosophy

Almost no config knobs. The team designed it that way: pick a style
and apply it consistently, don't bikeshed.

The few options worth tuning:

- **`printWidth`** — 80 (Prettier default), 100, or 120. Pick one.
- **`singleQuote`** — true for JS-leaning, false for HTML-leaning.
- **`trailingComma`** — "all" produces cleaner diffs.
- **`semi`** — true unless you have strong feelings.

After the team agrees, stop debating. Run Prettier in CI.

## ESLint flat config

ESLint 9+ uses **flat config** (`eslint.config.js`) — replaces
`.eslintrc`.

```js
// eslint.config.js
import js from '@eslint/js';
import ts from 'typescript-eslint';

export default [
  js.configs.recommended,
  ...ts.configs.recommended,
  {
    rules: {
      '@typescript-eslint/no-unused-vars': ['error', { argsIgnorePattern: '^_' }],
      'no-console': 'warn',
      'prefer-const': 'error',
    },
  },
  {
    ignores: ['dist/**', 'build/**', 'node_modules/**'],
  },
];
```

Flat config is an array of configs that apply in order. Each can
target specific files via `files` patterns. Simpler than the old
`.eslintrc` cascade.

## Rules worth enabling

A baseline for Node + TypeScript projects:

```js
{
  rules: {
    // TS-aware
    '@typescript-eslint/no-unused-vars': ['error', { argsIgnorePattern: '^_' }],
    '@typescript-eslint/no-explicit-any': 'warn',
    '@typescript-eslint/no-unsafe-assignment': 'warn',
    '@typescript-eslint/consistent-type-imports': ['error', { prefer: 'type-imports' }],

    // Bug catchers
    'eqeqeq': ['error', 'always', { null: 'ignore' }],
    'no-var': 'error',
    'prefer-const': 'error',
    'no-implicit-coercion': 'warn',

    // Async safety
    '@typescript-eslint/no-floating-promises': 'error',
    '@typescript-eslint/no-misused-promises': 'error',
    '@typescript-eslint/await-thenable': 'error',

    // Errors
    'no-throw-literal': 'error',
    '@typescript-eslint/only-throw-error': 'error',

    // Imports
    'no-duplicate-imports': 'error',
  },
}
```

The `@typescript-eslint/no-floating-promises` rule alone catches a
class of bug (forgotten `await`) that previously needed unit tests
to find.

## React-specific (for SPAs)

```js
import react from 'eslint-plugin-react';
import reactHooks from 'eslint-plugin-react-hooks';

export default [
  // ...
  react.configs.recommended,
  {
    plugins: { 'react-hooks': reactHooks },
    rules: {
      'react-hooks/rules-of-hooks': 'error',
      'react-hooks/exhaustive-deps': 'warn',
    },
  },
];
```

`exhaustive-deps` catches missing dependencies in `useEffect` and
similar — a common source of stale-closure bugs. **Turn it on.**

## Don't fight Prettier with ESLint

Old setups had ESLint rules for formatting (`indent`, `quotes`,
`semi`). With Prettier, these are redundant. Disable them with
**`eslint-config-prettier`**:

```js
import prettier from 'eslint-config-prettier';

export default [
  // ... other configs
  prettier,   // last — disables conflicting rules
];
```

Now ESLint focuses on logic; Prettier handles style. No fights.

## Running both

In CI:

```yaml
- run: pnpm format:check
- run: pnpm lint
- run: pnpm typecheck
- run: pnpm test
```

Format-check fails the build if anything's unformatted. Lint
catches code-quality issues. Typecheck (`tsc --noEmit`) catches
type errors. All three independently.

## In your editor

VS Code (most editors similar):

```json
// .vscode/settings.json
{
  "editor.formatOnSave": true,
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": "explicit"
  }
}
```

Save → Prettier formats → ESLint auto-fixes what it can. The team
gets consistent code without thinking.

## Pre-commit hooks

```bash
pnpm add -D husky lint-staged
```

`.husky/pre-commit`:

```bash
pnpm exec lint-staged
```

`package.json`:

```json
{
  "lint-staged": {
    "*.{ts,tsx,js,jsx}": ["prettier --write", "eslint --fix"],
    "*.{json,md,yml}": ["prettier --write"]
  }
}
```

On every `git commit`, staged files are auto-formatted and linted.
You can't commit unformatted code.

Some teams skip pre-commit hooks (slow, annoying) and rely on CI
checks. Either works; pick one.

## Editor config

```ini
# .editorconfig
root = true

[*]
indent_style = space
indent_size = 2
end_of_line = lf
charset = utf-8
trim_trailing_whitespace = true
insert_final_newline = true
```

EditorConfig works across IDEs, including those without Prettier
plugins. Pair with Prettier for full coverage.

## Common pitfalls

### Conflicting rules between Prettier and ESLint

Without `eslint-config-prettier`, ESLint and Prettier fight: ESLint
says use double quotes, Prettier reformats to single, ESLint fails
again. Always add `eslint-config-prettier` last in the config array.

### Disabling rules you don't understand

```js
// eslint-disable-next-line @typescript-eslint/no-floating-promises
fireAndForget();
```

`disable-next-line` is a tool, not a habit. Each one is "I'm
suppressing a check; here's why." If your codebase has many
suppressions, the rules are wrong or the code is.

### Running ESLint on built files

```
dist/index.js — 50,000 lines of minified output linted
```

Slow. Useless. Add `dist/**` to `ignores`.

### Forgetting to gitignore the ESLint cache

```
.eslintcache
```

ESLint caches results in `.eslintcache`. Gitignore it.

### Outdated config (eslintrc vs flat)

```
.eslintrc.json    — old format
eslint.config.js  — flat config (ESLint 9+)
```

If you have both, ESLint warns/errors. Migrate to flat config —
it's the modern standard.

### Auto-fixing dangerous things

```js
// In ESLint config
'@typescript-eslint/no-explicit-any': ['error', { fixToUnknown: true }]
```

Some auto-fixes change semantics. Read the fix before letting
`--fix` rewrite your code. For risky rules, set them to warn.

### Pre-commit hooks that take forever

If `lint-staged` runs all of ESLint on every commit, even single-
file changes feel slow. Configure it to only lint staged files:

```json
{
  "lint-staged": {
    "*.ts": ["prettier --write", "eslint --fix"]
  }
}
```

That's what the example above does — only the matching files.

### Inconsistent settings between editor and CI

If editor uses Prettier 2.x and CI uses 3.x, formatting bickers
back and forth on commits. Pin Prettier version in `package.json`
and use `pnpm exec prettier` everywhere (uses the installed
version, not a global one).

## When ESLint isn't enough

For more sophisticated static analysis:

- **TypeScript's `--strict`** flags (see **tsconfig**) catch more
  than ESLint can.
- **`type-coverage`** measures how much of your code is fully
  typed.
- **`knip`** finds unused exports, files, dependencies.
- **`madge`** detects circular dependencies.

These complement ESLint without replacing it.

## The mental model

Prettier + ESLint is **the formatter for style + the static
analyzer for logic**. They run independently; configure once;
forget.

For 2026 Node projects:

- **Prettier** with minimal config.
- **ESLint 9 flat config** with `@typescript-eslint`, `react-hooks`
  if applicable, `no-floating-promises` enabled.
- **`eslint-config-prettier`** to silence overlap.
- **Editor on-save** formats + auto-fixes.
- **CI** runs `format:check`, `lint`, `typecheck`.
- **(Optional) pre-commit hooks** for safety.

The win is consistent code that no one debates. PRs focus on logic,
not formatting. New developers see clean code and follow the style
automatically.
