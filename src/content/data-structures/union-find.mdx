# Union-find (disjoint sets)

Union-find tracks a partition of N items into disjoint sets, with two
operations: `find(x)` (which set does x belong to?) and `union(x, y)`
(merge their sets). With the two standard optimizations — union by
rank and path compression — both operations are **effectively
O(1)**: O(α(n)) where α is the inverse Ackermann function, which is
≤ 4 for any n that fits in the observable universe.

It's one of the most efficient data structures in computer science
and the implementation fits in 15 lines.

## When you reach for it

The signature pattern is "things merge over time; tell me what's
connected." Some real cases:

- **Kruskal's MST** — sort edges by weight, add each edge if its
  endpoints aren't already connected (`find(u) !== find(v)`).
- **Dependency cycle detection** — building a graph incrementally;
  if adding edge `(u, v)` would put `u` and `v` in the same
  component already, you have a cycle.
- **Image segmentation** — neighboring pixels with similar color get
  merged into the same region.
- **Incident grouping** — alerts that share a host, service, or
  trace get bucketed together as one incident.
- **Friend-of-a-friend / social grouping** — connected components
  of a friendship graph.
- **Online connectivity** — "are these two nodes in the same
  cluster?" as the cluster topology changes.

Anything where you only ask "same group?" — never "give me the
shortest path" or "iterate the group" — union-find beats every
other approach.

## The naive version

Each item points to a parent; the root of a tree represents its
set.

```js
class UnionFind {
  constructor(n) {
    this.parent = Array.from({ length: n }, (_, i) => i);
  }
  find(x) {
    while (this.parent[x] !== x) x = this.parent[x];
    return x;
  }
  union(x, y) {
    const rx = this.find(x), ry = this.find(y);
    if (rx !== ry) this.parent[rx] = ry;
  }
}
```

This works but degrades. After a bad sequence of unions, trees can
become linked lists — `find` walks all the way up, O(n).

Two fixes turn that into effectively O(1).

## Optimization 1: union by rank (or size)

When merging two trees, attach the **shorter** one under the
**taller**. This keeps trees shallow.

```js
union(x, y) {
  const rx = this.find(x), ry = this.find(y);
  if (rx === ry) return false;
  if (this.rank[rx] < this.rank[ry]) this.parent[rx] = ry;
  else if (this.rank[rx] > this.rank[ry]) this.parent[ry] = rx;
  else { this.parent[ry] = rx; this.rank[rx]++; }
  return true;
}
```

`rank` is an upper bound on the tree's depth. Union by **size**
(attach smaller-population to larger) works just as well and is
slightly easier to reason about.

With just union-by-rank (no path compression), every operation is
O(log n).

## Optimization 2: path compression

While walking up during `find`, point every visited node directly at
the root.

```js
find(x) {
  while (this.parent[x] !== x) {
    this.parent[x] = this.parent[this.parent[x]];  // path halving
    x = this.parent[x];
  }
  return x;
}
```

This is **path halving** — pointers leapfrog upward, flattening the
tree as a side effect of querying it. Cleaner than the full
two-pass version and almost as effective.

With both optimizations, the amortized cost is **O(α(n))** —
inverse Ackermann. For all practical n, α(n) ≤ 4. So:
effectively O(1).

## The complete implementation

```js
class UnionFind {
  constructor(n) {
    this.parent = Array.from({ length: n }, (_, i) => i);
    this.rank = new Uint8Array(n);
    this.count = n;
  }
  find(x) {
    while (this.parent[x] !== x) {
      this.parent[x] = this.parent[this.parent[x]];
      x = this.parent[x];
    }
    return x;
  }
  union(x, y) {
    const rx = this.find(x), ry = this.find(y);
    if (rx === ry) return false;
    if (this.rank[rx] < this.rank[ry]) this.parent[rx] = ry;
    else if (this.rank[rx] > this.rank[ry]) this.parent[ry] = rx;
    else { this.parent[ry] = rx; this.rank[rx]++; }
    this.count--;
    return true;
  }
  connected(x, y) {
    return this.find(x) === this.find(y);
  }
}
```

Fifteen lines, near-constant amortized per op, never deletes.

## Worked example: Kruskal's MST

```js
function kruskal(n, edges) {
  edges.sort((a, b) => a[2] - b[2]);   // by weight
  const uf = new UnionFind(n);
  const mst = [];
  let total = 0;
  for (const [u, v, w] of edges) {
    if (uf.union(u, v)) {
      mst.push([u, v, w]);
      total += w;
      if (mst.length === n - 1) break;
    }
  }
  return { mst, total };
}
```

The whole algorithm is "sort edges, take each one if it doesn't
form a cycle." Union-find makes the cycle check effectively free.

## Worked example: cycle detection in incremental graph

```js
function hasCycle(n, edges) {
  const uf = new UnionFind(n);
  for (const [u, v] of edges) {
    if (uf.connected(u, v)) return true;   // already connected → cycle
    uf.union(u, v);
  }
  return false;
}
```

Works for **undirected** graphs. For directed graphs, this is not
the right tool — use DFS-based cycle detection (white/gray/black
coloring) instead. Union-find can't distinguish edge direction.

## What it can't do

- **Split a set.** Once united, always united. There's no efficient
  undo.
- **Iterate a set's members.** You only know "same root" — you
  don't have a list of members. If you need that, maintain a
  separate `Map<root, members[]>` alongside (and update it on
  union).
- **Track per-set metadata that depends on history.** You can
  attach a value to each root and combine on union, but anything
  needing "the path from x to its root" doesn't fit.

For undo, look at **link/cut trees** or persistent union-find.
Different beast entirely.

## Weighted / rollback variants

- **Weighted union-find.** Each edge stores a "relative weight" to
  its parent. `find` accumulates as it walks. Useful for "is x
  exactly k units more than y?" problems and for some constraint
  systems.
- **Union-find with rollback.** Skip path compression (which makes
  ops non-undoable) and keep a stack of (parent, rank) changes.
  Each op is O(log n) instead of α(n), but you can undo. Used in
  some offline graph algorithms.

## Performance notes

- The `parent` array fits in cache for moderate n. `find` is a
  sequence of dependent loads but with path compression they
  flatten quickly.
- `Uint8Array` for `rank` is fine — rank rarely exceeds 30 even for
  billions of items, because each rank increment requires doubling
  the tree size.
- For huge n on the GPU or in WebAssembly, packed arrays + atomic
  CAS for concurrent union-find exist; they're a research area.

## Common mistakes

- **Forgetting path compression.** Without it, you have O(log n)
  ops, not O(α(n)). Usually fine, but the gap matters at scale.
- **Using on directed graphs for cycle detection.** Doesn't work.
- **Trying to delete an edge.** Standard union-find doesn't
  support it. If you need it, you're in different-data-structure
  territory.
- **Storing the array of members per set** and updating on every
  union. Defeats the point — now each union is O(set size).
  Recompute components from `find` at the end instead.
- **Calling `parent[x]` directly instead of `find(x)`.** Returns
  the immediate parent, not the root. Always go through `find`.

## The discipline

- "Things merge, tell me what's connected" → union-find.
- Always implement both union-by-rank and path compression. Together
  they're α(n); separately they're O(log n).
- Undirected only. Directed cycle detection wants DFS coloring.
- No delete, no split, no member iteration. Plan around it or
  pick a different structure.
- Fifteen lines of code, used in everything from Kruskal to
  incident grouping. One of the highest leverage data structures
  in the toolbox.
