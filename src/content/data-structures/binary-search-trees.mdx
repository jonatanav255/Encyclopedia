# Binary search trees

A binary search tree (BST) is an ordered map: keys live in a tree,
sorted left-to-right. You get O(log n) lookup, insert, delete, *and*
sorted iteration, range queries, predecessor/successor — things a
hash table can't do.

The catch: that O(log n) only holds if the tree is **balanced**. A
naive BST built from sorted input degrades to a linked list with
O(n) operations. Real-world BSTs are always self-balancing variants
(red-black, AVL, B-tree).

## The invariant

For every node:

- All keys in the left subtree are **less than** the node's key.
- All keys in the right subtree are **greater than** the node's key.

Equality is usually disallowed (a set) or handled by replacing the
value (a map).

```js
//        5
//       / \
//      3   8
//     / \   \
//    1   4   9
```

In-order traversal (left, node, right) yields keys in sorted order:
`1, 3, 4, 5, 8, 9`. This is the property that makes BSTs useful for
ordered data.

## Basic operations

```js
class BSTNode {
  constructor(key, value) {
    this.key = key;
    this.value = value;
    this.left = null;
    this.right = null;
  }
}

function get(root, key) {
  let n = root;
  while (n) {
    if (key < n.key) n = n.left;
    else if (key > n.key) n = n.right;
    else return n.value;
  }
  return undefined;
}

function insert(root, key, value) {
  if (!root) return new BSTNode(key, value);
  if (key < root.key) root.left = insert(root.left, key, value);
  else if (key > root.key) root.right = insert(root.right, key, value);
  else root.value = value;
  return root;
}
```

Both are O(h) where h is the tree height. For a balanced tree,
h ≈ log₂(n). For a degenerate tree, h = n.

## The unbalanced trap

Insert keys in sorted order into a plain BST:

```js
let root = null;
for (const k of [1, 2, 3, 4, 5]) root = insert(root, k, k);

//   1
//    \
//     2
//      \
//       3
//        \
//         4
//          \
//           5
```

That's not a tree, that's a linked list with extra steps. Every
operation is O(n). Real-world data is often partially sorted (event
logs by time, IDs in order), so naive BSTs are *especially* prone to
this in practice.

The fix: balancing.

## Self-balancing BSTs

After every insert/delete, perform local rotations to keep the tree
roughly balanced. Two classics:

- **AVL trees** — strict balance: heights of subtrees differ by at
  most 1. Faster lookups, slower inserts (more rotations).
- **Red-black trees** — looser balance: longest path is at most 2×
  the shortest. Fewer rotations, slightly taller. Better for
  insert-heavy workloads.

Both guarantee O(log n) operations. C++'s `std::map`, Java's
`TreeMap`, and most language standard libraries use **red-black
trees**. They're the workhorse.

You will almost never implement one. The interview question
("rotate left/rotate right") is real but the wins from implementing
your own vs. using a library are zero unless you're writing the
language runtime.

## Deletion is the annoying part

Insert is easy. Lookup is easy. Delete has three cases:

1. **Leaf node** — just remove it.
2. **One child** — replace the node with its child.
3. **Two children** — replace with the **in-order successor**
   (smallest key in the right subtree), then recursively delete
   that successor (which now has at most one child).

```js
function deleteKey(root, key) {
  if (!root) return null;
  if (key < root.key) root.left = deleteKey(root.left, key);
  else if (key > root.key) root.right = deleteKey(root.right, key);
  else {
    if (!root.left) return root.right;
    if (!root.right) return root.left;
    // Two children: copy successor's key/value, delete successor
    let succ = root.right;
    while (succ.left) succ = succ.left;
    root.key = succ.key;
    root.value = succ.value;
    root.right = deleteKey(root.right, succ.key);
  }
  return root;
}
```

In a self-balancing tree, you also have to rebalance after the
delete. That's where the code length explodes.

## Hash table vs BST: when to pick which

| | Hash table | Balanced BST |
|---|---|---|
| Lookup | O(1) average | O(log n) |
| Sorted iteration | No (or yes but unordered → sort first) | Yes, free |
| Range query (`x..y`) | No | Yes, O(log n + k) |
| Predecessor/successor | No | Yes, O(log n) |
| Min / max | O(n) | O(log n) |
| Worst-case lookup | O(n) | O(log n) guaranteed |

Pick a hash table when you only need by-key lookup. Pick a BST
(or its on-disk cousin, the B-tree) when you need ordering.

Examples that want a BST:

- "Show me events between 9:00 and 10:00."
- "Find the user just above this score."
- "Iterate scores in descending order."
- A sorted set where you also need O(log n) inserts.

JS has no built-in ordered map. You either use a library, simulate
with a sorted array (O(n) insert, fine for small n), or use a
`Map` and sort on read.

## Iteration patterns

**In-order**: left, node, right — yields keys in sorted order.

```js
function* inOrder(node) {
  if (!node) return;
  yield* inOrder(node.left);
  yield [node.key, node.value];
  yield* inOrder(node.right);
}
```

**Pre-order**: node, left, right — useful for cloning a tree
structure or serializing it.

**Post-order**: left, right, node — useful for deletion (delete
children before parent) or computing a function bottom-up.

**Level-order (BFS)**: queue-based, traverses by depth.

For a BST specifically, in-order is the one you'll use 90% of the
time.

## Range queries

"Give me all keys in `[lo, hi]`." A balanced BST handles this in
O(log n + k) where k is the result size:

```js
function* range(node, lo, hi) {
  if (!node) return;
  if (lo < node.key) yield* range(node.left, lo, hi);
  if (lo <= node.key && node.key <= hi) yield [node.key, node.value];
  if (node.key < hi) yield* range(node.right, lo, hi);
}
```

This is why databases use B-trees (a generalization of BSTs) for
indexes — range queries are first-class.

## Where balanced BSTs lead

- **B-trees** — wider branching factor, optimized for disk/SSD page
  size. The data structure behind nearly every relational database
  index. See `b-trees.mdx`.
- **Skip lists** — probabilistic alternative with similar
  guarantees, simpler to implement concurrently. Used in Redis
  sorted sets.
- **Treaps** — random priorities + BST + heap order. Surprisingly
  good in practice, very easy to implement.
- **Splay trees** — recently-accessed keys move to the root.
  Amortized O(log n). Used in some caches.

For most application code, you'll never reach for these directly —
your database does. But the BST mental model is the prerequisite
for understanding them.

## The discipline

- Plain BSTs are a trap on real data. Always assume self-balancing
  (red-black or AVL) in production.
- JS has no built-in ordered map. Use a library if you need one.
- In-order traversal of a BST is the sorted iteration. That's the
  killer feature.
- Range queries, predecessor/successor, min/max — these are the
  jobs hash tables can't do.
- For on-disk data (databases), B-trees win because of page-size
  considerations. See `b-trees.mdx`.
