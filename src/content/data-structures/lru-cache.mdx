# LRU cache

A cache with fixed capacity needs an **eviction policy** for when
it's full. LRU — *least recently used* — evicts whatever hasn't
been touched the longest. It's the default eviction policy almost
everywhere because it's simple, has good hit rates on realistic
workloads, and admits a clean O(1) implementation: a hash table
plus a doubly-linked list.

It's also the canonical interview question. The 30-line
implementation is worth memorizing.

## The shape

Two structures, sharing the same nodes:

- A **doubly-linked list** ordered by recency: head = most recent,
  tail = least recent.
- A **hash table** from key → list node, so you can find a node
  without walking the list.

```
hash       linked list (head ↔ ... ↔ tail)
key A -─┐
key B -─┼──>  [A] ↔ [B] ↔ [C] ↔ [D]
key C -─┤      ^                ^
key D -─┘     newest           oldest (evict next)
```

Every operation:

- **get(k)**: hash lookup → node. Move node to head. Return value.
- **put(k, v)**: hash lookup. If exists, update + move to head. If
  not, create node at head; if over capacity, drop the tail node
  and remove its key from the hash.

All O(1). The hash makes lookup constant; the doubly-linked list
makes "move to front" constant (you don't have to find the node's
neighbors — it already knows them).

## Why doubly-linked

Singly-linked lists can't do "remove this node in O(1)" — you'd
need the previous node, which means scanning. Doubly-linked nodes
hold `prev` and `next`, so splicing out is:

```js
node.prev.next = node.next;
node.next.prev = node.prev;
```

Constant time. This is the one place a linked list genuinely beats
an array (see `arrays-vs-linked-lists.mdx`) — you have node
references from the hash table, so you never traverse.

## The implementation

```js
class LRUCache {
  constructor(capacity) {
    this.capacity = capacity;
    this.map = new Map();
    // Sentinel head/tail simplify edge cases (no null checks)
    this.head = { key: null, value: null, prev: null, next: null };
    this.tail = { key: null, value: null, prev: null, next: null };
    this.head.next = this.tail;
    this.tail.prev = this.head;
  }
  _addToFront(node) {
    node.prev = this.head;
    node.next = this.head.next;
    this.head.next.prev = node;
    this.head.next = node;
  }
  _remove(node) {
    node.prev.next = node.next;
    node.next.prev = node.prev;
  }
  get(key) {
    const node = this.map.get(key);
    if (!node) return undefined;
    this._remove(node);
    this._addToFront(node);
    return node.value;
  }
  put(key, value) {
    const existing = this.map.get(key);
    if (existing) {
      existing.value = value;
      this._remove(existing);
      this._addToFront(existing);
      return;
    }
    if (this.map.size === this.capacity) {
      const lru = this.tail.prev;
      this._remove(lru);
      this.map.delete(lru.key);
    }
    const node = { key, value, prev: null, next: null };
    this._addToFront(node);
    this.map.set(key, node);
  }
}
```

The sentinel head/tail nodes are the trick that makes this clean.
Without them, every operation needs special-case null checks for
"is this the first/last node." With them, every node has real
`prev` and `next`, and the splicing code never branches.

Worth implementing once, by hand, without looking — that's the
muscle memory the interview wants.

## A cute one-liner (sort of)

JavaScript `Map` already remembers insertion order. So:

```js
class LRUCacheCheap {
  constructor(capacity) { this.capacity = capacity; this.map = new Map(); }
  get(key) {
    if (!this.map.has(key)) return undefined;
    const v = this.map.get(key);
    this.map.delete(key);
    this.map.set(key, v);    // re-insert to push to "most recent"
    return v;
  }
  put(key, value) {
    if (this.map.has(key)) this.map.delete(key);
    else if (this.map.size === this.capacity) {
      this.map.delete(this.map.keys().next().value);   // oldest
    }
    this.map.set(key, value);
  }
}
```

Real LRU semantics, ~12 lines, uses the spec-guaranteed insertion-
order property of `Map`. The performance is acceptable but slower
than the explicit list version (each touch does a delete + insert
in V8's ordered hash, which is more work than splicing two
pointers). Fine for prototypes; the explicit version is what you
ship for hot paths.

Don't bring this up first in an interview — it's a clever
side-step that some interviewers love and others view as dodging
the algorithm. Lead with the doubly-linked-list version, mention
this as an alternative.

## Eviction policies in the wild

LRU is one option among several. The real ones:

- **LRU** — drop the least recently used. Simple, good baseline,
  works on most workloads.
- **LFU** — least frequently used. Keep a hit count per entry,
  evict the lowest. Better on workloads with stable hot keys (a
  CDN serving the same popular files). Worse on workloads where
  popularity shifts.
- **FIFO** — drop the oldest inserted. Cheaper than LRU (no
  per-access update), worse hit rate.
- **Random** — pick a random entry to evict. Surprisingly
  competitive; used by Redis as `allkeys-random`.
- **LRU-K** — track the last K accesses, evict by K-th most
  recent. Better at distinguishing one-hit-wonders from real
  hot keys.
- **ARC (Adaptive Replacement Cache)** — IBM's hybrid that
  balances between recency (LRU) and frequency (LFU) adaptively.
  Patent-encumbered for years; now free. Used by ZFS.
- **2Q** — two queues: one for recently-seen-once items, one for
  proven-hot items. Cheap approximation of ARC.
- **TinyLFU / W-TinyLFU** — frequency sketch (Count-Min) for
  admission control. Used by Caffeine (Java) and most modern
  high-end caches. Often the best general-purpose policy as of
  the late 2010s.
- **CLOCK** — approximation of LRU using a bit per entry and a
  circular pointer. Used by OS page caches. Cheaper than true
  LRU because no list manipulation per access.

For application caches at small scale, LRU is fine. At cache-of-
many-millions scale, W-TinyLFU usually wins; Redis 4.0+ ships an
approximation (`allkeys-lfu`) for similar reasons.

## Redis's `maxmemory-policy`

Redis lets you pick eviction explicitly:

```
maxmemory-policy allkeys-lru        # evict any key by LRU
maxmemory-policy volatile-lru       # evict only keys with TTL
maxmemory-policy allkeys-lfu        # LFU (since 4.0)
maxmemory-policy volatile-lfu
maxmemory-policy allkeys-random
maxmemory-policy volatile-ttl       # evict shortest TTL
maxmemory-policy noeviction         # error on write when full
```

Redis's LRU and LFU are **sampled** — it doesn't maintain a true
LRU list (too expensive at Redis's scale). On eviction, it picks
N random entries and evicts the worst. The `maxmemory-samples`
knob defaults to **5** (unchanged across modern Redis versions);
as N goes up, behavior approaches true LRU at the cost of more
CPU per eviction.

This is the production-cache lesson: **true LRU is rarely worth
the cost**. Sampled or approximate policies hit nearly the same
ratio with much less bookkeeping.

## The classic interview pattern

"Design and implement an LRU cache with O(1) get and put."

The expected answer:

1. Identify the two-structures pattern: hash + doubly-linked list.
2. Sketch the data structures.
3. Walk through get / put logic.
4. Mention sentinel nodes as the code simplification.
5. Write it. ~30 lines.
6. Bonus points: mention the `Map`-insertion-order trick; mention
   that real systems use sampled LRU/LFU; mention thread safety
   (a global lock for single-machine caches, a shard-per-thread
   for high concurrency).

## Real-world design considerations

- **Concurrency.** The naive implementation isn't thread-safe.
  Real caches either lock globally (simple, contention-prone) or
  shard by hash (Caffeine, Guava) with a separate LRU per shard.
- **TTL** alongside LRU. Most production caches expire entries
  by both time and capacity. Eviction = "expired" OR "LRU-evicted
  when full."
- **Memory accounting.** "Capacity = N entries" is the easy
  version. "Capacity = M bytes" requires per-entry size tracking
  (because values vary wildly) and is what real caches do.
- **Negative caching.** Cache "this key doesn't exist" with a
  short TTL to avoid thundering herd on missing keys.
- **Stampede protection.** When a hot cache key expires, 1000
  concurrent requests can all rebuild it. Solutions: per-key
  locks, request coalescing, probabilistic early refresh.
- **Cache warmup.** On process start, the cache is cold and
  upstream takes the full load. Either preload the top-N keys
  or accept the warmup window.

The data structure is the easy part. The system around it is the
job.

## The discipline

- LRU = hash + doubly-linked list. O(1) get and put. ~30 lines.
- Sentinel head/tail nodes make the splicing code branch-free.
- The `Map`-insertion-order trick works but is slower in practice;
  use the explicit version for hot paths.
- Real production caches rarely use *true* LRU — sampled LRU or
  W-TinyLFU has nearly the same hit rate at a fraction of the
  per-access cost.
- Eviction policy choice depends on workload: LRU baseline, LFU
  for stable popularity, W-TinyLFU for the best general-purpose
  hit rate.
- The algorithmic design is the interview; the production design
  is concurrency, memory accounting, TTLs, and stampede
  protection.
