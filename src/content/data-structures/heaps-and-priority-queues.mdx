# Heaps and priority queues

A priority queue gives you "next-best item" in O(log n) — pop the
smallest (or largest), with new items arriving over time. The
canonical implementation is a **binary heap** stored as an array.
JS has no built-in priority queue, so you'll either write one or
pull in a library. The implementation is ~30 lines.

## What a heap is

A binary heap is a binary tree with two invariants:

1. **Shape**: complete tree (all levels full except possibly the
   last, which fills left to right).
2. **Order**: every parent is ≤ its children (min-heap) or ≥ its
   children (max-heap).

The shape invariant means you can store it in an array with no
pointers:

```js
// Array layout for a min-heap:
//   index 0 → root
//   index i → parent at (i-1)/2, children at 2i+1 and 2i+2
//
// Tree:                Array:
//        1               [1, 3, 2, 7, 5, 4]
//       / \
//      3   2
//     / \  /
//    7  5 4
```

No node objects, no pointers, no allocation per insert. Just one
array.

## The two core operations

**Push** (add an item):

1. Append to the end of the array.
2. *Sift up*: while the new item is smaller than its parent, swap
   them.

That's O(log n) — at most one swap per level, and the tree has
log₂(n) levels.

**Pop** (remove the min):

1. Save the root (the answer).
2. Move the last element to the root.
3. *Sift down*: while it's larger than its smallest child, swap.

Also O(log n).

## A minimal min-heap

```js
class MinHeap {
  constructor(compare = (a, b) => a - b) {
    this.data = [];
    this.compare = compare;
  }
  size() { return this.data.length; }
  peek() { return this.data[0]; }
  push(item) {
    this.data.push(item);
    this._siftUp(this.data.length - 1);
  }
  pop() {
    if (this.data.length === 0) return undefined;
    const top = this.data[0];
    const last = this.data.pop();
    if (this.data.length > 0) {
      this.data[0] = last;
      this._siftDown(0);
    }
    return top;
  }
  _siftUp(i) {
    while (i > 0) {
      const parent = (i - 1) >> 1;
      if (this.compare(this.data[i], this.data[parent]) >= 0) break;
      [this.data[i], this.data[parent]] = [this.data[parent], this.data[i]];
      i = parent;
    }
  }
  _siftDown(i) {
    const n = this.data.length;
    while (true) {
      const l = 2 * i + 1, r = 2 * i + 2;
      let smallest = i;
      if (l < n && this.compare(this.data[l], this.data[smallest]) < 0) smallest = l;
      if (r < n && this.compare(this.data[r], this.data[smallest]) < 0) smallest = r;
      if (smallest === i) break;
      [this.data[i], this.data[smallest]] = [this.data[smallest], this.data[i]];
      i = smallest;
    }
  }
}
```

That's a working priority queue. Memorize it; it shows up in
interviews and in code.

## Heapify: building a heap from an array

Naive: push n items, each O(log n) → O(n log n).

Smarter: start with the array as-is, sift-down from the last
non-leaf to the root. O(n). Why O(n)? Half the nodes are leaves
(sift-down is free for them), a quarter are one level up (one
swap), etc. The sum converges.

```js
static heapify(arr, compare) {
  const h = new MinHeap(compare);
  h.data = arr;
  for (let i = (arr.length >> 1) - 1; i >= 0; i--) h._siftDown(i);
  return h;
}
```

Use this when you have all items up front. Worth it for n > ~50.

## Top-K with a heap

"Find the K largest items in an array of N."

Naive: sort the array, take the last K. O(N log N).

Better with a **min-heap of size K**:

1. Push the first K items.
2. For each subsequent item: if it's larger than the heap's min,
   pop the min and push the new item.
3. The heap now holds the K largest.

```js
function topK(arr, k) {
  const heap = new MinHeap();
  for (const x of arr) {
    if (heap.size() < k) heap.push(x);
    else if (x > heap.peek()) {
      heap.pop();
      heap.push(x);
    }
  }
  return [...heap.data].sort((a, b) => b - a);
}
```

Complexity: O(N log K). For K much smaller than N (top-10 of a
million), this is dramatically faster than sorting.

This is also how **streaming top-K** works — you can't sort an
infinite stream, but you can maintain a size-K min-heap forever.

## Other classic uses

**Dijkstra's shortest path.** Pull the unvisited node with the
smallest tentative distance. Each `decrease-key` is push-a-better-
version-and-skip-stale-entries-on-pop (since binary heaps don't
have efficient decrease-key — Fibonacci heaps do, but they have
huge constants and are rarely worth it in practice).

```js
function dijkstra(graph, start) {
  const dist = new Map();
  dist.set(start, 0);
  const pq = new MinHeap((a, b) => a[0] - b[0]);
  pq.push([0, start]);
  while (pq.size()) {
    const [d, u] = pq.pop();
    if (d > (dist.get(u) ?? Infinity)) continue; // stale entry
    for (const [v, w] of graph.get(u) ?? []) {
      const nd = d + w;
      if (nd < (dist.get(v) ?? Infinity)) {
        dist.set(v, nd);
        pq.push([nd, v]);
      }
    }
  }
  return dist;
}
```

**Event scheduling / simulation.** Pull the next-scheduled event.
Game loops, discrete-event simulators, task scheduling all use a
priority queue keyed by timestamp.

**Heapsort.** Heapify then pop n times. O(n log n), in-place. Not
quite as fast as quicksort in practice (worse cache behavior) but
guaranteed worst-case O(n log n).

**Median maintenance.** Two heaps — a max-heap of the lower half
and a min-heap of the upper half. Median is the top of one or the
average of both tops. Each insert is O(log n).

## What JS gives you (nothing)

There's no `PriorityQueue` in the standard library. Options:

- Write one (above). 30 lines.
- Pull a library: `@datastructures-js/priority-queue` or
  `heap-js`.
- Use a sorted array — `arr.splice` to insert in order. O(n) per
  insert. Fine for `n < ~100`, bad above that.

If you reach for `arr.sort()` after every push, you're doing
O(n log n) work per push instead of O(log n). Don't.

## Decrease-key without decrease-key

Binary heaps can't efficiently find an existing element to lower
its priority (it would require O(n) search). The standard
workaround is the **stale-entry trick** shown in Dijkstra above:
push a new better entry, ignore the old one when it surfaces.

This wastes some memory (the heap can grow past the unique-key
count) but keeps everything O(log n). For graphs where the
average decrease count is small, it's fine. For dense graphs with
many updates, look at indexed heaps or Fibonacci heaps.

## d-ary heaps

A binary heap has 2 children per node. A d-ary heap has d. Sift-up
gets faster (fewer levels: log_d(n)); sift-down gets slower (have
to compare d children to find the smallest).

For Dijkstra on dense graphs, 4-ary heaps often outperform binary
heaps. For general use, binary is fine. Not worth tuning unless
profiling shows the heap is the bottleneck.

## The discipline

- Heap = "next smallest/largest in O(log n)". That's the whole API.
- Push, pop, peek. Heapify when you have n items at once.
- Don't sort an array per insert. Don't scan for max per pop.
- For Dijkstra, use the stale-entry trick — binary heaps don't
  do decrease-key.
- For top-K of a large stream, a size-K min-heap is the textbook
  answer and it's correct.
- JS doesn't ship one; write 30 lines or pull a library.
