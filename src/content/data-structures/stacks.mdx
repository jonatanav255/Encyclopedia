# Stacks

A stack is the simplest non-trivial data structure. Three operations,
all O(1), and one rule: the last thing you put in is the first thing
you take out. LIFO. That's it. Despite the smallness, it's everywhere
— the language runtime is using one to call the function you're reading
this from.

## The API, in full

```java
push(x);   // put x on top
pop();     // remove and return the top
peek();    // look at the top without removing
isEmpty(); // any items?
```

Four methods. Three of them mutate. All O(1). If your stack
implementation has anything else on its public surface, it's not a
stack anymore — it's a stack plus opinions.

## Two implementations

You can back a stack with either an array or a linked list. Both give
you the same Big-O. They don't give you the same wall-clock speed.

| | Array-backed | Linked-list-backed |
|---|---|---|
| Layout | Backing array + size counter | Chain of heap nodes |
| `push` | `arr[size++] = x` | New node points at old head; head = new node |
| `pop` | `return arr[--size]` | `head = head.next` |
| Growth | Resize geometrically when full | No resize — just allocate a node |
| Cache | Friendly — packed contiguously | Hostile — scattered across heap |
| Allocation | None per push (amortized) | One per push |
| Practical speed | Wins on modern CPUs | Almost always slower |

The textbook treats them as interchangeable. They aren't. The
array-backed version wins by a wide margin on real hardware, for the
same reasons covered in `arrays-vs-linked-lists.mdx` — cache lines,
prefetchers, and one allocation versus n. The linked-list version
only beats it when you cannot tolerate the occasional resize pause
(realtime systems) or when nodes are referenced from elsewhere.

A bare-bones array-backed stack:

```java
class IntStack {
    private int[] data = new int[16];
    private int size = 0;

    void push(int x) {
        if (size == data.length) {
            data = java.util.Arrays.copyOf(data, data.length * 2);
        }
        data[size++] = x;
    }

    int pop() {
        if (size == 0) throw new java.util.NoSuchElementException();
        return data[--size];
    }

    int peek() {
        if (size == 0) throw new java.util.NoSuchElementException();
        return data[size - 1];
    }

    boolean isEmpty() { return size == 0; }
}
```

That's a working stack. You will almost never write this in production
Java, for reasons in the next section.

## Don't use `java.util.Stack`

Java has a `java.util.Stack` class. Don't use it. The class extends
`java.util.Vector`, which is synchronized — every `push`, `pop`, and
`peek` acquires a monitor, even when only one thread will ever touch
the stack. That's pure overhead for the common case.

The Javadoc for `Stack` itself tells you this, almost in those words:

> A more complete and consistent set of LIFO stack operations is
> provided by the `Deque` interface and its implementations, which
> should be used in preference to this class.

When the standard library tells you not to use one of its own classes,
listen.

## The right answer: `ArrayDeque`

`ArrayDeque` is a resizable circular-array deque, and it's the
recommended stack in modern Java. No synchronization, no per-push
allocation, faster than `Stack` and faster than `LinkedList` for
stack-like access.

```java
Deque<Integer> stack = new ArrayDeque<>();
stack.push(1);
stack.push(2);
stack.pop();   // 2
stack.peek();  // 1
```

That's it. Program against `Deque<T>`, instantiate `ArrayDeque<>`, use
`push`/`pop`/`peek`. Don't reach for `LinkedList` either — same
interface, much worse cache behavior.

<Callout variant="info">
`ArrayDeque` rejects `null`. That's not a bug; it lets `peek()`
return `null` to mean "empty" without ambiguity. If you need a stack
of nullable references, wrap them in `Optional` or use a sentinel.
</Callout>

## The call stack is a stack

Every method call in the JVM pushes a **stack frame** onto the
thread's call stack: the return address, the local variables, and
the operand stack for that invocation. The method returns, and the
frame is popped. The whole control-flow mechanism of structured
programming is one big LIFO discipline.

This is the same JVM stack covered in `memory-and-pointers.mdx`. It's
fixed-size per thread (typically 512KB to 1MB; tunable with `-Xss`),
which is why infinite recursion blows up:

```java
int depth(int n) { return depth(n + 1); }
// java.lang.StackOverflowError
```

`StackOverflowError` doesn't mean your data is corrupted. It means
you pushed more frames than the stack has room for — usually because
your recursion has no base case, or because the recursion is correct
but the input is deeper than the stack can hold.

## Recursion and explicit stacks are the same shape

Any recursive algorithm can be rewritten as a loop with an explicit
stack. The shape of the problem is identical; what differs is *whose*
stack is doing the work — the JVM's, or one you allocate on the heap.

The heap version doesn't blow up on deep inputs. The heap is
gigabytes; the call stack is a megabyte. For traversals of
deeply-skewed trees or large graphs, you want the heap.

### DFS, two ways

Recursive:

```java
void dfs(Node n) {
    if (n == null || n.visited) return;
    n.visited = true;
    visit(n);
    for (Node c : n.children) dfs(c);
}
```

Iterative with an explicit stack:

```java
void dfs(Node start) {
    Deque<Node> stack = new ArrayDeque<>();
    stack.push(start);
    while (!stack.isEmpty()) {
        Node n = stack.pop();
        if (n == null || n.visited) continue;
        n.visited = true;
        visit(n);
        for (Node c : n.children) stack.push(c);
    }
}
```

Same traversal order (modulo push order of children). The iterative
version survives a million-node chain; the recursive one doesn't.

See `algorithms/bfs-and-dfs.mdx` for the broader picture. A queue
gives you BFS; a stack gives you DFS. That's the only difference.

### Iterative inorder traversal

The classic recursive inorder is three lines. The iterative version
is the standard interview answer for "show me you understand stacks":

```java
void inorder(TreeNode root) {
    Deque<TreeNode> stack = new ArrayDeque<>();
    TreeNode cur = root;
    while (cur != null || !stack.isEmpty()) {
        while (cur != null) {
            stack.push(cur);
            cur = cur.left;
        }
        cur = stack.pop();
        visit(cur);
        cur = cur.right;
    }
}
```

Walk left, pushing as you go. When you can't go left anymore, pop,
visit, and pivot right. The stack holds the "I'll come back to you"
list.

## Expression evaluation

The canonical use of a stack outside of language runtimes. To
evaluate a **postfix** (reverse Polish) expression like `3 4 + 2 *`:
scan left to right, push numbers, and when you hit an operator, pop
two operands and push the result.

```java
int evalPostfix(String[] tokens) {
    Deque<Integer> s = new ArrayDeque<>();
    for (String t : tokens) {
        switch (t) {
            case "+" -> { int b = s.pop(); int a = s.pop(); s.push(a + b); }
            case "-" -> { int b = s.pop(); int a = s.pop(); s.push(a - b); }
            case "*" -> { int b = s.pop(); int a = s.pop(); s.push(a * b); }
            case "/" -> { int b = s.pop(); int a = s.pop(); s.push(a / b); }
            default  -> s.push(Integer.parseInt(t));
        }
    }
    return s.pop();
}
```

That's a working expression evaluator. The full **shunting-yard
algorithm** converts infix (`3 + 4 * 2`) to postfix in one pass with
two stacks — one for operators, one for output — handling precedence
and parentheses. Every expression evaluator in every language,
spreadsheet, and calculator app is some variant of this. All one-pass,
all O(n).

Parentheses matching is the trivial version: push `(`, pop on `)`,
check they match. If the stack is non-empty at the end, you have
unbalanced parens.

## Undo/redo as two stacks

The standard implementation of undo/redo in editors, IDEs, and design
tools:

- Every user action creates a command object and pushes it onto the
  **undo stack**.
- Undo pops the top of the undo stack, reverses the action, and pushes
  the command onto the **redo stack**.
- Redo pops from the redo stack, reapplies, pushes back onto undo.
- Any **new** action clears the redo stack — you've branched off the
  history.

That's the entire model. The command object holds enough information
to invert itself (the old value, the deleted range, the moved
coordinates). The two stacks do the rest.

## The monotonic stack pattern

A stack you keep in monotonic (increasing or decreasing) order. When
a new element would violate the order, you pop until it doesn't. It's
an interview chestnut, but the underlying problems show up in real
analytics work: "next greater element," "largest rectangle in a
histogram," "daily temperatures," stock-span problems.

The trick is amortization. Each element gets pushed once and popped
at most once, so even though the inner loop *looks* O(n), the total
work across all iterations is O(n). Problems that look quadratic
become linear.

```java
// Next greater element to the right, for each index.
int[] nextGreater(int[] a) {
    int[] result = new int[a.length];
    java.util.Arrays.fill(result, -1);
    Deque<Integer> stack = new ArrayDeque<>(); // holds indices
    for (int i = 0; i < a.length; i++) {
        while (!stack.isEmpty() && a[stack.peek()] < a[i]) {
            result[stack.pop()] = a[i];
        }
        stack.push(i);
    }
    return result;
}
```

When you see "find the next/previous X for each element" in O(n),
this is the shape.

## Bounded vs unbounded

The `ArrayDeque` shown above is **unbounded** — it resizes on demand.
That's almost always what you want. The resize is amortized O(1), and
the brief copy is invisible at human time scales.

A **bounded** stack matters in three places: embedded systems where
you can't allocate at runtime, realtime systems where a resize pause
would miss a deadline, and security-sensitive code where an
attacker-controlled push count could OOM the process. In those cases
you pre-allocate a fixed array and reject `push` when full. For
ordinary backend code, don't bother.

## The discipline

- LIFO with three operations: push, pop, peek. That's the whole
  data structure.
- In Java, use `ArrayDeque` via the `Deque` interface. Not
  `java.util.Stack` (synchronized, deprecated in spirit), not
  `LinkedList` (cache-hostile).
- Array-backed beats list-backed in practice. Locality wins.
- Recursive code uses the JVM's call stack implicitly. Rewrite as
  a loop with an explicit `Deque` when depth could exceed `-Xss`.
- DFS, expression evaluation, undo/redo, parentheses matching,
  monotonic-stack tricks — all the same shape. Learn the shape once.
- `StackOverflowError` means your recursion is too deep, not that
  your data is wrong. Increase `-Xss` or go iterative.
