# Rendering patterns

CSR, SSR, SSG, ISR, Islands, RSC. The acronyms multiplied because
the trade-offs multiplied. Every rendering pattern is a different
answer to two questions: **when does HTML get built?** and **where
does JavaScript run?** Pick a pattern by reasoning about those, not
by following a framework's default.

## The five strategies

### CSR — client-side rendering

The server sends a near-empty HTML shell (`<div id="root"></div>`)
and a JS bundle. The browser downloads, parses, and runs the JS,
which fetches data and renders the DOM. Classic SPA.

- **Time-to-first-byte:** fast (static shell).
- **Time-to-interactive:** slow (JS download + execute + fetch).
- **SEO:** depends on crawler executing JS (Google does, most don't reliably).
- **Server cost:** near zero — static file hosting.
- **Best for:** apps behind a login (no SEO concern, repeat visitors
  cache the bundle), tools, dashboards.

### SSR — server-side rendering

The server runs the framework, queries the DB, builds the full HTML,
and streams it to the browser. The client then *hydrates* — attaches
event handlers to the existing DOM so it becomes interactive.

- **TTFB:** depends on server-side work — DB queries are now in the
  request critical path.
- **TTI:** much faster *perceived* (content visible before JS runs)
  but JS still has to load and hydrate to be interactive.
- **SEO:** great — crawlers see real HTML.
- **Server cost:** high — render-per-request, no shared HTML cache.
- **Best for:** content-heavy public pages where SEO and first-paint
  matter (marketing sites, news, e-commerce product pages).

### SSG — static site generation

Build-time: render every page to HTML, deploy the resulting flat
files. Same shape as SSR but the render happens once, at deploy,
not per request.

- **TTFB:** fastest of all options — CDN edge serves static files.
- **TTI:** same as SSR (hydration cost is identical).
- **SEO:** great.
- **Server cost:** zero per request; build cost scales with pages.
- **Best for:** docs sites (you're reading one), marketing sites,
  blogs — anywhere content changes infrequently and is the same for
  every visitor.

The break point: SSG falls over when you have 100k+ pages (long
builds) or content that updates faster than your deploy cycle.

### ISR — incremental static regeneration

Hybrid. Pages are statically generated, but a stale page is
regenerated on demand (background revalidation) or on a schedule.
Vercel's Next.js popularized the name; the same idea lives in
Astro, Gatsby, and as "stale-while-revalidate" at the CDN layer.

- **TTFB:** static-fast on cache hit, render-slow on miss.
- **Freshness:** configurable per page.
- **Server cost:** scales with miss rate, not request rate.
- **Best for:** e-commerce with thousands of product pages,
  documentation with frequent updates, anywhere SSG's freshness
  story falls short.

### Islands

The page is mostly static HTML. Small interactive zones —
*islands* — get their own JS bundles, hydrated independently. Astro
and Fresh popularized it; Qwik takes it further with *resumability*
(no hydration at all; the framework restores state from serialized
HTML).

- **TTI:** much faster than SSR because you ship JS only for the
  parts that need it.
- **JS bundle size:** drops dramatically — often 10× less.
- **Architecture cost:** every interactive component is a build-time
  boundary; cross-island state is awkward.
- **Best for:** mostly-static sites with sprinkled interactivity (a
  comment box on a blog, a buy button on a product page).

### RSC — React Server Components

Components render on the server (with database/file access in scope)
and stream serialized output to the client. The client never receives
the server component's code. Client components are clearly marked
(`'use client'`) and ship JS as before.

- **Bundle size:** drops because data-fetching components don't ship.
- **Data fetching:** moves out of `useEffect` and into component
  bodies that run on the server.
- **TTI:** depends on how much client-component work remains.
- **Mental model cost:** highest of any pattern — two component
  worlds with explicit boundaries.
- **Best for:** apps with a real backend, large component trees, and
  the engineering capacity to learn the model.

## Hydration — the cost everyone forgets

Any pattern that ships *rendered HTML + JS* (SSR, SSG, ISR, Islands,
RSC) faces a hydration cost: the browser still has to download and
execute the JS, walk the DOM, and attach handlers. Until that
finishes, the page looks interactive but isn't — buttons don't
respond.

Recent frameworks chip at this in different ways:

- **Partial hydration / Islands** — only hydrate marked components.
- **Progressive hydration** — hydrate above-the-fold first, below-the-fold
  later.
- **Selective hydration** — React 18+: hydrate based on user
  interaction (clicking an unhydrated component prioritizes it).
- **Resumability (Qwik)** — instead of replaying the render, serialize
  the framework state into the HTML and restore on event. Zero JS to
  reach interactivity for unused paths.

Hydration cost scales with the *component tree*, not with content
size. A page of marketing text with two interactive widgets hydrates
fast; a page of fifty rich components hydrates slowly even if the
text is identical.

## The decision tree

```
Is SEO required for these pages?
├─ NO  → CSR (cheapest, simplest, if first-paint is acceptable)
└─ YES → Is content the same for every visitor?
         ├─ YES → How often does it change?
         │       ├─ Rare → SSG
         │       └─ Often → ISR (or SSG with frequent rebuilds)
         └─ NO  → Is the page mostly static with a few interactive bits?
                  ├─ YES → Islands
                  └─ NO  → SSR (or RSC if you want server-only data fetches)
```

A real app usually picks **multiple** patterns. Marketing pages SSG,
product pages ISR, the authenticated dashboard CSR, the checkout
page SSR. Frameworks like Next.js, Astro, Nuxt, and Remix support
this per-route mixing as a first-class feature.

## What about API design?

The rendering pattern leaks into your API:

- **CSR** needs REST/GraphQL endpoints — the client talks to the
  backend over HTTP.
- **SSR/SSG** can call services directly from the render — no public
  API needed for data the server already has.
- **RSC** explicitly removes the "fetch on the client" step for
  server-fetched data.

For a public API consumed by mobile + web, you need REST/GraphQL
regardless. For an internal app, RSC's "no API for server data" is
genuinely freeing.

## Edge rendering

Some patterns (SSR, ISR, edge-rendered RSC) can run at CDN edge
locations instead of a single origin. Latency drops to ~50ms
worldwide, but:

- Edge runtimes are restricted — no native modules, often no Node
  built-ins, sometimes only Web APIs.
- DB latency becomes the bottleneck — your edge server in Tokyo
  hitting a Postgres in Virginia gives back what edge gained.
- Cold starts at the edge can be worse than at origin (more
  locations, more cold starts).

Edge SSR helps when your data layer is also distributed (Planetscale,
Turso, edge-cached APIs). Otherwise it can be slower than origin SSR
with a CDN in front.

## Streaming

SSR and RSC both stream — sending HTML in chunks as it renders
rather than waiting for the full page. The browser starts parsing
and rendering immediately, so first-content-paint can happen
before the server is done. React's `<Suspense>` boundaries declare
where streaming chunks can split.

Streaming pairs well with slow data fetches: the header renders
immediately, a Suspense boundary around the slow part shows a
fallback, and the real content streams in when the data arrives.
TTFB improves; total render time is the same.

## Common mistakes

- **CSR by default for content sites.** Anything search engines or
  link-previews need to read should not require JS execution.
- **SSR everywhere "for performance."** SSR is slower than SSG
  when content is shared, and hydration is the same cost. Use SSG
  unless you actually have per-request variation.
- **Islands without measuring.** If your page has 30 islands, you
  haven't reduced the JS bundle, you've just split it. Profile.
- **RSC because it's new.** RSC has the steepest mental-model cost
  of any pattern. If your app is small or your team is small,
  Islands or SSR may serve you better.
- **Hydration mismatch in production.** Server renders one thing,
  client renders another (timezones, randomness, conditional UA
  checks). The hydration error is loud in dev, silent in
  production, and produces broken UI.
