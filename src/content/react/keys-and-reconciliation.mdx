# Keys and reconciliation

When state changes, React calls your components again, producing a
new tree of elements. It compares this new tree against the previous
one and figures out the minimal set of DOM operations to update the
real DOM. This process is called **reconciliation**.

For lists, **keys** are how React matches new items to old ones.
Get them wrong and you get bugs that look like React "lost track."

## The basics

```jsx
{users.map((u) => <UserCard key={u.id} user={u} />)}
```

The `key` prop is mandatory for list items rendered from arrays.
React uses it to know which previous item each new item corresponds
to.

Requirements:

- **Unique among siblings.**
- **Stable across renders.** Don't generate fresh IDs each render.
- **Tied to the data**, not the position.

## Why keys matter

Without keys (or with bad keys), React matches items by **position**.
If the list reorders, React thinks "item at index 0 became item B" —
it doesn't realize they're the same item moved.

```jsx
// Index key — fine if list never changes order
{items.map((item, i) => <Row key={i} item={item} />)}

// User reorders — React thinks items at indices changed.
// State inside Row (input value, focus, animation) goes to the wrong row.
```

Real IDs match correctly:

```jsx
{items.map((item) => <Row key={item.id} item={item} />)}

// User reorders — React sees "item 3 moved from index 0 to index 5"
// and moves the DOM node. State stays with the right item.
```

## What can break

### Inputs lose focus or text

```jsx
function App() {
  const [items, setItems] = useState([
    { id: 1, name: 'a' },
    { id: 2, name: 'b' },
  ]);

  return (
    <>
      {items.map((item, i) => (
        <Row key={i} item={item} />   // BAD — index key
      ))}
      <button onClick={() => setItems([items[1], items[0]])}>Swap</button>
    </>
  );
}

function Row({ item }) {
  const [extra, setExtra] = useState('');
  return (
    <>
      <span>{item.name}</span>
      <input value={extra} onChange={(e) => setExtra(e.target.value)} />
    </>
  );
}
```

User types "hello" in the first row's input. Hits Swap. Now the
input that *was* in row B has the text "hello," because index 0 is
still the same component instance — just with different `item`
data.

Use `key={item.id}` and the input stays with item A as it moves.

### Animations restart

When position-based keys treat items as new, React unmounts the old
DOM and mounts new. Any CSS transition or mount animation reruns.

### Performance churn

Unnecessary DOM creation/destruction is slow. Stable keys let React
move existing nodes instead.

## When index keys are OK

- The list is **never reordered, filtered, or modified at the
  middle/start** (only appended at the end).
- Items have no local state, no input, no animation.
- The list is short and re-renders cheaply.

Examples: a fixed list of static labels, a list of strings you only
ever push to.

When in doubt, use a real ID. Generating stable IDs is cheap.

## Reconciliation rules

React's diffing algorithm makes simplifying assumptions:

1. **Different types unmount and remount.** `<div>` → `<section>`
   destroys the div and creates a section. State is lost.

```jsx
{isActive ? <div className="active">A</div> : <section>A</section>}
// ❌ Children completely rebuilt on toggle
```

Better:

```jsx
<div className={isActive ? 'active' : ''}>A</div>
// ✅ Same div, just changed class
```

2. **Same type updates props.** `<div className="a" />` →
   `<div className="b" />` just updates the className. The DOM node
   is reused.

3. **Children matched by key (in lists).** Without keys, by position.

These rules are why component identity matters. Two components that
look the same but have different types (e.g., conditionally
swapping `<A />` and `<B />`) lose state between renders.

## Different types ≠ different children

```jsx
{loggedIn ? <Profile /> : <Login />}
```

Switching between two components: each unmounts/mounts as you toggle.
Their states reset.

```jsx
<UserSection user={user} />   // same component, different prop
```

Same component throughout: state preserved as the prop changes.

## `key` to force a remount

You can use `key` to *intentionally* reset a component:

```jsx
<Form key={selectedUser.id} user={selectedUser} />
```

When `selectedUser.id` changes, React treats it as a different
component, unmounts the old, mounts a new one. All internal state
(inputs, scroll position, expansion state) resets.

Useful for "select a different user → form should reset" patterns.

## `Fragment` keys

Fragments shorthand (`<>...</>`) can't take keys. Use the named
form:

```jsx
{items.map((item) => (
  <React.Fragment key={item.id}>
    <dt>{item.label}</dt>
    <dd>{item.value}</dd>
  </React.Fragment>
))}
```

## React 19's key changes

React 19 still treats keys the same way. Nothing in the model
changed. The `key` rule is one of the oldest invariants in React.

## Common pitfalls

### Random keys

```jsx
{items.map((item) => <Row key={Math.random()} item={item} />)}
```

Every render generates new keys. React thinks every item is new.
Everything unmounts and remounts every render. State, animations,
inputs all break.

Don't generate keys at render time. Use stable IDs.

### Concatenated string keys

```jsx
{items.map((item) => <Row key={`${item.type}-${item.id}`} item={item} />)}
```

OK if `type` and `id` together are unique and stable. Sometimes you
need this when items come from different sources mixed together.
Just be sure the concatenation can't accidentally collide.

### Keys on non-list elements

```jsx
<div key={something}>...</div>   // unnecessary unless this is a list item
```

Keys only matter for siblings in an array. On a single child, the
key is ignored — except when you're using it intentionally to force
remount (as above).

### Keys not at the array level

```jsx
{items.map((item) => (
  <li>                            // BAD — key needs to be here
    <span key={item.id}>{item.name}</span>
  </li>
))}
```

The key must be on the **outermost** element returned by the map
callback (the `<li>` in this case), not on a nested child.

## When state survives reordering

With proper keys, when items reorder:

- React moves DOM nodes (instead of recreating them).
- Component state stays with the data (the right input keeps its
  value).
- Effects don't re-run unless their deps changed.
- Animations don't restart.

This is what stable keys buy you. The list visually reshuffles, but
each item is the same instance.

## The mental model

React diffs new trees against old, looking for "same component, same
key" matches. Matches get prop updates; mismatches get
unmount/remount.

For lists, keys are how you tell React "this item is the same as
before, just at a different index." Bad keys break that telling.

Three rules to live by:

1. Always provide `key` on list items rendered from arrays.
2. Use stable IDs from the data, not positions or random values.
3. Use `key` to intentionally force a remount when you want fresh
   state.

The bugs from bad keys are subtle — inputs forgetting their text,
animations restarting, components misbehaving "for no reason."
Knowing about keys saves hours of debugging.
