# `useRef`

`useRef` gives you a mutable object that survives across renders.
Two main uses: holding a value that shouldn't trigger re-renders, and
accessing a DOM node directly.

```jsx
const ref = useRef(initialValue);
ref.current;          // read
ref.current = newVal;  // mutate — no re-render
```

`ref` is the same object across all renders. `ref.current` is freely
mutable. Mutations don't trigger re-renders.

## Accessing a DOM node

The most common use:

```jsx
function TextInput() {
  const inputRef = useRef(null);

  function focusInput() {
    inputRef.current.focus();
  }

  return (
    <>
      <input ref={inputRef} />
      <button onClick={focusInput}>Focus</button>
    </>
  );
}
```

The `ref` prop on a DOM element receives the underlying DOM node.
`inputRef.current` is the `<input>` element — you can call
`.focus()`, `.scrollIntoView()`, `.getBoundingClientRect()`, etc.

Initial value: usually `null`. After the component mounts and the
input is in the DOM, `ref.current` points to it.

## Storing a value across renders

When you need a value that persists but shouldn't trigger renders:

```jsx
function Timer() {
  const intervalRef = useRef(null);

  function start() {
    intervalRef.current = setInterval(tick, 1000);
  }

  function stop() {
    clearInterval(intervalRef.current);
  }

  return ...;
}
```

The timer ID needs to survive renders (so `stop` can clear it), but
storing it in state would re-render on every change for no UI reason.
A ref is right.

Other use cases:

- Cached values that recompute on demand but don't drive UI.
- Tracking previous values (with `useEffect`).
- Storing event-handler counts, retry attempts, last-clicked-at.
- DOM dimensions captured for measurement.

## What `useRef` is NOT

- **Not state.** Mutating a ref doesn't trigger a re-render. If the
  value affects what's rendered, you need state.
- **Not for derived values.** Compute those in render.
- **Not for performance optimization.** Don't replace state with refs
  just to avoid renders — the result usually isn't visible.

The rule of thumb: if the user should see when the value changes,
use state. If it's internal bookkeeping, ref.

## Refs and effects

A common pattern: read DOM dimensions or focus on mount.

```jsx
function AutoFocus() {
  const ref = useRef(null);

  useEffect(() => {
    ref.current.focus();
  }, []);

  return <input ref={ref} />;
}
```

The effect runs after the DOM is committed, so `ref.current` is
populated.

## Forwarding refs

By default, custom components don't accept a `ref` — only DOM
elements do. To forward a ref to an internal element, accept it as
a prop (React 19+) or use `forwardRef` (React 18):

```tsx
// React 19+
type FancyInputProps = React.ComponentPropsWithRef<'input'>;

function FancyInput({ ref, ...rest }: FancyInputProps) {
  return <input ref={ref} className="fancy" {...rest} />;
}
```

```tsx
// React 18 (still common)
const FancyInput = React.forwardRef<HTMLInputElement, React.ComponentPropsWithoutRef<'input'>>(
  function FancyInput(props, ref) {
    return <input ref={ref} className="fancy" {...props} />;
  },
);
```

Now the parent can:

```tsx
const inputRef = useRef<HTMLInputElement>(null);
<FancyInput ref={inputRef} />
inputRef.current?.focus();
```

The forwarded ref bypasses the wrapper component and points at the
real input.

See **Forwarding refs and imperative handle** for the full treatment
(typing, migration, common pitfalls).

## `useImperativeHandle`

For exposing a custom API to the parent rather than the raw DOM:

```tsx
type FancyInputHandle = {
  focus: () => void;
  clear: () => void;
};

type FancyInputProps = React.ComponentPropsWithoutRef<'input'> & {
  ref?: React.Ref<FancyInputHandle>;
};

function FancyInput({ ref, ...props }: FancyInputProps) {
  const inputRef = useRef<HTMLInputElement>(null);

  useImperativeHandle(ref, () => ({
    focus: () => inputRef.current?.focus(),
    clear: () => { if (inputRef.current) inputRef.current.value = ''; },
  }), []);

  return <input ref={inputRef} {...props} />;
}
```

```tsx
const ref = useRef<FancyInputHandle>(null);
ref.current?.focus();
ref.current?.clear();
```

Useful when the parent shouldn't have arbitrary DOM access — exposes
only the methods you want. Generally a sign that the API should
maybe be props, but valid for imperative needs (focus management,
animation libraries). See **Forwarding refs and imperative handle**
for when this is the right tool.

## Callback refs

Refs can be functions instead of objects:

```jsx
function MeasureBox() {
  const [size, setSize] = useState(null);

  const ref = useCallback((node) => {
    if (node) {
      setSize({ width: node.offsetWidth, height: node.offsetHeight });
    }
  }, []);

  return <div ref={ref}>...</div>;
}
```

The function runs when the node is attached (passed the node) and
when detached (passed `null`). Useful when you need to react to the
node being available — not just have a stable reference to it.

## Common pitfalls

### Reading `ref.current` during render

```jsx
function Component() {
  const ref = useRef(null);

  return (
    <div>
      <input ref={ref} />
      {ref.current && <p>Value: {ref.current.value}</p>}
    </div>
  );
}
```

During the *first* render, `ref.current` is `null` — the DOM doesn't
exist yet. After mount, `ref.current` is set, but the component
won't re-render until something else triggers it.

If you need the value rendered, use state with `onChange`:

```jsx
const [value, setValue] = useState('');
<input value={value} onChange={(e) => setValue(e.target.value)} />
<p>Value: {value}</p>
```

### Trying to set state to a ref's value

```jsx
useEffect(() => {
  setRef(ref.current);   // ???
}, []);
```

If you want to "snapshot" the ref's value into state, use the value
directly, not the ref:

```jsx
function MyComponent() {
  const [el, setEl] = useState(null);
  return <div ref={setEl}>...</div>;
}
```

The callback ref form sets state when the node attaches. Now you
have both: the ref-equivalent (via `setEl`) and the state that
triggers re-renders.

### Using ref to track previous prop

A common need:

```jsx
function usePrevious(value) {
  const ref = useRef();
  useEffect(() => {
    ref.current = value;
  }, [value]);
  return ref.current;
}

function Component({ count }) {
  const prev = usePrevious(count);
  return <p>Now {count}, was {prev}</p>;
}
```

`useRef` + `useEffect` is the standard pattern for "previous value of
X." Library: `react-use` exports `usePrevious`.

### Reading `.current` immediately after setting

```jsx
ref.current = newValue;
console.log(ref.current);   // newValue
```

Works as expected — refs are synchronous. Unlike state setters, the
value is immediately available. This is why refs are useful for "I
need to communicate between renders without waiting for a
re-render."

### Treating refs as global state

Refs are scoped to the component instance. They don't share across
components without prop drilling or context. They're for local
imperative needs.

## When to reach for a ref

- DOM access (focus, scroll, measure).
- Storing timers, animation frame IDs, abort controllers.
- Tracking values that don't affect UI (last click time, render
  count).
- Caching expensive references (a memoized callback that needs to be
  the same across renders — though `useCallback` is more idiomatic).

## When NOT to

- Values that drive UI → use state.
- Derived values → compute during render.
- Cross-component sharing → state + context, or a state library.
- Global mutable singletons → that's not React's problem; module
  scope works.

## The mental model

`useRef` is a "drawer" attached to your component instance. It
persists across renders, holds a single value, and changing what's
in it doesn't trigger a render. Used right, refs are how you bridge
React's declarative model with imperative APIs (the DOM, third-party
libraries, timers).

The two questions to ask:

1. Does the value need to trigger a re-render when it changes?
   → State, not ref.
2. Is the value lifecycle-tied to the component but invisible to the
   user? → Ref, not module variable.
