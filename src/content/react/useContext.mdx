# `useContext`

Context lets a value flow from one component down to descendants
without passing through every layer of props. Used right, it
eliminates "prop drilling." Used wrong, it makes performance and
testing harder.

## The basic API

Three pieces:

1. **Create a context**:

```jsx
const ThemeContext = React.createContext('light');
```

2. **Provide a value** somewhere up the tree:

```jsx
<ThemeContext.Provider value="dark">
  <App />
</ThemeContext.Provider>
```

3. **Read it** anywhere below:

```jsx
function Button() {
  const theme = useContext(ThemeContext);
  return <button className={theme}>Click</button>;
}
```

Every component between provider and consumer can ignore the value.
The `Button` just receives it.

## When to use it

The clear wins:

- **Theme** (dark/light, palette).
- **Current user** (logged-in info, permissions).
- **Locale / i18n** settings.
- **Toast notifier / modal manager** (deeply nested code wants to
  trigger a UI action at the root).
- **Router state** (current URL, navigation function).

These are values that:

- Are "ambient" — used in many places.
- Don't change often.
- Don't need to be aware of each consumer specifically.

## When NOT to use it

- **Form state** — pass via props or use a form library.
- **Server data** — use a query library (TanStack Query). It's
  already a "global cache" with smarter invalidation.
- **Frequently-updating values** — every consumer re-renders on
  change. For high-frequency updates (mouse position, scroll),
  context is the wrong tool.
- **Global state in general** — context isn't a state library. Pair
  it with `useState`/`useReducer` if you need state-like behavior,
  but for complex global state, libraries like Zustand or Jotai are
  more efficient.

## The default value

```jsx
const Context = React.createContext('default');
```

Components that read the context **without a provider above them**
get the default. Two design philosophies:

**Philosophy A — meaningful default.** Set a real default that
consumers can safely use without a provider. Right when "no provider"
is a valid configuration:

```tsx
const ThemeContext = React.createContext<'light' | 'dark'>('light');
// Components rendered outside <ThemeProvider> still render fine in light mode.
```

**Philosophy B — null default + assert.** Treat "no provider" as a
bug. Use a custom hook that throws:

```tsx
const UserContext = React.createContext<User | null>(null);

export function useUser(): User {
  const user = useContext(UserContext);
  if (user === null) throw new Error('useUser must be inside <UserProvider>');
  return user;
}
```

This is the right choice for context that *requires* setup (current
user, router, theme that has no sensible fallback). Consumers get a
non-null type for free.

Pick A when "no provider" is acceptable (tests, storybook, partial
rendering). Pick B when missing the provider is always a bug —
which is most of the time.

## Pairing with state

A common pattern: context holds state + setters.

```jsx
const ThemeContext = React.createContext(null);

function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light');

  return (
    <ThemeContext.Provider value={{ theme, setTheme }}>
      {children}
    </ThemeContext.Provider>
  );
}

function useTheme() {
  return useContext(ThemeContext);
}

function Toggle() {
  const { theme, setTheme } = useTheme();
  return <button onClick={() => setTheme(theme === 'light' ? 'dark' : 'light')}>
    {theme}
  </button>;
}
```

`<ThemeProvider>` owns the state. Every consumer reads or updates
it. No prop drilling.

## The performance trap

```jsx
function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light');
  return (
    <ThemeContext.Provider value={{ theme, setTheme }}>
      {children}
    </ThemeContext.Provider>
  );
}
```

Every render of `ThemeProvider` creates a **new object** `{ theme,
setTheme }`. Every consumer re-renders because the value changed (by
reference).

The fix: memoize the value:

```jsx
const value = useMemo(() => ({ theme, setTheme }), [theme]);
```

Or destructure differently to avoid the object:

```jsx
// Two separate contexts — split read and write
<ThemeStateContext.Provider value={theme}>
  <ThemeDispatchContext.Provider value={setTheme}>
    {children}
  </ThemeDispatchContext.Provider>
</ThemeStateContext.Provider>
```

Two contexts: one for the value (changes when theme changes), one
for the setter (stable). Consumers subscribe only to what they need.

The single-context-with-object approach is common; understand the
re-render implication.

## Multiple contexts

Compose multiple providers:

```jsx
<ThemeProvider>
  <UserProvider>
    <I18nProvider>
      <App />
    </I18nProvider>
  </UserProvider>
</ThemeProvider>
```

Each is independent. A consumer can read any subset.

## Reading multiple contexts in one component

```jsx
function Profile() {
  const { theme } = useTheme();
  const { user } = useUser();
  const { t } = useI18n();
  return <div className={theme}>{t('hello')}, {user.name}</div>;
}
```

Multiple `useContext` calls (via custom hooks). Each subscribes
independently — changing one doesn't trigger re-render via the
others.

## Providing a value from a hook

You can build a provider purely from hooks:

```jsx
function useAuthValue() {
  const [user, setUser] = useState(null);
  const login = async (creds) => { /* ... */ setUser(...); };
  const logout = () => setUser(null);

  return { user, login, logout };
}

const AuthContext = React.createContext(null);
function AuthProvider({ children }) {
  const value = useAuthValue();
  return <AuthContext.Provider value={value}>{children}</AuthContext.Provider>;
}
```

The hook owns state and exposes methods. The provider just wires it.

## Common pitfalls

### New object every render

Covered above. Memoize or split into two contexts.

### Putting too much in one context

```jsx
const AppContext = React.createContext({ user, theme, cart, notifications, ... });
```

Any change to any field re-renders every consumer. Split into
focused contexts:

```jsx
<UserContext.Provider value={user}>
  <ThemeContext.Provider value={theme}>
    <CartContext.Provider value={cart}>
      ...
```

Each consumer subscribes to only what it needs.

### Default `null` without enforcement

```jsx
const Ctx = React.createContext(null);

function useCtx() {
  return useContext(Ctx);   // might be null!
}
```

Wrap in a custom hook that throws or returns a safe default:

```jsx
function useCtx() {
  const v = useContext(Ctx);
  if (!v) throw new Error('missing provider');
  return v;
}
```

### Using context for high-frequency state

```jsx
const MousePosContext = React.createContext({ x: 0, y: 0 });
// updates on every mousemove → every consumer re-renders
```

Don't. Use a state-management library that supports selectors
(Zustand, Jotai, Redux+useSelector). They subscribe to specific
slices and re-render only consumers of that slice.

## Context vs prop drilling

Prop drilling (passing props through layers that don't use them) is
the problem context solves. But:

- **2-3 layers** — prop drilling is usually fine. Adds a line per
  level, makes data flow explicit.
- **5+ layers** or "every screen uses this" — context wins. Less
  boilerplate, clearer intent.
- **Component-tree-wide state** — context is the bridge, but the
  storage might be a state library, query cache, or router.

Context isn't a replacement for state management — it's a
distribution mechanism.

## Context vs state libraries

| | Context | State library (Zustand, Redux) |
| --- | --- | --- |
| Built-in | Yes | No |
| Selectors / fine-grained | No | Yes |
| Devtools | No | Often yes |
| Persistence | Manual | Plugins |
| Async | Manual | Plugins (RTK Query) |

For simple shared state, context + useState is enough. For complex
apps with derived state, selectors, and devtools, a library scales
better.

Common modern stack: context for theme/auth/locale, a query library
for server data, Zustand or similar for UI state that needs
selectors.

## The mental model

Context is **dependency injection** at the React tree level. A
provider sets a value at a node; any descendant can read it.

The value lives wherever you store it — in `useState`, in module
scope, in a state library. Context just makes it reachable without
prop drilling.

Read sparingly. Each read is a subscription that re-renders the
consumer when the value changes. For performance-sensitive code,
limit context to truly ambient values and use selectors elsewhere.
