# What is React

React is a JavaScript library for building user interfaces by
composing **components**. You describe what the UI should look like
for a given state; React figures out how to update the DOM when the
state changes.

It's a library, not a framework — bring your own router, data
fetching, build tool, and state manager.

## The mental model

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      Clicked {count} times
    </button>
  );
}
```

Three concepts in nine lines:

1. **Component** — `Counter` is a function. It returns markup
   describing what should be on screen.
2. **State** — `useState(0)` makes the component remember a number
   between renders.
3. **Re-render** — `setCount` schedules a new render. React calls
   `Counter()` again, gets new markup, and updates only what
   changed in the DOM.

You write code that says "for this state, the UI looks like X."
React handles the bookkeeping of going from one X to the next.

## The trade

React's promise: **declarative UI**. Instead of writing imperative
DOM commands ("when the user clicks, create a new `<li>` and append
it"), you describe the desired state ("the list is items"). React
diffs the new description against the previous and applies the
minimal updates.

The cost:

- A virtual DOM step that does the diffing.
- A mental shift toward "state owns the truth; UI is a function of
  state."
- A learning curve around hooks, effects, and re-renders.

For complex UIs the trade is usually a clear win. For static
content it's overkill — a Markdown renderer is faster.

## Components

A React component is **a function that returns JSX**. Anywhere it
can be used as `<MyComponent />`.

```jsx
function Welcome({ name }) {
  return <h1>Hello, {name}</h1>;
}

function App() {
  return (
    <>
      <Welcome name="Jon" />
      <Welcome name="Jen" />
    </>
  );
}
```

The `<></>` is a "fragment" — React requires components to return
exactly one root, and a fragment lets you have multiple siblings
without an extra wrapping `<div>`.

Components compose freely. Each one owns its state, props, and
markup. Apps are trees of components: `<App>` containing `<Header>`,
`<Sidebar>`, `<Main>`, each containing more.

## Props vs state

**Props** are data passed *into* a component from its parent. They're
read-only. The component renders based on them.

**State** is data the component *owns*. It can change over time and
re-renders the component when it does.

```jsx
function User({ name, role }) {       // props
  const [online, setOnline] = useState(false);   // state
  return <div>{name} ({role}) — {online ? 'online' : 'offline'}</div>;
}
```

The parent decides `name` and `role`. The component itself tracks
`online`. Parents can pass callbacks (`onChange`) to let children
notify them of state changes, but the data flow is one-way:
parent → child via props, child → parent via callbacks.

## React doesn't ship a router or data fetcher

You'll add libraries for what most frameworks bundle:

- **Routing**: `react-router`, `tanstack-router`, `wouter`.
- **Data fetching / caching**: `tanstack-query`, `swr`, `apollo`.
- **State management**: `zustand`, `redux-toolkit`, `jotai`, or
  just `useState` + `useContext` for most needs.
- **Forms**: `react-hook-form`, `formik`.

This is the deal: React focuses on rendering. Everything else is
your choice.

## The big React frameworks

Modern React work often happens inside a higher-level framework:

- **Next.js** — file-system routing, server components, server-side
  rendering, image optimization, deployment.
- **Remix / React Router v7** — routing-first, data loaders/actions,
  progressive enhancement.
- **TanStack Start** — type-safe router-first framework.
- **Vite + React Router** — bring-your-own stack; lightest setup.

For learning, plain Vite + React Router is the cleanest. For
production apps, a framework is usually the right pick.

## When you'd use React

- **Interactive UIs** that update based on state — dashboards, forms,
  editors, real-time apps.
- **Component libraries** — shadcn/ui, MUI, Mantine — let you build
  consistent UIs by composition.
- **Cross-platform apps** — React Native uses the same model for
  iOS/Android.
- **Inside an existing app** — you can drop React into one section
  of a non-React app.

## When you wouldn't

- **Static content** — Markdown, blog, marketing — use a static
  generator (Astro, 11ty) and skip JS where possible.
- **Heavy DOM manipulation outside the React tree** — games,
  CAD-like editors. React fights you when you need direct,
  fine-grained DOM control.
- **No interactivity** — you don't need React for a contact form
  that submits to a server.

## The shift from class to function components

Old React had `class Component extends React.Component` with
`componentDidMount`, `componentDidUpdate`, `this.state`, etc.
Modern React is function components plus hooks (`useState`,
`useEffect`, etc.). The function form is shorter, easier to share
logic between components, and the only style new code uses.

You'll see class components in old codebases. New code: functions
only.

## What "hook" means

A hook is a function whose name starts with `use` that lets a
component "hook into" React's features. `useState` for state,
`useEffect` for side effects, `useContext` for shared values,
custom hooks (`useUser`, `useFetch`) for reusable logic.

Two rules:

1. Only call hooks at the top level of a component (or another
   hook). Not inside conditions, loops, or after early returns.
2. Only call hooks from React components or from other hooks. Not
   from regular functions.

These rules let React track hook calls by their *order*, which is
how the same `useState` returns the same state across renders.

## The mental shift

Coming from imperative DOM ("when X happens, change Y"), React asks
you to think:

- "What is the state of my UI?" — enumerate all the pieces of
  data.
- "What does the UI look like for each state?" — write components
  that render those states.
- "How does the state change?" — define events that update state.

Get those three right and the rendering happens automatically. The
"bug where the count says 5 but the badge says 4" disappears
because both come from the same source.
