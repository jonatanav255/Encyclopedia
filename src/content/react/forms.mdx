# Forms

Forms in React come in two flavors: **controlled** (React owns the
input value via state) and **uncontrolled** (the DOM owns it).
Controlled is the default for most cases; uncontrolled has its place.

See **Controlled vs uncontrolled** for the deeper comparison; this
page covers the everyday patterns.

## A simple controlled form

```jsx
function ContactForm() {
  const [name, setName] = useState('');
  const [email, setEmail] = useState('');

  function handleSubmit(e) {
    e.preventDefault();
    submit({ name, email });
  }

  return (
    <form onSubmit={handleSubmit}>
      <label>
        Name
        <input
          type="text"
          value={name}
          onChange={(e) => setName(e.target.value)}
        />
      </label>
      <label>
        Email
        <input
          type="email"
          value={email}
          onChange={(e) => setEmail(e.target.value)}
        />
      </label>
      <button type="submit">Submit</button>
    </form>
  );
}
```

The input is "controlled":

- `value={name}` — React state drives the input.
- `onChange` — every keystroke updates state.

The two are kept in sync. The state is the source of truth.

## `preventDefault` on submit

```jsx
function handleSubmit(e) {
  e.preventDefault();
  // ...
}
```

Without `preventDefault`, the browser submits the form and reloads
the page — almost never what you want in a React app.

`<form onSubmit>` is preferred over a button's `onClick` because it
also handles "Enter to submit" naturally. Pressing Enter in any
input fires submit; you don't have to wire it up.

## Different input types

```jsx
<input type="text" value={text} onChange={(e) => setText(e.target.value)} />
<input type="number" value={num} onChange={(e) => setNum(+e.target.value)} />
<input type="checkbox" checked={agreed} onChange={(e) => setAgreed(e.target.checked)} />
<input type="radio" name="size" value="s" checked={size === 's'} onChange={() => setSize('s')} />
<textarea value={body} onChange={(e) => setBody(e.target.value)} />
<select value={role} onChange={(e) => setRole(e.target.value)}>
  <option value="admin">Admin</option>
  <option value="user">User</option>
</select>
```

Notice:

- Checkboxes use `checked`, not `value`.
- Number inputs return strings — `+e.target.value` to get a number.
- `<textarea>` in React uses `value`, unlike HTML where the content is
  between the tags.
- `<select>` uses `value` on the select itself, not `selected` on the
  option.

## File inputs

File inputs are uncontrolled — you can't set their value
programmatically (security reason):

```jsx
function Upload() {
  const fileRef = useRef(null);

  function handleSubmit(e) {
    e.preventDefault();
    const file = fileRef.current.files[0];
    upload(file);
  }

  return (
    <form onSubmit={handleSubmit}>
      <input type="file" ref={fileRef} />
      <button type="submit">Upload</button>
    </form>
  );
}
```

Or use `onChange` to capture the file when it's selected:

```jsx
const [file, setFile] = useState(null);

<input type="file" onChange={(e) => setFile(e.target.files[0])} />
```

## Validation

Browser-native validation works:

```jsx
<input type="email" required />
<input type="number" min="0" max="100" />
<input pattern="\d{5}" />
```

The form won't submit if validation fails. The downside: limited
customization of error UI.

For richer validation, do it yourself:

```jsx
function ContactForm() {
  const [email, setEmail] = useState('');
  const [error, setError] = useState('');

  function handleSubmit(e) {
    e.preventDefault();
    if (!email.includes('@')) {
      setError('Invalid email');
      return;
    }
    submit(email);
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        type="email"
        value={email}
        onChange={(e) => { setEmail(e.target.value); setError(''); }}
      />
      {error && <span className="error">{error}</span>}
      <button type="submit">Submit</button>
    </form>
  );
}
```

For real forms with many fields, use a library.

## Form libraries

`useState` per field is fine for 2-3 fields. Beyond that:

- **`react-hook-form`** — lightweight, uses uncontrolled inputs +
  refs, very fast. Pairs well with Zod for validation.
- **`formik`** — established, controlled approach, slightly heavier.
- **`zod` + react-hook-form** — runtime validation + types from
  schema, single source of truth.

```jsx
import { useForm } from 'react-hook-form';
import { z } from 'zod';
import { zodResolver } from '@hookform/resolvers/zod';

const Schema = z.object({
  email: z.string().email(),
  age: z.number().min(18),
});

function MyForm() {
  const { register, handleSubmit, formState: { errors } } = useForm({
    resolver: zodResolver(Schema),
  });

  return (
    <form onSubmit={handleSubmit(submit)}>
      <input {...register('email')} />
      {errors.email && <span>{errors.email.message}</span>}

      <input type="number" {...register('age', { valueAsNumber: true })} />
      {errors.age && <span>{errors.age.message}</span>}

      <button type="submit">Submit</button>
    </form>
  );
}
```

`react-hook-form` doesn't re-render on every keystroke (it uses refs
internally), so large forms stay snappy.

## Submitting

```jsx
async function handleSubmit(e) {
  e.preventDefault();
  setSubmitting(true);
  try {
    await api.save(data);
    setSuccess(true);
  } catch (err) {
    setError(err.message);
  } finally {
    setSubmitting(false);
  }
}
```

While the request is in flight, disable the submit button and show a
spinner:

```jsx
<button type="submit" disabled={submitting}>
  {submitting ? 'Saving…' : 'Save'}
</button>
```

This is essential — without it, users double-submit on slow networks.

## Resetting

```jsx
function Form() {
  const [data, setData] = useState({ name: '', email: '' });

  function handleSubmit(e) {
    e.preventDefault();
    submit(data);
    setData({ name: '', email: '' });    // manual reset
  }

  return ...;
}
```

Or use the form's native reset:

```jsx
<form onSubmit={handleSubmit} onReset={() => setData(initial)}>
  ...
  <button type="reset">Reset</button>
</form>
```

For the `key` trick (unmount/remount to reset):

```jsx
{showForm && <Form key={formInstance} />}
```

Incrementing `formInstance` mounts a fresh `<Form>` with initial
state.

## Common pitfalls

### Mutating state directly

```jsx
const [user, setUser] = useState({ name: '', email: '' });

function handleNameChange(e) {
  user.name = e.target.value;   // mutates state — React doesn't re-render
  setUser(user);                 // same reference — no change detected
}
```

Always create a new object:

```jsx
setUser({ ...user, name: e.target.value });
```

Or split into separate states.

### Forgetting `preventDefault`

The form submits to the server (or reloads). Always
`e.preventDefault()` unless you actually want the browser to handle
it.

### Putting submit logic in the button's `onClick`

```jsx
<button onClick={handleSubmit}>Submit</button>   // bad — bypasses form
<form onSubmit={handleSubmit}><button type="submit">Submit</button></form>   // good
```

`onSubmit` is fired by Enter-in-input, by clicking submit, and
respects HTML form validation. `onClick` doesn't.

### Loose coupling between field and label

```jsx
<label>Name</label>
<input />
```

Without association, screen readers don't connect them. Either nest:

```jsx
<label>
  Name
  <input />
</label>
```

Or use `htmlFor`:

```jsx
<label htmlFor="name">Name</label>
<input id="name" />
```

### Storing too much in state

If you have 30 fields, 30 `useState` calls is repetitive. One object
in state can collapse them, but updates need spread:

```jsx
const [form, setForm] = useState({ name: '', email: '', ... });

function update(field, value) {
  setForm((prev) => ({ ...prev, [field]: value }));
}

<input value={form.name} onChange={(e) => update('name', e.target.value)} />
```

For real forms, reach for `react-hook-form` — it handles this
cleanly.

## Accessibility

- **Always** associate labels with inputs (`htmlFor`/`id` or
  nesting).
- Use semantic input types (`type="email"`, `type="tel"`) — mobile
  keyboards adapt.
- Show validation errors after submit attempt, not on every
  keystroke (avoids "you haven't finished typing yet" anxiety).
- Use `aria-invalid={!!error}` on inputs with errors.
- Use `aria-describedby` to point to error messages.

```jsx
<input
  id="email"
  aria-invalid={!!errors.email}
  aria-describedby="email-error"
/>
{errors.email && <span id="email-error">{errors.email}</span>}
```

## The mental model

A form's state lives somewhere — either in React state (controlled)
or in the DOM (uncontrolled). For controlled, every input has
`value` and `onChange`. For uncontrolled, you read values via refs
or `FormData` at submit time.

Most forms benefit from controlled inputs for instant validation and
conditional behavior. Heavy forms benefit from uncontrolled (less
re-rendering) — `react-hook-form` is the standard tool.

## The discipline

- Controlled inputs for everyday cases.
- `react-hook-form` (or similar) when forms get big.
- Always `preventDefault` on submit.
- Validate, show errors, prevent double-submission.
- Disable the submit button while in-flight.
- Associate labels with inputs.
