# Controlled vs uncontrolled

Form inputs in React come in two modes. **Controlled**: React state
drives the input. **Uncontrolled**: the DOM drives it; you read the
value when you need it.

Most React tutorials show controlled. Both have their place.

## Controlled

```jsx
function ControlledInput() {
  const [value, setValue] = useState('');

  return (
    <input
      value={value}
      onChange={(e) => setValue(e.target.value)}
    />
  );
}
```

The deal:

- `value={value}` tells React what's in the input.
- `onChange` updates state on every keystroke.
- The two are kept in sync.

You can now read, validate, transform, or restrict the value:

```jsx
<input
  value={value}
  onChange={(e) => setValue(e.target.value.toUpperCase())}
/>
```

The input shows the uppercased version. Try to type a lowercase
letter — React replaces it instantly.

## Uncontrolled

```jsx
function UncontrolledInput() {
  const inputRef = useRef(null);

  function handleSubmit() {
    alert(inputRef.current.value);
  }

  return (
    <>
      <input ref={inputRef} defaultValue="initial" />
      <button onClick={handleSubmit}>Submit</button>
    </>
  );
}
```

- `defaultValue` (not `value`) sets the initial value.
- No `onChange` — the DOM tracks the value internally.
- Read it via the ref when needed.

The state is in the DOM. React doesn't track it on every keystroke.

## When to use each

**Controlled** when:

- You need to react to every change (live validation, character
  count, conditional UI).
- You want to format or restrict input (capitalize, strip non-digits).
- The value drives other parts of the UI.
- You're rendering the input based on external state.

**Uncontrolled** when:

- You only care about the value at submit time.
- The form is large and re-renders would be wasteful.
- You're integrating with a non-React library that owns the input.
- You're working with file inputs (which are always uncontrolled).

In modern React, controlled is the default; uncontrolled is the
escape hatch. Libraries like `react-hook-form` lean uncontrolled
internally for performance — re-renders only when needed.

## File inputs are always uncontrolled

```jsx
<input
  type="file"
  value=""              // can't set this — security restriction
  onChange={...}        // works as normal
/>
```

You can't programmatically set a file input's value. Use a ref to
read it, or capture the file in `onChange`.

```jsx
const fileRef = useRef(null);

<input type="file" ref={fileRef} />
<button onClick={() => upload(fileRef.current.files[0])}>Upload</button>
```

## Common gotchas

### Mixing `value` and `defaultValue`

```jsx
<input value="" defaultValue="hi" />   // confused — value wins
```

`value` makes the input controlled. `defaultValue` sets initial for
uncontrolled. Don't pass both.

React will warn if you switch between them — going from controlled
to uncontrolled (or vice versa) loses state and produces console
warnings.

### `value={undefined}` is uncontrolled

```jsx
<input value={someVar} onChange={...} />
```

If `someVar` is `undefined` on first render, React treats the input
as uncontrolled, then "switches" to controlled when `someVar`
becomes defined. Warning in console.

Fix: ensure `value` is always a string (`?? ''`):

```jsx
<input value={someVar ?? ''} onChange={...} />
```

### `null` for `value`

Same issue. `null` makes the input uncontrolled. Coerce to `''`.

### Re-rendering the whole form on every keystroke

```jsx
function BigForm() {
  const [a, setA] = useState('');
  const [b, setB] = useState('');
  const [c, setC] = useState('');
  // 20 more fields...

  return (
    <>
      <input value={a} onChange={(e) => setA(e.target.value)} />
      <input value={b} onChange={(e) => setB(e.target.value)} />
      ...
    </>
  );
}
```

Each keystroke re-renders the entire form. For huge forms with
expensive children, this adds up. Two options:

1. Use uncontrolled inputs + refs.
2. Use `react-hook-form` — uncontrolled internally, surfaces only
   what changed.

For 10-20 simple inputs this isn't a real problem. For 100+ or with
heavy children, it can be.

## Controlled with `react-hook-form`

```jsx
import { useForm, Controller } from 'react-hook-form';

function Form() {
  const { control, handleSubmit } = useForm();

  return (
    <form onSubmit={handleSubmit(submit)}>
      <Controller
        name="email"
        control={control}
        render={({ field }) => <input {...field} />}
      />
    </form>
  );
}
```

For most inputs, RHF's `register` (uncontrolled) is enough. Use
`Controller` when you need controlled behavior (e.g., a third-party
component that requires `value`/`onChange`).

## "Don't fight the DOM" patterns

### `<select>` defaultValue

```jsx
<select defaultValue="medium">
  <option value="small">Small</option>
  <option value="medium">Medium</option>
  <option value="large">Large</option>
</select>
```

Cleaner than setting `selected` on each option. Same goes for
`<textarea defaultValue="..." />`.

### `<input type="checkbox">`

```jsx
// Controlled
<input type="checkbox" checked={agreed} onChange={(e) => setAgreed(e.target.checked)} />

// Uncontrolled
<input type="checkbox" defaultChecked />
```

Note: `checked`/`defaultChecked`, not `value`/`defaultValue`.

## The pattern in practice

For small forms (login, contact, signup):

```jsx
function LoginForm() {
  const [email, setEmail] = useState('');
  const [password, setPassword] = useState('');
  // controlled inputs, validate on submit
}
```

For big forms or admin panels:

```jsx
import { useForm } from 'react-hook-form';

function ProfileForm() {
  const { register, handleSubmit } = useForm();
  return (
    <form onSubmit={handleSubmit(submit)}>
      <input {...register('name')} />
      <input {...register('email')} />
      {/* 20 more — uncontrolled, fast */}
    </form>
  );
}
```

For React Server Components or progressive enhancement, uncontrolled
+ `<form action>` is increasingly idiomatic — works without JS,
React handles the rest.

## The mental model

Controlled = "React owns the value." Every change goes through
state. You can read, modify, derive, validate live.

Uncontrolled = "the DOM owns the value." You opt into reading it
when you need to. Less reactive, less overhead.

For interactive UIs that respond to every change, controlled. For
fire-and-forget forms, uncontrolled. For hybrid, libraries like
`react-hook-form` give you the best of both — uncontrolled defaults
with controlled escape hatches.
