# Fragments and portals

Two tools for shaping the rendered DOM independently of the React
tree.

## Fragments

A React component must return exactly one root element. When you
have siblings, wrap them in a fragment:

```jsx
function HeaderRow() {
  return (
    <>
      <th>Name</th>
      <th>Age</th>
      <th>Email</th>
    </>
  );
}
```

`<>...</>` is the shorthand. The full form:

```jsx
import { Fragment } from 'react';

<Fragment>
  <th>Name</th>
  <th>Age</th>
</Fragment>
```

Fragments emit no DOM. The browser sees just the three `<th>`
elements — no wrapping `<div>`.

## Why not just a `<div>`?

```jsx
return (
  <div>
    <th>Name</th>
    <th>Age</th>
  </div>
);
```

A `<div>` inside `<tr>` is invalid HTML — table semantics require
direct `<th>`/`<td>` children. Wrapping for React's "one root" rule
would break the layout.

Same with `<ul>` (which requires `<li>` children), list groupings,
and CSS Grid layouts. Fragments let you respect the parent's
expected children without React's element wrapping.

## Keys on fragments

The shorthand `<>...</>` can't take keys or other attributes. When
you need a key in a list, use the long form:

```jsx
{items.map((item) => (
  <Fragment key={item.id}>
    <dt>{item.label}</dt>
    <dd>{item.value}</dd>
  </Fragment>
))}
```

This is the most common reason to write `<Fragment>` instead of the
shorthand.

## Portals

A portal renders children into a **DOM node outside the parent
component's DOM hierarchy**, while keeping them in React's
component tree.

```jsx
import { createPortal } from 'react-dom';

function Modal({ children, isOpen }) {
  if (!isOpen) return null;

  return createPortal(
    <div className="modal-overlay">
      <div className="modal-content">{children}</div>
    </div>,
    document.body
  );
}
```

The modal's JSX is rendered as a child of `<body>`, not of the
component that called `<Modal>`. But it's still part of the React
tree for context, events, and re-renders.

## Why portals exist

CSS escapes. Some UI needs to break out of the parent's DOM
positioning:

- **Modals / dialogs** — should overlay everything, not respect a
  parent's `overflow: hidden`.
- **Tooltips** — should escape `overflow: hidden` clipping.
- **Dropdowns / popovers** — same reason.
- **Toast notifications** — should appear at viewport edge regardless
  of which component triggered them.

Without portals, your modal lives wherever you called it from in the
tree. CSS like `overflow: hidden`, `transform`, `z-index` on
ancestors can clip or stack badly. Portaling to `body` (or a
dedicated `<div id="modal-root">`) sidesteps all that.

## Events still propagate

The cleverest part of portals: even though the DOM child is
elsewhere, **React events bubble through the component tree**, not
the DOM tree.

```jsx
function Parent() {
  return (
    <div onClick={() => console.log('parent click')}>
      <Modal isOpen>
        <button>Click me</button>
      </Modal>
    </div>
  );
}
```

When you click the button (which lives in `<body>` via portal), the
parent's `onClick` fires. React events follow the JSX, not the DOM.

This is what makes portals usable — you can portal a tooltip out of
the parent's clipping context but still rely on context, focus
management, and event delegation as if it lived inside.

## Common portal targets

```jsx
// Most apps
createPortal(<Modal />, document.getElementById('modal-root'))
```

```html
<!-- index.html -->
<body>
  <div id="root"></div>
  <div id="modal-root"></div>
  <div id="tooltip-root"></div>
</body>
```

Dedicated portal targets keep the DOM tidy and let CSS target them
specifically (`#modal-root { ... }`).

## Portal pitfalls

### SSR

Portals render to a DOM node that doesn't exist during server-side
rendering (`document.body` is undefined). The component should
guard:

```jsx
function Modal({ children, isOpen }) {
  if (!isOpen) return null;
  if (typeof document === 'undefined') return null;
  return createPortal(...content, document.body);
}
```

For Next.js / Remix, you may need to render the modal's contents
server-side but skip the portal call. Frameworks handle this in
various ways; consult their docs.

### Focus management

When a modal opens via portal, focus should move into it (typically
to the first focusable element). When it closes, focus should
return to whatever triggered it.

The native `<dialog>` element handles this for you — React 19+ has
better integration with it. For pre-`<dialog>` code, libraries like
`focus-trap-react` or Radix's Dialog handle the tricky details
(focus return, Escape to close, click-outside).

### CSS isolation

Since portal contents live outside the parent's DOM, parent CSS
won't reach them. Use a CSS class or scoped styles on the portal
content directly.

### z-index wars

Portals don't automatically solve stacking. If two portals coexist
(modal + dropdown menu, say), they're both in `document.body` and
need explicit z-index ordering. Common pattern: a "stack" library
or shared z-index constants.

## When NOT to use portals

- **Anything that fits in the parent's flow.** Don't portal a
  component just because — it adds complexity (SSR, focus, styling).
- **Toasts that aren't full-viewport.** A fixed-positioned div might
  be enough.
- **Components that don't escape clipping.** Portals are overkill if
  the parent's `overflow` rules don't matter.

The litmus test: "would this element be broken by an ancestor's
CSS?" If yes, portal. If no, leave it inline.

## React 19 and `<dialog>`

React 19 plays better with the native `<dialog>` element:

```jsx
function Modal({ open, children }) {
  const ref = useRef(null);

  useEffect(() => {
    if (open) ref.current.showModal();
    else ref.current.close();
  }, [open]);

  return <dialog ref={ref}>{children}</dialog>;
}
```

`<dialog>` is portal-like by default — it overlays via the browser's
top layer (above all other content, no z-index needed). Comes with
focus management, Escape handling, and click-outside semantics. The
right primitive for modals in modern browsers.

You still might wrap it in `createPortal` if you need control over
where in the DOM tree it lives, but for "I want a modal" the
`<dialog>` element alone is increasingly the answer.

## The mental model

**Fragment** = "I need to return multiple children but don't want a
wrapper element." Emits no DOM.

**Portal** = "I want this child rendered elsewhere in the DOM,
while keeping it part of my React tree." Emits DOM in a different
place than the JSX suggests.

Fragments are everyday. Portals show up in libraries (modals,
tooltips, dropdowns) and in your own code when you need to escape
ancestor CSS. Most of the time you can use a library's primitives
(Radix, shadcn, Headless UI) rather than calling `createPortal`
directly.
