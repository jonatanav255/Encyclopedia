# `useReducer`

`useReducer` is `useState`'s big sibling. Instead of "set this value
to that," you describe state changes as **actions** and a **reducer**
function that applies them.

```jsx
function reducer(state, action) {
  switch (action.type) {
    case 'INCREMENT': return { count: state.count + 1 };
    case 'RESET':     return { count: 0 };
    default:           return state;
  }
}

function Counter() {
  const [state, dispatch] = useReducer(reducer, { count: 0 });

  return (
    <>
      <p>{state.count}</p>
      <button onClick={() => dispatch({ type: 'INCREMENT' })}>+</button>
      <button onClick={() => dispatch({ type: 'RESET' })}>Reset</button>
    </>
  );
}
```

Two things to provide: a reducer function and the initial state.
`dispatch(action)` sends an action; React calls `reducer(currentState,
action)` to compute the next state.

## Why use it over `useState`

`useState` is simpler for one value. `useReducer` shines when:

- **Multiple state fields update together.** Reducers consolidate
  the logic.
- **Updates depend on other state.** The reducer receives the
  current state, so no stale-closure problems.
- **Many ways to update.** A switch statement scales better than 10
  setters.
- **Updates are testable in isolation.** The reducer is a pure
  function — easy to unit-test.
- **You want to log actions** (for debugging, analytics, time-travel
  debugging).

## A more realistic example

```jsx
const initialState = {
  items: [],
  loading: false,
  error: null,
};

function reducer(state, action) {
  switch (action.type) {
    case 'FETCH_START':
      return { ...state, loading: true, error: null };
    case 'FETCH_SUCCESS':
      return { ...state, loading: false, items: action.payload };
    case 'FETCH_ERROR':
      return { ...state, loading: false, error: action.error };
    case 'ADD_ITEM':
      return { ...state, items: [...state.items, action.payload] };
    case 'REMOVE_ITEM':
      return { ...state, items: state.items.filter((i) => i.id !== action.id) };
    case 'CLEAR':
      return initialState;
    default:
      return state;
  }
}

function TodoList() {
  const [state, dispatch] = useReducer(reducer, initialState);

  async function loadTodos() {
    dispatch({ type: 'FETCH_START' });
    try {
      const items = await api.getTodos();
      dispatch({ type: 'FETCH_SUCCESS', payload: items });
    } catch (err) {
      dispatch({ type: 'FETCH_ERROR', error: err.message });
    }
  }

  return (
    <>
      <button onClick={loadTodos}>Load</button>
      {state.loading && <p>Loading…</p>}
      {state.error && <p>Error: {state.error}</p>}
      <ul>{state.items.map((i) => <li key={i.id}>{i.text}</li>)}</ul>
    </>
  );
}
```

The reducer captures all valid state transitions. The component just
dispatches events.

## Type-safe reducers (TypeScript)

```tsx
type State = { count: number };

type Action =
  | { type: 'INCREMENT' }
  | { type: 'DECREMENT' }
  | { type: 'SET'; value: number };

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case 'INCREMENT': return { count: state.count + 1 };
    case 'DECREMENT': return { count: state.count - 1 };
    case 'SET':       return { count: action.value };
    default:           return state;
  }
}
```

The discriminated union (`type: 'INCREMENT' | 'DECREMENT' | 'SET'`)
plus a `switch` gives you exhaustiveness checking — add a case to
the union and forget to handle it, the compiler complains.

## Lazy initialization

For expensive initial state:

```jsx
function init(initialCount) {
  return { count: initialCount, history: [] };
}

const [state, dispatch] = useReducer(reducer, 5, init);
```

The third argument is an init function. It's called once with the
second argument to produce the initial state. Same idea as
`useState(() => expensive())`.

## Dispatch is stable

```jsx
const [state, dispatch] = useReducer(reducer, initial);

useEffect(() => {
  // dispatch never changes between renders — safe to omit from deps
  fetchAndDispatch(dispatch);
}, []);
```

React guarantees `dispatch` has stable identity. You can leave it
out of effect dependencies, pass it down without `useCallback`, and
not worry about re-creating it.

## Combining with context

`useReducer` + context is the lightweight Redux replacement:

```jsx
const StateContext = React.createContext(null);
const DispatchContext = React.createContext(null);

function MyProvider({ children }) {
  const [state, dispatch] = useReducer(reducer, initialState);

  return (
    <StateContext.Provider value={state}>
      <DispatchContext.Provider value={dispatch}>
        {children}
      </DispatchContext.Provider>
    </StateContext.Provider>
  );
}

const useMyState = () => useContext(StateContext);
const useMyDispatch = () => useContext(DispatchContext);
```

Two contexts — readers of state re-render on state change; readers
of dispatch don't (dispatch is stable).

```jsx
function SomeComponent() {
  const dispatch = useMyDispatch();
  return <button onClick={() => dispatch({ type: 'CLICK' })}>Click</button>;
}
```

This component never re-renders on state changes — it only uses
dispatch.

## When to graduate to Redux / Zustand / Jotai

`useReducer` + context covers a lot. You'd graduate when:

- **Selectors** matter — you want a component to re-render only when
  a specific slice changes. Context is all-or-nothing per provider.
- **Devtools** matter — Redux Devtools is *very* good (time travel,
  action log, replay).
- **Middleware** matters — thunks, sagas, analytics, persistence.
- **Cross-store coordination** is complex.

Most apps don't reach this threshold. `useReducer` + context plus a
query library covers a startup-to-mid-size app.

## Side effects in a reducer (don't)

The reducer must be **pure**. No fetches, no `console.log`, no
random numbers, no Date.now, no mutations:

```jsx
function reducer(state, action) {
  switch (action.type) {
    case 'INCREMENT':
      console.log('incrementing');   // BAD — side effect
      return { count: state.count + 1 };
  }
}
```

Side effects belong in components (event handlers, effects) or
async middleware (in Redux). React relies on the reducer being pure
— it may call it twice in strict mode, and impure reducers break
that assumption.

## Common pitfalls

### Mutating state in the reducer

```jsx
case 'INCREMENT':
  state.count++;        // mutation!
  return state;          // same reference — React doesn't see change
```

Always return a new object:

```jsx
return { ...state, count: state.count + 1 };
```

### Action without a `type`

```jsx
dispatch({ count: 5 });   // no type — unhandled
```

The reducer's `switch` falls through to `default` and returns
unchanged state. Bugs in production. Use TypeScript discriminated
unions to enforce.

### Dispatching from a useEffect with stale deps

```jsx
useEffect(() => {
  dispatch({ type: 'SET', value: count });
}, []);
```

`count` is captured at mount. Use the dep array correctly:

```jsx
useEffect(() => {
  dispatch({ type: 'SET', value: count });
}, [count]);
```

### Forgetting `...state`

```jsx
case 'SET_NAME':
  return { name: action.name };   // BAD — drops every other field
```

Always spread:

```jsx
return { ...state, name: action.name };
```

Or use Immer for non-trivial nesting:

```jsx
import { produce } from 'immer';

const reducer = produce((draft, action) => {
  switch (action.type) {
    case 'SET_NAME':
      draft.name = action.name;   // looks mutating, isn't
      break;
  }
});
```

## When NOT to use it

- Single value → `useState`.
- Three-or-four toggles → `useState` × 4.
- Server data → query library.
- Simple form → `useState` per field or `react-hook-form`.

## The mental model

`useReducer` is **a state machine**. The state has shape; actions
trigger transitions; the reducer is the transition function. This
buys you:

- **Predictable transitions** — everything in one place.
- **Testable logic** — pure functions are easy to test.
- **Explicit intent** — "the user clicked" → dispatch an action
  named after the click, not a setter.

It's heavier than `useState` for simple cases but invaluable when
state gets complex. The line between "use state" and "use reducer"
is fuzzy; reach for the reducer when state changes start to feel
hard to follow.
