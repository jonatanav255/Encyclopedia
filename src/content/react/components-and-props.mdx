# Components and props

A React component is a function that returns JSX. Props are the
arguments — the data flowing into the component from its parent.

```jsx
function Welcome({ name, role = 'user' }) {
  return <p>Hello, {name} ({role})</p>;
}

<Welcome name="Jon" role="admin" />
<Welcome name="Jen" />
```

The two key rules:

1. **Components are pure**. For the same props (and state), they
   return the same JSX. No side effects in the body.
2. **Props are read-only**. Don't mutate them. If you need to change
   data, lift it to state.

## Naming

Components start with a **capital letter**:

```jsx
function Button() {}    // a component
function button() {}    // NOT a component — JSX treats <button /> as HTML
```

The convention is one component per file for non-trivial ones,
named the same as the file: `Button.jsx` exports `Button`.

## Destructuring props

```jsx
// Verbose
function Welcome(props) {
  return <p>{props.name}</p>;
}

// Idiomatic
function Welcome({ name }) {
  return <p>{name}</p>;
}

// With defaults and rename
function Welcome({ name = 'world', greeting: g = 'hi' }) {
  return <p>{g}, {name}</p>;
}
```

Destructure in the parameter list. Defaults handle "prop not
provided" (`undefined`); they don't fire on `null`.

## Required vs optional props

```tsx
type Props = {
  name: string;          // required
  role?: string;          // optional
};
```

TypeScript makes the contract explicit. Without TS, conventions and
runtime checks (or `prop-types` for legacy projects) carry the
weight.

## `children`

The special prop that holds whatever was between the component's
tags:

```jsx
function Card({ children }) {
  return <div className="card">{children}</div>;
}

<Card>
  <h2>Title</h2>
  <p>Body</p>
</Card>
```

`children` is the foundation of composition. The component is a
container; the caller decides what goes inside.

```tsx
type Props = { children: React.ReactNode };
```

`React.ReactNode` accepts anything renderable: strings, numbers,
elements, fragments, arrays, null.

## Render props

When you want the parent to define how to render an item:

```jsx
function List({ items, renderItem }) {
  return <ul>{items.map(renderItem)}</ul>;
}

<List
  items={users}
  renderItem={(u) => <li key={u.id}>{u.name}</li>}
/>
```

`children` can also be a function — "function as a child":

```jsx
function Tooltip({ children }) {
  const [open, setOpen] = useState(false);
  return children({ open, setOpen });
}

<Tooltip>
  {({ open, setOpen }) => (
    <button onClick={() => setOpen(!open)}>{open ? 'hide' : 'show'}</button>
  )}
</Tooltip>
```

Less common in modern code (custom hooks usually do the same thing
more cleanly), but you'll see the pattern.

## Spread props

```jsx
<input {...inputProps} />
```

Spreads each key as an attribute. Useful for transparent wrappers:

```jsx
function Input({ label, ...rest }) {
  return (
    <label>
      {label}
      <input {...rest} />
    </label>
  );
}

<Input label="Name" type="text" value={name} onChange={setName} />
```

`rest` collects everything not destructured (`type`, `value`,
`onChange`) and forwards it. The wrapper adds a label without
caring about input-specific props.

## Passing through `ref`

By default, refs aren't passed via props. React 19 made refs a normal
prop:

```jsx
// React 19+
function Input({ ref, ...rest }) {
  return <input ref={ref} {...rest} />;
}
```

In React 18 and earlier, you needed `React.forwardRef`:

```jsx
const Input = React.forwardRef((props, ref) => (
  <input ref={ref} {...props} />
));
```

Same idea, slightly different syntax. Most React 19+ codebases use
the prop form.

## Composition over inheritance

React has no inheritance for components. To share behavior:

```jsx
// Bad pattern from class-based days
class FancyButton extends Button { ... }

// React pattern: composition
function FancyButton({ children, ...rest }) {
  return (
    <Button {...rest} className="fancy">
      <Sparkle />
      {children}
    </Button>
  );
}
```

Compose smaller pieces. For shared logic, use a custom hook (see
**Custom hooks**).

## Component identity matters

```jsx
function App() {
  return showA ? <ComponentA /> : <ComponentB />;
}
```

When `showA` flips, React unmounts A and mounts B — they're
different types, so state, refs, and effects start fresh.

```jsx
function App() {
  return <Component name={showA ? 'A' : 'B'} />;
}
```

Same component, different prop — state is preserved across the
change. This matters when you have inputs or animations inside that
shouldn't reset.

## Function vs class components

```jsx
// Function (modern, idiomatic)
function Greet({ name }) {
  return <p>Hi {name}</p>;
}

// Class (legacy, rare in new code)
class Greet extends React.Component {
  render() {
    return <p>Hi {this.props.name}</p>;
  }
}
```

Class components are still supported but not recommended for new
code. Hooks (state, effects, context) work only in function
components, so picking a function is also picking into the modern
API.

## Pure rendering

```jsx
function Bad({ items }) {
  items.sort();                // mutates the prop!
  return <ul>{items.map(...)}</ul>;
}
```

Components must not mutate their props or any data outside their
scope. React assumes a component called with the same props gives
the same output — mutation breaks that.

```jsx
function Good({ items }) {
  const sorted = [...items].sort();
  return <ul>{sorted.map(...)}</ul>;
}
```

Treat props as read-only.

## Strict mode and double-render

React's strict mode (`<React.StrictMode>` in your root) double-renders
components in development to surface impure code:

```jsx
function Counter() {
  let count = 0;        // BAD — module-scope mutation that persists
  count++;
  return <div>{count}</div>;
}
```

In strict mode, this component renders twice and shows different
values — a clue that something impure is happening. Don't disable
strict mode; fix the code.

## When to extract a component

A useful heuristic: when a piece of JSX has its own concept name, or
when you'd want to test it in isolation. Don't extract for trivial
markup ("a `<div>` with two children"). Do extract when there's
shared logic, repeated use, or a distinct UX role.

```jsx
// Probably extract
<Header />
<UserCard user={u} />
<PaymentForm onSubmit={...} />

// Probably don't
<Container>
  <Wrapper>
    <Inner>{children}</Inner>
  </Wrapper>
</Container>
```

The "right" size varies. Components should be readable in one
screenful; if they're not, split.

## The discipline

- Components return JSX given props (and state).
- Props are read-only.
- Components must be pure — no side effects in render.
- Use composition (children, props) over inheritance.
- Lift state up when siblings need to share it.
- Extract components for clarity, not for the sake of decomposition.
