# State and `useState`

State is data a component owns and can change over time. When state
changes, React re-renders the component to reflect the new value.

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(count + 1)}>
      Clicked {count} times
    </button>
  );
}
```

`useState(initial)` returns a pair: the current value and a
**setter**. Calling the setter schedules a re-render with the new
value.

## The flow

1. First render: `useState(0)` returns `[0, setCount]`. The button
   shows "Clicked 0 times."
2. User clicks. `setCount(1)` queues an update.
3. React re-renders. `useState(0)` now returns `[1, setCount]`. The
   button shows "Clicked 1 times."
4. The initial value (`0`) is only stored on the *first* render.
   Subsequent renders still evaluate the argument and throw the
   result away — if it's expensive (e.g., `loadFromStorage()`),
   reach for lazy init (see below).

State is preserved per *component instance* — two `<Counter />`
siblings each have their own count.

## Setter forms

Two ways to update:

```jsx
// Value form
setCount(count + 1);

// Functional form (reads the previous state)
setCount((prev) => prev + 1);
```

The functional form is safer when:

- Multiple updates happen in the same handler.
- You're inside an async function and `count` might be stale.

```jsx
function increment() {
  setCount(count + 1);   // count is stale if called multiple times
  setCount(count + 1);   // still uses the same stale count!
}

function increment() {
  setCount((c) => c + 1);  // each receives the latest queued value
  setCount((c) => c + 1);
}
```

The first version doesn't double-increment because `count` is
captured at handler creation. The functional form does.

## State is a snapshot

```jsx
function Form() {
  const [name, setName] = useState('');

  function handleSubmit() {
    setName('Jon');
    console.log(name);   // logs '' — the OLD value!
  }
}
```

Setters don't change `name` immediately. They schedule a re-render.
The current render's `name` variable is frozen for that render. The
new value only appears in the next render.

If you need the new value in the same handler, use the setter argument
or a local variable:

```jsx
function handleSubmit() {
  const newName = 'Jon';
  setName(newName);
  console.log(newName);    // 'Jon'
}
```

## Batched updates

React batches multiple state updates inside a single event handler:

```jsx
function handleClick() {
  setA(a + 1);
  setB(b + 1);
  setC(c + 1);
  // ONE re-render, with all three updates applied
}
```

Since React 18, batching extends to async handlers, promises, and
timeouts too:

```jsx
async function handleClick() {
  await fetchData();
  setA(...);
  setB(...);    // still batched in React 18+
}
```

You usually don't think about batching — it just works.

## Initializing from a computation

If your initial value is expensive to compute, pass a function:

```jsx
// Computed every render (wasteful)
const [items, setItems] = useState(loadFromStorage());

// Computed once
const [items, setItems] = useState(() => loadFromStorage());
```

`useState(fn)` calls `fn` only on the first render. After that, it's
ignored. The function form is "lazy initialization."

## Updating objects and arrays

State is treated as immutable — you replace it, not mutate it.

```jsx
const [user, setUser] = useState({ name: 'Jon', age: 30 });

// Wrong — React doesn't see the change
user.age = 31;
setUser(user);

// Right — new object
setUser({ ...user, age: 31 });
```

The wrong version passes the same object reference; React's bail-out
optimization sees no change and skips the re-render. The right one
creates a new object.

For arrays:

```jsx
// Add
setItems([...items, newItem]);

// Remove
setItems(items.filter((i) => i.id !== removeId));

// Update one
setItems(items.map((i) => i.id === updateId ? { ...i, done: true } : i));
```

Never `push`, `splice`, `sort` (in place), or assign to indices on
state arrays. Use the immutable patterns.

For deeply nested state, libraries like Immer let you write code
that *looks* mutating but produces immutable updates:

```jsx
import { produce } from 'immer';

setUser(produce((draft) => {
  draft.profile.address.city = 'NYC';
}));
```

Or restructure state into smaller pieces so updates stay shallow.

## Multiple state variables vs one object

```jsx
// Option A — separate states
const [firstName, setFirstName] = useState('');
const [lastName, setLastName] = useState('');
const [email, setEmail] = useState('');

// Option B — one state object
const [user, setUser] = useState({ firstName: '', lastName: '', email: '' });
```

For independent values, separate `useState` calls are easier.
For values that always update together, one object is cleaner.

For complex state with relationships, see **`useReducer`**.

## Local vs lifted state

```jsx
// Local — each input is independent
function Input() {
  const [value, setValue] = useState('');
  return <input value={value} onChange={(e) => setValue(e.target.value)} />;
}
```

If a parent needs the value (to read it, to share it with siblings),
**lift state up**:

```jsx
function Form() {
  const [name, setName] = useState('');
  return (
    <>
      <NameInput value={name} onChange={setName} />
      <Greeting name={name} />
    </>
  );
}
```

The state lives in the common parent. Children become "controlled" —
they receive value and callback as props, no internal state.

## Resetting state

State is tied to component identity. Two ways to reset:

1. **Conditional render** — unmount and remount the component:

```jsx
{step === 1 && <Form key="1" />}
{step === 2 && <Form key="2" />}
```

Different `key` → React treats them as different components → fresh
state.

2. **Setter** — manually reset:

```jsx
function Form() {
  const [value, setValue] = useState('');
  return (
    <>
      <input value={value} onChange={(e) => setValue(e.target.value)} />
      <button onClick={() => setValue('')}>Reset</button>
    </>
  );
}
```

For "user moves to next step, form should clear," the `key`-based
reset is cleaner — no manual reset logic.

## When NOT to use `useState`

- **Derived data** — don't store what you can compute. `const fullName = first + ' ' + last`, not a state.
- **Server data** — use a query library (TanStack Query, SWR). They
  handle caching, refetching, and stale data correctly.
- **Refs** — when you need a value that persists across renders but
  doesn't trigger re-renders, use `useRef`.
- **Form data with many fields** — `react-hook-form` is more
  efficient than a `useState` per field.
- **Global app state** — Zustand, Redux, Jotai. `useState` is local;
  context + state can become global but it has trade-offs (see
  **`useContext`**).

## Common pitfalls

### Stale closures

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  useEffect(() => {
    const id = setInterval(() => setCount(count + 1), 1000);
    return () => clearInterval(id);
  }, []);     // empty deps — but count is captured!
}
```

The interval captures `count` at mount (which is 0) and never sees
updates. Result: counter ticks 0 → 1 → 1 → 1...

Fix with the functional setter:

```jsx
setCount((c) => c + 1);
```

This always receives the latest value, no stale closure.

### Forgetting that setters don't update immediately

```jsx
setCount(count + 1);
if (count > 10) { ... }   // checks OLD count
```

`count` is the value at the start of this render. After
`setCount`, the variable doesn't change — only the next render's
`count` is updated.

### Storing derived data

```jsx
const [items, setItems] = useState([]);
const [count, setCount] = useState(0);  // derived from items.length!
```

`count` will drift. Just compute: `const count = items.length`.

### Putting non-rendered data in state

If the value doesn't affect what's rendered, it doesn't need
to be state. Use a `useRef` instead — it persists across renders but
doesn't trigger re-renders.

## The mental model

State = "the data this component remembers across renders that, when
changed, causes a re-render."

Use it for:

- What the user has typed (uncontrolled forms aside).
- Toggles, open/closed states.
- Selected items, current step.
- Anything that changes in response to user action.

For other things — derived values, refs, server data, global state —
use other tools.

## The discipline

- Update state via the setter, never by mutation.
- Use the functional form when reading previous state in async or
  multiple-update scenarios.
- Treat state as immutable. Replace objects/arrays instead of
  mutating.
- Lift state up when siblings share it.
- Don't store derived values — compute them.
- Don't store server data in `useState` — use a query library.
