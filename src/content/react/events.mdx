# Event handlers

React events look like inline HTML handlers, but they're real
JavaScript functions wired up to a synthetic event system. Same DOM
events you know — `click`, `change`, `submit`, `keydown` — just with
camelCase names and React semantics.

## The basics

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  function handleClick() {
    setCount(count + 1);
  }

  return <button onClick={handleClick}>Click me</button>;
}
```

Three things to notice:

1. The attribute is `onClick`, not `onclick`. React uses camelCase
   for all event props.
2. You pass a **function reference** (`handleClick`), not the result
   of calling it. `onClick={handleClick()}` would call it during
   render and pass the return value (usually `undefined`).
3. The handler is just a function — define it inline, as a named
   function, or destructure it from a hook.

## Inline arrow functions

For simple handlers:

```jsx
<button onClick={() => setCount(count + 1)}>+</button>
```

Or to pass arguments:

```jsx
<button onClick={() => deleteUser(user.id)}>Delete</button>
```

The arrow creates a new function each render. For most cases that's
fine — React doesn't care. It matters when:

- You pass it to a memoized child (`React.memo`) — each new function
  defeats the memo. Wrap with `useCallback`.
- You're in a tight rendering loop with thousands of handlers.

For everyday code, inline arrows are idiomatic and cheap.

## The synthetic event

Handlers receive a **synthetic event** wrapping the native one:

```jsx
function handleInput(e) {
  console.log(e.target.value);   // text the user typed
  console.log(e.target);          // the input element
  console.log(e.currentTarget);   // the element the handler is on
  console.log(e.type);            // 'change'
}
```

Synthetic events normalize cross-browser quirks. The interesting
properties:

- `e.target` — the element that triggered the event (could be a
  child).
- `e.currentTarget` — the element the handler is attached to.
- `e.preventDefault()` — stop the browser's default behavior.
- `e.stopPropagation()` — stop bubbling to parent handlers.
- `e.key` (for keyboard) — `'Enter'`, `'Escape'`, `'a'`.
- `e.button` (for mouse) — 0 = left, 1 = middle, 2 = right.

## Common events

```jsx
<button onClick={fn}>Click</button>

<input
  onChange={fn}      // value changed (text inputs fire every keystroke)
  onInput={fn}        // similar — fires for every input change
  onBlur={fn}         // lost focus
  onFocus={fn}        // gained focus
  onKeyDown={fn}      // key pressed down
  onKeyUp={fn}        // key released
/>

<form onSubmit={fn}>...</form>

<div
  onMouseEnter={fn}
  onMouseLeave={fn}
  onScroll={fn}
/>
```

## `preventDefault` for forms

```jsx
function handleSubmit(e) {
  e.preventDefault();    // don't reload the page
  saveData(formData);
}

return <form onSubmit={handleSubmit}>...</form>;
```

Forms submit by reloading the page by default — almost never what
you want in a React app. `preventDefault` stops it.

Same applies to links if you want JS-driven navigation:

```jsx
<a href="/profile" onClick={(e) => { e.preventDefault(); navigate('/profile'); }}>
  Profile
</a>
```

(Use `<Link>` from your router instead; it does this for you.)

## Event delegation under the hood

React doesn't actually attach a `click` listener to every button.
It attaches one delegated listener at the root of the React tree and
dispatches to the right handler based on what was clicked. Mostly
invisible to you, but it explains why:

- React handlers fire in a slightly different order than native ones
  in some cases (rarely matters).
- `stopPropagation` on a native event listener won't stop a React
  handler (they're on different layers).
- Synthetic events are reused — old React (≤16) pooled them and
  required `e.persist()` to use them asynchronously. React 17+ no
  longer pools; you can use the event freely.

## Keyboard events

```jsx
<input
  onKeyDown={(e) => {
    if (e.key === 'Enter') submit();
    if (e.key === 'Escape') close();
    if (e.metaKey && e.key === 's') { e.preventDefault(); save(); }
  }}
/>
```

`e.key` gives you the symbolic key name. Modifier keys:
`e.metaKey` (Cmd), `e.ctrlKey`, `e.altKey`, `e.shiftKey`.

For accessibility, prefer `onKeyDown` over `onKeyPress` (deprecated).

## Custom event handlers as props

A component can accept event handlers as props:

```jsx
function Button({ onClick, children }) {
  return <button onClick={onClick}>{children}</button>;
}

<Button onClick={() => alert('hi')}>Click</Button>
```

The convention: prefix with `on` (`onChange`, `onSelect`, `onSubmit`).
Internally, you decide what triggers the call.

For events that aren't DOM events:

```jsx
function Toggle({ onToggle }) {
  const [on, setOn] = useState(false);
  return (
    <button onClick={() => {
      const next = !on;
      setOn(next);
      onToggle?.(next);
    }}>
      {on ? 'On' : 'Off'}
    </button>
  );
}
```

`onToggle?.(...)` only calls if the prop was provided. Lets callers
opt in.

## Passing arguments through handlers

```jsx
function deleteItem(id) { ... }

{items.map((item) => (
  <button key={item.id} onClick={() => deleteItem(item.id)}>
    Delete {item.name}
  </button>
))}
```

The arrow closes over `item.id`. Don't do `onClick={deleteItem(item.id)}`
— that calls during render.

Alternative: pass via `data-` attributes:

```jsx
<button data-id={item.id} onClick={(e) => deleteItem(e.currentTarget.dataset.id)}>
  Delete
</button>
```

Less common, but useful for very large lists where creating
thousands of arrow functions might matter.

## Common pitfalls

### Calling instead of passing

```jsx
<button onClick={handleClick()}>   // BAD — called immediately
<button onClick={handleClick}>     // GOOD — reference to function
<button onClick={() => handleClick()}>  // GOOD — arrow wraps it
```

The first form runs the handler during render, gets `undefined`, and
passes that as the click handler. The button effectively has no
handler. Common beginner bug.

### Forgetting `preventDefault` on forms

```jsx
<form onSubmit={(e) => save()}>   // reloads the page on submit
<form onSubmit={(e) => { e.preventDefault(); save(); }}>   // stays on page
```

### Stale closure in handlers

```jsx
function Counter() {
  const [count, setCount] = useState(0);

  function increment() {
    setCount(count + 1);    // captures count at handler creation
  }

  return <button onClick={increment}>+</button>;
}
```

For one-click handlers this is fine — each render creates a fresh
`increment`. The trap is when the function lives longer than the
render:

```jsx
useEffect(() => {
  const id = setInterval(increment, 1000);  // captures STALE increment
  return () => clearInterval(id);
}, []);
```

The interval calls a frozen `increment` forever. Fix with functional
setter: `setCount((c) => c + 1)`.

### `e.target` vs `e.currentTarget`

```jsx
<div onClick={(e) => {
  e.target;          // could be a nested span the user clicked
  e.currentTarget;   // always the div
}}>
  <span>click me</span>
</div>
```

For "what did I click on?" use `currentTarget`. For "what was the
exact thing clicked?" use `target`.

## Accessibility

Many handlers come with implicit accessibility requirements:

- Use `<button>`, not `<div onClick>`. The button is focusable and
  keyboard-activatable.
- `onClick` on non-interactive elements doesn't fire with keyboard.
  Add `tabIndex={0}` and `onKeyDown` if you really need to.
- `<form onSubmit>` is better than a "submit button"'s `onClick` —
  it handles Enter-to-submit naturally.

## The mental model

Event handlers are just functions React calls when the corresponding
DOM event fires. They receive a normalized event object. You can
call setters, navigate, fetch — anything you'd do anywhere else. The
component re-renders if state changes.

The two beginner traps are:

- Passing a function call (`onClick={fn()}`) instead of a reference
  (`onClick={fn}`).
- Forgetting `e.preventDefault()` on forms and links.

Once those are habit, everything else is just regular JavaScript
inside `onSomething={...}`.
