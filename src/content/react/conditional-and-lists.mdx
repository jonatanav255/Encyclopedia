# Conditional rendering and lists

Two of the most-used patterns in React. Both lean on plain
JavaScript inside JSX, but each has subtleties worth knowing.

## Conditional rendering

### Ternary

```jsx
<p>{isLoggedIn ? <Greeting /> : <Login />}</p>
```

Use when you have a "this or that" branch.

### `&&` short-circuit

```jsx
{isOpen && <Modal />}
```

If `isOpen` is truthy, render the modal. If falsy, render nothing
(React skips falsy children).

The classic bug:

```jsx
{items.length && <List items={items} />}
```

If `items.length` is 0, the expression evaluates to `0` — and `0` is
a renderable value. React renders the literal "0" on the page. Fix
with an explicit comparison:

```jsx
{items.length > 0 && <List items={items} />}
```

Or coerce to boolean:

```jsx
{Boolean(items.length) && <List items={items} />}
```

### Early return

```jsx
function Profile({ user }) {
  if (!user) return <Loading />;
  if (user.banned) return <Banned />;

  return <UserCard user={user} />;
}
```

Cleanest for "stop rendering if missing data" guards. Avoids deep
ternary nesting.

### Multiple conditions

For three or more branches, ternary chains get unreadable. Use
early returns, a switch, or a lookup:

```jsx
function Status({ kind }) {
  const map = {
    idle: <Idle />,
    loading: <Loading />,
    success: <Success />,
    error: <Error />,
  };
  return map[kind] ?? null;
}
```

Or:

```jsx
function Status({ kind }) {
  switch (kind) {
    case 'idle': return <Idle />;
    case 'loading': return <Loading />;
    case 'success': return <Success />;
    case 'error': return <Error />;
    default: return null;
  }
}
```

## Returning `null`

```jsx
function Banner({ show, message }) {
  if (!show) return null;
  return <div className="banner">{message}</div>;
}
```

Returning `null` (or `false`, or `undefined`) renders nothing.
Useful for components that sometimes shouldn't appear at all.

The component still runs — hooks fire, effects schedule — it just
produces no DOM. To skip the work entirely, don't render the
component:

```jsx
{show && <Banner message={message} />}
```

## Lists with `map`

```jsx
function UserList({ users }) {
  return (
    <ul>
      {users.map((u) => (
        <li key={u.id}>{u.name}</li>
      ))}
    </ul>
  );
}
```

`map` produces an array of elements. React renders each, in order.

The `key` prop is mandatory. React uses it to track which element
corresponds to which item across renders — see **Keys and
reconciliation** for the deep dive.

## Keys

```jsx
{items.map((item) => <Row key={item.id} data={item} />)}
```

Requirements:

- **Unique** among siblings (not globally).
- **Stable** across renders — don't generate fresh IDs each render.
- **Tied to the item**, not its position.

Bad keys:

```jsx
{items.map((item, i) => <Row key={i} ... />)}     // index — unstable on reorder
{items.map((item) => <Row key={Math.random()} ... />)}  // fresh each render
{items.map((item) => <Row key={item.name} ... />)}  // ok if names are unique and stable
```

Index keys are the most common trap. They work fine if the list
never reorders, but if the order changes (sort, filter, insert), React
matches the wrong items — state, focus, animations break.

Use a real ID. If you don't have one, generate stable IDs at item
creation, not at render.

## Filtering and transforming

```jsx
{users
  .filter((u) => u.active)
  .map((u) => <UserCard key={u.id} user={u} />)}
```

Chain `filter`/`map` for shape transformations. Compute outside the
JSX if it's getting hairy:

```jsx
function App({ users, query }) {
  const filtered = users
    .filter((u) => u.name.toLowerCase().includes(query.toLowerCase()))
    .sort((a, b) => a.name.localeCompare(b.name));

  return (
    <ul>
      {filtered.map((u) => <UserCard key={u.id} user={u} />)}
    </ul>
  );
}
```

Don't worry about performance for normal lists. For 10,000+ items,
look into virtualization (`react-window`, `react-virtual`).

## Empty states

```jsx
function UserList({ users }) {
  if (users.length === 0) return <p>No users yet.</p>;
  return (
    <ul>
      {users.map((u) => <li key={u.id}>{u.name}</li>)}
    </ul>
  );
}
```

Always handle empty. Lists of "nothing" are awkward UX — explicit
empty messages explain what's going on.

## Fragments in lists

If each item needs multiple top-level elements:

```jsx
{items.map((item) => (
  <React.Fragment key={item.id}>
    <dt>{item.label}</dt>
    <dd>{item.value}</dd>
  </React.Fragment>
))}
```

The shorthand `<>...</>` doesn't accept keys, so you need the full
form when keys are required.

## Rendering arrays directly

You can return an array from a component:

```jsx
function Row({ user }) {
  return [
    <td key="name">{user.name}</td>,
    <td key="age">{user.age}</td>,
  ];
}
```

Or use a fragment, which is usually clearer:

```jsx
function Row({ user }) {
  return (
    <>
      <td>{user.name}</td>
      <td>{user.age}</td>
    </>
  );
}
```

## Conditional CSS classes

```jsx
<button className={isActive ? 'active' : ''}>...</button>

<button className={`base ${isActive ? 'active' : ''} ${isDisabled ? 'disabled' : ''}`}>
  ...
</button>
```

For multiple conditional classes, use a helper:

```jsx
import clsx from 'clsx';

<button className={clsx('base', { active: isActive, disabled: isDisabled })}>
  ...
</button>
```

`clsx` (or `classnames`) is the idiomatic helper. Small dependency,
saves a lot of string juggling.

## Inline render decisions

```jsx
return (
  <div>
    <Header />
    {showSidebar && <Sidebar />}
    <Main>
      {loading ? (
        <Spinner />
      ) : error ? (
        <Error message={error} />
      ) : (
        <Content data={data} />
      )}
    </Main>
  </div>
);
```

Reasonable when there are 2-3 branches. Beyond that, extract into a
sub-component or use an early return.

## Common pitfalls

### Index keys with reorderable lists

```jsx
{todos.map((t, i) => <Todo key={i} todo={t} />)}
```

User reorders the list → React thinks items at indices 0, 1, 2 are
the same items as before, just with different data. Inputs lose
focus. Animations restart. State inside `Todo` corrupts.

Real IDs only.

### `&&` with numbers or strings

```jsx
{count && <Badge n={count} />}      // renders "0" if count is 0
{name && <h1>{name}</h1>}            // renders nothing if name is '' (which is fine here)
```

Be explicit with comparisons when the value might be falsy-but-meaningful.

### Forgetting to return JSX in `map`

```jsx
items.map((item) => {
  <li>{item}</li>     // NOT returned — returns undefined
});
```

With an arrow + braces, you need `return`. With parentheses, implicit
return:

```jsx
items.map((item) => <li>{item}</li>);
items.map((item) => (
  <li>{item.name} ({item.role})</li>
));
items.map((item) => {
  const label = format(item);
  return <li>{label}</li>;
});
```

### Logic in JSX gets unreadable

```jsx
return (
  <div>
    {a && b && c ? (
      <X
        prop1={complex(thing)}
        prop2={another(thing)}
      />
    ) : d || e ? (
      <Y />
    ) : null}
  </div>
);
```

Extract:

```jsx
function inner() {
  if (a && b && c) return <X prop1={complex(thing)} prop2={another(thing)} />;
  if (d || e) return <Y />;
  return null;
}

return <div>{inner()}</div>;
```

## The mental model

JSX is JavaScript with markup. Conditionals are JS expressions
(ternary, `&&`). Lists are JS arrays produced by `map`. Empty branch?
Return `null` or render nothing. Repeated items? Each needs a stable
unique key.

The patterns aren't framework-specific — they're just how
expressions work. The only thing React-specific is the key
requirement on list items.
