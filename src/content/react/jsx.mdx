# JSX

JSX is the markup-like syntax you write inside React components. It
looks like HTML, but it's actually JavaScript expressions in
disguise — transformed at build time into function calls.

```jsx
const greeting = <h1 className="title">Hello</h1>;
```

Becomes:

```js
const greeting = jsx('h1', { className: 'title', children: 'Hello' });
```

That's the whole trick. JSX is sugar.

## Why JSX

You could write React without JSX. It would be exhausting:

```js
React.createElement('div', null,
  React.createElement('h1', null, 'Hello'),
  React.createElement('p', null, 'world')
);
```

vs.

```jsx
<div>
  <h1>Hello</h1>
  <p>world</p>
</div>
```

JSX co-locates structure and logic. The full power of JavaScript is
available inside markup, and the markup is grep-able and editor-
friendly.

## Embedded expressions

Curly braces inject JavaScript:

```jsx
const name = 'Jon';
const showBadge = true;

return (
  <div>
    <h1>Hello, {name}</h1>
    <p>{Math.random() < 0.5 ? 'lucky' : 'unlucky'}</p>
    {showBadge && <span className="badge">New</span>}
  </div>
);
```

Anything that's an expression is allowed: function calls, ternaries,
property access, template literals. Statements (`if`, `for`) aren't —
use ternaries or extract to a variable.

## Attributes

JSX attributes use camelCase (`className`, `onClick`, `tabIndex`)
instead of HTML's lowercase (`class`, `onclick`, `tabindex`):

```jsx
<button className="primary" onClick={handleClick} disabled={!ready}>
  Submit
</button>
```

`class` is a reserved word in JS, so React uses `className`. `for` on
`<label>` becomes `htmlFor`.

String attributes use double quotes; JS expressions use braces:

```jsx
<img src="/logo.png" alt={altText} width={300} />
```

`width={300}` passes the number 300. `width="300"` passes the
string "300". Usually both work for HTML, but React passes whatever
type you wrote.

## Children

What's between the tags becomes the `children` prop:

```jsx
<Card>
  <h2>Title</h2>
  <p>Body</p>
</Card>
```

Inside `Card`, `props.children` is an array containing the `<h2>` and
`<p>` elements. You render them by including `{children}`:

```jsx
function Card({ children }) {
  return <div className="card">{children}</div>;
}
```

This is the foundation of composition — `Card` doesn't know what's
inside; the caller does.

## Conditional rendering

Three common patterns:

```jsx
// Ternary
{isLoggedIn ? <Profile /> : <Login />}

// && for "render if true"
{showError && <Error message={error} />}

// Early return
function Component({ data }) {
  if (!data) return <Loading />;
  return <div>{data.name}</div>;
}
```

The `&&` pattern has a gotcha: `0 && <X />` renders the literal "0"
because `0` is falsy but it's also a renderable value. For booleans
or maybe-zero counts, use a ternary or a `Boolean(count)` coercion.

## Lists

```jsx
<ul>
  {items.map((item) => (
    <li key={item.id}>{item.name}</li>
  ))}
</ul>
```

`map` produces an array of elements; React renders each. Every list
item needs a stable, unique **key** — see **Keys and reconciliation**.

## Fragments

JSX requires exactly one root element. To return siblings without an
extra wrapper:

```jsx
return (
  <>
    <Header />
    <Main />
    <Footer />
  </>
);
```

`<>...</>` is the short form of `<React.Fragment>...</React.Fragment>`.
Use the named form when you need a `key` (e.g., in a list of
fragments).

## Self-closing tags

Tags without children must self-close:

```jsx
<img src="/x.png" />     // not <img src="/x.png">
<input type="text" />    // not <input type="text">
<MyComponent />          // not <MyComponent>
```

This is stricter than HTML and a common build-time error.

## Comments

```jsx
return (
  <div>
    {/* This is a comment inside JSX */}
    <h1>Hello</h1>
  </div>
);
```

`{/* */}` is the JSX comment syntax. Outside JSX, regular JS comments
(`// ...` or `/* */`) work as expected.

## Style

Inline styles are objects, not strings:

```jsx
<div style={{ color: 'red', fontSize: 14, marginTop: 10 }}>Hi</div>
```

The double braces are: outer for JS expression, inner for the object
literal. Properties use camelCase (`fontSize`, not `font-size`).
Numeric values get `px` added by React (where appropriate).

For real styling, use CSS files, CSS modules, Tailwind, or a CSS-in-JS
library. Inline `style` is for one-offs and dynamic values.

## Class components in JSX

Components — both functions and classes — start with a **capital
letter**:

```jsx
<MyComponent />     // React component
<div />             // HTML element
```

The capitalization tells JSX whether to render an HTML element or a
React component. `<myComponent />` (lowercase) is treated as a
custom HTML tag, which usually isn't what you want.

## Spreading props

```jsx
const props = { src: '/x.png', alt: 'logo', width: 100 };
<img {...props} />
```

The `{...props}` spreads each key as an attribute. Useful for
passing through unknown props, but lose type safety quickly. Some
linters warn against it for that reason.

## JSX gotchas

### `&&` and falsy non-booleans

```jsx
{items.length && <List items={items} />}
```

If `items.length` is 0, React renders the literal "0" (because 0 is
falsy). Fix with `items.length > 0 && ...` or a ternary.

### Conditional with empty string

Same problem:

```jsx
{username && <p>{username}</p>}
```

If `username` is `''`, React renders nothing — but `''` is falsy so
no harm. The bug is with `0`.

### Forgetting `key`

React warns in the console. Always add a stable `key` for list
items. **Don't use the array index** unless the list never reorders
or doesn't have stable IDs.

### Comment outside expression

```jsx
<div>
  // not a comment — rendered as text!
</div>
```

Inside JSX, only `{/* */}` works.

### Multiline returns without parens

```jsx
return                   // returns undefined!
  <div>...</div>;
```

Insert a Linebreak after `return` and JS's ASI silently inserts a
semicolon. Wrap multi-line JSX in parentheses:

```jsx
return (
  <div>...</div>
);
```

## The compilation

Modern React uses the "automatic" JSX transform (React 17+). The
build tool inserts `import { jsx } from 'react/jsx-runtime'` for you
— you don't import React just to use JSX.

```jsx
// you write
const el = <h1>Hi</h1>;
```

```js
// compiles to (roughly)
import { jsx as _jsx } from 'react/jsx-runtime';
const el = _jsx('h1', { children: 'Hi' });
```

Old React (16 and earlier) required `import React from 'react'` in
every file because the compiled form was `React.createElement(...)`.
Modern setups (Vite, Next.js, Remix) don't.

## The mental model

JSX = JavaScript with markup notation. Inside the markup, curly
braces re-enter JavaScript. Tags map to HTML elements (lowercase) or
React components (capitalized). Attributes are React's idea of HTML
attributes (camelCase, JS expressions in braces).

Most of the "weird" feels go away once you remember it all compiles
to function calls. `<Card><h2>Hi</h2></Card>` is just
`jsx(Card, { children: jsx('h2', { children: 'Hi' }) })` in
disguise. JSX makes that readable.
