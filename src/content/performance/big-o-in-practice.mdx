# Big-O in practice

Big-O is the right tool for one job: deciding whether an algorithm
will still work when N grows by 100×. It is the wrong tool for
deciding whether function A is faster than function B at N=50.

The two failure modes in production code:

1. Picking an algorithm with the wrong asymptotic class because
   "the constant is small." At N=10M, the constant doesn't save
   you.
2. Picking an algorithm with the right asymptotic class but missing
   that the constant is enormous (a hash map with a 200-byte key
   isn't free).

Both are real bugs.

## What Big-O actually says

Big-O describes how the cost of an algorithm scales as input size
N grows toward infinity. It says nothing about absolute speed.

```
O(N)     — work grows in proportion to N
O(log N) — work grows much slower than N
O(N²)    — work explodes if N doubles, cost ×4
O(1)     — constant; independent of N
```

A `O(N)` linear scan of a small array beats a `O(1)` hash lookup
with a slow hash function for tiny N. Algorithm choice without
knowing N is guessing.

N=10⁶ is the boundary where O(N²) becomes painful (10¹² ops, hours
on a modern CPU). N=10⁹ is where O(N) starts to need streaming and
careful memory use.

## The "small N" lie

A common defense of an O(N²) loop: "N is always small." Three things
break that:

- **N grew.** The function was written when N was 50. It's now
  50,000. The original author left two years ago.
- **N is small *per call* but the function is hot.** 100 nested
  loops of N=20 on 10,000 requests/sec = 4×10⁷ ops/sec. Still cheap.
  Now N=200: 4×10⁹ ops/sec. Your CPU is one task.
- **N appears unrelated but is tied to user data.** "It's just the
  number of teams a user is in." Then a power user joins 5,000 teams
  and the page takes 30 seconds.

Treat "N is small" as something to **enforce** (a cap, an assertion)
rather than something to assume.

## The hidden-constant traps

Big-O ignores constants. The constants ignored are big:

- **Cache friendliness.** A linear scan of a contiguous array is
  often 10–20× faster than a linked list of the same length because
  the linked list misses cache on every node. Both are O(N).
- **Allocation cost.** O(N) with N allocations is dramatically slower
  than O(N) with no allocations. Allocations are 50–200ns each in
  managed languages; GC pressure makes it worse.
- **Hashing cost.** `Map<string, V>` lookups are O(1) in the textbook
  sense, but each lookup hashes the key. A 1KB key is much slower
  to look up than a 4-byte integer.

## A worked rewrite

The "find duplicates" interview problem, two ways:

```js
// O(N²) — nested loop
function hasDuplicate(arr) {
  for (let i = 0; i < arr.length; i++) {
    for (let j = i + 1; j < arr.length; j++) {
      if (arr[i] === arr[j]) return true;
    }
  }
  return false;
}

// O(N) — set
function hasDuplicate(arr) {
  const seen = new Set();
  for (const x of arr) {
    if (seen.has(x)) return true;
    seen.add(x);
  }
  return false;
}
```

At N=100, both finish in microseconds. At N=10,000, the O(N²) version
does 50M comparisons (still a few hundred ms); the set version is
about 100µs. At N=100,000, the O(N²) version is 5×10⁹ ops — minutes.
The set version is 1ms.

The fix wasn't to make the inner loop faster. It was to delete
the inner loop.

## When O(N²) is actually fine

- **N is bounded by something physical.** "Pieces in a chess game"
  caps at 32. The O(N²) loop is 1024 ops, always.
- **N is bounded by the platform.** "Items on a screen" might be
  capped at 200 by UI design.
- **The constant is dramatically lower.** Insertion sort beats
  quicksort for N < ~30 because the constant factor is small and
  the cache behavior is great.

For everything else, the rule of thumb: if your data is **a list a
user could grow without bound**, your algorithm should be O(N) or
O(N log N). If it's a fixed small set defined by domain rules, the
asymptotic class barely matters.

## How to talk about it

In code review or design: state both Big-O **and** what N is. "This
is O(N²) where N is the number of teams a user has joined; we
should put a cap on that or rewrite." Pure asymptotic talk without
N grounds nothing.
