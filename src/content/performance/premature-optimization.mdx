# Premature optimization

Knuth's full quote is more useful than the bumper sticker:

> "Programmers waste enormous amounts of time thinking about, or
> worrying about, the speed of noncritical parts of their programs,
> and these attempts at efficiency actually have a strong negative
> impact when debugging and maintenance are considered. We should
> forget about small efficiencies, say about 97% of the time:
> premature optimization is the root of all evil. Yet we should
> not pass up our opportunities in that critical 3%."
> — Donald Knuth, 1974

The interesting word is "**noncritical**." Knuth isn't saying
"don't optimize." He's saying don't optimize without knowing
where the critical 3% is.

## Three failure modes the quote attacks

<Steps>
<Step n="1" title="Optimizing the wrong code">
You spent two days inlining a function, only to discover the
profile says it's 0.4% of runtime. You moved a 0.4% number. The
other 99.6% is where the time went.
</Step>

<Step n="2" title="Optimizing without measurement">
"This loop *feels* slow, I'll rewrite it in a clever way." The
clever way is two pages longer, harder to read, and on the actual
input distribution it's the same speed or slower.
</Step>

<Step n="3" title="Trading clarity for nothing">
Code that's 5% faster but takes the next maintainer twice as long
to understand is a net negative. The maintainer cost is real and
spans years.
</Step>
</Steps>

## What's *not* premature

Picking the right approach when you sit down to write the code is
not premature optimization. It's design.

- Choosing a hash map over a linear scan when N might be large
  someday — fine. Big-O choices at design time are free, sometimes
  cheaper than the alternative.
- Adding pagination to an endpoint that returns a list — fine.
  Unbounded-list endpoints are a known footgun.
- Using a streaming JSON parser when you know the payload is huge —
  fine.
- Caching an expensive computed value that's read on every render
  — fine, if the cost is visibly noticeable.

These are **not** "premature" because:

- They cost the same as the naive version (or are easier to write).
- Reversing them later is expensive (you'd have to refactor the
  callers).
- The performance trap they avoid is well-known and predictable.

## What *is* premature

- Inlining all your function calls "for speed."
- Replacing readable code with bit-twiddling tricks before profiling.
- Manually caching things that the OS / DB / framework already
  caches.
- Adding a memoization layer to a function that runs once.
- Rewriting your hot path in C/Rust before knowing whether it's
  actually hot.
- Asking "what's faster, X or Y?" when both run in under a
  microsecond and the function is called once per request.

A reliable heuristic: if the optimization makes the code harder
to read or change, **you need a measurement to justify it**.

## The 90/10 (or 99/1) reality

A modern web service's CPU profile typically looks like:

- 30–60%: database driver and query parsing.
- 10–20%: JSON serialization / deserialization.
- 10–20%: framework middleware (auth, logging, routing).
- 10–30%: your actual business logic.

If you're optimizing your business logic by 2× and it's 20% of
runtime, you've moved the needle by 10%. If you're optimizing the
DB query by 2× and it's 40% of runtime, you've moved it by 20%
with the same effort. The framing isn't "is this code slow?", it's
"how much of the budget does this code spend?"

## The discipline

A repeatable loop that beats "I think this might be slow":

<Steps>
<Step n="1" title="Pick a target metric">
"This endpoint p95 is 800ms; I want it under 200ms." Concrete and
measurable. Without this, "make it faster" has no endpoint.
</Step>
<Step n="2" title="Measure the baseline">
With realistic data. Note the number.
</Step>
<Step n="3" title="Profile">
Use a profiler or detailed timing logs. Find the heaviest few
functions or DB calls. *Don't guess.*
</Step>
<Step n="4" title="Hypothesize and fix one thing">
Change one thing. Re-measure. Did it move?
</Step>
<Step n="5" title="Stop when the target is hit">
Don't keep optimizing past the target. The remaining work has
diminishing returns and ongoing maintenance cost.
</Step>
</Steps>

The biggest gains in real systems usually come from removing work
entirely (a query that didn't need to run, a serialization step
that could be skipped), not making existing work go faster.

## When the bumper sticker misleads

Two cases where "premature optimization is the root of all evil"
gets quoted to *avoid useful work*:

- Skipping pagination on a list endpoint because "we don't have
  many users yet." You'll get them. The bug will land at a bad
  time.
- Picking O(N²) "because it's simpler" without writing down what
  N is and why it stays small. Fine if N is bounded by a domain
  rule; not fine if N is "user-controlled input."

Knuth's "noncritical" is doing real work in that quote. Critical
paths *should* be designed with performance in mind. Everything
else can be the readable version.
