# OS overview and kernel architectures

An operating system does two things: **virtualize** the hardware so
multiple programs can pretend they own the machine, and **manage**
shared resources (CPU, memory, devices, files) so the pretending
doesn't end in tears. Everything else — schedulers, file systems,
drivers, IPC — is in service of those two jobs.

## The four jobs of a kernel

1. **CPU virtualization.** Give every process the illusion of its
   own CPU. Implemented by time-sharing + context switches.
2. **Memory virtualization.** Give every process the illusion of
   its own address space. Implemented by virtual memory + page
   tables.
3. **Persistence.** Survive reboots. Implemented by file systems
   on top of block devices.
4. **Protection.** Stop processes from clobbering each other or
   the kernel. Implemented by user/kernel mode + the MMU.

Arpaci-Dusseau call these "the three easy pieces" (lumping
persistence in with protection). The framing is what matters: every
OS concept is a way of doing one of these four things.

## Kernel architectures

```
Monolithic                Microkernel              Hybrid
┌────────────────┐        ┌──────┐  ┌──────┐      ┌──────────────┐
│ syscalls       │        │ FS   │  │ Net  │      │ syscalls     │
│ FS + Net + VM  │        │ srvr │  │ srvr │      │ FS + VM core │
│ + drivers      │  ←kernel│       │  │       │      │ (in kernel)  │
│ all in kernel  │        ├──────┴──┴──────┤      │  Net + drivers│
│                │        │  microkernel    │      │  in userspace │
└────────────────┘        └─────────────────┘      └──────────────┘
Linux, BSD, Win NT        MINIX 3, L4, QNX        macOS (XNU)
```

**Monolithic** — everything (drivers, FS, network stack, scheduler)
runs in one address space in kernel mode. Fast: a syscall is one
mode switch and then a function call. Fragile: a buggy driver can
panic the whole machine. Linux, FreeBSD, classic Unix.

**Microkernel** — only the bare minimum runs in kernel mode
(scheduling, IPC, basic memory). File systems, drivers, network
stacks run as user-mode *servers* that talk via message passing.
Survives driver crashes (the reincarnation server in MINIX 3
restarts dead servers). Slower: every FS call is now a context
switch plus an IPC round-trip.

**Hybrid** — large kernel like a monolith, but structured around
message-passing primitives and with some servers in user space.
macOS/XNU is Mach (microkernel) + BSD (monolithic) + IOKit.
Windows NT was *designed* as a microkernel but moved graphics and
window management into the kernel for performance.

In 2026, the dominant production kernels are all monolithic-ish.
The microkernel approach won in two narrow places: real-time
embedded (QNX) and formally verified kernels (seL4).

## What runs where

```
                    User mode
┌─────────────────────────────────────────┐
│  bash   nginx   postgres   your code    │
│  libc   openssl  zlib  ...              │
└──────────────┬──────────────────────────┘
               │ syscalls (int 0x80 / syscall / sysenter)
               ▼
┌─────────────────────────────────────────┐
│  VFS  scheduler  memory  net  drivers    │   Kernel mode
└─────────────────────────────────────────┘
               │
               ▼
              Hardware
```

The line between user and kernel mode is enforced by the CPU
itself (the protection rings on x86, EL0/EL1 on ARM). Crossing it
requires a syscall, an interrupt, or an exception — there's no
"just call the kernel function." This is what makes "the kernel"
a real boundary, not a software convention.

## Boot, briefly

What happens between power-on and you-have-a-shell:

1. **Firmware** (UEFI on modern x86, BIOS historically). Initializes
   hardware enough to read the disk.
2. **Bootloader** (GRUB, systemd-boot, Windows Boot Manager). Loads
   the kernel image into memory and jumps to it.
3. **Kernel init.** Detects hardware, sets up page tables, mounts
   the root filesystem, starts PID 1.
4. **PID 1** (systemd, init, launchd, sysvinit). Starts everything
   else — networking, services, getty, your login shell.

A "kernel panic" is when stage 3 or later hits an unrecoverable
error. Userspace can crash all day; the kernel can't.

## Why "the OS" is fuzzy

People say "operating system" to mean three different things:

- **Just the kernel** (Linus's "Linux is only the kernel").
- **Kernel + system libraries + core utilities** (a distribution —
  "Debian," "Ubuntu," "macOS Sonoma").
- **Everything that ships, including the GUI** (the user-facing
  product — "Windows 11," "iOS 17").

Inside one team it doesn't matter, but in a tech interview
clarify: "do you mean the kernel, or the whole distribution?"
The answer changes which abstractions are in scope.

## The mental model

A kernel is a single trusted program that mediates every access to
shared hardware. It has more privileges than any other program,
runs in a separate CPU mode, and is the only thing the hardware
*lets* talk directly to devices. Every other program — including
the shell, your database, the GUI — has to ask the kernel for
anything that touches the outside world.

The rest of OS theory is just naming the asks (syscalls), making
them fast (caches, schedulers), and stopping any one ask from
hurting the others (isolation, protection).
