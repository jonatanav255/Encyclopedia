# Multipart Internals

`multipart/form-data` is how HTML forms send file uploads — and how
any "send mixed fields + binary data" use case works. `multer`
parses it for you in Express, but knowing the wire format explains
why upload handling is the way it is.

## A multipart body, decoded

```
POST /upload HTTP/1.1
Content-Type: multipart/form-data; boundary=----WebKitFormBoundary7MA4YWxkTrZu0gW

------WebKitFormBoundary7MA4YWxkTrZu0gW
Content-Disposition: form-data; name="username"

jon
------WebKitFormBoundary7MA4YWxkTrZu0gW
Content-Disposition: form-data; name="avatar"; filename="me.jpg"
Content-Type: image/jpeg

<binary JPEG bytes>
------WebKitFormBoundary7MA4YWxkTrZu0gW--
```

Three things to notice:

1. **The boundary** in the `Content-Type` header is the literal
   separator. The browser generates a random one unlikely to appear
   in the payload.
2. **Each part has its own headers** — at minimum, `Content-Disposition`
   naming the field. Files also include `Content-Type` and `filename`.
3. **The body is one byte stream** with embedded boundary markers.
   The final boundary has a trailing `--`.

This is the actual on-the-wire format. Parsers like `multer` walk
this byte stream and split it into parts.

## Why this format

- **Binary-safe**: arbitrary bytes between the markers.
- **Multiple fields**: text and files mixed in one body.
- **Streaming-friendly**: parsers don't need to buffer the whole
  thing; they can emit parts as they're parsed.

The downsides:

- **Parsing complexity** — much more involved than URL-encoded forms.
- **No standard size limit** — easy to receive accidentally huge
  uploads if you don't cap.
- **Verbose** — lots of overhead per field (the headers add up).

## Why `express.json()` can't help

`express.json()` reads the body assuming it's JSON. A multipart
body has its own structure and includes binary data — it would
break trying to UTF-8 decode it. Different format, different parser.

That's why uploads need `multer` (or `busboy`, the lower-level
parser `multer` is built on):

```js
import multer from 'multer';
const upload = multer({ dest: 'uploads/' });

app.post('/upload', upload.single('avatar'), (req, res) => {
  // req.file is the parsed file, req.body is text fields
});
```

## The boundary parameter

`Content-Type: multipart/form-data; boundary=XYZ` — the server reads
`XYZ` and looks for it in the body. The boundary is preceded by
`\r\n--` and the final boundary is followed by `--\r\n`.

The boundary value is generated by the client to be unlikely to
appear in payload bytes. If it does appear (rare collision), the
upload corrupts. Modern clients use a long random boundary that
basically never collides.

## File metadata

`Content-Disposition: form-data; name="avatar"; filename="me.jpg"`

- `name` is the *form field name*. Required.
- `filename` is the *original client-side filename*. Optional, and
  **trusted at your own risk** — the client controls it. Don't use
  it as a path; sanitize or generate server-side.

The "filename* " variant supports non-ASCII names via RFC 5987:

```
Content-Disposition: form-data; name="avatar"; filename*=UTF-8''m%C3%B6.jpg
```

That's `mö.jpg` with the ö encoded as `%C3%B6`. Modern parsers
handle this transparently.

## Content-Type per part

For files: the browser supplies a guessed MIME type based on the
file extension. `multer` exposes this as `req.file.mimetype`.

**Don't trust it.** The client could send anything. For real safety,
check the magic bytes of the file with `file-type` — see **File
Uploads**.

## Field ordering

Fields arrive in the order the form serialized them. The browser
serializes in DOM order (top to bottom for HTML forms; iteration
order for `FormData`). Parsers preserve this.

Why it matters: if you want to validate text fields *before*
accepting the upload, the file must appear *after* the text fields
in the body. Otherwise the parser has already consumed the upload
by the time you read the text.

In practice, most uploads buffer or stream the file regardless of
order — but if you build a custom flow with `busboy`, this is the
sort of detail you have to manage.

## Streaming vs buffering

Two parsing strategies:

- **Buffer everything** — wait for the whole body, then parse. What
  `multer` with `dest:` does internally (writes to a temp file as it
  goes; effectively a stream-to-disk).
- **Stream parts** — emit each part as it's parsed, before the body
  ends. `busboy` works this way.

Streaming is more efficient for big uploads but requires you to
write code that handles parts one at a time. For most apps, `multer`
is the right level.

## Size limits

The protocol has no limit. **You** must set one:

```js
multer({
  dest: 'uploads/',
  limits: { fileSize: 5 * 1024 * 1024 },  // 5MB per file
});
```

Without it, a malicious client uploads 50GB and fills your disk.

For very large uploads, the right architecture is **direct-to-S3
signed URLs**: the client uploads directly to object storage, not
through your Node process. Your server only handles metadata.

## The `Content-Length` problem

The HTTP `Content-Length` header tells the server how big the body
is. Browsers set it; some clients (especially streaming uploads)
don't. Without it, the server reads until the connection closes —
or it uses chunked transfer encoding.

For uploads, you usually want a `Content-Length` so the server can
reject oversized requests upfront (before reading them).

## CSRF in multipart

`multipart/form-data` *is* a "simple" content type from CORS's
perspective — it doesn't trigger a preflight. So a malicious site
can POST a multipart body to your endpoint cross-origin, riding the
victim's cookies.

CSRF tokens still apply. See **Auth Basics**. SameSite cookies help.

## Common parsing bugs

- Forgetting size limits → DoS.
- Trusting `filename` → path traversal.
- Trusting `mimetype` → uploaded HTML serves as image but executes.
- Reading `req.file.buffer` when using disk storage (it's `path`,
  not `buffer`).
- Not cleaning up temp files when the handler errors after upload
  but before processing.

## In practice

For 99% of upload code, you'll use `multer` and never see the wire
format. The few times the format matters:

- Debugging "the upload reaches the server but fields are wrong" —
  inspect the raw body.
- Implementing a custom parser (large files, streaming transformations).
- Forwarding multipart bodies between services (the simplest way is
  to re-stream the parsed parts).
- Writing tests that send multipart bodies (`supertest`'s `.attach`).
