# HTTP Versions

HTTP/1.1 is what your Node service speaks. The web has long since
moved on — HTTP/2 and HTTP/3 are what browsers actually use. The
gap between them is bridged by the reverse proxy in front of your
Node process. Knowing what each version does (and what they don't)
explains a lot of production weirdness.

## HTTP/1.1 (1997)

The version Node's `http` module speaks natively. The shape:

- One TCP connection per request — or, with **keep-alive**, several
  requests serialized over one connection (head-of-line blocking:
  the slow request stalls the ones behind it).
- Plain text headers (`GET /users HTTP/1.1\r\n...`).
- Chunked transfer encoding for streaming.
- Pipelining was attempted, never worked in practice.

Most of what you "know" about HTTP — methods, status codes, the
mental model of request/response — is from /1.1. Everything since has
tweaked the wire format and concurrency, not the semantics.

## HTTP/2 (2015)

Binary, multiplexed, header-compressed. Same semantics, faster wire.

- **Multiplexing**: many requests in flight on one TCP connection,
  interleaved. No more head-of-line blocking *at the HTTP layer*.
- **Header compression** (HPACK): repeated headers don't waste
  bandwidth. Big win for APIs with chatty headers.
- **Server push**: server can preemptively send resources. Removed
  from Chrome in 2022 — turns out it didn't help much in practice.
- **Stream prioritization**: client hints which responses to send
  first.

The wire format is binary frames, not text. You can't `curl -v` and
read it raw.

### TCP head-of-line blocking remains

HTTP/2 fixes HOL at the *HTTP* layer, but the underlying TCP
connection still has packet ordering. One dropped packet stalls
*all* the multiplexed streams on that connection until retransmit.
For high-latency / lossy networks, this is a real problem — which
is what HTTP/3 fixes.

## HTTP/3 (2022)

Same semantics again. Same multiplexing, header compression. The
difference: it runs over **QUIC**, not TCP.

- **QUIC** is built on UDP, implements its own reliable, ordered
  delivery, congestion control, and TLS 1.3 baked in.
- **No TCP HOL blocking** — each stream has independent ordering.
- **0-RTT reconnects** — reuse a previous session to skip the
  handshake entirely.
- **Connection migration** — survives a client's IP changing
  (Wi-Fi to cellular).

HTTP/3 is meaningful for mobile, lossy, or long-RTT networks. For
LAN-to-LAN inside a datacenter, the gains are smaller than the cost
of the new infrastructure.

## What Node speaks

- **`node:http`** — HTTP/1.1.
- **`node:http2`** — HTTP/2 (with HTTP/1.1 compatibility mode).
- **HTTP/3 / QUIC** — not in the standard library (experimental
  modules exist). For production, terminate /3 at the proxy.

Express uses `node:http`. Running Express over HTTP/2 requires
either a different framework (Fastify supports it) or a shim that
bridges /2 streams to Express's /1.1-shaped API. In practice almost
nobody does this — the reverse proxy handles /2 and /3 and forwards
to Node as /1.1.

## The proxy bridges

```
internet ──(HTTP/3)──▶ CDN ──(HTTP/2)──▶ ALB ──(HTTP/1.1)──▶ Node
```

Each hop can speak a different version. Your Node code only deals
with the lowest one (/1.1). This is fine — the wire format
optimizations matter most where the latency is, which is the
client-to-edge leg.

## TLS

All modern HTTP is encrypted. HTTP/2 in practice (and HTTP/3 by
spec) requires TLS. Even when "not required" by the HTTP/2 spec, no
browser implements unencrypted HTTP/2.

TLS termination usually happens at the proxy too. Node sees plain
HTTP, and your `trust proxy` setting tells Express to honor the
original `X-Forwarded-Proto` header for `req.secure`.

For internal service-to-service calls, you may run TLS in Node
directly (`node:https` + `tls`). Less common; reach for it only when
the network between services isn't already trusted.

## What this means for Express code

For 99% of what you write, the HTTP version is invisible:

- Methods, status codes, headers — same.
- `req.headers` works regardless of how they arrived (text or
  HPACK-compressed).
- Streaming works (chunked /1.1, frames /2).
- Trailers and server push are /2-only features Node mostly hides.

What changes if your front-end actually speaks /2 or /3 to the
proxy:

- Connection coalescing — fewer TCP connections per user.
- Lower latency for chatty UIs.
- The `X-Forwarded-Proto` header reflects the original scheme.

For your code, nothing.

## Should you care?

- **As an app developer**: barely. Treat HTTP as protocol-shaped, code
  to the semantics. Let the proxy handle versions.
- **As an infra developer**: a lot. Pick the proxy, configure ALPN,
  tune connection pools, monitor protocol distribution.

The split matters because of how often Node app code is written
"defensively" against /2-specific behavior the developer doesn't
actually need to think about. Spend the thinking on the application
domain.

## A useful detail: ALPN

Application-Layer Protocol Negotiation — during the TLS handshake,
the client and server agree on the protocol. ALPN identifiers:

- `http/1.1` — HTTP/1.1
- `h2` — HTTP/2
- `h3` — HTTP/3

In a proxy config you'll often see `alpn h2 http/1.1` — the proxy
prefers /2 but falls back to /1.1 for older clients. Recognize this
when you read configs.

## TCP keep-alive vs HTTP keep-alive

Distinct things:

- **TCP keep-alive**: low-level periodic packets to detect dead
  peers. Set on the socket.
- **HTTP keep-alive**: reusing one TCP connection for multiple HTTP
  request/response pairs. The default in /1.1 and /2.

When tutorials say "enable keep-alive," they almost always mean the
HTTP form. It's already on by default.
