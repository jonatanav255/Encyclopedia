# Floating point

IEEE-754 defines the floating-point format used by essentially every
modern CPU. Once you understand the layout, the surprises stop being
surprises: `0.1 + 0.2 !== 0.3`, `NaN !== NaN`, the difference between
`+0` and `-0`, why "close enough" comparison needs ULPs or an
epsilon, when to reach for arbitrary-precision decimal.

<Callout type="note">
This page is the language-agnostic theory. For the JavaScript API
(`Number`, `Number.EPSILON`, `Math`, `bigint`, etc.) and JS-specific
pitfalls, see `javascript/numbers.mdx` — they overlap, deliberately.
</Callout>

## The bit layout (binary64 / "double")

64 bits, split as:

```
 sign   exponent (11 bits)    mantissa / significand (52 bits)
  S     EEEE EEEE EEE        MMMM MMMM MMMM ... MMMM
  1            11                       52
```

The value is roughly:

```
(-1)^S  *  1.MMMM...MMMM  *  2^(E - 1023)
```

The leading `1.` is implicit (called the "hidden bit") — only the
fractional part is stored. The exponent is **biased by 1023**: stored
value 1023 means real exponent 0, stored 1024 = real 1, etc. This
keeps the encoded form sortable as an unsigned integer.

Range: roughly `±1.8e308`. Precision: ~15-17 decimal digits.

There's also **binary32 / "float"** (1 sign, 8 exp, 23 mantissa,
~7 decimal digits, range ~3.4e38) — used in graphics, ML inference,
and some embedded.

## Why `0.1 + 0.2 !== 0.3`

`0.1` in decimal has no exact representation in binary. Just like
`1/3` in decimal is `0.333333...` repeating, `0.1` in binary is
`0.0001100110011...` repeating. The double-precision rounding gives
something close-but-not-exact to `0.1`.

The same for `0.2`. Add the two rounded approximations, and you get:

```js
0.1 + 0.2;          // 0.30000000000000004
```

Not a JS bug. Same answer in Python, Java, C, Ruby, Go — anywhere
that uses IEEE-754 binary floats. The values that **are** exactly
representable in binary are sums of powers of two: `0.5`, `0.25`,
`0.125`, `0.75`, `2.5`, `1.5`, ... Decimal-fraction-looking values
like `0.1`, `0.3`, `0.7` are not.

## The three special-ish values

### `Infinity` and `-Infinity`

```js
1 / 0;             //  Infinity
-1 / 0;            // -Infinity
Math.log(0);       // -Infinity
1e308 * 1e2;       //  Infinity   — overflow
```

Stored with the maximum exponent (all 1s) and a zero mantissa.
Arithmetic stays sane: `Infinity + 1 === Infinity`, `1 / Infinity ===
0`. Mostly useful as a sentinel for "bigger than anything else."

### `NaN` (not-a-number)

```js
0 / 0;             // NaN
Infinity - Infinity; // NaN
Math.sqrt(-1);     // NaN
Math.log(-1);      // NaN
parseInt('abc');   // NaN
```

Stored with max exponent and non-zero mantissa. The mantissa bits
let you encode a "payload" (most runtimes use this for diagnostics,
but the value just behaves as "NaN").

NaN is **contagious**: any arithmetic involving NaN is NaN.

And the famous quirk: **NaN is not equal to itself**.

```js
NaN === NaN;       // false
NaN !== NaN;       // true
```

To test for it, use `Number.isNaN(x)` (JS) or the IEEE `isnan(x)`
in C-family. Why? The IEEE designers needed a way to flag invalid
operations such that `x !== x` could be a "is this NaN?" test that
worked in any language, in any expression context, without needing
a dedicated keyword.

### Signed zero

```js
-0 === 0;          // true   — equality says yes
Object.is(-0, 0);  // false  — strict identity says no
1 / 0;             //  Infinity
1 / -0;            // -Infinity     — the sign survives in division!
```

`+0` and `-0` have distinct bit patterns. Most arithmetic treats
them the same, but division and `Math.atan2` and a few other
operations preserve the sign. Hardly ever matters in practice;
occasionally bites when computing limits.

## ULPs — the spacing between floats

ULP = "unit in the last place" = the distance between adjacent
representable floats at a given magnitude.

```js
// Around 1.0, ULP is ~2^-52  ≈ 2.22e-16:
Number.EPSILON;                 // 2.220446049250313e-16

// Around 1e6, ULP is ~2^-52 * 2^20 ≈ 2.33e-10.
// JS has no built-in nextAfter; you can poke the bit
// pattern via a Float64Array:
const buf = new Float64Array(1);
const u32 = new Uint32Array(buf.buffer);
buf[0] = 1e6;
u32[0] += 1;                    // bump the low mantissa bit
buf[0];                         // 1000000.0000002328 → ULP ≈ 2.33e-10
```

The spacing doubles every time you cross a power of two. At
magnitude `2^53` (~9e15), ULP is 1 — consecutive integers stop being
distinguishable. That's `Number.MAX_SAFE_INTEGER`.

For safe equality:

```js
function nearlyEqual(a, b, tol = Number.EPSILON) {
  return Math.abs(a - b) <= tol * Math.max(1, Math.abs(a), Math.abs(b));
}
```

The `Math.max(1, ...)` accounts for the fact that ULP scales with
magnitude. A flat `Math.abs(a - b) < 1e-9` only works for numbers
near zero.

## When floats betray you

### Money

`0.10` exists but doesn't behave. Add 10 of them, expect `1.0`, get
`0.9999999999999999`. Store currency as **integer minor units**
(cents, satoshis), or use a decimal library (`decimal.js`,
`big.js`, Java `BigDecimal`, Python `decimal.Decimal`, .NET
`decimal`). Never `float`.

### Accumulating sums

Naively summing `0.1` a million times gives a noticeably wrong total.
Use **compensated summation** (Kahan summation) when accuracy
matters at scale — keeps a running correction for what the rounding
threw away.

```js
function kahanSum(arr) {
  let sum = 0, c = 0;
  for (const x of arr) {
    const y = x - c;
    const t = sum + y;
    c = (t - sum) - y;
    sum = t;
  }
  return sum;
}
```

Significantly more accurate than `arr.reduce((s, x) => s + x, 0)`
on large arrays of small floats.

### Comparing equal-looking values

```js
0.1 + 0.2 === 0.3;       // false
(0.1 + 0.2).toFixed(15); // "0.300000000000000"     hides the trailing 4
```

Don't `===` two computed floats. Test against a tolerance.

### Subnormals (denormals)

Numbers smaller than the smallest "normal" float (~2.2e-308) are
represented as **subnormals** — with reduced precision. They're rare
in normal code but appear in numeric algorithms that produce
tiny intermediate values. On some CPUs, subnormal arithmetic is
**massively slower** (microcoded), which is why DSP code sometimes
flushes them to zero.

### NaN in collections

```js
new Set([NaN, NaN]).size;            // 1   — special-cased
[1, NaN, 3].indexOf(NaN);            // -1  — uses ===
[1, NaN, 3].includes(NaN);           // true — uses sameValueZero
```

`Set` and `Map` use a NaN-aware equality. `indexOf` doesn't.
`includes` does. Easy to get wrong.

## Decimal floating point

IEEE-754 also defines **decimal** types (`decimal32`, `decimal64`,
`decimal128`) that store base-10 mantissas exactly. Used in some
financial chips, IBM mainframes, .NET's `decimal`. They make `0.1
+ 0.2 === 0.3` true at the cost of slower arithmetic and a different
range. No JS support; for similar semantics use `decimal.js`.

## Rounding modes

IEEE defines four rounding modes for inexact results:

- **Round to nearest, ties to even** — the default. The famous
  "banker's rounding" — 2.5 rounds to 2, 3.5 rounds to 4. Removes
  the long-run bias of "always round up on ties."
- **Round toward zero** — truncate.
- **Round toward +Infinity**.
- **Round toward -Infinity**.

JS exposes only the default. C has `fesetround()`. Most code never
touches this.

## The takeaway

- **IEEE-754 binary doubles are exact for sums of powers of two,
  approximate for everything else.** This includes most decimal
  values you care about.
- **Compare floats with a tolerance.** Use `Math.abs(a - b) <=
  Number.EPSILON * max(1, |a|, |b|)`, not `===`.
- **Use integer cents or a decimal library for money.** Always.
- **`NaN !== NaN`. Test with `Number.isNaN`.** Same shape in any
  language.
- **`±Infinity`** is fine as a sentinel and arithmetic stays
  consistent.
- **Subnormals are rare but slow** — flush-to-zero is a real perf
  knob in numeric code.
