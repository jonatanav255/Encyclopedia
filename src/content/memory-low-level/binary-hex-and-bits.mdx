# Binary, hex, and bits

Computers don't store decimal. Everything is bits. Hex is the shorthand
humans use when staring at bits is too painful and decimal hides the
structure.

## The three bases you actually see

- **Binary (base 2)** — what the hardware uses. Prefix `0b` in JS.
- **Octal (base 8)** — three bits per digit. Mostly survives in
  `chmod 755` and not much else.
- **Hex (base 16)** — four bits per digit. The format of color codes
  (`#ff8800`), memory addresses, hashes, byte dumps. Prefix `0x`.

```js
0b1010;     // 10
0o17;       // 15  (octal, rarely useful)
0xff;       // 255
0xdeadbeef; // 3735928559
```

One hex digit = exactly four bits = one **nibble**. Two hex digits =
one byte. That's why hex dumps are so readable: each pair is one
byte, line lengths are predictable, and you can eyeball patterns
(repeating `00`, ASCII `41 42 43...` = "ABC").

```bash
$ xxd hello.bin
00000000: 4865 6c6c 6f20 776f 726c 6421 0a         Hello world!.
```

Left column is the byte offset (also hex). Middle is the bytes. Right
is the printable-ASCII rendering with `.` for non-printable.

## Converting in your head

A handful of values to memorize:

| bits | decimal | hex |
| --- | --- | --- |
| `1111` | 15 | `f` |
| `1 0000` | 16 | `10` |
| `1111 1111` | 255 | `ff` |
| `1 0000 0000` | 256 | `100` |
| `2^16 - 1` | 65535 | `ffff` |
| `2^32 - 1` | ~4.29B | `ffffffff` |

Powers of two you should recognize on sight: `1024` (`2^10`),
`65536` (`2^16`), `2^20 ~= 1M`, `2^30 ~= 1B`, `2^32 ~= 4.29B`,
`2^53 ~= 9e15` (JS safe integer ceiling), `2^63 ~= 9.2e18`
(signed 64-bit max).

## Bitwise operators

JS coerces to **32-bit signed integers** for bitwise ops. Same for
most C-family languages on `int`.

```js
0b1100 & 0b1010;   // 0b1000 = 8     — AND
0b1100 | 0b1010;   // 0b1110 = 14    — OR
0b1100 ^ 0b1010;   // 0b0110 = 6     — XOR
~0b1100;           // -13            — NOT (flips all 32 bits, sign too)
0b1 << 4;          // 16             — shift left
0b10000 >> 2;      // 4              — arithmetic right shift (sign-preserving)
0b10000 >>> 2;     // 4              — logical right shift (zero-fill)
```

`<<` is multiply by `2^n`. `>>` is divide by `2^n` (rounding toward
negative infinity for negatives).

## Flags and masks

The classic use of bitwise ops: pack a set of booleans into one
integer.

```js
const READ    = 1 << 0;   // 0b001 = 1
const WRITE   = 1 << 1;   // 0b010 = 2
const EXECUTE = 1 << 2;   // 0b100 = 4

let perms = READ | WRITE;        // 0b011 = 3

// Check a flag:
(perms & READ) !== 0;            // true

// Set a flag:
perms |= EXECUTE;                // 0b111 = 7

// Clear a flag:
perms &= ~WRITE;                 // 0b101 = 5

// Toggle:
perms ^= READ;                   // 0b100 = 4
```

This is exactly how `chmod` works — `7 = rwx`, `5 = r-x`, `0 = ---`.
And how event masks work (`epoll`, `kqueue`), and CPU feature flags,
and HTTP method allowlists in some routers.

## Masking — keep only some bits

```js
const byte = 0xab;             // 0b10101011
byte & 0x0f;                   // 0x0b — low nibble
(byte & 0xf0) >> 4;            // 0x0a — high nibble
byte & 0xff;                   // forces to 8-bit (no-op here)
```

To extract bytes from a 32-bit int:

```js
const n = 0xdeadbeef;
(n >>> 24) & 0xff;   // 0xde
(n >>> 16) & 0xff;   // 0xad
(n >>>  8) & 0xff;   // 0xbe
 n         & 0xff;   // 0xef
```

`>>>` not `>>` so the sign bit doesn't propagate.

## Power-of-two checks

```js
function isPowerOfTwo(n) {
  return n > 0 && (n & (n - 1)) === 0;
}
```

Why it works: a power of two has exactly one bit set. `n - 1` flips
that bit off and turns all lower bits on. `AND` gives zero.

```
n     = 0b01000   (8)
n - 1 = 0b00111   (7)
n & (n-1) = 0     -> power of two
```

The same trick clears the lowest set bit (`n & (n-1)`), which is the
basis of efficient `popcount` (count set bits).

## Alignment

`n & ~(align - 1)` rounds `n` down to a multiple of `align` (when
`align` is a power of two):

```js
const align = 16;
(123 & ~(align - 1));   // 112 — round down
((123 + align - 1) & ~(align - 1));  // 128 — round up
```

Allocators do this constantly. Pointer alignment, page alignment,
cache-line alignment — all the same trick.

## Reading hex dumps

```bash
$ xxd -c 16 packet.bin
00000000: 4745 5420 2f20 4854 5450 2f31 2e31 0d0a  GET / HTTP/1.1..
00000010: 486f 7374 3a20 6578 616d 706c 652e 636f  Host: example.co
00000020: 6d0d 0a0d 0a                             m....
```

Things to spot: `0d 0a` is CRLF (`\r\n`), `20` is space, ASCII letters
are `0x41-0x5a` (uppercase) and `0x61-0x7a` (lowercase). A run of
`00`s is padding or an uninitialized struct field. A magic number at
offset 0 tells you the file type (`7f 45 4c 46` = ELF, `89 50 4e 47`
= PNG, `ff d8 ff` = JPEG).

## When this comes up

You're a working engineer, not a kernel hacker. But:

- Permission bits (`0o755`), feature flags, protocol headers,
  bitmap indexes — all bit-twiddling.
- Debugging a binary protocol with `tcpdump -X` or `xxd`.
- Color manipulation (`(r << 16) | (g << 8) | b`).
- Hash and ID formats — Twitter snowflake IDs pack timestamp, worker,
  and sequence into a single 64-bit int with shifts and masks.

If you can read a hex dump and convert `0xff` to 255 without thinking,
half of the "scary low-level stuff" stops being scary.
