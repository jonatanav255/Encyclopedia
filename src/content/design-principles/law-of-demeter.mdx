# Law of Demeter

> Only talk to your immediate friends. — Ian Holland et al.,
> Northeastern University, 1987

Also called the **Principle of Least Knowledge**. A method `m` on
object `O` should only call methods on:

- `O` itself,
- `O`'s fields,
- objects passed in as parameters,
- objects `m` creates itself.

What it forbids: reaching through one object to call something on
another object that came back from the first. The smell is *chain
calls*:

```ts
order.getCustomer().getAddress().getCity().toUpperCase();
```

`order` is talking to a stranger — `City` — by reaching through
two intermediaries. Three levels of coupling for one operation.

## Why it matters

The depth of the chain is the depth of the coupling. Anything that
changes in `Customer`, `Address`, or `City` can break `order`'s
caller. The signature of `order` doesn't reveal this; you only find
out at refactor time.

A Demeter-respecting version moves the operation to where the data
lives:

```ts
order.shippingCityUpper();   // order asks customer; customer asks
                             // address; details stay localized.
```

The chain still exists at runtime, but each step is between
neighbors that know each other. Replacing `Address.City` with
`Address.Region` only forces `Address` (and maybe `Customer`) to
change, not `order`'s caller.

## "Tell, don't ask"

A related slogan. Instead of asking an object for its state and
acting on it externally, tell it what to do:

```ts
// Asking.
if (user.getAccount().getBalance() > 0) {
  user.getAccount().setBalance(0);
  audit.log('zeroed out balance');
}

// Telling.
user.zeroBalance(audit);
```

The second form keeps the balance-manipulation logic next to the
data it touches; the caller doesn't know about `Account`
internals. Demeter falls out naturally when you write code this way.

## When chains are fine

Demeter is a heuristic, not a law of physics. Two common exceptions:

- **Pipelines of values.** Fluent APIs and method chains on data
  collections aren't Demeter violations because each step returns
  the same kind of object (a builder, a stream, an iterator). The
  "chain" is along one axis, not across unrelated types.

  ```ts
  rows.filter(active).map(byName).slice(0, 10);
  ```

  Three calls; one type lineage. Demeter is comfortable with this.

- **Plain data / DTOs.** A struct with public fields isn't an
  object in the Demeter sense — there's no encapsulation to
  violate. `record.user.name` is reading two field accesses, not
  delegating two method calls. Trying to "fix" this with
  `record.getUserName()` just adds noise.

The Demeter complaint is specifically about chained calls on
*objects that encapsulate behavior*.

## How Demeter shapes design

Following the rule pushes designs in a particular direction:

- **Methods migrate to where the data lives.** Logic ends up on
  the class that owns the relevant fields.
- **Wrapper / facade methods proliferate.** `order.shippingCity()`
  delegates to its customer. The wrapper is the cost.
- **Object graphs flatten.** Deep nesting becomes painful, so you
  end up restructuring or denormalizing the model.

The trade-off: more named methods on each class, fewer cross-object
reaches. For domain models that evolve over time, this is usually
worth it. For tiny scripts, it's overkill.

## A common over-application

Demeter is sometimes read as "no `.` chains at all," producing
absurdities:

```java
// Over-applied: hiding a simple field access behind a method.
double total = order.computePriceFromAllItems();
// when the original was just:
double total = order.items.stream()
  .mapToDouble(Item::price).sum();
```

If the chain is across cooperating types in a clear pipeline, the
fluent form is often clearer. The rule wants to prevent *reach-
through* coupling, not all method chaining.

## Demeter and tests

A Demeter-friendly design is easier to test in isolation. To test
`order.shippingCityUpper()`, you fake `customer` once. The
non-Demeter version requires faking customer *and* its address
*and* its city — three layers of test setup for one behavior.

If you find yourself building deep fake graphs to test something,
that's often a Demeter signal: the production code is reaching
through layers you have to recreate.

## Relationship to other principles

- **Encapsulation**: Demeter is encapsulation enforced at the
  call-site level.
- **SRP**: respecting Demeter pushes operations onto the class
  that owns the data — typically a single-responsibility win.
- **DIP**: Demeter doesn't require interfaces, but interfaces make
  it easier to honor — you talk to one collaborator behind a
  contract, not to its internal graph.
