# Interface Segregation

> Clients should not be forced to depend on methods they do not
> use. — Robert C. Martin

A fat interface — one with many unrelated methods — couples every
client to every method, even those the client never calls. Splitting
the interface into smaller, role-shaped pieces lets each client
depend only on what it actually needs.

## The motivating shape

A `Worker` interface with everything a person at a company might
do:

```ts
interface Worker {
  work(): void;
  eat(): void;
  sleep(): void;
}
```

A `Robot` implements `Worker` — but robots don't eat or sleep:

```ts
class Robot implements Worker {
  work() { /* ... */ }
  eat()  { throw new Error('robots do not eat'); }
  sleep(){ throw new Error('robots do not sleep'); }
}
```

Now every consumer of `Worker` has methods that may throw, even if
they only meant to call `work()`. Splitting:

```ts
interface Workable { work(): void; }
interface Eater    { eat(): void; }
interface Sleeper  { sleep(): void; }

class Robot  implements Workable {}
class Human  implements Workable, Eater, Sleeper {}
```

A function that only schedules work takes `Workable`. It can accept
robots without contortions, and it doesn't need to know about
eating.

## Why fat interfaces hurt

- **Forced no-op implementations.** Implementers either throw or
  silently no-op for methods that don't apply. Both are lies the
  type system can't catch.
- **Caller coupling.** Anything that imports `Worker` for `work()`
  is now also coupled to `eat()` and `sleep()` — change those and
  the whole transitive client set may rebuild and re-test.
- **Mocking pain.** Writing a fake `Worker` for a test that only
  needs `work()` still requires stubbing `eat` and `sleep`.

## "Role interfaces" — Fowler's framing

Martin Fowler distinguishes:

- **Header interfaces** — extracted from a class, mirror its full
  public surface. Often fat. (`UserRepository` with 30 methods.)
- **Role interfaces** — extracted from a client, contain only the
  methods that client calls. Small. (`UserLookup` with one
  `findById`.)

ISP is the rule that says: prefer role interfaces. The class can
implement many of them; each client depends only on its role.

This also makes the codebase **easier to evolve**. Adding a method
to a role interface only touches the implementers and the clients
that asked for it.

## ISP in structurally typed languages

In Go and TypeScript, this principle is almost free because of
structural typing.

```go
// Go: small interfaces composed at the call site.
type Reader interface { Read(p []byte) (n int, err error) }
type Closer interface { Close() error }
type ReadCloser interface { Reader; Closer }
```

A function that only needs reading takes `Reader`. Anything with a
`Read` method satisfies it; no inheritance declaration required.
The Go standard library is a tour of role interfaces — `io.Reader`,
`io.Writer`, `io.Closer`, `io.Seeker`, all small and composable.

TypeScript has the same property: a function typed to accept
`{ findById(id: string): User }` accepts any object with that
shape.

In nominally typed OO languages (Java, C#), ISP requires more
explicit declarations: each implementer says which interfaces it
satisfies.

## When *not* to split

Two small interfaces are not better than one when they always
travel together:

```ts
// Probably over-segregated.
interface NameSetter   { setName(n: string): void; }
interface NameGetter   { getName(): string; }
```

If every client uses both, you've added types without reducing
coupling. A single `Named` interface is clearer. ISP is about
*independently varying* roles, not splitting for the sake of small
files.

A sanity check: can you imagine a real implementer that has one of
the interfaces and not the other? If not, the split is theoretical
and probably premature.

## ISP and the database repository

A common bad pattern: a `UserRepository` with `findById`,
`findByEmail`, `searchByName`, `listAll`, `create`, `update`,
`delete`, `archive`, plus 12 more. Every service that touches users
imports the whole thing.

Better: small role interfaces (`UserLookup`, `UserCreator`,
`UserArchiver`) — or, in many cases, no shared "repository" type at
all. Each service defines the *role interface it needs*, and any
implementer that exposes those methods satisfies it.

This is the same idea as Mark Seemann's framing: **let the consumer
define the interface, not the provider.**

## Relationship to other principles

- **SRP** at the class level often produces ISP at the interface
  level for free: a class with one responsibility tends to have a
  small public surface.
- **DIP** uses interfaces as inversion points; smaller interfaces
  are easier to substitute, easier to fake in tests.
- **LSP**: smaller interfaces are easier to subtype honestly.
  Splitting a fat `List` into `Readable` and `Writable` lets you
  give read-only lists a type that doesn't lie.
