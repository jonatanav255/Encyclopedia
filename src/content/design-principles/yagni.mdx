# YAGNI — You Aren't Gonna Need It

Coined inside Extreme Programming in the late 1990s. The full
slogan, as Ron Jeffries used to put it: **"Always implement things
when you actually need them, never when you just foresee that you
need them."**

YAGNI is the temporal cousin of KISS. KISS says *don't make today's
design more complex than today's problem*. YAGNI says *don't build
for tomorrow's hypothetical problem at all*.

## Why "I'll need this later" is usually wrong

Several reasons that hold up across decades of empirical
observation (Fowler has a detailed essay on this):

- **The foreseen need doesn't arrive.** Estimates over which
  features matter are routinely 50%+ off. Code built for the
  unneeded feature is dead-weight you still maintain.
- **The foreseen need arrives in a different shape.** When the
  feature does come, the requirements have shifted; the
  pre-built abstraction fits the imagined version, not the real
  one. You refactor anyway.
- **Cost of carry.** Until the feature lands, the speculative code
  costs you in build times, test runs, surface area to read, and
  bugs introduced by code that isn't being used (and therefore
  isn't being battle-tested).
- **Delay shrinks the option's value.** The right time to make a
  design decision is when you have the most information about what
  you need. That's not "now," it's "later."

The combination: you pay carrying cost on something whose eventual
shape you can't predict.

## What YAGNI is *not* arguing against

YAGNI is misread as "don't think ahead." That's not it.

- **Architectural decisions** that are expensive to change (a
  database choice, an authentication model, a public API contract)
  still need forethought. The cost of doing them later isn't just
  "now I write the code"; it's a migration on production data.
- **Cross-cutting concerns** like observability, error handling,
  and security are not features added at the end. They're shapes
  the whole codebase takes.
- **Reversibility matters.** YAGNI is most defensible for
  *reversible* decisions: anything you can rip out and replace
  cheaply. For irreversible decisions, more upfront thought is
  warranted.

The frame: **be conservative about adding code; be thoughtful about
choosing shapes.**

## Common YAGNI violations

- An admin dashboard with role-based permissions for "future
  multi-tenant support" when the app has one customer.
- A plugin system for extension points no one has asked to extend.
- A queue and worker between two services that are sized for
  in-process call volume.
- A `BaseEntity` abstract class with fields that "every entity will
  probably need eventually."
- Configuration flags that ship without any code path actually
  using their non-default values.
- Internationalization scaffolding for a product that will be
  English-only for the next two years.

In each case, the speculative code adds cost now for a benefit
that's uncertain and far off.

## Applying YAGNI without being shortsighted

Some practical tactics:

- **Add the feature when the second request arrives**, not the
  first imagined one. The first request is information; the second
  is a pattern.
- **Leave seams, not implementations.** A function call where you
  *might* later add caching is much cheaper than building a cache
  layer you don't need yet. The seam is free; the cache isn't.
- **Use the type system / tests as a memory aid.** A `TODO` plus a
  failing test marker is enough to remember "we'll need this when
  X happens" — you don't have to build it.
- **Make decisions late.** "We'll switch to Kafka when our queue
  depth justifies it" is a more defensible position than "we
  started with Kafka because we knew we'd grow."

## YAGNI and refactoring

YAGNI works *because* you're willing to refactor. If touching code
is expensive (because of poor test coverage, fragile design, or
team ownership friction), the option-value of waiting drops, and
"build it now while I'm in here" starts to look reasonable. So
YAGNI assumes a baseline of refactor-friendliness.

If your codebase is hostile to change, the answer isn't "stop using
YAGNI." It's "fix the refactor-friendliness," because losing that
ability means every decision becomes a one-shot.

## YAGNI as a team norm

In code review, YAGNI is the principle that gives you license to
say: **"this is speculative — let's remove it until we need it."**
The author doesn't have to defend the speculation; the burden of
proof is on the addition.

This norm is more important than any individual instance, because
speculative code accumulates. A codebase with a strong YAGNI
culture stays surprisingly small.

## Relationship to other principles

- **KISS**: YAGNI prevents the kind of premature complexity KISS
  warns about.
- **DRY**: don't deduplicate code that doesn't share knowledge
  *yet*. The shared abstraction might not match when the third
  caller arrives.
- **OCP**: don't add extension points for variants that don't
  exist. The wrong axis of variation is worse than no axis.
