# Composition over Inheritance

> Favor object composition over class inheritance. — *Design
> Patterns*, Gamma/Helm/Johnson/Vlissides (GoF), 1994

Inheritance models *is-a*: a `Dog` is an `Animal`. Composition
models *has-a*: a `Car` has an `Engine`. The principle says: when
both are plausible, prefer composition — has-a relationships are
more flexible than is-a hierarchies, with fewer surprises down the
road.

## What goes wrong with inheritance

Deep class hierarchies have known pathologies:

- **Brittle base class problem.** Changing a method on a base
  class can silently break subclasses that overrode (or relied on
  the old behavior of) related methods. The coupling is implicit.
- **The diamond / multi-axis problem.** Real domains usually have
  multiple independent axes of variation (size, color, behavior,
  pricing tier). Single inheritance forces you to pick one axis for
  the hierarchy; the others have to be modeled some other way.
- **Tight coupling, fixed at compile time.** A subclass is welded
  to its parent. You can't pick a different behavior at runtime.
- **Liskov hazards.** Subclasses that "almost fit" the base
  contract break callers in non-obvious ways.

Composition sidesteps all four:

- The composed object exposes a well-defined interface; replacing
  it doesn't ripple into your class internals.
- You can compose along several axes — a `Car` can hold an
  `Engine`, a `Transmission`, a `BatteryPack`, all independent.
- Composition is configurable at runtime — pass in a different
  `Engine` to get a different car.
- The contract is the composed interface, not the parent's full
  surface.

## A worked example

A naive inheritance hierarchy for game characters:

```ts
class Character { walk() { /* ... */ } }
class FlyingCharacter extends Character { fly() { /* ... */ } }
class SwimmingCharacter extends Character { swim() { /* ... */ } }
// Now we need a flying-and-swimming duck. ???
```

Single inheritance can't express "flies and swims" without messy
multi-inheritance, copy-paste, or moving everything onto the base.

Composition:

```ts
interface Movement { perform(): void; }
class Walk implements Movement { perform() {/*...*/} }
class Fly  implements Movement { perform() {/*...*/} }
class Swim implements Movement { perform() {/*...*/} }

class Character {
  constructor(private moves: Movement[]) {}
  move() { for (const m of this.moves) m.perform(); }
}

const duck = new Character([new Walk(), new Fly(), new Swim()]);
```

Each behavior is a separate, replaceable component. New behaviors
are new classes; new combinations don't require new subclasses.

This is the **Strategy pattern**, and it's the most common
composition-over-inheritance refactor in practice.

## When inheritance is genuinely the right tool

The principle is "favor," not "never use." Inheritance is fine when:

- The relationship is genuinely *is-a* in both data and behavior,
  not just structure. `LinkedHashMap` truly is a `HashMap`.
- The base class is small, stable, and well-tested.
- The hierarchy is shallow — one or two levels at most.
- Subclasses share a real LSP-respecting contract.

Frameworks often legitimately ask you to extend a base class
(`React.Component`, `Activity`, etc.), and that's fine — the
framework owns the contract and tests it.

The cases where inheritance *fails* are usually:

- Reuse-by-inheritance: extending a class just to grab a few
  utility methods.
- Deep hierarchies (3+ levels). Each level adds invisible coupling.
- Inheritance across module boundaries you don't own.

## "Mixins" and traits

Some languages soften the rule with mechanisms that let you reuse
behavior without classical single-inheritance:

- **Traits (Rust)**, **interface default methods (Java 8+)**,
  **mixins (Ruby)**. Add behavior without a deep hierarchy.
- **Composition via delegation** (Kotlin's `by` keyword). The
  class holds another object and forwards specific methods.

These give you many of composition's benefits with less boilerplate.
They're closer in spirit to composition than to classical
inheritance.

## In functional style

Composition is already the default in functional code: behaviors
are functions; you build complex behaviors by composing simpler
ones (`pipe(parse, validate, save)`). There's no "inheritance" to
prefer *against* — the principle becomes trivially true.

This is one reason functional codebases tend to feel less tangled
than deeply inherited OO ones: there's no implicit method-override
graph to keep in your head.

## The trade-off composition asks for

Composition isn't free. You pay in:

- **More files / types.** Each component is its own class or
  function.
- **More wiring.** Someone has to instantiate the parts and
  assemble them — usually a factory or DI container.
- **A bit more verbosity.** `character.move()` -> `for (m of moves)
  m.perform()` is longer than a single overridden method.

The trade is usually worth it for non-trivial systems because the
flexibility compounds: every new feature combines existing
components instead of forcing a new subclass.

## Relationship to other principles

- **OCP**: composition is the standard tool for extension. New
  behaviors are new components, not edits to base classes.
- **DIP**: composition is natural with dependency injection — you
  hand the composed objects in.
- **LSP**: if you're tempted to use inheritance only to share code,
  the result rarely respects LSP. Composition removes the
  temptation.
