# Liskov Substitution

> If `S` is a subtype of `T`, then objects of type `T` may be
> replaced with objects of type `S` without altering the
> correctness of the program. â€” Barbara Liskov, 1987

The most precise of the SOLID letters. LSP is a *behavioral*
contract, not a syntactic one â€” a subclass that compiles cleanly
can still violate LSP if it changes what callers depend on.

## The canonical counterexample: Square / Rectangle

Looks like inheritance:

```ts
class Rectangle {
  width: number; height: number;
  setWidth(w: number)  { this.width  = w; }
  setHeight(h: number) { this.height = h; }
  area() { return this.width * this.height; }
}

class Square extends Rectangle {
  setWidth(w: number)  { this.width = w; this.height = w; }
  setHeight(h: number) { this.width = h; this.height = h; }
}
```

A caller written against `Rectangle`:

```ts
function resize(r: Rectangle) {
  r.setWidth(5);
  r.setHeight(4);
  assert(r.area() === 20);   // holds for Rectangle, fails for Square
}
```

`Square` passes the type checker but breaks an invariant the caller
relied on (`setWidth` doesn't affect height). LSP violation. The
fix isn't "test for Square inside `resize`" â€” that defeats
polymorphism. The fix is that `Square` is not a behavioral subtype
of `Rectangle`; modeling it as one was wrong.

## The behavioral rules (informal)

A subtype can:

- **Weaken preconditions** (accept inputs the base rejected).
- **Strengthen postconditions** (return guarantees the base didn't).
- **Preserve invariants** of the base.
- **Not throw new kinds of exceptions** the base didn't declare.

A subtype must *not*:

- Strengthen preconditions (require more than the base did).
- Weaken postconditions (return less than the base promised).
- Break invariants.
- Make caller-visible state worse (e.g., closing a connection the
  base wouldn't have closed).

In one sentence: **a subclass may do more, but never less.**

## Why this is hard

Inheritance hierarchies make it easy to express *is-a-kind-of*
relationships that aren't actually substitutable:

- A `ReadOnlyList` extending `List` and throwing on `add()`. The
  base says "you can add"; the subtype says "no, you can't." That's
  strengthening preconditions â€” LSP violation. The Java
  `Collections.unmodifiableList` family has this issue and it's
  documented as a known anti-pattern.
- A `Penguin` extending `Bird` with a `fly()` method that throws.
  Same shape.
- A `Stream` subclass whose `read()` returns nulls under conditions
  the base didn't allow.

The pattern is the same: the subclass *narrows* what callers can
do, even though the type system says it widens.

## How to design *with* LSP rather than against it

Two practical moves:

**Prefer composition or interface segregation.** Instead of
`ReadOnlyList extends List`, define `Readable` and `Writable` as
separate interfaces; `ReadOnlyList` implements only `Readable`. The
type system now correctly refuses code that tries to `.add()` to a
read-only list.

**Make invariants part of the interface, not the implementation.**
If `Account.withdraw(n)` must never produce a negative balance, the
base class enforces it (or documents it as a precondition). A
subclass cannot then *relax* it.

## LSP in modern languages

Languages with structural typing or traits change the picture:

- **TypeScript** has structural compatibility. There's no `extends`
  required â€” anything with the right shape is assignable. LSP still
  applies to *behavior*, but you don't have to set up a class
  hierarchy to violate it.
- **Rust** doesn't have classical inheritance; traits enforce a
  behavior contract and (with default implementations) you can't
  weaken postconditions. LSP-style violations are mostly impossible
  at the type level.
- **Go**'s interface satisfaction is also structural. Same
  conclusion as TS.

In all three, the LSP question becomes: "does this
implementation/trait-impl actually meet the documented contract of
the interface?" rather than "is `B extends A` semantically
honest?".

## Tests as contracts

A well-tested base class typically has tests phrased in terms of
its interface ("any `Stream` shouldâ€¦"). Subclasses must pass the
same suite. This is a cheap LSP enforcement mechanism: if the
subclass can't pass the base's tests, the inheritance is wrong.

## When LSP is over-applied

The rule does not say "never specialize." It says specialization
must preserve the *observable contract*. A subclass can:

- Override a method for efficiency, as long as the output is
  semantically the same.
- Add new methods or fields.
- Tighten internal invariants the base didn't expose.

LSP only kicks in for what callers can see through the base type.

## Relationship to other principles

- **OCP** depends on LSP being respected â€” extension points that
  accept any subtype are only safe if subtypes really substitute.
- **ISP**: splitting interfaces is often the right LSP fix. A
  smaller interface is easier to subtype honestly.
- **Composition over inheritance**: the easiest way to avoid LSP
  violations is to not subtype at all when the relationship isn't
  cleanly behavioral.
