# Open/Closed Principle

> Software entities should be open for extension, but closed for
> modification. — Robert C. Martin (1996)

The principle itself was introduced by **Bertrand Meyer** in
*Object-Oriented Software Construction* (1988), in an
inheritance-centric framing: a class is "closed" because clients
depend on its current behavior, and "open" because new classes can
use it as a parent. The pithy modern wording above is Martin's
reformulation, which generalized the idea beyond inheritance.

A module is **open for extension** if you can add new behavior
without changing the module's existing code. It's **closed for
modification** if its existing clients don't need to recompile or
retest when the new behavior is added.

The shape this principle pushes you toward: a stable core that
accepts plug-in variants through a well-defined extension point.

## Why the rule exists

Two kinds of pain motivate OCP:

- **Ripple edits.** A `switch` over shape types that lives in five
  different methods. Adding `Hexagon` means editing all five. Easy
  to miss one. Easy to break the others.
- **Re-test cost.** Editing existing code re-opens the question
  "did I break the existing callers?" New code added through a
  documented extension point only re-opens "does the new variant
  work?"

The classic motivating example: a `Renderer` with a switch on
shape type.

```ts
class Renderer {
  draw(shape: Shape) {
    switch (shape.kind) {
      case 'circle':    drawCircle(shape);    break;
      case 'square':    drawSquare(shape);    break;
      case 'triangle':  drawTriangle(shape);  break;
    }
  }
}
```

Adding `Hexagon` forces an edit to `Renderer.draw` and any other
switch-on-kind site. OCP-shaped version:

```ts
interface Shape { draw(): void; }

class Circle   implements Shape { draw() { /* ... */ } }
class Square   implements Shape { draw() { /* ... */ } }
class Hexagon  implements Shape { draw() { /* ... */ } }

class Renderer {
  render(shapes: Shape[]) { for (const s of shapes) s.draw(); }
}
```

`Renderer` is now closed — adding `Hexagon` doesn't touch it.

## The trap: premature OCP

OCP is the principle most often applied prematurely. Every
extension point is a guess about what will vary. Guesses are often
wrong, and the cost of a wrong guess is *worse* than a switch
statement:

- An interface with one implementation is harder to read than a
  concrete class.
- An interface shaped around the *wrong* axis of variation forces
  you to refactor later anyway, on top of the abstraction.

A heuristic from Sandi Metz: **prefer duplication over the wrong
abstraction.** Wait until you see the second or third variant
before introducing the extension point. The variants tell you what
the axis of variation actually is.

## Forms OCP takes in modern code

The textbook form is "inheritance plus polymorphism," but in
practice OCP shows up as:

- **Strategy pattern.** Pass a function (or function-shaped object)
  to vary one step of an algorithm.
- **Plug-in registries.** A map from key to handler that callers
  add to without editing dispatch.
- **Tagged-union + exhaustiveness.** In TypeScript or Rust, a
  discriminated union with a `never` fallback gives you OCP-like
  safety while still using a switch — the compiler errors when a
  new variant isn't handled.
- **Hooks/middleware.** Frameworks like Express or Vite expose
  extension via middleware/plugin lists, not by editing the core.

The Rust/TS variant is interesting because it inverts the OCP
trade-off: a `switch` with exhaustiveness *is* a stable extension
point, because adding a variant is a compile error in every call
site you forgot. You get the "no missed edits" property without the
interface.

## A subtler version of "closed"

OCP isn't just about *source* not changing. It's about **the
contract not changing.** A class can have its private internals
edited freely as long as its public behavior — what callers rely
on — doesn't shift. So a refactor that improves performance without
changing return values doesn't violate OCP. A "fix" that quietly
changes the meaning of `discount()` for existing callers does,
extension point or not.

This is also where OCP meets **semver**: a "closed" module is one
where you can ship new minor versions without breaking consumers.

## When to break OCP deliberately

- **The cost of the abstraction exceeds the cost of editing.**
  Three places use a constant; just change the constant.
- **The "extension" is one-off.** A bug-fix conditional in a single
  call site doesn't need to be a Strategy.
- **The variants don't share a meaningful interface.** Forcing
  unrelated handlers to implement a common `Handler` interface
  produces empty methods and lies.

## Relationship to other principles

- **SRP** makes OCP cheap: a one-reason class is small enough to
  shape into an extension point without bloating other concerns.
- **LSP** is the *correctness* half of OCP: extension is only safe
  if subtypes actually behave as the base interface promises.
- **DIP** often appears alongside OCP — the stable core depends on
  the abstraction; the variants implement it.
