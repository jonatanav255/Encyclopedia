# DRY — Don't Repeat Yourself

> Every piece of knowledge must have a single, unambiguous,
> authoritative representation within a system. — Andy Hunt &
> Dave Thomas, *The Pragmatic Programmer*

DRY is the most misquoted principle in software. The popular
version — "never type the same line twice" — leads to bad
abstractions and unnecessary coupling. The original is about
**knowledge**, not syntax.

## Knowledge vs code

Two functions can look almost identical and still represent two
different pieces of knowledge:

```ts
function priceForCustomer(c: Customer) {
  return c.basePrice * (1 - c.discount);
}

function payoutForEmployee(e: Employee) {
  return e.salary * (1 - e.taxRate);
}
```

The shape is the same: `x * (1 - y)`. The *knowledge* isn't —
one is pricing policy, the other is payroll policy. Extracting a
shared `multiplyByOneMinus(x, y)` couples two rules that may evolve
independently. The next change to the pricing model now has to
consider whether it breaks payroll.

The real DRY violation is when **the same fact** is encoded in two
places. Examples:

- A regex for valid email lives in three forms.
- Tax brackets exist in code *and* in a docstring *and* in a CSV
  used by the report generator.
- A database column name is duplicated in an ORM mapping and a raw
  SQL query.

Each is one piece of knowledge in multiple places. Change one,
forget the others, get bugs. That's the pain DRY treats.

## The rule of three

A pragmatic heuristic widely attributed to *Refactoring* by Martin
Fowler:

1. **First time** you write something — write it.
2. **Second time** you write something similar — note the duplication.
3. **Third time** — extract it, *if* the three uses really are the
   same idea.

Premature extraction at step two often produces an abstraction
fitted to two examples that doesn't generalize when the third
arrives. Worse: you've coupled two callers that didn't need to
share a fate.

## The wrong-abstraction tax

Sandi Metz's much-cited line: **prefer duplication over the wrong
abstraction.** Three near-copies are honest; you can edit one and
leave the others alone. A wrong abstraction is a *lie* — it
suggests these things are the same when they aren't, and editing
the abstraction quietly affects callers that have nothing to do
with the change.

Symptoms of a wrong abstraction:

- The shared function takes many flags or modes to handle each
  caller's special case.
- Callers pass weird defaults to make the abstraction fit.
- The abstraction's name describes mechanics ("processItem") rather
  than meaning ("calculateTax").

In these cases, **inlining** (collapsing the abstraction back into
its callers) is often the right move. Duplication you can fix later
is cheaper than coupling you can't.

## Where DRY pays off cleanly

These are nearly always real DRY violations worth fixing:

- **Constants** (timeouts, URLs, magic numbers). One definition,
  imported everywhere.
- **Schema definitions** (types/validators/SQL). Tools like Zod,
  Prisma, or sqlc give you one source of truth.
- **Configuration**. Environment-derived values should live in one
  module that everyone imports.
- **Documentation that derives from code**. Generate it; don't
  retype it.

## DRY across boundaries

DRY *inside a module* is much safer than DRY *across modules* or
*across services*. Two services that share code today may need to
deploy independently tomorrow; deduplicating between them creates a
hidden coupling that breaks the autonomy you bought by splitting
them.

A common pattern: keep duplication between services, and use code
generation or schema sharing for the parts that *must* be aligned
(API contracts, message schemas).

## What DRY is not

- **Not "small files / short functions."** A 30-line function is
  not a DRY violation by virtue of being long.
- **Not the same as "abstract everything."** Abstraction has costs:
  indirection, harder reading, harder debugging. DRY doesn't ask
  you to pay that cost when there's no shared knowledge to
  consolidate.
- **Not a substitute for naming.** Sometimes the duplication is
  just three uses of an idea that doesn't have a name yet.
  Sometimes naming the idea (without extracting it) is enough.

## Relationship to other principles

- **KISS**: DRY done wrong violates KISS — premature abstraction is
  more complex than the duplication it replaces.
- **SRP**: if you're tempted to extract code that comes from
  different responsibilities, SRP says: don't. They change for
  different reasons.
- **YAGNI**: DRY about *current* knowledge, not knowledge you
  imagine you'll have later.
