# SOLID

Five object-oriented design principles assembled (and named) by
Robert C. Martin in the early 2000s, drawing on earlier work by
Bertrand Meyer, Barbara Liskov, and others. The acronym is a
mnemonic, not a ranking — the five are independent rules of thumb.

| Letter | Principle | One-line gist |
| --- | --- | --- |
| S | Single Responsibility | A class should have one reason to change. |
| O | Open/Closed | Open for extension, closed for modification. |
| L | Liskov Substitution | Subtypes must be usable wherever the base is. |
| I | Interface Segregation | Many small interfaces beat one fat one. |
| D | Dependency Inversion | Depend on abstractions, not concretions. |

## What SOLID is — and isn't

SOLID is a set of *heuristics* for managing change in object-oriented
codebases. It pre-dates microservices, modern functional languages,
and most of today's framework conventions, and it shows: every
principle is phrased in terms of classes, interfaces, and
inheritance.

What it's good for:

- **Code that will live a long time** and absorb changing
  requirements. The pain SOLID treats — rippling edits, fragile
  inheritance hierarchies, modules that can't be tested in
  isolation — is real and well-known.
- **Talking about design.** Even if you don't apply every letter,
  the shared vocabulary ("that violates ISP", "this is a Liskov
  problem") is useful in reviews.

What it isn't:

- **Not a checklist.** "I applied all five" doesn't mean the design
  is good. You can satisfy SOLID with an overengineered mess of
  interfaces.
- **Not language-neutral.** In a language with first-class functions
  and structural typing (TypeScript, Go), several letters collapse
  or look different. ISP and DIP get easier; OCP often becomes
  "pass a function."
- **Not a substitute for KISS/YAGNI.** SOLID can be applied
  prematurely. Three concrete classes are often clearer than three
  interfaces plus three implementations.

## Common misreadings

- **SRP ≠ "one method per class."** It's about *reasons to change*,
  not method count.
- **OCP ≠ "never edit code."** It's about not editing *stable* code
  to add new variants.
- **LSP ≠ "inheritance is fine if signatures match."** Behavior must
  also match — strengthening preconditions or weakening
  postconditions in a subclass breaks LSP even when the compiler
  accepts it.
- **ISP ≠ "split everything."** A coherent interface is fine; the
  rule targets *fat* interfaces with unrelated capabilities.
- **DIP ≠ "use a DI framework."** Dependency inversion is a design
  shape; dependency injection is a mechanism. You can have one
  without the other.

## A worked example

A `ReportService` that pulls rows from a database, formats them as
PDF, and emails them.

```ts
class ReportService {
  generateAndEmail(userId: string) {
    const rows = db.query('SELECT * FROM events WHERE user=?', userId);
    const pdf = buildPdf(rows);
    smtp.send(userId, pdf);
  }
}
```

Three reasons to change (schema, PDF layout, email transport) live
in one class — SRP violation. The class hard-codes `db`, `smtp`, and
the PDF builder — DIP violation, and impossible to test without
hitting all three. Adding a CSV export means editing the method —
OCP violation.

A SOLID-shaped version:

```ts
interface RowSource { fetch(userId: string): Row[]; }
interface Formatter { format(rows: Row[]): Buffer; }
interface Sink { deliver(userId: string, payload: Buffer): void; }

class ReportService {
  constructor(
    private source: RowSource,
    private formatter: Formatter,
    private sink: Sink,
  ) {}
  run(userId: string) {
    this.sink.deliver(userId, this.formatter.format(this.source.fetch(userId)));
  }
}
```

Each collaborator is an interface; concrete implementations
(`PostgresEventSource`, `PdfFormatter`, `SmtpSink`, `CsvFormatter`,
`S3Sink`) are wired in at the edges. Tests pass in-memory fakes.
Adding CSV adds a class instead of editing `ReportService`.

The cost: more types, more wiring, more files. If you only ever
need PDF+SMTP, the first version is fine. **SOLID buys flexibility
you pay for upfront.** Apply it when the flexibility is likely to
pay back.

## Reading order on this site

If you're new to these, the pages roughly build on each other:

1. **Single Responsibility** — the foundation.
2. **Open/Closed** — needs SRP to make sense.
3. **Liskov Substitution** — about inheritance behavior.
4. **Interface Segregation** — about interface shape.
5. **Dependency Inversion** — about direction of imports.

Then the non-SOLID principles (DRY, KISS, YAGNI, Demeter,
composition-over-inheritance, separation of concerns) are
complementary, not derivative.
