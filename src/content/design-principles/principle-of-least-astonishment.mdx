# Principle of Least Astonishment

Sometimes credited to Geoffrey James (*The Tao of Programming*),
sometimes to the PL/I language committee in the 1970s. The phrasing
varies; the rule is the same:

> A component of a system should behave in a way that most users
> will expect it to behave. The behavior should not astonish or
> surprise users.

The principle is about **interface design** — both UIs and APIs.
"Users" includes programmers calling your library, operators reading
your logs, and end-users clicking your buttons.

## What "astonishment" actually means

Astonishment is a *gap* between what someone expected and what
happened. Two interfaces with identical behavior can have different
amounts of astonishment depending on the surrounding conventions and
the user's mental model.

A non-exhaustive list of common surprises:

- `delete user` quietly cascades and deletes their orders too.
- `length()` on a "string" returns bytes, not characters.
- `sort()` mutates the array instead of returning a new one.
- `setLogLevel("debug")` also turns on remote telemetry.
- A `GET` endpoint changes server state.
- A button labelled "Save" navigates away from the page.
- A function named `validate(x)` throws on invalid input instead of
  returning a boolean.
- A library that politely calls `process.exit(1)` from a parser
  function when input is malformed.

In each case the *behavior* may be defensible; the *naming and
context* set an expectation that the behavior breaks.

## Where astonishment comes from

The frequent root causes:

- **Names that don't match behavior.** `getUser()` triggering a
  network call. `parse()` writing to disk. `isValid()` having side
  effects.
- **Hidden side effects.** A property *getter* (`user.profile`) that
  starts a fetch and caches the result. A "config" function that
  silently mutates `process.env`.
- **Surprising defaults.** `JSON.parse('null')` is a valid call
  returning `null`. `Number('')` returns `0`, not `NaN`. Defaults
  that mean "off" in one place and "on" in another.
- **Local conventions that fight the language's.** A method named
  `next()` on a class that's not an iterator. An `async` function
  that returns a plain value instead of a Promise.
- **Inconsistency across siblings.** Two endpoints in the same API,
  one returning `{ data: [...] }` and the other returning `[...]`
  directly. Two CLI commands, one taking `--verbose` and the other
  `-v`.

## How to apply it

The principle is easy to invoke and hard to operationalize.
Concrete moves that help:

- **Name for behavior, not implementation.** `parse` reads and
  returns; if it also writes, it's `parseAndSave`. `get` is cheap and
  pure; if it makes a network call, it's `fetch`. The most expensive
  thing the function does should be visible in the name.
- **Prefer explicit options to clever inference.** A function that
  "auto-detects" the format from the input is less astonishing as
  `parse(input, { format: 'json' })`. The auto-detection logic is the
  surprise.
- **Don't break language conventions.** In JS, methods named with
  past tense return new collections (`map`, `filter`, `sorted`); ones
  named with the verb form mutate (`sort`, `push`). Match the
  convention even if your code "knows better."
- **Make side effects visible.** If a function writes to disk, sends
  an email, mutates a parameter, or starts a timer, its name and
  documentation should say so. The Haskell community formalized this
  with the `IO` type; lower-tech versions like a `*Async`, `*Sync`,
  `*Effect` suffix work.
- **Pick one shape and use it everywhere.** Sibling APIs should
  return the same envelope shape, accept the same error format, use
  the same naming convention. Inconsistency is astonishment with no
  upside.
- **Steal expectations from the surrounding ecosystem.** In a Rails
  app, a method named `find` raises on missing; `find_by` returns
  nil. New code in that app should follow the same pattern — even if
  you personally would have done it differently. The audience's
  mental model is the standard, not your taste.

## The hard part: whose expectations?

The principle is local to a *user population*. "Expected" by a
Python programmer can be "astonishing" to a Haskell one. A REST API
shape that astonishes a GraphQL user can be exactly what a REST user
expected.

A few resolution heuristics:

- **The largest audience wins for public APIs.** If your library is
  used across language ecosystems, conform to the most common
  expectation (usually: idiomatic in the host language).
- **Internal conventions beat external ones for internal code.**
  Your codebase's existing patterns are what new contributors will
  read first. Match them before reaching for outside idioms.
- **Document the surprises you keep.** Sometimes the unsurprising
  behavior is genuinely worse (slower, less safe, harder to extend).
  In that case, keep the surprising behavior *and write a paragraph
  next to it explaining why*. The cost of astonishment is paid in
  every reader; the documentation pays it back once per reader.

## Related principles

- **Convention over Configuration** (Rails, etc.) is a special case
  of POLA at the framework level: the conventions ARE the
  expectations. Following them eliminates astonishment for everyone
  who knows the framework.
- **Composition over inheritance** reduces astonishment by removing
  one of inheritance's prime sources: behavior inherited from a
  base class without local evidence.
- **Fail-fast** (covered in its own page) is sometimes in tension
  with POLA — a system that crashes on unexpected input is
  surprising to a caller expecting forgiveness, but the alternative
  (silently producing wrong output) is usually worse. Pick a
  failure mode and document it.

<Callout type="info">
The simplest day-to-day test: **read your code's names without
opening the bodies. Does the behavior match what the name promises?**
If you'd be surprised by what's inside, your callers will be too.
</Callout>
