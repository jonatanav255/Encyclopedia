# Single Responsibility

> A class should have only one reason to change. — Robert C. Martin

The most quoted and most misread of the SOLID letters. "Single
responsibility" sounds like "one method" or "one job," but the
operative phrase is **reason to change**. SRP is about coupling
unrelated *axes of change* into the same module.

## The Martin definition

The motivation comes from organizational pain. Suppose an
`Employee` class has three methods used by three departments:

- `calculatePay()` — used by Finance.
- `reportHours()` — used by HR.
- `save()` — used by the DBA team.

Three teams now share one file. A Finance change to pay rules and
an HR change to time-off accrual can land in the same class on the
same day, merge-conflict, and one team can silently break the other.
The class has **three reasons to change**, owned by three different
stakeholders. That's the SRP violation.

The fix: split the data from the operations, or split the operations
along the lines of who asks for them:

```ts
class EmployeeData { /* fields only */ }

class PayCalculator     { compute(e: EmployeeData): Money { /* ... */ } }
class HoursReporter     { hours(e: EmployeeData): number  { /* ... */ } }
class EmployeeRepository{ save(e: EmployeeData): void     { /* ... */ } }
```

Each class now has one stakeholder, one reason to change.

## What SRP is *not*

- **Not "one method per class."** A `UrlParser` with `parse`,
  `parseQuery`, and `parseFragment` has one responsibility (parsing
  URLs). Three methods, one reason to change.
- **Not "small files."** A 400-line class with one cohesive purpose
  is fine. A 40-line class that mixes parsing and persistence is
  not.
- **Not just for OO classes.** A module, a function, a service —
  the same logic applies. A function that fetches *and* formats
  *and* emails is doing three things.

## A practical sniff test

If you can finish the sentence "this class changes when…" with more
than one clause joined by *and*, suspect a violation:

- "…the API contract changes **and** when the database schema
  changes **and** when the email template changes."

Each *and* is a candidate seam. The next question is whether the
seams correspond to *different stakeholders or different deploy
cadences*. If they do, splitting pays off. If all three change
together because one team owns all three, splitting is paperwork.

## Cohesion vs coupling

SRP is the class-level statement of an older idea: **high cohesion,
low coupling.** A cohesive module's pieces all serve one purpose; a
loosely coupled module doesn't reach into its neighbors. SRP
optimizes cohesion *by* lowering coupling between unrelated
concerns.

A useful frame: ask whether two methods would naturally evolve
together or apart. `formatDate` and `parseDate` evolve together.
`formatDate` and `sendInvoiceEmail` do not.

## When to ignore SRP

Two common cases where the principle steers you wrong:

- **Tiny scripts and prototypes.** A 60-line CLI that fetches, parses,
  and prints can stay one file. SRP overhead has to earn its keep.
- **"Data classes" / DTOs.** A struct with fields and a couple of
  derived getters is not "doing too much" by having both
  serialization and validation helpers attached. It depends on
  whether each helper has a different owner.

The pragmatic version of the rule: **separate things that change
for different reasons, owned by different people, on different
schedules.** That's what makes the principle a force-multiplier
rather than busywork.

## Relationship to other principles

- **Open/Closed**: easier to honor when each class has one
  responsibility — you extend by adding new SRP-shaped classes
  rather than editing one fat class.
- **Dependency Inversion**: an SRP-shaped class typically has a
  small set of dependencies, which makes inverting them tractable.
- **DRY**: tension here. Two classes with the same code may not be
  the same responsibility. Deduplicating across responsibilities
  creates coupling that SRP is trying to remove.
