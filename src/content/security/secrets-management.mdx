# Secrets Management

A secret is anything an attacker could use to impersonate you,
read your data, or take destructive action. API keys, database
passwords, JWT signing keys, OAuth client secrets, encryption keys.
The "how do you store and rotate these without writing them in
git" problem.

## The first rule

**Never commit secrets to git.**

Once a secret enters git history, it's effectively public — even if
you `git rm`, the old commit still contains it. People scan public
repos for leaked secrets within minutes of push.

If you ever accidentally commit a secret:

1. **Rotate it immediately.** Assume it's compromised.
2. Remove it from history (`git filter-repo`) only if the repo is
   private.
3. For public repos, the rotation is the only thing that matters —
   the history is already cached by GitHub's mirror and scrapers.

## Environment variables — the baseline

```js
const dbUrl = process.env.DATABASE_URL;
const jwtSecret = process.env.JWT_SECRET;
```

In **development**: a `.env` file (committed `.env.example` for
the variable names, `.env` itself is gitignored). Loaded with
`dotenv` or Node 20.6+'s `--env-file`.

In **production**: env vars set by the platform — Kubernetes secrets,
AWS Secrets Manager, GCP Secret Manager, Vercel/Fly/Render
dashboards, GitHub Actions repository secrets.

The same code; the *source* of the env vars differs per environment.

## Secret managers

For real production, use a dedicated secret manager:

- **AWS Secrets Manager** / **Parameter Store** — IAM-gated, with
  rotation lambdas, KMS-encrypted at rest.
- **GCP Secret Manager**.
- **Azure Key Vault**.
- **HashiCorp Vault** — cloud-agnostic, has dynamic secret
  generation (issues per-request DB credentials, etc.).
- **1Password / Doppler / Infisical** — developer-friendly with
  CI/CD integrations.

Benefits over plain env vars:

- **Access control via IAM** — only specific services / IAM roles
  can read.
- **Rotation** — change the secret without redeploying (your app
  re-fetches periodically or on signal).
- **Audit log** — who read which secret when.
- **Encryption at rest** managed by the platform.

The cost: complexity. For small projects, env vars set by your
hosting platform are usually fine.

## Loading secrets at startup vs lazily

**At startup** — fail fast if a secret is missing:

```js
function required(name) {
  const v = process.env[name];
  if (!v) throw new Error(`${name} is required`);
  return v;
}

export const config = {
  dbUrl: required('DATABASE_URL'),
  jwtSecret: required('JWT_SECRET'),
};
```

Crashes the process at boot rather than serving traffic without the
secret and failing when the first request needs it.

**Lazily** — fetch from a secret manager on demand:

```js
import { SecretsManagerClient, GetSecretValueCommand } from '@aws-sdk/client-secrets-manager';

const sm = new SecretsManagerClient();

async function getSecret(id) {
  const r = await sm.send(new GetSecretValueCommand({ SecretId: id }));
  return r.SecretString;
}
```

Slower; cache aggressively. Useful when secrets rotate at runtime
and you can't redeploy on every rotation.

## Rotation

Every secret should be rotatable. Best practice: rotate routinely
(quarterly is a common cadence) and after any incident.

The mechanism:

1. Generate a new secret.
2. Update the secret store.
3. Either restart the app (env-var-based) or trigger a re-fetch
   (manager-based).
4. Verify it works.
5. Revoke the old secret.

For database passwords: many secret managers can rotate Postgres
passwords automatically via a Lambda/Cloud Function. Worth setting
up for sensitive DBs.

For JWT signing keys: rotation is harder — old tokens signed with
the old key need to keep working until expiry. Maintain a small
keyring; sign with the newest; verify against any in the ring.

## Detecting leaks

- **GitHub Secret Scanning** — built-in for public repos, scans
  every push for known patterns (AWS keys, GitHub tokens, etc.).
- **`trufflehog`**, **`gitleaks`** — open-source scanners; run in
  pre-commit hooks and CI.
- **AWS GuardDuty**, **CloudTrail** — detect anomalous use of AWS
  credentials.

The faster you detect, the cheaper the response. Pre-commit hooks
catch leaks before they're pushed.

## What lives in env vars vs config

Env vars are for things that **change per environment** or are
**sensitive**. Constants belong in code.

```js
// In env
DATABASE_URL=...
JWT_SECRET=...
LOG_LEVEL=info

// In code
const MAX_UPLOAD_SIZE = 5 * 1024 * 1024;
const ALLOWED_FILE_TYPES = ['image/png', 'image/jpeg'];
const API_BASE_PATH = '/api/v1';
```

Constants in env vars become "config drift" — different values per
environment leading to environment-specific bugs. Keep code
constants in code.

## Secret hygiene

- **Least privilege**: each service has the minimum permissions
  needed. The user API doesn't need write access to the audit log.
- **Per-environment secrets**: staging and prod have different
  values for everything. Compromising one doesn't compromise the
  other.
- **Per-service secrets**: don't share one DB password across all
  services. One leak shouldn't expose everything.
- **No secrets in logs**: configure your logger to redact known
  secret fields (`req.headers.authorization`, etc.).
- **No secrets in error messages**: a 500 response leaking the DB
  password in a stack trace is a real bug.

## Encryption at rest

If you store secrets in your database (e.g., users' API keys to
third-party services):

```js
import { createCipheriv, createDecipheriv, randomBytes } from 'node:crypto';

function encrypt(plaintext, key) {
  const iv = randomBytes(12);
  const cipher = createCipheriv('aes-256-gcm', key, iv);
  const encrypted = Buffer.concat([cipher.update(plaintext, 'utf8'), cipher.final()]);
  const tag = cipher.getAuthTag();
  return Buffer.concat([iv, tag, encrypted]).toString('base64');
}

function decrypt(payload, key) {
  const buf = Buffer.from(payload, 'base64');
  const iv = buf.subarray(0, 12);
  const tag = buf.subarray(12, 28);
  const encrypted = buf.subarray(28);
  const decipher = createDecipheriv('aes-256-gcm', key, iv);
  decipher.setAuthTag(tag);
  return Buffer.concat([decipher.update(encrypted), decipher.final()]).toString('utf8');
}
```

Use **AES-256-GCM** (authenticated encryption — detects
tampering). The key (`key`) lives in your secret manager. The
encrypted blob lives in the database. An attacker who steals the
database can't read the values without the key.

For high-stakes data, consider using KMS (AWS / GCP) directly so
the encryption key never leaves their service.

## .env in production?

In a containerized world, mostly no — env vars come from the
orchestrator. `.env` files are a dev-time convenience.

The exception: some PaaS providers (older Heroku, etc.) historically
worked fine with `.env`-style config but managed it for you.

## Quick checklist

- [ ] `.env` is gitignored.
- [ ] `.env.example` is committed.
- [ ] Production secrets come from the platform/secret manager.
- [ ] Required secrets validated at startup.
- [ ] Logger redacts sensitive fields.
- [ ] Each environment has different secrets.
- [ ] Pre-commit hook scans for accidental commits.
- [ ] Rotation plan documented.

The discipline isn't hard; it's just discipline. The hard part is
catching the accidental leak — make tooling do that for you.
