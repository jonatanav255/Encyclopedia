# Numbers

JavaScript has one number type for almost everything: `number`, an
IEEE 754 double-precision float. There's also `bigint` for arbitrarily
large integers. Together they cover the language's numeric needs —
with some sharp edges.

## `number` — the double

```js
typeof 1;              // "number"
typeof 1.5;            // "number"
typeof (0.1 + 0.2);    // "number"
typeof NaN;            // "number"
typeof Infinity;       // "number"
```

It's all the same type. Integers up to `2^53 - 1` are represented
exactly; beyond that you lose precision:

```js
Number.MAX_SAFE_INTEGER;        // 9007199254740991 (2^53 - 1)
Number.MAX_SAFE_INTEGER + 1;    // 9007199254740992
Number.MAX_SAFE_INTEGER + 2;    // 9007199254740992 — same!  oops
```

Beyond `MAX_SAFE_INTEGER`, consecutive integers stop being
distinguishable. If you handle IDs or timestamps that could exceed it,
use `bigint` or strings.

## Floating point quirks

```js
0.1 + 0.2;             // 0.30000000000000004
0.1 + 0.2 === 0.3;     // false
```

Not a JavaScript bug — IEEE 754 can't represent `0.1` exactly. Same
in Python, Java, Ruby, etc.

For currency:

- Store cents (or smaller) as integers: 199 instead of 1.99.
- Use a decimal library (`decimal.js`, `big.js`) if you need actual
  arbitrary-precision decimals.
- `bigint` for integer-only currency.

For tolerant comparison:

```js
Math.abs(a - b) < Number.EPSILON;  // approximate equality
```

`Number.EPSILON` is the smallest difference between 1 and the next
larger double.

## Special values

```js
Infinity;             // 1 / 0
-Infinity;            // -1 / 0
NaN;                  // 0 / 0, "abc" * 1, etc.

Number.MAX_VALUE;     // largest finite — ~1.8e308
Number.MIN_VALUE;     // smallest POSITIVE — ~5e-324 (not most negative!)
```

`NaN` is contagious — any math with `NaN` is `NaN`. And:

```js
NaN === NaN;          // false
Number.isNaN(NaN);    // true   — preferred check
Number.isNaN('abc');  // false  — only checks if value IS NaN, not "would coerce"
isNaN('abc');         // true   — global, coerces first (avoid)
```

`Number.isNaN(x)` is strict. `isNaN(x)` coerces first ("abc" becomes
`NaN`, returns true). Always use `Number.isNaN`.

## Integer checks

```js
Number.isInteger(1);    // true
Number.isInteger(1.5);  // false
Number.isInteger('1');  // false   — strict, no coercion

Number.isSafeInteger(2 ** 53); // false — past MAX_SAFE_INTEGER
```

## Conversion

```js
Number('42');         // 42
Number('42abc');      // NaN
Number('');           // 0   — surprising
Number('  42  ');     // 42  — trims whitespace
Number('0x10');       // 16
Number('1e3');        // 1000

parseInt('42abc', 10); // 42  — stops at first non-digit
parseInt('   42');     // 42  — trims
parseFloat('1.5x');    // 1.5

+'42';                // 42  — unary plus shorthand
```

**Always pass a radix to `parseInt`**:

```js
parseInt('08', 10);  // 8 — explicit
parseInt('08');      // 8 in modern engines, 0 in some old ones (octal)
```

`Number()` rejects non-numeric strings entirely (`Number('42abc')` →
`NaN`), while `parseInt` is lenient (`parseInt('42abc')` → `42`). Pick
based on whether you want strict or "take what you can."

## Math methods

```js
Math.abs(-5);         // 5
Math.min(1, 2, 3);    // 1   — variadic
Math.min(...[1, 2, 3]); // 1
Math.max(...arr);
Math.floor(1.9);      // 1
Math.ceil(1.1);       // 2
Math.round(1.5);      // 2   — rounds half toward +Infinity (so -1.5 rounds to -1, not -2)
Math.trunc(1.9);      // 1   — toward zero
Math.sign(-3);        // -1, 0, or 1
Math.pow(2, 10);      // 1024
2 ** 10;              // 1024 — operator form
Math.sqrt(16);        // 4
Math.random();        // [0, 1)
```

`Math.random()` is not cryptographically secure. For security-relevant
randomness, use `crypto.getRandomValues` (browser/Node) or
`crypto.randomInt`.

## Formatting

```js
(1234.5).toFixed(2);     // "1234.50"
(1234.5).toExponential(); // "1.2345e+3"
(1234.5).toPrecision(3); // "1.23e+3"
```

These all return strings, not numbers.

For locale-aware:

```js
new Intl.NumberFormat('en-US').format(1234.5);     // "1,234.5"
new Intl.NumberFormat('de-DE').format(1234.5);     // "1.234,5"
new Intl.NumberFormat('en-US', { style: 'currency', currency: 'USD' })
  .format(1234.5);                                   // "$1,234.50"
new Intl.NumberFormat('en-US', { style: 'percent' })
  .format(0.25);                                     // "25%"
```

Always use `Intl.NumberFormat` for user-facing numbers.

## `BigInt`

For integers beyond `MAX_SAFE_INTEGER` (or when you want exact
integer math):

```js
const big = 9007199254740993n;   // n suffix
const fromString = BigInt('9007199254740993');

big + 1n;                         // 9007199254740994n
typeof big;                       // "bigint"
```

Rules:

- Operands of math operators must both be `bigint` (or both `number`)
  — mixing throws.
- Comparison works across types (`5n === 5` is false, `5n == 5` is
  true, `5n < 10` is true).
- `JSON.stringify` doesn't support `bigint` — throws `TypeError`.
  Convert to string for serialization.
- Math functions (`Math.abs`, etc.) only accept `number`.

Use `bigint` for:

- Large IDs (Twitter snowflake, database `bigint` columns).
- Cryptographic math.
- High-precision currency (in lowest unit — millicents, etc.).

## Bitwise operators

```js
5 & 3;     // 1
5 | 3;     // 7
5 ^ 3;     // 6   — XOR
~5;        // -6  — bitwise NOT
5 << 1;    // 10
5 >> 1;    // 2   — sign-preserving right shift
5 >>> 1;   // 2   — zero-fill right shift
```

These coerce to **32-bit signed integers**. So:

```js
(2 ** 33) | 0;   // 0   — overflow, truncated
```

Use only for flag/bit manipulation in known 32-bit contexts. For
larger integers, `BigInt` supports `&`, `|`, `^`, `<<`, `>>` (no
`>>>`).

The `| 0` and `~~x` tricks ("convert to int") work for the
truncation behavior but lose precision past 32 bits. `Math.trunc(x)`
is clearer.

## Common pitfalls

### `+` is concatenation when one side is string

```js
'1' + 2;       // '12'   — not 3
'1' - 2;       // -1     — minus coerces to number
+'1' + 2;      // 3      — unary + first
```

### `JSON` doesn't preserve `NaN` or `Infinity`

```js
JSON.stringify({ x: NaN, y: Infinity, z: -Infinity });
// '{"x":null,"y":null,"z":null}'
```

Decide on a sentinel (`null`, `"NaN"`) and document it.

### Coercion to number

```js
Number(true);       // 1
Number(false);      // 0
Number(null);       // 0
Number(undefined);  // NaN
Number([]);         // 0     — empty array → "" → 0
Number([42]);       // 42    — single-element array
Number([1, 2]);     // NaN
Number({});         // NaN
```

These are why `==` is unreliable. Use `===` and explicit conversions.

### `Math.random()` isn't secure

For tokens, IDs, cryptographic seeds, use `crypto.getRandomValues(new
Uint32Array(1))[0]` (browser) or `crypto.randomInt` /
`crypto.randomBytes` (Node).

## The discipline

- Default to `number`. `bigint` only when you need it.
- Use `Number.isInteger`, `Number.isSafeInteger`, `Number.isNaN` —
  never the legacy globals.
- Store currency in integer minor units.
- Format numbers with `Intl.NumberFormat`, not by hand.
- Don't trust `===` for floats. Compare with `Math.abs(a - b) <
  EPSILON` if needed.
