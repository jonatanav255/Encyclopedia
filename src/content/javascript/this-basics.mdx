# `this` basics

`this` is the most-confused keyword in JavaScript. The rule is simple
once you internalize it: **`this` is determined by how a function is
*called*, not where it's defined.**

The exception, as always, is arrow functions — they inherit `this`
from where they're written. That exception is what makes them useful
for callbacks.

## The four call modes

Every function call falls into one of these four shapes:

### 1. Method call: `obj.fn()`

`this` is `obj`.

```js
const user = {
  name: 'jon',
  greet() { return `hi ${this.name}`; },
};
user.greet(); // "hi jon"   — `this` is `user`
```

### 2. Plain call: `fn()`

`this` is `undefined` in strict mode (every module and class is strict
by default), or the global object in sloppy mode.

```js
function plain() { return this; }
plain();        // undefined (strict) or window/global (sloppy)
```

### 3. Constructor call: `new Fn()`

`this` is a new object, freshly created with `Fn`'s prototype.

```js
function User(name) { this.name = name; }
const u = new User('jon');  // `this` was the new object
u.name; // "jon"
```

### 4. Explicit call: `fn.call(thisArg, ...)` / `fn.apply` / `fn.bind`

`this` is whatever you pass.

```js
function greet() { return `hi ${this.name}`; }
greet.call({ name: 'jen' });   // "hi jen"
greet.apply({ name: 'jen' });  // same, but args passed as array
const bound = greet.bind({ name: 'jen' });
bound();                       // "hi jen"
```

## Losing `this`

The most common bug: a method passed somewhere loses its binding.

```js
const user = {
  name: 'jon',
  greet() { return `hi ${this.name}`; },
};

const g = user.greet;
g();    // "hi undefined" — called as plain function, `this` is undefined

setTimeout(user.greet, 100); // also broken
```

The function reference `user.greet` is just a function. When you call
it without the `user.` prefix, the method-call rule doesn't apply.

### Fixes

```js
// 1. Bind explicitly
const g = user.greet.bind(user);
g(); // "hi jon"

// 2. Wrap in an arrow (closes over `user`)
setTimeout(() => user.greet(), 100);

// 3. Use an arrow on the object (if appropriate)
const user2 = {
  name: 'jon',
  greet: () => `hi ${user2.name}`,  // can't use `this` here
};
```

## Arrow functions

Arrow functions have **no `this`** of their own. They look it up in
the enclosing lexical scope.

```js
const obj = {
  name: 'jon',
  delayed() {
    setTimeout(() => console.log(this.name), 100); // works — arrow inherits delayed's `this`
  },
  broken() {
    setTimeout(function() { console.log(this.name); }, 100); // undefined — plain call
  },
};
```

In `delayed`, the arrow inherits `this` from `delayed`, which is `obj`
(method call). In `broken`, the inner regular function gets its own
`this` (undefined in strict mode).

This is why callbacks almost always want to be arrow functions.

## Classes

Inside a class, methods follow the method-call rule:

```js
class Counter {
  count = 0;
  inc() { this.count++; }
}

const c = new Counter();
c.inc();                   // works — `this` is c

const detached = c.inc;
detached();                // TypeError: Cannot read 'count' of undefined
```

Two common solutions:

```js
// 1. Arrow as class field — bound to the instance at construction
class Counter {
  count = 0;
  inc = () => { this.count++; };
}

// 2. Bind in the constructor
class Counter {
  count = 0;
  constructor() { this.inc = this.inc.bind(this); }
  inc() { this.count++; }
}
```

The arrow-field form is concise and widely used in React class
components (back when those were common). Some style guides prefer
binding in the constructor for memory reasons (arrow fields create a
fresh function per instance).

## The mental model

When you see `fn()` ask: **how is this called?**

- `obj.fn()` → `this` is `obj`.
- `fn()` → `this` is undefined (strict) or global.
- `new Fn()` → `this` is the new object.
- `fn.call(x)` → `this` is `x`.
- Arrow → inherit from outer scope, ignoring all of the above.

If you can't see the call site, you can't predict `this`. That's why
methods passed as callbacks lose their binding — the original call
site disappears, replaced by `setTimeout`'s invocation.

## When `this` matters less than you'd think

In a lot of modern code, `this` barely comes up:

- **Modules** of plain functions don't use `this` at all.
- **React function components** don't have `this`.
- **Most utility code** is arrow functions over modules.

`this` is mostly about classes and event handlers. The rest of the
language has been moving away from it for a decade.
