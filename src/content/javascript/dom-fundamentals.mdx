# DOM fundamentals

The DOM (Document Object Model) is the browser's tree-shaped
representation of the page. JavaScript reads from and writes to it
via the `document` object and DOM nodes.

Most modern apps use a framework (React, Vue, Svelte) that
abstracts DOM access — but knowing the underlying API helps for:

- Vanilla JS for small UI.
- Library internals — knowing what React actually does.
- DOM-heavy work (extensions, scrapers, canvas integrations).
- Performance debugging.

## The tree

A page like `<body><h1>Hi</h1><p>Yo</p></body>` becomes:

```
document
  ↓
<html>
  ↓
<body>
  ↓
<h1>  ← textContent: "Hi"
<p>   ← textContent: "Yo"
```

Each node is an object. Element nodes (`<h1>`), text nodes ("Hi"),
comment nodes — all distinct types. The `document` is the root.

## Finding elements

```js
document.getElementById('my-id');               // by id
document.getElementsByClassName('my-class');    // HTMLCollection (live)
document.getElementsByTagName('div');           // HTMLCollection (live)
document.querySelector('.menu > li');           // first match
document.querySelectorAll('a[href^="https://"]'); // all matches (static NodeList)
```

**`querySelector` / `querySelectorAll`** are the modern picks —
CSS selectors, predictable return types.

**Difference to know**: `getElementsByClassName` returns a *live*
collection — if you add a matching element, the collection updates.
`querySelectorAll` returns a *static* snapshot.

```js
const live = document.getElementsByClassName('item');
const snap = document.querySelectorAll('.item');

document.body.appendChild(itemEl);

live.length;   // updated
snap.length;   // unchanged
```

For most code, static is what you want.

## Creating and modifying

```js
const div = document.createElement('div');
div.className = 'card';
div.textContent = 'Hello';

document.body.appendChild(div);
```

Modern alternatives:

```js
const parent = document.querySelector('.list');
parent.append(div);              // adds at end (multiple allowed)
parent.prepend(div);             // adds at start
parent.before(div);              // inserts before parent
parent.after(div);               // inserts after parent
parent.replaceChildren(div);     // clears and replaces

div.remove();                     // removes from parent
```

`append` accepts strings, multiple nodes, mix-and-match:

```js
parent.append(div, ' separator ', span);
```

Cleaner than the old `parent.appendChild(parent.insertBefore(...))`
dance.

## Reading and writing attributes

```js
el.id;                    // 'foo'
el.className;             // 'a b c'
el.classList;             // DOMTokenList
el.classList.add('new');
el.classList.remove('old');
el.classList.toggle('active');
el.classList.contains('hidden');

el.getAttribute('data-id');         // string or null
el.setAttribute('data-id', '42');
el.removeAttribute('data-id');

el.dataset.userId;        // reads data-user-id (camelCase)
el.dataset.userId = '42';

el.style.color = 'red';   // inline style
el.style.fontSize = '14px';
```

`classList` is the modern way to manage classes. `dataset` reads/
writes `data-*` attributes as camelCase properties.

For setting many styles, prefer **classes** over `style` — easier
to manage, themeable, CSP-friendly.

## Text content vs HTML

```js
el.textContent = 'Hi';                // safe, sets text
el.innerHTML = '<b>Hi</b>';           // parses HTML, can be XSS
el.innerText = 'Hi';                  // similar to textContent but
                                       // honors CSS visibility
```

**`innerHTML` parses HTML** — if the string contains user input
without sanitization, that's an XSS vulnerability. Use `textContent`
for plain text.

For safe HTML insertion: a templating library, React, or **DOMPurify**:

```js
import DOMPurify from 'dompurify';
el.innerHTML = DOMPurify.sanitize(userHtml);
```

## Events

```js
button.addEventListener('click', (event) => {
  event.preventDefault();
  console.log('clicked');
});

button.addEventListener('keydown', (e) => {
  if (e.key === 'Escape') close();
});
```

The `event` object has:

- `event.target` — the element that fired.
- `event.currentTarget` — the element the listener is on.
- `event.preventDefault()` — stop default behavior.
- `event.stopPropagation()` — stop bubbling to ancestors.
- `event.key`, `event.code`, `event.metaKey`, etc. for keyboard.

### Event delegation

For many child elements, attach one listener at a common parent:

```js
list.addEventListener('click', (e) => {
  const item = e.target.closest('li');
  if (!item) return;
  console.log('clicked item:', item.dataset.id);
});
```

`closest(selector)` walks up to find the matching ancestor. Pattern:
let events bubble; identify the actual target with `closest`. Saves
listener memory and works for items added later.

### Removing listeners

```js
const handler = () => {};
button.addEventListener('click', handler);
button.removeEventListener('click', handler);   // must be the same function
```

Anonymous arrows can't be removed (you don't have the reference).
Save the handler for removal.

For one-shot listeners:

```js
button.addEventListener('click', handler, { once: true });
```

Auto-removes after first fire.

### `AbortController` for listener cleanup

```js
const ac = new AbortController();

el.addEventListener('click', handler, { signal: ac.signal });
el.addEventListener('keydown', otherHandler, { signal: ac.signal });

// Remove both at once
ac.abort();
```

Pattern: one signal removes many listeners. Useful in React effects
or component cleanup.

## Traversing the tree

```js
el.parentNode;
el.parentElement;          // null if parent isn't an element
el.children;                // HTMLCollection of element children
el.childNodes;              // NodeList of all children (incl text)
el.firstElementChild;
el.lastElementChild;
el.nextElementSibling;
el.previousElementSibling;
el.closest('.menu');        // walk up to first matching ancestor
```

`children` (elements) vs `childNodes` (everything incl text/comment)
— remember the distinction.

`closest` walks the ancestor chain looking for a CSS-selector
match. Replaces the manual `while (el && !el.matches(...)) el = el.parentElement`.

## Forms

```js
const form = document.querySelector('form');

form.addEventListener('submit', (e) => {
  e.preventDefault();
  const data = new FormData(form);
  const json = Object.fromEntries(data);
  console.log(json);
});
```

`FormData` extracts form values cleanly. `Object.fromEntries`
converts to a plain object (for non-multi-value fields).

For inputs:

```js
input.value;              // current value
input.value = 'hi';
input.checked;            // checkbox / radio
input.files;              // file inputs
input.focus();
input.blur();
input.select();           // select all text
```

## Measurement

```js
el.getBoundingClientRect();   // { top, left, right, bottom, width, height }
el.offsetWidth;                // border-box width
el.offsetHeight;
el.scrollTop;
el.scrollLeft;
```

`getBoundingClientRect` is the modern measure. Includes transforms
and is relative to the viewport.

**Reading layout forces synchronous layout calculation** — if you
modify the DOM then read layout in the same frame, the browser must
recompute everything (the "layout thrash"):

```js
// BAD
for (const item of items) {
  item.style.width = '100px';
  console.log(item.offsetWidth);   // forces layout calc each time
}

// Better
for (const item of items) item.style.width = '100px';
for (const item of items) console.log(item.offsetWidth);   // one layout pass
```

Batch writes; batch reads. Don't interleave.

## Animation and timing

```js
requestAnimationFrame(() => {
  el.style.transform = 'translateY(10px)';
});
```

`requestAnimationFrame` fires before the next paint. Use for any
visual update — sets up the change to be batched with other DOM
work in one frame.

For CSS transitions:

```js
el.style.transition = 'opacity 200ms';
el.style.opacity = '1';
```

The browser handles the interpolation. Smooth.

For more complex animation, **Web Animations API**:

```js
el.animate(
  [{ transform: 'translateY(0)' }, { transform: 'translateY(20px)' }],
  { duration: 300, easing: 'ease-out' }
);
```

Native, GPU-accelerated, returns a promise that resolves on
completion.

## IntersectionObserver / MutationObserver / ResizeObserver

Modern APIs for reactive DOM:

```js
// Fires when element becomes visible
const io = new IntersectionObserver((entries) => {
  for (const entry of entries) {
    if (entry.isIntersecting) loadLazy(entry.target);
  }
});
io.observe(img);
```

```js
// Fires when DOM mutations happen
const mo = new MutationObserver((mutations) => {
  for (const m of mutations) console.log(m.type, m.target);
});
mo.observe(el, { childList: true, subtree: true });
```

```js
// Fires when element resizes
const ro = new ResizeObserver((entries) => {
  for (const entry of entries) console.log(entry.contentRect);
});
ro.observe(el);
```

These replace polling-based "did the layout change?" checks. Each is
batched, efficient, and provides the data you need.

## Common pitfalls

### `innerHTML` with user input

```js
el.innerHTML = `<p>${userName}</p>`;   // XSS if userName contains HTML
```

Always sanitize, or use `textContent`:

```js
const p = document.createElement('p');
p.textContent = userName;
el.appendChild(p);
```

### `getElementsByClassName` returning live collection

```js
const items = document.getElementsByClassName('item');
for (let i = 0; i < items.length; i++) {
  document.body.appendChild(items[i].cloneNode());   // collection grows!
  // infinite loop or weird behavior
}
```

Use `querySelectorAll` (static) when you'll modify the DOM during
iteration.

### `addEventListener` without cleanup

Long-lived listeners on long-lived elements (window, document, body)
leak memory if your component is gone. Always pair with `removeEventListener`
or use the `signal` option.

### Layout thrashing

Covered above. Reads and writes in alternation are slow.

### Forgetting `preventDefault` for forms / links

```html
<form onSubmit="...">
```

Without `e.preventDefault()`, the browser submits the form and
reloads. Almost always wrong in modern JS.

### `event.target` vs `event.currentTarget`

```js
ul.addEventListener('click', (e) => {
  e.target;          // could be a nested span the user clicked
  e.currentTarget;   // always the ul
});
```

For event delegation (using `closest`), check whether `target`
matches; use `currentTarget` for the handler's owner.

### Synchronous heavy work in event handlers

```js
button.addEventListener('click', () => {
  for (let i = 0; i < 1e7; i++) heavyWork();
  // UI frozen for seconds
});
```

Defer with `setTimeout`, break into chunks with `requestIdleCallback`,
or move to a Web Worker.

## Performance — when DOM matters

For dynamic UI:

- **Minimize layout reads** — batch them, use `requestAnimationFrame`.
- **Use `DocumentFragment`** for bulk insertions:
  ```js
  const fragment = document.createDocumentFragment();
  for (const data of items) {
    const el = document.createElement('div');
    el.textContent = data.name;
    fragment.appendChild(el);
  }
  list.appendChild(fragment);   // one DOM mutation
  ```
- **Use `IntersectionObserver`** for "load when visible" patterns
  instead of scroll listeners.
- **For lists of 100+ items**, consider virtualization
  (`react-window`, `virtual-scroller`).

## The mental model

The DOM is **a tree of objects mirroring the page's HTML**.
JavaScript reads and writes; the browser renders.

Modern DOM work breaks down to:

- **Find** (`querySelector` family).
- **Create / Modify** (`createElement`, `append`, `replaceChildren`,
  `textContent`).
- **Style** (CSS classes via `classList`, not inline styles).
- **Listen** (`addEventListener` + delegation + `signal` cleanup).
- **Measure** (`getBoundingClientRect`, batched reads).
- **Observe** (Intersection / Mutation / Resize for reactivity).

For component-driven apps, a framework (React, Vue, Svelte) abstracts
all this. For small widgets or library internals, vanilla DOM is
fast and dependency-free. The patterns above are the bedrock — every
framework eventually does these things under the hood.
