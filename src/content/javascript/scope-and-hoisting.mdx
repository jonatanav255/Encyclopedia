# Scope and hoisting

**Scope** is where in your code a variable is visible. **Hoisting** is
how the engine pre-processes declarations before running the code.

## The scopes

```js
// global scope
const APP = 'encyclopedia';

function outer() {
  // function scope
  const a = 1;

  if (true) {
    // block scope
    const b = 2;
    console.log(a, b); // both visible
  }

  console.log(b); // ReferenceError — block-scoped
}
```

Lexical scoping: inner scopes see outer scopes; outer scopes don't see
inner ones. The scope chain is determined by *where* the code is
written, not where it's called from.

## The scope chain

When you read a variable, the engine walks outward:

```js
const name = 'global';

function outer() {
  const name = 'outer';

  function inner() {
    console.log(name); // walks up: inner → outer → global, finds "outer"
  }

  inner();
}
```

The lookup stops at the first match. Inner declarations **shadow**
outer ones.

## Hoisting

Before any code in a scope runs, the engine scans declarations and
"hoists" them to the top of their scope. What gets hoisted depends on
the declaration type:

```js
// Function declarations: fully hoisted (name + body)
hi();             // "hi" — works
function hi() { console.log('hi'); }

// var: hoisted, initialized to undefined
console.log(x);   // undefined
var x = 1;

// let / const: hoisted, but unreadable until declaration (TDZ)
console.log(y);   // ReferenceError
let y = 1;

// Function expressions / arrow functions: only the variable is hoisted
hello();          // TypeError: hello is not a function
var hello = () => console.log('hello');
```

The order:

1. Function declarations → hoisted with their bodies.
2. `var` declarations → hoisted, initialized to `undefined`.
3. `let` / `const` / `class` declarations → hoisted, in the *temporal
   dead zone* until the declaration line.

## Why hoisting exists

So you can call functions before they're declared:

```js
const total = sum(1, 2, 3);

function sum(...nums) {
  return nums.reduce((a, b) => a + b, 0);
}
```

This pattern is idiomatic — declare your helpers below the main flow,
read the main flow first. Hoisting makes it work.

## What "hoisting" really means

The engine doesn't physically move code. It just creates the bindings
before executing the block. The mental model "imagine the declarations
moved to the top" is a useful shortcut.

## Practical lessons

- Declarations are hoisted; **assignments are not**.
  ```js
  console.log(x); // undefined
  var x = 5;       // assignment happens here, not at the top
  ```
- Function declarations are special — both name and body are hoisted.
- Using `let`/`const` everywhere eliminates the "undefined surprise"
  that `var` causes. The TDZ throws instead of returning `undefined`.
- Inner scopes shadow outer scopes silently. Don't reuse names across
  nested scopes unless you're intentionally shadowing.

## Block-scoped function declarations

Strict mode + `let`-style block scope changed how function declarations
behave inside blocks:

```js
if (cond) {
  function f() {}   // technically scoped to the if-block in strict mode
}
f();                // ReferenceError in strict mode
```

Engines historically disagreed here. Don't declare functions inside
blocks; use function expressions assigned to `const` instead.

## The discipline

- Use `let`/`const` everywhere — block scope + TDZ catches the bugs
  that `var` hides.
- Declare variables as close to first use as possible.
- Don't shadow outer names unless intentional.
- Function declarations at the top of their scope (or wherever makes
  the code most readable, since they hoist).
