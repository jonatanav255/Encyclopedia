# `var`, `let`, `const`

Three ways to declare a variable. Use `const` by default, `let` when
you have to reassign, and `var` never.

## `const`

Block-scoped, can't be reassigned, must be initialized at declaration.

```js
const PI = 3.14;
PI = 3.15;          // TypeError: Assignment to constant variable

const user = { name: 'jon' };
user.name = 'jen';  // OK — the binding is const, not the object
user = {};          // TypeError
```

The thing `const` protects is the **binding** — what variable points
to. It does not freeze the value. A `const` array still has mutable
contents. Use `Object.freeze(obj)` (shallow) if you need immutability,
or a library like `immer` for deep.

## `let`

Block-scoped, can be reassigned, can be declared without initializer.

```js
let count = 0;
count = 1;          // OK

if (true) {
  let inside = 'block';
}
console.log(inside); // ReferenceError — let is block-scoped
```

Reach for `let` when you have to reassign — accumulators, counters,
values that genuinely change.

## `var` (don't use this)

Function-scoped, hoisted, can be redeclared. All three are bad:

```js
function f() {
  if (true) {
    var x = 1;
  }
  console.log(x); // 1 — leaked out of the if block
}

console.log(y);  // undefined — hoisted declaration
var y = 5;

var z = 1;
var z = 2;       // no error
```

Function-scoping causes the classic loop bug:

```js
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
// Logs: 3, 3, 3  (one `i`, captured by closure, ends at 3)

for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
// Logs: 0, 1, 2  (fresh binding each iteration)
```

`var` exists for backwards compatibility. No new code should use it.

## Block scope vs function scope

```js
{
  let x = 1;
  const y = 2;
  var z = 3;
}
console.log(z); // 3   — var leaks out
console.log(x); // ReferenceError
console.log(y); // ReferenceError
```

Block scope is what every other modern language does. JavaScript
finally got it in 2015 with `let`/`const`.

## The temporal dead zone

`let` and `const` are hoisted — the engine knows about them at the top
of the block — but they're not initialized until the declaration line.
Accessing them before that throws:

```js
console.log(x); // ReferenceError: Cannot access 'x' before initialization
let x = 1;
```

Contrast with `var`:

```js
console.log(x); // undefined — no error
var x = 1;
```

And functions, which are fully hoisted:

```js
sayHi(); // works
function sayHi() { console.log('hi'); }
```

The TDZ exists to catch a class of bugs that `var`'s "silently
undefined" behavior masked.

## The discipline

- **`const` by default**, even for objects you'll mutate the contents
  of. The binding doesn't change, so use `const`.
- **`let`** when you need to reassign.
- **Never `var`** in modern code. Linters can enforce this
  (`no-var`).
- Declare variables as close to first use as possible. Block scope
  makes this easy.
