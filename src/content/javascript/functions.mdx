# Functions

Functions are first-class values in JavaScript — you can assign them
to variables, pass them as arguments, return them from other
functions, and store them in arrays or objects. Half the language is
about wielding that flexibility well.

## Three ways to define a function

```js
// 1. Function declaration — hoisted
function add(a, b) {
  return a + b;
}

// 2. Function expression — assigned to a variable
const sub = function(a, b) {
  return a - b;
};

// 3. Arrow function — concise, lexical `this`
const mul = (a, b) => a * b;
```

Differences:

- **Declarations** are hoisted (both name and body), so you can call
  them before they appear in the source.
- **Expressions** can be anonymous or named; only the *variable* is
  hoisted, not the function — so calling them above their assignment
  fails.
- **Arrow functions** have no `this` binding of their own — they
  inherit `this` from the enclosing scope. They also can't be used as
  constructors and don't have an `arguments` object.

## Parameters

### Defaults

```js
function greet(name = 'world') {
  return `hello, ${name}`;
}

greet();         // "hello, world"
greet('jon');    // "hello, jon"
greet(undefined); // "hello, world" — undefined triggers default
greet(null);     // "hello, null"  — null does not
```

Defaults trigger only on `undefined`, not `null`. This matters when
you destructure HTTP query params or JSON.

### Rest

```js
function sum(...nums) {
  return nums.reduce((a, b) => a + b, 0);
}
sum(1, 2, 3);    // 6
```

`...nums` collects all remaining arguments into an array. Must be the
last parameter.

### Destructuring

```js
function fetchPage({ url, headers = {}, timeout = 5000 }) { ... }

fetchPage({ url: '/api', timeout: 10_000 });
```

Pass options as an object so callers can name what they're sending.
Way better than positional `fetchPage(url, headers, timeout, retries, ...)`.

## The `arguments` object

In non-arrow functions, `arguments` is an array-like containing all
passed arguments:

```js
function legacy() {
  console.log(arguments.length, arguments[0]);
}
legacy('a', 'b');  // 2 'a'
```

In modern code, use rest parameters instead:

```js
function modern(...args) { ... }
```

Arrow functions don't have `arguments`. If you destructure them
inside an arrow you need to use rest.

## Closures

A function "closes over" the variables in its enclosing scope. They
remain accessible even after the outer function has returned.

```js
function counter() {
  let n = 0;
  return () => ++n;
}

const next = counter();
next(); // 1
next(); // 2
```

The inner arrow captures `n`. Each call to `counter()` creates a fresh
`n`. See **Closures** for the deeper treatment.

## Arrow functions in detail

```js
const id = x => x;                 // single param, no parens needed
const add = (a, b) => a + b;       // multi param, parens required
const obj = () => ({ x: 1 });      // returning an object literal: wrap in ()
const log = msg => { console.log(msg); }; // block body: explicit return
```

The body has two forms:

- **Expression body** — `x => x * 2`. Implicit return.
- **Block body** — `x => { return x * 2; }`. Explicit return required.

The `() => ({...})` gotcha: arrow + curly braces is a *block*, not an
object literal. Wrap in `()` to force expression interpretation.

## `this` in arrows vs regular functions

```js
const obj = {
  name: 'jon',
  greetRegular: function() { return `hi ${this.name}`; },
  greetArrow: () => `hi ${this?.name}`,
};

obj.greetRegular(); // "hi jon"
obj.greetArrow();   // arrow's `this` is the outer scope, not obj —
                    // in a module/strict context `this` is undefined
                    // (the optional chaining above guards against that);
                    // without it, `this.name` would throw TypeError.
```

Arrow functions inherit `this` from where they're defined, not where
they're called. They're great for callbacks that need the *outer*
`this`:

```js
class Timer {
  constructor() {
    this.count = 0;
    setInterval(() => this.count++, 1000);  // arrow keeps Timer's `this`
  }
}
```

Without the arrow, `this` inside the callback would be the timer's
context (in browsers, the global object or undefined in strict mode).

## Returning multiple values

JavaScript functions return one value. Multiple "returns" use an array
(positional) or object (named):

```js
function divmod(a, b) {
  return [Math.floor(a / b), a % b];   // array
}
const [q, r] = divmod(10, 3);

function range(...) {
  return { start, end, step };          // object — preferred
}
const { start, end } = range(...);
```

Objects scale better — callers can ignore fields they don't need and
you can add new ones without breaking existing destructuring.

## Higher-order functions

A function that takes or returns another function:

```js
const greaterThan = n => x => x > n;
const isPositive = greaterThan(0);
isPositive(5);   // true

const withRetry = (fn, n = 3) => async (...args) => {
  for (let i = 0; i < n; i++) {
    try { return await fn(...args); }
    catch (err) { if (i === n - 1) throw err; }
  }
};
```

`map`, `filter`, `reduce`, event handlers, `setTimeout` — all
higher-order. The pattern is the bread and butter of JavaScript.

## Pure vs impure

A **pure** function:

1. Returns the same output for the same input.
2. Has no side effects (no I/O, no mutation of outside state).

Pure functions are easier to test, cache, and reason about. They
compose. Side effects (logging, DB writes, network) belong at the
edges of your code, not buried in helpers.

## Recursion

```js
function factorial(n) {
  return n <= 1 ? 1 : n * factorial(n - 1);
}
```

JavaScript doesn't optimize tail calls in most engines (the spec
requires it; the implementations don't). Deep recursion will blow the
stack. Convert to iteration or use a trampoline if you need depth.

## Naming

- Verbs for actions: `fetchUser`, `parseConfig`, `sendEmail`.
- `is`/`has`/`can` prefixes for booleans: `isAdmin`, `hasAccess`.
- Avoid generic helpers in the global namespace; group them in
  modules.

A good function name makes the call site readable without looking at
the implementation.
