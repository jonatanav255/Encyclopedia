# `Intl` — formatting for humans

`Intl` is the built-in API for locale-aware formatting of numbers,
dates, currencies, lists, relative times, and more. It's been
shipping in browsers and Node for years and is almost always the
right answer over manual formatting or third-party libraries.

## Why use it

A lot of code has variations of this:

```js
// Bad — locale-blind, doesn't handle currency rules, plurals, etc.
function formatPrice(n) { return `$${n.toFixed(2)}`; }
```

That works for US English. It breaks for:

- European locales using `.` and `,` swapped.
- Currencies with different decimal rules (Japanese yen has no
  decimals).
- Right-to-left languages.
- Plurals ("1 item" vs "2 items" vs Russian's three-plural-form system).
- Date formats (`05/20/2026` vs `20/05/2026` vs `2026年5月20日`).

`Intl` handles all of these. It's already in the runtime — no library
needed.

## `Intl.NumberFormat`

```js
const fmt = new Intl.NumberFormat('en-US');
fmt.format(1234.5);                    // "1,234.5"

new Intl.NumberFormat('de-DE').format(1234.5);  // "1.234,5"
new Intl.NumberFormat('fr-FR').format(1234.5);  // "1 234,5"
```

### Currency

```js
new Intl.NumberFormat('en-US', { style: 'currency', currency: 'USD' })
  .format(1234.5);     // "$1,234.50"

new Intl.NumberFormat('de-DE', { style: 'currency', currency: 'EUR' })
  .format(1234.5);     // "1.234,50 €"

new Intl.NumberFormat('ja-JP', { style: 'currency', currency: 'JPY' })
  .format(1234);       // "￥1,234"  — no decimals (yen rule)
```

The locale controls grouping/decimal separators and symbol position.
The `currency` option controls precision (yen vs dollar vs Bitcoin).

### Percent

```js
new Intl.NumberFormat('en-US', { style: 'percent' }).format(0.25); // "25%"
new Intl.NumberFormat('en-US', { style: 'percent', minimumFractionDigits: 1 })
  .format(0.255);     // "25.5%"
```

### Precision

```js
new Intl.NumberFormat('en-US', {
  minimumFractionDigits: 2,
  maximumFractionDigits: 2,
}).format(1.5);       // "1.50"

new Intl.NumberFormat('en-US', { notation: 'compact' }).format(123_456);
// "123K"

new Intl.NumberFormat('en-US', { notation: 'scientific' }).format(1234);
// "1.234E3"
```

`compact` notation gives you `1.2K`, `3.4M`, `5.6B` — great for
charts and summaries.

## `Intl.DateTimeFormat`

```js
const d = new Date('2026-05-20T12:30:00Z');

new Intl.DateTimeFormat('en-US').format(d);      // "5/20/2026"
new Intl.DateTimeFormat('de-DE').format(d);      // "20.5.2026"
new Intl.DateTimeFormat('ja-JP').format(d);      // "2026/5/20"
new Intl.DateTimeFormat('en-GB').format(d);      // "20/05/2026"
```

### Style presets

```js
new Intl.DateTimeFormat('en-US', { dateStyle: 'full' }).format(d);
// "Wednesday, May 20, 2026"

new Intl.DateTimeFormat('en-US', { dateStyle: 'long' }).format(d);
// "May 20, 2026"

new Intl.DateTimeFormat('en-US', { dateStyle: 'medium' }).format(d);
// "May 20, 2026"

new Intl.DateTimeFormat('en-US', { dateStyle: 'short' }).format(d);
// "5/20/26"
```

Same for `timeStyle`:

```js
new Intl.DateTimeFormat('en-US', { timeStyle: 'short' }).format(d);
// "12:30 PM"

new Intl.DateTimeFormat('en-US', { dateStyle: 'medium', timeStyle: 'short' })
  .format(d);
// "May 20, 2026, 12:30 PM"
```

### Custom

```js
new Intl.DateTimeFormat('en-US', {
  year: 'numeric',
  month: 'long',
  day: 'numeric',
  hour: 'numeric',
  minute: '2-digit',
  hour12: true,
}).format(d);
// "May 20, 2026, 12:30 PM"
```

### Time zone

```js
new Intl.DateTimeFormat('en-US', {
  dateStyle: 'short',
  timeStyle: 'short',
  timeZone: 'America/Los_Angeles',
}).format(d);
// "5/20/26, 5:30 AM"
```

`timeZone` is the right way to format a UTC instant in any zone —
without touching the underlying `Date`.

### `formatToParts`

```js
new Intl.DateTimeFormat('en-US').formatToParts(d);
// [
//   { type: 'month', value: '5' },
//   { type: 'literal', value: '/' },
//   { type: 'day', value: '20' },
//   { type: 'literal', value: '/' },
//   { type: 'year', value: '2026' },
// ]
```

Returns the structured pieces. Useful when you want to insert custom
markup between parts (highlighting, links).

## `Intl.RelativeTimeFormat`

```js
const rtf = new Intl.RelativeTimeFormat('en', { numeric: 'auto' });

rtf.format(-1, 'day');      // "yesterday"
rtf.format(0, 'day');       // "today"
rtf.format(1, 'day');       // "tomorrow"
rtf.format(-3, 'day');      // "3 days ago"
rtf.format(2, 'hour');      // "in 2 hours"

new Intl.RelativeTimeFormat('es').format(-1, 'day');  // "hace 1 día"
```

`numeric: 'auto'` produces "yesterday"/"tomorrow"/"now" when
appropriate; `'always'` forces "1 day ago"/"in 1 day."

The unit can be `year`, `quarter`, `month`, `week`, `day`, `hour`,
`minute`, `second`.

## `Intl.ListFormat`

```js
new Intl.ListFormat('en').format(['apples', 'oranges', 'bananas']);
// "apples, oranges, and bananas"

new Intl.ListFormat('de').format(['Äpfel', 'Birnen', 'Orangen']);
// "Äpfel, Birnen und Orangen"

new Intl.ListFormat('en', { type: 'disjunction' })
  .format(['red', 'green', 'blue']);
// "red, green, or blue"

new Intl.ListFormat('en', { style: 'short' }).format(['a', 'b', 'c']);
// "a, b, & c"
```

Stop building `arr.join(', ')` lists. Use `ListFormat` and you get
Oxford comma rules, locale-appropriate separators, and conjunction/
disjunction handling.

## `Intl.PluralRules`

For picking the right form ("1 file" vs "2 files"):

```js
const pr = new Intl.PluralRules('en');
pr.select(1);   // "one"
pr.select(2);   // "other"
pr.select(0.5); // "other"

const ru = new Intl.PluralRules('ru');
ru.select(1);   // "one"
ru.select(2);   // "few"
ru.select(5);   // "many"
ru.select(11);  // "many"  (Russian's special rule)
```

Combine with a lookup table:

```js
const forms = { one: '1 file', few: '{n} files', many: '{n} files', other: '{n} files' };
const cat = pr.select(count);
const s = forms[cat].replace('{n}', count);
```

In practice you'll use an i18n library (`react-intl`, `i18next`) that
wraps `PluralRules` for you. But knowing it exists explains how those
libraries work.

## `Intl.Collator`

For locale-aware string comparison and sorting:

```js
const c = new Intl.Collator('en');
[3, 1, 11].sort(c.compare);   // doesn't matter for numbers

const words = ['Ångström', 'angstrom', 'apple'];
words.sort(new Intl.Collator('sv').compare);
// Swedish puts Å after Z, so different from English
```

```js
const ci = new Intl.Collator('en', { sensitivity: 'base' });
ci.compare('café', 'cafe');     // 0 — case + accent insensitive
```

Way better than `'a' < 'b'`, which sorts by Unicode code point (so
`'b' < 'a'` is false but `'B' < 'a'` is true — uppercase comes first).

## `Intl.DisplayNames`

Localized names for languages, regions, currencies, scripts:

```js
const dn = new Intl.DisplayNames('en', { type: 'language' });
dn.of('fr');           // "French"
dn.of('zh-CN');        // "Chinese (China)"

new Intl.DisplayNames('es', { type: 'region' }).of('US');  // "Estados Unidos"
new Intl.DisplayNames('en', { type: 'currency' }).of('JPY'); // "Japanese Yen"
```

For language selectors, country selectors, currency dropdowns.

## `Intl.Segmenter`

Locale-correct word and grapheme splitting (browser-only or recent
Node):

```js
const segmenter = new Intl.Segmenter('en', { granularity: 'word' });
const it = segmenter.segment('hello, world!');
[...it].map(s => s.segment).filter(s => /\S/.test(s));
// ['hello', 'world']
```

Real Unicode segmentation handles emoji, surrogate pairs, and
non-Latin scripts (Japanese, Thai) where word boundaries aren't
spaces. Don't use `.split(' ')` for anything user-facing.

## Performance

`Intl` constructors are expensive — building the formatter pulls
locale data. Cache them:

```js
// Bad — rebuilds every call
function fmt(n) { return new Intl.NumberFormat('en-US').format(n); }

// Good — built once
const FMT = new Intl.NumberFormat('en-US');
function fmt(n) { return FMT.format(n); }
```

For per-component instances in React, `useMemo` or hoist out of
render.

## Locale negotiation

Pass a list of preferred locales; the runtime picks the first it
supports:

```js
new Intl.NumberFormat(['fr-CA', 'fr', 'en']).format(1234.5);
// uses fr-CA if available, otherwise fr, otherwise en
```

In browsers, `navigator.language` and `navigator.languages` are the
user's preferences. In Node, `process.env.LANG` or your i18n
library's locale.

## What `Intl` doesn't do

- **Translation** — `Intl` formats; it doesn't translate strings.
  Use an i18n library (`i18next`, `react-intl`, `lingui`).
- **Right-to-left layout** — `Intl` produces text; layout is CSS
  (`dir="rtl"`).
- **Calendars beyond Gregorian** — partial support; check browser
  matrix.
- **Old browsers** — IE11 and pre-2018 mobile browsers lack much of
  `Intl`. Polyfill via `formatjs` if you support them. Modern code
  doesn't need to.

## The discipline

- Format every user-facing number, date, currency, and list with
  `Intl`. Never build them by hand.
- Cache formatter instances.
- Always pass an explicit locale — don't rely on default.
- Use `timeZone` to display dates in user zones, not server zones.
- Pair `Intl.PluralRules` with your i18n strings for plurals.

There's almost no formatting need an `Intl` API doesn't cover. Reach
for a library only when you need things `Intl` doesn't: translation,
markup, dynamic locale switching across many components.
