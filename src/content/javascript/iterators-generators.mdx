# Iterators and generators

Iterators are how JavaScript walks sequences. Generators are a concise
way to write iterators.

## The iterator protocol

An object is an **iterator** if it has a `next()` method that returns
`{ value, done }`:

```js
const it = {
  i: 0,
  next() {
    if (this.i < 3) return { value: this.i++, done: false };
    return { value: undefined, done: true };
  },
};

it.next(); // { value: 0, done: false }
it.next(); // { value: 1, done: false }
it.next(); // { value: 2, done: false }
it.next(); // { value: undefined, done: true }
```

An object is **iterable** if it has a `[Symbol.iterator]()` method
that returns an iterator:

```js
const range = {
  from: 1,
  to: 3,
  [Symbol.iterator]() {
    let i = this.from;
    const last = this.to;
    return {
      next() {
        if (i <= last) return { value: i++, done: false };
        return { value: undefined, done: true };
      },
    };
  },
};

for (const n of range) console.log(n);  // 1, 2, 3
[...range];                              // [1, 2, 3]
```

Built-in iterables: arrays, strings, `Map`, `Set`, `arguments`,
`NodeList`, `Buffer`. Anything you can `for...of` over is iterable.

## Where iterators show up

```js
for (const x of iterable) { ... }         // for...of
[...iterable]                              // spread
const [a, b, c] = iterable;               // destructure
Array.from(iterable);                     // convert
new Map(iterableOfPairs);                 // build a map
new Set(iterable);                        // build a set
Promise.all(iterable);                    // accepts any iterable, not just array
```

All of these consume the iterator protocol.

## Generators

A function declared with `function*` returns a generator object — an
iterator you can write declaratively:

```js
function* range(from, to) {
  for (let i = from; i <= to; i++) {
    yield i;
  }
}

for (const n of range(1, 3)) console.log(n);  // 1, 2, 3
[...range(1, 3)];                              // [1, 2, 3]
```

The generator pauses at each `yield`, returning that value. The next
`next()` call resumes execution until the next `yield` (or the
function returns).

### Two-way communication

`yield` is also an expression — `next(arg)` sends a value back in:

```js
function* dialog() {
  const name = yield 'what is your name?';
  yield `hi ${name}`;
}

const d = dialog();
d.next();          // { value: 'what is your name?', done: false }
d.next('jon');     // { value: 'hi jon', done: false }
d.next();          // { value: undefined, done: true }
```

The first `next()` runs to the first yield. The argument passed to the
*second* `next()` becomes the result of the first `yield` expression.

### Delegating with `yield*`

```js
function* inner() { yield 1; yield 2; }
function* outer() {
  yield 0;
  yield* inner();   // delegate
  yield 3;
}

[...outer()]; // [0, 1, 2, 3]
```

Useful for composing generators.

### Early termination

`return()` and `throw()` stop the generator early:

```js
const it = range(1, 100);
it.next();      // { value: 1, done: false }
it.return('stopped'); // { value: 'stopped', done: true }
it.next();      // { value: undefined, done: true }
```

Generators that own resources (file handles, sockets) can use the
`try {} finally {}` pattern to clean up on `return()`:

```js
function* readLines(file) {
  const fd = openSync(file);
  try {
    while (true) {
      const line = readLine(fd);
      if (line === null) return;
      yield line;
    }
  } finally {
    closeSync(fd);  // runs even if consumer breaks out of for...of
  }
}
```

## Async iteration

Iterators that return promises. Object has `[Symbol.asyncIterator]()`;
consumed with `for await...of`:

```js
async function* fetchPages(url) {
  let next = url;
  while (next) {
    const res = await fetch(next);
    const json = await res.json();
    yield json.items;
    next = json.nextPage;
  }
}

for await (const items of fetchPages('/api/feed')) {
  for (const item of items) console.log(item);
}
```

Async generators are the natural shape for paginated APIs, log
streams, and any "values arrive over time."

Built-in async iterables: Node streams (`Readable`), Web
`ReadableStream`, response bodies.

## Generators vs arrays

```js
function* infinite() {
  let i = 0;
  while (true) yield i++;
}

for (const n of infinite()) {
  if (n > 5) break;
  console.log(n);
}
```

You can't put this in an array — it never ends. Generators are lazy:
each value is computed on demand. They're how you express "the next
value of an infinite sequence" without precomputing.

## When to use them

- **Lazy sequences** — pagination, log tailing, large file lines.
- **Custom iterables** — make your class iterable by adding
  `[Symbol.iterator]`.
- **Async streams** — Node, fetch, file reads.
- **Coroutine-like control flow** — co-routines, parsers,
  state machines. Rarely needed in app code but useful in libraries.

## What to avoid

- **Using a generator when an array would do.** Arrays are simpler,
  random-access, and predictable.
- **Mixing sync and async iteration on the same object.** Pick one.
- **Long-lived generators holding resources.** If a `for...of` never
  finishes (consumer breaks out), the generator's `finally` runs, but
  you should still close resources explicitly when possible.

## The mental model

An iterator is a paused computation that emits values one at a time.
A generator is the easy way to write one. Async generators do the
same thing, except each yield may take time to produce.

`for...of`, `for await...of`, spread, destructure, `Array.from`,
`Map`, `Set` — all of these speak the protocol. Implementing it makes
your data structure interoperable with everything in the language.
