# Currying and function composition

Two functional-programming techniques that pay off in JavaScript:
**currying** (transforming a multi-arg function into a chain of
single-arg functions) and **composition** (combining small functions
into bigger ones).

Not the soul of FP — JS isn't a pure language — but useful patterns
in data pipelines, middleware chains, and React/Redux selectors.

## Currying

```js
// Uncurried
function add(a, b) { return a + b; }
add(2, 3);   // 5

// Curried
const addCurried = a => b => a + b;
addCurried(2)(3);   // 5
```

The curried form returns a function expecting the next argument.
You can partially apply by calling with some arguments and saving
the result.

```js
const add5 = addCurried(5);
add5(3);   // 8
add5(7);   // 12
```

`add5` is "add with 5 pre-filled." Reusable.

## Why currying matters

Useful when you want to:

- **Pre-configure** a function for a specific use case.
- **Plug into pipelines** (`map(fn)`, `filter(fn)`) where the
  pipeline expects a single-arg function.
- **Express intent** — `applyDiscount(0.1)(price)` reads better than
  `applyDiscount(0.1, price)` in some contexts.

```js
const tax = (rate) => (amount) => amount * (1 + rate);
const usTax = tax(0.08);
const ukVAT = tax(0.20);

[100, 200, 300].map(usTax);   // [108, 216, 324]
```

The pre-configured `usTax` slots into `map` cleanly.

## Auto-currying

For arbitrary arity:

```js
function curry(fn) {
  return function curried(...args) {
    if (args.length >= fn.length) {
      return fn(...args);
    }
    return (...more) => curried(...args, ...more);
  };
}

const add3 = curry((a, b, c) => a + b + c);

add3(1)(2)(3);     // 6
add3(1, 2)(3);     // 6
add3(1)(2, 3);     // 6
add3(1, 2, 3);     // 6
```

`curry(fn)` returns a function that collects arguments until it has
enough, then invokes. Each "partial" is also curried.

Libraries: `ramda.curry`, `lodash.curry`. Auto-currying is rarely
worth a dep for one function; build a custom for cases you use.

## Composition

```js
const trim = (s) => s.trim();
const lower = (s) => s.toLowerCase();
const exclaim = (s) => s + '!';

// Manual
exclaim(lower(trim('  Hello  ')));   // 'hello!'

// Composed
const greet = (s) => exclaim(lower(trim(s)));
greet('  Hello  ');   // 'hello!'
```

Composition is just nested function calls. The "composition
operator" is implicit in JS.

A `compose` helper makes the chain explicit:

```js
const compose = (...fns) => (x) => fns.reduceRight((acc, fn) => fn(acc), x);

const greet = compose(exclaim, lower, trim);
greet('  Hello  ');   // 'hello!'
```

Right-to-left ("compose" mathematically — `f ∘ g(x) = f(g(x))`).

For left-to-right reading, `pipe`:

```js
const pipe = (...fns) => (x) => fns.reduce((acc, fn) => fn(acc), x);

const greet = pipe(trim, lower, exclaim);
greet('  Hello  ');   // 'hello!'
```

Reads top-to-bottom. Most JS codebases prefer `pipe`.

## Where composition shines

Data pipelines:

```js
const processOrders = pipe(
  filter(o => o.status === 'paid'),
  map(o => ({ ...o, total: o.items.reduce((s, i) => s + i.price, 0) })),
  sortBy(o => o.total),
  take(10),
);

const top10 = processOrders(orders);
```

Each step is one function. The pipeline reads as a sentence. Easy
to insert / remove / reorder steps.

For this style at scale, libraries like **Ramda**, **lodash/fp**,
and **fp-ts** provide curried versions of every common operation.

## TC39 pipeline operator

The pipeline operator (`|>`) is at Stage 2 in TC39 as of 2026 —
parked there for years due to a split between two competing forms:
**Hack-style** (`x |> fn(^)` with a placeholder) and **F#-style**
(`x |> fn` implicit single-argument). The committee hasn\'t
resolved which to standardize, so the proposal hasn\'t advanced.

```js
// Hypothetical syntax (not yet shipped)
const result = '  Hello  '
  |> trim
  |> lower
  |> exclaim;
```

Native pipe. Eliminates the helper function. Babel can transpile
it; most production code doesn't use it yet.

## Currying vs partial application

Two related but distinct concepts:

```js
// Partial application — fix some args, return a function
function partial(fn, ...preset) {
  return (...rest) => fn(...preset, ...rest);
}

const greet = (greeting, name) => `${greeting}, ${name}`;
const hello = partial(greet, 'Hello');
hello('Jon');   // 'Hello, Jon'
```

`bind` is partial application:

```js
const hello = greet.bind(null, 'Hello');
```

Currying is a *specific kind* of partial application: one argument
at a time. Most "currying" libraries actually do partial
application — they let you pass multiple args at once.

## When currying / composition hurts

These styles aren't free:

- **Stack frames per call** — each curried call is a function
  invocation. Hot loops with deeply curried operations have
  overhead.
- **Debugging** — stack traces show nested wrapper functions; the
  failing logic is several frames deep.
- **Argument order matters** — curried functions must accept the
  "data" argument last (so partial application binds config
  upfront). Forces a different mental model from standard JS.

For a 30-line script processing a CSV, plain code is clearer. For a
library with many composable primitives, currying earns its keep.

## Composition in React

Higher-order components were the original React composition:

```jsx
const withAuth = (Component) => (props) =>
  props.user ? <Component {...props} /> : <Login />;

const withLogging = (Component) => (props) => {
  console.log('render', Component.name);
  return <Component {...props} />;
};

const Enhanced = withAuth(withLogging(MyComponent));
```

Modern React uses **hooks** for the same job — composition via
function calls inside a component, not function-wrapping. Hooks are
the React-y composition.

```jsx
function MyComponent() {
  const user = useUser();
  useLogging('MyComponent');
  if (!user) return <Login />;
  // ...
}
```

Cleaner. The pattern of "composing behavior" is preserved; the
mechanism changed.

## Composition in Express

Middleware chains are composition:

```js
app.use(logger);
app.use(auth);
app.use(parseBody);
app.use(routes);
```

Each middleware wraps the next. The result is a function pipeline
where each step decides whether to continue.

## Patterns to avoid

### Currying everything

```js
const isAdult = curry((age) => age >= 18);
```

Single-argument functions don't need currying. The wrap adds
overhead without benefit.

### Deep `compose` chains for trivial work

```js
const upper = compose(s => s + '!', s => s.toLowerCase(), s => s.trim());
```

For three operations on one value, just write them inline:

```js
const upper = s => (s.trim().toLowerCase() + '!');
```

Composition pays off when you have many small, named operations
worth composing — not when you're cramming three transformations
into one helper.

### Mixing curry / uncurried libraries

```js
const result = map(filter(arr, isAdult), addOne);
```

If `filter` is uncurried (`filter(arr, fn)`) and `map` is curried
(`map(fn)(arr)`), the call order is confusing. Pick one style per
codebase.

### Forgetting argument order

```js
const filterAdults = filter(arr);   // for uncurried filter, this passes arr as fn
```

Function-first vs data-first matters. Lodash is data-first
(`_.filter(arr, fn)`). Lodash/fp and Ramda are function-first
(`R.filter(fn)(arr)`). Reading the library docs first saves
confusion.

## The mental model

**Currying**: rewrite `f(a, b, c)` as `f(a)(b)(c)`. Enables partial
application and slot-into-pipeline use.

**Composition**: `f(g(x))` = applying `g` then `f`. `pipe(g, f)` is
the same idea written left-to-right. Build complex transforms from
simple parts.

Both styles are about **expressing data flow as small, named,
reusable steps**. JavaScript supports them natively but doesn't
optimize for them — pure-FP languages (Haskell, Elm) make these
the default; JS makes them an option.

In 2026 JS:

- Use composition / pipe for data pipelines with named steps.
- Use partial application (or `bind`) for "preconfigure a function."
- Don't auto-curry by default. Curry specific functions when the
  ergonomics call for it.
- For full FP, reach for `fp-ts` or `effect-ts` — but understand
  the learning curve and bundle cost.

For most JS code, the win is in **clear naming + small functions**,
not formal FP. Currying and composition are tools — use them where
they make code clearer, not as ideology.
