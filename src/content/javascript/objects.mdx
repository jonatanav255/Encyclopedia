# Objects

Objects are JavaScript's universal container — keyed collections of
values. Almost everything that isn't a primitive is an object: arrays,
functions, dates, regexes, class instances.

## Creating objects

```js
const a = {};                    // empty
const b = { name: 'jon', age: 30 };   // literal
const c = Object.create(null);   // no prototype — pure dictionary
const d = new SomeClass();
const e = Object.fromEntries([['a', 1], ['b', 2]]); // { a: 1, b: 2 }
```

Object literals are by far the most common. Reach for
`Object.create(null)` when you need a "pure" dictionary with no
inherited methods like `hasOwnProperty` — handy when keys come from
user input.

## Accessing properties

```js
const user = { name: 'jon', 'first-name': 'jon' };

user.name;            // "jon"  — dot notation
user['name'];         // "jon"  — bracket notation
user['first-name'];   // "jon"  — required for non-identifier keys

const key = 'name';
user[key];            // "jon"  — dynamic key
```

Dot notation when you know the key statically; bracket when the key
is dynamic, contains hyphens, or starts with a number.

## Shorthand and computed keys

```js
const name = 'jon';
const age = 30;

const user = { name, age };   // shorthand: { name: name, age: age }

const k = 'level';
const obj = { [k]: 'senior' }; // computed key: { level: 'senior' }

const greet = {
  hi() { return 'hi'; },       // method shorthand
};
```

## Spread, rest, destructuring

```js
const a = { x: 1, y: 2 };
const b = { ...a, z: 3 };      // { x: 1, y: 2, z: 3 } — shallow copy + add

const { x, ...rest } = a;       // x = 1, rest = { y: 2 }

// Rename + default while destructuring
const { x: xx, missing = 'fallback' } = a;
// xx = 1, missing = 'fallback'

// Nested destructuring
const config = { db: { host: 'localhost', port: 5432 } };
const { db: { host, port } } = config;
```

Destructuring + defaults + renaming is the swiss army knife of object
unpacking — most modern code reaches for it instead of repeated
`config.db.host` chains.

## Common operations

```js
Object.keys(obj);     // ['name', 'age']
Object.values(obj);   // ['jon', 30]
Object.entries(obj);  // [['name', 'jon'], ['age', 30]]

Object.assign({}, a, b); // merge — like { ...a, ...b }
Object.fromEntries(arr); // entries back to object

'name' in user;       // true   — own or inherited
user.hasOwnProperty('name'); // true — own only
Object.hasOwn(user, 'name'); // safer — works on null-prototype objects

delete user.age;      // removes property
```

`Object.hasOwn` (ES2022) is the safer replacement for
`hasOwnProperty` — it works even if the object has no prototype or has
overridden the method.

## Property descriptors

Every property has metadata — writability, enumerability,
configurability — accessed via descriptors:

```js
Object.defineProperty(obj, 'id', {
  value: 1,
  writable: false,
  enumerable: false,
  configurable: false,
});

Object.getOwnPropertyDescriptor(obj, 'id');
// { value: 1, writable: false, enumerable: false, configurable: false }
```

You rarely use these directly — class fields, `Object.freeze`, and
private fields cover the common cases. But knowing they exist explains
why `for...in` sometimes skips properties (non-enumerable).

## Freezing

```js
const obj = Object.freeze({ a: 1 });
obj.a = 2;            // silently fails (throws in strict mode)
obj.b = 3;            // silently fails
Object.isFrozen(obj); // true
```

`Object.freeze` is **shallow**. Nested objects are still mutable.

```js
const config = Object.freeze({ db: { host: 'x' } });
config.db.host = 'y';   // works! db is not frozen
```

For deep freeze, recurse manually or use a library.

## Iterating

```js
const user = { name: 'jon', age: 30 };

for (const key of Object.keys(user)) {
  console.log(key, user[key]);
}

for (const [key, value] of Object.entries(user)) {
  console.log(key, value);
}

// for...in iterates own + inherited enumerable string keys — rarely what you want
for (const key in user) {
  if (Object.hasOwn(user, key)) {
    console.log(key);
  }
}
```

`for...in` is the legacy way and has the "walks inherited properties"
gotcha. Use `Object.keys/values/entries` for own properties only.

## Map vs object

Both store key-value pairs. Differences:

| | Object | Map |
| --- | --- | --- |
| Keys | strings/symbols only | any value (including objects) |
| Iteration order | insertion (mostly) | insertion (guaranteed) |
| Size | `Object.keys(o).length` | `m.size` |
| Default prototype | inherits from Object | none |
| JSON | yes | no — must convert |

Use `Map` when:
- Keys are not strings.
- You add/remove frequently (Map is optimized for this).
- You need guaranteed iteration order.

Use plain objects for everything else — JSON-like data, configuration,
typed records.

## Common gotchas

### Key coercion

Object keys are strings (or symbols). Anything else gets stringified:

```js
const o = {};
o[1] = 'a';
o['1'] = 'b';
console.log(o[1], o['1']);  // 'b' 'b' — same key

o[{}] = 'x';
o[{ different: true }] = 'y';
console.log(o);     // { '[object Object]': 'y' } — both keys stringified the same
```

Use a `Map` if you want object keys to stay distinct.

### Equality is by reference

```js
const a = { x: 1 };
const b = { x: 1 };
a === b;            // false
JSON.stringify(a) === JSON.stringify(b); // true — crude deep equal
```

### Shallow copies leak references

```js
const a = { nested: { count: 1 } };
const b = { ...a };
b.nested.count = 99;
console.log(a.nested.count); // 99 — same nested object
```

`{ ...obj }` clones one level. Deep copies need `structuredClone(obj)`
or a library.

### `undefined` vs missing

```js
const a = {};
a.x === undefined;  // true
'x' in a;           // false

const b = { x: undefined };
b.x === undefined;  // true
'x' in b;           // true
```

If you care about the distinction (e.g., JSON serialization treats
them differently), check `in` or `Object.hasOwn`.
