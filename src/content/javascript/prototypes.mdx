# Prototypes

Every JavaScript object has an internal link to another object called
its **prototype**. When you access a property the engine doesn't find
on the object itself, it walks up the prototype chain looking for it.
That chain is how inheritance works — there are no classes underneath,
just object links.

## The prototype chain

```js
const obj = {};
Object.getPrototypeOf(obj);     // Object.prototype
Object.getPrototypeOf(Object.prototype); // null — end of chain

const arr = [];
Object.getPrototypeOf(arr);     // Array.prototype
Object.getPrototypeOf(Array.prototype); // Object.prototype
```

When you call `arr.map(...)`, the engine:

1. Checks `arr` for a `map` property — not there.
2. Walks to `Array.prototype` — finds `map`. Uses it.

`arr.toString` walks one more level, to `Object.prototype`.
`arr.somethingNonExistent` walks all the way to `null` and returns
`undefined`.

## Classes are syntactic sugar

```js
class Animal {
  constructor(name) { this.name = name; }
  speak() { return `${this.name} makes a sound`; }
}

class Dog extends Animal {
  speak() { return `${this.name} barks`; }
}

const d = new Dog('rex');
d.speak();   // "rex barks"
```

Under the hood:

- `Dog.prototype.__proto__ === Animal.prototype` — inheritance.
- `d.__proto__ === Dog.prototype` — instance link.
- Method lookup: `d` → `Dog.prototype` (finds `speak`) → stops.

The same can be written with prototypes directly (don't, but
understand it):

```js
function Animal(name) { this.name = name; }
Animal.prototype.speak = function() { return `${this.name} makes a sound`; };

function Dog(name) { Animal.call(this, name); }
Dog.prototype = Object.create(Animal.prototype);
Dog.prototype.constructor = Dog;
Dog.prototype.speak = function() { return `${this.name} barks`; };

const d = new Dog('rex');
```

Classes give you the same chain with a saner syntax.

## Own vs inherited properties

```js
const arr = [1, 2, 3];

arr.hasOwnProperty('length'); // true — own
arr.hasOwnProperty('map');    // false — inherited
arr.length;                   // 3
arr.map;                      // function, from Array.prototype

Object.keys(arr);             // ['0', '1', '2'] — own enumerable
Object.getOwnPropertyNames(arr); // ['0', '1', '2', 'length']
```

`Object.keys`, `JSON.stringify`, and `for...in` (with a guard) only
see own properties. The prototype chain is consulted only for value
lookup, not enumeration.

Use `Object.hasOwn(obj, key)` instead of `obj.hasOwnProperty(key)` —
safer and works on null-prototype objects.

## Creating with a specific prototype

```js
const animal = { speak: function() { return 'sound'; } };
const dog = Object.create(animal);
dog.speak();             // "sound" — inherited

const pure = Object.create(null);
pure.toString;            // undefined — no prototype, no Object methods
```

`Object.create(null)` is the way to make a "pure dictionary" that
won't accidentally inherit `hasOwnProperty`, `toString`, etc. Useful
when keys come from user input — guards against
prototype-pollution-style mistakes.

## `__proto__` vs `prototype`

Two confusingly similar names:

- `__proto__` (or `Object.getPrototypeOf(obj)`): the link on an
  *instance* pointing to its prototype.
- `Function.prototype`: the object that becomes the `__proto__` of
  instances created with `new`.

```js
function Dog() {}
const d = new Dog();

d.__proto__ === Dog.prototype;       // true
Dog.__proto__ === Function.prototype; // true — Dog itself is a function
```

`__proto__` is technically deprecated as a property accessor; use
`Object.getPrototypeOf` / `Object.setPrototypeOf` for correctness.

## Don't mutate prototypes of built-ins

```js
Array.prototype.last = function() { return this[this.length - 1]; };
[1, 2, 3].last(); // 3
```

Tempting. Don't. Reasons:

- Conflicts with future spec additions (`Array.prototype.last` could
  become a standard method).
- Breaks `for...in` on arrays (added properties are enumerable by
  default).
- Surprises library code that didn't expect prototype additions.

If you want array-like helpers, use plain functions: `last(arr)`.

## Shadowing

```js
class Base { greet() { return 'base'; } }
class Sub extends Base { greet() { return 'sub'; } }

const s = new Sub();
s.greet();     // "sub" — own method found first
```

`Sub`'s `greet` shadows `Base`'s. Call the parent's version
explicitly with `super`:

```js
class Sub extends Base {
  greet() { return super.greet() + ' (extended)'; }
}
```

## `instanceof`

Walks the prototype chain looking for the constructor's `prototype`:

```js
[] instanceof Array;       // true
[] instanceof Object;      // true — Array inherits from Object
new Date() instanceof Date; // true
```

False if the prototype was swapped or the object came from another
realm (iframe, vm.Module). For cross-realm-safe type checks use
`Array.isArray(x)`, `typeof`, or `Object.prototype.toString.call(x)`.

## Performance

V8 (and other engines) optimize property access aggressively. The hot
path assumes the prototype chain is stable. Mutating a prototype after
many instances exist (`Foo.prototype.newMethod = ...`) invalidates
caches and slows everything down.

Define the full shape at class definition, don't tack on methods at
runtime.

## When prototypes still matter

In modern code, you mostly think in classes and modules. But
prototype-shaped thinking comes up:

- **Mixins** — copying methods between prototypes to combine behaviors.
- **Subclassing built-ins** — `class MyArray extends Array` works
  because of prototype chains.
- **Debugging mysterious "is not a function" errors** — usually a
  method that's on a prototype the object doesn't actually inherit
  from (passed something that quacks like an array but isn't).
- **`Object.create(null)`** — when you want a dict with no inherited
  surface area.
- **Performance investigations** — V8's inline caches break when
  prototype shapes change.

The class syntax hides the chain but doesn't remove it. Understanding
prototypes is what makes "weird" inheritance bugs straightforward to
diagnose.
