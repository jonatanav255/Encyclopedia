# Modules

JavaScript has two main module systems: **ES modules (ESM)** — the
standard — and **CommonJS (CJS)** — the legacy Node format. Modern
code uses ESM; you'll see CJS in older Node code and some npm
packages.

## ES modules (ESM)

The standard. Used in browsers (`<script type="module">`), Node (with
`.mjs` or `"type": "module"` in `package.json`), and every modern
bundler.

```js
// math.js
export function add(a, b) { return a + b; }
export const PI = 3.14;
export default function sub(a, b) { return a - b; }
```

```js
// app.js
import sub, { add, PI } from './math.js';

add(1, 2);  // 3
sub(5, 2);  // 3
```

### Named vs default exports

```js
// Named exports — any number per file
export const foo = 1;
export function bar() {}
export { someInternal as renamed };

// Default export — at most one per file
export default class Widget {}
```

Importers choose names independently of defaults:

```js
import Anything from './widget.js'; // default — pick your name
import { foo, bar } from './math.js'; // named — must match exactly
import { foo as f } from './math.js'; // rename on import
```

Defaults are convenient for "one thing per file" patterns (React
components, classes). Named exports scale better — they're greppable,
support multiple exports per file, and play well with auto-imports in
editors. Many codebases ban default exports for that reason.

### Re-exports

```js
// barrel.js
export * from './math.js';
export { Widget } from './widget.js';
export { default as Reader } from './reader.js';
```

Useful for "index" modules that consolidate exports, but use
sparingly — large barrels hurt tree-shaking (bundlers can't always
prove which re-exports are unused).

### Live bindings

Imports are **live bindings**, not copies:

```js
// counter.js
export let count = 0;
export function inc() { count++; }
```

```js
// app.js
import { count, inc } from './counter.js';
console.log(count); // 0
inc();
console.log(count); // 1 — sees the update!
```

You can't reassign an import (`count = 5` throws), but its value
updates when the source module changes it. CJS doesn't do this.

### Dynamic import

```js
const { add } = await import('./math.js');
add(1, 2);
```

Returns a promise. Useful for code-splitting (only load when needed)
and for conditional imports.

```js
if (cond) {
  const heavy = await import('./heavy.js');
  heavy.run();
}
```

### `import.meta`

The module's own metadata:

```js
import.meta.url;       // "file:///path/to/file.js"
import.meta.dirname;   // (recent Node) the directory
```

In Vite, `import.meta.glob` and `import.meta.env` are powerful
bundler-specific extensions.

## CommonJS (CJS)

The original Node module system. Synchronous, file-by-file.

```js
// math.js
function add(a, b) { return a + b; }
function sub(a, b) { return a - b; }

module.exports = { add, sub };
// or: exports.add = add;
```

```js
// app.js
const { add, sub } = require('./math.js');
const math = require('./math.js'); // entire object
```

`require` is synchronous and dynamic — you can call it inside an `if`.
That's its biggest difference from ESM.

### Interop

The bumpy seam:

```js
// CJS file
module.exports = { foo: 1 };

// ESM importing CJS
import cjs from './cjs.js';            // gets { foo: 1 }
import { foo } from './cjs.js';        // works in Node, depending on version

// CJS importing ESM
const esm = require('./esm.mjs');     // synchronous require of async ESM — error
const esm = await import('./esm.mjs'); // works (top-level await needed)
```

Node has gotten better at this in recent versions, but interop edges
are still rough. The clean move is to pick ESM for all new code and
treat CJS dependencies as "they work, but I can't import them
synchronously from ESM if they have top-level `await`."

### `__dirname` and `__filename`

Available in CJS:

```js
console.log(__dirname);  // directory of this file
console.log(__filename); // full path
```

Not available in ESM. Use:

```js
import { dirname } from 'node:path';
import { fileURLToPath } from 'node:url';
const __dirname = dirname(fileURLToPath(import.meta.url));
```

Recent Node added `import.meta.dirname` and `import.meta.filename` to
make this less painful.

## Module resolution

When you write `import x from 'lib'`, the runtime/bundler has to find
`lib`. The rules:

1. **Relative** (`./foo`, `../bar`) — resolved against the importing
   file's directory.
2. **Absolute** (`/foo`) — rare; resolved from project root in
   bundlers, filesystem root in Node.
3. **Bare specifier** (`react`, `node:fs`) — looked up in
   `node_modules` (climbing parent directories) or as a Node built-in
   (`node:` prefix).

In ESM, file extensions are **required**: `./foo.js` not `./foo`. CJS
historically allowed extension omission. Bundlers add their own
resolution layers (TypeScript path aliases, `tsconfig.paths`).

## Tree shaking

Modern bundlers can drop unused exports if they can prove a module is
side-effect-free. ESM's static `import`/`export` syntax makes this
possible; CJS's dynamic `require` makes it nearly impossible.

```js
// good — static, tree-shakable
import { specific } from 'big-lib';

// bad — pulls everything
import * as everything from 'big-lib';

// CJS — bundler usually keeps the whole module
const big = require('big-lib');
```

In `package.json`, `"sideEffects": false` tells bundlers the package
is safe to tree-shake. List any side-effectful files explicitly if
not.

## `package.json` exports

The modern way to declare what a package exposes:

```json
{
  "name": "my-lib",
  "type": "module",
  "exports": {
    ".": {
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs",
      "types": "./dist/index.d.ts"
    },
    "./utils": "./dist/utils.mjs"
  }
}
```

This gates which files importers can reach. `import 'my-lib/internal'`
fails unless `"internal"` is in `exports`. The `import`/`require`
conditional fields support dual-package (ESM + CJS) publishing.

## Common pitfalls

### Circular imports

```js
// a.js
import { b } from './b.js';
export const a = 1;

// b.js
import { a } from './a.js';
export const b = a + 1;  // a might be undefined if a is still loading
```

ESM handles this better than CJS (live bindings give you the eventual
value), but circular imports usually signal that two modules should be
one or that a third should hold shared state.

### Top-level `await`

ESM supports `await` at the top level of a module:

```js
const config = await loadConfig();
export { config };
```

Anything importing this module waits for the `await` to settle.
Useful for setup, but it can serialize startup if overused.

### File extensions

Native ESM in Node and browsers requires explicit `.js` (or `.mjs`)
extensions:

```js
import { foo } from './foo';     // works in TS, fails at runtime
import { foo } from './foo.js';  // works
```

TypeScript's `--moduleResolution NodeNext` setting enforces this even
when you write `.ts` source.

### The "default-of-a-named" trap

```js
// CJS
module.exports = { foo: 1 };

// ESM importing
import mod from './cjs.js';  // { foo: 1 } — or fails, depending on Node version
import { foo } from './cjs.js'; // works in some versions, error in others
```

When in doubt, `import mod from ...` and then `mod.foo`. The more
explicit form is portable.

## The discipline

- Default to ESM.
- Named exports over defaults.
- Use `node:` prefix for built-ins (`node:fs`, `node:path`).
- Keep modules small and focused — one concept per file.
- Use barrels sparingly; they break tree-shaking and obscure imports.
- For libraries, publish ESM + types, optionally CJS for legacy.
