# Regular expressions

Regex matches patterns in strings. JavaScript's regex engine is fast
for typical patterns, but has gotchas around state, escaping, and
performance.

## Syntax

```js
const re = /hello/;        // literal — recommended
const re2 = new RegExp('hello'); // string-built — for dynamic patterns
```

Literals are preferred — clearer, syntax-highlighted, and you don't
have to double-escape backslashes:

```js
const literal = /\d+/;
const dynamic = new RegExp('\\d+');  // double-escaped
```

Flags after the closing `/`:

- `g` — global, find all matches.
- `i` — case-insensitive.
- `m` — multiline (`^`/`$` match per line).
- `s` — dotall (`.` matches newlines).
- `u` — Unicode mode.
- `y` — sticky.

## Basic matching

```js
/cat/.test('the cat sat');   // true — does it match?
'the cat sat'.match(/cat/);  // ['cat', index: 4, ...]

'the cat sat'.replace(/cat/, 'dog'); // "the dog sat"
'the cat'.split(/\s+/);              // ['the', 'cat']

'aa bb cc'.matchAll(/(\w+)/g);
// iterator over all matches, with capture groups
for (const m of 'aa bb cc'.matchAll(/(\w+)/g)) console.log(m[0]);
// "aa", "bb", "cc"
```

## Character classes

```js
/[abc]/         // a, b, or c
/[^abc]/        // not a, b, or c
/[a-z]/         // any lowercase
/[a-zA-Z0-9_]/  // word character — same as \w
/./             // any char except newline (unless s flag)
/\d/  /\D/      // digit / non-digit
/\w/  /\W/      // word / non-word
/\s/  /\S/      // whitespace / non-whitespace
```

`\d`, `\w`, `\s` are short for character classes — ASCII by default,
extended in Unicode mode.

## Quantifiers

```js
/a*/   // zero or more (greedy)
/a+/   // one or more
/a?/   // zero or one
/a{3}/ // exactly 3
/a{2,5}/ // 2 to 5
/a{2,}/  // 2 or more
```

By default, quantifiers are **greedy** — they grab as much as
possible. Add `?` to make them **lazy**:

```js
'<a><b>'.match(/<.+>/);   // ['<a><b>']    — greedy
'<a><b>'.match(/<.+?>/);  // ['<a>']        — lazy
```

## Anchors

```js
/^hello/     // beginning of string (or line with `m`)
/world$/     // end of string (or line with `m`)
/\bcat\b/    // word boundary
```

## Capture groups

```js
const m = '2026-05-20'.match(/(\d{4})-(\d{2})-(\d{2})/);
m[0]; // "2026-05-20"  — full match
m[1]; // "2026"        — group 1
m[2]; // "05"
m[3]; // "20"
```

Named groups (ES2018+) are clearer:

```js
const m = '2026-05-20'.match(/(?<year>\d{4})-(?<month>\d{2})-(?<day>\d{2})/);
m.groups.year;  // "2026"
m.groups.month; // "05"
```

Non-capturing group: `(?:...)` matches without storing.

## Backreferences

```js
/(\w+) \1/  // matches "hi hi", "the the" — repeated word
```

Inside the pattern, `\1` matches whatever group 1 matched. Useful for
HTML/XML pair matching at simple levels (but use a real parser for
real markup).

## Lookahead / lookbehind

Assertions that match without consuming:

```js
/\d+(?=px)/    // digits FOLLOWED BY 'px'   — "10px" → "10"
/\d+(?!px)/    // digits NOT followed by 'px'
/(?<=\$)\d+/   // digits PRECEDED BY '$'    — "$100" → "100"
/(?<!\$)\d+/   // digits NOT preceded by '$'
```

## Flags in detail

### `g` (global)

Required for "find all matches" via `match`, `replaceAll`, `matchAll`.
Also enables the **`lastIndex` state** on the regex:

```js
const re = /\d+/g;
re.exec('1 2 3'); // ['1', index: 0, ...]   — lastIndex now 1
re.exec('1 2 3'); // ['2', index: 2, ...]
re.exec('1 2 3'); // ['3', index: 4, ...]
re.exec('1 2 3'); // null                     — exhausted
re.exec('1 2 3'); // ['1', index: 0, ...]   — resets
```

This state can bite you — calling `re.test(str)` advances `lastIndex`
too. Don't share global regex instances across unrelated calls.

### `i`, `m`, `s`

```js
/hello/i.test('HELLO');        // true
/^line/m.test('foo\nline two'); // true — m makes ^ match line starts
/a.b/s.test('a\nb');            // true — s makes . match newlines
```

### `u` (Unicode)

Enables proper Unicode support. Without it, `\d` is still
ASCII-only, but pattern syntax matters:

```js
/\u{1F600}/u.test('😀');   // true
/[😀-😎]/u.test('😀');     // surrogate-aware range
```

For real internationalized text, use `u` (or `v`, the newer variant).

### `y` (sticky)

Match only at `lastIndex`, not anywhere later. Useful for tokenizers
that walk through a string position-by-position.

## Replace with function

```js
'hello WORLD'.replace(/\w+/g, (m) => m.toUpperCase());
// "HELLO WORLD"

'2026-05-20'.replace(/(\d{4})-(\d{2})-(\d{2})/, (_, y, m, d) => `${d}/${m}/${y}`);
// "20/05/2026"
```

The callback receives the full match, capture groups, the index, and
the original string. Whatever it returns is inserted.

## Performance: catastrophic backtracking

The big footgun. Some patterns can take exponential time on
adversarial input:

```js
/^(a+)+$/.test('aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaab');
// can take seconds or minutes
```

The engine tries many ways to split the `a`s between the inner and
outer `+`. Each `a` doubles the search space.

This is **ReDoS** — Regular expression Denial of Service. If user
input feeds into a vulnerable regex, an attacker can hang your
process.

### Avoiding it

- Avoid nested quantifiers (`(a+)+`, `(.*)*`, `(.*).+`).
- Avoid alternations with overlapping branches (`(foo|foo)`).
- Anchor patterns (`^`, `$`) when you expect specific positions.
- Test patterns with adversarial inputs — `'a'.repeat(50) + '!'`.

Tools: `safe-regex`, `eslint-plugin-redos`, or migrate to engines like
`re2` (no backtracking, no exponential blowup).

## Common pitfalls

### Escaping in `new RegExp`

```js
function escapeRe(s) {
  return s.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
}

new RegExp(escapeRe(userInput) + '$');  // safe-ish
```

Always escape user-supplied strings before building a regex.

### `replaceAll` vs `replace` with `g`

```js
'a.b.c'.replace('.', '-');       // "a-b.c"  — first match only
'a.b.c'.replace(/\./g, '-');     // "a-b-c"
'a.b.c'.replaceAll('.', '-');    // "a-b-c"  — preferred for strings
```

`replaceAll` with a string argument is clearer. `replaceAll` with a
regex requires the `g` flag (throws otherwise).

### Regex object reuse with `g`

```js
const re = /a/g;
re.test('a');  // true, lastIndex now 1
re.test('a');  // false! position 1, "a" already past
```

If you're using `g` and `test`/`exec`, don't share the instance across
unrelated calls. Or reset `re.lastIndex = 0` between uses. Easier:
build a fresh regex each time.

## When NOT to use regex

- **Parsing HTML, XML, JSON, code.** Use a parser.
- **URL parsing.** Use `URL`.
- **CSV parsing.** Use a CSV library (quoted strings, escaped
  commas).
- **Anything with nested structures.** Regex is a flat language; it
  can't count parens.

Regex is great for tokenizing flat patterns (digits, emails *roughly*,
phone numbers in a known format). For everything else, use a parser
or a more specialized tool.
