# Content negotiation

Content negotiation lets a single URL serve different
representations based on what the client wants — JSON vs HTML,
English vs French, gzipped vs not. The HTTP-level protocol
(`Accept`, `Accept-Language`, `Accept-Encoding`, `q=` weights,
`Vary`, the `406` status) is covered in
**http/content-negotiation**. This page is the Express-specific
view: `res.format`, `req.accepts`, `res.vary`, and the gotchas
that bite Express apps in particular.

For most APIs that only serve JSON, you barely think about it. The
moment you serve both API + HTML, or multiple languages, or
compressed + uncompressed, the Express mechanics matter.

## `Accept` and `res.format`

```js
app.get('/users/:id', async (req, res) => {
  const user = await db.users.findById(req.params.id);

  res.format({
    'application/json': () => res.json(user),
    'text/html':        () => res.render('user', { user }),
    'text/plain':       () => res.send(`${user.name}\n${user.email}`),
    default:            () => res.status(406).send('Not Acceptable'),
  });
});
```

The client sends `Accept: text/html` or `Accept: application/json`.
`res.format` picks the matching handler. The `default` runs if no
declared type matches.

`res.format` parses the full `Accept` header (including `q=`
weights — see the HTTP page) and dispatches to the matching
handler. For browser-default `Accept` headers (heavy on
`text/html`), this typically routes to `text/html`; for SDK
clients sending `Accept: application/json`, it routes to JSON.

## `req.accepts`

```js
const type = req.accepts(['json', 'html', 'text']);
// type === 'json' or 'html' or 'text' or false
```

Returns the highest-priority acceptable type that's in the list, or
`false` if none matches. Useful when you want different logic per
type without `res.format`'s callback structure.

```js
const type = req.accepts(['html', 'json']);
if (type === 'html') return res.render(...);
if (type === 'json') return res.json(...);
res.sendStatus(406);
```

## `Vary` — Express won't set it for you

The `Vary` header tells caches "this response depended on these
request headers." (See **http/content-negotiation** for what
goes wrong without it.) The Express-specific footgun: **`res.format`
does not automatically set `Vary: Accept`** (as of Express 5). You
have to add it:

```js
res.vary('Accept');
res.format({ ... });
```

`compression()` middleware *does* set `Vary: Accept-Encoding`
itself. Most other negotiation axes are on you.

## `req.acceptsLanguages` and `req.acceptsCharsets`

Same shape as `req.accepts`, scoped to the corresponding header.
`req.acceptsLanguages(['en', 'es', 'fr'])` returns the
best-matching language string or `false`. Don't forget
`res.vary('Accept-Language')`. For real i18n, use a library
(`i18next`, `lingui`, `react-intl`); negotiation only picks the
key.

## `compression()` and streaming

`compression` middleware reads `Accept-Encoding`, picks gzip /
brotli / deflate, sets `Content-Encoding` and
`Vary: Accept-Encoding`. The Express-specific gotcha: **it
buffers**, so it breaks SSE, large downloads, and any other
streamed response. Skip it for streaming routes:

```js
app.use((req, res, next) => {
  if (req.url.startsWith('/stream/')) return next();
  return compression()(req, res, next);
});
```

## `Content-Type` (response)

The server tells the client what was actually returned:

```js
res.type('application/json');
res.type('json');          // shortcut — same as above
res.type('html');
res.type('xml');
res.json(obj);             // sets Content-Type: application/json automatically
```

If your `res.format` handlers don't set `Content-Type`, set it:

```js
res.format({
  'application/json': () => {
    res.type('application/json');
    res.send(JSON.stringify(user));
  },
});
```

Browsers and APIs both care.

## CORS and content negotiation

```http
Access-Control-Expose-Headers: Content-Type, Vary
```

Browsers don't expose all response headers to JS unless allowed.
Expose `Content-Type` if your client code reads it. Expose `Vary`
if it matters for client caching.

## Putting it together — a multi-format endpoint

```js
app.get('/users/:id', async (req, res) => {
  const user = await db.users.findById(req.params.id);
  if (!user) return res.sendStatus(404);

  res.vary('Accept');

  res.format({
    'application/json': () => res.json(user),

    'text/html': () => res.render('user', { user }),

    'text/csv': () => {
      res.type('text/csv');
      res.send(`name,email\n${user.name},${user.email}`);
    },

    default: () => res.sendStatus(406),
  });
});
```

JSON for API clients; HTML for browsers; CSV for the curl-and-pipe
crowd. Same URL.

In practice, most APIs just serve JSON and use a separate frontend
that consumes the API. Multi-format on one URL is more common for
data download endpoints (export as CSV / JSON / XLSX).

## `406 Not Acceptable` in Express

`res.format` calls the `default` handler when nothing matches; if
you don't provide one, Express returns `406` automatically. See
the HTTP page for whether `406` is the right policy for your API.

## Common pitfalls

### Forgetting `Vary`

Most common bug. Symptoms: CDN serves the wrong content to half
your users. Fix: `res.vary('Accept')` (and any other negotiated
header).

### `res.format` with overlapping types

```js
res.format({
  'application/json': () => res.json(a),
  'application/*':    () => res.json(b),
});
```

If the client sends `Accept: application/json`, both match. Express
picks the most specific (`application/json`). Less specific patterns
are fallbacks.

### Forgetting that browsers don't always send what you expect

```
Accept: text/html,application/xhtml+xml,application/xml;q=0.9,image/avif,image/webp,image/apng,*/*;q=0.8,application/signed-exchange;v=b3;q=0.7
```

Real-world browser Accept headers are messy. `res.format` and
`req.accepts` handle the parsing.

### Compression buffering streamed responses

`compression()` buffers. For SSE, file downloads, anything streamed,
either disable compression or use streaming compression options.

### Content negotiation in API design

If you only serve JSON, don't bother with `res.format`. It adds
complexity for no benefit. `res.json` is enough.

Content negotiation matters when:

- You serve both API + web pages.
- You serve data in multiple formats (JSON, CSV, XML).
- You support multiple API versions via Accept.
- You serve compressed and uncompressed.

For everything else: `res.json(obj)` and move on.

## The mental model

Content negotiation is **a polite protocol**: client says "I prefer
these, in this order"; server picks the best match it can produce.
Caches respect the `Vary` header to store separate copies per
negotiation axis.

For Express:

- `res.format({ type: fn })` — picks the handler.
- `req.accepts([...])` — direct query for "what should I serve?".
- `res.vary('Accept')` — tell caches about it.
- `compression()` — handles `Accept-Encoding` automatically.

Most APIs only need this for compression and don't think about it.
Multi-format endpoints need every part. Either way, the mechanics
are straightforward once you've seen them.
