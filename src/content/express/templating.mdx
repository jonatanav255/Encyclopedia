# Templating

Express can render HTML server-side via a view engine. You hand it data
plus a template name; it produces an HTML string and sends it. This was
the dominant way to build web apps before SPAs took over — and it's
still the right call for content-driven sites, admin dashboards, and
anything where a SPA is overkill.

## The setup

```js
import express from 'express';
import path from 'node:path';

const app = express();

app.set('views', path.join(import.meta.dirname, 'views'));
app.set('view engine', 'ejs');
```

Two settings:

- **`views`** — directory containing your template files.
- **`view engine`** — the engine name. Express looks for an installed
  package matching that name (`pnpm add ejs`).

Now `res.render('home', { user })` looks for `views/home.ejs` and
renders it with `user` as the data.

## A minimal template (EJS)

```html
<!-- views/home.ejs -->
<!DOCTYPE html>
<html>
  <body>
    <h1>Hello, <%= user.name %></h1>
    <% if (user.isAdmin) { %>
      <a href="/admin">Admin</a>
    <% } %>
  </body>
</html>
```

The handler:

```js
app.get('/', (req, res) => {
  res.render('home', { user: { name: 'Jon', isAdmin: true } });
});
```

`res.render(view, locals)` reads the file, compiles it (once — cached
in production), runs it with `locals`, and sends the resulting string
with `Content-Type: text/html`.

## Picking an engine

| Engine | Style | When to pick |
| --- | --- | --- |
| **EJS** | Inline JS in HTML (`<% %>`, `<%= %>`) | You want HTML that looks like HTML |
| **Pug** | Indented, no closing tags | You like terse syntax, don't mind learning it |
| **Handlebars** | `{{ }}` placeholders, logic via helpers | You want logic-light templates |
| **Nunjucks** | Jinja2-style, powerful inheritance | Coming from Python/Django |

All four are widely used; pick the one you'll enjoy reading six months
later. **EJS** is the safest default — it's just HTML with `<% %>`.

## App-wide locals

Variables you want available in every template:

```js
app.locals.siteName = 'Encyclopedia';
app.locals.currentYear = new Date().getFullYear();

app.use((req, res, next) => {
  res.locals.user = req.user;       // per-request
  next();
});
```

- `app.locals.*` — set once, available everywhere, for static data.
- `res.locals.*` — set per request, available to the template that runs
  for this request.

## Layouts and partials

EJS itself doesn't have built-in layouts — you use `<%- include('partials/header') %>`
for partials. Other engines (Pug, Nunjucks) have inheritance:

```pug
//- layout.pug
html
  body
    block content

//- home.pug
extends layout
block content
  h1= title
```

For non-trivial apps, partials/layouts are essential. Pick an engine
with the inheritance model you want.

## Caching

In production Express caches compiled templates automatically. In
development it recompiles every request, so edits show up without a
restart. The toggle is `NODE_ENV` — set it to `production` and template
caching turns on. If you need caching off in production (rare), set
`app.set('view cache', false)` explicitly.

## Where SSR fits today

For modern apps:

- **Templating** is great for: static sites with server-side data,
  email rendering (yes — `nodemailer` + EJS), admin dashboards, simple
  CRUD interfaces.
- **SPAs** (React, Vue, Svelte) are great for: highly interactive UIs,
  apps that already need a JSON API for mobile, things with rich
  client-side state.
- **Meta-frameworks** (Next.js, Remix, SvelteKit) replace both: they
  do SSR + SPA hydration with one codebase. If you're choosing now and
  you need both, start there instead of "Express + a view engine."

Express's templating story is from an older era. It still works, and
sometimes "just HTML + a sprinkle of JS" is the right tool. But for new
projects with serious frontend ambitions, a meta-framework is usually
better.

## Escaping

Every engine escapes by default — `<%= user.name %>` HTML-escapes the
output. The unescaped version is a separate, scarier syntax
(`<%- user.bio %>` in EJS). Don't use it on user input; that's how XSS
happens.
