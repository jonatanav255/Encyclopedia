# Getting Started

Express is a thin layer on top of Node's built-in `http` module. The smallest
useful app is about six lines: install the package, create an `app`, register
one route, and call `listen` to start the server.

## Installation

```bash
npm install express
```

Express has no runtime build step — it's just a regular npm package. You import
it, configure routes and middleware, then start listening.

## The smallest app

```js
import express from 'express';

const app = express();

app.get('/', (req, res) => {
  res.send('Hello, world!');
});

app.listen(3000, () => {
  console.log('Listening on http://localhost:3000');
});
```

What's actually happening:

1. `express()` returns an `app` object — internally, an ordered list of
   middleware/route handlers, plus methods to register more.
2. `app.get(path, handler)` adds a route to that list. It only fires for
   `GET` requests whose URL matches `path`.
3. `app.listen(port)` is a convenience for `http.createServer(app).listen(port)`.
   The Node `http` server uses `app` as its request handler.

## What `app` actually is

```js
console.log(typeof app); // "function"
```

`app` is itself a function — `(req, res) => ...`. That's why you can pass it
straight to `http.createServer(app)` or mount it as middleware in another app:

```js
import http from 'node:http';
http.createServer(app).listen(3000);
```

Knowing this demystifies a lot of Express — it's not magic, it's a function
that walks an array of other functions in order.

## A slightly more realistic shape

```js
import express from 'express';

const app = express();

app.use(express.json());                  // parse JSON bodies

app.get('/api/health', (req, res) => {    // a route
  res.json({ status: 'ok' });
});

app.use((err, req, res, next) => {        // error handler
  console.error(err);
  res.status(500).json({ error: 'internal' });
});

const port = process.env.PORT ?? 3000;
app.listen(port, () => {
  console.log(`listening on ${port}`);
});
```

This is the shape almost every Express app converges to:

- A few `app.use(...)` calls registering global middleware (body parsing,
  logging, CORS, etc.).
- A series of routes (`app.get`, `app.post`, ...).
- A final error-handling middleware at the bottom.
- `app.listen` reading the port from the environment.

<Callout variant="tip">
You don't *have* to call `app.listen`. Exporting `app` and letting your tests
import it (without listening) is the standard pattern for `supertest`-style
integration tests — see the **Testing** topic.
</Callout>

## Hot reload while developing

Express won't reload itself when files change. Use a watcher:

```bash
npm install -D nodemon
npx nodemon server.js
```

Or with newer Node (≥ 20.6):

```bash
node --watch server.js
```

Both restart the process on file changes. Pick one — they're functionally
equivalent for a beginner.

## What you don't get out of the box

Express deliberately ships almost nothing beyond routing and middleware
plumbing. You'll add packages for:

- Bodies other than JSON (`multer`, `express.urlencoded`)
- Cookies (`cookie-parser`)
- Sessions (`express-session`)
- CORS (`cors`)
- Templating (`ejs`, `pug`, ...)
- Logging (`morgan`, `pino`)

This is by design — the framework stays small and you compose what you need.
The cost is more decisions upfront; the benefit is no surprise behaviour.
