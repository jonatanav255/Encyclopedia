# HTTP Basics

Every Express handler is dealing in HTTP — a request/response protocol over
TCP. Knowing the parts of a request, the parts of a response, and which methods
mean what is what lets you reason about Express without memorizing API
surface.

## A request, decomposed

```
GET /users/42?include=posts HTTP/1.1
Host: api.example.com
Accept: application/json
Authorization: Bearer abc123
```

Five things:

- **Method** — `GET`. The verb. What the client wants to do.
- **Path** — `/users/42`. Where on the server.
- **Query string** — `?include=posts`. Optional key/value pairs.
- **Headers** — `Host`, `Accept`, `Authorization`, etc. Metadata about the
  request.
- **Body** (not shown — `GET` requests usually don't have one). The payload.

In Express, those map to `req.method`, `req.path`, `req.query`,
`req.headers`, and `req.body`.

## A response, decomposed

```
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 23

{"id":42,"name":"Jon"}
```

Three things:

- **Status code** — `200`. Outcome category (see **Status Codes**).
- **Headers** — `Content-Type`, `Content-Length`, etc.
- **Body** — the payload (JSON, HTML, an image, anything).

In Express: `res.status(200)`, `res.set('Content-Type', 'application/json')`,
`res.send(body)`. Most of the time you use higher-level helpers like
`res.json(obj)` which set the header and serialize for you.

## The methods you'll actually use

  columns={['Method', 'Meaning', 'Has body?', 'Idempotent?']}
  rows={[
    ['GET', 'Read a resource', 'No', 'Yes'],
    ['POST', 'Create a resource', 'Yes', 'No'],
    ['PUT', 'Replace a resource', 'Yes', 'Yes'],
    ['PATCH', 'Partially update', 'Yes', 'No (by convention)'],
    ['DELETE', 'Remove a resource', 'Sometimes', 'Yes'],
    ['HEAD', 'Like GET but headers only', 'No', 'Yes'],
    ['OPTIONS', 'Capability check (CORS preflight)', 'No', 'Yes'],
  ]}

**Idempotent** means doing it N times has the same effect as doing it once.
`GET /users/42` ten times changes nothing. `POST /users` ten times creates
ten users. The browser and proxies treat idempotent methods as safe to retry
on network failure — non-idempotent ones aren't.

## Request/response is stateless

Each HTTP request is independent. The server doesn't remember the previous
one. If you want continuity ("this is the same user as before"), the client
has to carry that state with it — usually a cookie (sessions) or a header
(`Authorization: Bearer ...`).

Express is just a request/response machine. State lives in cookies, headers,
or your database.

## Where Express fits

```
client → TCP → HTTP parser → Node http server → Express app → your handler
```

Node's `http` module parses the bytes into `req` and `res` objects. Express
wraps those, adds conveniences (`req.params`, `req.query`, `res.json`, ...),
and routes the call to the right handler.

<Callout variant="tip">
Express's `req` and `res` are **subclasses** of Node's `IncomingMessage` and
`ServerResponse`. Anything you can do with the native Node API still works in
Express. The framework just adds sugar on top — it doesn't replace the
underlying objects.
</Callout>

## HTTP/1.1 vs HTTP/2

You'll typically write Express as if everything is HTTP/1.1, because that's
the abstraction. HTTP/2 (multiplexing, header compression) is usually
terminated at the reverse proxy (nginx, ALB) in front of Node — by the time
the request reaches Express it's been downgraded to HTTP/1.1 anyway.
Knowing this matters when you start tuning performance.
