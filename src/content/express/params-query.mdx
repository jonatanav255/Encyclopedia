# Params & Query

Two ways routes get input from the URL: **path parameters** (encoded in the
URL structure) and **query strings** (key/value pairs after `?`). Different
purposes, different APIs, both end up on `req`.

## Path parameters

You declare them with `:` in the route path. Express captures them into
`req.params`.

```js
app.get('/users/:id', (req, res) => {
  console.log(req.params.id); // "42"
});
```

`GET /users/42` → `req.params = { id: '42' }`.

Multiple params:

```js
app.get('/users/:userId/posts/:postId', (req, res) => {
  // req.params = { userId, postId }
});
```

<Callout variant="tip">
Params are **always strings**. `req.params.id` is `'42'`, not `42`. Use
`Number(req.params.id)` (and validate it's not `NaN`) before treating it as
a number.
</Callout>

## Query strings

Everything after the `?` in a URL, parsed into an object on `req.query`.

```js
app.get('/search', (req, res) => {
  // GET /search?q=express&page=2
  console.log(req.query); // { q: 'express', page: '2' }
});
```

Like params, query values are strings. They can also be:

- **Repeated keys**: `?tag=js&tag=ts` → `req.query.tag = ['js', 'ts']`
- **Nested** (with the default `qs` parser): `?filter[type]=user` →
  `req.query.filter = { type: 'user' }`

You can disable nesting by configuring the parser:

```js
app.set('query parser', 'simple');
```

## Params vs query — when to use which

| Use a path param when... | Use a query string when... |
| --- | --- |
| The value identifies a resource | The value filters or modifies a result |
| It's required for the route to mean anything | It's optional |
| `/users/42` (this specific user) | `/users?role=admin` (a filter) |

Rule of thumb: if removing it would 404, it's a param. If removing it still
returns a valid (broader) response, it's a query.

## Optional params (Express 4 syntax)

```js
app.get('/users/:id?', (req, res) => {
  if (!req.params.id) return res.json(allUsers);
  res.json(findUser(req.params.id));
});
```

In Express 5 the optional-param syntax changed slightly (uses brackets:
`/users{/:id}`) because the underlying `path-to-regexp` library was bumped.
See **Express 5 Migration**.

## Wildcard params

In Express 4: `*` captures everything after.

```js
app.get('/files/*', (req, res) => {
  console.log(req.params[0]); // "deep/path/file.txt"
});
```

In Express 5 the `*` syntax is gone — you'd write `/files/*splat` or use a
named wildcard. The old form is a common gotcha if you upgrade.

## URL decoding

Express decodes URL-encoded characters automatically.

```
GET /users/hello%20world
→ req.params.id === "hello world"
```

If a param value contains a `/`, it's *not* captured by a single segment.
`GET /files/a/b/c` matches `/files/:name` as `name = 'a'` — `b/c` is the rest
of the path. Use a wildcard for multi-segment captures.

## Casting and validating

Always assume params and queries are user-controlled hostile input.

```js
app.get('/users/:id', (req, res) => {
  const id = Number(req.params.id);
  if (!Number.isInteger(id) || id < 1) {
    return res.status(400).json({ error: 'invalid id' });
  }
  // ... safe to use as a number now
});
```

For anything beyond trivial routes you'll want a schema validator (zod,
joi, ajv) — see **Validation**.

## What `req.path` and `req.url` give you separately

- `req.path` — just the path, no query string: `/users/42`
- `req.url` — path + query: `/users/42?include=posts`
- `req.originalUrl` — the URL before mount paths were stripped (relevant
  when this code runs inside a sub-router)

You rarely need `req.url` directly. `req.params` and `req.query` cover most
real use.
