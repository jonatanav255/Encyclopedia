# Graceful Shutdown

When a deploy starts or an orchestrator scales down, your Node process
gets a `SIGTERM`. Done wrong, the process exits immediately and any
in-flight requests are cut off mid-response — clients see connection
resets, half-written data, dropped uploads. Done right, the process
stops accepting new requests, finishes the in-flight ones, and exits
cleanly.

## The basic pattern

```js
import { app } from './app.js';

const server = app.listen(process.env.PORT ?? 3000);

function shutdown(signal) {
  console.log(`${signal} received, shutting down`);

  server.close((err) => {
    if (err) {
      console.error('error during close', err);
      process.exit(1);
    }
    console.log('http server closed');
    process.exit(0);
  });

  // Hard timeout in case close hangs
  setTimeout(() => {
    console.error('forced shutdown after 30s');
    process.exit(1);
  }, 30_000).unref();
}

process.on('SIGTERM', () => shutdown('SIGTERM'));
process.on('SIGINT', () => shutdown('SIGINT'));
```

What `server.close` does:

1. Stops accepting new connections immediately.
2. Waits for **existing** requests to finish.
3. Calls the callback when the last connection is gone.

It does **not** force-close existing connections. That's the graceful
part.

## The full picture

Most production services have more than just `http`:

```js
async function shutdown(signal) {
  logger.info({ signal }, 'shutting down');

  // 1. Stop the HTTP server (no new requests)
  await new Promise((resolve) => server.close(resolve));

  // 2. Close database connections
  await db.end();

  // 3. Close other resources
  await redis.disconnect();
  await queue.close();

  // 4. Flush logs
  await logger.flush?.();

  process.exit(0);
}

process.on('SIGTERM', () => shutdown('SIGTERM'));
process.on('SIGINT', () => shutdown('SIGINT'));
```

Order matters: stop accepting requests before closing the database,
otherwise the next handler crashes.

## Keep-alive is the gotcha

`server.close` waits for connections to be idle, not requests to
finish. With HTTP keep-alive enabled (default), an idle browser
connection can hold the close pending indefinitely.

In Node 18.2+ you can use `server.closeIdleConnections()` and
`server.closeAllConnections()`:

```js
await new Promise((resolve) => {
  server.close(resolve);
  server.closeIdleConnections();          // close idle keep-alives now
  setTimeout(() => server.closeAllConnections(), 25_000); // hard cap
});
```

For older Node, the `http-graceful-shutdown` package or `stoppable`
do the same thing with manual connection tracking.

## Drain period

Orchestrators send `SIGTERM` and wait a configurable grace period
before forcing `SIGKILL`. Defaults:

- Kubernetes: 30s
- AWS ECS: 30s
- Heroku: 30s

Make sure your shutdown finishes inside that window, or include a
hard timeout that exits before the orchestrator kills you.

If your in-flight requests can legitimately take longer than the
grace period (long uploads, slow queries), increase the orchestrator
grace period instead of letting requests get cut off.

## Health checks during shutdown

Once you've started shutting down, fail the readiness probe so the
load balancer stops sending new requests *before* it would notice via
connection errors:

```js
let healthy = true;

app.get('/health', (req, res) => {
  res.status(healthy ? 200 : 503).json({ ok: healthy });
});

async function shutdown(signal) {
  healthy = false;                          // fail health check first
  await new Promise(r => setTimeout(r, 5_000));  // let LB notice
  // ... rest of shutdown
}
```

The 5-second sleep gives the LB time to mark the instance unhealthy.
Without it, requests routed in the last few seconds before close get
dropped.

## What can go wrong

- **Forgetting `process.on('SIGTERM')`**: Node's default `SIGTERM`
  handler is to exit immediately. You need an explicit handler.
- **Hanging on a slow query**: an in-flight request that takes 10
  minutes will hold close open. Either don't have such requests, or
  set a hard timeout.
- **Closing the DB before requests finish**: handlers in mid-flight
  now error on every query. Close in order: server, then resources.
- **Not testing it**: a graceful shutdown that has never been tested
  is broken. Send `SIGTERM` locally and watch what happens.

## Testing

```bash
node server.js &
PID=$!
sleep 2
curl http://localhost:3000/slow &     # start a slow request
kill -TERM $PID
wait $PID
```

The slow request should complete. The process should exit 0 within
your grace window.

## Don't fight the orchestrator

Long-running jobs, queue workers, scheduled tasks — keep these
**outside** the HTTP process. A queue worker is shut down
differently (drain the current job, ack it, exit). Mixing them with
HTTP makes both shutdowns harder.

If you have an in-process job runner (cron, `node-cron`), stop it in
shutdown:

```js
job.stop();
```

Before closing the DB it was talking to.
