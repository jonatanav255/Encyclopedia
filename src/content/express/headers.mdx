# Headers

HTTP headers carry metadata about the request and response — content type,
authentication, caching, CORS, you name it. Express gives you simple APIs
to read and write them.

## Reading request headers

```js
app.get('/', (req, res) => {
  const auth = req.get('Authorization');       // case-insensitive
  const auth2 = req.headers.authorization;     // lowercased key on the raw object
  const ua = req.get('User-Agent');
});
```

Three things to know:

- **Header names are case-insensitive** at the protocol level. Express
  normalizes them to lowercase on `req.headers` for consistency.
  `req.headers['Content-Type']` is `undefined`; `req.headers['content-type']`
  is the value. `req.get(name)` does the case-insensitive lookup for you.
- **Repeated headers** (rare but legal): `req.headers.set-cookie` is an
  array. Most others are a single string.
- **Whatever the client sent is what you read.** Anything claiming
  authority — `User-Agent`, `Referer`, `X-Forwarded-For` — is
  client-controlled. Don't trust it for security decisions.

## Writing response headers

```js
res.set('Content-Type', 'application/json');
res.set({                                      // batch
  'X-Custom': 'value',
  'Cache-Control': 'no-store',
});
res.append('Set-Cookie', 'a=1');              // adds, doesn't replace
res.get('Content-Type');                       // read what's been set so far
res.removeHeader('X-Powered-By');
```

`set` overwrites. `append` adds another value (for headers like
`Set-Cookie` that can repeat). Once `res.send` / `res.json` / `res.end`
has been called, headers are flushed and further attempts throw
`ERR_HTTP_HEADERS_SENT`.

## Headers Express sets for you

By default Express adds:

- `X-Powered-By: Express` — basically advertising. Disable it:
  ```js
  app.disable('x-powered-by');
  ```
  This doesn't add security, but it does stop telling attackers your stack.
- `Content-Type` — auto-set by `res.json` (`application/json`), `res.send`
  (`text/html` for strings), `res.sendFile` (by extension), etc.
- `Content-Length` / `Transfer-Encoding` — Node figures out one of these.
- `ETag` — Express sets a weak ETag on responses for caching. Configure
  with `app.set('etag', ...)`. See **Caching**.

## Headers you almost never set yourself

Let the framework or Node handle these:

- `Content-Length` — Node computes it from the buffer.
- `Connection`, `Transfer-Encoding` — protocol plumbing.
- `Date` — Node sets it.

## Custom headers

By convention, custom headers start with `X-` (e.g. `X-Request-Id`) — though
RFC 6648 deprecated this convention years ago, it persists in the wild.
Either form is fine; just be consistent within your API.

```js
app.use((req, res, next) => {
  res.set('X-Request-Id', crypto.randomUUID());
  next();
});
```

## CORS headers are just headers

`Access-Control-Allow-Origin`, `Access-Control-Allow-Methods`, etc., are
response headers. The `cors` middleware sets them for you, but there's no
magic — you can set them manually if you understand the protocol. See
**CORS**.

## Reading headers in middleware vs handlers

Headers are available on `req` from the very first middleware. They arrive
before the body, so middleware doesn't need to wait for parsing to inspect
them. That's why auth middleware (which checks `Authorization`) can run
before `express.json()`.

<Callout variant="tip">
If you build APIs that other people consume, document which headers you
accept and which you set. "Hidden" required headers are an integration
nightmare — at minimum, document the auth header and any required
`Content-Type` constraints.
</Callout>
