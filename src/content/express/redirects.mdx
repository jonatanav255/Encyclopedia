# Redirects

A redirect is a response that tells the client "the thing you want is over
there instead." It's a 3xx status code plus a `Location` header.

## The API

```js
res.redirect('/login');                    // default: 302
res.redirect(301, '/new-url');             // explicit status
res.redirect('https://example.com/about'); // absolute is fine too
```

That's the whole thing — Express sets the `Location` header for you and
sends an empty body.

## Picking the status code

| Code | Name | When to use |
| --- | --- | --- |
| **301** | Moved Permanently | The resource lives at a new URL forever. Browsers and SEO crawlers cache this. |
| **302** | Found | Temporary redirect. Default in Express. Most common for "after login, go here." |
| **303** | See Other | Redirect after a POST. Browser switches to GET on follow. Use this for post-redirect-get. |
| **307** | Temporary Redirect | Like 302, but preserves the original method (POST stays POST). |
| **308** | Permanent Redirect | Like 301, but preserves the method. |

The 302/303/307 distinction trips people up because old browsers used to
treat 302 like 303 (silently switch POST → GET), but that's not what the
spec says. Modern browsers respect it correctly.

## Post/Redirect/Get

The classic pattern for HTML form submissions:

```js
app.post('/submit', (req, res) => {
  saveForm(req.body);
  res.redirect(303, '/thanks');  // browser follows with GET /thanks
});
```

Why 303 and not 302: this guarantees the browser switches to GET. Reloading
`/thanks` re-runs the GET, not the POST — no duplicate submissions.

## Permanent vs temporary

Use **301 / 308** sparingly. Browsers cache them aggressively — once the
browser has seen the redirect, it stops asking your server entirely and
goes straight to the new URL. Reversing a 301 you regretted is painful.

Use **302 / 303 / 307** for anything you might change. The cost is one
extra round trip; the benefit is you can switch behaviour without browser
caches fighting you.

<Callout variant="warn">
**Open redirect vulnerability.** Never redirect to a user-supplied URL
without validating it:

```js
app.get('/go', (req, res) => {
  res.redirect(req.query.to);  // ← attacker sends ?to=https://evil.com
});
```

Phishing campaigns love this. Allow-list the destinations, or only allow
relative URLs (start with `/` and don't start with `//`).
</Callout>

## Relative vs absolute

Both work:

```js
res.redirect('/login');                  // relative to current host
res.redirect('https://other.example/');  // absolute
res.redirect('../');                     // also valid
```

For absolute URLs the browser does the protocol/host swap. For relative,
the browser resolves against the current page.

## Redirecting back

A common pattern: send the user back where they came from.

```js
res.redirect('back'); // uses the Referer header, falls back to '/'
```

Treat the Referer with the same suspicion as any user input — it's a
header the client controls. For sensitive flows, store the destination
server-side (in the session) rather than trusting `Referer`.
