# Status Codes

HTTP status codes are how the server tells the client the *category* of the
result. Three digits, organized into five families. Pick them correctly and
your API is self-documenting; pick them carelessly and clients have to look
at the body to know what happened.

## The ones you'll actually reach for

### 2xx — success

- **200 OK** — Generic success with a body. `GET` returning data,
  `PUT` / `PATCH` returning the updated resource.
- **201 Created** — A `POST` that created a resource. Idiomatic to also
  set a `Location` header pointing to the new resource.
- **204 No Content** — Success, but no body. Common for `DELETE`, or for
  `PUT` when the client doesn't need the updated resource echoed back.

```js
res.status(201).json(newUser);
res.status(204).end();
```

### 4xx — the client did something wrong

- **400 Bad Request** — The request itself is malformed (bad JSON, missing
  required field). The client can't retry the same way and expect success.
- **401 Unauthorized** — Authentication is missing or invalid. ("Who are
  you?")
- **403 Forbidden** — Authenticated, but not allowed. ("I know who you are,
  and you can't do this.")
- **404 Not Found** — The resource doesn't exist.
- **409 Conflict** — Request collides with current state (e.g. trying to
  create something that already exists).
- **422 Unprocessable Entity** — Syntactically valid but semantically wrong
  (e.g. JSON parsed fine but the email is not a valid email). Some APIs use
  400 for this — pick one and be consistent.
- **429 Too Many Requests** — Rate limit hit. Almost always paired with a
  `Retry-After` header.

### 5xx — the server did something wrong

- **500 Internal Server Error** — Catch-all "something broke." Your error
  handler returns this when nothing more specific applies.
- **502 Bad Gateway** — You proxied to something upstream and it returned
  garbage.
- **503 Service Unavailable** — You're temporarily down (overloaded,
  shutting down). Often paired with `Retry-After`.
- **504 Gateway Timeout** — Upstream took too long.

## Setting the status in Express

```js
res.status(201).json(data);          // explicit
res.sendStatus(204);                  // status + plain-text body matching
res.status(404).json({ error: 'not found' });
```

`res.status()` only stages the code. The response isn't actually sent until
something like `.json()`, `.send()`, `.end()`, or `.redirect()` is called.

If you call `.json()` / `.send()` without `.status()`, the default is `200`.

## 401 vs 403 — the one people get wrong

- **401**: "I don't know who you are." No credentials, expired token,
  invalid signature. The client should authenticate and try again.
- **403**: "I know who you are, but no." Right credentials, wrong
  permissions. Re-authenticating won't help.

If your handler doesn't even read the token, return 401 when it's missing.
Only after you've validated identity and decided the *action* isn't allowed
should you return 403.

## 200 with `{ error: ... }` is a lie

Don't do this:

```js
res.status(200).json({ error: 'user not found' });
```

The status code is the API contract. Returning 200 for an error breaks
every middleware, log aggregator, and monitoring tool that expects 4xx/5xx
to mean "bad." Use the right status code and put the message in the body
*and* the status.

<Callout variant="tip">
For HTML pages, returning 200 with a "page not found" template is sometimes
done deliberately so search engines don't unindex the URL. For JSON APIs,
always use the right code.
</Callout>

## 304 Not Modified — caching's status code

Browsers and HTTP caches use 304 to mean "you sent an `If-None-Match` or
`If-Modified-Since` header and nothing has changed — keep using your cached
copy." Express handles this for `express.static`; for custom routes you set
it yourself via `res.set('ETag', ...)` and check `req.fresh`. See **Caching**.
