# Express vs Fastify vs Hono

Three Node web frameworks dominate in 2026. The choice isn't about
performance benchmarks (they're all faster than your database). It's
about validation, ecosystem, runtime portability, and DX.

This page is a 2026 decision matrix written from inside an
Express-heavy world.

## The contenders

- **Express** — the granddaddy. Express 5 (mid-2024) finally
  shipped, fixed async error handling, modernized routing. Massive
  ecosystem, every tutorial assumes it, every LLM trained on it.
- **Fastify** — schema-first, JSON-schema validation + serialization
  built in, plugin model, ~2× Express throughput in microbenchmarks
  that no real service hits.
- **Hono** — multi-runtime (Node, Bun, Deno, Cloudflare Workers,
  Vercel Edge). TypeScript-first, tiny core, modern web standards
  (Request/Response).

A few honorable mentions:

- **NestJS** — opinionated full-framework on top of Express or
  Fastify. Big in enterprise, controversial elsewhere.
- **tRPC** — not a framework but a layer: type-safe RPC over
  Express/Fastify/Hono.

## Throughput is rarely the answer

```
"Fastify is 2× faster than Express!"
```

True in microbenchmarks: `GET /` returning a static JSON object,
no middleware, no DB.

In reality, a typical request spends:

- 1–5 ms in framework code.
- 5–50 ms in middleware + handler logic.
- 20–500 ms in DB/network calls.

Framework overhead is < 5% of real-world latency. Picking a framework
for throughput is the wrong axis.

## What each does well

### Express

**Strengths:**

- Ubiquitous. Stack Overflow answers, books, every tutorial.
- LLMs generate good Express code on the first try.
- The entire Node middleware ecosystem (`helmet`, `multer`,
  `cors`, `compression`, `pino-http`) is Express-first.
- Express 5 fixed the long-standing async-error pain.
- Stable, slow-moving — your code from 2018 still runs.

**Weaknesses:**

- No built-in validation. Bring your own (Zod, Joi).
- No first-class TypeScript. Types work but the API isn't TS-idiomatic.
- Schema-less JSON parsing/serialization (`express.json()` just parses).
- Routing matcher (`path-to-regexp`) is fine but not exotic.

Pick when: shipping a Node service today, your team knows Express,
you don't need exotic perf, you have an existing middleware story.

### Fastify

**Strengths:**

- **JSON-schema everywhere**: schemas validate input and *generate
  the fastest possible response serializer* (avoids `JSON.stringify`
  reflection).
- Plugin system with encapsulation (a plugin's hooks don't leak
  outside).
- First-class TypeScript and types-derived-from-schema (`fastify-type-provider-zod`).
- Built-in async error handling (predates Express 5 by years).
- Logging via Pino built in. Hooks for everything.

**Weaknesses:**

- Smaller ecosystem; many libraries assume Express.
- Plugin model is unfamiliar at first; mistakes around encapsulation
  leak.
- LLM code generation is solid but trips on the plugin API more
  often.

Pick when: starting fresh, you want validation-as-architecture, your
team is comfortable learning a new model, you serialize a lot of
JSON.

### Hono

**Strengths:**

- **Multi-runtime**: same code runs on Node, Bun, Deno, Workers, Edge.
- TypeScript-first. The router infers route param types into your
  handlers automatically.
- Tiny core (~12 KB). No middleware unless you opt in.
- Uses Web platform primitives — `Request`/`Response`, not custom
  `req`/`res`.
- Good for "I might deploy this to Workers later" or "I want one
  framework across my serverless functions and my main server."

**Weaknesses:**

- Smaller ecosystem still. Node-specific middleware needs adapters.
- The Node adapter (`@hono/node-server`) is good but slightly thinner
  than running Express directly.
- LLM code generation is improving but less reliable than Express.

Pick when: you want runtime portability, you're greenfield, you want
TypeScript-derived routing types, or you're deploying to the edge.

## Express 5 vs Fastify in 2026

This is the practical question for most Node teams. Three things
have changed:

1. **Express 5 finally awaits async handlers.** The "wrap every
   handler in `asyncHandler`" pain is gone. `express-async-errors`
   is no longer needed.
2. **`req.query` and `req.params` are null-prototype objects.**
   Prototype-pollution attacks via query/params are now structurally
   blocked.
3. **`path-to-regexp` v8 is stricter.** Named wildcards, no bare
   `*`. A short migration headache, lasting safety win.

Fastify still wins on:

- Validation-as-architecture (the JSON schema *is* the contract).
- Faster serialization (when output is schema'd).
- Built-in encapsulation (a plugin can't accidentally pollute
  globals).

Express still wins on:

- Familiarity, ecosystem, training-set coverage.
- Lower cognitive load for "just hand me a function that runs on
  requests."
- Better LLM and tutorial coverage.

If you're starting today with no preference, Fastify is a defensible
default. If you're inheriting Express, the upgrade to Express 5
is more practical than a rewrite.

## Hono on Node

Hono runs on Node via `@hono/node-server`:

```js
import { Hono } from 'hono';
import { serve } from '@hono/node-server';

const app = new Hono();

app.get('/users/:id', (c) => {
  const id = c.req.param('id');   // typed as string
  return c.json({ id });
});

serve({ fetch: app.fetch, port: 3000 });
```

Notice: the route param `id` is typed because Hono's router uses
template-literal types. No manual generic, no runtime cast.

Performance on Node is comparable to Express; the win is portability.
Same `app` code deploys to Bun, Cloudflare Workers, Deno Deploy with
different adapters.

## When NestJS makes sense

NestJS is "Angular for backends": modules, controllers, DI,
decorators, guards, pipes.

**Pros:**

- Strong structure imposed by the framework. Useful for large teams.
- DI container helps with testing (mock dependencies).
- Good for "build me an enterprise CRUD app" patterns.

**Cons:**

- Heavy. Boilerplate per endpoint vs Express's "function".
- Magic via decorators. Hard to grep.
- Performance overhead (it's Express/Fastify under the hood plus
  more layers).
- Trains people in a NestJS-specific paradigm, not raw web.

Pick when: large team, strict architecture preferences, want a
batteries-included framework with DI. Otherwise, Fastify or Express
+ deliberate structure is lighter.

## When tRPC slots in

tRPC layers on top of any framework. It's not a framework choice —
it's a "how do my client and server agree on types" choice.

```ts
// server.ts
export const appRouter = router({
  user: router({
    getById: publicProcedure
      .input(z.string())
      .query(({ input }) => db.users.find(input)),
  }),
});

export type AppRouter = typeof appRouter;
```

```ts
// client.ts
const trpc = createTRPCClient<AppRouter>({ ... });
const user = await trpc.user.getById.query('42');
// user is fully typed
```

For internal APIs (your own frontend + backend), this is a strong
contender. Replaces hand-written REST + types. Works with Express or
Fastify under the hood.

## Decision matrix

| | Express | Fastify | Hono |
| --- | --- | --- | --- |
| Ecosystem size | huge | medium | small |
| Validation | bring own | built-in (JSON schema) | bring own (Zod usually) |
| TypeScript | works | first-class | first-class with type inference |
| Multi-runtime | Node only | Node + Bun | Node, Bun, Deno, Workers, Edge |
| Logging | bring own (pino-http) | built-in (Pino) | bring own |
| Async errors | fixed in v5 | always handled | always handled |
| Learning curve | flat | medium | medium |
| LLM code quality | excellent | good | growing |
| Best for | most Node teams | new schema-first teams | edge / portable |

## The honest recommendation

- **Maintenance / existing Express codebase**: upgrade to Express 5,
  stay. Cost of rewrite > value of switch.
- **Greenfield Node service, small/medium team**: Fastify. The
  schema-first model pays off as the API grows.
- **Greenfield, want edge / portable / TypeScript heaven**: Hono.
- **Internal full-stack TypeScript with a frontend you control**:
  Express/Fastify + tRPC.
- **Large enterprise with imposed structure preference**: NestJS.

There's no wrong answer. Most teams that switch frameworks regret the
cost more than the migration was worth. Pick once, optimize the
codebase under it.
