# Environment

The same code runs in development, staging, and production — but with
different ports, database URLs, API keys, and feature flags. The clean
separation is **environment variables**: configuration lives outside the
code, in the process environment.

## Reading env vars

Node exposes them on `process.env`:

```js
const port = process.env.PORT ?? 3000;
const dbUrl = process.env.DATABASE_URL;
const nodeEnv = process.env.NODE_ENV ?? 'development';

app.listen(port);
```

Every value on `process.env` is a **string** (or `undefined`). Always coerce
numbers and booleans yourself:

```js
const port = Number(process.env.PORT ?? 3000);
const debug = process.env.DEBUG === 'true';
```

## `NODE_ENV`

A convention, not a Node feature. Common values: `development`, `test`,
`production`. Express checks it internally to choose defaults:

- `production` → cached view templates, less verbose errors.
- anything else → no caching, full stack traces in error responses.

```js
if (process.env.NODE_ENV === 'production') {
  app.set('trust proxy', 1);
  // ...other production-only setup
}
```

A lot of frameworks check this too — set it correctly in deployment.

## `.env` files

For local development, hardcoding env vars in your shell or `package.json`
gets old. `dotenv` loads them from a file:

```bash
npm install dotenv
```

```
# .env
PORT=3000
DATABASE_URL=postgres://localhost/myapp_dev
SESSION_SECRET=dev-only-secret
```

```js
import 'dotenv/config';   // must be the first import
import express from 'express';
// process.env.PORT is now '3000'
```

<Callout variant="warning">
**Don't commit `.env` to git.** Add it to `.gitignore` from day one. Commit
a `.env.example` with the variable names but no values, so teammates know
what's needed.
</Callout>

Newer Node (≥ 20.6) has `--env-file` built in:

```bash
node --env-file=.env server.js
```

No `dotenv` package needed.

## What belongs in env vars

- **Anything secret**: API keys, DB passwords, JWT signing keys, encryption
  keys.
- **Anything that changes per environment**: ports, URLs, log levels,
  feature flags.
- **Anything that shouldn't be in version control**.

## What doesn't

- App constants (max upload size, route paths) — those are code.
- Anything you want type-checked. `process.env` is `string | undefined` for
  every key; if you need typed config, parse env vars into a typed
  object at startup:

```js
const config = {
  port: Number(process.env.PORT ?? 3000),
  dbUrl: required('DATABASE_URL'),
  isProd: process.env.NODE_ENV === 'production',
};

function required(name) {
  const v = process.env[name];
  if (!v) throw new Error(`${name} is required`);
  return v;
}
```

Crash on startup if a required env var is missing. Better than discovering
it three hours into a deploy when a request finally tries to use it.

## In production

You don't ship a `.env` file. Production env vars come from:

- The hosting platform's secret manager (Railway, Fly, Render variables;
  AWS Secrets Manager; Kubernetes secrets).
- The orchestrator (Docker `-e`, Kubernetes pod env).
- The CI/CD pipeline injecting them at deploy time.

The Node code doesn't change — it still reads `process.env`. Only the
*source* of the vars changes between environments.
