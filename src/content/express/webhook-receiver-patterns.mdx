# Webhook receiver patterns

Receiving a webhook looks easy ("accept POST, return 200"). Done
wrong, it leaks secrets, processes the same event twice, and
silently drops events during outages. Done right, it's
**verified, idempotent, and durable.**

## The four properties to get right

A production webhook receiver:

1. **Verifies the signature.** Reject forgeries.
2. **Reads the raw body**, not the parsed JSON. Signature is over
   bytes.
3. **Acknowledges fast (< 5 seconds)**, processes async. The sender
   gives up otherwise.
4. **Is idempotent.** The sender retries; you must not double-process.

Missing any one of these is a production-grade bug.

## Verifying the signature

Stripe, GitHub, Shopify, Slack — every webhook provider signs
payloads with HMAC. The receiver re-computes and compares.

### Stripe example

```js
import crypto from 'node:crypto';

app.post('/webhooks/stripe',
  express.raw({ type: '*/*' }),   // raw body, not JSON
  (req, res) => {
    const sig = req.headers['stripe-signature'];   // "t=...,v1=..."
    if (!sig) return res.sendStatus(400);

    // Parse "t=timestamp,v1=hex"
    const parts = Object.fromEntries(
      sig.split(',').map(p => p.split('='))
    );
    const timestamp = parts.t;
    const v1 = parts.v1;
    if (!timestamp || !v1) return res.sendStatus(400);

    // Reject old payloads (replay defense)
    if (Math.abs(Date.now() / 1000 - +timestamp) > 300) {
      return res.sendStatus(400);   // older than 5 minutes
    }

    // Re-compute HMAC over "<timestamp>.<rawBody>"
    const payload = `${timestamp}.${req.body.toString('utf8')}`;
    const expected = crypto
      .createHmac('sha256', process.env.STRIPE_WEBHOOK_SECRET)
      .update(payload)
      .digest('hex');

    // Constant-time compare
    const expectedBuf = Buffer.from(expected, 'hex');
    const actualBuf = Buffer.from(v1, 'hex');
    if (expectedBuf.length !== actualBuf.length) return res.sendStatus(400);
    if (!crypto.timingSafeEqual(expectedBuf, actualBuf)) return res.sendStatus(400);

    // Now parse and process
    const event = JSON.parse(req.body.toString('utf8'));
    queueForProcessing(event);
    res.sendStatus(200);
  }
);
```

Key details:

1. **`express.raw()` not `express.json()`**. JSON parsing might
   reformat (canonicalization, key order), breaking the signature
   comparison.
2. **Timestamp check.** Reject events older than ~5 minutes. Prevents
   replay if a signature is ever exposed.
3. **`crypto.timingSafeEqual`**, not `===`. Constant-time comparison
   defeats timing attacks.
4. **Length-check before `timingSafeEqual`** — it throws
   `RangeError` on mismatched lengths.
5. Parse the JSON **after** verification, not before. Don't trust
   the data until you've verified.

### GitHub example

```js
const sig = req.headers['x-hub-signature-256'];   // "sha256=hex"
if (!sig) return res.sendStatus(400);

const expected = 'sha256=' + crypto
  .createHmac('sha256', secret)
  .update(req.body)
  .digest('hex');

const sigBuf = Buffer.from(sig);
const expBuf = Buffer.from(expected);

if (sigBuf.length !== expBuf.length) return res.sendStatus(400);
if (!crypto.timingSafeEqual(sigBuf, expBuf)) return res.sendStatus(400);
```

Per-provider format varies; the pattern is the same. **Always
length-check before `timingSafeEqual`** (it throws on mismatch) and
**always check that `sig` exists** (an attacker sending no header
would otherwise crash the handler).

## Why raw body

```js
// BAD — JSON middleware consumes the stream
app.use(express.json());
app.post('/webhooks/stripe', ...);    // req.body is parsed; raw bytes lost
```

`express.json()` reads the request stream and produces a parsed
object. The original bytes are gone. Signature verification fails
because the HMAC was over the original bytes, and `JSON.stringify(req.body)`
doesn't reproduce them (key order, whitespace, etc.).

The fix: route-specific raw parsing **before** any JSON middleware.

```js
// Order matters
app.post('/webhooks/stripe', express.raw({ type: '*/*' }), stripeHandler);
app.use(express.json());   // for all other routes
```

Or use `express.json` with a `verify` callback that captures the raw
body:

```js
app.use(express.json({
  verify(req, res, buf) {
    req.rawBody = buf;
  },
}));
```

Both approaches work. The first is cleaner for known-webhook routes.

## Idempotency

The webhook sender will retry on:

- Network failure.
- Timeout (typically 5–10 seconds).
- Non-2xx response.

Your handler must process the same event ID at most once.

### Idempotency-key in Redis

```js
async function handleEvent(event) {
  const key = `webhook:processed:${event.id}`;
  const set = await redis.set(key, '1', 'EX', 86_400, 'NX');
  if (!set) {
    log.info({ id: event.id }, 'duplicate webhook, skipping');
    return;
  }
  await processEvent(event);
}
```

`SET ... NX EX 86400` atomically: set if not exists, with a 24-hour
TTL. If another worker already processed it, NX fails.

### Idempotency-key in DB

```js
async function handleEvent(event) {
  const inserted = await db.query(
    'INSERT INTO processed_webhooks (event_id, received_at) VALUES ($1, NOW()) ON CONFLICT DO NOTHING RETURNING event_id',
    [event.id]
  );
  if (inserted.rows.length === 0) return;   // already processed
  await processEvent(event);
}
```

`ON CONFLICT DO NOTHING` is the Postgres idiom. The unique index on
`event_id` enforces it; a duplicate insert fails silently.

For databases that don't have ON CONFLICT, use a transactional check.

### Idempotency in the processing logic

For some processors, idempotency is built into the operation:

```js
async function applyEvent(event) {
  // "Set user's email to X" — naturally idempotent
  await db.query('UPDATE users SET email = $1 WHERE id = $2', [event.email, event.userId]);
}
```

Where it's idempotent at the data level (UPDATE, UPSERT, SET status),
external dedup might not be needed. But for emit-side-effects
operations (send email, call third-party API), you must dedup.

## Respond fast, process async

```js
app.post('/webhooks/stripe', verifySig, async (req, res) => {
  // ~50ms: verify, queue, return
  await queue.add('webhook', { eventId: event.id, payload: event });
  res.sendStatus(200);
});
```

Acknowledge quickly. If you take 30 seconds to do the actual work,
the sender thinks you're down and retries — now you have two copies
of the same job.

Queue patterns:

- **BullMQ** + Redis: durable, retries, monitoring.
- **AWS SQS**: managed, durable.
- **Postgres-backed**: simple if you already use Postgres
  (`pg-boss`).

The queue worker processes the job idempotently. If it fails,
retry. If it succeeds, mark the job done.

### When you can't queue

For very simple handlers (no slow downstream calls, no external
side effects), processing inline is fine:

```js
app.post('/webhooks/simple', verifySig, async (req, res) => {
  await applyUpdate(event);   // < 1 second
  res.sendStatus(200);
});
```

The threshold is: can you complete the work plus a safety margin in
less than the sender's timeout? Usually 5 seconds. Anything more
than that, queue.

## The outbox pattern (when you also emit)

If processing the webhook means producing your own events:

```js
async function processWebhook(event) {
  await db.transaction(async (tx) => {
    await tx.query('UPDATE accounts SET balance = ... WHERE id = ...', ...);

    // Write to outbox in the same transaction
    await tx.query('INSERT INTO outbox (event_type, payload) VALUES (...)');
  });
}
```

A separate worker reads from the outbox and publishes to your event
bus. Either both happen (DB write committed, outbox entry persisted)
or neither (rollback). No "DB updated but downstream message lost."

This is the durable cross-system consistency pattern. See
**Background jobs** for the full treatment.

## Replay defense

```js
const timestamp = parts.t;
if (Math.abs(Date.now() / 1000 - +timestamp) > 300) {
  return res.sendStatus(400);
}
```

Reject events older than ~5 minutes. Prevents replay attacks where
an attacker captures a valid request and replays it later (e.g., to
trigger a withdrawal twice).

5 minutes is the typical window — enough for legitimate clock skew
+ network delay, short enough that replay opportunities are tiny.

## HTTPS only

Webhooks contain potentially sensitive data and a valid HMAC. Over
HTTP, they're trivially intercepted and replayed.

Force HTTPS:

```js
app.use((req, res, next) => {
  if (req.protocol !== 'https' && !isLocalhost(req)) {
    return res.sendStatus(403);
  }
  next();
});
```

Most providers refuse to send webhooks to HTTP endpoints anyway.

## Logging

Log every webhook with sender, event ID, signature status:

```js
log.info({
  webhook: 'stripe',
  eventId: event.id,
  eventType: event.type,
  signatureValid: true,
}, 'webhook received');
```

Don't log:

- Full payload (could contain PII, credentials).
- Signature secret.
- Raw body (could contain credentials).

For debugging, you can log the *event ID* + first few keys. For
deep inspection, replay from the provider's dashboard.

## Common pitfalls

### `express.json()` consuming the body

The classic. Order middleware correctly: raw body capture for
webhook routes **before** `express.json()`.

### Comparing signatures with `===`

```js
if (received === computed) ...    // timing attack possible
```

Use `crypto.timingSafeEqual(Buffer.from(received), Buffer.from(computed))`.
The constant-time compare prevents an attacker from learning the
signature byte-by-byte through timing.

### Skipping length check before `timingSafeEqual`

```js
crypto.timingSafeEqual(Buffer.from(short), Buffer.from(long));
// RangeError: Input buffers must have the same length
```

Length check first. Mismatched length is a verification failure
anyway.

### Forgetting `JSON.parse` after verification

```js
verifyAndProcess(req.body);   // body is a Buffer
```

After raw-body verification, `req.body` is a Buffer. Parse it:
`JSON.parse(req.body.toString('utf8'))`.

### Processing inline, slow

```js
app.post('/webhooks/x', async (req, res) => {
  await callThirdPartyApi(...);   // 8 seconds
  await sendEmail(...);             // 3 seconds
  res.sendStatus(200);              // 11 seconds later
});
```

Sender times out at 5 seconds. Retries. You now have two copies.

Always queue work that exceeds 1–2 seconds.

### Ignoring retries when fixing

After a fix, the queue has piled-up retries from the broken period.
Processing them might mass-trigger emails or notifications. Pause
the queue or implement event-time-based filtering ("ignore events
older than yesterday morning").

### Single secret across environments

```env
STRIPE_WEBHOOK_SECRET=whsec_...
```

If dev and prod share the secret, a dev webhook to your prod
endpoint validates. Use **per-environment secrets**. Stripe and
others let you configure separate endpoints with different secrets.

### `req.headers['stripe-signature']` typing

Headers are strings or arrays of strings in raw Node. Express
gives you `req.headers` typed as `string | undefined`. For TypeScript,
narrow before use.

## Provider-specific gotchas

### Stripe

Signature format: `t=timestamp,v1=hex,v0=hex_legacy`. You verify
`v1`. The signed payload is `${t}.${rawBody}`.

### GitHub

Two headers: `X-Hub-Signature` (SHA-1, deprecated) and
`X-Hub-Signature-256` (SHA-256). Use 256. Signed payload is the
raw body.

### Shopify

`X-Shopify-Hmac-SHA256` header, base64-encoded. Signed payload is
the raw body.

### Slack

`X-Slack-Signature` and `X-Slack-Request-Timestamp`. Signed payload
is `v0:${timestamp}:${rawBody}`.

Each provider has docs; the pattern is the same (HMAC over
`timestamp + body`).

## The mental model

A webhook receiver is **HTTP + HMAC + queue**:

1. **HTTP** — receive the request.
2. **HMAC** — verify the signature (raw body, length-checked,
   timing-safe compare, timestamp check).
3. **Queue** — push the verified event for async processing.
4. **Idempotency** — dedupe by event ID in Redis or DB before
   processing.
5. **Ack** — return 200 quickly so the sender doesn't retry.

Five steps; each one is a place where production-grade code differs
from the tutorial. Skip any one and you have an outage waiting.

For 2026 Node + Express, this is the standard playbook. Stripe,
GitHub, Shopify, Slack all expect you to follow it.
