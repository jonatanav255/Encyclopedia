# Unions and intersections

Two ways to build types out of other types. They're duals:

- **Union** (`A | B`) — "value is either A or B." Like `or`.
- **Intersection** (`A & B`) — "value is both A and B." Like `and`.

Together they cover most of the "combine these types" logic in a
codebase.

## Unions

```ts
type Status = 'idle' | 'loading' | 'success' | 'error';
type StringOrNumber = string | number;
type MaybeUser = User | null;
```

A value of type `A | B` is either an A or a B. To use it, you need to
**narrow** (see **Narrowing**) — figure out which side you're on.

```ts
function format(x: string | number): string {
  if (typeof x === 'string') return x.toUpperCase();
  return x.toFixed(2);
}
```

The methods you can call on a union without narrowing are the
*intersection of methods on each member*:

```ts
type S = string | string[];

function f(s: S) {
  s.length;           // ok — both have .length
  s.toUpperCase();    // error — only string has it
}
```

`length` works because both `string` and `string[]` have it.
`toUpperCase` is string-only — you need to narrow.

## Discriminated unions

The most useful union pattern. Each variant has a literal field that
identifies it:

```ts
type Result =
  | { kind: 'success'; data: User }
  | { kind: 'error'; message: string };

function handle(r: Result) {
  switch (r.kind) {
    case 'success': return r.data.name;
    case 'error':   return r.message;
  }
}
```

TypeScript uses the discriminant (`kind`) to narrow within each
branch. This pattern is the bread-and-butter of:

- Redux/state-machine actions.
- API response shapes.
- Result types (`Ok | Err`).
- Event types in pub-sub systems.

```ts
type Action =
  | { type: 'INCREMENT' }
  | { type: 'SET'; value: number }
  | { type: 'RESET' };

function reducer(state: number, action: Action): number {
  switch (action.type) {
    case 'INCREMENT': return state + 1;
    case 'SET': return action.value;
    case 'RESET': return 0;
  }
}
```

The discriminant can be any field name (`kind`, `type`, `tag`,
`_tag`, `status`). Be consistent within a project.

## Exhaustiveness

```ts
function handle(r: Result) {
  switch (r.kind) {
    case 'success': return r.data;
    // forgot 'error' — bug
  }
}
```

To force the compiler to catch missing cases:

```ts
function handle(r: Result): User | string {
  switch (r.kind) {
    case 'success': return r.data;
    case 'error':   return r.message;
    default: {
      const _exhaustive: never = r;
      return _exhaustive;
    }
  }
}
```

If you later add a variant to `Result` and forget to handle it,
`_exhaustive` is no longer `never` and the compiler errors. This is
the single most valuable pattern when working with state machines.

## Intersections

```ts
type Named = { name: string };
type Aged = { age: number };
type Person = Named & Aged;   // { name: string; age: number }

const p: Person = { name: 'jon', age: 30 };
```

A value of type `A & B` has all properties of both A and B. Used
for:

- **Composing object shapes**: `Base & WithTimestamps`.
- **Mixing in capabilities**: `Component & WithStyles`.
- **Adding props**: `BaseProps & { extra: number }`.

```ts
type WithTimestamps = { createdAt: Date; updatedAt: Date };
type User = Named & WithTimestamps;
```

## Intersection vs `extends`

```ts
interface User extends Named { age: number; }
type User = Named & { age: number };
```

For object shapes, these are almost equivalent. `extends` gives you
better error messages and prevents overlapping fields from creating
`never`s silently.

```ts
type Bad = { x: number } & { x: string };   // x: never
```

That's a quiet bug — the property `x` is `number & string`, which is
`never`. With `interface extends`, you'd get a compile error instead.

## Mixing unions and intersections

```ts
type Admin = User & { role: 'admin'; permissions: string[] };
type Guest = User & { role: 'guest' };
type Account = Admin | Guest;
```

Common pattern: base type intersected with role-specific extras,
then unioned to make a closed set.

## Union widening

A subtle gotcha:

```ts
type Mode = 'edit' | 'view';

function setMode(m: Mode) {}

const arr = ['edit', 'view'];   // string[]
setMode(arr[0]);                 // error — arr[0] is string, not Mode

const arr2 = ['edit', 'view'] as const;  // readonly ['edit', 'view']
setMode(arr2[0]);                          // ok — arr2[0] is 'edit'
```

Arrays of literals widen to general types unless you use `as const`.

## Distributive conditional types

Conditional types over unions automatically distribute:

```ts
type ToArray<T> = T extends any ? T[] : never;

type R = ToArray<string | number>;
// (string[]) | (number[])   — distributed
```

If you don't want distribution:

```ts
type ToArray<T> = [T] extends [any] ? T[] : never;
type R = ToArray<string | number>;
// (string | number)[]
```

This becomes important in advanced type-level code. See **Conditional
types**.

## Common pitfalls

### `any` swallows unions

```ts
const x: any = ...;
const u: User | null = x;     // accepted with no warning
```

`any` defeats union narrowing. Use `unknown` instead.

### Intersection with primitives

```ts
type T = string & number;   // never — no value is both
```

Intersecting incompatible types produces `never`. Sometimes useful
for "this branch shouldn't exist" patterns, often a sign of a type
error.

### Spreading vs intersecting

```ts
type A = { x: number };
type B = { x: string };

const obj: A & B = { x: ??? };   // error — x is never

const obj = { ...{ x: 1 }, ...{ x: 'hi' } };  // last wins at runtime
```

`A & B` requires both, which fails. Spread at runtime overwrites.
These are different operations — don't conflate "merge objects" with
"merge types."

## When to use what

- **Union** — "value can be one of these." Status codes, API
  results, polymorphic returns.
- **Discriminated union** — "value is one of these, and I can tell
  which." Reducers, parsers, ASTs.
- **Intersection** — "value must satisfy several contracts."
  Composing shapes, mixins, augmenting props.

In practice you'll write tons of unions and a moderate number of
intersections. Intersections beyond `Base & { extra }` are rare in
app code — they tend to live in libraries.

## The mental model

- `A | B` is "one of." Use it for choices.
- `A & B` is "all of." Use it for combining requirements.

The narrowing toolkit (`typeof`, `in`, discriminant fields) is what
makes unions usable. Without narrowing you can only call methods that
exist on every variant.

For closed sets — roles, modes, status codes — unions of string
literals (`'admin' | 'user' | 'guest'`) are the workhorse. Use them
instead of `string` to make illegal states unrepresentable.
