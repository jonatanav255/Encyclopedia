# What is TypeScript

TypeScript is JavaScript with a type system. You write `.ts` (or
`.tsx`) files; the TypeScript compiler (`tsc`) checks your types and
emits plain `.js` for the runtime to execute.

The runtime sees only JavaScript. Every type — `string`, `User`,
`Array<number>` — is **erased** during compilation. No metadata
survives to runtime.

## The deal

You add annotations:

```ts
function greet(name: string): string {
  return `hi, ${name}`;
}

greet('jon');        // ok
greet(42);            // compile error
```

In return, the compiler catches a large class of bugs *before* you
run the code:

- Passing the wrong argument type.
- Reading a missing property.
- Calling a method that doesn't exist.
- Forgetting to handle `null` or `undefined`.
- Mismatched return values.
- Refactoring that leaves callers broken.

It also powers editor features: autocompletion, jump-to-definition,
rename-symbol, inline error squiggles. Your IDE knows what type
everything is.

## What it isn't

- **A runtime checker.** TypeScript doesn't validate inputs at
  runtime. JSON from an API is `any` until you check it. For that,
  use Zod, Valibot, or hand-written guards.
- **A new language.** Everything that runs in JS runs in TS. TS is a
  superset.
- **Magic.** Types reflect what *you* tell the compiler. Bad types
  → bad checking.

## Setup

```bash
pnpm add -D typescript
pnpm exec tsc --init
```

That creates `tsconfig.json` — the config file. Common starting
settings:

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "noImplicitOverride": true,
    "skipLibCheck": true,
    "esModuleInterop": true,
    "isolatedModules": true,
    "jsx": "preserve",
    "outDir": "./dist"
  }
}
```

`strict: true` enables a bundle of flags (`noImplicitAny`,
`strictNullChecks`, etc.) — turn it on from day one.

## Type inference

You don't have to annotate everything. The compiler infers a lot:

```ts
let n = 5;             // inferred number
const name = 'jon';    // inferred 'jon' (literal type)
const arr = [1, 2];    // inferred number[]
function double(n: number) { return n * 2; }  // return type inferred as number
```

The discipline is to **annotate boundaries** (function parameters,
public exports, API contracts) and let the compiler infer the
internals.

```ts
// Annotate the surface
export function fetchUser(id: number): Promise<User> {
  // Let it infer the rest
  return fetch(`/api/users/${id}`).then(r => r.json());
}
```

## Strict null checks

The single most valuable thing TypeScript catches. With strict mode:

```ts
function name(user: { name?: string }): string {
  return user.name.toUpperCase();   // ERROR — name might be undefined
}
```

You have to handle the absent case:

```ts
function name(user: { name?: string }): string {
  return user.name?.toUpperCase() ?? 'anonymous';
}
```

This is why "switching a codebase to TypeScript catches dozens of
real bugs": JS treats `undefined.toUpperCase()` as a runtime error;
TS catches it at type-check time.

## The compile-time / runtime gap

```ts
const data: User = JSON.parse(json);
data.name;    // typed as string — but is it?
```

The compiler trusts your annotation. It has no way of knowing what's
actually in `json`. If the API sends `{ "name": null }`, the type
lies.

That's why every input from outside (HTTP, files, environment
variables, FFI) should be **validated**, not just typed:

```ts
import { z } from 'zod';

const UserSchema = z.object({
  id: z.number(),
  name: z.string(),
});

const data = UserSchema.parse(JSON.parse(json));
// data is now typed AND validated
```

For internal code (your own modules), trust the types. For data
crossing into your program, validate.

## What gets compiled away

```ts
interface User { id: number; name: string; }
type ID = number;
function f(x: number): void { console.log(x); }
const u: User = { id: 1, name: 'jon' };
```

Compiled output:

```js
function f(x) { console.log(x); }
const u = { id: 1, name: 'jon' };
```

`interface`, `type`, and parameter annotations vanish. Only the
JavaScript-level constructs remain.

This means:

- **Types can't be used at runtime.** No `instanceof MyInterface`,
  no `typeof someType`.
- **`import type` is free.** It's a hint to the compiler; nothing
  is imported at runtime.
- **`const enum`** can be inlined, removing the runtime cost.
- **Decorators** (when not erased) are actual runtime code.

## When to add types

- **Always** for shared functions, public exports, API responses.
- **Often** for complex internal data structures.
- **Sometimes** for simple internals — `let x = 5` is fine without
  `: number`.
- **Never** when it adds noise without value. Don't `: string` every
  `const greeting = 'hi'`.

The goal is "the types are obviously right." Over-annotating makes
code harder to read; under-annotating misses bugs.

## Common TS-specific things

- **`as` casting** — `value as User`. Tells the compiler to trust
  you. Use sparingly; it bypasses checks.
- **`!` non-null assertion** — `user!.name`. "I promise this isn't
  null." Same — use only when you're certain and the compiler can't
  see it.
- **`unknown`** — the type-safe `any`. Forces you to narrow before
  using.
- **`never`** — the bottom type, used for exhaustiveness checks.

## The mental model

TypeScript is **structural typing layered over JavaScript**, with
nullability tracked. The compiler is your reviewer for "did you
remember to handle null? Did you pass the right thing? Did you spell
the property correctly?"

Runtime behavior is unchanged. If you'd remember everything yourself,
TypeScript adds nothing. Nobody does — that's why everyone uses it.
