# Conditional types

Conditional types let you express "type X if Y, otherwise Z" at the
type level. Combined with `infer`, they let you extract or compute
types from other types — the foundation of most TypeScript magic.

## The syntax

```ts
type IsString<T> = T extends string ? true : false;

type A = IsString<'hi'>;     // true
type B = IsString<42>;       // false
```

Read as: "If T is assignable to string, the type is true; otherwise
false."

## Distributive conditionals

The big quirk: when T is a union, the conditional **distributes
over the union** automatically.

```ts
type ToArray<T> = T extends any ? T[] : never;

type R = ToArray<string | number>;
// (string[]) | (number[])  — distributed!
```

Each member of the union is evaluated separately, then the results
are unioned. This is usually what you want.

To prevent distribution, wrap in a tuple:

```ts
type ToArray<T> = [T] extends [any] ? T[] : never;

type R = ToArray<string | number>;
// (string | number)[]  — not distributed
```

The `[T] extends [...]` trick is the canonical way to opt out.

## `infer` — extracting types

`infer` lets you bind a name to part of the type being checked:

```ts
type ReturnType<T> = T extends (...args: any[]) => infer R ? R : never;

type A = ReturnType<() => string>;       // string
type B = ReturnType<() => Promise<User>>; // Promise<User>
```

The `infer R` says "let R be whatever the return type is." Inside
the conditional you can use R.

More examples:

```ts
type ArgsOf<T> = T extends (...args: infer A) => any ? A : never;
type FirstArg<T> = T extends (a: infer A, ...rest: any[]) => any ? A : never;
type ElementOf<T> = T extends (infer E)[] ? E : never;
type Awaited<T> = T extends Promise<infer U> ? Awaited<U> : T;   // recursive!
type Unboxed<T> = T extends Box<infer V> ? V : never;
```

Most of TypeScript's built-in `ReturnType`, `Parameters`, `Awaited`,
`Promise`-unwrapping utilities are implemented with `infer`.

## Common conditional patterns

### `NonNullable<T>`

```ts
type NonNullable<T> = T extends null | undefined ? never : T;

type R = NonNullable<string | null | undefined>;   // string
```

Strip nullables from a type. Built-in.

### `Exclude<T, U>` and `Extract<T, U>`

```ts
type Exclude<T, U> = T extends U ? never : T;
type Extract<T, U> = T extends U ? T : never;

type R = Exclude<'a' | 'b' | 'c', 'a'>;   // 'b' | 'c'
type R2 = Extract<'a' | 'b' | 'c', 'a' | 'b'>;  // 'a' | 'b'
```

Set difference and intersection at the union level. Both built-in.

### `Omit<T, K>` and `Pick<T, K>`

```ts
type Pick<T, K extends keyof T> = { [P in K]: T[P] };
type Omit<T, K extends keyof any> = Pick<T, Exclude<keyof T, K>>;
```

Pick is mapped; Omit is Pick + Exclude. Both built-in.

### Function inference

```ts
function delayed<T extends (...args: any[]) => any>(fn: T): T {
  return ((...args: Parameters<T>) => fn(...args)) as T;
}
```

`Parameters<T>` and `ReturnType<T>` are key when typing wrappers.

## Recursive conditional types

Conditionals can refer to themselves (with depth limits):

```ts
type Flatten<T> = T extends (infer E)[] ? Flatten<E> : T;

type A = Flatten<number[]>;          // number
type B = Flatten<number[][][]>;      // number
type C = Flatten<number>;            // number
```

Useful for unwrapping nested structures, summing tuple types, parsing
template strings.

TS limits recursion depth (~100 levels in modern versions) to prevent
infinite loops. For deep recursion, you sometimes need tail-recursive
forms.

## Distributing over an object

To "map" a conditional over an object's values, combine with mapped
types:

```ts
type StringValuesOnly<T> = {
  [K in keyof T as T[K] extends string ? K : never]: T[K];
};

type R = StringValuesOnly<{ id: number; name: string; admin: boolean }>;
// { name: string }
```

The `as` clause in the mapped type filters keys; mapping a key to
`never` removes it.

## `never` in unions

`never` is the empty union — it has no inhabitants. In unions, it
disappears:

```ts
type T = string | never;   // string
```

This is what makes the "map to `never` to filter" pattern work — the
filtered-out cases vanish.

## Common gotchas

### Forgetting distributivity

```ts
type IsArray<T> = T extends unknown[] ? true : false;

type R = IsArray<number[] | string>;
// boolean (true | false) — distributed!
```

If you wanted to ask "is T (as a whole) an array," wrap to prevent
distribution:

```ts
type IsArray<T> = [T] extends [unknown[]] ? true : false;
type R = IsArray<number[] | string>;
// false
```

### Inferring inside arrays

```ts
type Last<T extends unknown[]> = T extends [...any, infer L] ? L : never;

type A = Last<[1, 2, 3]>;   // 3
```

Variadic tuple inference is powerful. Combined with recursion, you
can write things like `Reverse<T>`, `Concat<T, U>`, etc.

### Conditional types that depend on a generic

```ts
function take<T>(arr: T[], n: number): T extends string ? string : T[] {
  // body
  return arr.slice(0, n) as any;
}
```

The compiler often can't validate the body satisfies the conditional
return — you need an `as` cast. Conditional return types are
expressive but require manual confirmation.

### Type instantiation depth

```ts
type Loop<N, A extends 0[] = []> =
  A['length'] extends N ? A : Loop<N, [...A, 0]>;
```

For large N this blows the recursion limit. The compiler signals
"Type instantiation is excessively deep." Refactor or accept the
limit.

## Performance

Conditional types are evaluated lazily, but heavy use slows
typechecking. Symptoms: editor lag, `tsc` running for minutes.

Mitigations:

- Cache complex types with intermediate aliases.
- Use simpler shapes (avoid massive unions).
- Don't recurse arbitrarily.
- For libraries, profile and document the cost.

The TypeScript compiler has flags like `--extendedDiagnostics` that
show timing per type — useful when hunting expensive types.

## The mental model

A conditional type is an `if/else` at the type level. The body is
restricted (no statements; just type expressions). With `infer`, you
get to bind names to parts of the input — like pattern matching.

The distribution rule (unions get split) is the most surprising
behavior. The `[T] extends [...]` wrap is the universal escape.

Most of the time you'll consume conditional types via the built-in
utilities (`NonNullable`, `Omit`, `ReturnType`, etc.). You'll write
your own when you need:

- A library-quality utility type.
- A typed wrapper that preserves complex relationships.
- Type-level computation over template strings or tuples.

Save them for the cases where simpler patterns don't fit. Most app
code rarely needs them.
