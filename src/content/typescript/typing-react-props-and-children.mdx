# Typing React props and children

The TypeScript-and-React interface has more "wait, which type do I
use?" moments than any other corner of the language. Three things are
usually getting tangled: how to type `children`, how to extend a
built-in element's props, and how to type events.

## children: ReactNode is almost always right

```tsx
type CardProps = {
  children: React.ReactNode;
};
```

`React.ReactNode` accepts anything you can render: strings, numbers,
elements, arrays, fragments, portals, `null`, `undefined`,
`boolean`. That's almost always what `children` should be.

The narrower siblings exist but are rarely correct:

<Cheatsheet
  columns={['Type', 'What it accepts', 'When to use']}
  rows={[
    ['ReactNode', 'Anything renderable', 'children — default choice'],
    ['ReactElement', 'A single JSX element', 'When you specifically need to inspect / clone children'],
    ['JSX.Element', 'Same as ReactElement but JSX-flavored', 'Rarely — slightly stricter than ReactElement'],
    ['ReactNode[]', 'Array of nodes', 'Never useful — ReactNode already includes arrays'],
    ['string', 'Just strings', 'When the API really only accepts text'],
  ]}
/>

The trap most people fall into:

```tsx
// Looks reasonable
type Props = { children: React.JSX.Element };

<Component>hello</Component>           // ✗ string is not JSX.Element
<Component>{a} {b}</Component>         // ✗ multiple children → fragment, not JSX.Element
<Component>{null}</Component>          // ✗ null is not JSX.Element
```

If you need to clone or inspect a single child, type it as
`React.ReactElement`. For anything else, `ReactNode`.

### Optional children

```tsx
// Optional — caller may omit children entirely
type Props = { children?: React.ReactNode };

// Required — caller must pass something (even null counts)
type Props = { children: React.ReactNode };
```

`React.PropsWithChildren<T>` is shorthand for "T plus optional
children":

```tsx
type Props = React.PropsWithChildren<{ title: string }>;
// equivalent to: { title: string; children?: React.ReactNode }
```

Convenient but optional; spell it out if you find the shorthand
opaque.

### Render-prop children

```tsx
type Props = {
  children: (state: { open: boolean }) => React.ReactNode;
};
```

A function-as-children. The type tells consumers exactly what to
return.

## Extending an HTML element's props

The common case: build a wrapper that should accept everything a
`<button>` accepts, plus a few of your own:

```tsx
type ButtonProps = {
  variant?: 'primary' | 'secondary';
} & React.ComponentPropsWithoutRef<'button'>;

function Button({ variant = 'primary', className, ...rest }: ButtonProps) {
  return <button className={`btn btn-${variant} ${className ?? ''}`} {...rest} />;
}
```

`ComponentPropsWithoutRef<'button'>` gets you every HTML attribute,
every event handler, every aria/data attribute the browser supports —
correctly typed, with autocompletion.

The two flavors:

- `ComponentPropsWithoutRef<T>` — props minus `ref`. Use when you
  handle refs separately (or don't).
- `ComponentPropsWithRef<T>` — props including `ref`. Use when you're
  building a polymorphic component (see **Polymorphic components**).

For non-element components, point at the component value:

```tsx
type LinkProps = React.ComponentProps<typeof Link>;
```

This pulls in whatever props `Link` declared, useful when wrapping a
library component.

### Why not `React.HTMLAttributes<HTMLButtonElement>`?

It exists, but `ComponentPropsWithoutRef<'button'>` is more precise.
The `HTMLAttributes` family doesn't include element-specific props
(like `type` on `<button>`, `href` on `<a>`). The `ComponentProps`
family does.

## Typing event handlers

The pattern is `EventName<Element>`:

```tsx
type Props = {
  onChange: (e: React.ChangeEvent<HTMLInputElement>) => void;
  onClick: (e: React.MouseEvent<HTMLButtonElement>) => void;
  onSubmit: (e: React.FormEvent<HTMLFormElement>) => void;
  onKeyDown: (e: React.KeyboardEvent<HTMLInputElement>) => void;
  onFocus: (e: React.FocusEvent<HTMLInputElement>) => void;
};
```

The element type controls what `e.currentTarget` is. `e.target` is
typed less specifically because the event can bubble from a child —
React doesn't know what that child is.

```tsx
function onChange(e: React.ChangeEvent<HTMLInputElement>) {
  e.currentTarget.value;    // string — guaranteed
  e.target.value;            // string in TS, but technically could be other element types at runtime
}
```

Use `currentTarget` when you need the element the handler is attached
to. Use `target` when you actually want the source of the event
(rare — usually for delegation).

### Inline handlers don't need types

```tsx
<input onChange={(e) => setValue(e.target.value)} />
```

TypeScript infers `e: ChangeEvent<HTMLInputElement>` from the
`onChange` slot. Only annotate when you're extracting the handler:

```tsx
function handleChange(e: React.ChangeEvent<HTMLInputElement>) {
  setValue(e.target.value);
}

<input onChange={handleChange} />
```

## React.FC: skip it

`React.FC<Props>` is the old way:

```tsx
const Greet: React.FC<{ name: string }> = ({ name }) => <p>{name}</p>;
```

Most React-with-TypeScript style guides now recommend against it.
Reasons:

- It used to imply optional `children` automatically (until
  `@types/react` 18 removed this), causing silent breakage when
  upgrading.
- It awkwardly types defaults and generics.
- The plain function signature is shorter and inference is better:

```tsx
function Greet({ name }: { name: string }) {
  return <p>{name}</p>;
}
```

Not strictly wrong. Just not needed.

## Generic components

```tsx
type SelectProps<T> = {
  options: T[];
  value: T;
  onChange: (value: T) => void;
  renderOption: (option: T) => React.ReactNode;
};

function Select<T>({ options, value, onChange, renderOption }: SelectProps<T>) {
  return (
    <ul>
      {options.map((opt, i) => (
        <li key={i} onClick={() => onChange(opt)} aria-selected={opt === value}>
          {renderOption(opt)}
        </li>
      ))}
    </ul>
  );
}

<Select<User>
  options={users}
  value={selected}
  onChange={setSelected}
  renderOption={(u) => u.name}
/>
```

The JSX-call generic syntax (`<Select<User>>`) only works in `.tsx`
files. Inference usually fills `T` in from `options` and you don't
need to spell it out.

The constraint to be aware of: the props for a generic component are
inferred from the **first** usage point that has full type info. If
the first use is `value={undefined}`, T may infer too widely. Keep
the props that pin T (`options`, `value`) un-optional, or pass T
explicitly.

## Discriminated-union props

When some props only make sense together, model them as a union:

```tsx
type ButtonProps =
  | { variant: 'icon'; icon: React.ReactNode; label: string }
  | { variant: 'text'; children: string };

function Button(props: ButtonProps) {
  if (props.variant === 'icon') {
    return <button aria-label={props.label}>{props.icon}</button>;
  }
  return <button>{props.children}</button>;
}

<Button variant="icon" icon={<Star />} label="Favorite" />
<Button variant="text">Save</Button>
<Button variant="icon">Save</Button>                  // ✗ icon variant needs `icon` and `label`
```

Better than `icon?: ReactNode; label?: string` — the compiler enforces
"if variant is icon, you must pass icon and label."

## Common pitfalls

<Compare
  bad={`// Loses everything from <input> — no type, no value, no aria-*

}`}
  good={`// Inherits every <input> prop
type Props = React.ComponentPropsWithoutRef<'input'>;

function Input(props: Props) {
  return <input {...props} />;
}`}

/>

  bad={`// children typed as a single element — breaks for text or arrays

}`}

## The discipline

- `React.ReactNode` for `children`, almost always. Reach for
  `ReactElement` only when you must clone or inspect.
- Use `ComponentPropsWithoutRef<T>` (or `ComponentProps<typeof X>`)
  to extend an element/component's props — don't redeclare them.
- Skip `React.FC`. A plain function declaration with a typed param is
  shorter and inferred better.
- Don't annotate inline handlers — let inference do its job. Annotate
  extracted handlers explicitly.
- When some props go together, model them as a discriminated union.
  The compiler will enforce the relationship at use sites.
