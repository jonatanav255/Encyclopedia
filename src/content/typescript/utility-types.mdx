# Utility types

The TypeScript standard library ships with a set of pre-defined
generic types that cover the most common transformations. Knowing
them saves you reinventing wheels.

## `Partial<T>` â€” make every field optional

```ts
type User = { id: number; name: string };
type UserPatch = Partial<User>;
// { id?: number; name?: string }

function update(id: number, patch: Partial<User>) {
  // patch can be { name: 'jen' } or {} or anything
}
```

The classic use case: PATCH endpoint payloads, form drafts, partial
config overrides.

`Partial` is shallow. For nested partials, write `DeepPartial`
yourself or use a library.

## `Required<T>` â€” make every field required

```ts
type Config = { port?: number; host?: string };
type StrictConfig = Required<Config>;
// { port: number; host: string }
```

Useful after defaults are applied:

```ts
function applyDefaults(c: Config): Required<Config> {
  return { port: c.port ?? 3000, host: c.host ?? 'localhost' };
}
```

## `Readonly<T>` â€” lock all fields

```ts
type Locked = Readonly<{ x: number; y: number }>;
// { readonly x: number; readonly y: number }
```

Shallow. For deep, write `DeepReadonly` (mapped + recursive).

## `Pick<T, K>` â€” select specific keys

```ts
type User = { id: number; name: string; email: string; password: string };
type PublicUser = Pick<User, 'id' | 'name'>;
// { id: number; name: string }
```

Use when you want only some fields of a larger type. Better than
copy-pasting the fields.

## `Omit<T, K>` â€” drop specific keys

```ts
type User = { id: number; name: string; password: string };
type SafeUser = Omit<User, 'password'>;
// { id: number; name: string }
```

The inverse of `Pick`. Often used in response shapes ("everything
except sensitive fields").

`Omit` doesn't validate that the keys exist in T (unlike `Pick`,
which does). `Omit<User, 'banana'>` is just `User`. This is by
design â€” useful for "omit if present" patterns.

## `Record<K, V>` â€” typed dictionary

```ts
type StatusMap = Record<'idle' | 'loading' | 'done', string>;
// { idle: string; loading: string; done: string }

type StringDict = Record<string, number>;
// { [key: string]: number }
```

Use when you have a fixed set of keys and a uniform value type, or
when you want a typed string-keyed map.

## `Exclude<T, U>` and `Extract<T, U>` â€” set ops on unions

```ts
type Roles = 'admin' | 'user' | 'guest';
type NonAdmin = Exclude<Roles, 'admin'>;     // 'user' | 'guest'
type Privileged = Extract<Roles, 'admin'>;   // 'admin'
```

`Exclude` removes; `Extract` keeps. Both distribute over unions.

## `NonNullable<T>` â€” strip null/undefined

```ts
type Maybe = string | null | undefined;
type Definite = NonNullable<Maybe>;   // string
```

Very useful inside generic constraints:

```ts
function require<T>(value: T): NonNullable<T> {
  if (value == null) throw new Error();
  return value;
}
```

## `ReturnType<T>` â€” function's return

```ts
function getUser() { return { id: 1, name: 'jon' }; }

type User = ReturnType<typeof getUser>;
// { id: number; name: string }
```

Pair with `typeof` to derive the type without writing it twice.

## `Parameters<T>` â€” function's args as tuple

```ts
function fetchUser(id: number, opts: { cache: boolean }) {}

type Args = Parameters<typeof fetchUser>;
// [id: number, opts: { cache: boolean }]
```

Useful when wrapping functions:

```ts
function withLogging<T extends (...args: any[]) => any>(fn: T) {
  return (...args: Parameters<T>): ReturnType<T> => {
    console.log('called with', args);
    return fn(...args);
  };
}
```

## `ConstructorParameters<T>` â€” class constructor args

```ts
class User { constructor(id: number, name: string) {} }
type Args = ConstructorParameters<typeof User>;   // [number, string]
```

For factory functions and dependency injection.

## `InstanceType<T>` â€” class instance from constructor

```ts
class User { id = 1; }
type U = InstanceType<typeof User>;   // User
```

Goes from `typeof Class` (constructor) back to instance type. Used
for typing factory APIs.

## `Awaited<T>` â€” unwrap promises (deeply)

```ts
type A = Awaited<Promise<string>>;          // string
type B = Awaited<Promise<Promise<number>>>; // number
type C = Awaited<string>;                    // string
```

Recursive â€” unwraps nested promises until it hits a non-promise. The
return of an `async` function is its inner type wrapped in
`Promise<T>`; `Awaited<ReturnType<typeof fn>>` gets the inner.

## `Uppercase<S>`, `Lowercase<S>`, `Capitalize<S>`, `Uncapitalize<S>`

```ts
type A = Uppercase<'hello'>;     // 'HELLO'
type B = Lowercase<'HELLO'>;     // 'hello'
type C = Capitalize<'hello'>;    // 'Hello'
type D = Uncapitalize<'Hello'>;  // 'hello'
```

For template literal types â€” see that page.

## `NoInfer<T>` (TS 5.4+) â€” block inference on one parameter

By default, a generic parameter is inferred from every position
it appears in. Sometimes you want it inferred from one position
and treated as a fixed constraint at another:

```ts
function pick<K extends string>(options: K[], chosen: K): K {
  return chosen;
}

pick(['a', 'b', 'c'], 'd');   // đźź˘ â€” TS widens K to 'a' | 'b' | 'c' | 'd'
```

Both arguments feed inference, so `K` widens to include `'d'` and
the typo passes. `NoInfer` blocks inference at the `chosen`
position:

```ts
function pick<K extends string>(options: K[], chosen: NoInfer<K>): K {
  return chosen;
}

pick(['a', 'b', 'c'], 'd');   // đźš« 'd' is not assignable to 'a' | 'b' | 'c'
```

Now `K` is inferred only from `options`; `chosen` is checked
against the inferred `K` as a constraint. Lets you make one
parameter the "source of truth" and others conform to it.

Common uses: default value parameters that should match an inferred
union, callbacks whose signature should match an earlier argument,
config builders.

## `Prettify<T>` â€” flatten types in hover tooltips

Not built-in, but the most-copied snippet in TS-land:

```ts
type Prettify<T> = { [K in keyof T]: T[K] } & {};
```

The mapped type walks the keys; the `& {}` is a compiler quirk
(undocumented but stable) that reliably forces shape resolution in
hovers. Wrapping a type in `Prettify` makes IDE hovers show the
resolved shape instead of the alias chain:

```ts
type A = { a: 1 };
type B = { b: 2 };
type Combined = A & B;            // hover: A & B
type CombinedPretty = Prettify<A & B>;
// hover: { a: 1; b: 2 }
```

Zero runtime cost. Use on public-facing utility types where the
"display" of the type matters for DX. Don't use on every type â€”
the extra computation slows the checker on huge intersections.

## Common compositions

### Public user â€” strip sensitive fields and make readonly

```ts
type PublicUser = Readonly<Omit<User, 'password' | 'salt'>>;
```

### Partial update with required id

```ts
type Update<T> = Partial<T> & Pick<T, 'id'>;

function update(payload: Update<User>) {
  // payload.id is required, everything else optional
}
```

### Form data â€” all strings

```ts
type FormData<T> = { [K in keyof T]: string };

type UserForm = FormData<User>;
// { id: string; name: string; ... }
```

### Action types from a handler map

```ts
const handlers = {
  ADD: (state: number, action: { value: number }) => state + action.value,
  RESET: (state: number) => 0,
};

type Action = {
  [K in keyof typeof handlers]: Parameters<typeof handlers[K]>[1] extends undefined
    ? { type: K }
    : { type: K } & NonNullable<Parameters<typeof handlers[K]>[1]>;
}[keyof typeof handlers];
```

That's a mouthful, but the result is exactly the actions your reducer
accepts â€” derived from the handler map. Type-safe, no duplication.

## Built-in vs custom utility libraries

Built-in: `Partial`, `Required`, `Readonly`, `Pick`, `Omit`, `Record`,
`Exclude`, `Extract`, `NonNullable`, `Parameters`, `ReturnType`,
`ConstructorParameters`, `InstanceType`, `Awaited`, `Uppercase`,
`Lowercase`, `Capitalize`, `Uncapitalize`, `ThisParameterType`,
`OmitThisParameter`, `ThisType`.

Libraries like `type-fest`, `ts-toolbelt`, `utility-types` add
hundreds more: `DeepPartial`, `DeepReadonly`, `Mutable`, `XOR`,
`SetOptional`, `RequireAtLeastOne`, etc. For most projects,
`type-fest` is a good lightweight addition.

## When to write your own

When a project repeatedly applies the same transformation, extract a
named utility:

```ts
// good
type CreateInput<T> = Omit<T, 'id' | 'createdAt' | 'updatedAt'>;

function createUser(input: CreateInput<User>) { ... }
function createPost(input: CreateInput<Post>) { ... }
```

The name "CreateInput" tells the next reader what's happening. The
explicit `Omit<...>` would obscure intent.

## Pitfalls

### `Omit` doesn't enforce keys

```ts
type X = Omit<User, 'nonexistent'>;   // just User â€” no error
```

`Pick` does enforce; `Omit` doesn't. Mostly fine; occasionally
masks typos.

### `Partial` is shallow

```ts
type Config = { db: { host: string } };
type ConfigPatch = Partial<Config>;
// { db?: { host: string } } â€” db is optional, but db.host is still required
```

For deep partial, you need a recursive type or a library.

### Built-ins on functions

```ts
function f(a: number, b: string) {}
type R = Readonly<typeof f>;
// readonly versions of `length`, `prototype`, etc. â€” not what you wanted
```

Most utilities apply to objects, not functions. For function
manipulation use `Parameters`, `ReturnType`, etc.

## The discipline

- Reach for built-in utilities first. The built-ins cover almost
  everything routine.
- Compose them: `Readonly<Omit<...>>`, `Partial<Pick<...>>`.
- Extract named utilities (`CreateInput<T>`) when a pattern repeats.
- Use `type-fest` for advanced needs rather than rolling your own.
- Don't over-engineer types â€” most app code does fine with simple
  shapes plus a sprinkle of utilities.
