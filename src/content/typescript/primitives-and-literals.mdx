# Primitives and literals

TypeScript's primitive types match JavaScript's: `string`, `number`,
`boolean`, `bigint`, `symbol`, `null`, `undefined`. It also has
**literal types** that narrow primitives down to specific values, and
the top/bottom types `any`, `unknown`, `never`, and `void`.

## The primitive types

```ts
let s: string = 'hi';
let n: number = 42;
let b: boolean = true;
let big: bigint = 10n;
let sym: symbol = Symbol();
let u: undefined = undefined;
let nu: null = null;
```

Avoid the uppercase `String`, `Number`, `Boolean` — those refer to the
object wrapper types and are nearly always wrong:

```ts
let bad: String = 'hi';    // boxed object type — don't use
let good: string = 'hi';   // primitive — yes
```

## Literal types

A type can be a specific value, not just a category:

```ts
type Role = 'admin' | 'user' | 'guest';
const role: Role = 'admin';
const oops: Role = 'banana';  // error
```

`'admin'` is both a value and a type. The type `'admin'` has exactly
one inhabitant: the string `'admin'`.

Literal numbers, booleans, and bigints work the same:

```ts
type Bit = 0 | 1;
type AlwaysTrue = true;
```

Unions of literals are how you express "one of these specific values"
— roles, status codes, theme names. Way safer than `string`.

## When does TS infer a literal vs a wide type?

```ts
let x = 'hi';        // inferred: string
const y = 'hi';      // inferred: 'hi'

function f(s: string) {}
function g(s: 'hi') {}

f(y);   // ok
g(y);   // ok
g(x);   // error — x might be any string, not just 'hi'
```

`let` widens (because you can reassign), `const` keeps the literal
(immutable binding).

For objects:

```ts
const o = { mode: 'edit' };
// inferred: { mode: string }  — not { mode: 'edit' }

const o2 = { mode: 'edit' as const };
// inferred: { mode: 'edit' }

const o3 = { mode: 'edit' } as const;
// inferred: { readonly mode: 'edit' }
```

`as const` is the escape hatch — "treat this as deeply literal."
Useful for config objects, action types, anything you want the
compiler to track precisely.

## `any` — the escape hatch

`any` opts out of type checking. Anything goes:

```ts
let x: any = 5;
x = 'hi';
x = [1, 2, 3];
x.foo.bar.baz;       // no error, but will crash at runtime
```

`any` is contagious — anything assigned from `any` becomes `any`
unless you cast.

Use `any` only when:

- You're prototyping and will tighten later.
- You're interoperating with truly untyped JS.
- A type is genuinely too complex to express and you accept the cost.

Don't use `any` to silence errors you don't understand. Reach for
`unknown` instead — it gives you the "I don't know yet" semantic
without disabling checks.

## `unknown` — the safe `any`

```ts
let x: unknown = JSON.parse(input);
x.toUpperCase();   // error — must narrow first

if (typeof x === 'string') {
  x.toUpperCase(); // ok — narrowed to string
}
```

`unknown` is the right type for "we don't know what this is yet —
narrow before using." The compiler forces you to check before doing
anything.

For `JSON.parse`, `fetch().json()`, `process.argv` parsing, any
boundary where data comes in unknown — use `unknown` and validate.

## `void` and `undefined`

```ts
function log(msg: string): void {
  console.log(msg);
}
```

`void` is for "this function doesn't return anything you should care
about." Functionally similar to `undefined`, but with one twist:

```ts
type Cb = () => void;
const cb: Cb = () => 42;   // OK — `void` means "ignore the return"
```

A function typed `() => void` can still return values; they're
ignored. This matters for callbacks (e.g., `forEach`) — you can pass
a function that returns something, and TS doesn't complain.

Don't conflate `void` and `undefined`:

```ts
function f(): undefined { return undefined; } // must explicitly return
function g(): void { /* anything */ }
```

For "this returns nothing," use `void`. For "this returns the value
`undefined`," use `undefined`.

## `never`

The bottom type. Inhabited by nothing. Functions that never return
(throw or loop forever):

```ts
function fail(msg: string): never {
  throw new Error(msg);
}

function loop(): never {
  while (true) {}
}
```

`never` is mostly useful for exhaustiveness:

```ts
type Shape = { kind: 'circle' } | { kind: 'square' };

function area(s: Shape) {
  switch (s.kind) {
    case 'circle': return Math.PI;
    case 'square': return 1;
    default: {
      const _exhaustive: never = s;   // catches "I added a case to Shape but not here"
      return _exhaustive;
    }
  }
}
```

If you add `{ kind: 'triangle' }` to `Shape`, `_exhaustive` stops
being `never` and the compiler errors.

## Type vs value

TypeScript has two namespaces: types and values.

```ts
class User { name = ''; }     // creates both a value (constructor) and a type (instances)
type User2 = { name: string }; // type only
const u: User = new User();    // User as type, User as value
```

Most of the time you don't notice this. When you do — usually with
classes, enums, or `typeof` — it's worth knowing.

## `typeof` (the type operator)

In TypeScript, `typeof x` (when used in a type position) refers to
the type of the value `x`:

```ts
const config = { port: 3000, host: 'localhost' };

type Config = typeof config;
// { port: number; host: string }
```

Useful for deriving types from values you already have — config
objects, API client instances, etc. (Not to be confused with the
runtime `typeof` operator.)

## The hierarchy

```
any        — top, opts out of checking
unknown    — top, safe (must narrow)
   ...
'admin' | 'user' — literal union
   ...
string | number — primitives
   ...
never      — bottom
```

`unknown` is the type-safe top. `never` is the bottom. Everything
else lives in between.

`any` is a "trapdoor" — anything goes both in and out. Avoid it
unless you have a specific reason.

## The discipline

- Use literal types (`'admin' | 'user'`) for closed sets, not
  `string`.
- Use `unknown` instead of `any` for "I don't know yet."
- Use `as const` for deeply literal config objects.
- Use `never` for exhaustiveness checks.
- Avoid the wrapper types (`String`, `Number`, `Boolean`).
- Avoid `any` — it disables the entire point of TypeScript.
