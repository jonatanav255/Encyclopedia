# `tsconfig.json`

The configuration file that tells the TypeScript compiler what to
compile and how. Get this right early — bad defaults cost you weeks
of bugs that strict mode would have caught.

## Minimal modern config

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "lib": ["ES2022", "DOM"],
    "jsx": "preserve",

    "strict": true,
    "noUncheckedIndexedAccess": true,
    "noImplicitOverride": true,
    "noFallthroughCasesInSwitch": true,
    "exactOptionalPropertyTypes": true,
    "noImplicitReturns": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true,

    "esModuleInterop": true,
    "isolatedModules": true,
    "verbatimModuleSyntax": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,

    "resolveJsonModule": true,
    "allowJs": false,

    "outDir": "./dist",
    "rootDir": "./src",
    "declaration": true,
    "declarationMap": true,
    "sourceMap": true
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}
```

That's a serious starting point. Let's walk through the categories.

## The strictness suite

`strict: true` is a meta-flag that enables a bundle:

- `noImplicitAny` — error on parameters/variables typed as `any`
  implicitly.
- `strictNullChecks` — null and undefined are separate types.
- `strictFunctionTypes` — function parameter compatibility checked
  strictly (contravariance).
- `strictBindCallApply` — `bind`/`call`/`apply` check argument
  types.
- `strictPropertyInitialization` — class fields must be assigned in
  the constructor (or marked `?`/`!`).
- `noImplicitThis` — error on `this` of implicit `any` type.
- `alwaysStrict` — emit `"use strict"`.
- `useUnknownInCatchVariables` — `catch (err)` types `err` as
  `unknown` (must narrow before use).

Turn `strict` on from day one. Migrating a non-strict codebase is
painful; starting strict is free.

## Extra strict flags worth enabling

### `noUncheckedIndexedAccess`

```ts
const arr = [1, 2, 3];
const x = arr[10];   // x: number | undefined (with the flag)
```

Catches assumptions about array indices being present. One of the
most valuable flags TS doesn't include in `strict`.

### `exactOptionalPropertyTypes`

```ts
type User = { name?: string };

const u: User = { name: undefined };   // error — must be "missing" or string
const u: User = {};                     // ok
```

Distinguishes "key absent" from "key present but undefined." If you
care about that distinction (often in JSON serialization), turn it
on.

### `noImplicitOverride`

```ts
class Base { foo() {} }
class Sub extends Base {
  foo() {}        // error without `override`
  override foo() {} // ok
}
```

Forces explicit `override` when subclasses override. Catches typos
("I added `bar`, but I meant `foo`") and refactor breaks.

### `noUnusedLocals`, `noUnusedParameters`

Errors for unused variables and parameters. Loud about dead code.
Some teams disable these because they're noisy during refactoring;
others love them.

For parameters, prefix with `_` to opt out:

```ts
function f(_unused: string, used: number) { return used; }
```

### `noFallthroughCasesInSwitch`

```ts
switch (x) {
  case 1: doA();
  case 2: doB();   // error: case 1 falls through
}
```

`break` or `return` to silence. The pattern only works for
intentional fallthrough (rare), which this flag forces you to comment
on.

## Module flags

### `module` and `target`

- `target` — the JS version output. `ES2022` is modern, well-supported
  in Node 18+ and all evergreen browsers.
- `module` — the module system to emit. `ESNext` keeps ESM as-is;
  `NodeNext` produces ESM with Node-specific conventions.

Choose based on output: bundler input → `ESNext`. Raw Node ESM →
`NodeNext`. Library shipping CJS → `CommonJS`.

### `moduleResolution`

- `Bundler` — TS 5.0+, the right pick for Vite, Webpack, esbuild.
- `NodeNext` — strict Node ESM rules (extensions required,
  `package.json` exports honored).
- `Node` — legacy CJS resolution. Avoid for new code.
- `Classic` — really old, don't use.

### `esModuleInterop`, `allowSyntheticDefaultImports`

```ts
import express from 'express';   // works only with esModuleInterop
```

Allows default imports from CJS modules. Turn on. The second flag
is implied by the first.

### `isolatedModules`, `verbatimModuleSyntax`

For fast non-`tsc` build pipelines (Vite, esbuild, swc). Forces
clearer import/export syntax — see **Modules and imports**.

### `resolveJsonModule`

```ts
import config from './config.json';
```

Allows importing JSON files as values. Useful for data, config,
static dictionaries.

## Output flags

### `outDir`, `rootDir`

- `rootDir` — the source root (e.g., `./src`).
- `outDir` — where to put compiled output (`./dist`).

Together they preserve the directory structure: `src/foo/bar.ts`
becomes `dist/foo/bar.js`.

### `declaration`, `declarationMap`

```json
{ "declaration": true, "declarationMap": true }
```

Emits `.d.ts` files alongside `.js`. Required for libraries. The
`.map` files let IDEs jump from a consumer to your TS source.

### `sourceMap`, `inlineSourceMap`, `inlineSources`

Source maps let debuggers show TS source in stack traces. Always
enable for development. `inlineSourceMap` embeds the map in the JS
(simpler deploys, larger files).

### `incremental`, `tsBuildInfoFile`

```json
{ "incremental": true, "tsBuildInfoFile": "./.tsbuildinfo" }
```

Caches type information between runs for fast rebuilds. Almost free
to enable. The `.tsbuildinfo` file should be in `.gitignore`.

## `lib` and `types`

```json
{ "lib": ["ES2022", "DOM", "DOM.Iterable"] }
```

Which ambient APIs are available. `ES2022` brings ES2022 globals
(`Object.hasOwn`, etc.). `DOM` brings browser globals (window,
document). Omit `DOM` for Node-only code.

```json
{ "types": ["node", "vitest/globals"] }
```

Whitelist of `@types/*` packages to include. By default, all
installed `@types` are included. Set explicitly to control the
ambient set — important for libraries.

## Source includes

```json
{
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist", "**/*.test.ts"]
}
```

- `include` — file patterns to compile.
- `exclude` — patterns to skip (applied to `include`).
- `files` — explicit list (rare; use `include` instead).

For monorepos and large projects, see **Project references**.

## `paths`

```json
{
  "baseUrl": ".",
  "paths": {
    "@/*": ["src/*"]
  }
}
```

Path aliases — see **Modules and imports**. Need matching bundler
config.

## JSX

```json
{ "jsx": "preserve" }
```

- `preserve` — leave JSX as-is for a downstream tool to transform.
- `react` — emit `React.createElement` calls (legacy).
- `react-jsx` — emit `_jsx`/`_jsxs` from the new JSX transform
  (React 17+).
- `react-jsxdev` — dev-mode JSX transform.

For React 17+ apps with a bundler that handles JSX (Vite), use
`preserve`. For libraries, `react-jsx`.

## `skipLibCheck`

```json
{ "skipLibCheck": true }
```

Skip type checking of declaration files (`*.d.ts`). Hugely improves
speed; some libraries have invalid types that don't affect you, and
this lets you proceed.

Almost always enabled in practice. Turn off only if you maintain a
type-checking-strict library.

## `forceConsistentCasingInFileNames`

```json
{ "forceConsistentCasingInFileNames": true }
```

`import './Foo'` vs `./foo` are different files on Linux, same on
macOS/Windows. This flag catches the mismatch — so a Linux CI doesn't
break code that worked locally.

## Project structure conventions

For libraries:

```
src/        — source
dist/       — compiled output (gitignored)
types/      — additional .d.ts (augmentations)
tsconfig.json
tsconfig.build.json (for build only — extends base)
```

For apps:

```
src/        — source
dist/       — built artifacts (or whatever your bundler emits)
tsconfig.json
```

Multiple tsconfigs are common: a base, then `tsconfig.build.json`,
`tsconfig.test.json`, etc. Use `extends`:

```json
// tsconfig.build.json
{
  "extends": "./tsconfig.json",
  "include": ["src/**/*"],
  "exclude": ["src/**/*.test.ts"]
}
```

## Common gotchas

### `strict` not actually doing much

Make sure your code is being checked. `tsc --noEmit` runs the
compiler without emitting — useful in CI. If `tsc --noEmit` passes
but your IDE shows errors, you're picking up different configs.

### Mixing `Node` and `Bundler` resolution

Pick one. The right one for your environment. Mixing causes "file
not found" surprises.

### `allowJs` for migration

```json
{ "allowJs": true, "checkJs": false }
```

Lets you have `.js` files alongside `.ts`. Useful during migration;
turn off in finalized TS projects.

### Out-of-date `target` matching `tsc` defaults

Old `target: ES5` is still common in legacy projects. It drags in
polyfills you don't need. Set to a modern target for modern code —
fewer transformations, smaller output.

## The discipline

- `strict: true` from day one.
- Plus: `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`,
  `noImplicitOverride`.
- `skipLibCheck: true`. Always.
- Modern `target` (`ES2022`+) — your audience supports it.
- `moduleResolution: "Bundler"` for Vite/Webpack; `NodeNext` for raw
  Node ESM.
- `verbatimModuleSyntax: true` to force explicit type imports.
- Source maps and declaration maps on.
- Keep `tsconfig.json` short and use `extends` for variations.
