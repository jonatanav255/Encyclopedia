# Generics

Generics let you write code that works for any type while preserving
the relationship between inputs and outputs. Without generics you'd
either lose type information (`any`) or have to duplicate the code
for every type.

## The basic idea

```ts
function identity<T>(value: T): T {
  return value;
}

identity(5);       // T inferred as number → returns number
identity('hi');    // T inferred as string → returns string
identity<User>(u); // T explicit
```

`<T>` declares a *type parameter*. When you call the function, T
gets bound to whatever you passed.

The win over `any`:

```ts
function badIdentity(value: any): any { return value; }
const x = badIdentity(5);   // x: any — type lost
const y = identity(5);      // y: number — preserved
```

## Naming conventions

Single-letter for one parameter:

- `T` — Type
- `U`, `V` — additional types
- `K` — Key
- `V` — Value
- `E` — Element
- `R` — Return
- `P` — Props

For complex generic functions, full names are fine:

```ts
function paginate<TItem, TCursor>(...) { ... }
```

The compiler doesn't care about the name.

## Generic functions

```ts
function map<T, U>(arr: T[], fn: (x: T) => U): U[] {
  return arr.map(fn);
}

const lengths = map(['hi', 'world'], s => s.length);
// T = string, U = number → number[]
```

The compiler infers both T and U from arguments. Explicit form:

```ts
const lengths = map<string, number>(['hi', 'world'], s => s.length);
```

Almost always you can let inference do the work.

## Generic constraints

`extends` constrains what T can be:

```ts
function getKey<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

const user = { id: 1, name: 'jon' };
const name = getKey(user, 'name');   // string
const bad = getKey(user, 'missing'); // error — not a key of user
```

The constraint `K extends keyof T` means K must be one of T's keys.
The return type `T[K]` indexes into T with K, giving the value type.

Common constraints:

- `T extends object` — any non-primitive.
- `T extends string` — any string (literal or wide).
- `T extends { id: number }` — must have an `id` field.
- `T extends readonly unknown[]` — any tuple or array.

## Generic types

```ts
type Box<T> = { value: T };

const a: Box<number> = { value: 5 };
const b: Box<string> = { value: 'hi' };

type StringBox = Box<string>;
```

Generics for type aliases work the same as for functions — pass T at
the point of use.

```ts
interface Repository<T> {
  get(id: number): Promise<T>;
  save(item: T): Promise<void>;
  delete(id: number): Promise<void>;
}

class UserRepository implements Repository<User> { ... }
```

## Generic classes

```ts
class Stack<T> {
  private items: T[] = [];
  push(item: T) { this.items.push(item); }
  pop(): T | undefined { return this.items.pop(); }
}

const s = new Stack<number>();
s.push(1);
s.push('hi'); // error
```

## Default type parameters

```ts
type Response<T = unknown> = { ok: true; data: T } | { ok: false; error: string };

const r: Response = ...;             // T defaults to unknown
const r2: Response<User> = ...;       // T explicit
```

Defaults make generic types easier to use when you don't always have
specific information.

## Generic constraints with inference

```ts
function pick<T, K extends keyof T>(obj: T, keys: K[]): Pick<T, K> {
  const out = {} as Pick<T, K>;
  for (const k of keys) out[k] = obj[k];
  return out;
}

const user = { id: 1, name: 'jon', email: 'j@x.com' };
const sub = pick(user, ['id', 'name']);
// typed as { id: number; name: string }
```

`Pick<T, K>` is a built-in utility — see **Utility types**.

## When generics inference goes wrong

Sometimes TS infers too widely:

```ts
function arr<T>(...items: T[]): T[] { return items; }

const a = arr('a', 'b');           // string[] — fine
const b = arr<'a' | 'b'>('a', 'b'); // ('a' | 'b')[] — narrowed
```

The fix: use `as const`, explicit type parameter, or a tuple-rest
pattern:

```ts
function arr<const T extends readonly unknown[]>(...items: T): T { return items; }

const c = arr('a', 'b');           // readonly ['a', 'b']
```

The `const` modifier (TS 5.0+) prevents widening at the call site.

## Constrained inference and `extends infer`

```ts
type Awaited<T> = T extends Promise<infer U> ? U : T;
```

`infer U` extracts the inner type — see **Conditional types** for the
full treatment. The `extends` and `infer` keywords let you compute
types from other types.

## Generic constraints with branded types

```ts
type ValidId<T extends string> = string & { __brand: T };
type UserId = ValidId<'User'>;
type PostId = ValidId<'Post'>;

function getUser(id: UserId) { ... }

getUser('1' as UserId);   // ok
getUser('1' as PostId);   // error
```

See **Branded types** for more.

## When NOT to use generics

```ts
// Bad — T is used once, no relationship
function log<T>(x: T): void { console.log(x); }

// Better
function log(x: unknown): void { console.log(x); }
```

The rule of thumb: **use a generic when the input and output types
are related**. If T appears only once and the function doesn't return
or compose it in a typed way, don't make it generic.

```ts
// Good — output depends on input
function first<T>(arr: T[]): T | undefined { ... }

// Good — relationships between params
function pair<A, B>(a: A, b: B): [A, B] { ... }

// Bad — T unused
function send<T>(payload: T, url: string): void { fetch(url, { body: JSON.stringify(payload) }); }
```

The "bad" example doesn't benefit from being generic — `payload:
unknown` works fine since the function only serializes.

## Higher-kinded gotcha

```ts
function compose<A, B, C>(f: (a: A) => B, g: (b: B) => C): (a: A) => C {
  return a => g(f(a));
}
```

TS handles this well. But:

```ts
function map<F>(arr: F[], fn: F => F) { ... }   // can't abstract F to "the same container type"
```

TypeScript doesn't have higher-kinded types — you can't write
"`Container<T>` for any container." Tools like `fp-ts` simulate them
with HKT encodings, which are powerful but verbose.

## Generic React components

```tsx
function Select<T>({
  options,
  onChange,
}: {
  options: T[];
  onChange: (value: T) => void;
}) {
  return <select onChange={e => onChange(options[+e.target.value])}>{...}</select>;
}

<Select<string> options={['a', 'b']} onChange={s => ...} />
<Select<User> options={users} onChange={u => ...} />
```

Generic components flow types from props through to handlers. The
JSX syntax `<Select<string> ...>` is supported in `.tsx` files.

## Generic constraints in declarations

```ts
class Repository<T extends { id: number }> {
  find(id: number): Promise<T | null> { ... }
}

class UserRepository extends Repository<{ id: number; name: string }> { ... }
```

The constraint at the class level applies to every method.

## Common patterns

### Builder

```ts
class QueryBuilder<T> {
  where<K extends keyof T>(key: K, value: T[K]): this { ... }
}
```

Returning `this` (or `this & Built<...>`) is how typed builders work.
See **Type-level programming** for fancier versions.

### Tagged result

```ts
type Result<T, E = Error> =
  | { ok: true; value: T }
  | { ok: false; error: E };

function ok<T>(value: T): Result<T> { return { ok: true, value }; }
function err<E>(error: E): Result<never, E> { return { ok: false, error }; }
```

### Container

```ts
class Cache<T> {
  private map = new Map<string, T>();
  get(key: string): T | undefined { return this.map.get(key); }
  set(key: string, value: T): void { this.map.set(key, value); }
}
```

## The discipline

- Use generics when the input/output types are related.
- Let inference do the work; rarely write explicit type arguments.
- Constrain (`T extends ...`) when the body needs specific
  capabilities.
- Don't generify when `unknown` works.
- Name parameters descriptively if the function has multiple.
