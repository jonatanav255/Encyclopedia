# Declaration files (`.d.ts`)

`.d.ts` files describe the shape of JavaScript code without
implementing it. They're how TypeScript ships types for libraries,
how you augment third-party packages, and how you tell the compiler
about ambient values like globals.

## What's in a `.d.ts`

Only declarations. No runtime code:

```ts
// types.d.ts
export interface User {
  id: number;
  name: string;
}

export function fetchUser(id: number): Promise<User>;

export const VERSION: string;
```

The `.d.ts` file says "these things exist with these types." A
matching `.js` file implements them at runtime.

```js
// types.js
export function fetchUser(id) { ... }
export const VERSION = '1.0';
```

The compiler trusts the `.d.ts`. If it lies about the JS, you get
runtime errors.

## When you write them

### As a library author

When you publish a library written in TS, emit declaration files:

```json
{
  "compilerOptions": {
    "declaration": true,
    "declarationMap": true,
    "outDir": "./dist"
  }
}
```

`pnpm exec tsc` produces matching `.d.ts` files for every emitted
`.js`. Ship both.

In `package.json`:

```json
{
  "main": "./dist/index.js",
  "types": "./dist/index.d.ts",
  "exports": {
    ".": {
      "import": "./dist/index.mjs",
      "require": "./dist/index.cjs",
      "types": "./dist/index.d.ts"
    }
  }
}
```

### As a library consumer (untyped library)

If a library is plain JS with no types:

```ts
// my-lib.d.ts
declare module 'untyped-library' {
  export function doThing(input: string): number;
  export const VERSION: string;
}
```

The compiler picks this up via the `include` in `tsconfig.json`.

### For ambient values

```ts
// globals.d.ts
declare global {
  const __VERSION__: string;
  var $: JQuery;
}

declare const ENV: 'development' | 'production';
```

`__VERSION__` is replaced at build time by Vite/Webpack;
`globalThis.$` is a jQuery global. Both need to be declared
ambiently — they don't `import` from anywhere.

The `export {}` (or any `import`/`export`) makes the file a module;
without it, top-level `declare` is treated as global.

## `declare` keyword

`declare` says "this exists; don't compile it":

```ts
declare const PI: number;
declare function add(a: number, b: number): number;
declare class User { id: number; }
declare module 'untyped-library' { ... }
declare namespace Express { interface Request {} }
```

No body, no implementation. Pure type information.

## `.d.ts` vs `.ts`

Same syntax for types. The difference:

- `.d.ts` — declarations only. No runtime code allowed (no
  function bodies, no const initializers).
- `.ts` — declarations plus implementations.

The compiler treats `.d.ts` as "trust this." Errors in `.d.ts` are
also harder to surface — `skipLibCheck` in tsconfig usually skips
checking declaration files for speed.

## Bundling declarations

For libraries, you want a single `.d.ts` file (not many fragmented
ones). Tools:

- **`tsc` with `declaration: true`** — emits per-file `.d.ts`.
  Functional, but messy for consumers.
- **`api-extractor`** — Microsoft's tool. Rolls many `.d.ts` into
  one with a stable public API.
- **`tsup`** — bundler that emits both JS and bundled `.d.ts`.
- **`rollup-plugin-dts`** — Rollup plugin for declaration bundling.

`tsup` is the easy choice for new libraries:

```bash
pnpm add -D tsup
pnpm exec tsup src/index.ts --format cjs,esm --dts
```

One file in, both formats out plus a `.d.ts`.

## Triple-slash directives

Old-school way to bring in types:

```ts
/// <reference types="node" />
/// <reference path="./other.d.ts" />
/// <reference lib="dom" />
```

Largely obsolete now — `tsconfig.json`'s `types` field is the modern
equivalent for `@types/*` packages.

You'll still see them in:

- Old codebases.
- Library entry points (`index.d.ts` referencing companion files).
- Tools that need explicit ordering.

## Augmenting other libraries

The most common `.d.ts` use case in app code. See **Declaration
merging** for the deep dive.

```ts
// types/augmentations.d.ts
import 'express-session';

declare module 'express-session' {
  interface SessionData {
    userId?: number;
  }
}
```

```ts
// types/express.d.ts
declare global {
  namespace Express {
    interface Request {
      user?: { id: number; email: string };
    }
  }
}

export {};
```

## Common patterns

### CSS modules

```ts
// css-modules.d.ts
declare module '*.module.css' {
  const classes: { readonly [key: string]: string };
  export default classes;
}
```

Now `import s from './foo.module.css'` is typed.

### Asset imports (Vite)

```ts
// assets.d.ts
declare module '*.svg' {
  const src: string;
  export default src;
}
declare module '*.png' {
  const src: string;
  export default src;
}
```

For typed asset imports.

### Environment variables

```ts
// env.d.ts
declare namespace NodeJS {
  interface ProcessEnv {
    DATABASE_URL: string;
    NODE_ENV: 'development' | 'production' | 'test';
    PORT?: string;
  }
}
```

Or for Vite:

```ts
interface ImportMetaEnv {
  readonly VITE_API_URL: string;
}
interface ImportMeta {
  readonly env: ImportMetaEnv;
}
```

These types **don't validate at runtime**. They're a hint; pair with
Zod or similar for actual checks.

### Polyfill / experimental APIs

```ts
declare global {
  interface Array<T> {
    toSorted(compareFn?: (a: T, b: T) => number): T[];
  }
}
```

If you're using a feature your lib version doesn't yet know about.

## `@types/*` packages

For untyped npm packages, the community publishes types:

```bash
pnpm add -D @types/lodash @types/node
```

`@types/lodash` provides `lodash.d.ts`. The TS compiler finds them
automatically (in `node_modules/@types/`).

You usually don't write your own `@types/*`. If you do, the
`DefinitelyTyped` repository is where they live.

## Common pitfalls

### `.d.ts` is loaded but ignored

If your `.d.ts` doesn't seem to take effect:

- Check `tsconfig.json` `include`/`exclude` — is the file included?
- Check `types` and `typeRoots` — are you whitelisting only some?
- Restart your editor's TS server.

### Augmentations not merging

For `declare module 'X'` to *augment* an existing module, the file
must be a module (have `import`/`export`). For `declare global`, the
file must also be a module, otherwise the declarations become
top-level and conflict.

```ts
declare global { ... }
export {};   // makes this a module
```

### Implementation file out of sync

If your `.d.ts` says `foo: string` but the JS sets `foo` to a number,
TypeScript doesn't catch it. Manually maintained `.d.ts` files drift.

Either:

- Generate `.d.ts` from TS source (so they're always in sync).
- Use Zod schemas as the source of truth and infer the type.
- Write extensive tests for the JS implementation matching the
  declarations.

### Over-reliance on `any`

```ts
declare function untyped(input: any): any;
```

That's basically not adding types. Better: figure out the real shape
and write it. Or use `unknown` to force the caller to narrow.

## The discipline

- For library code: emit `.d.ts` via `tsc`/`tsup` rather than
  hand-writing.
- For app augmentation: keep `.d.ts` files in `src/types/` and
  include them in `tsconfig.json`.
- Use `declare global` sparingly — pollutes everything.
- Pair declaration files with runtime validation for any boundary
  data (env vars, response shapes).
- Trust the types of `@types/*` packages; report bugs upstream when
  they're wrong.

`.d.ts` is the bridge between JS and TS. Used well, you get rich
typing for code that doesn't have it natively. Used poorly, you get
fragile lies the compiler will happily believe.
