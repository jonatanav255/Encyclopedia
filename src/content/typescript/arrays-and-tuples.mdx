# Arrays and tuples

## Arrays

Two equivalent syntaxes:

```ts
const a: number[] = [1, 2, 3];
const b: Array<number> = [1, 2, 3];
```

`number[]` is the more common form. `Array<T>` is the same thing —
useful when `T` is itself complex (e.g., `Array<{ id: number }>` reads
more easily than `{ id: number }[]`).

Both describe "an array of any length where every element is the
specified type."

```ts
const ids: number[] = [1, 2, 3];
ids.push(4);            // ok
ids.push('hi');         // error — not a number
ids[10];                // typed as number (but actually undefined)
```

That last point is a footgun fixed by a flag.

## `noUncheckedIndexedAccess`

By default, `arr[i]` is typed as `T` — the compiler assumes the index
is valid. Set this flag and TS becomes honest:

```json
// tsconfig.json
{
  "compilerOptions": { "noUncheckedIndexedAccess": true }
}
```

```ts
const arr: number[] = [1, 2, 3];
const x = arr[10];        // x: number | undefined  (with the flag)
```

This is one of the most valuable flags in a strict config. Turn it
on for new projects — it catches a real class of bugs (assuming an
array index always returns something).

## Readonly arrays

```ts
const xs: readonly number[] = [1, 2, 3];
xs.push(4);            // error — push doesn't exist on readonly
xs[0] = 99;            // error — can't assign
```

`readonly T[]` (or `ReadonlyArray<T>`) is a separate type from `T[]`.
You can assign `T[]` to `readonly T[]` (widening), not vice versa.

Use for function parameters that don't mutate:

```ts
function sum(nums: readonly number[]): number {
  return nums.reduce((a, b) => a + b, 0);
}
```

This signals intent and prevents accidental mutation. Callers can
pass either kind of array.

## Inferring array types

```ts
const xs = [1, 2, 3];          // number[]
const ys = [1, 'two', true];   // (string | number | boolean)[]
const empty = [];              // any[]  — too wide
```

Empty arrays inherit `any[]` unless you annotate. Once you push
something to them, they become `any[]` — defeating type checking.

```ts
const xs: number[] = [];    // explicit
const ys = [] as number[];  // assertion (same effect)
```

## Tuples

A tuple is an array with a known length and per-position types:

```ts
const pair: [string, number] = ['hi', 42];
pair[0];     // string
pair[1];     // number
pair[2];     // error — index out of range
```

Common uses:

- Return multiple values from a function: `[result, error]`.
- React `useState`: `[value, setValue]`.
- Coordinate pairs, named anchors.

```ts
function divmod(a: number, b: number): [number, number] {
  return [Math.floor(a / b), a % b];
}

const [q, r] = divmod(10, 3);
```

## Tuples with rest

```ts
type Variadic = [first: string, ...rest: number[]];

const a: Variadic = ['hi', 1, 2, 3];
const b: Variadic = ['hi'];           // ok — rest can be empty
const c: Variadic = [1, 2, 3];        // error — first must be string
```

The leading element is fixed; the rest is `number[]`.

## Named tuple elements

```ts
type Range = [start: number, end: number];

const r: Range = [0, 100];
// hovering r[0] shows "start: number" in editors
```

Pure documentation — names don't change behavior. But they make
`useState` and similar APIs read better.

## Tuples are subtypes of arrays

```ts
function first<T>(arr: T[]): T | undefined {
  return arr[0];
}

const p: [string, number] = ['a', 1];
first(p);     // T = string | number
```

A tuple `[A, B, C]` is assignable to `(A | B | C)[]` (with the loss
of length/position info). The other direction needs an assertion.

## `as const` for literal tuples

```ts
const dir = [1, 0];               // number[]
const dir2 = [1, 0] as const;     // readonly [1, 0]
```

`as const` narrows everything to literal types and makes it readonly.
The result is a *tuple of literals*, perfect for things like:

```ts
const DIRECTIONS = {
  up: [0, -1],
  down: [0, 1],
  left: [-1, 0],
  right: [1, 0],
} as const;

type Direction = keyof typeof DIRECTIONS;   // 'up' | 'down' | 'left' | 'right'
type Vector = typeof DIRECTIONS[Direction];  // readonly [...]
```

This pattern shows up everywhere in real codebases: derive a union
type from a config object without listing each member twice.

## Array methods and types

Most array methods preserve types correctly:

```ts
const xs = [1, 2, 3];
xs.map(x => x.toString());   // string[]
xs.filter(x => x > 1);        // number[]
xs.reduce((a, b) => a + b);   // number
xs.find(x => x > 1);          // number | undefined
xs.indexOf(2);                // number
```

`filter` doesn't narrow types unless you give it a type guard:

```ts
const mixed = [1, 'a', null, 2, null];
const cleaned = mixed.filter(x => x !== null);
// typed as (string | number | null)[]   — filter doesn't narrow!

const cleanedTyped = mixed.filter((x): x is string | number => x !== null);
// typed as (string | number)[]
```

This is one of TypeScript's notable rough edges. The user-defined
guard fixes it.

## Tuple type tricks

You can extract pieces of a tuple type:

```ts
type T = [string, number, boolean];

type First = T[0];          // string
type Rest = T[1];           // number
type Length = T['length'];  // 3
type All = T[number];       // string | number | boolean
```

`T[number]` is "the type of any element of T." Useful for converting
tuples to unions.

## `tuple` for variadic generics

```ts
function tuple<T extends unknown[]>(...args: T): T {
  return args;
}

const t = tuple(1, 'a', true);   // typed [number, string, boolean]
```

Variadic generics let you preserve tuple types through function
calls. Useful in functional helpers and routers.

## Don't over-tuple

Tuples are great for fixed-arity returns and React-style pairs. They
become awkward when:

- You'd want named fields. Use an object — `{ value, setValue }`
  reads better than `[value, setValue]` in app code.
- The arity might grow. Tuples are fixed.
- You'd reorder them. Tuples lock position.

```ts
// Better as object
function user(): { id: number; name: string } { ... }

// Reasonable as tuple
function divmod(a: number, b: number): [quotient: number, remainder: number] { ... }
```

## The discipline

- Default to `T[]`.
- Use tuples for fixed pairs and `useState`-style pairs.
- Use `readonly T[]` for parameters that don't mutate.
- Turn on `noUncheckedIndexedAccess` — it catches real bugs.
- Use `as const` to derive literal tuples from arrays.
- Use type guards in `filter` to narrow.
- Don't tuple what should be an object.
