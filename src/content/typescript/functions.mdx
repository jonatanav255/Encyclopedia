# Functions in TypeScript

TypeScript types function parameters, return values, and the
function itself. The annotations look natural — most JS function
declarations need just a few additions.

## Annotating parameters and return

```ts
function add(a: number, b: number): number {
  return a + b;
}

const sub = (a: number, b: number): number => a - b;
```

The `: number` after the parameter list is the return type. You can
usually omit it and let TypeScript infer:

```ts
function add(a: number, b: number) {
  return a + b;        // inferred as number
}
```

**When to annotate the return:**

- **Public exports** — be explicit about your function's contract.
- **Recursive functions** — sometimes TS can't infer.
- **Functions that always return the same kind** — clarity.

**When to skip:**

- Local helpers — inference handles it.
- Simple one-liners — annotation noise.

## Optional and default parameters

```ts
function greet(name: string, greeting?: string): string {
  return `${greeting ?? 'hi'}, ${name}`;
}

function add(a: number, b = 0): number {
  return a + b;
}
```

`b = 0` infers `b: number`. Optional parameters (`b?: number`) are
typed as `number | undefined`.

Required parameters can't come after optional ones:

```ts
function bad(a?: string, b: number) {}  // error
function good(a: number, b?: string) {} // ok
```

For complex option sets, use an object:

```ts
function fetchPage({
  url,
  timeout = 5000,
  retries = 3,
}: { url: string; timeout?: number; retries?: number }) {
  // ...
}

fetchPage({ url: '/api' });
```

Named-options pattern. Way better than positional parameters once you
have more than 2-3.

## Rest parameters

```ts
function sum(...nums: number[]): number {
  return nums.reduce((a, b) => a + b, 0);
}

sum(1, 2, 3);   // 6
sum(...[1, 2, 3]);
```

Or with tuples for typed rest:

```ts
function event(name: string, ...args: [number, string]) {}
event('foo', 1, 'bar');
event('foo', 1);    // error — missing string
```

## Function types

```ts
type Callback = (err: Error | null, data?: string) => void;

const onDone: Callback = (err, data) => {
  if (err) console.error(err);
  else console.log(data);
};
```

You can describe a function type with `(params) => ReturnType`. Use
it for callbacks, event handlers, factory parameters.

Alternative `interface` form:

```ts
interface Callback {
  (err: Error | null, data?: string): void;
}
```

Same thing. `type` is more common for function aliases.

## Method shorthand vs property

```ts
type Foo = {
  method(): void;       // method shorthand
  prop: () => void;     // property of function type
};
```

Functionally similar with one subtle difference: method shorthand has
bivariant parameter checking (looser), the property form has
strict checking under `--strictFunctionTypes`. In practice the
property form is safer. Use it unless you need declaration merging.

## Overloads

When one function's behavior depends on its argument shape:

```ts
function get(id: number): User;
function get(name: string): User[];
function get(query: number | string): User | User[] {
  if (typeof query === 'number') return loadById(query);
  return loadByName(query);
}

const a = get(1);        // User
const b = get('jon');    // User[]
```

The first declarations are *overload signatures* — callers see them.
The last (with body) is the *implementation signature* — only the
compiler sees it.

Overloads are awkward; prefer unions and discriminated returns when
you can:

```ts
function find(
  query: { id: number } | { name: string },
): { kind: 'one'; user: User } | { kind: 'many'; users: User[] }
```

Less elegant but type-safe and easy to consume.

## `this` parameter

In TS you can declare the expected `this` for a function:

```ts
function greet(this: { name: string }): string {
  return `hi ${this.name}`;
}

greet.call({ name: 'jon' });  // ok
greet.call({});                // error — missing name
```

The `this` parameter doesn't appear at runtime. It's purely for type
checking.

For arrow functions, `this` is captured from the outer scope, so the
`this` parameter doesn't apply.

## Generic functions

```ts
function identity<T>(value: T): T {
  return value;
}

identity(5);          // T = number → returns number
identity('hi');       // T = string → returns string
identity<User>(u);    // explicit
```

`<T>` declares a type parameter. The compiler infers it from how you
call the function.

Constraints with `extends`:

```ts
function getKeys<T extends object>(obj: T): (keyof T)[] {
  return Object.keys(obj) as (keyof T)[];
}
```

`T extends object` limits T to object types.

See **Generics** for the deep dive.

## Async functions

```ts
async function fetchUser(id: number): Promise<User> {
  const res = await fetch(`/api/${id}`);
  return res.json();
}
```

Return type is `Promise<T>`, not `T`. TypeScript wraps it for you if
you mark the function `async`.

If you forget the `async` and try to return a `Promise<T>` directly:

```ts
function fetchUser(id: number): Promise<User> {
  return fetch(`/api/${id}`).then(r => r.json());
}
```

Both are fine. The `async`/`await` form just reads more linearly.

## Higher-order functions

```ts
function withRetry<T>(fn: () => Promise<T>, n: number = 3): () => Promise<T> {
  return async () => {
    for (let i = 0; i < n; i++) {
      try { return await fn(); }
      catch (err) { if (i === n - 1) throw err; }
    }
    throw new Error('unreachable');
  };
}

const getUser = withRetry(() => fetch('/api').then(r => r.json()));
```

Generics + function types let you build composable utilities without
losing type safety.

## Function returning a function

```ts
const greet = (greeting: string) => (name: string) => `${greeting}, ${name}`;

greet('hi')('jon');   // "hi, jon"
const hi = greet('hi');
hi('jon');             // "hi, jon"
```

TS infers the chained types. Currying is rare in app code, common in
functional-style libraries.

## `void` return

```ts
function logAndReturn(s: string): void {
  console.log(s);
  return s;     // error — void function can't return a value
}
```

Wait — earlier I said `void` return types can return values when used
in callback positions. Both are true:

```ts
function exec(cb: () => void) {
  cb();
}

exec(() => 42);   // OK — return value ignored
```

When `void` is in a *function type*, it means "I don't care about the
return value." When you *declare* a function with `: void`, the
compiler still lets you `return` (you just can't return non-undefined
explicitly without lying).

The distinction rarely matters; just don't return values from
explicit `void` functions.

## `never` return

```ts
function fail(msg: string): never {
  throw new Error(msg);
}

function loop(): never {
  while (true) {}
}
```

`never` means "this function never returns normally." Useful for:

- Exhaustiveness checks (see **Narrowing**).
- Type-safe error helpers.

## Function inference at call sites

```ts
type Comparator<T> = (a: T, b: T) => number;

function sort<T>(arr: T[], cmp: Comparator<T>): T[] {
  return [...arr].sort(cmp);
}

sort([3, 1, 2], (a, b) => a - b);
// TS infers T = number, so a and b in the comparator are numbers
```

This contextual typing is what makes TS feel "smart" — you write
ordinary code, and the compiler fills in the types.

## The discipline

- Annotate function parameters (always).
- Annotate return type for public/exported functions; let it infer
  for local helpers.
- Use object-options for 3+ parameters.
- Prefer unions/discriminated returns over overloads.
- Use generics when the function works "for any T," not when you
  could just use `unknown`.
