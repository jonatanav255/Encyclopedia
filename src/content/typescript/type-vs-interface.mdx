# `type` vs `interface`

Both declare a named type. For object shapes they're almost
interchangeable. The differences are real but small.

## They mostly do the same thing

```ts
type User = {
  id: number;
  name: string;
};

interface User {
  id: number;
  name: string;
}
```

Either declaration lets you write `function get(u: User) {}`. Both
support optional fields, readonly, generics, methods.

```ts
type Box<T> = { value: T };
interface Box<T> { value: T; }
```

For most everyday use, pick one and stick with it. Most teams default
to `interface` for objects/contracts and `type` for unions, mapped
types, or anything `interface` can't express.

## What `type` can do that `interface` can't

### Unions

```ts
type Status = 'idle' | 'loading' | 'success' | 'error';
type StringOrNumber = string | number;
```

`interface` can't be a union. This alone covers most "use `type`"
cases.

### Tuples

```ts
type Pair = [string, number];
```

### Mapped types

```ts
type ReadonlyUser = { readonly [K in keyof User]: User[K] };
```

### Conditional types

```ts
type NonNullable<T> = T extends null | undefined ? never : T;
```

### Aliases for primitives

```ts
type ID = string;
type Age = number;
```

`interface` can only describe object/function/class shapes.

## What `interface` can do that `type` can't

### Declaration merging

```ts
interface User { id: number; }
interface User { name: string; }
// User is now { id: number; name: string }
```

Two `interface` declarations with the same name merge. Useful for:

- Augmenting third-party types (`declare module 'express' { interface Request { user?: ... } }`).
- Extending the global namespace (`interface Window { myThing: ... }`).
- Library authors splitting a large surface into multiple files.

`type` can't do this — two `type X = ...` declarations conflict.

### Self-referential extension via `extends`

Both can extend:

```ts
interface Admin extends User { permissions: string[]; }

type Admin = User & { permissions: string[] };   // same thing with intersection
```

`interface extends` is slightly more readable and produces clearer
error messages. Functionally they're equivalent for object shapes.

## Subtle differences

### Implements

A class can `implements` either:

```ts
class UserImpl implements User { id = 1; name = 'jon'; }
```

Both work. `interface` is the conventional choice here because of the
historical Java/C# baggage; the compiler treats them equivalently.

### Performance

Decade-old TypeScript advice: `interface` is faster to type-check
because the compiler caches it more aggressively. In modern TS
(5.x+), the difference is negligible for most codebases. Don't pick
based on perf.

### Error messages

`interface`-based errors tend to be a touch clearer because the
compiler can refer to the interface by name. With `type`, you
sometimes see the expanded form, which can be verbose.

```
// interface error
Type 'X' is not assignable to type 'User'.

// type error (sometimes)
Type 'X' is not assignable to type '{ id: number; name: string; ... }'.
```

Not a major factor.

## Recommendations

The community has converged on a fuzzy convention:

- **`type`** for unions, intersections, tuples, primitives, mapped
  types, conditional types, and computed types.
- **`interface`** for object-shape contracts (entities, props,
  options bags) — especially when others might extend them.

```ts
// Good fits for interface
interface User { id: number; name: string; }
interface ButtonProps { label: string; onClick: () => void; }

// Good fits for type
type Status = 'idle' | 'loading' | 'success';
type Coord = [number, number];
type Nullable<T> = T | null;
```

Some teams use `type` exclusively because it's more general. Both
styles are common and reasonable. The key is consistency.

## When it really matters: third-party augmentation

```ts
// global.d.ts
declare global {
  namespace Express {
    interface Request {
      user?: User;
    }
  }
}
```

This works because `interface` merges. If `Express.Request` were a
`type`, you couldn't add fields to it from your own code.

If you're a library author who wants users to extend your types, use
`interface`. If you're a library author who wants users to *not* be
able to extend yours, use `type`.

## When extending across many files

```ts
// in feature-1/types.ts
interface AppConfig { feature1: boolean; }

// in feature-2/types.ts
interface AppConfig { feature2: boolean; }
```

`interface AppConfig` is now both fields. Useful in plugin
architectures, less useful in normal codebases.

`type` would refuse to merge — which is usually what you want for
ordinary application code, because accidental merging is a footgun.

## The mental model

`interface` = an "open" named shape that can be extended and merged.
`type` = a "closed" named alias that can express more kinds of types.

For day-to-day "describe an object" work, pick one and be consistent.
For unions and computed types, `type` is the only option. For
extending third-party libraries' types, `interface` is the only
option.

In practice, most modern codebases mix them based on the rules above
and nobody notices. Don't agonize over it.
