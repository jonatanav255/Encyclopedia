# Modules and imports

TypeScript shares JavaScript's module system but adds **type-only**
imports/exports and module resolution settings that affect how
`.ts` files find each other. The mechanics matter — get them wrong
and your build can be slow, fail at runtime, or import types you
don't intend.

## ESM is the default

```ts
import { add } from './math.js';
import type { User } from './types.js';
import defaultExport from './module.js';

export const PI = 3.14;
export function multiply(a: number, b: number) { return a * b; }
export default class Widget { ... }
export type { User } from './types.js';
```

Use ESM (`import`/`export`) syntax. Avoid `require`/`module.exports`
unless you have a specific reason (legacy CJS, bundler quirks).

## Type-only imports

```ts
import type { User } from './types.js';
import type { Component, FC } from 'react';
```

`import type` tells the compiler "this import is purely for type
information." It's erased at compile time — no runtime import. Use
it when you need a type but not a value.

Why this matters:

- **Faster compilation** — the compiler can skip loading the module
  for types-only imports.
- **No circular import issues at runtime** — if A and B import each
  other's types, `import type` breaks the cycle.
- **Cleaner output** — no leftover imports that pull in modules just
  for their types.

The reverse:

```ts
import { type User, fetchUser } from './users.js';
```

Mixed — some symbols are values, some are types. The `type` keyword
in the import clarifies which is which.

For exports:

```ts
export type { User } from './types.js';
```

Re-exports types without bringing values along.

## `isolatedModules`

```json
// tsconfig.json
{ "compilerOptions": { "isolatedModules": true } }
```

This flag requires that every file can be transpiled in isolation —
no cross-file type information needed. It's required by tools like
Babel, esbuild, swc, and Vite that compile one file at a time.

Implications:

- `export type` and `import type` matter — the compiler can't see
  which imports are types-only without these annotations.
- `const enum` is restricted (can't be elided in isolation).
- Re-exports must be explicit about value vs type.

Set this in any project using a fast non-`tsc` build pipeline.

## `verbatimModuleSyntax`

TS 5.0+ replacement for several older flags:

```json
{ "compilerOptions": { "verbatimModuleSyntax": true } }
```

Forces explicit `import type` / `export type` annotations. Output JS
matches input TS one-to-one (no surprise elisions). This is the
strict mode for module syntax — adopt it in new projects.

## File extensions

Modern Node and ESM require **`.js` extensions** on imports — even
when you're writing `.ts`:

```ts
import { add } from './math.js';   // resolves to math.ts during build
```

Yes, you write `.js` even though the source is `.ts`. The compiler
preserves the extension; runtime needs it.

For bundlers (Vite, webpack, esbuild), extensions are often
optional — they resolve `./math` to `math.ts`. The portable
recommendation is to always write the extension to match what
runtime/build expects.

`moduleResolution` controls this:

- `node` — legacy CJS resolution (no extensions needed). Older
  default.
- `node16`/`nodenext` — strict, requires extensions, distinguishes
  ESM vs CJS based on `package.json` `"type"`.
- `bundler` — TS 5.0+, lenient like bundlers expect, no extensions
  required. The right pick for Vite/Webpack-based projects.

```json
{
  "compilerOptions": {
    "moduleResolution": "Bundler",
    "module": "ESNext",
    "target": "ES2022"
  }
}
```

## Path aliases

```json
{
  "compilerOptions": {
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"],
      "@components/*": ["src/components/*"]
    }
  }
}
```

```ts
import { Button } from '@components/Button';
import { fetchUser } from '@/lib/api';
```

TS understands paths, but **bundlers don't automatically inherit
them**. You need matching config in Vite/Webpack:

```ts
// vite.config.ts
export default defineConfig({
  resolve: {
    alias: { '@': '/src', '@components': '/src/components' },
  },
});
```

Or use a plugin (`vite-tsconfig-paths`, `tsconfig-paths` for Node).

## Default vs named exports

```ts
// math.ts
export const PI = 3.14;
export function add(a: number, b: number) { return a + b; }
export default function sub(a: number, b: number) { return a - b; }
```

```ts
import sub, { PI, add } from './math.js';
import { add as plus } from './math.js';
```

In TypeScript, default exports lose:

- Auto-import suggestions in some editors (the importer picks the
  name, not the export).
- Grep-ability — `import Foo` doesn't appear at the def site.
- Refactor-rename precision.

Many style guides ban default exports for these reasons. Use named
exports unless you have a specific reason (one main class per file,
matching a framework convention like React components).

## Re-exports

```ts
export { add, sub } from './math.js';
export * from './math.js';
export type { User } from './types.js';
export { default as MathWidget } from './widget.js';
```

Barrel files (`index.ts` that re-exports everything):

```ts
// src/components/index.ts
export * from './Button.js';
export * from './Input.js';
```

Convenient but harmful at scale:

- Tree-shaking breaks if any file has side effects.
- Editor jump-to-def becomes a maze.
- Circular dependency hotspots.

Use sparingly. For large folders, prefer importing the specific file.

## Interop with CJS

Old packages export via `module.exports`. To import from TS:

```ts
import express from 'express';   // works if esModuleInterop is true
import * as express from 'express';   // works without esModuleInterop
```

Set `esModuleInterop: true` in tsconfig — basically always. It
synthesizes default imports from CJS modules so you can write
`import x from 'cjs'` and get the right thing.

`allowSyntheticDefaultImports` is the type-checker side of the same
deal. `esModuleInterop` implies it.

## Top-level await

Supported in ESM modules with `target: ES2022` or later:

```ts
const config = await loadConfig();
export { config };
```

Modules that import this wait for it to resolve. Don't overuse —
serializing startup adds latency.

## `import.meta`

```ts
import.meta.url;       // "file:///path/to/file.js"
import.meta.dirname;   // (recent Node) the directory
import.meta.env;       // (Vite) bundler-injected env
```

The set of fields depends on environment. TS knows about the standard
ones; bundler-specific ones need ambient declarations:

```ts
// env.d.ts
interface ImportMetaEnv {
  readonly VITE_API_URL: string;
}
interface ImportMeta {
  readonly env: ImportMetaEnv;
}
```

## Common pitfalls

### Importing a type as a value

```ts
import { User } from './types.js';

if (x instanceof User) { ... }   // error — User has no runtime form
```

If `User` is a type alias or interface, there's no value to use at
runtime. Convert to a class or a string discriminant.

### Circular imports

```ts
// a.ts
import { B } from './b.js';
export const A = new B();
```

```ts
// b.ts
import { A } from './a.js';   // A might be undefined at this point
export class B {}
```

ESM handles cycles better than CJS (live bindings), but the
underlying problem is structural — split shared types into a third
file:

```ts
// a.ts → imports ./shared, ./b
// b.ts → imports ./shared (not ./a)
// shared.ts → just types
```

### `import type` and runtime usage

```ts
import type { Logger } from './logger.js';

const l = new Logger();   // error — Logger isn't imported as value
```

If you need the class at runtime, drop the `type`. If you only need
the type, keep it.

### Missing file extension at runtime

```ts
import { add } from './math';   // works in Vite, fails in raw Node ESM
```

With Node ESM, you need `./math.js`. Some bundlers are lenient. The
safer rule: always write the extension.

## The discipline

- Use ESM syntax. Avoid `require` in `.ts` files.
- Use `import type` for type-only imports.
- Set `verbatimModuleSyntax` or at least `isolatedModules` for new
  projects.
- Use `moduleResolution: "Bundler"` for Vite/Webpack, `nodenext` for
  raw Node.
- Prefer named exports over default.
- Avoid barrel files for large folders.
- Always include `.js` extensions for ESM portability.
