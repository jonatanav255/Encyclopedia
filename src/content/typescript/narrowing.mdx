# Narrowing

A value can have a *wide* type — `string | number | undefined`, for
example. **Narrowing** is how TypeScript figures out which member of
the union you have in a particular branch of your code, so it can let
you use type-specific methods.

This is one of the most important everyday concepts. The compiler
does most of the work; you just need to write the checks it
recognizes.

## `typeof` guards

```ts
function format(x: string | number): string {
  if (typeof x === 'string') {
    return x.toUpperCase();   // x is string here
  }
  return x.toFixed(2);         // x is number here
}
```

TS knows `typeof x === 'string'` narrows to `string`. The same for
`'number'`, `'boolean'`, `'function'`, `'object'`, `'undefined'`,
`'symbol'`, `'bigint'`.

Caveat: `typeof null === 'object'` — TS knows this and won't claim
the value is non-null just because `typeof x === 'object'` passed.

## Truthiness

```ts
function f(s: string | null) {
  if (s) {
    s.toUpperCase();   // s is string here
  }
}
```

A truthy check narrows out `null`, `undefined`, `''`, `0`, `false`,
`NaN`. Be careful with strings/numbers — `if (n)` excludes `0`, which
is often valid.

For strict nullable checks:

```ts
if (s !== null && s !== undefined) { ... }
if (s != null) { ... }       // shortcut for both
```

## Equality narrowing

```ts
function compare(a: string | number, b: string | boolean) {
  if (a === b) {
    // both a and b must be string (only overlapping type)
  }
}
```

When two values are `===`, their types must overlap. The compiler
narrows both to the overlap.

```ts
const ROLE_LIST = ['admin', 'user'] as const;
type Role = typeof ROLE_LIST[number];   // 'admin' | 'user'

function isRole(s: string): s is Role {
  return ROLE_LIST.includes(s as Role);
}
```

## `in` operator

Checks for property existence:

```ts
type Fish = { swim: () => void };
type Bird = { fly: () => void };

function move(animal: Fish | Bird) {
  if ('swim' in animal) {
    animal.swim();        // animal is Fish
  } else {
    animal.fly();         // animal is Bird
  }
}
```

The compiler narrows based on which union member has the property.

## `instanceof`

```ts
function f(err: Error | string) {
  if (err instanceof Error) {
    err.stack;       // err is Error
  } else {
    err.length;      // err is string
  }
}
```

Works for classes (and prototypes). Doesn't work for interfaces or
type aliases — those have no runtime form.

## Discriminated unions

The cleanest narrowing pattern in real codebases:

```ts
type Result =
  | { kind: 'success'; data: User }
  | { kind: 'error'; message: string };

function handle(r: Result) {
  switch (r.kind) {
    case 'success':
      return r.data.name;       // r is the success branch
    case 'error':
      return r.message;          // r is the error branch
  }
}
```

The common field (`kind`) lets the compiler pick the right branch.
This is the standard pattern for state machines, API responses,
Redux actions, anything with a "type tag."

You can use any field as discriminant — `type`, `status`, `_tag`,
`kind`. Just be consistent.

## User-defined type guards

When TS can't narrow automatically, you can teach it:

```ts
function isUser(x: unknown): x is User {
  return typeof x === 'object'
    && x !== null
    && typeof (x as any).id === 'number'
    && typeof (x as any).name === 'string';
}

const data: unknown = JSON.parse(input);
if (isUser(data)) {
  data.name;   // data is User here
}
```

The `x is User` return type tells the compiler "if this returns true,
treat the argument as User from here on."

User-defined guards are how libraries like Zod work — the
`.parse(value)` returns a typed value, and `.safeParse` returns
`{ success: true, data: T } | { success: false; error }`.

Be careful: TS trusts your guard. If the body lies, you'll get
runtime errors despite "passing" the type check.

## Assertion functions

```ts
function assertString(x: unknown): asserts x is string {
  if (typeof x !== 'string') throw new Error('expected string');
}

function f(x: unknown) {
  assertString(x);
  x.toUpperCase();    // x is string from here on
}
```

`asserts` functions narrow without returning a boolean — they throw
instead. Useful for input validators.

```ts
function assertDefined<T>(x: T | undefined): asserts x is T {
  if (x === undefined) throw new Error('expected defined');
}
```

## Non-null assertion: `!`

```ts
function f(x: string | undefined) {
  x!.toUpperCase();   // "trust me, it's defined"
}
```

`!` tells the compiler "I know this isn't null/undefined." No
narrowing happens — you're just suppressing the error.

Use when:

- The compiler can't see what you know (e.g., across an async
  boundary).
- You're certain and would rather not write an unnecessary check.

Don't use as a habit. Each `!` is a place where you've claimed
something the compiler couldn't verify. If you're wrong, you crash.

## Optional chaining + nullish coalescing

The most common narrowing tools in practice:

```ts
const name = user?.profile?.name ?? 'anonymous';
```

`?.` short-circuits to `undefined` if anything in the chain is null
or undefined. `??` falls back when the left side is null/undefined
(but not `0`, `''`, or `false` — those are kept).

```ts
const port = config.port ?? 3000;       // 0 → 0, not 3000
const port = config.port || 3000;       // 0 → 3000  (wrong!)
```

`??` is the right operator when "set to a falsy value" should be
respected. Use `||` only when "treat any falsy value as missing" is
intentional.

## Control flow narrowing

The compiler tracks narrowing through if/else, switch, early returns,
ternaries:

```ts
function f(x: string | null) {
  if (x === null) return;
  // x is string from here
  x.toUpperCase();
}
```

It also un-narrows after operations that might mutate:

```ts
let x: string | null = getMaybe();
if (x === null) return;
// x is string here
await async();
// x is still string — unless TS can't see what async() does
```

Async function calls don't generally un-narrow, but mutations through
references (e.g., calling a function that holds a reference to the
parent object) can.

## Common narrowing mistakes

### Aliased narrowing — variables hold, properties don't (always)

```ts
type R = { user: User | null };

function f(r: R) {
  if (r.user !== null) {
    r.user.email;     // ✅ narrowed
    later(() => {
      r.user.email;   // 🚫 not narrowed — could have been mutated
    });
  }
}
```

Narrowing on a property reference holds only as long as TS can
prove no reassignment happened. Inside callbacks, after function
calls that *could* see `r`, or after `await`, the narrowing
expires.

The fix is to alias to a local:

```ts
function f(r: R) {
  const user = r.user;
  if (user !== null) {
    user.email;
    later(() => {
      user.email;     // ✅ — `user` is a local const, can't be reassigned
    });
  }
}
```

The local `const` is provably immutable for the function's
lifetime. TS keeps the narrowing across closures and async hops.

This is the single most common "I narrowed it, why doesn't it work
in this callback?" surprise. The rule: **narrow the local, not
the property**.

### Re-narrowing after a function call

```ts
function f(x: string | null) {
  if (x === null) return;
  doThing();          // could change `x` via shared state? TS doesn't track that
  x.toUpperCase();    // still works — TS keeps the narrowing
}
```

The compiler assumes function calls don't mutate locals (because
locals can't be mutated from outside). If you're using closures with
shared state, this assumption can be wrong — read-uppercase might
break at runtime.

### Discriminant in nested object

```ts
type R = { user: { status: 'in' | 'out' } };

function f(r: R) {
  if (r.user.status === 'in') {
    // works in modern TS
  }
}
```

Modern TS narrows nested discriminants. Old TS sometimes failed at
this — you needed to destructure first.

### Truthy check on `0`

```ts
function f(count: number | undefined) {
  if (count) {        // BUG: count === 0 falls through
    process(count);
  }
}

if (count !== undefined) { ... }   // better
```

`if (x)` on a number/string excludes `0`/`''` — almost never what
you want. Be explicit.

## The discipline

- Start with narrow types — `'admin' | 'user'`, not `string`. Then
  narrowing isn't needed.
- For runtime input, use `unknown` + a validator (Zod), then narrow
  once at the boundary.
- Prefer discriminated unions for state. The `switch` exhaustiveness
  pattern catches missed cases.
- Use `?.` and `??` for nullish chains.
- Use `!` sparingly. Each one is a point where types and runtime can
  diverge.
- Reach for type guards / assertion functions only when the built-ins
  don't fit.
