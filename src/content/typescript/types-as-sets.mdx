# Types as sets

The mental model that makes TypeScript click: **a type is a set of
possible values**. Once you internalize this, most of the type
system's behavior (assignability, narrowing, unions, intersections,
`never`) becomes intuitive.

This frame is championed in *Total TypeScript* (Pocock) and
*TypeScript in 50 Lessons* (Baumgartner) — and it's the right one.

## Types are sets, values are elements

```ts
type X = string;
```

Read as: "X is the set of all strings."

```ts
const a: string = 'hi';   // 'hi' is in the set "string"
```

```ts
type Y = 'admin' | 'user';   // a set with exactly two elements
const b: Y = 'admin';   // 'admin' is in the set
const c: Y = 'other';   // ERROR — 'other' is not in the set
```

A *literal* type is a set with one element. A *union* type is the
union of sets. The compiler is just doing set membership checks.

## Assignability = subset

```ts
type Narrow = 'admin';
type Wide = 'admin' | 'user';

const a: Narrow = 'admin';
const b: Wide = a;   // OK — narrow is a subset of wide

const c: Wide = 'user';
const d: Narrow = c;   // ERROR — wide isn't a subset of narrow
```

You can assign a value of a narrower type (smaller set) to a wider
type (larger set). Subset → superset always works. The reverse
doesn't.

This is **the rule** that explains every "Type X is not assignable
to type Y" error: X's set must be a subset of Y's set.

## Unions are union of sets

```ts
type A = 'foo' | 'bar';
type B = 'bar' | 'baz';
type Both = A | B;   // 'foo' | 'bar' | 'baz' — union of sets
```

Three elements. Standard set union.

For object types:

```ts
type X = { name: string };
type Y = { age: number };
type Either = X | Y;
```

`Either` is "either X or Y." A value satisfies it if it has the
fields of X OR the fields of Y.

But when reading from a `Either`-typed value, you only have access
to **common** fields:

```ts
function f(v: Either) {
  v.name;   // ERROR — Y values don't have name
  v.age;    // ERROR — X values don't have age
}
```

The compiler is conservative: it only lets you access fields that
exist in *both* X and Y (the intersection of their property sets).

## Intersections are intersection of sets

For literal types, intersection is set intersection:

```ts
type A = 'foo' | 'bar';
type B = 'bar' | 'baz';
type Both = A & B;   // 'bar' — only the overlap
```

Just `'bar'`. The single element in both sets.

For object types, intersection means "must have both shapes":

```ts
type X = { name: string };
type Y = { age: number };
type Combined = X & Y;   // { name: string; age: number }
```

A value of `Combined` must have **both** fields. So the *intersection*
of object types actually *expands* the required fields — opposite
of how you might intuit "intersection."

Why? Because the set of values that satisfy "has a name string" is
huge; the set that "has both name AND age" is smaller. The
intersection of those sets gives you the more constrained shape.

## `never` is the empty set

```ts
type Impossible = string & number;   // never
```

No value is both a string and a number. The intersection is empty.
That's `never`.

`never` is a subtype of everything (you can assign `never` to
anything because `never` has no elements to violate the constraint):

```ts
function impossible(): never { throw new Error(); }
const x: string = impossible();   // OK
```

The empty set is a subset of every set.

## `unknown` and `any` — the top types

`unknown` is the set of *all possible values*. Every type is a
subset.

```ts
let x: unknown = 5;
x = 'hi';
x = { foo: 1 };   // all OK
```

But you can't *read* from `unknown` without narrowing:

```ts
const s: string = x;   // ERROR — not every unknown is a string
if (typeof x === 'string') {
  const s: string = x;   // OK — narrowed
}
```

`any` is the same set but **opts out of the rules**. You can
assign to anything; you can read anything. It's `unknown` with
disabled type-checking.

```ts
const a: any = 5;
const s: string = a;   // OK — any opts out
a.does.not.exist;       // OK — any opts out
```

`any` is a trap. Use `unknown` and narrow.

## `void` is "I don't care"

```ts
function log(x: string): void { console.log(x); }
```

`void` is a special type. As a return type, it means "I don't make
guarantees about what's returned" — so callers shouldn't rely on
the return value.

In *function type position*, a `void` return type accepts any
return:

```ts
type Callback = () => void;
const cb: Callback = () => 42;   // OK — void means "ignored"
```

This is what makes `forEach((x) => result.push(x))` work even though
`push` returns a number.

## Narrowing = restricting the set

```ts
function f(x: string | number) {
  if (typeof x === 'string') {
    x;   // narrowed to string — the subset where typeof is 'string'
  } else {
    x;   // narrowed to number — the complement
  }
}
```

In each branch, the compiler has *narrowed* the type — restricted
the set to a smaller subset based on the runtime check.

Narrowing techniques (covered in **Narrowing**):

- `typeof` — narrows by JS type.
- `instanceof` — narrows by class.
- `in` operator — narrows by property existence.
- Equality checks against a literal.
- Discriminant fields.
- User-defined type guards.

Each is "filter the set to elements that pass this check."

## Generics — sets with a parameter

```ts
type Box<T> = { value: T };
```

`Box<T>` is "the set of objects with a `value` field of type T."
`Box<string>` is a specific subset (where value is a string);
`Box<number>` is another.

Constraints:

```ts
function first<T extends unknown[]>(arr: T): T[0] {
  return arr[0];
}
```

`T extends unknown[]` reads as "T is a subset of the set of arrays."
The function works for any specific array type.

## Conditional types — sets in branches

```ts
type Wrap<T> = T extends string ? { value: T } : { error: T };
```

For T in the set "string," return one type; otherwise return
another. Sets in, type out — chosen by membership.

For unions, conditional types **distribute**:

```ts
type R = Wrap<'a' | 1>;
// = Wrap<'a'> | Wrap<1>
// = { value: 'a' } | { error: 1 }
```

Each element of the input union is mapped separately, then the
results are unioned. This *is* set-theoretic mapping.

## Function types — set of all functions matching a signature

```ts
type Fn = (a: number) => string;
```

The set of all functions that take a number and return a string.

A specific function `const f = (n: number) => n.toString()` is in
that set. A function with a different signature is not.

For function subtyping:

- **Parameters are contravariant**: a function that accepts a wider
  set of inputs is more useful than one that accepts a narrower
  set. `(x: Animal) => void` is a subset of `(x: Dog) => void`
  (the former accepts everything the latter does, plus more).
- **Returns are covariant**: a function that returns a narrower
  set is more useful. `() => Dog` is a subset of `() => Animal`.

See **Variance** for the full treatment.

## Why this model helps

Internalizing "types are sets" demystifies confusing TypeScript
behavior:

- **"Why is `Type X` not assignable to `Type Y`?"** — Because X's
  set has values that aren't in Y's set.
- **"Why does `A & B` have all the fields of both?"** — Because to
  be in both sets, you must satisfy both shape constraints.
- **"Why is `never` the bottom?"** — Empty set; subset of all sets.
- **"Why does conditional distribute over unions?"** — Set-by-set
  mapping.
- **"Why does narrowing make my type smaller?"** — You've filtered
  the set.

When you read a type error, ask: "What are the two sets, and why
isn't one a subset of the other?" The error message usually
follows directly.

## A common confusion

```ts
type X = { name: string };
type Y = { name: string; age: number };

const x: X = { name: 'jon' };
const y: Y = x;   // ERROR
```

"Y has more fields than X, so isn't Y narrower? Shouldn't I be
able to assign X to Y?"

No. Y is the set of objects with both name AND age. X is the set
of objects with at least name. Y is *more restrictive* — a smaller
set. So you can assign Y to X (narrower to wider), not X to Y.

The intuition is reversed from "more fields = bigger" — more fields
mean *more restrictive*, *smaller set*.

## Common pitfalls

### Confusing "wider" with "more elements"

A type with one literal (`'admin'`) is narrower than a string
union (`string`). It has fewer elements; it's a subset; it's more
specific.

A union (`'a' | 'b' | 'c'`) is *narrower* than `string` because
it has fewer elements.

### Reading `any` as "anything"

`any` is "the type system gives up on this value." Not "this is the
top type." Use `unknown` for "I don't know the type yet."

### Treating intersection like field-by-field merge

```ts
type X = { mode: 'edit' };
type Y = { mode: 'view' };
type Both = X & Y;   // { mode: never } — impossible
```

The intersection of `'edit'` and `'view'` is empty. The intersected
type can't exist. The compiler computes `never` for the `mode`
field. You can't construct a value of this type.

### Forgetting `unknown` exists

```ts
const data: any = JSON.parse(input);
```

`any` everywhere is the lazy escape. `unknown` forces you to narrow
before using — that's the safety.

## The mental model

A type is a set. Values are elements of sets. Assignability checks
"is this value's set a subset of the target set?"

- **Wider types**: more elements (string, unknown).
- **Narrower types**: fewer elements (literals, restricted unions).
- **Union**: set union.
- **Intersection**: set intersection (which is "more fields" for
  objects — counterintuitively).
- **`never`**: empty set.
- **`unknown`**: universal set.
- **`any`**: universal set with type-checking disabled.

Once this model clicks, TypeScript stops being a series of magic
incantations. Every "why doesn't this compile?" question becomes
"are these sets compatible, and which direction am I trying to
assign?"

The advanced features (conditional types, mapped types, distributive
patterns) all derive from this same set-theoretic foundation.
