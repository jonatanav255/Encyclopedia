# Monolith vs microservices

The honest version: **a monolith is the right default**. Microservices
solve organizational problems (independent deploys, team autonomy,
scaling parts of the system) by paying a large operational tax
(network failures, distributed transactions, observability, deploy
choreography). If you don't have the organizational problem, you're
buying complexity nobody wants.

Newman's *Building Microservices* is unambiguous: pick microservices
when you can articulate which **specific problem** they solve. "We
want to be like Netflix" is not a problem.

## What each really is

<Compare a="Monolith" b="Microservices">
<>
**One deployable unit.** All code, in one process (or one cluster of
identical processes behind a load balancer), against one or a small
number of databases. Calls between modules are in-memory function
calls. Transactions are local. Refactors are atomic.
</>
<>
**Many small services**, each independently deployable, owning their
own data. Calls between modules are network calls. Transactions span
services (sagas, compensating actions). Refactors require coordinated
deploys.
</>
</Compare>

A "modular monolith" is not microservices — it's a monolith with
careful internal boundaries. That's often the right destination, not
a way station to microservices.

## What microservices buy you

<Cheatsheet>
| Benefit | What it actually means |
|---|---|
| Independent deploys | Team A ships without Team B's approval, code freeze, or release train. |
| Team scaling | Conway's law in your favour: one team per service, no cross-team merge conflicts. |
| Independent scaling | Scale only the hot service, not the whole app. |
| Heterogeneous stacks | Different runtimes per service. Real value: replacing legacy gradually. |
| Fault isolation | A bug in service B can't crash service A's process. |
| Replaceability | Smaller blast radius for rewrites. |
</Cheatsheet>

Note what's NOT on the list: "better code," "faster development,"
"easier to understand." Microservices are organizationally cheaper
and technically more expensive.

## What microservices cost you

<Cheatsheet>
| Cost | Where it bites |
|---|---|
| Network in the request path | Every cross-service call is a partial failure waiting to happen. |
| Distributed transactions | No `BEGIN; ... COMMIT;` across services. Sagas, idempotency, compensations. |
| Eventual consistency | Reads after writes can lie. Every UI now has to handle "not yet." |
| Observability tax | Per-request traces across N services. Without tracing, debugging is archaeology. |
| Deploy choreography | Schema changes require expand-contract across multiple services. |
| Local dev complexity | Running 20 services on a laptop is not running 1. |
| On-call burden | More services = more pagers = more runbooks. |
| Infrastructure | Service mesh, gateway, registry, secrets, CI per service. |
</Cheatsheet>

Fowler calls this the "**microservices premium**" in *Production-Ready
Microservices*: every microservice needs logging, metrics, tracing,
deploy pipelines, health checks, on-call rotation, security review.
Multiply by N services. That's the bill.

## When microservices pay off

Three honest signals, from Newman:

<Steps>
<Step n={1} title="You have >50 engineers on one codebase.">
Below that, the monolith's merge-conflict and release-coordination
cost is small. Above it, deploy bottlenecks become real and team
autonomy starts to matter more than technical purity.
</Step>
<Step n={2} title="You can articulate the bounded context.">
You know where the natural cuts in the domain are — and they don't
generate chatty cross-service traffic on the hot path. If "list
orders for a user" requires calling 5 services, the boundary is
wrong, not the architecture.
</Step>
<Step n={3} title="You can pay the operational tax.">
You have, or will have within 6 months: distributed tracing,
centralized logging, a deploy platform, a service mesh or equivalent,
on-call rotations, and an SRE/platform team. Without these, you're
building a distributed monolith.
</Step>
</Steps>

If you can't check all three, you're not ready. Stay monolith. Make
the **modular monolith** good enough to extract from later (Newman's
"Monolith to Microservices" is the playbook).

## The distributed-systems tax

The shift isn't "monolith plus HTTP." It's a category change. From
*Release It!* (Nygard), the new failure modes you inherit:

- **Partial failure.** Half the system works; the other half times
  out. Your code must handle every downstream being down,
  intermittently, simultaneously.
- **Cascading failure.** One slow service backs up its callers, which
  back up theirs. Without circuit breakers, one service can take down
  the system.
- **Network non-determinism.** "It worked locally" is irrelevant —
  prod has latency spikes, packet loss, DNS failures, TLS handshake
  storms.
- **State drift.** Service A's view of "user 42" is 3 seconds older
  than Service B's. Every read might lie.

These problems don't exist in a monolith. They're not "advanced" —
they're the baseline cost of crossing a process boundary.

## Migration: not all at once

Newman's *Monolith to Microservices* is a book-length answer, but the
shape is this:

```
1. Don't rewrite. Extract.
2. Start with a leaf module — one with few inbound dependencies.
3. Build it side-by-side. Route traffic via a strangler.
4. Move its data with CDC; cut over reads first, then writes.
5. Delete the old code only after the new path is stable for weeks.
6. Then pick the next module. Repeat.
```

Big-bang rewrites fail. Incremental extraction works. See the
**Strangler fig pattern** and **Service decomposition strategies**.

## The middle ground: modular monolith

The honest answer for most teams isn't "monolith or microservices."
It's a **well-modularized monolith** with:

- Strict internal boundaries (no cross-module imports below a
  certain layer).
- One database, but one schema per module — no cross-schema joins.
- Domain events for cross-module reactions, even in-process.
- A clear extraction story for any module that needs to leave.

This gets you 80% of the deploy-independence value (you can extract
when needed) at 20% of the operational cost. Shopify and Stack
Overflow famously run massive modular monoliths.

## The signal that you got it wrong

You think you have microservices. You actually have a **distributed
monolith**:

- Every deploy requires deploying ≥3 services in sequence.
- Services share a database.
- One service "owns" the schema; others read its tables directly.
- A single user request touches 7 services synchronously.
- You can't run one service in isolation locally.

See the dedicated page on **The distributed monolith** for diagnosis
and rescue.

<Callout variant="warning">
The worst architecture is microservices done at the wrong time. You
pay the operational cost without getting team autonomy (because the
boundaries are wrong) and lose the monolith's atomic refactor
ability. This is where most struggling "microservices migrations"
end up.
</Callout>

## A decision frame

Use this in design reviews:

```
- How many engineers will touch this codebase in 18 months? (<50: monolith)
- Are deploy bottlenecks a real measured problem today? (no: monolith)
- Can we name the bounded contexts confidently? (no: monolith)
- Do we have observability, tracing, on-call already? (no: not yet)
- Is there ONE specific module that needs independent scaling/teams? (yes: extract just that one)
```

Microservices are not a destination. They're a tool for specific
organizational and scaling problems. Use the tool when you have the
problem; don't use it because the tool is famous.

## In summary

- Monolith is the correct default. Modular monolith is the correct
  destination for most teams.
- Microservices solve organizational problems at a real operational
  cost; you need >50 engineers, clear bounded contexts, and platform
  maturity before they pay off.
- The distributed-systems tax (partial failure, cascading failure,
  state drift, observability) is inherent — you can't engineer it
  away, only manage it.
- Migrate incrementally with the strangler pattern, never in a
  big-bang rewrite.
- The worst outcome is a distributed monolith — microservices' cost
  without microservices' benefit.
