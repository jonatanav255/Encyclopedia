# Health checks at the load balancer

Health checks decide which backends the LB sends traffic to. A
healthy check means "ready for requests"; a failing check means
"skip this one." Sounds simple. In practice, health checks
themselves cause some of the worst production outages, because
they amplify partial failures into full outages.

The story of "the entire cluster failed because every backend
got marked unhealthy at the same time" is depressingly common.

## Active vs passive checks

### Active

The LB periodically sends a request to each backend (e.g.,
`GET /health` every 10 seconds). If M consecutive checks fail,
mark the backend unhealthy. If N consecutive checks succeed
after that, mark it healthy again.

```
   LB ──── GET /health ────► backend
       ◄────── 200 OK ──────
```

Pros: definitive — you know exactly whether the backend
responds. Detects backends that are running but broken.

Cons: probe traffic adds load. Detection lag = interval ×
threshold (e.g., 10s × 3 = 30s before declaring down).

### Passive

The LB watches real traffic. If a backend returns errors
(5xx) or times out, mark it temporarily down. After a cooldown,
try again.

```
   real request ─► backend ─► 500 error
                            └► mark unhealthy for 30s
```

Pros: zero probe overhead, instant reaction to user-visible
failures.

Cons: requires real traffic to detect (cold backends with no
traffic aren't tested). False positives from one-off errors
(retries help).

Production: use both. Active to catch silent failures and
test cold backends; passive to react instantly to real
problems.

## Shallow vs deep checks

A health endpoint can answer two very different questions:

### Shallow

```
GET /health  →  200 OK  (always, if the process is running)
```

The endpoint returns 200 as long as the HTTP server is up. It
doesn't check the database, the cache, downstream services,
or anything else.

Pros:

- Fast (no work).
- Doesn't fail because of downstream issues.
- Doesn't cause cascading failures (more on that below).

Cons:

- The backend can return 200 to `/health` while every real
  request fails (database connection lost, dependency down).

### Deep

```
GET /health/deep  →  checks DB connection, cache, downstreams
```

Returns 200 only if all critical dependencies are reachable
and recent operations succeeded.

Pros:

- Catches "process is up but useless" states.
- Validates the full request path.

Cons:

- **Cascading failure risk**. If the DB has a 2-second blip,
  every backend's deep health check fails simultaneously. The
  LB marks them all unhealthy. The LB has no backends to send
  traffic to → full outage from a transient DB hiccup.

## The cascading-failure trap

This is the classic disaster. A diagram:

```
   500 requests/sec
         │
         ▼
   ┌────[LB]────┐
   │    │       │
   ▼    ▼       ▼
 [app] [app] [app]   ← every app does GET /health → DB
   │    │       │       (deep check)
   └────┼───────┘
        ▼
       [DB]   ← brief slowdown
```

The DB has a 5-second slowdown (vacuum, network glitch,
unrelated heavy query). Every app's `/health` query slows.
The LB sees the slow responses, declares all backends
unhealthy, and returns 503s to clients. The DB recovers in
10 seconds — but the LB takes another 30 seconds to
re-declare backends healthy (3 consecutive passes × 10s
interval). Total user-facing outage: 40 seconds, for a 5-
second DB blip.

Avoiding this is the central craft of health-check design.

## Rules for good health checks

<Steps>
<Step n={1} title="Use shallow checks for LB routing decisions.">
The LB's check should answer "is this process responsive?"
Not "is everything downstream working?" Process health is
local and binary; system health is shared and gradient.
</Step>
<Step n={2} title="Use deep checks for observability — not for routing.">
Run a separate `/health/deep` endpoint that monitoring
scrapes. Alert on degraded health, page on outages — but
don't let the LB take backends out of rotation based on it.
</Step>
<Step n={3} title="Fail closed: small fraction of backends, not all.">
If a deep-check-based system has to fail backends, ensure
the LB can't take down more than X% of the fleet at once. A
"max unhealthy" threshold prevents whole-cluster ejection.
</Step>
<Step n={4} title="Slow down the recovery declaration.">
Going from healthy to unhealthy: fast (1–3 failures). Going
back to healthy: slow (10+ successes). Prevents flapping when
a backend is partially recovered.
</Step>
<Step n={5} title="Health check cost matters.">
A `/health` that's expensive (queries the DB, runs a function)
adds load you can't escape. Keep it cheap: cached state +
liveness only.
</Step>
</Steps>

## Liveness vs readiness

Kubernetes formalized a useful distinction:

- **Liveness**: is the process alive? If not, restart it.
- **Readiness**: is the process ready to serve traffic? If not,
  remove from LB rotation but don't restart.

Why split them:

- A process that's slow because the DB is slow is *alive* but
  not *ready*. Restarting it doesn't help; it'll come back
  and hit the same slow DB.
- A process that's deadlocked is alive (TCP port responds) but
  not actually serving. Restarting it does help.

Liveness should be a **shallow** check that fails only when
the process is truly broken (deadlock, OOM-pending). Readiness
can be slightly deeper but should still be local-only — does
the process have its config loaded, has it warmed up, can it
accept connections.

Neither should depend on shared downstream services.

## Startup probes

Modern systems (K8s, ECS) added **startup probes** that run
*before* liveness/readiness during process startup. Startup
probes have a long timeout and accept "not yet" as an answer.
Once the startup probe passes, liveness/readiness take over.

Without startup probes, a slow-starting backend (e.g., Java
service warming JIT for 30 seconds) gets killed by liveness
checks before it ever serves traffic. Then it restarts, gets
killed again — a death spiral.

## What to put in a real /health endpoint

For most services, the right /health is:

```js
app.get('/health', (_, res) => {
  res.status(200).send('ok');
});
```

That's it. The process is up; it can serve HTTP; the LB
should send traffic. Don't add a DB call here unless you have
a really good reason.

For deeper checks (scraped by monitoring, not by the LB):

```js
app.get('/health/deep', async (_, res) => {
  const checks = {
    db: await tryDbPing().catch(() => false),
    cache: await tryCachePing().catch(() => false),
    upstream: await tryUpstreamPing().catch(() => false),
  };
  const ok = Object.values(checks).every(Boolean);
  res.status(ok ? 200 : 503).json(checks);
});
```

Run this on a non-LB-routing endpoint. Alert on it. Don't
gate routing on it.

## Outlier detection

Modern LBs (Envoy, Linkerd, AWS ALB) add **outlier detection**
on top of health checks. Instead of just checking `/health`,
they watch real traffic and eject backends that are returning
errors at a higher rate than peers — or are dramatically
slower than peers.

```
Eject if:
- 5xx rate > 50% over last 30s
- p99 latency > 5× cluster average
```

Outlier detection is more responsive to real failure than
periodic health checks. It also has bounds (don't eject more
than 10% of the fleet) to avoid the cascading-failure trap.

This is where "health checks" and "circuit breaking" blur.
Envoy's outlier detection is effectively a per-backend circuit
breaker driven by the LB.

## Circuit breaking at the LB level

The LB can implement circuit-breaker patterns:

- **Maximum connections per backend**: don't open more than N
  concurrent connections to one backend, even if it claims to
  be healthy.
- **Maximum pending requests per backend**: queue depth limit.
- **Maximum retries**: don't amplify load via aggressive
  retries.

These limits protect backends from being overwhelmed when
they're partially degraded. Without them, a slightly-slow
backend gets a deeper queue, which makes it slower, which
deepens the queue further — runaway.

Envoy's circuit breakers, AWS ALB's target group attributes,
and Linkerd's per-service limits all implement this.

## Heath check interval and threshold tuning

The math:

```
time to detect failure = interval × (failure_threshold + 1)
```

```
time to recover = interval × success_threshold
```

Typical defaults:

- Interval: 10–30 seconds.
- Failure threshold: 2–3.
- Success threshold: 2–5.

Failure detection: 20–90 seconds. Recovery: 20–150 seconds.

If you need faster detection, use shorter intervals — but
remember the probe traffic scales: a 1-second interval with
50 backends and 10 LB nodes = 500 probes/second.

## Common pitfalls

### Deep health check that takes down the cluster

Already covered. The single most common production
catastrophe involving health checks. Keep LB-routing checks
shallow.

### Aggressive failure threshold + retries = death spiral

Threshold of 1 failure marks a backend down. The LB retries to
another backend, which gets the extra load, which slows it
down, which trips its threshold. The fleet topples one
backend at a time. Use threshold ≥ 2 and rate-limit retries.

### Health check that doesn't exercise the real code path

Backend's `/health` returns 200, but real `/api/users/:id`
crashes due to a missing config. The LB sees green; users
see red. Add a smoke test of a real endpoint to your
monitoring (not to the LB's health check).

### Forgetting timeouts on health check requests

Health check with no timeout = a slow backend holds a probe
open forever. The LB never declares it down; new traffic
continues to be sent. Always set probe timeouts (1–3
seconds) shorter than the interval.

### Health check that depends on a single SPOF

Health check pings `analytics-service` because "we want to
make sure analytics is working." Analytics goes down → every
backend's check fails → fleet ejected → full outage. Don't
let your hot path depend on a non-critical dependency.

### No `max_unhealthy` threshold

The LB will eject 100% of backends if 100% fail their
checks. If your check is even slightly correlated with
shared infrastructure (the DB, the resolver, the LB itself),
all of them will fail at once. Set `max_unhealthy` to 25–50%
of the fleet — if more than that fails, something is
catastrophically wrong and continuing to serve from
"unhealthy" backends is better than serving nothing.

### Confusing liveness, readiness, and startup

K8s has all three for a reason. Liveness → restart. Readiness
→ remove from rotation. Startup → grace period during boot.
Mixing them up causes restart loops or cold-start failures.

## In summary

- Health checks decide which backends serve traffic.
- Active + passive together cover periodic probing and
  real-time error detection.
- **Keep LB-routing checks shallow.** Deep checks belong in
  monitoring, not in routing decisions.
- Outlier detection and circuit breakers at the LB level
  catch what health checks miss.
- Bound the fraction of unhealthy backends the LB can eject
  at once — never let the LB take the entire fleet down.
- Liveness vs readiness vs startup — distinct concepts, distinct
  endpoints. Mixing them up is a classic source of bad
  outages.
