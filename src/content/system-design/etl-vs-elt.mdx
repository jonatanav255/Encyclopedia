# ETL vs ELT

Two orderings of the same three steps in a data pipeline:

- **ETL (Extract, Transform, Load)**: pull data from sources,
  transform in a dedicated tool, load into the warehouse.
  The classic data-warehouse pattern.
- **ELT (Extract, Load, Transform)**: pull data, load raw
  into the warehouse, transform inside the warehouse using
  its SQL engine. The cloud-warehouse pattern.

The shift from ETL to ELT is one of the biggest changes in
data platforms in the last decade.

## ETL: the classic order

```
sources → ETL tool (Spark, Informatica) → transform → load → warehouse
```

- Transform is done in a dedicated ETL engine.
- Data lands in the warehouse already structured.
- The warehouse never sees raw data.

Pros:

- Warehouse stays clean; only curated data lives there.
- Strict schema; columns ready for BI.
- Validation can happen pre-load.

Cons:

- Transform logic in a separate tool with separate
  vocabulary.
- Changing requirements means re-transforming and reloading.
- Initial pipeline development is heavy.

Common ETL tools: Informatica, Talend, Apache NiFi, AWS Glue,
Spark for custom code.

## ELT: the modern order

```
sources → load raw → warehouse → transform via SQL → models
```

- Load raw data into a staging area.
- Use the warehouse's SQL engine to transform into curated
  models.
- The "transform" step is just SQL.

Pros:

- Warehouses (Snowflake, BigQuery, Redshift) are now fast
  enough that doing transforms in-warehouse is viable.
- Raw data preserved — can re-derive new models without
  re-extracting.
- Transform logic in SQL, where analysts and analytics
  engineers can read and modify it.
- dbt-style version-controlled transform models.

Cons:

- Warehouse storage cost (for raw data).
- Warehouse compute cost (running transform queries).
- Complex semi-structured data harder to transform in SQL.

Modern stacks: Fivetran/Airbyte (extract+load) → Snowflake/BigQuery
(load+transform) → dbt (models).

## Why ELT won (mostly)

A few changes made ELT practical:

1. **Cloud warehouses are fast and cheap.** Snowflake,
   BigQuery, Databricks can process billions of rows in
   seconds. Storage is cheap (object storage).
2. **Compute is decoupled from storage.** Pay for what you
   use; spin up for transforms, down after.
3. **dbt and SQL-first modeling** popularized version-controlled,
   testable, documented transformations.
4. **Schema-on-read** is friendlier to evolving data;
   transforming downstream is more flexible than transforming
   on ingest.

For most modern data platforms, ELT is the default.

## When ETL still wins

- **Tight compliance / data residency**: transform sensitive
  fields before they hit the warehouse.
- **Real-time transformation**: stream processing
  (Flink, Kafka Streams) is technically ETL — transform in
  the stream before landing.
- **Format conversion**: incoming binary protocols need
  decoding before SQL can touch them.
- **Heavy ML / vector processing**: warehouses are SQL; if
  you need PyTorch in the pipeline, ETL with Spark.
- **Legacy systems**: replacing existing ETL is months of
  work; sometimes you just keep it.

## The hybrid: EtLT

Extract, light-transform, load, full-transform. Light
transformations (decoding, filtering, basic enrichment)
happen during load. Heavy transformations (joins,
aggregations, business logic) happen in the warehouse.

This is what most "ELT" pipelines really do in practice.

## dbt and the analytics-engineer role

dbt (data build tool) is the canonical ELT-transform tool.
It treats SQL transformations as code:

```sql
-- models/marts/orders.sql
{{ config(materialized='table') }}

WITH raw_orders AS (
  SELECT * FROM {{ source('shopify', 'orders') }}
),
enriched AS (
  SELECT
    o.*,
    c.country,
    c.tier
  FROM raw_orders o
  LEFT JOIN {{ ref('customers') }} c ON o.customer_id = c.id
)
SELECT * FROM enriched
```

Versioned in git; tested (assertions); documented; lineage
graphed. The "analytics engineer" role exists because of dbt
and ELT.

## The data lineage problem

Both ETL and ELT have it: which transformations produced
which fields? In an outage, you need to trace bad data back
to its source.

Tools: OpenLineage, dbt's auto-generated docs, Marquez,
DataHub. The most-overlooked piece of data infrastructure;
spending time here pays back forever.

## Common antipatterns

### "We need to keep using ETL because we always have"

Pre-cloud warehouses couldn't handle ELT loads. Modern
warehouses can. The legacy ETL tool is often more expensive
and slower than the warehouse it feeds.

### "Just dump everything raw into one big table"

Even with ELT, you need staging schemas, governance, naming
conventions. Without those, the warehouse becomes a data
swamp.

### Doing ML-style transforms in SQL

If your transformation needs Python / R / vector math, SQL
in the warehouse is awkward. Use the warehouse for the
SQL-shaped work; Spark or Python notebooks for the rest.

### Running raw data through dashboards

Dashboards directly on raw source tables get slow and break
when the source schema changes. Always materialize curated
models for downstream consumption.

## In summary

- ETL: transform before loading; clean warehouse,
  dedicated tool.
- ELT: load raw, transform in-warehouse via SQL;
  flexible, dbt-friendly.
- ELT won for most use cases — cloud warehouses are fast +
  cheap enough.
- ETL still wins for compliance, real-time, heavy ML, format
  conversion.
- Most production pipelines are EtLT in practice.
- Data lineage is overlooked — invest in it early.
