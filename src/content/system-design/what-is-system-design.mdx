# What is system design

"System design" is the layer of decisions between **product
requirements** and **code**. Given "we want a chat app for 100M
users," what components exist, what do they look like from the
outside, how do they talk to each other, where does data live,
what happens when something breaks?

It's the part of engineering that doesn't show up in a unit test
but determines whether the system survives.

## Design vs architecture vs implementation

People use these words interchangeably; they're not the same.

<Compare a="Architecture" b="System design">
<>
The **highest-level** structure: the major subsystems
(services, data stores, message buses, edges) and the
constraints that bind them (latency budgets, consistency
guarantees, deployment topology). Architecture is what survives
a rewrite.
</>
<>
The next layer down: **how each subsystem is shaped** to meet
its responsibilities. APIs, data models, sharding strategy,
caching layout, replication mode, failure handling. System
design is what you draw on the whiteboard in an interview.
</>
</Compare>

**Implementation** is the layer below that — the actual code,
library choices, query plans, struct layouts. Implementation
can change weekly; system design changes quarterly;
architecture changes once a year (and painfully).

The skill at each level is different:

- **Implementation**: language, framework, library knowledge.
- **System design**: trade-off recognition. There's no single
  right answer — every choice exchanges one cost for another.
- **Architecture**: organizational fit. Who owns what, how teams
  evolve, what the next 3 years look like.

## What system design actually asks

Every system-design problem reduces to the same handful of
questions. The names of the components change; the questions
don't.

<Steps>
<Step n={1} title="What are we building?">
The product requirements, the user actions, the SLAs you're
promising. Without this, no design choice has a basis.
</Step>
<Step n={2} title="How big?">
Users, requests, data volume, read/write ratio, growth rate.
"100 QPS" and "1M QPS" force completely different
architectures.
</Step>
<Step n={3} title="What can go wrong?">
What if a server dies? A region? The database leader? A
network partition between you and your payment provider? The
answers shape replication, retries, and failover.
</Step>
<Step n={4} title="What can we tolerate?">
Stale reads? Lost writes? 200ms vs 2s latency? The honest
answer per-feature is the foundation of every consistency,
caching, and replication choice.
</Step>
<Step n={5} title="What does it cost?">
Servers, bandwidth, storage, engineering time, on-call burden.
The most elegant design is irrelevant if it bankrupts the
company.
</Step>
</Steps>

## The vocabulary of decisions

A good design isn't a drawing — it's an **argued chain of
decisions**. Each decision sounds like:

> "We chose **X** over **Y** because we need **property Z** at
> this scale. We accept the cost of **W**."

For example:

- "We chose **cursor pagination** over **offset** because the
  feed has 10M+ items and offset's seek time blows past our
  100ms budget. We accept that clients can't jump to arbitrary
  pages."
- "We chose **eventual consistency** for the like counter
  because losing one count is invisible to users; we keep
  **strong consistency** for the wallet balance because losing
  one cent is a regulatory event."

If you can't articulate a design as a chain of these decisions,
you don't have a design — you have a sketch.

## Why this is hard

Three reasons system design is harder than it looks:

### 1. The constraints are invisible

A junior engineer designs for the happy path. A senior engineer
designs for the failure path — which doesn't appear in the
requirements doc. Network partitions, GC pauses, slow
dependencies, retry storms, hot keys. These are the constraints
that actually shape the system, and they're never written down.

### 2. Every choice has multiple consequences

Adding a cache reduces DB load (good), introduces a consistency
problem (bad), adds a moving part to operate (worse), and
changes the failure mode (now you have to think about cache
unavailability separately). One arrow on the diagram costs you
weeks of careful thought.

### 3. The right answer depends on context

There is no "best" database, "best" message broker, "best" API
style. The answer is always "for **this** workload, **this**
team, **this** SLA, **this** scale." Pattern-matching from
"what Google does" usually misleads you because you don't have
Google's traffic, team, or budget.

## What this section is not

- **Not a substitute for the books.** The 50 canonical books
  (Designing Data-Intensive Applications, System Design
  Interview Vol 1/2, Designing Distributed Systems, Building
  Microservices, the SRE books, Release It!, Database
  Internals, etc.) go deeper on every topic. Use this as the
  map; use the books for the territory.
- **Not opinionated style guidance.** This isn't "the right way
  to design systems." It's a vocabulary and a survey. Strong
  opinions belong in your own design docs.
- **Not language-specific.** System design lives above the
  language. Examples may use HTTP/JSON syntax for clarity but
  apply equally to gRPC, Thrift, or anything else.

## In summary

System design sits between product and code. It's the layer
where you choose components, draw boundaries, and reason about
failures. The work is mostly trade-offs, and the deliverable is
a defensible chain of decisions — not a clever diagram.
