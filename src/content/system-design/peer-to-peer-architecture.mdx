# Peer-to-peer architecture

No central coordinator. Every node is equal — each can read,
write, route, store, gossip. Nodes discover each other,
distribute work and data across themselves, and reach consensus
without a designated leader.

The opposite of leader-follower. Where leader-follower
centralizes coordination, P2P distributes it.

## Examples

- **Cassandra, Riak** — Dynamo-style key-value stores with no
  leader.
- **BitTorrent** — file sharing with no central server.
- **Blockchain (Bitcoin, Ethereum)** — anonymous nodes agreeing
  on a global ledger.
- **IPFS** — content-addressed distributed filesystem.
- **Distributed hash tables (Chord, Kademlia)** — lookup
  service without a server.

## Core primitives

P2P systems combine a small toolkit:

<Cheatsheet>
| Primitive | Purpose |
|---|---|
| Consistent hashing | Decides which node owns which key |
| Gossip protocol | Membership, failure detection, metadata propagation |
| Vector clocks | Detecting concurrent updates |
| Quorum reads/writes | Tunable consistency |
| Anti-entropy | Repair drift between replicas (Merkle trees) |
| DHT lookup | Routing requests to the right node |
</Cheatsheet>

These show up across all P2P systems with minor variations.

## DHT: routing without a central directory

In a system with N nodes, "where does key K live?" must be
answerable without a directory. Consistent hashing + a DHT
(distributed hash table) is the answer:

- Hash the key to a ring position.
- The node "responsible" for that range owns the key.
- Each node knows about O(log N) other nodes (the "finger
  table" in Chord), so any node can route any lookup in O(log
  N) hops.

For datacenter-scale P2P (Cassandra), every node knows about
every other node — no DHT routing needed. For internet-scale
P2P (BitTorrent, IPFS), DHT is essential.

## What P2P gives you

- **No SPOF.** Any node can die without taking down the
  system.
- **Add capacity by adding nodes.** Each new node takes a
  share of data and traffic; no master to coordinate.
- **High write availability.** Writes can hit any node;
  there's no leader to fail over.
- **Geographic flexibility.** Nodes can live anywhere; no
  central region.

## What P2P costs you

- **Eventual consistency by default.** Strong consistency
  requires extra mechanisms (quorum reads, consensus).
- **Conflict resolution is your problem.** Two concurrent
  writes need a merge strategy (LWW, CRDT, app-level).
- **Hard to debug.** "Why is this node returning stale data?"
  is harder when every node could be the issue.
- **No global view.** You can't easily ask "what's the current
  state of the system?" without polling many nodes.

## Membership and failure detection

Nodes need to know which other nodes are alive.

- **Gossip-based** (Cassandra, Riak, Consul): per-node random
  exchanges propagate membership.
- **DHT-based** (Chord, Kademlia): each node maintains a
  routing table; failure detection happens on lookup.
- **Beacon-based** (BitTorrent): periodically announce
  presence to known peers.

(See gossip-protocols and failure-detectors.)

## Writes and replication

Each key has N replicas. Writes can hit any node:

- The coordinator (the node the client contacts) forwards to

Reads similarly: R reads, return the most recent (vector
clock or timestamp).

## P2P at internet scale: BitTorrent, IPFS

When N is millions of nodes, full membership knowledge is
impossible. Use DHT for lookup; each node tracks only its
neighbors.

- **BitTorrent**: trackers (or trackerless DHT) tell peers
  which other peers have which pieces.
- **IPFS**: content-addressed; lookup by content hash via
  DHT; data fetched in parallel from many peers.

## P2P at datacenter scale: Cassandra, Riak

Hundreds to thousands of nodes; everyone knows about
everyone. Gossip propagates membership. Consistent hashing
+ replication factor decide where data lives.

The key win: linear scale-out and write availability across
data centers.

## When P2P fits

- **Massive write availability needed.** Single-leader can't
  keep up.
- **Geographic distribution is core.** Different regions need
  to write independently.
- **No tolerance for SPOF.** Even momentary central failure
  is unacceptable.
- **Eventual consistency is acceptable.** Or strong
  consistency is needed only on a small subset of operations
  (LWT path).

## When it doesn't

- **Strong consistency by default.** Coordination-free P2P
  can't give you linearizable single-object reads cheaply.
- **Multi-key transactions.** P2P stores almost universally
  lack them.
- **Small systems.** A 3-node Raft cluster is simpler than a
  3-node P2P system.

## In summary

- Peer-to-peer: no central coordinator; every node equal.
- Built from consistent hashing + gossip + vector clocks +
  quorums + anti-entropy + DHT.
- Wins: no SPOF, write availability, geographic flexibility,
  horizontal scale.
- Costs: eventual consistency by default, conflict resolution,
  harder debugging, no global view.
- Examples: Cassandra, Riak (datacenter); BitTorrent, IPFS,
  blockchain (internet).
- Pick P2P when leaderlessness is the requirement, not
  because it sounds modern.
