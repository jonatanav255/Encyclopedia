# Recommendation system architecture

A system that picks **which items to show a user**: the
Netflix homepage, YouTube next-up, Amazon "you may also
like," Spotify Discover Weekly, Twitter's For You feed.

The shape is similar across all of these. This page is about
the **system**, not the algorithms.

## The overall shape

```
user request → feature service → recommendation service
                                       │
              ┌────────────────────────┼────────────────────┐
              ▼                        ▼                    ▼
         candidate generation     light ranking        heavy ranking
              │                        │                    │
              ▼                        ▼                    ▼
              └──────────────── re-ranking ─────────────────┘
                                       ▼
                                user-facing list
```

Same ranking pipeline as search (see ranking-pipelines), with
recommendation-specific candidate generation.

## Candidate generation strategies

For recommendations, "what could we possibly show?" has many
sources:

### Collaborative filtering

Users who liked X also liked Y. Build co-occurrence matrices
from interaction history. Recommend items frequently liked
together with what the user already likes.

Implementations: matrix factorization, alternating least
squares.

### Content-based

Item embeddings (text, image, audio features). Recommend
items similar to ones the user engaged with.

Implementations: trained encoders (BERT for text, ResNet
for images), pre-trained foundation models.

### Two-tower (modern default)

Train a user-tower (turns user features + history into a
vector) and an item-tower (turns item features into a vector).
Recommendations = nearest items to the user's vector.

```
user (features) → user-tower → user vector
item (features) → item-tower → item vector

score = user vector · item vector
```

Used by YouTube, Spotify, many production systems. Combines
content + collaborative signals via training.

### Graph-based

Random walks on a user-item graph; items reached
frequently are candidates. Used by Pinterest, Twitter.

### Popularity

The default for cold-start users. "Trending now," "what's
popular in your country."

Most systems combine several sources, deduplicate, and pass
the union to ranking.

## Ranking

Same as search (recall → light → heavy → re-rank). Specific
signals for recommendations:

- **Click-through rate** for the user × item pair (predicted).
- **Engagement features** (likes, comments, shares).
- **Freshness** (recent items boost).
- **Recency of interaction** with similar items.

## Feature store

Both online (serving) and offline (training) need the same
features. A **feature store** keeps them consistent:

- Compute features in batch (daily) and stream (realtime).
- Serve features at sub-ms latency online.
- Time-aligned snapshots for training.

Tools: Feast, Tecton, Vertex AI Feature Store, custom (Uber's
Michelangelo, LinkedIn's Frame).

## Embedding store

For two-tower retrieval, item embeddings live in a vector
database (Pinecone, Weaviate, Vespa, Milvus) or a custom
approximate-nearest-neighbor index (FAISS, ScaNN, HNSW).

The user's vector is computed online; nearest-N items
retrieved in milliseconds even with billions of items.

## Cold start

A new user has no history; a new item has no engagement.

For new users:

- Default to popularity.
- Quickly learn from first few interactions.
- Use registration signals (country, age, declared interests).

For new items:

- Use content-based signals (title, description, tags).
- Boost initially to gather engagement signal.
- Pair with established items to "explore" them.

## Diversity and serendipity

Pure relevance ranking produces filter bubbles. Add
explicit diversity:

- **Maximal marginal relevance**: each next item must be
  different enough from already-shown.
- **Re-rank with diversity constraint**: maximum N items per
  source / category.
- **Exploration**: a small fraction of slots filled with
  less-confident picks.

## Real-time updates

User just watched something. The next recommendation should
reflect it.

- **Realtime feature update**: feature store receives the
  event and updates user features within seconds.
- **Realtime candidate re-generation**: certain candidates
  may now match better.
- **Realtime model adjustment**: contextual signals influence
  the ranking.

Most production systems do realtime at the feature level,
with model updates batched.

## Personalization vs context

A recommendation depends on both:

- **Who the user is**: long-term preferences.
- **What the context is**: time of day, device, source page.

Encode both in the user-tower features. A user might want
"work focus music" at 9am and "party music" at 10pm — same
user, different context, different recommendations.

## Logging and offline evaluation

Every recommendation served must be logged with:

- Candidates considered (with scores at each stage).
- User context.
- Outcomes (click, dwell, like, share, skip).

Used for:

- Training the next model.
- Offline replay of new ranking strategies.
- A/B test analysis.

Without logging, no progress.

## Evaluation

Offline: NDCG, AUC, precision-at-K. Useful for picking
models to A/B test.

Online: A/B testing against control. Metrics: engagement,
conversion, retention, revenue.

Online wins. Offline gives a candidate; online tests if it's
better.

## In summary

- Recommendation system: candidate generation + ranking
  pipeline + re-rank.
- Candidates from collaborative, content-based, two-tower,
  graph, popularity sources.
- Two-tower with embedding-based retrieval is the modern
  default.
- Feature store keeps features consistent between training
  and serving.
- Diversity constraints break filter bubbles; cold-start
  needs explicit handling.
- Real-time updates at feature level; model updates
  batched.
- Logging is non-negotiable; A/B tests are the truth.
