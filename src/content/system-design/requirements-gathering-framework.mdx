# Requirements gathering for system design

Before drawing a single box, you need a small, repeatable script
that turns "design X" into a list of constraints you can defend.
The script is the same in interviews and on real teams; it's just
slower in real teams.

If you skip this step you'll spend the next hour optimizing
something the interviewer never asked for.

## The shape of the script

<Steps>
<Step n={1} title="Restate the problem in one sentence.">
"You want a system that lets users do A, where B people use it
at C scale, and the most important property is D." If the
interviewer corrects you, you've already won 5 minutes.
</Step>
<Step n={2} title="Enumerate the core user actions (FRs).">
3-7 verbs, no more. Anything bigger is two systems.
</Step>
<Step n={3} title="Lock in the scale (NFRs).">
DAU, read/write ratio, peak factor, geography, latency budget,
availability target. Numbers, not adjectives.
</Step>
<Step n={4} title="State what is OUT of scope.">
Auth, payments, abuse, admin tools — whatever doesn't matter
for the next 40 minutes. This is the most-skipped step.
</Step>
<Step n={5} title="Confirm before designing.">
"So: A, B, C, D, with NOT-E and NOT-F. Sound right?" If yes,
draw. If no, repeat.
</Step>
</Steps>

You should be ~5-7 minutes in at this point. The whiteboard is
still blank. That's correct.

## The question bank

Memorize this list. You don't ask all of them every time — you
ask the ones that aren't obvious from the prompt.

### About the users

- Who are the users? Consumers, businesses, internal?
- How many? Today and projected in 1-2 years.
- Where are they? One region, several, global?
- How active are they? DAU/MAU ratio, sessions per day.

### About the actions

- What can users *do*? List verbs.
- For each: how often? Once a day or every second?
- What's the **read/write ratio**?
- Is there an obvious **fanout** (one action affects many)?
- Are any actions **strictly real-time** (chat, trading) vs
  **eventual** (notifications, analytics)?

### About scale

- DAU and growth rate.
- Peak QPS (use peak/avg = 3x as a default).
- Data volume per record and total.
- Bandwidth per request and total.
- Concurrent connections, if long-lived.

### About guarantees

- Latency budget for the user-facing path (p50, p99).
- Availability target. **Three nines or four?**
- Durability — can we lose the last second? Hour? Never?
- Consistency — strong, read-your-writes, or eventual?
- Geographic resilience — survive one AZ? One region?

### About constraints

- Budget — order of magnitude.
- Compliance — PCI, HIPAA, GDPR, SOC2?
- Existing systems we must integrate with.
- Team size and on-call capacity.

### About scope

- What is explicitly **not** being designed?
- Auth, payments, search ranking, mobile clients,
  internationalization — name what's out so you don't
  accidentally design it.

## What to do in an interview when answers are missing

The interviewer wants you to **make reasonable defaults and
state them**. Don't stall asking "what should this be?" for every
NFR. Use this default table and let them correct you.

<Cheatsheet>
| Variable | Default to assume | Adjust if... |
|---|---|---|
| DAU | 10M (mid-scale) | Specifically Twitter/Netflix → 100M+ |
| Read/write ratio | 100:1 for content | Chat 1:1; transactional ~3:1 |
| Peak factor | 3x average | Live events 10x; retail 100x |
| p99 latency | 200ms for reads | Trading under 10ms; analytics 1s+ |
| Availability | 99.9% | Money path 99.99%, fun path 99% |
| Consistency | Eventual for reads, strong for writes-to-self | Money strong everywhere |
| Geography | Single region active, multi-AZ | "Global" only if explicitly asked |
| Retention | 1 year hot, indefinite cold | Logs 30 days; messages forever |
</Cheatsheet>

State each one as you make it: "I'll assume 10M DAU and a 100:1
read/write ratio; correct me if that's wrong." That gets a yes/no
in 5 seconds and unblocks the design.

## The two-page output

For real-world projects, the framework produces a short doc:

```
Problem
-------
What the user can do. One paragraph.

Functional requirements
-----------------------
FR1, FR2, FR3 ... (5-7 max)

Non-functional requirements
---------------------------
Scale, latency, availability, durability, consistency,
security, cost.

Out of scope
------------
Things this system will NOT do.

Open questions
--------------
Things we can't yet answer; who needs to answer them; by when.
```

Two pages. Maybe one. If you can't get the requirements onto two
pages, you're describing the implementation, not the
requirements.

<Callout variant="info">
The "open questions" section is the single most useful part. It
forces you to *flag what you don't know* rather than picking an
answer at random. Real architects live in the open questions list.
</Callout>

## How requirements drive design choices

Make the connection explicit; don't make the reviewer guess.

<Compare a="Requirement" b="Design consequence">
<>
"99.99% availability for redirects."

"100M URLs at 10K writes/sec peak."

"p99 < 100ms globally."

"Lose at most 1 second of data."

"Strong consistency on user balance."

"PCI scope minimized."
</>
<>
Multi-AZ replicas; failover < 60s.

Sharded KV store, not a single Postgres.

CDN edge cache; pre-resolve in 200+ POPs.

Synchronous WAL replication, sub-second.

Single-leader Postgres for the wallet, no eventual reads.

Card data never touches our infra; tokenize at the gateway.
</>
</Compare>

Each row of the right column is **forced** by the left. That
chain — "we chose X because requirement Y" — is the audit trail
of the design.

## Common mistakes

### 1. Starting with the design

"Let's put nginx in front of three app servers..." Stop. You
don't know what you're scaling for yet.

### 2. Asking too many questions

Asking 20 questions before drawing anything is the other failure
mode. Five focused questions plus stated defaults is enough.

### 3. Vague NFRs

"It should be fast" / "It should be reliable." These are not
requirements. Quantify or drop them.

### 4. No out-of-scope list

Without one, the design grows until it's everything. Naming
what's out is half the work.

### 5. Treating requirements as static

Requirements drift as you design. If you discover a constraint
mid-design, **go back** and add it to the requirements list,
then check the rest of the design against it. Don't smuggle it
in silently.

## The 60-second version (for warm starts)

When the prompt is familiar — URL shortener, chat, feed — you
can compress this to about 60 seconds:

```
- Restate.
- 5 FRs.
- DAU, R/W ratio, peak.
- Latency target, availability target, consistency posture.
- "Out of scope: <three things>."
- "Proceeding to estimate scale; speak up if any of those are off."
```

Anything longer is being pedantic; anything shorter is skipping
the work.

## In summary

- A repeatable script beats winging it; ask the same 15-20
  questions every time.
- Restate the problem in one sentence before doing anything else.
- Quantify NFRs; vague NFRs aren't useful.
- State your defaults out loud so the interviewer can correct
  them in 5 seconds.
- Always write the "out of scope" list — what you're NOT
  building is half the design.
- The whiteboard stays empty for the first 5-7 minutes. That's
  correct.
