# Sequential consistency

A weakening of linearizability defined by Leslie Lamport in
1979. The system behaves as if all operations executed in
**some** total order, and every process sees that order
preserved for its own operations.

Critically, that order need not match real time. T1 can finish
before T2 begins, and the system can still pick an order where
T2 came first — as long as every process sees the same global
order, and each process's own operations stay in their
issued order.

## Linearizable vs sequential

<Compare a="Linearizable" b="Sequential">
<>
There is a single global order **and** it respects real time.
Once an op completes, every subsequent op sees it.
</>
<>
There is a single global order; per-process order is
preserved within it. Real time doesn't enter the definition.
</>
</Compare>

Concrete example: Client A writes X=1 at 10:00:00, returns
success. Client B reads X at 10:00:05 and sees X=0.

- A linearizable system: **bug**. The write returned before
  the read; the read must see X=1 or later.
- A sequential system: **legal**. The total order can place B's
  read before A's write as long as no process saw a violation
  of its own program order.

In practice that ambiguity is too weird for most applications,
which is why "strong" almost always means linearizable.

## What it's used for

Mostly **hardware memory models**, not application databases.
SPARC's "Total Store Order" is roughly sequential consistency.
Java's volatile + happens-before gives a sequentially
consistent view of marked memory.

For data systems, sequential consistency is rarely a target on
its own — you either go all the way to linearizable, or drop
to causal+ for the geo-distributed scale wins.

## Why it isn't more popular

Two reasons systems skip directly from linearizable to causal:

### 1. The cost is almost the same

Sequential consistency still requires a global agreement on
order — you need consensus or a leader. You've paid most of
the linearizability cost without getting the recency guarantee
clients usually expect.

### 2. The semantics are confusing

"Real time doesn't matter" is hard to explain to a product
team. "If your customer support agent writes the case note and
then asks the user to refresh, the user might still see the
old version — that's intended" doesn't fly.

## A model you'll see in lock-free programming

If you write lock-free C++ or Java, sequential consistency is
the default memory ordering for atomics. Every store/load
happens in some global order that all threads agree on.

The compiler/CPU can reorder operations *within* a thread for
performance, but not across SC atomics. Switching to weaker
orderings (`memory_order_acquire`, `memory_order_release`,
`memory_order_relaxed`) gives the compiler more freedom and
costs you global ordering — common in hot paths.

## How it relates to "linearizable for single key"

Some databases (Cassandra without LWT, DynamoDB strong reads)
claim sequential consistency per-key. That means:

- Two clients reading key K see a consistent timeline for K.
- But there's no cross-key guarantee. K1's update and K2's
  update can appear in different relative orders to different
  readers.

For independent keys this is fine; for keys that imply each
other (account balance + ledger entry), it isn't.

## In summary

- Sequential consistency: a single global order, per-process
  order preserved within it, no real-time guarantee.
- Lamport's original definition; mostly relevant for hardware
  memory models today.
- Costs essentially the same as linearizable in a distributed
  system but doesn't give you recency — so data systems
  usually skip it.
- Useful concept for understanding what linearizability adds
  on top.
