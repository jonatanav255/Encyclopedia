# Chaos engineering

Inject failure in production (or production-like environments)
to **validate that your system actually handles the failures
you designed for**.

Netflix coined the term with Chaos Monkey (2011), which
randomly killed EC2 instances during business hours to force
the system to tolerate them. The discipline has grown into a
field with tooling, methodology, and case studies.

## Why this is necessary

You designed for "any single AZ can fail." Did you verify
that the system actually behaves correctly when an AZ fails?

Without testing, no. The fallback paths you wrote 6 months
ago — circuit breakers, retries, graceful degradation —
have probably bit-rotted. Production traffic exposes assumptions
the design didn't.

Chaos engineering is the deliberate discipline of testing
those assumptions.

## The principles

Adapted from the Principles of Chaos Engineering:

<Steps>
<Step n={1} title="Define 'normal' (steady state).">
What does the system look like when healthy? Specific
metrics: error rate under 0.1%, p95 latency under 500ms,
order success rate above 99%.
</Step>
<Step n={2} title="Hypothesize that chaos won't change it.">
"Killing one DB read replica shouldn't change the steady
state." Specific, testable.
</Step>
<Step n={3} title="Inject real-world failures.">
Kill a replica. Add 500ms of network latency. Drop 1% of
packets. Fill a disk.
</Step>
<Step n={4} title="Measure.">
Did the steady state hold? If yes, the hypothesis is
validated. If no, you've found a real problem.
</Step>
<Step n={5} title="Fix or accept.">
Either fix the system or update your understanding of what
"survives" means.
</Step>
</Steps>

## Common failure modes to inject

<Cheatsheet>
| Layer | Examples |
|---|---|
| Compute | Kill instance, kill pod, OOM |
| Network | Latency, packet loss, partition, DNS failure |
| Storage | Slow disk, fill disk, kill replica, force failover |
| Dependency | Make downstream slow, return 503, return malformed data |
| Time | Clock skew, NTP outage |
| Resource | CPU saturation, memory pressure |
| Config | Bad config rollout, missing secret |
| Region | Take a whole region offline |
</Cheatsheet>

Each surface a different class of bug. Pod-kill tests deploy
+ failover. Network latency tests timeout configurations.
Region-down tests multi-region failover.

## Tooling

- **Chaos Monkey / Chaos Engineering Suite** (Netflix
  original).
- **Gremlin** — commercial chaos platform.
- **Litmus** — CNCF graduated, K8s-native chaos.
- **Chaos Mesh** — also K8s-native.
- **PowerfulSeal** — open source, infrastructure-level.
- **AWS Fault Injection Service** — cloud-native fault
  injection.
- **Pumba** — Docker-level chaos.

Most start with a simple kill-pod cron job and graduate to
tools as the discipline matures.

## Blast radius

A test in production can cause an outage if it goes wrong.
Control blast radius:

- **Start in staging.** Always.
- **In production, start with the smallest possible scope**:
  one pod, one host, one tenant.
- **Time-bound the experiment** (don't leave failures running
  forever).
- **Have a kill switch** — one button to stop the chaos.
- **Run during business hours** — engineers are watching, on
  call, can respond.

Chaos engineering that takes down a customer is the wrong
kind of test.

## Game days

A scheduled, planned chaos exercise. The team gathers, picks
a scenario, injects, watches the system respond, debriefs.

```
Game Day: "What happens if the auth service is down for 30 minutes?"

Plan:
  - Inject 500 errors from auth service.
  - Watch downstream services.
  - Watch user-facing impact.
  - Practice incident response.

Hypothesis: cached sessions allow users to continue browsing;
new logins fail with a clear error.

Result: cached sessions worked, but the page footer was
broken because it lazy-loaded auth info. Bug filed.
```

Game days are training wheels for production chaos.
Eventually you graduate to ambient chaos (automated, daily).

## Steady-state metrics

You can't test chaos without baseline metrics:

- **Business metrics**: signups, orders, revenue per minute.
- **System metrics**: error rate, p99 latency.
- **User experience metrics**: time-to-first-byte, fail rate.

If injecting failure doesn't move these metrics, the system
truly survives. If it does, you've found a problem.

## Common findings

What chaos engineering reliably uncovers:

- **Timeouts too long**: a "1-second" call actually takes 30s
  because the timeout was unset.
- **Missing fallbacks**: an optional dependency takes down a
  required path.
- **Cascading retries**: clients retry storms when a downstream
  is slow.
- **Resource exhaustion**: pool sizes too small or large for
  the actual workload.
- **Monitoring gaps**: an outage is invisible because the
  alerting was based on the wrong signal.
- **Documentation lies**: the runbook says "click X to
  failover" but X doesn't exist anymore.

## Production vs staging

Many failures only show up in production:

- Real traffic shape (not synthetic load).
- Real datasets (different from staging).
- Real dependencies (third-party APIs).
- Real scale.

Staging chaos catches the easy bugs. Production chaos catches
the rest. Both are necessary; production is where the
real value is.

## Cultural prerequisites

Chaos engineering in a culture that punishes failures is
malpractice. You can't ask people to introduce failure into
production if their bonus depends on uptime.

Prerequisites:

- **Blameless postmortems**. Failures are learning, not
  punishment.
- **Error budgets**. The team has explicit room to spend on
  reliability work.
- **Observability**. You can detect and diagnose failures
  quickly.
- **Practiced incident response**. The team knows how to
  respond when chaos finds something.

Without these, chaos engineering is reckless.

## In summary

- Chaos engineering: deliberately inject failure to validate
  resilience assumptions.
- Hypothesis-driven: define steady state, hypothesize about
  failure, inject, measure.
- Common injections: kill instances, network latency, slow
  dependencies, region outage.
- Game days as training wheels; ambient daily chaos as
  graduation.
- Control blast radius — start small, time-bound, have a kill
  switch.
- Requires a culture of blameless learning and observability;
  without those, it's malpractice.
