# Lamport clocks

How do you order events across machines when their physical
clocks disagree? Leslie Lamport's 1978 answer: don't try.
Order by **happens-before**, which doesn't need clocks.

A Lamport timestamp is a simple integer counter that captures
causality. Two rules:

1. Every event on a process increments the local counter.
2. Every message sent carries the sender's counter. The
   receiver sets its counter to `max(local, received) + 1`.

That's it. Two events `a` and `b` with `LC(a) < LC(b)` mean
`a` *might* have caused `b`. Two events with `LC(a) == LC(b)`
or `LC(a) > LC(b)` mean `a` *didn't* cause `b` (or it's the
reverse).

## What Lamport clocks give you

A **partial order** consistent with happens-before. Given any
two events, you can tell:

- Whether one happened before the other (causally), OR
- That they're concurrent.

Use this for:

- Ordering events in a distributed log.
- Detecting concurrent updates that need conflict resolution.
- Determining causality in event sourcing.

## What they don't give you

### A total order

Two unrelated events can have the same Lamport timestamp.
Lamport's rule for breaking ties: append the process ID. Now
you have a total order: `(LC, pid)`. But it's an **arbitrary**
total order — not real time, not happens-before, just
deterministic.

### Concurrency detection

A Lamport timestamp `LC(a) < LC(b)` doesn't tell you whether
`a` caused `b` or they're just unrelated and one ticked
faster. For that, you need **vector clocks**.

## Example walkthrough

Three processes P1, P2, P3. Each starts at LC=0.

```
P1: event A (LC=1)
P1: send msg to P2, tag with LC=2
P2: receive msg, LC = max(0,2)+1 = 3, event B (LC=3)
P3: event C (LC=1)
P3: send msg to P2, tag with LC=2
P2: receive msg, LC = max(3,2)+1 = 4, event D (LC=4)
```

- LC(A)=1, LC(B)=3, LC(C)=1, LC(D)=4.
- A happens before B (LC and the message say so).
- A and C have LC=1 — they're concurrent. Lamport correctly
  says "not ordered" but tie-break by pid would say (1,
  P1) < (1, P3) → A first. That ordering is **invented**, not
  real.

## Compared to physical clocks

| Property | Wall clock | Lamport clock |
|---|---|---|
| Captures real time | Yes | No |
| Survives clock skew | No | Yes — there's no clock to skew |
| Detects causality | No (can't trust skew) | Yes |
| Total order | Yes | Only with tie-break |
| Cost per event | Free | One integer compare + max |

## Where Lamport clocks are used

- **Distributed databases** for ordering writes when leaders
  fail over. Cassandra's commit log ordering.
- **Event sourcing systems** for causal ordering of events.
- **Distributed mutual exclusion** (Lamport's bakery
  algorithm).
- **Operational transformation** in collaborative editors
  (though CRDTs have largely replaced this).

## Limitations

- **Only captures one direction.** "A happens before B" might
  mean "A directly caused B" or "A and B are on completely
  different branches of execution that don't actually
  influence each other." Vector clocks fix this.
- **Counters grow unbounded.** Not usually a problem in
  practice (64-bit integers last ~600 years at billion
  events/sec).
- **Requires every message to carry the timestamp.** Cheap but
  not free.

## Lamport clocks → vector clocks → HLC

Three steps in the evolution of distributed clocks:

<Steps>
<Step n={1} title="Lamport clock — single integer.">
Captures partial order. Can't distinguish "happens before"
from "happened at the same logical step."
</Step>
<Step n={2} title="Vector clock — one integer per node.">
Captures the full happens-before relation. Larger; can detect
true concurrency.
</Step>
<Step n={3} title="Hybrid logical clock (HLC) — physical + logical.">
Bounded size; aligned with wall-clock time so humans can
debug. Used in CockroachDB, MongoDB.
</Step>
</Steps>

For most modern systems, HLC is the right default. Lamport
clocks remain pedagogically important and are simpler to
implement when you don't need to distinguish concurrency.

## In summary

- Lamport clocks: a single integer per process, incremented on
  events and synchronized via message piggyback.
- Captures the happens-before partial order without physical
  clocks.
- Can establish a total order by tie-breaking with process ID,
  but the order is arbitrary beyond causality.
- Doesn't detect concurrency — vector clocks needed for that.
- Foundation for vector clocks and HLC.
