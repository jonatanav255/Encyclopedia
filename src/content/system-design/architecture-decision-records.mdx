# Architecture decision records (ADRs)

A lightweight markdown record of an architecturally significant
decision: **what was chosen, what was rejected, and why**.
Saved next to the code (typically `docs/adr/`), versioned in
git, immutable once accepted.

The discipline is decades old (Michael Nygard's 2011 post
popularized it). The payoff: six months later, when someone
asks "why did we pick X over Y?", there's an answer.

## What goes in an ADR

A typical ADR is ~1 page. The standard sections:

```
# ADR-0042: Use Postgres for event log instead of Kafka

Date: 2026-05-15
Status: Accepted
Deciders: backend team

## Context
We need to persist domain events for replay and downstream
consumers. Our scale is ~1K events/sec.

## Decision
Use a Postgres `events` table with an outbox pattern. Defer
Kafka until we hit ~10K events/sec sustained.

## Consequences
- (+) No new infrastructure to operate.
- (+) ACID with the main domain writes.
- (+) Easy to query and debug.
- (−) Limits ingest throughput to Postgres' ceiling.
- (−) Future migration to Kafka required if scale grows.
- (−) Downstream consumers must use polling not push.

## Alternatives considered
- Kafka now: rejected. Operating Kafka for 1K events/sec is
  overkill at our team size.
- Redis Streams: rejected. Non-durable across restart.
```

That's it. Tight, dated, immutable.

## What makes ADRs different from "design docs"

A design doc describes a system; an ADR records a **decision**.

- Design doc: "Here's the architecture of the order service."
- ADR: "We're using Postgres for the event log."

You can have a design doc that contains many decisions, but
each significant decision should have its own ADR for
discoverability.

## When to write one

Not every commit needs an ADR. Write one when:

- A choice has **long-term consequences**.
- It would be hard to reverse (e.g., picking a database).
- Team members might forget the reasoning.
- The trade-offs are non-obvious.

Concrete triggers:

- Choosing a new data store, message broker, RPC framework.
- Changing the deploy topology.
- Introducing a new architectural pattern (CQRS, event
  sourcing, sharding).
- Standardizing on a library across the codebase.
- Naming conventions for APIs / events.

Don't write one for: bug fixes, refactors, minor library
upgrades.

## The status field

A small but important detail. Possible values:

- **Proposed** — under discussion.
- **Accepted** — current standard.
- **Deprecated** — superseded by a newer ADR; reference the
  successor.
- **Rejected** — discussed and ruled out (still useful — explains
  why a tempting option wasn't chosen).

Old ADRs **stay in the repo** with status updates. They're
the historical record, not a stack you keep popping.

## Numbering and discoverability

Sequential numbers (`0001-...`, `0002-...`) so chronology is
visible. Each ADR has a stable filename so it can be linked
externally.

```
docs/adr/
├── 0001-record-architecture-decisions.md  (an ADR about ADRs themselves)
├── 0002-use-postgres-for-primary-db.md
├── 0003-eventually-consistent-search-index.md
└── 0042-use-postgres-for-event-log.md
```

For larger orgs, ADRs per service or per domain (`docs/adr/`
in each repo).

## What ADRs are not

### Not implementation details

How exactly the outbox pattern is implemented goes in the
code or a runbook. The ADR explains "we use outbox vs CDC"
at the level of choice.

### Not a permission slip

ADRs document decisions; they don't authorize you to make
them. Org culture decides who can write an accepted ADR.

### Not a substitute for design review

Big decisions go through synchronous review (design doc,
team meeting). The ADR records the outcome.

## Common ADR mistakes

### Writing one only for the "good" decision

"We chose X." But the decision was 80% X, 20% Y, and the
reader wonders about Y forever. Always list the rejected
options with brief reasons.

### Writing them after the fact

Three months after the decision is made, someone writes the
ADR. The context drift makes it half-accurate. Write at
decision time.

### Updating instead of superseding

Don't edit an accepted ADR to reflect a new decision. Mark
it deprecated and write a new one pointing back. Otherwise
git history becomes the only place to see what was once
decided.

### Bikeshedding the template

Don't spend a month picking the perfect ADR template. The
classic Nygard format is fine. Write the first one in 20
minutes.

## Tooling

Light tools that help:

- **adr-tools** (CLI) — creates new ADRs from template,
  manages indexes.
- **MADR template** — modern fork with sections for "decision
  drivers" and "consequences" structured.
- **MkDocs / Docusaurus** — for nice rendering of the ADR
  directory.

But ADRs are markdown files in git. Tooling is optional.

## In summary

- ADR: a short markdown file recording an architectural
  decision, including alternatives considered.
- Stored in the repo, versioned, immutable once accepted.
- Each significant decision gets one; trivial changes don't.
- Status field: proposed / accepted / deprecated / rejected.
- Superseded ADRs stay in the repo with pointers to
  successors.
- Tools optional; the discipline is what matters.
