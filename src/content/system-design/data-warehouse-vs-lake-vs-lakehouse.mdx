# Warehouse vs lake vs lakehouse

Three storage paradigms for analytical data, converging
toward each other over time.

- **Data warehouse**: schema-on-write, structured, fast SQL.
- **Data lake**: schema-on-read, unstructured / semi-structured,
  cheap.
- **Lakehouse**: structure on top of object storage, combining
  warehouse semantics with lake economics.

The 2020s have largely been about the lakehouse rising.

## Data warehouse

The classic enterprise analytics platform. Schema is defined
upfront; data is loaded into structured tables; SQL queries
are fast.

Examples: Snowflake, Google BigQuery, Amazon Redshift, Teradata,
Vertica.

Pros:

- Excellent SQL performance via columnar storage and indexes.
- Schema enforcement prevents garbage in.
- Mature governance, security, lineage tools.
- BI tools (Looker, Tableau) work out of the box.

Cons:

- Storage cost is higher than object storage.
- Schema rigidity hurts for evolving data shapes.
- Loading unstructured data (PDFs, images) is painful.
- Vendor lock-in via proprietary file formats.

## Data lake

A pile of files in object storage (S3, ADLS, GCS), in
various formats (parquet, avro, json, csv, even raw logs).
Schema applied at query time.

Examples: AWS S3 + Athena, Azure Data Lake Storage, HDFS.

Pros:

- Cheap (object storage).
- Stores anything — structured, semi-structured, unstructured.
- No upfront schema commitment; flexible.
- Compute decoupled from storage.

Cons:

- "Data swamp" risk: without governance, becomes unsearchable.
- Slow queries on raw files (no indexes).
- Schema drift goes unnoticed.
- ACID semantics non-trivial.

## Lakehouse

A layer on top of object storage that provides
warehouse-like semantics: ACID transactions, schema
enforcement, time travel, indexes.

The data still lives as files in S3, but a metadata layer
makes it behave like a warehouse.

Implementations:

- **Delta Lake** (Databricks).
- **Apache Iceberg** (Netflix-origin, now widely adopted).
- **Apache Hudi** (Uber-origin).

Pros:

- Object-storage cost.
- ACID transactions, schema enforcement, time travel,
  upserts, deletes.
- Multiple compute engines (Spark, Trino, Snowflake, Athena)
  can query the same data.
- Open file format (parquet), no vendor lock-in.

Cons:

- More moving parts than a pure warehouse.
- Newer; tooling still maturing.
- Compute still expensive.

## How they compare

<Cheatsheet>
| Property | Warehouse | Lake | Lakehouse |
|---|---|---|---|
| Storage cost | High ($) | Low ($) | Low ($) |
| Compute cost | High | Variable | Variable |
| Schema | On write | On read | On write (enforced via metadata) |
| ACID | Yes | No | Yes |
| Time travel | Limited | No | Yes |
| BI tool support | Excellent | OK | Good |
| Multi-engine | Locked to one | Yes | Yes |
| Schema enforcement | Strict | Loose | Strict |
| Vendor lock-in | High | None | Low |
| Unstructured data | Bad | Good | Limited |
</Cheatsheet>

## Decision

For most teams in 2026:

- **Start simple**: a single managed warehouse (Snowflake,
  BigQuery) covers everything for the first few years.
- **Add a lake** if you have unstructured / semi-structured
  data the warehouse can't handle.
- **Move to lakehouse** when storage cost becomes a problem
  or you want multi-engine flexibility.

Going straight to lakehouse can work, but warrants extra
engineering capacity.

## Open table formats

Iceberg, Delta, Hudi all do similar things differently:

- **Iceberg**: open spec, broad engine support (Snowflake,
  Trino, Spark, Flink, Dremio). Strong on hidden partitioning,
  schema evolution.
- **Delta Lake**: open format but Databricks-driven. Tight
  integration with Spark; supports Z-ordering.
- **Hudi**: best for write-heavy / upsert workloads; built
  for CDC ingestion.

If you're choosing a new lakehouse format in 2026, **Iceberg**
has the most open ecosystem and the most adoption from
warehouses (BigQuery, Snowflake, Redshift all support it).

## Common architecture

A modern data platform often looks like:

```
sources (DBs, APIs, events)
  ▼
extract + load (Fivetran, Airbyte, custom)
  ▼
landing zone (object storage, raw)
  ▼
ELT / dbt models (Iceberg / Delta tables in object storage)
  ▼
compute engines:
   - dashboards (Snowflake or Trino)
   - ML training (Spark)
   - ad-hoc queries (Athena, Trino)
```

The lakehouse data is the source of truth; multiple engines
read it.

## Common antipatterns

### Data swamp

Lake without governance: thousands of CSV files with no
schema, no documentation. Nobody knows what's where.

Fix: catalog (DataHub, Glue catalog), naming conventions,
ownership.

### Multiple sources of truth

Different teams maintain their own warehouses; the same
"customer count" differs across dashboards. Trust collapses.

Fix: single canonical source per metric; downstream consumers
use that, not their own derivation.

### Lakehouse overhead without benefit

A small dataset that fits in Postgres: putting it in
Iceberg for "modern architecture" buys you 10× the operational
overhead for no benefit.

Fix: scale-match. Use warehouses or even Postgres for
small datasets.

## In summary

- Warehouse: schema-on-write, expensive, BI-friendly,
  vendor-locked.
- Lake: cheap, flexible, easy to fill with garbage.
- Lakehouse: lake economics + warehouse semantics via
  Iceberg / Delta / Hudi on object storage.
- 2020s direction: lakehouse with Iceberg as the open
  standard.
- Start with a warehouse; add lake/lakehouse as scale
  demands.
- Watch for: data swamps, multiple sources of truth,
  premature complexity.
