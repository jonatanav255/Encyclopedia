# Layered architecture

The default architecture for most enterprise applications.
Code is organized into horizontal layers, each calling only
the layer below.

```
┌────────────────────────────────┐
│ Presentation (controllers/UI)  │
├────────────────────────────────┤
│ Application (use cases)        │
├────────────────────────────────┤
│ Domain (business logic)        │
├────────────────────────────────┤
│ Persistence (repositories)     │
├────────────────────────────────┤
│ Database                       │
└────────────────────────────────┘
```

Rails, Django, Spring, ASP.NET MVC — all variations on
layered architecture. It's "boring" because it works.

## What each layer does

- **Presentation** — HTTP routing, request parsing, response
  shaping. Knows about HTTP/JSON; nothing else.
- **Application** — orchestrates business use cases. "Place an
  order" calls the domain to validate, the persistence layer
  to save, the mailer to confirm. Thin.
- **Domain** — the business rules. Entities, invariants,
  workflows. Pure logic; doesn't know about HTTP or DB.
- **Persistence** — read/write the database. Repository
  pattern, ORM, raw SQL — implementation detail.

## Why people pick it

- **Easy to find things.** New engineer asks "where does X
  live?" — controllers, services, models. They've seen this
  shape before.
- **Easy to test.** Each layer has clear inputs/outputs.
- **Easy to scale teams.** Different people can work on
  different layers.
- **Matches the frameworks.** Most web frameworks assume
  layered.

## When it stops working

- **Cross-cutting concerns proliferate.** Auth, logging,
  tenancy live everywhere; layers stop helping.
- **Business logic leaks into other layers.** Validation logic
  in controllers; queries in domain code. Now layers are
  smudged.
- **Performance crosses layers.** A query that needs hand-written
  joins ends up bypassing the persistence layer with raw SQL.
- **Multiple inbound channels.** HTTP + gRPC + queue
  consumer — all duplicate the same presentation logic.

When these add up, hexagonal-architecture or vertical-slice
patterns start to look attractive.

## Strict vs relaxed layering

<Compare a="Strict layering" b="Relaxed layering">
<>
Each layer calls **only the layer directly below**. To use
persistence from presentation, you must go through application
+ domain. Maximum decoupling, maximum verbosity.
</>
<>
Each layer can call **any layer below**, skipping levels.
Easier to write; layers begin to merge into each other over
time.
</>
</Compare>

Most real systems are relaxed. Strict is too painful for the
99% case.

## Variations

### Three-layer

Presentation / Business / Data. Skips the "application" layer.
Adequate for simple CRUD; too coarse for complex domains.

### Five-layer (DDD style)

Adds "infrastructure" alongside persistence (caches, queues,
external clients). Domain stays pure; everything stateful is
infrastructure.

### Vertical slices

Instead of layers, organize by **feature**. Each feature has
its own controller, service, and repository in one folder.
Easier to understand a single feature; harder to find common
patterns. Popular in DDD-style projects.

## A subtle pitfall: anaemic domain

Layered tends to push behavior **out** of entities and into
"services":

```ts
// Anaemic
class Order { items: Item[]; status: string }
class OrderService {
  cancel(order: Order) {
    if (order.status === 'shipped') throw new Error();
    order.status = 'cancelled';
  }
}
```

The "domain" has no behavior; it's a data bag. The "service"
holds all the logic. You've got functions and structs, not
objects.

This is fine if you embrace it (procedural code is OK), but
it's usually accidental — the result of layered habits
pushing all logic to services.

A richer domain:

```ts
class Order {
  cancel() {
    if (this.status === 'shipped') throw new Error();
    this.status = 'cancelled';
  }
}
```

The service becomes a thin orchestrator: load, call,
persist.

## In summary

- Layered architecture: presentation → application → domain →
  persistence → DB.
- Default for most enterprise apps; works because everyone
  knows it.
- Fails when cross-cutting concerns, multiple inbound channels,
  or complex business logic push you past its assumptions.
- Watch for anaemic domain — pushing all behavior into
  "services" is a layered antipattern.
- For projects beyond CRUD, consider hexagonal or vertical
  slices.
