# Functional vs non-functional requirements

Every system has two kinds of requirements. **Functional** ones
describe what the system *does*. **Non-functional** ones describe
*how well* it does it. The functional ones look like the
interesting part — they're the product. The non-functional ones
are what actually drives the architecture.

In an interview or a design doc, separating them in the first
five minutes is the difference between a defensible design and a
sketch.

## What each is

<Compare a="Functional requirements (FR)" b="Non-functional requirements (NFR)">
<>
**Verbs and nouns.** "Users can post tweets," "admins can
delete accounts," "payments charge a card and refund within
24 hours." Each FR maps to an API, a screen, a workflow. Pass/fail
is testable with a unit or integration test.
</>
<>
**Properties of the whole system.** Latency, throughput,
availability, durability, consistency, security, cost,
maintainability. Pass/fail is testable only with load tests,
chaos tests, or measurement over time.
</>
</Compare>

A useful sniff test: an FR fails if a feature is missing; an NFR
fails when the feature is present but **the experience is wrong**
— too slow, too lossy, too unreliable, too expensive.

## Why NFRs drive architecture more than FRs

Almost any FR can be implemented many ways. A "post a tweet"
endpoint is 30 lines of code against any database. Add the NFRs —
"500K writes/sec, 99.99% availability, 100ms p99, durable across
region failure" — and 95% of the implementation options disappear.

The NFRs are the **constraint set**. Architecture is what
satisfies the constraint set. The FRs are mostly the wiring.

<Callout variant="info">
Interview hint: if a candidate jumps straight to drawing services
without asking about scale, latency, or availability, they're
designing in a vacuum. Always ask for NFRs first; the design
follows from them.
</Callout>

## The canonical NFR categories

The list isn't fixed, but these eight cover most of the space.

<Cheatsheet>
| NFR | Question it answers | Typical metric |
|---|---|---|
| Performance | How fast? | p50/p95/p99 latency, throughput (QPS) |
| Availability | How often is it up? | "Three nines" / "four nines" uptime |
| Durability | How safely is data kept? | 11 nines (S3), data-loss probability/year |
| Consistency | How fresh are reads? | Strong / read-your-writes / eventual |
| Scalability | How does it cope with growth? | "10x users without rewrite" |
| Security | Who can do what? | Auth model, threat surface, compliance scope |
| Maintainability | How fast can we change it? | Deploy frequency, MTTR |
| Cost | What does it cost to run? | $ per user, $ per transaction |
</Cheatsheet>

Other NFRs people add: **observability** (can we see what's
happening?), **portability** (can we move clouds?),
**accessibility** (works for assistive tech?), **localization**
(works in N languages and currencies?).

## Worked example: URL shortener

Junior version of the requirements:

- "Users can shorten a long URL."
- "Users can visit the short URL and get redirected."

That's the FRs. Now NFRs:

```
- 100M URLs/year created, 10:1 read:write at peak.
- p99 redirect latency < 100ms globally.
- 99.99% availability for redirects (clicks are the money path).
- Durable: a shortened URL must never resolve to nothing.
- Strong consistency: the shortened URL must work the moment
  the API returns success.
- Cost ceiling: < $X/month at this scale.
```

With just the FRs, "POST /shorten + GET /:code against any DB"
is acceptable. With the NFRs added, you immediately need: a
distributed key-value store sharded by code, a CDN-cached
redirect path, a multi-region deployment, and an unguessable code
generator. Same product. Completely different architecture.

## The trap of "obvious" NFRs

The most expensive architectural mistakes come from NFRs that
**weren't written down because they felt obvious**:

<Steps>
<Step n={1} title="Availability">
"Of course it has to be up." But how many nines? 99% is 3.65
days/year of downtime. 99.99% is 52 minutes/year. The
architecture for each is wildly different.
</Step>
<Step n={2} title="Data loss tolerance">
"Of course we can't lose data." But can you lose the last 1
second? The last 5 minutes? The first answer requires
synchronous replication; the second allows async backups.
</Step>
<Step n={3} title="Latency expectation">
"Should feel fast." Define it. 50ms p99? 500ms p99? The
caching, indexing, and replication choices flip on that
number.
</Step>
<Step n={4} title="Geographic scope">
"It should work globally." Multi-region active-active is 10x
the complexity of single-region. Most products don't need it
on day one but assume they do.
</Step>
</Steps>

If you don't extract these as explicit numbers, you'll over- or
under-build by 10x.

## Quantifying NFRs

Every NFR worth keeping is **measurable**. The shape:

```
"<property> shall be <number> as measured by <method>
under <conditions>."
```

Examples:

- "p99 redirect latency shall be < 100ms as measured by RUM
  under peak load (50K QPS)."
- "Availability shall be ≥ 99.99% as measured by external
  blackbox probes, monthly."
- "Durability shall be such that the probability of losing any
  byte in a 12-month window is < 10⁻¹¹."

A bare "should be fast" is not an NFR; it's a feeling.

## FRs and NFRs interact

Some FRs are only **achievable** with the right NFR posture:

- "Show a personalized feed for 100M users" requires NFRs on
  precomputation latency and storage cost.
- "Allow users to undo within 30 seconds" requires NFRs on
  event-log durability and replay.
- "Show real-time scores during a live match" requires NFRs on
  fanout latency and connection scaling.

Skipping the NFR analysis means promising FRs you can't ship.

## Where each one lives in the design

<Compare a="What FRs affect" b="What NFRs affect">
<>
APIs, data model, business logic, screens, workflows, role
definitions, error semantics. Mostly product-shaped.
</>
<>
Replication topology, caching layout, database choice, region
strategy, deploy frequency, on-call burden, hardware budget.
Mostly infrastructure-shaped.
</>
</Compare>

You write FRs to **explain the product**. You write NFRs to
**constrain the architecture**.

## A practical template

Use this skeleton in any design doc or interview:

```
Functional requirements
-----------------------
- FR1: Users can <action>.
- FR2: Admins can <action>.
- FR3: System emits <event> when <trigger>.

Non-functional requirements
---------------------------
- Scale:        <DAU>, <QPS read>, <QPS write>, <storage/year>
- Latency:      <p99 target> for <operation>
- Availability: <nines>; failure tolerance: <region/AZ/node>
- Durability:   max acceptable data loss = <RPO>
- Consistency:  <per-operation guarantees>
- Security:     <auth model>, <compliance: PCI/HIPAA/GDPR>
- Cost ceiling: <$ per month at projected scale>

Out of scope
------------
- <things explicitly NOT being built>
```

Three lists, written **before** drawing boxes. Reviewers can
challenge them. The design must satisfy them.

## In summary

- FRs say what the system does; NFRs say how well.
- NFRs drive architecture far more than FRs do.
- The biggest design mistakes come from NFRs nobody wrote down.
- Quantify every NFR — bare adjectives ("fast," "reliable") are
  not requirements.
- Extract both lists in the first five minutes of any system
  design exercise. The design follows from them, not the other
  way around.
