# Secrets management

A **secret** is anything that, if disclosed, lets an attacker
impersonate you, read your data, or take destructive action.
Database passwords, API keys, signing keys, OAuth client
secrets, encryption keys. Everything else (config, hostnames,
feature flags) is not a secret and shouldn't ride in the same
system.

The architecture question is: how do secrets get from a vault to
a workload **safely**, get **rotated** routinely, and **never**
end up in source control, log files, or container images?

There's a separate, smaller security page on the developer
workflow side. This page is the architecture-level view: how
storage, distribution, identity, and rotation fit together.

## The lifecycle

<Cheatsheet>
| Phase | What happens | Where it can leak |
|---|---|---|
| Generation | A secret is created (random, signed, derived) | Weak entropy, predictable patterns |
| Storage | Encrypted at rest in a secret store | Backups, snapshots, replicas |
| Distribution | Delivered to workloads at startup or on demand | Env vars in process listing, leaked logs |
| Use | Consumed by code | Stack traces, log messages, error reporters |
| Rotation | Replaced periodically or on incident | Stale copies in caches, configs |
| Revocation | Old secret invalidated | Long-lived sessions still authorized |
| Audit | Every access logged | Audit log itself becomes a target |

</Cheatsheet>

Every step has a failure mode. A "secrets management" practice
is the set of policies that minimize each.

## Storage: dedicated secret stores

Production secrets live in a **purpose-built store**, not in a
config file in S3, not in a git-encrypted YAML, not in
environment variables baked into a container image.

<Cheatsheet>
| System | Strength |
|---|---|
| HashiCorp Vault | Cloud-agnostic, dynamic secrets (per-request DB creds), best-in-class policy |
| AWS Secrets Manager | Tight IAM integration, automatic rotation lambdas, KMS-encrypted |
| AWS Parameter Store | Cheaper, simpler, supports both plain and secure params |
| GCP Secret Manager | IAM-integrated, automatic versioning, CMEK |
| Azure Key Vault | IAM-integrated, FIPS-compliant HSM tier |
| Kubernetes Secrets | **Not encryption by default** — base64 ≠ encryption. Pair with KMS-backed etcd encryption |

</Cheatsheet>

Plain Kubernetes Secrets are stored base64-encoded in etcd. If
etcd isn't encrypted at rest with a KMS-backed key, anyone who
reads etcd reads every secret. Most managed Kubernetes platforms
now enable KMS encryption by default, but verify.

## Identity: who can read what

The store is only as good as its access policy. The discipline:

<Steps>
<Step n={1} title="Workload identity, not shared keys.">
The `payment-service` workload has an identity (IAM role, SPIFFE
ID, service account) that's allowed to read its own secrets and
nothing else.
</Step>
<Step n={2} title="No shared production identity.">
The fastest path to a breach: every service uses the same key
or role. One leak compromises everything.
</Step>
<Step n={3} title="Per-environment isolation.">
Staging and production secrets are different values **in
different stores** with different access roles. Compromising one
doesn't compromise the other.
</Step>
<Step n={4} title="Per-service scope.">
`payment-service` doesn't need access to `analytics-warehouse`
credentials. Least privilege at the secret level.
</Step>
<Step n={5} title="Audit every read.">
Who fetched which secret when. Use this in postmortems and to
detect unauthorized access.
</Step>
</Steps>

The store enforces this — IAM in AWS, Vault policies, Kubernetes
RBAC. The cost is configuration; the payoff is blast-radius
limitation.

## Distribution: how secrets reach the workload

Three patterns:

<Cheatsheet>
| Pattern | How | Trade-offs |
|---|---|---|
| Env vars at start | Orchestrator reads secret store, sets `DATABASE_URL`, etc. | Easy; visible in `/proc/<pid>/environ`; needs redeploy to rotate |
| Mounted file | Secret written to a tmpfs mount; app reads file on demand | Avoids env var leakage; supports hot reload; needs FS watcher |
| In-process fetch | App library calls the secret store at runtime | Most dynamic; adds startup latency; depends on store availability |

</Cheatsheet>

The most operationally robust pattern: **a sidecar or agent**
fetches secrets from the store and writes them to a tmpfs
volume the app reads. Vault Agent and the AWS Secrets and
Configuration Provider (ASCP) do exactly this. Rotation
becomes "agent re-fetches; app re-reads file."

## Dynamic secrets

Vault's strongest feature, increasingly common elsewhere:
secrets are **generated on demand** with short TTL.

```
service requests DB credentials
  ↓
Vault generates a unique Postgres user + password
  ↓
Vault grants the role; returns credentials with TTL=1h
  ↓
service uses them
  ↓
TTL expires; Vault drops the Postgres user
```

Properties:

- Every workload instance has a unique credential.
- Compromise of one credential ages out in an hour.
- A leaked credential in logs is useless after the TTL.
- Per-instance audit trail in the DB itself ("who connected as
  user_abc123?").

The downside: more moving parts. Worth it for high-sensitivity
data stores; overkill for low-stakes secrets.

## Rotation

Every secret must be rotatable. Rotate routinely (quarterly is
typical), and **always after a suspected incident**.

Two flavors:

- **Static rotation** — generate a new value, update the store,
  redeploy or signal apps to re-fetch. Old value continues
  working briefly to avoid downtime, then gets revoked. Both
  values must coexist briefly.
- **Dynamic rotation** — already happening on every TTL. There's
  nothing to "rotate" in the traditional sense.

Some stores rotate automatically. AWS Secrets Manager's
"rotation Lambda" can change an RDS password and update the
secret atomically. Similar features exist in Vault and others.

Specific cases that need extra care:

- **JWT signing keys.** Already-issued JWTs need to keep
  validating until they expire. Maintain a **keyring** — sign
  with the newest, verify against any in the ring.
- **TLS certificates.** Same pattern, longer windows.
- **OAuth client secrets.** Coordinate with the provider's
  rotation flow; not all support overlap.

## What never goes in source

A short list of things that **must not** reach git:

- Production passwords or API keys.
- Internal private keys.
- Anything matching the format `[A-Za-z0-9_-]{20,}` that you
  can't identify — assume secret, check before commit.
- `.env` files with real values (commit `.env.example` with
  placeholders).

Detection in depth:

- **Pre-commit hooks** (gitleaks, trufflehog) — block the leak
  before push.
- **CI scans** — catch what slips past the hook.
- **GitHub Secret Scanning** — built-in on public repos; flags
  known secret patterns.
- **External honeytokens** — fake credentials whose use lights
  up an alert. If anyone tries them, you know.

When a real secret is committed:

<Steps>
<Step n={1} title="Rotate immediately.">
Assume compromised. The rotation is the only thing that
matters — the cleanup of git history is secondary.
</Step>
<Step n={2} title="Don't rely on history cleanup.">
Even after `git filter-repo`, mirrors, forks, scrapers, and
backups still have it. Rotation is non-negotiable.
</Step>
<Step n={3} title="Postmortem.">
What process should have caught this? Add the missing scan.
</Step>
</Steps>

## Logs and error reporting

The classic accidental leak: an exception with a stack trace
that includes the DB URL containing the password. The stack
trace goes to Sentry, to logs, to email alerts.

Defenses:

- **Logger redaction**: configure the logger to strip known
  secret-bearing fields (`Authorization`, `cookie`,
  `password`, etc.).
- **Error sanitization**: error reporters (Sentry, Rollbar)
  support scrubbing rules. Use them.
- **Connection-string handling**: don't pass `DATABASE_URL`
  through anywhere it might be logged. Parse it into a config
  object at startup and discard the original string.

## KMS and envelope encryption

For application-layer encryption (encrypting user data fields
before storage), use a Key Management Service. The pattern:

- Customer data is encrypted with a **data encryption key (DEK)**,
  generated per record or per partition.
- The DEK is encrypted with a **key encryption key (KEK)** that
  lives in KMS and never leaves it.
- The encrypted DEK is stored alongside the data.
- To decrypt: ask KMS to unwrap the DEK, decrypt the data, drop
  the plaintext DEK from memory.

This way: a stolen database lacks the KEK; an attacker with KMS
access still needs the DB; a key compromise lets you
**re-wrap** DEKs with a new KEK without re-encrypting every
record.

AWS KMS, GCP Cloud KMS, Azure Key Vault Keys all support this
pattern natively.

<Callout variant="warning">
Don't roll your own crypto. Use AES-256-GCM for at-rest
encryption, KMS for key management, libsodium / NaCl for
anything novel. "Cleverness" in cryptography is how breaches
happen.
</Callout>

## In summary

- Secrets live in a dedicated store, not config files, not
  container images, not env vars baked into builds.
- Workload identity (IAM role, SPIFFE) authorizes access; never
  shared "production" credentials.
- Distribute via tmpfs files, mounted by a sidecar that handles
  fetch and refresh.
- Dynamic, short-lived credentials shrink blast radius — Vault's
  per-request DB users are the gold standard.
- Rotate routinely; always rotate on incident, immediately.
- Pre-commit + CI + provider scans together prevent the
  accidental git commit.
- For data-at-rest, use envelope encryption with KMS.
