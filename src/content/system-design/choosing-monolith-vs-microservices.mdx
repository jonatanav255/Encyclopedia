# How to choose: monolith vs microservices

Start with a monolith. Move to microservices only when you
have a *specific, measured* reason and the team to operate
them.

This is the most-broken decision in the industry. "Modern
companies use microservices" leads to teams of 5 running 20
services they can't keep up with. The result is a slow
monolith with extra network hops.

## When microservices actually pay off

- **Multiple teams** (typically 30+ engineers) that can't deploy
  together without coordination overhead.
- **Different parts have different scaling profiles.** The
  payment service handles 10K QPS; the admin tool handles 10.
  Sharing infrastructure wastes money.
- **Different parts have different reliability requirements.**
  Payments need 99.99%; the marketing site can be 99%.
- **Polyglot tech requirements.** ML inference in Python,
  trading engine in C++, web in TypeScript.
- **Independent deploy cadence.** Mobile services that ship
  weekly vs reporting that ships monthly.

## When a monolith is still right

- **Single team, fewer than ~30 engineers.**
- **Single language and runtime is fine for all features.**
- **Deploy cadence is consistent across the product.**
- **Operational team is small.** Microservices need investment
  in CI/CD, observability, service mesh, deploy infrastructure
  — usually a full SRE team.

For most startups, a monolith is the right answer until ~50
engineers.

## The middle ground: modular monolith

A monolith with **strict module boundaries**: clear interfaces
between modules, no shared mutable state, separate databases
or schemas per module. Logical microservices in a single
deployable unit.

```
monolith
├── billing/    (own DB schema, public interface, internal types)
├── orders/
├── users/
└── inventory/
```

You get:

- Deploy independence within the monolith (recompile, test,
  deploy once).
- Module-level fault isolation (a bug in one module is
  contained).
- Easy split-out later — when you do need microservices,
  module boundaries become service boundaries.

Used at Shopify, Stack Overflow, GitHub for years before
considering microservices.

## The cost of microservices

Each new service adds:

- A repo (or directory) + CI/CD pipeline.
- Deployment infrastructure (K8s manifests, secrets, configs).
- Network calls instead of in-process calls (latency, failure
  modes).
- Distributed tracing setup.
- Schema versioning between services.
- New ways to fail (timeouts, retries, circuit breakers,
  cascading failures).
- On-call burden.

For 5 services this is manageable. For 50 it's a full-time
job for an SRE team.

## The "distributed monolith" antipattern

Microservices that **must deploy together**, **share a
database**, or **call each other synchronously in long
chains**.

```
client → service A → service B → service C → service D → DB
```

A change to service D requires synchronous coordination with
A, B, C. You've got the operational cost of microservices
without the deploy independence. (See distributed-monolith-antipattern.)

## The honest decision questions

<Steps>
<Step n={1} title="Why are you considering microservices?">
"Because they're modern" — wrong reason. "Because team X can't
ship without merging through team Y" — right reason.
</Step>
<Step n={2} title="Do you have observability today?">
Distributed tracing, log aggregation, structured logging,
service-level metrics. Without these, microservices are
debuggability hell.
</Step>
<Step n={3} title="Do you have CI/CD for fast independent deploys?">
Microservices without per-service CI take hours per deploy
across the fleet. You'll regret it weekly.
</Step>
<Step n={4} title="Are your service boundaries clear?">
If not, you'll cut them wrong, end up with chatty cross-service
calls, then have to merge them. Take a year to find boundaries
in a modular monolith first.
</Step>
<Step n={5} title="Do you have an SRE team for operations?">
If your "ops" is one person, microservices will burn them
out.
</Step>
</Steps>

If you can't say yes to 1, 2, 3, 4, 5 — stay monolith.

## Conway's law

> Organizations design systems that mirror their communication
> structure.

Microservices reflect a team-oriented organization. If your
org is one big team that talks constantly, your microservices
will be tightly coupled too. The architecture won't fix the
org problem.

## Strangler fig migration

When you do migrate, don't rewrite. Use the strangler-fig pattern:

- New features go into a new service.
- Slowly route old endpoints through a proxy to a new service
  while the old monolith handles the rest.
- When all endpoints have moved, retire the monolith.

## In summary

- Default to monolith. Most teams don't need microservices.
- Move when team size, deploy independence, or scaling
  profiles diverge sharply.
- Modular monolith is the middle ground: clear boundaries,
  one deployable, easy to split later.
- Microservices cost: CI/CD, observability, service mesh, SRE
  capacity. Real, ongoing.
- Anti-patterns: microservices because "modern,"
  distributed-monolith, Conway's-law-mismatch.
- Migrate via strangler fig, not a rewrite.
