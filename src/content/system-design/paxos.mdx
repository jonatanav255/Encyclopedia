# Paxos

Paxos is the consensus algorithm against which all others are
compared. Leslie Lamport designed it in the late 1980s, but
the original parody-style paper ("The Part-Time Parliament")
sat in rejection for nearly a decade and didn't appear in ACM
TOCS until 1998; the 2001 simplification "Paxos Made Simple"
became the canonical reference. It is the
foundation of Google's Chubby, Megastore, Spanner, and many
other large-scale systems — and notoriously hard to implement
correctly.

Today Raft is the more common choice for new systems, but
Paxos remains worth understanding: a great deal of the
distributed-systems literature, and several massively
deployed production systems, are built on it.

## What Paxos solves

The same problem as Raft: a set of processes propose values,
and the protocol picks one such that:

- **Only a proposed value can be chosen.**
- **At most one value is chosen.**
- **A process learns the chosen value only after it has been
  chosen.**

This is **single-value consensus**. Multi-Paxos extends it to
agreeing on a sequence of values (replicated log).

## The three roles

Paxos describes the protocol in terms of three logical roles
that nodes can play (often the same physical node plays
several):

- **Proposers**: propose values.
- **Acceptors**: vote on proposals.
- **Learners**: learn the chosen value.

A node may be all three. In Multi-Paxos, one designated
proposer-leader does most of the work.

## The protocol

Paxos runs in two phases.

### Phase 1: Prepare / Promise

```
Proposer:
  pick a ballot number n (must be greater than any used before)
  send PREPARE(n) to a majority of acceptors

Acceptor receives PREPARE(n):
  if n > highest prepare seen:
    promise not to accept any proposal with ballot < n
    return PROMISE(n, accepted_n, accepted_value)
      where accepted_n / accepted_value are the highest-numbered
      proposal previously accepted (if any)
  else: ignore (or NAK)
```

### Phase 2: Accept / Accepted

```
Proposer, on receiving PROMISE from a majority:
  pick value v:
    if any acceptor returned accepted_value, use the one
      with the highest accepted_n
    else: use proposer's own desired value
  send ACCEPT(n, v) to a majority

Acceptor receives ACCEPT(n, v):
  if n >= highest prepare seen:
    accept the proposal; record (n, v)
    return ACCEPTED(n, v)

Once a majority has accepted (n, v), v is chosen.
```

The choice of `v` in phase 2 is the subtle bit: a proposer
must adopt any previously-accepted value to preserve safety.

## Why it works

The intuition behind safety:

<Steps>
<Step n={1} title="Any two majorities intersect.">
A value chosen by majority M1 with ballot n1 is known to any
later majority M2 — at least one acceptor is in both.
</Step>
<Step n={2} title="A proposer with a higher ballot learns prior decisions.">
Phase 1 brings back any previously-accepted value via PROMISE
messages. The proposer must propose that value (or one with
the highest ballot if multiple).
</Step>
<Step n={3} title="Therefore only one value can ever be chosen.">
Once a value v is chosen with ballot n, any subsequent ballot
n' > n will see v in PROMISE responses and propose v itself.
Conflicting values are impossible.
</Step>
</Steps>

Liveness depends on partial synchrony — proposers must
eventually be able to complete a round without being preempted
by a higher ballot. Two competing proposers can "duel" by
preempting each other's ballots indefinitely; in practice, a
leader-elected protocol (Multi-Paxos) avoids this.

## Multi-Paxos

Single-value Paxos requires two round trips per decision —
too slow for a replicated log. Multi-Paxos amortizes:

- **Phase 1 is run once per leader term.** The leader prepares
  for a range of slots.
- **Phase 2 runs per slot.** Each new value is committed in
  one round trip.

The leader's role becomes:

1. Win an election (a one-time phase 1 for an unbounded range
   of slots).
2. For each new client request, send ACCEPT to acceptors with
   the next slot number.
3. Once a majority accepts, the value is chosen.

This is functionally identical to Raft's log replication. The
difference is mostly presentational.

## The hard parts

What makes Paxos famously hard to implement:

### 1. Leader election is unspecified

The classic paper says "elect a leader by any means." Real
implementations need a concrete election protocol; many
hand-rolled solutions had subtle bugs. Raft, in contrast,
defines election as part of the protocol.

### 2. Membership changes

Adding or removing acceptors mid-flight risks losing the
majority-intersection invariant. The standard solution is
**joint consensus** — temporarily run with both old and new
configurations, requiring a majority of each. Complex to get
right.

### 3. Snapshot and log compaction

The protocol describes commitment but not how to garbage
collect the log. Production systems must define snapshot
formats, transfer protocols, and the rules for installing them
on lagging acceptors.

### 4. Reconciling many slots concurrently

A leader may have many in-flight accept rounds across many
slots. Tracking ballot numbers, partial accepts, and
out-of-order arrivals is intricate.

### 5. Proving correctness

Subtle bugs in proposer/acceptor logic are easy to introduce.
TLA+ specifications exist; production systems rely on them
plus extensive Jepsen-style testing.

## Variants and optimizations

### Fast Paxos

Skip phase 1 in the common case: clients send values directly
to acceptors. Achieves one round trip instead of two — but
requires a larger quorum (3/4 instead of 1/2 + 1) to maintain
safety, and complicates conflict handling.

### Generalized Paxos

Allows commands that commute (e.g., independent updates to
different keys) to be committed without ordering. Higher
throughput at the cost of complexity.

### EPaxos (Egalitarian Paxos)

No leader; any replica can commit. Lower WAN latency because
clients hit the nearest replica. The conflict graph for
non-commutative operations adds complexity.

### Flexible Paxos

The realization that prepare and accept quorums don't have to
be the same size — they only need to intersect. Lets you trade
durability for performance in specific ways.

### Disk-Paxos / Cheap Paxos

Variants that reduce hardware requirements or use shared
storage as a coordination medium.

## Multi-Paxos in production: the real story

Single-decree Paxos — the version in "Paxos Made Simple" — is
not what any real system runs. It commits one value with two
round trips, then starts over. Useful for understanding the
safety argument; useless as a replicated log.

Production Paxos systems run **Multi-Paxos with a stable
leader**, and the optimization that matters more than any
other is the **leader lease**.

### The leader lease optimization

Once a leader has completed phase 1 for a range of slots, it
holds an implicit promise from a majority of acceptors not to
acknowledge any other proposer. Production systems make this
promise *explicit* and *time-bounded*:

- The leader periodically renews a **lease** — typically by
  exchanging heartbeats with a quorum.
- Each acceptor records "I will not vote for anyone else
  until time T."
- While the lease holds, the leader can commit new values
  with **one round trip** (phase 2 only) — phase 1 is skipped
  entirely.
- The leader can even serve **linearizable reads from local
  state** without coordinating, because no other leader can
  have committed anything during the lease window.

This is the optimization that makes Paxos competitive with
Raft in steady state. Both end up at "one RTT + one fsync" per
commit; the difference is mostly in how leader-change is
specified.

### Worked example: Chubby's master lease

Chubby's leader holds a renewable **master lease** of 12
seconds by default (per Burrows OSDI 2006), configurable and
extendable under load. The lease is short enough that a true
partition recovers within an SLA budget, long enough that
typical NTP-bounded clock skew is dwarfed by it.

While the lease is valid:

- Clients can route reads directly to the master without a
  quorum read — the master serves from its local replicated
  state.
- Writes still go through Paxos commit, but skip phase 1.
- A failed lease renewal forces the master to step down; a new
  election begins.

The trick: clients caching the master's identity work because
the lease expiration provides a hard upper bound on stale
results.

### Textbook Multi-Paxos vs. real Multi-Paxos

What's missing from "Multi-Paxos" as described in the papers
versus what every production deployment actually ships:

<Cheatsheet>
| Concern | Textbook Multi-Paxos | Spanner / Chubby / Megastore |
|---|---|---|
| Leader election | "Pick one somehow" | Lease-based with stable terms; explicit fencing tokens |
| Configuration changes | Vague | Joint consensus (run with both old and new majorities simultaneously) |
| Log compaction | Unmentioned | Periodic snapshots + InstallSnapshot RPC for lagging replicas |
| Pipelining | Implicit | Explicit: many slots in flight, out-of-order ACKs, careful in-doubt tracking |
| Read paths | Quorum reads only | Lease reads, follower reads with bounded staleness, read indexes |
| Disk format | Unmentioned | Tight WAL formats, group commit, parallel fsync |
</Cheatsheet>

Each of these is a paper in itself. "Paxos Made Live"
(Chandra, Griesemer, Redstone, 2007) is the most honest writeup
of how much engineering sits between the algorithm and a
production system — orders of magnitude more code than the
algorithm itself.

### Why CockroachDB and TiKV chose Raft per partition, not EPaxos

Both CockroachDB and TiKV evaluated EPaxos and Flexible Paxos
(see flexible-paxos-and-epaxos), then chose **Raft per range /
region** anyway. The reasoning:

- **Library maturity.** etcd-raft and tikv-raft had years of
  production hardening and Jepsen testing by 2015. No
  comparable EPaxos library existed.
- **Implementation cost.** EPaxos's conflict graph and
  dependency tracking add complexity that scales with command
  rate. The team estimated several engineer-years to match
  Raft's correctness story.
- **Many small groups beat one big group.** CockroachDB runs
  thousands of Raft groups per cluster (one per range, ~64 MB
  each); TiKV is similar. Throughput scales by adding ranges,
  not by making any single group faster. EPaxos's "one global
  leaderless consensus" optimization is irrelevant when you
  already have parallelism through sharding.
- **WAN latency wins are marginal at the SQL layer.** The
  cross-region cost of a transaction is dominated by the
  number of ranges it touches, not the per-range commit
  latency. Sharding + smart placement beats leaderless WAN
  consensus for the common workload.

This is the pattern across the industry in 2026: pick a
boring, leader-based, per-shard consensus algorithm and scale
horizontally. Save the exotic variants for the specific
workloads where they pay off (geo-distributed CRDTs,
single-shard databases with global writes).

## Spanner: Paxos at scale

Spanner runs Paxos groups per "tablet" — a range of data —
with replicas placed across regions. Each tablet has its own
Paxos leader; the global system runs millions of groups.

Key Spanner-specific contributions:

- **TrueTime** integration for external consistency: reads
  carry a timestamp; writes wait through clock uncertainty
  before committing to ensure ordering.
- **Two-phase commit between Paxos groups** for cross-tablet
  transactions: each group is reliable; the 2PC coordinator
  uses Paxos for fault tolerance.

The combination is "global linearizable transactions" — which
is what Spanner sells.

## Paxos vs Raft, head to head

<Cheatsheet>
| Aspect | Paxos | Raft |
|---|---|---|
| Year | 1989 (1998 published) | 2014 |
| Specification | Several papers, often confusing | One paper |
| Roles | Proposer / acceptor / learner | Leader / follower / candidate |
| Leader | Optional in single-value; required in Multi-Paxos | Always required |
| Log management | Less prescriptive | Sequential, with backtracking |
| Membership | Joint consensus (complex) | Single-server changes (simple) |
| Common quorum | Majority | Majority |
| Implementations | Spanner, Chubby, Cassandra LWT | etcd, Consul, CockroachDB, TiKV, ... |
| Reputation | "Famously hard" | "Understandable" |
</Cheatsheet>

For a new project: Raft. For understanding the literature
(Spanner, Chubby, classical papers): Paxos.

## Why notoriously hard

Concrete reasons Paxos earned its reputation:

- **Original paper presentation**. The parable framing was
  considered impenetrable. Lamport published "Paxos Made
  Simple" 12 years later acknowledging this.
- **No reference algorithm for Multi-Paxos.** Engineers had to
  derive their own; minor mistakes broke safety.
- **Optimizations conflate with the base protocol.** Production
  systems include leader leases, batching, pipelining,
  read-only optimizations, snapshot transfer, all built atop
  bare Paxos. Each is correct in isolation; composing them
  while preserving safety is hard.
- **Failure modes are subtle.** Network partitions, slow
  nodes, dueling proposers — each requires careful handling.
- **Verification effort is enormous.** TLA+ models of Paxos
  exist; real implementations rarely match them line-for-line.

The Google Chubby team published "Paxos Made Live"
documenting how their implementation diverged from the paper
for unobvious reasons. It's the best paper on the gap between
theory and practice in consensus.

## When to choose Paxos today

Honestly: rarely.

- You're working inside a system that already uses Paxos
  (Spanner, Chubby, Cassandra LWT).
- You want very specific Paxos-variant optimizations (Fast
  Paxos, EPaxos for WAN).
- You're doing research or building a new variant.

For everyone else, Raft is the practical choice. The
implementations are mature, the literature is clear, and the
operational story is well-trodden.

## In summary

- Paxos: Lamport's consensus protocol (published 1998, "Paxos
  Made Simple" 2001); two phases (prepare / accept), majority
  quorum, monotonic ballots.
- Solves single-value consensus; Multi-Paxos extends to logs
  by amortizing phase 1.
- Used by Chubby, Megastore, Spanner, Cassandra LWT — large
  systems and the foundations of the field.
- Famously hard to implement: leader election unspecified,
  membership changes complex, log compaction implicit.
- For new systems, use Raft. For understanding the
  foundations, read Paxos.
