# How to choose: REST vs gRPC vs GraphQL

The three dominant API styles for service-to-service and
client-to-service communication. Each has a sweet spot; none
is universally best.

## Quick rubric

<Cheatsheet>
| Need | Best fit |
|---|---|
| Public API, third-party consumers | REST |
| Internal microservice → microservice, latency-sensitive | gRPC |
| Mobile/web client aggregating many backends | GraphQL |
| Real-time streaming (server → client) | gRPC streams or SSE |
| Tooling-rich, IDE autocomplete | gRPC (codegen from .proto) |
| Browser-cacheable | REST (HTTP semantics work) |
| Different shapes per client (mobile vs web vs admin) | GraphQL |
| Simple CRUD, broad ecosystem | REST |
</Cheatsheet>

## REST

HTTP verbs (`GET`, `POST`, `PUT`, `DELETE`), resource URLs
(`/users/42`), JSON payloads.

### Strengths

- **Universal.** Every language, every browser, every tool
  knows HTTP/JSON.
- **HTTP semantics work.** Status codes, caching headers,
  redirects, idempotency.
- **Discoverable.** A new developer can read URLs and
  understand the model.
- **CDN-friendly.** GETs cache; reverse proxies route by URL.
- **Standardized error handling** via status codes + Problem
  Details (RFC 7807).

### Weaknesses

- **Verbose.** JSON over HTTP/1.1 has overhead per request.
- **No native streaming.** Need SSE or chunked encoding for
  server-push.
- **No native schema.** OpenAPI helps but adds a step.
- **Over-fetching/under-fetching.** A page needs 3 endpoints;
  each returns more than needed.

### When to use

- Public APIs for third parties.
- Server-to-server when latency isn't critical.
- Browser-facing endpoints (auth, CRUD, simple pages).
- Anything where HTTP caching helps.

## gRPC

Binary RPC over HTTP/2 with protobuf schemas. Strongly typed,
codegen for clients in any language.

### Strengths

- **Schema-first.** `.proto` file is the source of truth;
  generates clients and servers.
- **Tiny payloads.** Protobuf binary encoding ~3× smaller
  than JSON, ~5–10× faster to parse.
- **HTTP/2 multiplexing.** Many concurrent calls share one
  connection.
- **Streaming.** Server-streaming, client-streaming, and
  bidirectional streaming all first-class.
- **Codegen ergonomics.** IDE knows every method, every
  field, every error.

### Weaknesses

- **Browser support is poor.** gRPC-Web exists but requires a
  proxy and has limited streaming.
- **Not human-readable.** Debugging requires reflection tools
  or logging interceptors.
- **HTTP infra doesn't help.** CDN caching, simple curl
  debugging — gone.
- **Schema evolution discipline required.** Forgetting `reserved`
  numbers breaks compatibility.

### When to use

- Service-to-service internal APIs where latency and
  throughput matter.
- Polyglot environments (need consistent client codegen
  across Go, Java, Python, etc.).
- Streaming workloads.

## GraphQL

A query language for APIs: clients send a query that describes
exactly what they need; the server returns exactly that.

### Strengths

- **Single endpoint.** `POST /graphql` handles every read
  and write.
- **Client-driven shape.** A mobile screen requests only the
  fields it shows; the web requests more.
- **Schema introspection.** Built-in. Generates types and
  docs for any language.
- **Aggregation across services.** Federated GraphQL (Apollo,
  Hot Chocolate) stitches multiple services into one
  endpoint for the client.
- **Subscriptions** for realtime over WebSocket.

### Weaknesses

- **HTTP caching is broken.** Every query is a `POST`; no
  CDN caching. (Mitigation: persisted queries.)
- **N+1 query explosion** without DataLoader.
- **Easier to write expensive queries.** A client can request
  a deeply nested tree; server must compute it. Mitigations:
  query depth limits, cost analysis.
- **Schema is a single point of coordination.** Every team
  contributing types must agree on naming, evolution, deprecation.

### When to use

- Mobile + web + admin clients with different shapes from
  the same backend.
- Aggregating multiple backends into one API for clients.
- Rapid frontend development where shape changes often.

Avoid for:

- Service-to-service (gRPC or REST is simpler).
- Public APIs (third parties don't want to learn your schema).
- CDN-heavy workloads.

## Mixing them

Common production setup:

- **gRPC** for service-to-service inside the datacenter.
- **REST** for public APIs and webhook ingress.
- **GraphQL** for the BFF layer between mobile/web and the
  internal services.

Each at its sweet spot. Nothing forces you to pick one for
everything.

## Common mistakes

### "GraphQL for our microservices"

GraphQL between services is rare; the type system and
resolver complexity slow you down without buying anything.
gRPC or REST is simpler internally.

### "gRPC from browsers"

Browser support is awkward. gRPC-Web works but you've added a
proxy layer. If the consumer is browsers, REST or GraphQL is
usually the right call.

### "REST is old / outdated"

REST is fine. Most apps don't have the scale or polyglot need
that justifies gRPC. HTTP/JSON is universal for a reason.

### "Pick one for everything"

The right answer is usually multiple. Public REST, internal
gRPC, optional GraphQL BFF.

## In summary

- REST: default for public APIs and HTTP-cache-friendly
  traffic.
- gRPC: best for internal, latency-sensitive, streaming, or
  polyglot service mesh.
- GraphQL: best when clients need flexible shapes or
  aggregation across backends.
- Most production systems use a mix; don't force one style
  everywhere.
- Anti-patterns: GraphQL between services, gRPC from
  browsers, "REST is old."
