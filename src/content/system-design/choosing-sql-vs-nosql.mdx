# How to choose: SQL vs NoSQL

Default to SQL. Reach for NoSQL when you have a specific
reason backed by measurement.

The "NoSQL revolution" of 2010–2015 produced great tools for
specific problems. It also produced a lot of teams who picked
DynamoDB because it sounded modern, then spent years working
around the fact that they actually needed Postgres.

## Decision tree

<Steps>
<Step n={1} title="Do you need ACID transactions across multiple rows?">
Yes → SQL. NoSQL stores can do per-key transactions, not
multi-key.
</Step>
<Step n={2} title="Do you need joins?">
Yes (and you can't trivially denormalize) → SQL. Most NoSQL
forces you to denormalize, which costs writes and storage.
</Step>
<Step n={3} title="Do you have less than ~1 TB of data and less than ~10K QPS?">
Yes → SQL. Postgres or MySQL handle this comfortably.
NoSQL's scaling advantages don't apply.
</Step>
<Step n={4} title="Is the schema highly volatile or document-shaped?">
Yes → Document store (MongoDB, DynamoDB). Schema-on-read is
faster to evolve.
</Step>
<Step n={5} title="Massive write throughput needed (>100K writes/sec)?">
Yes → Wide-column (Cassandra, ScyllaDB) or KV (DynamoDB,
Riak). SQL can't keep up here without expensive sharding.
</Step>
<Step n={6} title="Graph-shaped traversals dominate?">
Yes → Graph DB (Neo4j, Neptune). SQL recursive CTEs work
but get expensive on deep traversals.
</Step>
<Step n={7} title="Time-series with high write volume + range queries?">
Yes → TSDB (InfluxDB, TimescaleDB, ClickHouse). Postgres
works at modest scale; TSDBs at large.
</Step>
<Step n={8} title="Full-text search?">
Yes → Search engine (Elasticsearch, OpenSearch). Postgres FTS
covers basic; specialized engines beat it at scale.
</Step>
</Steps>

If you're past step 1–3, you usually don't need NoSQL.

## The honest defaults

### SQL (Postgres, MySQL)

- Default for ~80% of new systems.
- ACID, joins, indexes, materialized views, transactions,
  rich query language.
- JSON columns let you do "document" storage when needed
  without giving up relational power.
- Scales to a TB or two and 10K+ QPS without much effort;
  beyond that requires care (partitioning, read replicas,
  Citus-style sharding).

### Document (MongoDB, DynamoDB)

- Schema is part of the document, not the table.
- Easy to evolve fields per-document.
- Per-document transactions; multi-document is recent + slow.
- Strong for catalog data, user preferences, event sourcing
  payloads.

### Wide-column (Cassandra, ScyllaDB, HBase)

- Massive write scale; tunable consistency.
- Query patterns must be designed in advance — model around
  the queries.
- Eventual consistency by default; LWT for strong.
- Time-series, write-heavy event logs, social-graph fan-out.

### Key-value (Redis, DynamoDB, Riak)

- Single-key access by primary key. No queries beyond that.
- Tiny per-operation latency (Redis: under 1 ms).
- Caches, session stores, rate limit counters.

### Graph (Neo4j, JanusGraph, Neptune)

- First-class node + edge model.
- Cheap traversals; expensive global queries.
- Social networks, recommendation graphs, knowledge graphs.

### Search (Elasticsearch, OpenSearch, Vespa)

- Inverted indexes; full-text + faceted search.
- Eventually consistent; not a source of truth.
- Search, log analytics, observability.

### Time-series (Influx, Timescale, ClickHouse)

- Write-optimized for append; aggregation-optimized for read.
- Columnar storage, range queries fast.
- Metrics, IoT, financial data.

## Common mistakes

### "We need to scale" → DynamoDB at 10 QPS

You don't need scale. You need joins. Postgres for years.

### "Schema flexibility" → MongoDB → 5 years of consistency bugs

You wanted flexibility for development. You got it. You also
got "the user's email is a string in 90% of documents and
an object in 10%." Schema-on-read shifts the cost from "DBA
review" to "runtime bug per query."

### "Cassandra for everything"

Cassandra is great when you've **modeled around the query
shape upfront**. It's terrible when product requirements
change and the existing model no longer supports the new
query. SQL's flexibility costs more at write time and pays
back at every product pivot.

### "Postgres can do JSON, so we don't need MongoDB"

Often true. Postgres JSONB is excellent. If you don't have a
specific reason for Mongo (large clusters of true document
shape, mobile sync), Postgres with JSONB is usually enough.

## When in doubt

Pick Postgres. You can usually add a NoSQL store later for a
specific use case (cache, search index, analytics) without
ripping out Postgres. The reverse (replace your NoSQL store
with Postgres) is months of migration work.

## In summary

- Default to SQL. Most apps never need NoSQL.
- Pick NoSQL only with a measured reason: scale, schema
  shape, query shape, latency.
- Document for evolving schemas; wide-column for write
  throughput; KV for low-latency single-key; graph for
  traversals; search for FTS; TSDB for time-series.
- Postgres + JSONB covers many "we need a document store"
  cases without leaving SQL.
- Adding NoSQL alongside SQL is easy; replacing SQL with
  NoSQL is expensive.
