# Dimensional modeling

Ralph Kimball's approach to designing data-warehouse schemas
for analytics. Two table types:

- **Fact tables**: events that happened. Sales, page views,
  orders. Lots of rows, narrow.
- **Dimension tables**: attributes describing those events.
  Customers, products, dates, stores. Few rows (millions, not
  billions), wide.

The classic schema shape is the **star schema**: facts in the
middle, dimensions joined to them.

## Star schema

```
              dim_date
                  │
dim_customer ─ fact_sales ─ dim_product
                  │
              dim_store
```

`fact_sales` contains:

- Foreign keys to all dimensions.
- Measures (quantity, revenue, discount).
- Maybe degenerate dimensions (transaction ID).

Each dimension contains its descriptive attributes
(`dim_customer.name`, `dim_customer.country`, etc.).

Queries are simple: join fact + dimensions, group by
attributes, aggregate measures.

```js
const sql = `
  SELECT
    dim_date.year,
    dim_product.category,
    SUM(fact_sales.revenue) AS revenue
  FROM fact_sales
  JOIN dim_date ON fact_sales.date_key = dim_date.key
  JOIN dim_product ON fact_sales.product_key = dim_product.key
  GROUP BY 1, 2
`;
```

## Why this design

- **Optimized for analytics queries**. Star schemas join
  efficiently in columnar engines.
- **Business-readable**. "What did we sell in Q3 by product
  category?" maps directly to the SQL.
- **Stable**. Adding new measures or dimensions doesn't break
  existing queries.
- **Industry standard**. Every BI tool understands it.

## Fact table grain

The **grain** is what one row represents:

- One row per order line item? (`fact_order_lines`)
- One row per order? (`fact_orders`)
- One row per daily aggregate? (`fact_daily_sales`)

The grain determines what aggregations are possible. Coarser
grain = smaller table, lower flexibility. Finer grain =
flexibility but storage cost.

Common practice: maintain multiple fact tables at different
grains for different query patterns.

## Types of fact tables

### Transaction fact

One row per event. Most common shape. `fact_sales`.

### Snapshot fact

State at a point in time. Daily customer balance.
`fact_account_balance_daily`.

### Accumulating snapshot

Each row represents an entity's lifetime (filling in as
events happen). `fact_orders` with columns
`placed_at`, `paid_at`, `shipped_at`, `delivered_at` — null
until each event happens.

### Factless fact

A row indicating an event happened, no measures. "Student
attended class" — useful for counting, no quantity to sum.

## Dimension types

### Conformed dimensions

Shared across multiple fact tables. `dim_date` used by
`fact_sales`, `fact_traffic`, `fact_inventory`. Consistent
definitions across the business.

### Junk dimensions

Combine many low-cardinality boolean flags into one table to
avoid a wide fact. "Order had express shipping AND was a
gift AND ..." becomes one junk-dim row per combination.

### Degenerate dimensions

Identifiers that exist on the fact but aren't worth their
own dimension table. Order number, ticket number.

### Role-playing dimensions

One table used multiple times. `dim_date` joined to
`fact_orders` as `order_date`, `ship_date`, `payment_date`.

## Snowflake schema

If dimensions themselves have hierarchies, you can normalize
them:

```
dim_product → dim_category → dim_department
```

Pros: less storage if categories are reused; explicit
hierarchy.
Cons: more joins per query; slower.

In modern columnar warehouses, the storage saving is
negligible; star schemas win on query simplicity.

## Slowly changing dimensions (SCD)

When dimension attributes change over time, what do you do?

- **Type 1**: overwrite. History lost.
- **Type 2**: new row with effective dates. Full history.
- **Type 3**: previous value column. Limited history.

(See slowly-changing-dimensions.)

## Surrogate keys

Most dimension tables use **surrogate keys** (auto-incrementing
integers) instead of natural keys (the business identifier).

Why:

- Insulates from natural-key changes (email rename, SSN
  reissue).
- Type 2 SCD requires multiple rows per natural key.
- Smaller integer = faster joins than UUIDs / strings.

Fact tables reference dimension surrogate keys.

## When dimensional modeling doesn't fit

- **Streaming / real-time queries** with millisecond
  latency. Star joins are too slow; flat denormalized
  events win.
- **Highly nested / hierarchical data**. JSON or document
  store is closer to the shape.
- **Truly unstructured data**. Logs, raw text — don't try to
  force into stars.

For traditional BI and reporting, star schemas remain
unmatched.

## Modern adoption

dbt, Looker, and most warehouses still encourage dimensional
modeling for the curated analytics layer. The raw / staging
layer might be flatter, but the "marts" presented to BI tools
are usually star schemas.

The Kimball book (1996, with updates) is still the
authoritative source.

## In summary

- Dimensional modeling: facts (events) + dimensions
  (attributes), joined as a star schema.
- Star schema: facts in center, dimensions on the edges.
  Optimal for BI queries.
- Fact table grain matters (one row per what?).
- Slowly changing dimensions: type 1 (overwrite), type 2
  (new row), type 3 (prev column).
- Conformed dimensions: shared across multiple facts for
  consistency.
- Surrogate keys (integers) over natural keys.
- Still the default for curated analytics layers in modern
  warehouses.
