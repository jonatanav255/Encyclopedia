# Byzantine vs crash fault tolerance

Distributed systems can fail in two fundamentally different
ways. The choice of fault model shapes which algorithms you
need, how many nodes you must run, and how expensive
correctness will be.

## The two models

- **Crash fault**: a node either follows the protocol or
  stops. Silent failure. The node doesn't lie, just goes
  away.
- **Byzantine fault**: a node may do *anything* — send
  contradictory messages, lie about its state, behave
  arbitrarily. Could be a bug, a hacked node, or a malicious
  actor.

In a datacenter, CFT is usually enough; nodes you own don't
lie to you. In cross-organizational systems (blockchain,
multi-party computation), nodes can't be trusted, and you
need BFT.

## The cost difference

To tolerate **F** simultaneous faults:

<Cheatsheet>
| Fault model | Min nodes needed |
|---|---|
| Crash fault | 2F + 1 |
| Byzantine fault | 3F + 1 |
| Byzantine fault (signed messages) | 2F + 1 |
</Cheatsheet>

Concrete: tolerating 1 fault needs **3** nodes (CFT) or **4**
nodes (BFT). Tolerating 2 faults needs 5 (CFT) or 7 (BFT).
Plus: BFT messages are typically signed, larger, and require
much more cross-node communication.

A Raft cluster of 5 nodes (2F+1 with F=2) is plenty for a
datacenter; a 7-node PBFT cluster (3F+1 with F=2) is the
Byzantine equivalent.

## When you need BFT

- **Public blockchains.** Bitcoin, Ethereum — nodes are
  operated by anonymous parties who may be adversarial.
- **Permissioned blockchains across organizations.**
  Hyperledger Fabric, R3 Corda — different companies run
  nodes; you can't assume they won't try to cheat.
- **Critical multi-party systems.** Air-traffic control,
  weapons systems, election infrastructure.
- **Untrusted hardware.** Cloud providers' enclaves running on
  potentially compromised hosts.

## When you don't

- **Single-organization datacenter.** Your servers don't lie
  to you. Bugs cause crashes, not arbitrary behavior. Use Raft
  or Paxos.
- **Most CRUD apps.** Your DB, queue, cache — CFT is
  sufficient.
- **SRE-managed clouds.** Cloud providers run their own
  failure-tolerance under the hood; from your perspective the
  service either responds or crashes.

If you're not sure which model you need, you almost certainly
need CFT. BFT is roughly 10–100× more expensive (more nodes,
more messages, more crypto) and only useful when adversarial
behavior is in scope.

## Byzantine algorithms

The classic algorithms:

- **PBFT (Practical Byzantine Fault Tolerance)**: Castro &
  Liskov 1999. 3F+1 nodes, three-phase commit per request,
  O(N²) messages per request. Reasonable for ~10–100 node
  clusters.
- **HotStuff**: linear message complexity per request, basis of
  Diem/Libra and several modern BFT chains.
- **Tendermint**: BFT used in Cosmos.
- **Nakamoto consensus**: Bitcoin's proof-of-work. BFT under
  the assumption that no single party controls more than 50%
  of computational power.
- **Proof-of-stake variants**: Casper, Algorand, Solana — BFT
  under assumption that no party controls more than 1/3 of
  stake.

## Crash-fault algorithms

The standard tools:

- **Raft**: leader-based replication, easier to understand than
  Paxos.
- **Multi-Paxos**: original consensus algorithm, used by
  Spanner, Chubby, ZooKeeper (variant: ZAB).
- **Viewstamped Replication**: equivalent to Raft, predates
  it.

These all assume nodes follow the protocol — they might
crash, but they don't lie.

## The asynchrony dimension

Both CFT and BFT live inside an **asynchrony assumption**:

- **Synchronous**: messages have bounded delay. Possible to
  build perfect consensus.
- **Partially synchronous**: messages eventually deliver but
  no bound. Most real systems. Consensus possible (Raft,
  Paxos, PBFT).
- **Fully asynchronous**: no bound at all. FLP impossibility
  applies — no deterministic consensus possible if even one
  node can crash. Real systems sidestep this by relaxing
  asynchrony or accepting probabilistic termination.

## Common confusions

### "BFT means hack-proof"

BFT tolerates up to F Byzantine nodes among N=3F+1. If the
attacker controls more than F, BFT gives no guarantee. So a
4-node BFT cluster collapses if 2 nodes are compromised.

### "Blockchain needs BFT, but it's still slow because of consensus"

Blockchains are slow for **multiple** reasons: BFT messaging
overhead, large network size, proof-of-work or stake
verification. BFT is just one factor.

### "We're in a datacenter, but use BFT to be safe"

The cost is 1.5× more nodes and 5–10× more messages for
protection against a fault mode that doesn't happen in your
environment. Almost never worth it.

## In summary

- Crash fault: nodes fail silently. Tolerate F with 2F+1
  nodes. Use Raft/Paxos.
- Byzantine fault: nodes can lie arbitrarily. Tolerate F with
  3F+1 nodes. Use PBFT/HotStuff/Tendermint.
- BFT is roughly 10–100× more expensive in resources and
  bandwidth.
- Use BFT when you're crossing organizational trust boundaries
  (blockchain, multi-org). Otherwise CFT.
- Both models still depend on asynchrony assumptions (FLP); no
  consensus is perfectly safe under unbounded delay.
