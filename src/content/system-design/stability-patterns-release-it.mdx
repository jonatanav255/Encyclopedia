# Stability patterns (Release It!)

Michael Nygard's *Release It!* catalogued patterns and
antipatterns for production stability after a career of
debugging outages. The book is from 2007; the 2nd edition
(2018) updates everything. The patterns are universal.

This page summarizes the catalog. Each pattern has its own
deeper page in this topic.

## Stability antipatterns (what kills systems)

### Integration points

Every connection to another system is a potential
failure. They're the dominant source of production failures.

A service that calls 10 downstreams has 10 ways to fail.
Without timeouts, circuit breakers, and fallbacks, each
becomes a path to outage.

### Chain reactions

A failure in one component triggers failures in others.
Server A is overloaded; its replicas now take all its traffic;
they overload; the cluster collapses.

Each instance compensating for failed peers is the mechanism.

### Cascading failures

A failure in one service propagates upstream. Service C fails;
B's circuit breaker opens (good); B starts returning errors
to A (eventually causing A to fail too).

Without proper isolation and graceful degradation, every
component eventually depends on every other.

### Users (especially traffic spikes)

Real users do unexpected things at unexpected rates. Sale
events, viral content, bot traffic. Systems must handle
3–100× normal load on bad days.

### Blocked threads

A thread waiting on an unresponsive socket holds resources.
Multiply by hundreds; the service has no threads left.

The original case for **timeouts everywhere**.

### Attacks of self-denial

The system uses up its own resources. Memory leak. File
descriptors. DB connections never released. The system
crashes itself.

### Scaling effects

What works for 10 services breaks at 1000. N² communication,
broadcast storms, gossip overload. Scale finds different
failures than load.

### Unbalanced capacities

A's call to B is fast. B's call to C is slow. C is the
bottleneck. But B is the one being blamed for "being slow."
Capacity must match across the chain.

### Dogpile

When a service comes up after restart, all clients reconnect
simultaneously. Service crashes again. Repeat.

Mitigation: jittered reconnect backoff.

### Slow responses

Often worse than no response. A 30-second response holds
client threads; clients keep retrying; the system spirals.

Fail fast > respond slowly.

### SLA inversion

Your SLA is 99.99%. You depend on a third-party with a 99%
SLA. Math doesn't allow you to beat your weakest dependency.

Either redundancy (multiple providers), aggressive caching,
or graceful degradation.

### Unbounded result sets

`SELECT * FROM orders WHERE user_id = X` — works fine for a
typical user. A power user has 100K orders; query returns 100
MB; memory exhausts.

Always paginate. (See pagination-patterns.)

## Stability patterns (what to build in)

### Timeouts

Every network call gets one. (See timeouts-everywhere.)

### Circuit breaker

Trip when downstream fails repeatedly; serve fallback. (See
circuit-breaker.)

### Bulkhead

Partition resources so one failing dependency doesn't
starve others. (See bulkhead-pattern.)

### Steady state

The system can run indefinitely without intervention. Logs
rotate. Caches evict. Sessions expire. Disk usage stays
bounded.

The opposite of "needs a daily restart."

### Fail fast

If you can detect ahead of time that a request will fail
(missing data, exceeded quota), reject immediately. Don't make
the user wait for the inevitable.

### Handshaking

Slow consumers signal upstream to slow down. (See
backpressure.) Don't push faster than your downstream can
process.

### Test harness

Production-like test environment with the ability to inject
failures. The foundation for chaos engineering.

### Decoupling middleware

Brokers, queues, or async messaging between services so a
failure in one doesn't immediately block the other.

### Shed load

Drop low-priority work under overload. (See load-shedding.)

### Create back pressure

Bounded queues with explicit rejection when full. Forces the
problem to surface rather than hiding in unbounded memory.

### Governor

A rate limit or quota on operations that might cause runaway
behavior. The "circuit breaker for resource use" rather than
"for downstream calls."

## Why this catalog matters

Production failures repeat. The same shapes have killed
systems for decades. Nygard's contribution was naming them
so teams could talk about them.

When someone says "we have an SLA inversion problem with our
payment provider," everyone in the room knows what that
means and what the fixes are.

## How to use the catalog

The antipattern list is a debugging tool: "what's killing
us?" usually maps to one or two of these.

The pattern list is a design checklist: "do we have timeouts?
circuit breakers? bulkheads? steady state?" — for every new
service before launch.

For incident reviews, identify which antipatterns were
present and which patterns were missing. The fix isn't "be
more careful next time"; it's "implement bulkhead pattern in
service X."

## In summary

- Release It! catalogs stability antipatterns and patterns
  from years of production experience.
- Antipatterns: integration points, chain reactions,
  cascading failures, blocked threads, slow responses, SLA
  inversion, unbounded result sets, dogpile, scaling effects,
  unbalanced capacities, users.
- Patterns: timeouts, circuit breaker, bulkhead, steady state,
  fail fast, handshaking, test harness, decoupling middleware,
  shed load, create back pressure, governor.
- Use as a design checklist for new services and a diagnostic
  framework for incidents.
