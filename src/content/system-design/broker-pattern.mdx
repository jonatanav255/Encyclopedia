# Broker pattern

A central intermediary decouples producers from consumers.
Producers don't know who consumes; consumers don't know who
produced. The broker routes, buffers, and sometimes transforms.

This is the architectural family that includes message brokers
(Kafka, RabbitMQ, NATS), service buses (Mulesoft, Apache
Camel), API gateways, and pub/sub services.

## The shape

```
producers ──► broker ──► consumers
                │
                ├─ routing rules
                ├─ buffering (queue or log)
                ├─ delivery guarantees
                └─ transformations (optional)
```

The broker owns the addressing problem. A producer says "this
event happened" without knowing who cares. Subscribers
register interest with the broker.

## Three families

### Message brokers

Async, durable, large fan-out. Kafka, RabbitMQ, AWS SNS/SQS,
Google Pub/Sub. (See `pubsub-vs-queue` and
`message-broker-choice`.)

### Service buses / ESBs

Enterprise integration with built-in transformation and
routing logic. Mulesoft, Apache Camel, BizTalk. Heavier than
brokers; usually overkill outside large legacy environments.

### API gateways

Sync, request/response, with auth/rate-limit/routing on top.
Kong, Apigee, AWS API Gateway. (See api-gateway.)

All three are brokers in spirit — they sit between callers
and callees and centralize cross-cutting concerns.

## Trade-offs

### Pros

- **Loose coupling.** Producers and consumers evolve
  independently.
- **Fan-out.** One producer event reaches N consumers without
  the producer doing N calls.
- **Buffering.** A slow consumer doesn't block the producer.
- **Cross-cutting concerns.** Auth, rate limiting, observability
  centralized at the broker layer.

### Cons

- **Single point of operational concern.** The broker must be
  highly available; an outage takes down many services.
- **Latency tax.** Every hop costs ms-to-tens-of-ms.
- **Operator complexity.** Brokers need to be tuned,
  monitored, scaled, upgraded.
- **Schema coordination.** Producers and consumers must agree
  on payload shape, evolution, versioning.

## Common antipatterns

### The broker as DB

Storing application state in the broker (Kafka used as a
forever log of "facts that should be queryable") works briefly,
then collapses. Brokers aren't designed for ad-hoc reads.

Fix: project events from the broker into a real database for
queries. (See log-as-database for the nuanced version.)

### The broker as a synchronous RPC

Sending a request through a broker and waiting for a response
via a reply queue. You've reinvented HTTP RPC with extra
hops. If you need request/response, use HTTP/gRPC.

### Logic in the broker

ESBs often grow large XSLT or rule-based pipelines inside the
broker. Now the broker is your most-stateful service, the
hardest to deploy, the one that takes down everything.

Fix: keep the broker dumb. Logic lives in producers or
consumers, not in routing.

## When to use it

A broker pays off when:

- You have **multiple consumers** of the same event.
- You need **asynchronous** processing (the producer doesn't
  wait).
- Your throughput requires **buffering** (consumer is slower
  than producer).
- You need to **add consumers later** without changing the
  producer.

Don't use it when:

- You need **request/response** semantics.
- The broker would be the only stateful component, and a
  direct point-to-point call is simpler.
- The team isn't ready to operate a broker (Kafka is real
  work).

P2P is simpler when you have few consumers and they don't
change often. Brokers win when consumer set grows.

## Modern variants

- **Cloud-native pub/sub** (AWS SNS, Google Pub/Sub, Azure
  Service Bus) — managed brokers; no ops, scale at provider
  speed.
- **Event mesh** — a network of interconnected brokers across
  clouds and on-prem. Solace, Confluent Multi-Region.
- **Edge brokers** — MQTT brokers for IoT, often with
  bridge-to-cloud.
- **gRPC streams** as P2P "broker-lite" between specific
  services.

## In summary

- Broker pattern: central intermediary decouples producers
  from consumers.
- Three flavors: message brokers (async, durable), service
  buses (rich routing), API gateways (sync request/response).
- Pros: loose coupling, fan-out, buffering, cross-cutting
  concerns.
- Cons: ops burden, latency tax, schema coordination.
- Antipatterns: broker-as-DB, broker-as-RPC, business logic
  in the broker.
- Best for multi-consumer async; overkill for simple
  point-to-point.
