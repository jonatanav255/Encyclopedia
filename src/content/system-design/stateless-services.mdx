# Stateless services

A stateless service holds **no per-request memory between
requests**. Any replica can serve any request. The local
process knows about the configuration, the code, and whatever
it just loaded from a backing store — nothing else.

This sounds boring. It's the single biggest enabler of
horizontal scaling, blue-green deploys, and graceful failover.
Most production scaling problems trace back to a service that
*thinks* it's stateless but isn't.

## What "stateless" actually means

A service is stateless if all three are true:

1. **No in-memory session data** that other replicas don't have.
2. **No local disk dependency** that other replicas can't see.
3. **No process-local caches** that affect correctness (only
   performance).

State that lives somewhere shared — a database, Redis, S3, a
message broker — is fine. State that lives in *this process's
heap or this disk* is the problem.

A 30-second test: kill this replica mid-request. Does the next
request, routed to a different replica, work correctly? If yes,
stateless. If it produces an error, hands the user a stale
session, or loses data — stateful.

## Why it matters

<Steps>
<Step n={1} title="Horizontal scaling becomes trivial.">
Stateless replicas are interchangeable. Add five more behind
the load balancer; traffic rebalances; done. No coordination,
no cache warming, no migration.
</Step>
<Step n={2} title="Deploys are safe.">
Rolling deploys, blue-green, canary — all assume you can shift
traffic between replicas without breaking sessions. Sticky
state on a single replica makes every deploy a hazard.
</Step>
<Step n={3} title="Failover is automatic.">
A replica dies. The load balancer health-checks it out. Its
in-flight requests fail and retry against a healthy replica.
Users never notice. With state pinned locally, those users
lose their session or their work.
</Step>
<Step n={4} title="Autoscaling works.">
Autoscaling scales in *and out*. If a scale-in destroys a
replica that held the only copy of a user's session, the user
gets logged out at 3 AM. Stateless replicas absorb scale-in
without consequence.
</Step>
<Step n={5} title="Multi-region is feasible.">
Stateless services replicate to a new region for free.
Stateful services need replication design before they go
multi-region. The state layer is the hard part.
</Step>
</Steps>

## Common forms of accidental state

The places state sneaks in:

### In-memory session

```ts
const sessions = new Map<string, Session>();

app.post('/login', (req, res) => {
  const id = crypto.randomUUID();
  sessions.set(id, { userId, createdAt: Date.now() });
  res.cookie('sid', id);
});
```

Convenient. Works in dev. The first time you add a second
replica, half of `/me` requests fail because `sessions` is
local to one process. See session-management-at-scale for the
fix.

### Local file uploads

```ts
app.post('/upload', upload.single('photo'), (req, res) => {
  // req.file.path is on this server's disk
  saveToDb({ userId, path: req.file.path });
});
```

A subsequent `/photos/:id` request routed to a different
replica can't find the file. Either upload directly to object
storage (S3, GCS) or stream through the app to object storage
with a signed URL.

### Process-local caches that affect behavior

```ts
const featureFlags = new Map();
setInterval(refreshFlags, 60_000);
```

If replicas refresh independently, two replicas can see
different flag values for several seconds during a flag flip.
Sometimes acceptable; sometimes catastrophic (e.g., a flag
gating PII redaction). Treat process-local caches as eventually
consistent and design accordingly.

### WebSocket or SSE connection state

A WebSocket lives on one replica. Messages destined for that
user must reach that replica. Either:

- Pin the user to that replica (sticky session, complicates
  deploys), OR
- Route messages through a pub/sub layer (Redis, NATS) so any
  replica can deliver to any connected user.

The latter is the modern standard. See sticky-sessions for the
tradeoff.

### Long-running per-user state machines

Multi-step wizards, OAuth flows, payment flows that store
intermediate state in the process. Persist intermediate state
in a database or Redis with a TTL. The process should be able
to crash between steps without breaking the flow.

## Patterns for pushing state out

### Externalize sessions

Sign session content into the cookie itself (JWT, signed
cookie) — no server state at all — *or* store sessions in
Redis keyed by session ID. Both work; both have tradeoffs (see
session-management-at-scale).

### Use object storage for files

Direct browser → S3 with a presigned POST URL. The app server
never touches the bytes. Scales to terabytes for the cost of
a single API call.

### Distributed cache for shared lookups

Redis for hot keys. Local cache fronts Redis for further
latency reduction, with a small TTL (1–30 seconds) so all
replicas converge quickly.

### Persist work-in-progress

Wizard step 3 of 5 → save partial state to DB with `status =
'in_progress'`. Step 4 can land on any replica.

### Move long jobs to a queue

Don't compute a 30-second PDF inside the request. Drop a job
on a queue; return a job ID; let workers (also stateless)
process. The request finishes in 10 ms regardless.

## Stateless ≠ pure

A stateless service can still:

- **Cache** (as long as eviction or staleness is safe).
- **Connect** to databases (the pool itself is local; the data
  isn't).
- **Open** files (read-only configs, static assets).
- **Maintain** counters and metrics (those are observability,
  not correctness state).

The rule is about request-affecting state, not all in-process
data. A connection pool to Postgres is stateless from the
LB's perspective — drop the replica, the next replica opens
its own pool from the same config.

## The 12-factor view

The 12-factor app guidelines codify this:

- **VI. Processes**: execute the app as one or more stateless
  processes.
- **IV. Backing services**: treat databases, caches, queues as
  attached resources, swappable by config.
- **IX. Disposability**: processes should start fast and shut
  down gracefully.

All three flow from stateless design. They aren't separate
principles; they're consequences of "any replica can serve any
request."

## Stateful services exist and are fine

Some things are inherently stateful: databases, message brokers,
caches with non-trivial in-memory state (Redis, Memcached),
search indexes, in-memory analytical engines.

You don't make these stateless — you make them **explicitly
stateful** with care:

- Persistent storage with replication.
- Coordinated cluster membership (Raft, Paxos, gossip).
- Slow, careful scaling (Kafka rebalances, Cassandra streaming).
- Failover plans and runbooks.

The mistake is treating a *stateful* service as if it were
stateless. The opposite — treating a stateless app server as if
it needs all the stateful-service machinery — wastes weeks.

## Detecting accidental state

Useful checks in a code review:

<Cheatsheet>
| Smell | Probable problem |
|---|---|
| `Map<sessionId, ...>` or `new Set()` at module scope | In-memory session/state. |
| `fs.writeFile(...)` for user data | Local disk dependency. |
| `setInterval` for refresh that affects behavior | Per-process timing. |
| `path.join('/tmp', ...)` for anything persistent | Disk that other replicas can't see. |
| WebSocket message routing that assumes local map | Won't work across replicas without pub/sub. |
| `process.env.LOCAL_SOMETHING` set differently per box | Config drift between replicas. |
</Cheatsheet>

None of these are bugs in isolation. Each is a smell that
someone assumed "one process forever."

## The exception: deliberate single-leader services

A handful of services genuinely need a single leader: leader
election for distributed locks, certain stateful workflow
engines, real-time game servers with per-room state. Don't
fight this — embrace it with the right machinery:

- Leader election via etcd/ZooKeeper/Consul.
- Sharding by entity (user, room, document) so each shard has
  one leader.
- A failover protocol.

These services aren't stateless and shouldn't pretend to be.
But they're also rare. 95% of services in a typical backend
should be stateless.

## Common pitfalls

### "We'll just use sticky sessions"

Sticky sessions hide statefulness — they don't fix it. The
moment a replica dies or a deploy rolls, users lose their
session anyway. See sticky-sessions for the full critique.

### Local in-memory rate limiters

```ts
const counts = new Map();
```

A 100 RPM rate limit on one of 10 replicas effectively becomes
1000 RPM globally — each replica counts only its own traffic.
Use a shared store (Redis with `INCR + EXPIRE`).

### Cache warmth as a soft requirement

If your service is unusably slow after a deploy because the
local cache is cold, you're stateful — just informally. Warm
the cache from a shared source, or design for cold-start
performance.

### Logging request context across replicas

If a user's request spans three replicas (frontend, API,
worker), make sure your trace ID propagates. Otherwise debugging
is impossible. See observability for the standard pattern.

## In summary

- Stateless = any replica serves any request, identically.
- The enemy is local memory, local disk, and local timers that
  affect correctness.
- Move sessions to a shared store; uploads to object storage;
  WebSocket routing to pub/sub; long jobs to a queue.
- Stateful services are fine — but call them stateful and run
  them with replication, failover, and care.
- Once your services are truly stateless, every other scaling
  decision gets easier.
