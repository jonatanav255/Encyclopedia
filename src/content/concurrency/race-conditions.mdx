# Race conditions

A race condition is when the **correctness of a program depends
on the timing or interleaving of concurrent operations**. The
program isn't wrong in the sense of "this line of code is
wrong" — every individual line is fine. It's wrong because the
*order* in which lines from different tasks execute happens to
produce a value the developer didn't expect.

## The canonical example

Two threads increment a counter:

```c
int counter = 0;

void increment() {
    counter = counter + 1;   // looks atomic. isn't.
}
```

`counter = counter + 1` is at least three machine instructions:

```
1. load   counter -> R1
2. add    R1, 1   -> R1
3. store  R1      -> counter
```

If two threads run `increment()` concurrently, the interleaving
might be:

```
T1: load  counter (0)
T2: load  counter (0)        ← T2 read before T1 stored
T1: add   1 -> 1
T2: add   1 -> 1
T1: store 1
T2: store 1                  ← overwrites T1's store
```

Two increments, one result. The lost update is invisible: no
error, no exception, just a wrong number. Run a million
increments across many threads and you'll consistently see a
final value lower than a million. The bigger the thread count,
the bigger the gap.

## Three kinds of race

Not all races look the same. Three useful categories:

<Steps>
<Step n="1" title="Read-modify-write race (the lost-update class)">

The counter example. A thread reads, computes, writes — and
in between, another thread did the same. The fix is to make
the whole read-modify-write atomic: a mutex, an atomic
fetch-and-add, a CAS loop. Most "counter is wrong" bugs are
this.

</Step>
<Step n="2" title="Check-then-act race (the TOCTOU class)">

```python
if not os.path.exists(path):
    create(path)             # someone else created it in between
```

Two operations that were supposed to be atomic — "test the
condition, then act on it" — but in between, another thread
or process changed the condition. The acronym **TOCTOU**
(time-of-check to time-of-use) is the security-flavored name;
the same shape is endemic in everyday code. The fix is
usually to combine the check and the action into one atomic
call (`O_CREAT | O_EXCL`, `INSERT … ON CONFLICT`,
`putIfAbsent`).

</Step>
<Step n="3" title="Publication race (the visibility class)">

One thread writes; another reads; the writer's store may not
yet be visible. This is a subtler memory-model issue:

```c
// T1: init
data = compute();
ready = true;          // T2 might see ready=true but data=stale

// T2: read
if (ready) use(data);
```

Even with no overlap, T2 can see `ready=true` before `data`
finished being written, because of CPU/compiler reordering.
The fix is a memory barrier, an atomic-release/atomic-acquire
pair, or a higher-level primitive that publishes the data
safely (mutex, channel, immutable swap).

</Step>
</Steps>

## "But I have a single-threaded language"

Even single-threaded async languages have races. The pattern is
different: not "between OS threads" but "between two tasks that
yield at an `await`."

```js
async function transfer(from, to, amount) {
  const a = await getBalance(from);
  if (a < amount) throw new Error('insufficient');
  await setBalance(from, a - amount);    // ← another task can run here
  await setBalance(to,   (await getBalance(to)) + amount);
}
```

While the first `await` is pending, the event loop is free to
schedule another `transfer` call on the same account. Both can
read the same balance, both pass the check, both decrement —
classic check-then-act in async clothing. The fix is the same:
serialize the operation (per-account mutex, atomic DB
transaction, queue).

## How to spot them

Some smells in a code review:

- Any unguarded read followed by a write to the same location
  shared by multiple actors.
- `if (cache[k] == null) cache[k] = compute(k);` — every cache
  with that shape has a duplication race; sometimes you don't
  care, sometimes you do.
- `if (condition) await ...; if (still_condition) commit();` —
  the condition can change during the await.
- "We add the row, then we update the count." Two writers
  doing this against the same count is a lost-update race.
- Memoization with concurrent readers but no lock around the
  fill.

## Why they're hard to debug

Three properties make races infamous:

1. **Non-determinism**. The bug only happens on certain
   interleavings, which depend on CPU load, scheduler choices,
   GC, OS noise. The test that fails in CI passes locally a
   thousand times.
2. **Silent corruption**. Most races don't throw — they
   produce a wrong-but-plausible value, which then propagates
   through your program. By the time something downstream
   breaks, the original race is hours gone from the logs.
3. **Adding logging fixes it**. Logs slow code down enough to
   change the timing. The Heisenbug effect is real.

## How to prevent them

In order of preference:

- **Don't share mutable state**. The fastest-and-safest fix is
  not having the bug. Immutable data, message passing, copying
  inputs, per-task ownership — these all eliminate whole bug
  classes by construction. Actor model, channels, functional
  cores: same idea in different clothing.
- **If you must share, share through a primitive built for it**.
  Atomics, mutexes, RWLocks, semaphores, channels, queues. The
  primitive's job is to make the operation atomic at a level
  meaningful to your program.
- **Make critical sections small**. The mutex's contract is
  cheap to misuse — hold it for a network call and you've
  serialized your whole system.
- **Use the language's race detector**. Go's `-race`, Rust's
  type system (`Sync`/`Send`), Java's `ThreadSanitizer`. They
  don't find all races but they find many before they reach
  production.
- **Test concurrently**. A test that increments a counter
  10,000 times from 100 threads will catch most read-modify-
  write races. `loom` (Rust), `pthread` stress tests, Jepsen
  (distributed) — pick the level that matches your code.

## The takeaway

A race is a behavior, not a bug location. It happens when two
or more concurrent actors touch shared state in an order the
code didn't pin down. The line of code that races looks fine in
isolation; the bug lives in the *gap* between operations that
should have been one. The fix is either to eliminate the
sharing or to make the sequence atomic — there's no
"defensive copy" or "if (rare) retry" middle ground that
actually works.
